## 最终目标（产品/工程口径）
- UI 查询路径 ：打开下拉框/搜索/翻页 → 后端查 MySQL → 返回 JSON（稳定、毫秒级、0 FB 配额）。
- FB API 只负责“进货” ：后台定时同步 + 按需手动刷新（可控、可限流、可观测）。
- 默认过滤逻辑（AdsPolar 口径） ：默认展示 ACTIVE/PAUSED/PENDING_REVIEW/DISAPPROVED/IN_PROCESS/WITH_ISSUES ，默认隐藏 DELETED/ARCHIVED ；当用户输入搜索词（q 非空）时，允许把隐藏状态也搜出来（精准可控）。
- 与现有数据口径兼容 ：你们的 ad_snapshots 仍然可以继续 “today 只落 spend>0”（这是指标热数据口径）；结构镜像表独立存在， 不受 spend=0 影响 ，专门服务选择器与规则作用对象。
## 方案总览（两条线并行）
### 1）结构镜像（Structure Mirror DB，新增三张表）
新增并维护三张“结构镜像表”（可以叫 structure_campaigns / structure_adsets / structure_ads ，避免和现有业务表混淆）：

- structure_campaigns : account_id, campaign_id, name, effective_status, status, updated_time, last_synced_at
- structure_adsets : account_id, adset_id, campaign_id, name, effective_status, status, daily_budget, updated_time, last_synced_at
- structure_ads : account_id, ad_id, adset_id, campaign_id, name, effective_status, status, configured_status, updated_time, last_synced_at
索引建议（核心是 account 维度 + status + name/id 搜索）：

- (account_id, effective_status, name)
- (account_id, name) / (account_id, ad_id) / (account_id, adset_id) 等
- 如果你们数据量很大，再加 FULLTEXT(name) （可选）
### 2）同步管线（Sync Remote）
新增一条“结构同步”管线（模式完全复用你们已有的 cron + 限流思想，见 cronService.js ）：

- 定时同步 ：每 30min / 60min 按账户跑一次（低频、可控）。
- 增量补全 ：你们 today 同步本来就会发现“新 ad_id”，目前已经会调用 resolveObjectsByIds 补元数据（见 ingestorService.js:L295-L389 ）。把这里的“补全结果”升级为 upsert 到 structure_ads（并顺带 upsert 对应 adset/campaign） ，能把“鸡生蛋”彻底打通。
- 手动刷新 ：UI 一个“刷新列表”按钮 → 调后端触发该 account 的 structure sync（带账户锁+冷却，例如 2 分钟内不允许重复触发）。
## 后端 API 设计（关键：从 FB 改为读库）
把当前这两条接口都切到“读本地镜像表”，避免任何 UI 行为直打 FB：

### A. GET /api/structure/:level（从 DB 查）
行为：

- q 为空 ：默认只返回“运营需要的状态集合”（隐藏 DELETED/ARCHIVED）。
- q 非空 ：放宽状态（允许把 ARCHIVED/DELETED 搜出来），但仍然分页+limit 防爆。
- 分页：用 DB cursor（推荐 keyset： (name,id) 或 updated_time,id ），不要 OFFSET（大表会慢）。
返回：

- items: [{ id, name, effective_status, status, ...relationIds }]
- meta: { source: 'db', last_synced_at }
### B. GET /api/structure/resolve?ids=...（优先从 DB resolve）
你们现在的 resolve 会打 FB：见 index.js:L1911-L1925 。应改为：

- 先在本地镜像表按 id 批量查（ads/adsets/campaigns 三表 union）。
- DB 找不到的 id：返回 { id, missing: true } （前端提示“可能是新建/刚改名，点刷新”），而不是立刻去 FB 兜底（否则编辑规则一打开也会穿透）。
## 前端选择器（只做必要改造）
你们 RuleManager 当前已经具备：

- q 搜索输入、300ms debounce、分页 after、Refresh 列表按钮位（已经有“刷新列表”按钮 UI 入口：见 RuleManager.vue:L131-L136 ）
建议改造为：

- “刷新列表”按钮语义从“重新拉 FB”改成“刷新本地镜像（触发一次 sync）+ 然后重新查库”。
- 仍然调用 /api/structure/:level ，但它现在是 DB 查询，所以交互会非常丝滑，且不会耗 FB 配额。
## 落地顺序（最小风险、最快见效）
- Phase 0（立刻止血，可选） ：如果你现在只关心 Ad 级选择器，短期可以先让选择器改用你们已有的 GET /api/ads （它已经是查 ad_snapshots ，0 FB 配额）见 index.js:L1981-L2032 。缺点是它跟随热数据口径，可能只覆盖 spend>0/已出现过的广告。
- Phase 1（标准 SaaS 形态） ：上结构镜像三表 + 定时同步 + /api/structure 读库（彻底解决慢/贵/不可控）。
- Phase 2（体验完善） ：resolve 读库 + 手动刷新 + 同步可观测（记录每账户 last_synced_at、条数、耗时、失败原因）。
## 验收标准（你要的“不会自杀”可量化）
- q 为空：打开选择器、翻页、搜索（DB LIKE）全程 不产生 FB API 调用 （只看你们服务端日志即可验证）。
- 延迟：同账户同层级首次打开 < 100ms（通常 10–30ms），翻页 < 50ms。
- 同步：定时同步或手动刷新时才会调用 FB，且受你们现有的限流/并发策略控制（你们 ingestor 已有一整套 rate limit 防护，可复用）。
- 数据一致性：FB 新建广告在下次同步后可搜到；若运营立即要配规则，点一次“刷新列表”即可出现。
