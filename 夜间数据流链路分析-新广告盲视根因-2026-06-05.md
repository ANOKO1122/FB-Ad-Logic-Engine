# 夜间 vs 白天数据流对比 — 新广告盲视根因分析

> 日期：2026-06-05
> 关联：`6.5方案A-扩展嗅探覆盖时段-实现方案-2026-06-05.md`

---

## 一、核心问题

**新广告在夜间（00:00-06:00 本地时间）创建后，热数据表 `ad_snapshots` 不更新，规则引擎无法对其评估，直到 06:00 切换到白天路径后才"突然可见"，导致规则集中雪崩。**

---

## 二、白天 vs 夜间 完整数据流对比

### 2.1 白天路径：`syncAccountTodayStats`（Insights First）

```
syncAccountTodayStats(accountId)           // ingestorService.js:344
│
├─ ① 获取账户时区（优先 DB 缓存，无则 FB API 查询）
│
├─ ② 【核心】getAdInsights(accountId, {preset:'today'}, {       // 第 430 行
│        level: 'ad',
│        spendGreaterThanZero: true      // FB 端过滤：只要 spend>0
│        //  ⚠️ 关键: 不传 ad_id 参数！
│        //  FB 返回该账户下「所有 spend>0 的广告」，老广告和新广告一视同仁
│    })
│    └→ insights = [所有 spend>0 广告的 Today 数据]   ✅ 新广告直接在这里
│
├─ ③ 本地二次过滤 spend>0（兜底，防止 FB filtering 失效）
│    └→ if filtered.length===0 → return（合理：整个账户今天没花钱）
│
├─ ④ resolveObjectsByIds(insights中提取的ad_id列表)     // 获取 status/name
│    └→ 补全 campaign_id, adset_id, effective_status
│
├─ ⑤ enqueueWrite('SNAPSHOT', insights, {...})           // 入写入队列
│    └→ 异步写入 ad_snapshots 表                        ✅ 热数据落盘
│
└─ ⑥ 返回 { syncedCount: N }                             // 结束
```

**白天路径特点**：第一步就向 FB 发起「无 ad_id 限定」的账户级查询，FB 返回所有广告 → 不存在"先知道 ID 才能查"的问题 → 新广告天然覆盖。

---

### 2.2 夜间路径：`syncAccountSlidingWindow`（filterActiveAds + 嗅探）

```
syncAccountSlidingWindow(accountId, ownerId, tz, daysBack=7)   // ingestorService.js:940
│
├─ ① 获取账户时区（同白天）
│
├─ ② filterActiveAds(accountId, [], tz, daysBack)              // 第 1005 行
│    │  SELECT DISTINCT ad_id FROM daily_stats   WHERE spend>0 AND date>=近7天
│    │  UNION
│    │  SELECT DISTINCT ad_id FROM ad_snapshots  WHERE spend>0 AND data_date>=近7天
│    │
│    │  ⚠️ 查的是 DB 中"曾经花过钱"的广告。今天才创建的新广告不在 DB 中 → 查不到。
│    │  ⚠️ 但对持续运营的账户，过去7天的 daytime 同步已在 DB 中留下了老广告记录。
│    │     所以 filterActiveAds 通常会返回几十到几百个老广告 ID，不会返回0。
│    │     只有在全新账户或长期无花费账户上才会返回0。
│    │
│    └→ targetAdIds = [老广告ID1, 老广告ID2, ...]   // 不含今天的新广告
│
├─ ③ if (targetAdIds.length === 0) → return              // 第 1007 行 🚫
│    │  ⚠️ 边界情况：账户从未有过花费 → 直接退出
│    │  此时嗅探代码（步骤④）永远不会执行
│    │  影响范围：仅全新/长期无花费账户（natie 不属于此类，它有779个老广告）
│    └→ 退出
│
├─ ④ 【嗅探修复】getAdInsights(accountId, {preset:'today'}, {  // 第 1024 行
│        level: 'ad',
│        spendGreaterThanZero: true
│        //  和白天一样的账户级查询！理论上能发现新广告
│    })
│    │  ⚠️ 但这里有个关键局限：
│    │  1) FB API 有数据延迟（新广告创建后可能需要几分钟到几十分钟才有数据）
│    │  2) spendGreaterThanZero 过滤: 新广告 spend=0 时不会被返回
│    │  3) 如果 FB API 报错 → try/catch 静默降级
│    │
│    └→ discoveredIds = FB 返回的 spend>0 广告 ID（可能不含新广告）
│       newIds = discoveredIds - targetAdIds
│       if newIds.length>0 → targetAdIds 合并新 ID
│       if newIds.length==0 → targetAdIds 不变（仍是老广告列表）
│
├─ ⑤ resolveObjectsByIds(targetAdIds)                      // 获取 status
│    └→ 只解析 targetAdIds 中的广告
│
├─ ⑥ fetchInsightsInBatches(accountId, targetAdIds, ...)    // 第 1065 行 🚫
│    │  for each ad_id in targetAdIds:
│    │    GET /{ad_id}/insights?date_preset=today...
│    │
│    │  ⚠️ 致命限制：这里按 ad_id 逐个拉取。
│    │  如果新广告的 ID 不在 targetAdIds 中 → 永远不会被请求
│    │  新广告的 Today 数据 → 永远不会被拉取 → 永远不会写入 ad_snapshots
│    │
│    └→ todayInsights = [只含 targetAdIds 中广告的 Today 数据]
│
├─ ⑦ filter(todayInsights, spend>0)                         // 第 1070 行 🚫
│    │  if filtered.length===0 → 跳过写入，不调 saveSnapshotsToDb
│    │  ⚠️ 即使新广告 ID 已通过嗅探加入 targetAdIds，
│    │     如果它的 spend=0，也会在这一步被过滤掉
│    │
│    └─ ⑦ saveSnapshotsToDb(filtered, ...)                 // 第 1084 行
│         写入 ad_snapshots（热数据）
│         ⚠️ 只含 targetAdIds 中的老广告，不含新广告
│
└─ ⑧ fetchInsightsByDay + updateDailyStatsFromInsights      // 第 1092 行
      写入 daily_stats（冷数据，历史回补）
```

**夜间路径特点**：整个链路围绕「已知 ad_id 列表」运转。新广告 ID 只有步骤④一个入口，且受 FB API 延迟和 spend>0 限制。一旦步骤④未发现新广告，后续所有步骤都不会处理它。

## 三、为什么 filterActiveAds 夜间不会返回 0？

**结论：对持续运营的账户，`filterActiveAds` 夜间通常返回几十到几百个广告 ID，不会返回 0。**

原因：

```
filterActiveAds 查询的是"近 N 天内有 spend>0 的广告"
  │
  ├─ daily_stats    ← 白天 syncAccountTodayStats 写入了昨天的数据
  │                   昨天有花费的广告 → 在 daily_stats 中有记录
  │
  └─ ad_snapshots   ← 白天 syncAccountTodayStats 写入了今天的快照（如果今天白天有同步过）
                      今天凌晨之前的数据来自前一天的 daytime 同步
```

| 场景 | filterActiveAds 结果 | 举例 |
|------|---------------------|------|
| 持续运营的账户 | 返回 N 个（N≥1） | natie：779 个已知广告 |
| 24h 内刚激活的新账户 | 可能返回 0 | 刚绑定的全新 FB 账户 |
| 长期无花费后重新投放 | 可能返回 0 | 停投 2 周后重新开启 |
| 所有广告都在昨天被暂停 | 返回 N 个（历史有花费） | 暂停不影响历史数据 |

**natie 案例如下**：6月4日 05:00（上海），`filterActiveAds` 返回 **779 个**老广告 ID，正常通过了步骤③的检查，嗅探也执行了。问题不在步骤③的早期 return，而在步骤④的嗅探没有发现新广告（新广告此时可能 spend=0）。

## 四、夜间真正的问题：三个阻断点的实际影响

| 阻断点 | 位置 | 对 natie(活跃账户) | 对全新账户 | 对一般活跃账户 |
|:---:|------|:---:|:---:|:---:|
| ① len=0 提前 return | 第 1007 行 | ❌ 不触发（779>0） | 🔴 触发 | ❌ 不触发 |
| ② 嗅探未发现新广告 | 第 1024 行 | 🔴 **主因** | N/A(已被①阻断) | 🔴 **主因** |
| ③ spend>0 过滤跳过 | 第 1070 行 | ❌ 不触发（老广告有 spend） | N/A | ⚠️ 新广告 spend=0 时触发 |

**对 natie 这类活跃账户，核心问题是阻断点②：嗅探没有发现新广告。** 原因可能是：
- 新广告在凌晨 00:00-05:00 创建，FB API 数据有延迟
- 新广告刚开始投放，前几个小时 spend 为 0 或极小
- `spendGreaterThanZero: true` 过滤掉了 spend=0 的新广告

## 六、白话总结：到底发生了什么

### 6.1 一个比喻：图书馆的两种找书法

**白天模式（直接问管理员）**：
你走到图书馆前台说"把今天所有借出去的书列给我"。管理员查系统，1秒钟给你完整清单。不管书是什么时候入库的，只要今天有人借，你都能知道。

**夜间模式（先翻自己的小本本）**：
你先翻自己口袋里的笔记本，上面记着"之前借过的书有哪些"。然后拿着这个清单去找管理员，逐一问"这本书今天借出去了吗？" 如果有一本书你笔记本上没记，你就永远不会去问管理员这本书的情况。

→ 这就是白天和夜间的本质区别。白天直接问 FB"有哪些广告"，夜间先查自己数据库"我知道哪些广告"，再逐个去问 FB。

### 6.2 新广告为什么夜间"隐身"

```
假设一个广告在晚上被 FB 批准通过，凌晨 2 点开始有花费：

凌晨 2:00  广告开始投放，产生了 $0.5 的花费
           │
凌晨 2:15  心跳触发夜间同步:
           │  ① 查数据库 → "我知道的广告列表" → 没有这个新广告（今天刚有花费）
           │  ③ 嗅探: 问 FB "今天哪些广告花钱了" → FB 可能返回它
           │     ⚠️ 但 FB 数据有延迟，刚产生花费的广告可能要 30 分钟才能查到
           │     ⚠️ 且系统每小时只成功跑 1-2 轮（62个账户排队太慢）
           │  → 大概率本轮错过了这个新广告
           │
凌晨 3:00  心跳可能又在排队/跳过...
           │
凌晨 5:00  终于又跑了一轮同步，但此时新广告的数据在 FB 端可能已经更新了
           │  嗅探发现了它！
           │  → 广告数据写入 ad_snapshots ✅
           │  → 下分钟规则引擎评估 → 命中条件 → 暂停 ✅
           │
          （理想情况：新广告在 5:00 就被发现和暂停）
          
          （实际情况：6月4日 natie 的 125 个广告，在 05:00 的嗅探中没被发现
             FB API 返回的 spend>0 广告里没有它们 → 可能花费还不够或数据延迟）
```

### 6.3 为什么夜间只跑 1-2 次嗅探

不是系统不想跑，是**跑不过来**：

- 系统有 64 个账户要处理
- 夜间时段（比如北京时间凌晨 0-6 点），**62 个亚洲账户同时处于夜间**
- 每个账户的同步需要 30-60 秒（要查 DB、调 FB API、写数据）
- 系统同时最多处理 6 个账户
- 一轮完整同步所有账户 → 需要 5-10 分钟
- 如果一轮没跑完，下一轮来了 → 跳过（防止重叠）
- 6 小时的夜间窗口里，可能只有 1-2 轮真正全部跑完
- **每轮跑完时才会对每个账户执行一次嗅探**
- → 所以每个账户的嗅探在夜间只跑了 1-2 次

### 6.4 广告不是夜间创建的，为什么也受影响？

你说得对，广告不一定是凌晨新建的。关键是：**夜间同步不看你广告什么时候建的，只看它是否已经在你的数据库"笔记本"里。**

一个广告可能昨天白天就创建了，但：
- 昨天白天花费为 0 → 白天同步没拉它 → DB 里没记录
- 到了晚上才开始产生花费 → 但夜间同步查 DB 找不到它 → 嗅探可能发现也可能错过
- 直到第二天早上 6 点，白天模式启动，直接问 FB"今天有哪些广告" → 全部揪出来

**所以不是"夜间创建的广告"有问题，而是"夜间首次产生花费、但 DB 还没记录的广告"有问题。**

### 6.5 方案A 做了什么

在夜间（UTC 15-23，对应北京时间 23:00-07:00）新增一条**独立通道**，每 15 分钟对所有账户跑一次滑动窗口同步（含嗅探）。

```
原来: 夜间 6 小时 → 1-2 轮嗅探 → 新广告可能 5 小时都没被发现
现在: 夜间 6 小时 → 24+ 轮嗅探 → 新广告最多 15 分钟就能被发现
```

**效果**：新广告一旦产生花费，15 分钟内就会被嗅探发现 → 写入数据库 → 规则引擎下分钟就能评估 → 该暂停就暂停，不会再积压到早上 6 点集中触发。
