## 问题3：统一心跳锁状态异常（严重问题）

### 问题描述
- **现象**：锁状态显示"既不是空闲，也不是被占用，但无法获取"
- **影响**：导致统一心跳任务无法执行，系统卡住
- **位置**：`server/services/ingestorService.js` 第2610-2637行 和 第2984-2993行

### 根本原因

**核心问题**：MySQL 的 `GET_LOCK` 和 `RELEASE_LOCK` 必须在**同一个连接**上操作！

**问题代码流程**：
1. 第2614行：使用 `connection = await pool.getConnection()` 获取连接A
2. 第2615行：在连接A上获取锁 `GET_LOCK`
3. 第2635行：在 finally 中立即 `connection.release()`，**但此时锁还在连接A上**
4. 第2987行：使用 `releaseConnection = await pool.getConnection()` 获取了**新的连接B**
5. 第2988行：在连接B上尝试 `RELEASE_LOCK`

**问题分析**：
- 连接A已经释放回连接池，锁被连接A持有
- 连接B是新连接，无法释放连接A上的锁
- 导致锁处于"被占用但无法释放"的异常状态

### 永久解决方案

**修复策略**：
1. 使用专用连接获取和释放锁（在整个函数执行期间保持连接）
2. 必须在获取锁的同一连接上释放锁
3. 不能在获取锁后立即释放连接，必须等到释放锁后再释放连接

**修复代码**：

```javascript
// 修复前（错误）
let connection = null
try {
  connection = await pool.getConnection()
  const [lockRows] = await connection.query(`SELECT GET_LOCK(?, 0) AS acquired`, [HEARTBEAT_LOCK_NAME])
  lockAcquired = lockRows[0]?.acquired === 1
} finally {
  if (connection) {
    connection.release()  // ❌ 错误：锁还在连接上就释放了连接
  }
}

// ... 执行任务 ...

finally {
  if (lockAcquired) {
    const releaseConnection = await pool.getConnection()  // ❌ 错误：新连接无法释放旧连接的锁
    await releaseConnection.query(`SELECT RELEASE_LOCK(?)`, [HEARTBEAT_LOCK_NAME])
    releaseConnection.release()
  }
}

// 修复后（正确）
let lockConnection = null
try {
  lockConnection = await pool.getConnection()
  const [lockRows] = await lockConnection.query(`SELECT GET_LOCK(?, 0) AS acquired`, [HEARTBEAT_LOCK_NAME])
  lockAcquired = lockRows[0]?.acquired === 1
  // ✅ 正确：获取锁后，不释放连接，保持连接直到释放锁
} catch (lockError) {
  if (lockConnection) {
    lockConnection.release()  // ✅ 正确：只有获取锁失败时才释放连接
    lockConnection = null
  }
}

// ... 执行任务 ...

finally {
  if (lockAcquired && lockConnection) {
    try {
      // ✅ 正确：在同一连接上释放锁
      await lockConnection.query(`SELECT RELEASE_LOCK(?)`, [HEARTBEAT_LOCK_NAME])
    } finally {
      // ✅ 正确：释放锁后再释放连接
      lockConnection.release()
      lockConnection = null
    }
  }
}
```

**修改位置**：
- `server/services/ingestorService.js` 第2610-2637行（获取锁）
- `server/services/ingestorService.js` 第2984-2993行（释放锁）

## 预防措施

### 锁管理最佳实践
1. **同一连接原则**：`GET_LOCK` 和 `RELEASE_LOCK` 必须在同一连接上操作
2. **连接生命周期**：获取锁后，连接必须保持到释放锁
3. **异常处理**：确保在 finally 中正确释放锁和连接
4. **锁超时机制**：考虑为锁设置 TTL，自动释放（未来优化）

### 代码审查检查点
- [ ] 获取锁和释放锁是否在同一连接上？
- [ ] 连接是否在释放锁之前就被释放？
