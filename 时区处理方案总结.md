## 📋 时区使用策略

### 1. 数据存储时区（后端）

**原则：数据时区优先**

- **`ad_snapshots.timezone_name`**：存储数据同步时的时区（从 Facebook API 获取）
- **`daily_stats.timezone_name`**：存储归档时的时区（与数据时区一致）
- **`account_mappings.timezone_name`**：存储账户配置时区（从 Facebook API 自动更新）

**查询策略：**
- `today` 查询：优先使用 `ad_snapshots` 最新快照的 `timezone_name`（数据时区优先）
- `last_7_days` / `last_30_days` 查询：使用 `daily_stats.timezone_name` 或 `account_mappings.timezone_name`

---

### 2. 规则执行时区（后端）

**原则：使用账户时区**

- 规则执行时，使用 `account_mappings.timezone_name` 或从 Facebook API 获取的时区
- 确保规则判断的时间窗口与账户实际时区对齐
- 例如：`today` 规则在 `Asia/Shanghai` 时区下，指的是北京时间 00:00-23:59

---

### 3. 前端展示时区（前端）

**原则：统一转换为北京时区（Asia/Shanghai）**

- **数据展示**：所有时间、日期统一转换为 `Asia/Shanghai` 时区显示
- **用户界面**：前端显示的时间都是北京时间，便于用户理解
- **数据来源**：后端返回的数据可能使用账户时区，前端负责转换

**实现建议：**
```javascript
// 前端时区转换示例（使用 Luxon 或 dayjs）
import { DateTime } from 'luxon'

// 将账户时区的时间转换为北京时区显示
function formatTimeForDisplay(timestamp, accountTimezone) {
  const dt = DateTime.fromISO(timestamp, { zone: accountTimezone })
  return dt.setZone('Asia/Shanghai').toFormat('yyyy-MM-dd HH:mm:ss')
}
```

## 🔄 时区更新流程

### 自动更新流程

1. **定时同步触发**（每 15 分钟）
   - `cronService.js` 调用 `syncAllAccountsTodayStats()`
   - 遍历所有活跃账户，调用 `syncAccountTodayStats()`
   - 每个账户同步时，自动从 Facebook API 获取时区
   - 如果时区不一致，自动更新 `account_mappings.timezone_name`

2. **手动同步触发**
   - 调用 `syncAccountTodayStats(accountId, ownerId, timezoneName)`
   - 即使传入 `timezoneName`，也会从 API 获取最新时区并更新

3. **滑动窗口同步**（如果使用）
   - `syncAccountSlidingWindow()` 也会自动更新时区

## 🎯 前端时区转换实现建议

### 方案 1：前端统一转换（推荐）

**优点：**
- 用户体验一致（所有时间都是北京时间）
- 前端逻辑简单（统一转换函数）

**实现：**
```javascript
// utils/timezone.js
import { DateTime } from 'luxon'

/**
 * 将任意时区的时间转换为北京时区显示
 * @param {string|Date} timestamp - 时间戳或日期对象
 * @param {string} sourceTimezone - 源时区（如 'America/Los_Angeles'）
 * @returns {string} 格式化后的北京时间字符串
 */
export function toBeijingTime(timestamp, sourceTimezone = 'UTC') {
  const dt = DateTime.fromJSDate(
    timestamp instanceof Date ? timestamp : new Date(timestamp),
    { zone: sourceTimezone }
  )
  return dt.setZone('Asia/Shanghai').toFormat('yyyy-MM-dd HH:mm:ss')
}

/**
 * 将任意时区的时间转换为北京时区的日期字符串
 * @param {string|Date} timestamp - 时间戳或日期对象
 * @param {string} sourceTimezone - 源时区
 * @returns {string} 格式化后的日期字符串（yyyy-MM-dd）
 */
export function toBeijingDate(timestamp, sourceTimezone = 'UTC') {
  const dt = DateTime.fromJSDate(
    timestamp instanceof Date ? timestamp : new Date(timestamp),
    { zone: sourceTimezone }
  )
  return dt.setZone('Asia/Shanghai').toFormat('yyyy-MM-dd')
}
```

**使用示例：**
```vue
<template>
  <div>
    <p>同步时间：{{ displaySyncTime(ad.synced_at, accountTimezone) }}</p>
    <p>日期：{{ displayDate(ad.date, accountTimezone) }}</p>
  </div>
</template>

<script>
import { toBeijingTime, toBeijingDate } from '@/utils/timezone'

export default {
  methods: {
    displaySyncTime(timestamp, accountTimezone) {
      return toBeijingTime(timestamp, accountTimezone)
    },
    displayDate(date, accountTimezone) {
      return toBeijingDate(date, accountTimezone)
    }
  }
}
</script>
```

### 方案 2：后端返回已转换的时间（备选）

**优点：**
- 前端无需转换逻辑
- 后端统一处理，减少前端复杂度

**缺点：**
- 后端需要知道前端展示时区（耦合）
- 如果前端需要支持多时区，需要修改后端

**实现：**
```javascript
// server/index.js 或 server/services/ruleDataService.js
// 在返回数据前，将时间转换为前端展示时区
const displayTimezone = 'Asia/Shanghai'
const displayTime = DateTime.fromJSDate(data.synced_at, { zone: data.timezone_name })
  .setZone(displayTimezone)
  .toISO()
```

## 📝 注意事项

### 1. 历史数据时区

- **历史数据可能保留旧时区**（如 `UTC`），这是正常的
- **查询时会使用数据时区优先策略**，确保查询结果正确
- **不需要手动更新历史数据的时区**

### 2. 多账户时区

- **不同账户可能有不同时区**（如 `Asia/Shanghai`、`America/Los_Angeles`）
- **前端展示时统一转换为 `Asia/Shanghai`**
- **规则执行时使用账户实际时区**

### 3. 时区更新频率

- **每次数据同步时都会检查并更新时区**
- **如果 Facebook API 返回的时区与数据库不一致，会自动更新**
- **不需要手动维护时区配置**
