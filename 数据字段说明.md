# 数据字段说明文档

> 本文档说明系统中收集的数据字段及其处理逻辑

---

## 一、数据收集字段清单

系统目前收集以下 **10 个核心字段**（全部可以直接从 Facebook API 获取）：

| 序号 | 字段名 | 数据库字段 | 数据来源 | 说明 |
|------|--------|-----------|---------|------|
| 1 | 状态 | `status` | `/ads` API 的 `effective_status` 或 `status` | 广告当前投放状态（ACTIVE/PAUSED 等） |
| 2 | 花费 | `spend` | `insights` API 的 `spend` | 该广告累计消耗的金额 |
| 3 | 成效(购买) | `purchases` | `insights` API 的 `actions` 字段中提取 | `offsite_conversion.fb_pixel_purchase` 次数 |
| 4 | CPA | `cpa` | `insights` API 的 `cost_per_action_type` | 单次购买成本 |
| 5 | ROAS | - | `insights` API 的 `roas` 或 `action_values` | **不入库**，见下方说明 |
| 6 | CPC | - | `insights` API 的 `cpc` | **不入库**，见下方说明 |
| 7 | uCPC | `ucpc` | `insights` API 的 `cost_per_unique_link_click` | 独立单次链接点击费用 |
| 8 | 加购费 | `add_to_cart_cost` | `insights` API 的 `cost_per_action_type` | 单次加入购物车成本 |
| 9 | 结账费 | `checkout_cost` | `insights` API 的 `cost_per_action_type` | 单次发起结账成本 |
| 10 | 支付费 | `payment_cost` | `insights` API 的 `cost_per_action_type` | 单次添加支付信息成本 |

## 二、特殊字段处理逻辑

### 2.1 CPC（单次点击成本）

#### 处理原则
**CPC 不入库**，采用"当天从 API 获取，历史数据通过计算"的策略。

#### 当天数据
- **数据来源**：直接从 Facebook API 的 `insights` 接口获取 `cpc` 字段
- **获取方式**：在 Batch API 请求中包含 `cpc` 字段
- **存储方式**：**不存储到数据库**，直接从 API 返回给前端展示

#### 历史数据
- **计算逻辑**：`CPC = 花费 / 点击量`
- **数据来源**：从 `daily_stats` 表或历史数据中获取：
  - 花费：`daily_stats.spend`
  - 点击量：需要从 `daily_stats` 表的 `actions` JSON 字段中提取 `link_click` 的次数
- **计算时机**：查询历史数据时实时计算

#### 为什么不入库？
- **时间范围问题**：看过去一周的数据时，需要计算那一周的总花费除以总点击量，才能得出正确的 CPC
- **存储冗余**：如果存储每天的 CPC，无法准确反映一周的 CPC（因为 CPC 是平均值，不是累加值）
- **计算更准确**：通过计算得出的 CPC 更符合业务需求（例如：一周总花费 $100，总点击 200 次，CPC = $0.5）

---

### 2.2 ROAS（广告支出回报率）

#### 处理原则
**ROAS 不入库**，采用"当天从 API 获取，历史数据通过计算"的策略。

#### 当天数据
- **数据来源**：
  1. 优先使用 Facebook API 直接返回的 `roas` 字段
  2. 如果没有，从 `action_values` 计算：`ROAS = 购买总金额 / 花费`
- **获取方式**：在 Batch API 请求中包含 `roas` 和 `action_values` 字段
- **存储方式**：**不存储到数据库**，直接从 API 返回给前端展示

#### 历史数据
- **计算逻辑**：`ROAS = 购买总金额 / 花费`
- **数据来源**：从 `daily_stats` 表或历史数据中获取：
  - 购买总金额：从 `daily_stats` 表的 `action_values` JSON 字段中提取 `offsite_conversion.fb_pixel_purchase` 的 `value`
  - 花费：`daily_stats.spend`
- **计算时机**：查询历史数据时实时计算

#### 为什么不入库？
- **时间范围问题**：看过去一周的数据时，需要计算那一周的总购买金额除以总花费，才能得出正确的 ROAS
- **存储冗余**：如果存储每天的 ROAS，无法准确反映一周的 ROAS（因为 ROAS 是比值，不是累加值）
- **计算更准确**：通过计算得出的 ROAS 更符合业务需求（例如：一周总花费 $100，总购买金额 $250，ROAS = 2.5）

## 三、数据获取方式

### 3.1 Facebook API 请求字段

在 `server/services/ingestorService.js` 的 `fetchInsightsInBatches` 函数中，Batch API 请求包含以下字段：

```javascript
const fields = 'ad_id,ad_name,spend,cpc,roas,actions,action_values,cost_per_action_type,cost_per_unique_link_click,cost_per_unique_inline_link_click'
```

**注意**：
- 虽然请求了 `cpc` 和 `roas`，但这两个字段**不会写入数据库**
- 这两个字段仅用于当天数据的实时展示

### 3.2 数据提取逻辑

#### 从 `cost_per_action_type` 提取成本字段

`cost_per_action_type` 是一个数组，包含各种 action_type 的成本。使用 `pickCostPerActionType` 函数提取：

```javascript
// CPA：从 cost_per_action_type 中提取 purchase 的成本
const cpa = pickCostPerActionType(costPerActionType, [
  'offsite_conversion.fb_pixel_purchase',
  'purchase'
])

// 加购费：从 cost_per_action_type 中提取 add_to_cart 的成本
const addToCartCost = pickCostPerActionType(costPerActionType, [
  'offsite_conversion.fb_pixel_add_to_cart',
  'add_to_cart'
])

// 结账费：从 cost_per_action_type 中提取 initiate_checkout 的成本
const checkoutCost = pickCostPerActionType(costPerActionType, [
  'offsite_conversion.fb_pixel_initiate_checkout',
  'initiate_checkout'
])

// 支付费：从 cost_per_action_type 中提取 add_payment_info 的成本
const paymentCost = pickCostPerActionType(costPerActionType, [
  'offsite_conversion.fb_pixel_add_payment_info',
  'add_payment_info'
])
```

#### 从 `actions` 提取购买次数

```javascript
// 从 actions 数组中提取购买次数
const purchases = parseActions(actions)
// parseActions 函数会精确匹配 'offsite_conversion.fb_pixel_purchase'
```

#### 提取 uCPC

```javascript
// uCPC：优先使用 cost_per_unique_link_click，如果没有则使用 cost_per_unique_inline_link_click
const ucpc = extractUcpc(insight)
```

## 四、防御性编程要点

### 4.1 Facebook API 的"不值不显"原则

**重要**：Facebook API 有一个"不值不显"的原则：
- 如果某个指标的值是 0，API 会直接把这个字段藏起来（不返回），而不是返回 0
- 所以在写代码时，必须做好防御性编程

#### ❌ 错误写法
```javascript
// 如果今天没单，这行代码会报错导致页面白屏
const cpa = data.insights[0].cost_per_action_type.find(x => x.action_type === 'purchase').value;
```

#### ✅ 正确写法
```javascript
// 先找，找不到就给默认值 null（而不是 0）
const cpaObj = data.insights[0].cost_per_action_type?.find(x => x.action_type === 'purchase');
const cpa = cpaObj ? parseFloat(cpaObj.value) : null;
```

### 4.2 代码中的防御性编程

所有成本字段的提取都使用了防御性编程：

```javascript
function pickCostPerActionType(costPerActionType, actionTypes) {
  // 防御性编程：如果 costPerActionType 不存在或不是数组，返回 null
  if (!costPerActionType || !Array.isArray(costPerActionType)) {
    return null
  }
  
  // ... 查找逻辑 ...
  
  // 如果都没找到，返回 null（而不是 0，因为 Facebook API 可能不返回该字段）
  return null
}
```

## 五、数据库表结构

### `ad_snapshots` 表（热数据：Today）

```sql
CREATE TABLE `ad_snapshots` (
  `id` INT AUTO_INCREMENT PRIMARY KEY,
  `account_id` VARCHAR(50) NOT NULL,
  `ad_id` VARCHAR(50) NOT NULL,
  `ad_name` VARCHAR(500),
  `status` VARCHAR(50),                    -- 广告状态
  `owner_id` INT NOT NULL,
  `spend` DECIMAL(10,2) DEFAULT 0.00,     -- 花费
  `ucpc` DECIMAL(10,4),                    -- 独立单次链接点击费用
  `cpa` DECIMAL(10,4),                     -- 单次购买成本
  `actions` JSON,                          -- 所有动作数据
  `purchases` INT DEFAULT 0,              -- 购买次数
  `add_to_cart_cost` DECIMAL(10,4),       -- 加购费
  `checkout_cost` DECIMAL(10,4),           -- 结账费
  `payment_cost` DECIMAL(10,4),            -- 支付费
  `sync_session_id` VARCHAR(100) NOT NULL,
  `synced_at` DATETIME NOT NULL,
  `timezone_name` VARCHAR(50) DEFAULT 'UTC',
  `mute_until` DATETIME,
  `mute_reason` VARCHAR(255),
  `is_simulation` TINYINT(1) DEFAULT 0,
  UNIQUE KEY `uk_ad_session` (`ad_id`, `sync_session_id`),
  INDEX `idx_account_id` (`account_id`),
  INDEX `idx_owner_id` (`owner_id`),
  INDEX `idx_synced_at` (`synced_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**注意**：表中**不包含** `cpc` 和 `roas` 字段，这两个字段不入库。

## 六、查询历史数据时的计算逻辑

### 6.1 查询 CPC（历史数据）

```sql
-- 示例：查询过去一周的 CPC
SELECT 
  ad_id,
  ad_name,
  SUM(spend) as total_spend,
  -- 从 actions JSON 中提取 link_click 的总次数
  JSON_EXTRACT(actions, '$[*].value') as clicks,  -- 需要根据实际 JSON 结构调整
  CASE 
    WHEN clicks > 0 THEN SUM(spend) / clicks
    ELSE NULL
  END as cpc
FROM daily_stats
WHERE date >= DATE_SUB(CURDATE(), INTERVAL 7 DAY)
GROUP BY ad_id, ad_name;
```

### 6.2 查询 ROAS（历史数据）

```sql
-- 示例：查询过去一周的 ROAS
SELECT 
  ad_id,
  ad_name,
  SUM(spend) as total_spend,
  -- 从 action_values JSON 中提取 purchase 的总金额
  JSON_EXTRACT(action_values, '$[*].value') as purchase_value,  -- 需要根据实际 JSON 结构调整
  CASE 
    WHEN total_spend > 0 THEN purchase_value / total_spend
    ELSE NULL
  END as roas
FROM daily_stats
WHERE date >= DATE_SUB(CURDATE(), INTERVAL 7 DAY)
GROUP BY ad_id, ad_name;
```

## 七、总结

1. **CPC 和 ROAS 不入库**：这两个字段当天数据从 API 获取，历史数据通过计算得出
2. **其他 8 个字段入库**：状态、花费、成效(购买)、CPA、uCPC、加购费、结账费、支付费
3. **防御性编程**：所有字段提取都做好空值判断，处理 Facebook API 的"不值不显"原则
4. **历史数据计算**：查询历史数据时，CPC 和 ROAS 通过 SQL 计算得出，确保时间范围内的准确性
