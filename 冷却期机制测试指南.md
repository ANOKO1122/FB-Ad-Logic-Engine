# 冷却期机制测试指南

## ✅ 可以使用前端创建规则进行测试

是的，完全可以通过前端创建一个模拟运行的规则来测试冷却期机制！

---

## 🎯 测试步骤

### 步骤 1：通过前端创建测试规则

1. **打开前端页面**：访问规则管理页面（通常是 `/rules` 或规则管理菜单）

2. **点击"新建规则"按钮**

3. **填写规则信息**：
   - **规则名称**：`测试规则-冷却期验证`
   - **条件**：任意条件（例如：`spend > 0`）
   - **操作**：任意操作（例如：`暂停广告`）
   - **模拟运行**：✅ **勾选"模拟运行"**（重要！这样不会真正执行操作）
   - **启用**：✅ **勾选"启用"**（重要！只有启用的规则才会被执行）

4. **保存规则**

---

### 步骤 2：验证规则已创建

在 MySQL 中执行：

```sql
-- 检查规则是否创建成功
SELECT id, rule_name, enabled, is_simulation, last_executed_at 
FROM rules 
WHERE enabled = 1 
ORDER BY id DESC 
LIMIT 5;
```

**预期结果**：
- 应该能看到刚才创建的规则
- `enabled = 1`（已启用）
- `is_simulation = 1`（模拟运行模式）
- `last_executed_at = NULL`（从未执行过）

---

### 步骤 3：验证冷却期机制

在 Node.js REPL 中执行：

```javascript
import('./server/services/cronService.js').then(async (m) => {
  console.log('=== 第一次执行（非强制模式，遵循冷却期）===')
  await m.manualExecute(false)
  
  console.log('\n=== 等待 2 秒后第二次执行（应该被冷却期过滤）===')
  await new Promise(resolve => setTimeout(resolve, 2000))
  await m.manualExecute(false)
})
```

**预期结果**：

## 📊 验证要点

### 1. 第一次执行应该成功

- ✅ 应该看到规则执行日志
- ✅ 应该看到"冷却期过滤: X 条规则 → Y 条可执行"的日志
- ✅ 规则会被执行（模拟运行模式，不会真正执行操作）

### 2. 第二次执行应该被过滤

- ✅ 应该看到"所有规则都在冷却期内，跳过执行"的日志
- ✅ 规则不会被执行（因为15分钟冷却期未到）

### 3. 强制模式应该忽略冷却期

## ⚠️ 注意事项

1. **模拟运行模式**：
   - 勾选"模拟运行"后，规则不会真正执行操作（不会暂停广告）
   - 只会记录日志，用于测试验证

2. **规则必须启用**：
   - 只有 `enabled = 1` 的规则才会被执行
   - 创建规则时记得勾选"启用"

3. **冷却期时间**：
   - 冷却期为 15 分钟
   - 如果第二次执行在 15 分钟内，应该被过滤
   - 如果等待超过 15 分钟，可以再次执行

## 🎯 完整测试流程

### 测试 1：验证冷却期机制（基本）

1. 前端创建一条测试规则（模拟运行 + 启用）
2. 第一次执行：`manualExecute(false)`
3. 立即第二次执行：`manualExecute(false)`（应该被过滤）

### 测试 2：验证强制模式（忽略冷却期）

1. 第一次执行：`manualExecute(false)`
2. 立即第二次执行：`manualExecute(true)`（应该执行，因为强制模式）

### 测试 3：验证冷却期过期

1. 第一次执行：`manualExecute(false)`
2. 等待 16 分钟（超过15分钟冷却期）
3. 第二次执行：`manualExecute(false)`（应该执行，因为冷却期已过）
