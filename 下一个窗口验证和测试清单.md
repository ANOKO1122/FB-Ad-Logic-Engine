## 🔍 步骤4：SQL 综合验证

### 4.1 验证 data_date 查询口径

**执行 SQL**（在 MySQL 中执行 `验证SQL查询.sql` 的第三个查询）：
```sql
SELECT 
  account_id,
  COUNT(DISTINCT ad_id) as ad_count,
  SUM(spend) as total_spend,
  MAX(data_date) as latest_data_date
FROM ad_snapshots
WHERE data_date = CURDATE()
GROUP BY account_id
ORDER BY account_id
LIMIT 20;
```

**检查点**：
- [ ] 应该能看到今天的数据，latest_data_date 应该是今天

**预期结果**：
- ✅ data_date 查询口径正确

### 4.2 验证没有新的僵尸数据

**执行 SQL**（在 MySQL 中执行 `验证SQL查询.sql` 的第四个查询）：
```sql
SELECT 
  account_id,
  COUNT(*) as zombie_count
FROM ad_snapshots
WHERE data_date = CURDATE()
  AND spend = 0
  AND (link_clicks IS NULL OR link_clicks = 0)
  AND (purchases IS NULL OR purchases = 0)
GROUP BY account_id
HAVING zombie_count > 0;
```

**检查点**：
- [ ] 应该返回 0 行（或只有历史数据，data_date 不是今天）

**预期结果**：
- ✅ 没有新的僵尸数据

### 4.3 对比 synced_at 和 data_date 的差异

**执行 SQL**（在 MySQL 中执行 `验证SQL查询.sql` 的第五个查询）：
```sql
SELECT 
  account_id,
  ad_id,
  data_date,
  DATE(synced_at) as synced_date_utc,
  spend,
  synced_at
FROM ad_snapshots
WHERE data_date < DATE(synced_at)
  AND synced_at >= DATE_SUB(NOW(), INTERVAL 24 HOUR)
ORDER BY synced_at DESC
LIMIT 20;
```

**检查点**：
- [ ] 这些是历史数据（data_date 是昨天或更早），但 synced_at 被更新了
- [ ] 这解释了为什么之前查询 `synced_at >= 24小时前` 会看到 spend=0 的历史数据

**预期结果**：
- ✅ 理解 synced_at 和 data_date 的差异是正常的
