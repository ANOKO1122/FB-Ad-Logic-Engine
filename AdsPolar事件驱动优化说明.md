# AdsPolar 事件驱动优化说明

## 🎯 优化目标

根据 AdsPolar 的设计理念，将规则执行从"批量模式"改为"事件驱动模式"：
- **优化前**：数据同步完成后，批量触发所有账户的规则执行
- **优化后**：每个账户数据同步完成后，立即触发该账户的规则执行

---

## 📊 优化对比

### 优化前（批量模式）

```
统一心跳 (10:00)
  ├─ 同步账户 A 数据 ✅
  ├─ 同步账户 B 数据 ✅
  ├─ 同步账户 C 数据 ✅
  └─ ... (36个账户)
  
  等待所有账户同步完成...
  
  批量触发规则执行
  ├─ 执行账户 A 规则
  ├─ 执行账户 B 规则
  └─ 执行账户 C 规则
```

**问题**：
- 需要等待所有账户同步完成
- 批量执行可能导致资源竞争
- 26个账户超时（5分钟）

---

### 优化后（事件驱动模式）

```
统一心跳 (10:00)
  ├─ 同步账户 A 数据 ✅
  │   └─ 立即触发账户 A 规则执行 ✅
  ├─ 同步账户 B 数据 ✅
  │   └─ 立即触发账户 B 规则执行 ✅
  └─ 同步账户 C 数据 ✅
      └─ 立即触发账户 C 规则执行 ✅
```

**优势**：
- ✅ 数据刚落库，立即评估规则（最准确）
- ✅ 分散执行，避免批量竞争
- ✅ 不会超时（每个账户独立执行）
- ✅ 符合 AdsPolar "顺手触发"理念

---

## 🔧 实现方案

### 1. 反向索引优化（已完成）

**位置**：`server/services/cronService.js` - `executeAllRules`

**优化**：
- 先查询所有启用的规则
- 分析规则，确定需要执行的账户
- 只在实际需要的账户上执行

**效果**：
- 从36个账户减少到1个账户（如果规则指定了账户）
- 减少35个账户的不必要执行

---

### 2. 事件驱动优化（新增）

**位置**：`server/services/ingestorService.js` - `unifiedHeartbeatSync`

**优化**：
- 每个账户数据同步完成后，立即触发规则执行
- 使用 `setImmediate` 异步执行，不阻塞数据同步流程

**代码**：

```javascript
// 2.4 AdsPolar 事件驱动优化：数据同步完成后立即触发规则执行
if (synced && dataUpdated) {
  // 异步触发规则执行（不等待完成，避免阻塞数据同步）
  setImmediate(async () => {
    try {
      const { executeRulesForAccount } = await import('./cronService.js')
      console.log(`   🔄 [${accountId}] 数据同步完成，立即触发规则执行（AdsPolar 事件驱动）`)
      await executeRulesForAccount(accountId, { force: false })
    } catch (ruleError) {
      console.error(`   ⚠️  [${accountId}] 规则执行失败:`, ruleError.message)
    }
  })
}
```

## 📝 保留批量模式（作为兜底）

### 为什么保留？

1. **兼容性**：某些场景可能需要批量执行
2. **兜底机制**：如果事件驱动失败，批量模式可以作为备份
3. **手动触发**：支持手动触发所有规则

### 优化后的批量模式

**位置**：`server/services/cronService.js` - `startCronJob`

**逻辑**：
- 数据同步完成后，检查是否有账户需要执行规则
- 使用反向索引，只在实际需要的账户上执行
- 避免遍历所有账户

## 📝 总结

### 已实现的优化

1. **反向索引** ✅

2. **事件驱动** ✅

   - 分散执行，避免批量竞争

3. **保留批量模式** ✅
   - 作为兜底机制
   - 支持手动触发

### 性能提升

- ✅ 从36个账户减少到1个账户（如果规则指定了账户）
- ✅ 从批量执行改为事件驱动（分散执行）
- ✅ 避免超时问题（每个账户独立执行）
- ✅ 符合 AdsPolar "大道至简"理念
