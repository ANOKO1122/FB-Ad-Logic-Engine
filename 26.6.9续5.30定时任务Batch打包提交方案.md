# 定时任务 Batch 打包提交方案

> 版本：v1.0  
> 日期：2026-06-09  
> 状态：方案设计（未实施）  
> 目标：将所有定时任务的动作请求通过 Meta Batch API 打包提交，替代当前逐条调用的方式

---

## 1. 背景与现状

### 1.1 当前执行模式

定时任务（`scheduledTaskService.js` → `executeDueScheduledTasks`）当前是**纯逐条调用**：

```
for each dueTask
  → for each account  (多账户)
    → for each targetId  (多目标)
      → executeAction(api, task, targetId, budgetObjectId)
        → api.pauseAd(id)                 // 1次 HTTP 请求
        → api.activateAd(id)              // 1次 HTTP 请求
        → api.pauseAdset(id)              // 1次 HTTP 请求
        → api.updateAdsetBudget(id, c)    // 1次 HTTP 请求
        → api.updateCampaignBudget(id, c) // 1次 HTTP 请求
```

**问题**：
- N 个目标 = N 次独立 HTTP 请求，网络开销大
- 容易触发 Meta API 的 QPS 限制（每小时调用额度）
- 无法利用 Meta Batch API 的并行处理和低积分消耗优势

### 1.2 对标：规则管理已有的 Batch 能力

规则管理（`actionExecutorService.js`）已有**有限 Batch**：
- 仅在 `ENABLE_ACTION_BATCH=1` 且单规则+单状态动作+ad 层级时启用
- 预算类动作仍未 Batch
- 定时任务侧完全未接入

### 1.3 鉴权确认：系统使用万能 Token ✅

经过代码审查确认：

- `FacebookMarketingAPI` 构造函数接收**单一** `accessToken`（`server/index.js` 第 257 行）
- 所有 API 调用统一使用 `this.accessToken`，不做账户级别的 Token 区分
- 现有 `batchRequests()` 方法在所有子请求中**共用同一个 token**（第 403 行）
- 该 Token 是绑定在同一个 BM 下的**系统用户 Token**，天然具备跨账户操作权限

**结论**：系统已使用万能 Token 鉴权，Meta Batch API 的跨账户批处理方案可以直接实施，无需额外鉴权改造。

## 2. 目标

| 目标 | 说明 |
|------|------|
| 全部动作 Batch | 暂停、激活、设置预算、增加预算、减少预算，全部走 Batch |
| 跨账户 | 一个 Batch 请求可同时操作不同广告账户下的对象 |
| 多对象 | 支持动态筛选匹配的大量目标（可能数百个） |
| 自动分片 | 超过 50 个子请求时，自动切分为多个批次顺序发送 |
| 条件预读 | 只有 increase_budget / decrease_budget 才在执行前批量查询当前预算（需实时值做计算）；pause/activate/set_budget 跳过读阶段，直接写入 |
| 容错兼容 | 单个子请求失败不影响同批次其他子请求；保持审计日志完整性 |

## 3. 方案设计

### 3.1 整体流程

**核心判断逻辑**：只有 `increase_budget` / `decrease_budget` 需要走 Read Phase（因为"上调 10%"需要知道当前预算才能算新值），`pause_ad` / `activate_ad` / `set_budget` 直接走 Write Phase（不依赖 FB API 实时查询结果）。

```
┌─────────────────────────────────────────────────────────┐
│  1. 收集阶段 (Collection)                                │
│     查询所有到期任务 → 解析每个任务的目标对象列表           │
│     → 按动作类型分类：直接执行 / 需先读预算                  │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│  2. 预算读取阶段 (Read Phase)                            │
│     ⚠️ 条件触发：仅当存在 increase_budget/decrease_budget │
│     收集所有需要当前预算的目标 → 构造 GET Batch            │
│     → 分片执行 batchRequests() → 计算新预算值              │
│     pause/activate/set_budget 直接跳过此阶段              │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│  3. 写入执行阶段 (Write Phase)                            │
│     收集所有动作项（状态 + set_budget + 已算好的增减预算）  │
│     → 构造 POST Batch → 分片执行 batchRequests() → 解析结果│
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│  4. 结果处理阶段 (Post-processing)                        │
│     按任务汇总结果 → 更新任务状态（成功/失败/重试）          │
│     → 批量写入审计日志                                     │
└─────────────────────────────────────────────────────────┘
```

### 3.3 阶段一：收集（Collection）

**流程**：
1. 查询到期任务（与现有一致）
2. 对每个任务：
   - 获取所有目标账户 → 解析每个账户的目标对象列表
   - 解析每个目标的 `budgetObjectId`（ad 级预算动作需要查 `adset_id`）
   - 按 `actionType` 分流入队：

| actionType | 入队目标 | 原因 |
|-----------|---------|------|
| `pause_ad` / `activate_ad` | **状态写入队列** | 启停只需改 status 字段，不依赖 API 返回值计算 |
| `set_budget` | **预算写入队列** | 设置为固定值（如 $30），无需查询当前预算即可计算 |
| `increase_budget` / `decrease_budget` | **预算读取队列** | 需要先查当前预算才能计算"上调 10%""下调 $5"的目标值 |

```js
// 伪代码
const statusWriteQueue = []      // Item[]
const budgetWriteQueue = []     // Item[]
const budgetReadQueue = []      // Item[]

for (const task of dueTasks) {
  for (const accountId of getTaskAccountIds(task)) {
    const targetIds = await resolveTargetIdsForTask(task, accountId)
    for (const targetId of targetIds) {
      const budgetObjectId = await resolveBudgetObjectId(accountId, task, targetId)
      const objectName = await resolveObjectName(accountId, targetLevel, targetId)
      const itemBase = { taskId, accountId, targetId, targetLevel, objectName }

      if (['pause_ad', 'activate_ad'].includes(task.actionType)) {
        const actionType = task.actionType
        const statusMap = {
          pause_ad: { newStatus: 'PAUSED', objectId: targetId },
          activate_ad: { newStatus: 'ACTIVE', objectId: targetId }
        }
        const resolved = resolveStatusActionLevel(task, targetId)
        statusWriteQueue.push({
          kind: 'status',
          ...itemBase,
          objectId: resolved.objectId,
          targetLevel: resolved.targetLevel,
          newStatus: statusMap[actionType].newStatus
        })
      }

      else if (task.actionType === 'set_budget') {
        const newCents = computeSetBudgetCents(task.actionParams)
        budgetWriteQueue.push({
          kind: 'budget',
          ...itemBase,
          objectId: budgetObjectId,
          newBudgetCents: newCents,
          actionType: task.actionType
        })
      }

      else if (['increase_budget', 'decrease_budget'].includes(task.actionType)) {
        budgetReadQueue.push({
          kind: 'read_budget',
          ...itemBase,
          objectId: budgetObjectId,
          actionType: task.actionType,
          actionParams: task.actionParams
        })
      }
    }
  }
}
```

### 3.4 阶段二：预算读取阶段（Read Phase） — 条件触发

> ⚠️ **此阶段仅在收集到 `increase_budget` 或 `decrease_budget` 类型的任务时触发。**  
> 如果本轮到期任务全部为 `pause_ad` / `activate_ad` / `set_budget`，**直接跳过此阶段进入 Write Phase**。

**触发条件**：
```js
if (budgetReadQueue.length === 0) {
  // 没有 increase/decrease 任务 → 跳过 Read Phase → 直接进入 Write Phase
  logger.info('[ScheduledTask Batch] 无增减预算任务，跳过 Read Phase')
  return new Map()
}
```

**为什么只有 increase/decrease 需要预读？**
- `pause_ad`：改 `status=PAUSED`，不依赖 API 返回值
- `activate_ad`：改 `status=ACTIVE`，不依赖 API 返回值
- `set_budget`：设置为固定美元值（如 $30），前端已传入目标值，直接 `30 * 100 = 3000美分` 即可，无需知道当前预算
- `increase_budget`：上调 $5 或 10%，必须知道当前预算才能计算目标值 → **需要 Read Phase**
- `decrease_budget`：下调 $5 或 10%，必须知道当前预算才能计算目标值 → **需要 Read Phase**

**目标**：批量获取所有 `increase_budget` / `decrease_budget` 目标的当前预算。

**Batch 请求格式（GET）**：
```json
[
  { "method": "GET", "relative_url": "adset_123?fields=daily_budget,lifetime_budget" },
  { "method": "GET", "relative_url": "adset_456?fields=daily_budget,lifetime_budget" },
  { "method": "GET", "relative_url": "campaign_789?fields=daily_budget,lifetime_budget" }
]
```

**实现**：
```js
async function executeBudgetReadPhase(budgetReadQueue, api) {
  if (budgetReadQueue.length === 0) return new Map()

  // 1. 构造 batch request items
  const requests = budgetReadQueue.map(item => ({
    method: 'GET',
    relative_url: `${item.objectId}?fields=daily_budget,lifetime_budget`
  }))

  // 2. 发送 batch 请求（自动分片，每批 ≤50）
  const results = await api.batchRequests(requests, {
    priority: 'action',
    label: 'scheduled_budget_read'
  })

  // 3. 解析结果 → 计算新预算
  const budgetMap = new Map()  // key → newCents
  for (let i = 0; i < budgetReadQueue.length; i++) {
    const item = budgetReadQueue[i]
    const res = results[i]
    const key = `${item.taskId}:${item.targetId}`

    if (Number(res?.code || 0) >= 200 && Number(res?.code || 0) < 300 && res?.body) {
      const detail = res.body
      const raw = detail.daily_budget > 0 ? detail.daily_budget : detail.lifetime_budget
      const currentCents = Math.round(Number(raw))
      const newCents = computeNewBudgetCentsOnce(currentCents, item.actionParams)
      budgetMap.set(key, { currentCents, newCents, isDaily: detail.daily_budget > 0 })
    } else {
      budgetMap.set(key, { error: true, message: res?.body?.error?.message || `code=${res?.code}` })
    }
  }

  return budgetMap
}
```

### 3.5 阶段三：写入执行阶段（Write Phase）

**目标**：将所有状态动作和预算动作统一通过 POST Batch 发送。

**Batch 请求格式（POST）**：

| 动作类型 | relative_url | body |
|---------|-------------|------|
| 暂停 ad | `ad_id` | `status=PAUSED` |
| 激活 ad | `ad_id` | `status=ACTIVE` |
| 暂停 adset | `adset_id` | `status=PAUSED` |
| 激活 adset | `adset_id` | `status=ACTIVE` |
| 暂停 campaign | `campaign_id` | `status=PAUSED` |
| 激活 campaign | `campaign_id` | `status=ACTIVE` |
| 设置/调整 adset 预算 | `adset_id` | `daily_budget=5000` |
| 设置/调整 campaign 预算 | `campaign_id` | `daily_budget=5000` |

> **注意**：不可以使用 `act_xxx/adsets` 作为 relative_url。Meta Batch API 中 POST 操作的 `relative_url` 需直接使用目标对象的 ID（如 `act_xxx/campaigns` 用于创建 campaign，而更新现有对象直接用 `{object_id}`）。所有更新操作统一使用 `{object_id}` 作为 `relative_url`，`body` 中指定要修改的字段。

**预算写入整合**：
- `set_budget`：收集阶段直接计算好 `newBudgetCents`（固定值 × 100），直接进入 Write Phase
- `increase_budget` / `decrease_budget`：Read Phase 结束后从 `budgetMap` 中取出计算结果；若 `newCents === currentCents`（无需调整）则跳过不发送
- `pause_ad` / `activate_ad`：无预算计算，直接构造 status body

**三类动作的 Batch body 构造方式**：

| 动作 | 是否需要 Read Phase | body 构造方式 |
|------|-------------------|-------------|
| `pause_ad` / `activate_ad` | ❌ 不需要 | `status=PAUSED` 或 `status=ACTIVE`（固定值） |
| `set_budget` | ❌ 不需要 | `daily_budget=<value×100>`（固定值，前端传入） |
| `increase_budget` / `decrease_budget` | ✅ 需要 | `daily_budget=<Read Phase 计算出的 newCents>`（依赖当前预算） |

```js
async function executeWritePhase(statusItems, budgetItems, api) {
  // 1. 合并所有写入项
  const allItems = []

  for (const item of statusItems) {
    const statusValue = item.newStatus  // 'PAUSED' | 'ACTIVE'
    allItems.push({
      meta: item,
      request: {
        method: 'POST',
        relative_url: String(item.objectId),
        body: `status=${statusValue}`
      }
    })
  }

  for (const item of budgetItems) {
    const field = item.isDaily ? 'daily_budget' : 'lifetime_budget'
    allItems.push({
      meta: item,
      request: {
        method: 'POST',
        relative_url: String(item.objectId),
        body: `${field}=${Math.round(item.newBudgetCents)}`
      }
    })
  }

  // 2. 提取纯 request 数组，走 batchRequests（内部自动按 50 分片）
  const requests = allItems.map(i => i.request)
  const results = await api.batchRequests(requests, {
    priority: 'action',
    label: 'scheduled_write_batch'
  })

  // 3. 按 taskId 分组收集结果
  const taskResults = new Map()  // taskId → { items: [], success: N, fail: N }

  for (let i = 0; i < allItems.length; i++) {
    const { meta } = allItems[i]
    const res = results[i]
    const ok = Number(res?.code || 0) >= 200 && Number(res?.code || 0) < 300 && !res?.body?.error

    const entry = {
      taskId: meta.taskId,
      targetId: meta.targetId,
      accountId: meta.accountId,
      objectName: meta.objectName || null,
      actionType: meta.actionType,
      success: ok,
      errorMessage: ok ? null : (res?.body?.error?.message || `code=${res?.code}`),
      apiRequest: JSON.stringify(allItems[i].request),
      apiResponse: JSON.stringify(res?.body || res?.raw || null)
    }

    if (!taskResults.has(meta.taskId)) {
      taskResults.set(meta.taskId, { items: [], successCount: 0, failCount: 0 })
    }
    const tr = taskResults.get(meta.taskId)
    tr.items.push(entry)
    if (ok) tr.successCount++
    else tr.failCount++
  }

  return taskResults
}
```

### 3.6 分片策略

利用现有的 `batchRequests()` 方法，该方法已实现**自动 50 条分片**：

```js
// server/index.js 第 392-443 行（现有实现）
async batchRequests(requests, opts = {}) {
  const BATCH_LIMIT = 50
  const allResults = []

  for (let i = 0; i < requests.length; i += BATCH_LIMIT) {
    const chunk = requests.slice(i, i + BATCH_LIMIT)
    // 每个 chunk 发送一次 HTTP POST
    const response = await this.makeRequest(batchUrl, { batch: JSON.stringify(chunk), access_token }, ...)
    // 解析结果
    allResults.push(...chunkResults)
  }

  return allResults
}
```

**分片行为**：
- 当请求数 ≤ 50：1 次 HTTP POST，直接返回
- 当请求数 51~100：第 1 次 HTTP POST 50条 → 第 2 次 HTTP POST 剩余条
- 当请求数 N：`⌈N/50⌉` 次 HTTP POST，顺序执行

**结果索引映射**：`batchRequests` 按分片顺序拼接结果，因此返回的 `allResults[i]` 与原始 `requests[i]` 严格对应，无需手动维护索引。

### 3.7 结果处理与任务状态更新

每个任务的结果汇总规则（与当前一致）：

| 任务结果 | 处理 |
|---------|------|
| 全部对象成功 | `updateTaskAfterExecute(task)` → 清空重试，计算下次执行时间 |
| 部分/全部失败 | `updateTaskAfterFailure(task)` → 指数退避重试或自动禁用 |
| 全部跳过 | `updateTaskAfterSkip(task)` → 前推 next_execute_at |

### 3.8 审计日志：整合为任务级单条日志

#### 3.8.1 设计动机

当前规则管理是 **1 个对象 = 1 条审计日志**（`executeActionsForAd` → 每个 matchedAd 写一条 `automation_logs`）。

但定时任务 Batch 模式下，一个任务可能操作数十乃至数百个目标对象——如果仍然每对象一条日志，会导致：
- 日志表膨胀严重（500 个目标 = 500 行）
- 前端日志列表无法阅读（一个任务刷出一整页日志）
- 语义不对齐：Batch 本身就是"一次请求多对象"，日志也应是一次记录

**设计原则**：**一个定时任务 × 一次执行 = 一条审计日志**。所有分片的所有目标对象信息整合在这一条中。

#### 3.8.2 日志粒度：整个任务一条日志

无论任务有多少个目标对象、拆了多少个分片，最终只写入 **1 条** `automation_logs` 记录。

```
任务 #12 "凌晨关停亏损广告" 执行：
  → 匹配 120 个目标对象
  → 3 个分片 (50+50+20)
  → 写 1 条 automation_logs 记录（← 而不是 120 条或 3 条）
```

#### 3.8.3 数据库字段映射

| automation_logs 字段 | Batch 模式下填入的值 | 说明 |
|----------------------|---------------------|------|
| `accountId` | 该任务的主账户 ID | 保持与任务表一致 |
| `adId` | `"batch_summary"` | 固定值，标记这是批量日志 |
| `objectType` | 任务的目标层级 (`ad`/`adset`/`campaign`) | 与任务一致 |
| `objectId` | `"batch:{N}个对象"` | 如 `"batch:120个对象"`，概括对象数量 |
| `objectName` | `null` | 无需单对象名称 |
| `actionType` | `"BATCH_PAUSE_AD"` / `"BATCH_ACTIVATE_AD"` / `"BATCH_SET_BUDGET"` / `"BATCH_INCREASE_BUDGET"` / `"BATCH_DECREASE_BUDGET"` | 加 `BATCH_` 前缀区分 |
| `actionPayload` | **JSON: 每个目标对象的执行结果数组** | 见下方数据结构 |
| `apiRequest` | **JSON: 所有分片的 batch 请求原文** | 大 JSON，含每个 chunk 的 requests 数组 |
| `apiResponse` | **JSON: 所有分片的 batch 响应原文** | 大 JSON，含每个 chunk 的 results 数组 |
| `status` | `success` / `fail` / `skipped` | 按汇总规则判定（见 3.8.5） |
| `errorMessage` | 汇总错误信息 | 如 `"成功120, 失败3: [ad_1: xxx, ad_2: yyy]"` |
| `explanation` | `{ trigger_type: 'scheduled_batch', task_id, task_name, total_objects, total_chunks, ... }` | 记录 batch 执行元信息 |
| `preflightMode` | 仅 Dry Run 时 `"preflight"` | — |
| `isSimulation` | 任务 `is_simulation` 值 | — |

```json
      "errorMessage": "Permission error",
      "chunkIndex": 0
    }
  ]
}
```

#### 3.8.5 日志 status 汇总判定规则

| 目标执行结果 | 日志 status |
|------------|-----------|
| 全部 `success`（含部分 `skipped`） | `success` |
| 有 `fail`，但也有 `success` | `fail`（errorMessage 列出失败项） |
| 全部 `fail` | `fail` |
| 全部 `skipped` | `skipped` |
| 一个都没执行（队列空） | `skipped` |

#### 3.8.6 实现代码

```js
/**
 * 写入 Batch 审计日志（一条任务一条日志）
 * @param {Object} task - 任务对象
 * @param {Array} allChunkRequests - 所有分片的请求数组（二维: [[chunk0], [chunk1], ...]）
 * @param {Array} allChunkResults - 所有分片的响应数组（二维: [[chunk0], [chunk1], ...]）
 * @param {Array} targetResults - 每个目标的执行结果（见 actionPayload.targets 格式）
 */
async function writeBatchAuditLog(task, allChunkRequests, allChunkResults, targetResults) {
  const total = targetResults.length
  const successCount = targetResults.filter(t => t.status === 'success').length
  const failCount = targetResults.filter(t => t.status === 'fail').length
  const skippedCount = targetResults.filter(t => t.status === 'skipped').length

  // 判定整体 status
  let overallStatus = 'success'
  if (failCount > 0) overallStatus = 'fail'
  else if (successCount === 0 && skippedCount > 0) overallStatus = 'skipped'
  else if (total === 0) overallStatus = 'skipped'

  // 汇总错误信息
  let errorMessage = null
  if (failCount > 0) {
    const failItems = targetResults.filter(t => t.status === 'fail')
    const failSummary = failItems.slice(0, 10).map(t =>
      `${t.objectId}: ${t.errorMessage}`
    ).join('; ')
    errorMessage = `成功${successCount}, 失败${failCount}: [${failSummary}]`
    if (failItems.length > 10) {
      errorMessage += ` ...还有${failItems.length - 10}条`
    }
  }

  const actionType = task.actionType || task.action_type
  const batchActionType = `BATCH_${String(actionType).toUpperCase()}`

  await db.insert(automationLogs).values({
    accountId: String(task.accountId || task.account_id || ''),
    adId: 'batch_summary',
    objectType: (task.targetLevel || task.target_level || 'ad').toLowerCase(),
    objectId: `batch:${total}个对象`,
    objectName: null,
    actionType: batchActionType,
    actionPayload: {
      batch_summary: {
        total,
        success: successCount,
        fail: failCount,
        skipped: skippedCount,
        chunks: allChunkRequests.length
      },
      targets: targetResults
    },
    apiRequest: JSON.stringify(allChunkRequests),
    apiResponse: JSON.stringify(allChunkResults),
    explanation: {
      trigger_type: 'scheduled_batch',
      schedule_type: task.scheduleType || task.schedule_type,
      schedule_at: task.scheduleAt || task.schedule_at,
      task_id: task.id,
      task_name: task.taskName || task.task_name || null
    },
    preflightMode: task.isSimulation ? 'preflight' : null,
    status: overallStatus,
    errorMessage: errorMessage,
    isSimulation: !!(task.isSimulation || task.is_simulation),
    triggeredAt: new Date(),
    ownerId: task.ownerId || task.owner_id || 0
  })
}
```

## 5. 容错与边界情况

### 5.1 Batch 中单条失败

Meta Batch API 的特性是**每条子请求独立执行**，单条失败不影响同批其他子请求。

处理策略：
- 逐条检查 `res.code` 和 `res.body.error`
- 成功的 task 走 `updateTaskAfterExecute`，失败的走 `updateTaskAfterFailure`
- 审计日志中失败条目标记 `status='fail'`，记录 API 错误信息

### 5.2 部分对象"无需调整"（预算不变）

`increase_budget` / `decrease_budget` 计算后发现 `newCents === currentCents`：
- 不加入 Write 队列
- 审计日志标记 `status='skipped'`，`errorMessage='目标已达成（budget_already_at_target）'`

### 5.3 账户未激活

收集阶段过滤：
- 在校验账户 `is_active` 时，跳过该账户的所有目标
- 审计日志记录为 `skipped`

### 5.4 空队列

- `budgetReadQueue.length === 0`：跳过 Read Phase
- `statusWriteQueue + budgetWriteQueue.length === 0`：跳过 Write Phase
- 全部空：直接返回 `{ executed: 0, skipped: 0, errors: 0 }`

### 5.5 Token 缺失

- 与现有逻辑一致：检查 `FACEBOOK_ACCESS_TOKEN`
- 所有非 Dry Run 任务跳过，记录 warn

### 5.6 Dry Run 模式

- `isSimulation=true` 的任务不参与 Batch
- 走独立逻辑：仅写入审计日志，标记 `preflightMode='preflight'`

### 5.7 互斥锁

- 与现有 `_scheduledTaskRunning` 逻辑一致
- 整个两阶段过程都在互斥锁保护下执行

## 6. 性能预期

| 场景 | 当前（逐条） | 改造后（Batch） | 改善 |
|------|------------|----------------|------|
| 1个任务，10个目标（暂停/激活/设预算） | 10 次 HTTP POST | 1 次 HTTP POST（无 Read Phase） | **10x** |
| 1个任务，10个目标（上调10%） | 10 GET + 10 POST = 20 次 | ceil(10/50) GET + ceil(10/50) POST = 2 次 | **10x** |
| 5个任务，共200个目标（混合动作） | 200+ 次 HTTP | ceil(200/50) POST + 增减再额外 GET | **~50x** |
| 跨3个账户，各50个目标（全增减预算） | 150 GET + 150 POST = 300 次 | ceil(150/50) x 2 = 6 次 | **~50x** |
| 单个大任务500个目标（纯暂停） | 500 次 HTTP POST | ceil(500/50) = 10 次 POST | **~50x** |
| 单个大任务500个目标（纯上调10%） | 500 GET + 500 POST = 1000 次 | ceil(500/50) x 2 = 20 次 | **~50x** |

**Meta 积分消耗**：Batch 中 50 个子请求仅消耗约 3-5 个积分（vs 逐条约 50 个积分），大幅降低 API 调用额度压力。

## 7. 实施注意事项

### 7.1 Batch body 格式

Meta Batch API 的 `body` 字段为**字符串**，需使用 **URL-encoded** 格式：
- ✅ `"status=PAUSED"` （正确）
- ✅ `"daily_budget=5000"` （正确）
- ❌ `"{\"status\":\"PAUSED\"}"` （避免使用 JSON 字符串）
- ❌ `{ status: "PAUSED" }` （不能传对象）

### 7.2 超时控制

`batchRequests()` 已设置 60 秒超时（`server/index.js` 第 410 行），对于大批次（如 500 条切 10 个分片），每个分片独立超时。

### 7.3 并发安全

在 `flushAuditBatch` 之前，每个任务的结果处理完成后立即写入审计日志（与现有 v3.2 行为一致），防止服务重启丢失。

### 7.4 向后兼容

- `forceExecuteTask` 手动执行路径与 Cron 路径**共用同一套 Batch 执行逻辑**（`collectActionItems` → `executeWritePhase` → `writeBatchAuditLog`）
- 旧的 `executeAction()` 逐条执行函数**保留但标记为 deprecated**，不再被任何路径调用
- 通过环境变量 `SCHEDULED_BATCH_ENABLED=0/1` 控制是否启用 Batch；关闭时回退到旧的逐条模式（兼容过渡期）

### 7.5 建议的环境变量

```bash
# 定时任务 Batch 模式开关（默认启用）
SCHEDULED_BATCH_ENABLED=1

# 设为 0 时：Cron 和手动执行均回退到旧的逐条模式（executeAction + writeAuditLog 每对象一条日志）
# 设为 1 时：Cron 和手动执行均走 Batch + writeBatchAuditLog（每任务一条整合日志）
```

## 9. 改造风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Batch API 超时导致整批失败 | 多个目标的动作一起失败 | 每个分片独立超时（60s），分片间相互不影响 |
| 预算读取 Batch 返回异常数据 | 预算计算结果错误 | 对每个 GET 结果做校验，异常项跳过并标记 fail |
| 大量目标导致 Batch 分片过多 | 本轮执行时间变长 | 已有 LIMIT 100 任务上限 + 每个分片写审计日志；极端情况可加总超时 |
| Cron 和手动执行路径统一 | 两套执行逻辑维护成本高 | 共用同一套 Batch 执行逻辑，`executeAction()` 标记 deprecated 保留但不调用 |

## 10. 总结

本方案将定时任务的执行路径从"逐条 HTTP 调用"改造为"两阶段 Batch 模式"：

1. **收集阶段**：遍历到期任务，按动作类型分类为三个队列
2. **Read Phase**：批量 GET 拉取需调整预算的对象的当前预算（仅 increase/decrease 触发）
3. **Write Phase**：所有动作统一打包为 POST Batch，自动按 50 分片
4. **结果处理**：按任务汇总 → 更新状态
5. **审计日志**：每个任务写 1 条整合日志（而非每对象一条），包含 batch 请求/响应原文和每个目标的执行结果；Cron 和手动执行统一走此路径

核心优势：
- HTTP 请求数从 O(N) 降至 O(N/50)
- Meta API 积分消耗大幅降低
- 跨账户天然支持（复用现有万能 Token + `batchRequests`）
- 审计日志行数从 N 条降至 1 条/任务，避免日志表膨胀
- 前端通过摘要+气泡模式友好展示批量执行结果

改造集中在 `scheduledTaskService.js` 一个文件（后端），`Logs.vue` 需小幅调整（前端日志展示）。数据库表结构无需改动。
