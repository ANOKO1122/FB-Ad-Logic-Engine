#### 1.1 权限控制统一修复

**问题**：
- Summary 接口：错误使用 `req.user.id` 过滤 `automation_logs.owner_id`（应该用 `users.owner_id`）
- 详情接口：错误使用 `req.user.id` 过滤 `automation_logs.owner_id`（应该用 `users.owner_id`）
- 列表接口：正确使用 `users.owner_id` 过滤，但 summary/详情不一致

**根本原因**：
- `automation_logs.owner_id` 记录的是 `owners.id`（从 `user.owner_id` 写入）
- 但接口错误地用了 `users.id` 去过滤，导致权限错判

**解决方案**：
- Summary 接口：先查询 `users.owner_id`，再用它过滤 `automation_logs.owner_id`
- 详情接口：同样用 `users.owner_id` 过滤 `automation_logs.owner_id`
- 统一三个接口的权限口径：都用 `users.owner_id` → `automation_logs.owner_id`

**验证结果**：
- ✅ pytest 测试全部通过（6 passed）
- ✅ 权限控制正确：非 admin 只能看到自己 owner_id 的日志
- ✅ Admin 可以查看所有日志

## 遇到的问题与解决方法

### 问题1：Thunder Client 环境变量配置问题

**问题**：
- 用户无法使用 Thunder Client 测试，因为 Env 里没有配置环境变量
- 请求使用 `{{staffToken}}` 但环境变量为空，导致 token 过期错误

**解决方法**：
- 方案1：指导用户在 Thunder Client 中创建环境并配置变量
- 方案2：修改测试用例集合，使用固定 URL 和手动粘贴 token 的方式（最终采用）

**结果**：
- ✅ 测试用例集合改为不依赖环境变量
- ✅ 用户可以正常使用 Thunder Client 测试

### 问题2：drizzle.js 文件第一行误加字符

**问题**：
- 后端启动失败：`ReferenceError: h is not defined`
- 文件第一行误加了字符 `h`：`h// Drizzle ORM 数据库连接配置`

**解决方法**：
- 删除第一行的 `h` 字符

**结果**：
- ✅ 后端正常启动

### 问题3：查询接口 SQL 语法错误

**问题1**：查询列表接口报错 `Incorrect arguments to mysqld_stmt_execute`
- **原因**：MySQL 的 LIMIT 和 OFFSET 用占位符 `?` 绑定可能有问题
- **解决**：改为直接拼接整数 `LIMIT ${limitNum} OFFSET ${offset}`

**问题2**：查询统计接口 SQL 语法错误
- **原因**：当 `whereClause` 为空时，SQL 变成了 `... AND skip_reason IS NOT NULL`（缺少 WHERE）
- **解决**：判断 `whereClause` 是否为空，分别处理

**结果**：
- ✅ 查询列表接口正常工作
- ✅ 查询统计接口正常工作

### 问题4：skip_details JSON 解析错误

**问题**：
- MySQL JSON 列有时返回对象（不是字符串），直接 `JSON.parse()` 会报错
- 错误：`JSON.parse("[object Object]")` 失败

**解决方法**：
- 兼容三种情况：
  1. 已经是对象：直接使用
  2. 是字符串：JSON.parse
  3. 是 Buffer：toString 再 JSON.parse

**结果**：
- ✅ skip_details 正确解析为对象

### 问题5：evaluated_at 时区转换问题

**问题**：
- 用户发现数据库里的时间（UTC）和控制台显示的时间不一致
- 怀疑是时区问题

**分析结果**：
- `Etc/GMT-8` = UTC+8（中国），进程时区没有错
- 系统时区逻辑正确：存 UTC，算账户时区，前端显示北京时区
- evaluated_at 写入时已修复为显式 UTC 转换

**解决方法**：
- 在 `ruleExecutionSummaryService.js` 中显式将 `evaluatedAt` 转换为 UTC 时间字符串
- 确保不依赖进程时区

**结果**：
- ✅ evaluated_at 正确存储为 UTC
- ✅ 前端统一显示为北京时区

## 技术细节

### 规则执行摘要表设计

**表名**：`rule_execution_summaries`

**核心字段**：
- `run_id`：运行批次ID（同一轮执行共享）
- `rule_id`、`rule_name`：规则信息
- `account_id`、`user_id`、`owner_id`：账户和用户信息
- `matched_count`、`executed_count`、`failed_count`、`skipped_count`：统计字段
- `status`：执行状态（matched/no_match/skipped/error）
- `skip_reason`：跳过原因枚举（cooldown/no_permission/account_mismatch/user_not_found/no_match/error）
- `skip_details`：跳过详情（JSON 对象）
- `error_message`：错误信息（已脱敏）
- `duration_ms`：执行耗时
- `evaluated_at`：评估时间（UTC）

**索引设计**：
- `idx_run_id`：按运行批次查询
- `idx_account_evaluated`：按账户 + 时间倒序
- `idx_rule_evaluated`：按规则 + 时间倒序
- `idx_owner_evaluated`：按所有者 + 时间倒序
- `idx_user_evaluated`：按用户 + 时间倒序
- `idx_status_evaluated`：按状态 + 时间倒序
- `idx_skip_reason_evaluated`：按跳过原因 + 时间倒序

### 前端时区工具

**文件**：`src/utils/dateTime.js`

**功能**：
- `parseUTC(dateStr)`：解析后端 UTC 时间字符串
- `formatTimeBeijing(dateStr)`：仅时间 HH:mm:ss
- `formatDateBeijing(dateStr)`：仅日期 MM/dd
- `formatDateTimeBeijing(dateStr)`：完整 yyyy-MM-dd HH:mm:ss
- `formatDateTimeBeijingLocale(dateStr)`：中文友好格式

**使用**：
- 所有前端时间展示统一调用这些函数
- 确保无论用户浏览器在哪个时区，前端都显示北京时区

## 经验总结

### 1. 权限控制统一的重要性

- 三个接口（summary/list/detail）必须使用同一套权限口径
- `users.owner_id` 和 `users.id` 不能混用
- 需要统一测试，确保权限控制一致

### 2. 时区处理的复杂性

- 数据库存 UTC，业务逻辑用账户时区，前端显示用固定时区
- 需要明确每一层的时区处理方式
- 写入时间戳时要显式转换为 UTC，避免依赖进程时区

### 3. MySQL JSON 列的兼容性

- MySQL JSON 列在不同驱动下可能返回对象或字符串
- 需要兼容多种返回格式，避免解析错误

### 4. SQL 参数绑定的注意事项

- LIMIT 和 OFFSET 在某些情况下用占位符可能有问题
- 对于安全的整数参数，可以直接拼接
- WHERE 子句为空时的 SQL 拼接要特别注意

### 5. 系统层可观测性的价值

- 展示层（用户看）和系统层（排障用）分离是好的设计
- 摘要记录可以回答"为什么规则没动静"的问题
- 不需要前端页面也能排障（API + 数据库查询）
