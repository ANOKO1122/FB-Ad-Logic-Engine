### 3. 方案 B+ 最终优化设计

#### 3.1 核心设计理念
- **智能路由**：根据时间窗口自动选择数据源（today → ad_snapshots，历史 → daily_stats）
- **原始计数存储**：存储分子和分母，支持多天动态计算
- **完全离线执行**：规则引擎不再调用 Facebook API，速度提升 20 倍
- **动态聚合计算**：多天窗口使用"总分子/总分母"重算，避免辛普森悖论

#### 3.2 关键优化点（已采纳所有建议）

1. **冷数据落盘口径（重要修正）**
   - 错误：对同一日的多次快照求和（SUM）→ 会重复累计，严重高估
   - 正确：取每个广告当日的最后快照
   - 实现：使用 `ROW_NUMBER() OVER (PARTITION BY account_id, ad_id ORDER BY synced_at DESC)`

2. **操作元数据字段**
   - 新增：`ad_set_id`（广告组 ID）
   - 理由：规则动作如增减预算需要 `adset_id`，避免在规则执行时额外查询 Ads API

3. **时间窗口命名一致性**
   - 统一使用：`last_3_days`、`last_7_days`（不是 `last_3d`、`last_7d`）

4. **时区一致性**
   - 统一从 `account_mappings` 表获取时区
   - 路由选择、冷数据落盘、时间窗口计算都使用账户时区

5. **归因设置口径**
   - Insights 请求显式开启 `use_account_attribution_setting: true`

6. **索引与约束**
   - `ad_snapshots`：`(account_id, synced_at)`、`(ad_id, synced_at)`、`(ad_id, synced_at DESC)`
   - `daily_stats`：`(account_id, date)`、`(ad_id, date)`

7. **分母为零防护**
   - 单天计算也要做除零保护
   - 多天聚合也要做除零保护

## 🐛 遇到的问题与解决方法

### 问题 1：测试循环依赖导致失败

**问题描述**：
- `health.test.js` 和 `rules.test.js` 失败
- 错误：`TypeError: Cannot read properties of undefined (reading 'get')`
- 位置：`server/index.js:1299`

**原因分析**：
- 循环依赖：`app.js` → `admin.js` → `cronService.js` → `index.js` → `app.js`
- 当测试导入 `app.js` 时，`index.js` 中的 `app` 对象还是 `undefined`
- `index.js` 在模块加载时执行 `app.get(...)`，导致错误

**解决方法**：
- 将 `index.js` 中的路由注册改为条件执行：
  ```javascript
  if (app) {
    app.get('/api/accounts', ...)
    // ... 其他路由
  }
  ```

**验证结果**：
- ✅ 所有测试通过（45 个测试用例）

---

### 问题 2：MySQL JSON 字段解析问题

**问题描述**：
- 测试失败：`Unexpected token 'a', "adset_123,adset_456" is not valid JSON`
- 位置：`server/tests/rules.test.js:125`

**原因分析**：
- MySQL 返回的 JSON 字段可能是字符串，也可能是已解析的对象
- 需要处理两种情况

**解决方法**：
- 添加类型判断：
  ```javascript
  const targetIds = typeof rows[0].target_ids === 'string' 
    ? JSON.parse(rows[0].target_ids) 
    : rows[0].target_ids
  ```

**验证结果**：
- ✅ 所有测试通过

---

### 问题 3：时区字段缺失

**问题描述**：
- `account_mappings` 表中没有 `timezone_name` 字段
- 代码中使用默认值 `'UTC'`，无法支持多时区

**解决方法**：
- 添加 `timezone_name` 字段到 `account_mappings` 表
- 更新 `ingestorService.js`：从数据库读取 `timezone_name`

**验证结果**：
- ✅ 字段已添加
- ✅ 代码已更新

---

### 问题 4：Windows 备份命令兼容性

**问题描述**：
- 备份命令失败：`date` 命令在 Windows PowerShell 中不存在

**解决方法**：

  ```bash

  ```

- ✅ 备份成功
- ✅ 迁移执行成功

## 💡 经验总结

### 1. 测试循环依赖问题
- **问题**：模块循环依赖导致测试失败
- **解决**：使用条件执行（`if (app) { ... }`）避免在测试时执行路由注册
- **经验**：架构分离很重要，避免模块间的循环依赖

### 2. MySQL JSON 字段处理
- **问题**：MySQL 返回的 JSON 字段可能是字符串，也可能是已解析的对象
- **解决**：添加类型判断，兼容两种情况
- **经验**：数据库字段类型要统一处理，做好防御性编程

### 3. 时区处理的重要性
- **问题**：硬编码时区无法支持多时区账户
- **解决**：统一从 `account_mappings` 表获取时区
- **经验**：时区处理要统一，避免跨日边界偏差

### 4. 冷数据落盘口径的重要性
- **问题**：对快照求和会导致重复累计，严重高估
- **解决**：取最后快照，不是 SUM
- **经验**：理解数据含义很重要，快照是累计值，不能直接求和
