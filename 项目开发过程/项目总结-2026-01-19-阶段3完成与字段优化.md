### 1. 阶段3：更新冷数据落盘（取最后快照）

#### 1.1 问题发现与修正
- **问题**：冷数据落盘使用 `SUM()` 聚合，导致重复累计（严重高估）
- **原因**：快照是累计值，对快照求 SUM 会发生"重复累计"
- **示例**：
  - 10:00 快照：spend = $100
  - 14:00 快照：spend = $150
  - 错误做法：SUM = $100 + $150 = $250 ❌
  - 正确做法：取最后快照 = $150 ✅

#### 1.2 实现方案
- **MySQL 8.0+**：使用 `ROW_NUMBER() OVER (PARTITION BY account_id, ad_id ORDER BY synced_at DESC)`，取 `rn = 1`
- **非 8.0 兼容**：使用"最大时间戳连接法"（先取 MAX(synced_at)，再连接原表取该时间点的完整字段）
- **时区处理**：使用 Luxon 库计算"昨日"边界，基于账户的 `timezone_name`

#### 1.3 代码实现
- 修改 `archiveDailyStats` 函数（`server/services/ingestorService.js`）
- 实现 `queryLastSnapshotCompatible` 函数（兼容非 MySQL 8.0）
- 更新 SQL 插入语句，包含 7 个原始计数字段和 `ad_set_id`
- 实现昨日 API 数据拉取（优先使用 API 的 cpc/roas，缺失时用原始计数计算）

#### 1.4 测试验证
- 创建 `test-archive-with-mock-data.js` 测试脚本
- 插入 3 个测试快照（同一广告同日不同时间点）
- 验证落盘值等于最后快照（35.8），而不是 SUM（71.6）
- **测试结果**：✅ 验证通过

## 🐛 遇到的问题与解决方法

### 问题1：冷数据落盘使用 SUM 导致重复累计

**现象**：
- 同一广告同日多次快照，落盘值 = SUM（错误）
- 例如：10:00 快照 $100 + 14:00 快照 $150 = $250（错误）

**原因**：
- 快照是累计值，不是增量值
- 使用 SUM 会重复累计，导致严重高估

**解决方法**：
- 改为取"最后快照"：使用 `ROW_NUMBER()` 或最大时间戳连接法
- 验证：落盘值 = 最后快照值（35.8），而不是 SUM（71.6）

---

**原因**：
- Facebook Insights API 不支持直接请求 `roas` 字段
- 实际字段名是 `purchase_roas`（数组格式）

**解决方法**：
- 更新字段请求：从 `roas` 改为 `purchase_roas`
- 实现 `extractRoas` 函数：从数组中提取值
- 添加降级策略：如果 `purchase_roas` 不可用，用 `purchase_value / spend` 计算
