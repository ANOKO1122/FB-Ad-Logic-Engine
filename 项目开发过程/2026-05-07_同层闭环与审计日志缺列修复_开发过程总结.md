## 4. 本轮关键问题与解决方法汇总

### 4.1 问题：执行日志缺失但动作已生效

- 现象：`FB` 后台确认动作成功；前端执行日志为空；后端仅出现“写入审计日志失败”。
- 根因：`automation_logs` 表结构未同步到最新代码合同，缺 `explanation` 字段。
- 解决方法：
  1. 补执行迁移：`server/db/migrations/048_add_explanation_to_automation_logs.sql`。
  2. 在运行库直接补齐 `explanation` 列并复验。  
  3. 对 `automation_logs` 关键列做环境一致性校验（发布前 SQL + 启动自检）。

### 4.2 问题：错误日志信息不足，排障成本高

- 现象：日志仅输出 `error.message` 或模糊文本，缺少 SQL 级错误细节。
- 根因：审计写入异常日志未使用统一结构化字段输出。
- 解决方法：
  1. 统一改造三条写入链路的 catch 日志结构：
     - `server/services/actionExecutorService.js`
     - `server/services/cronService.js`
  2. 输出标准字段：`code`、`errno`、`sqlState`、`sqlMessage`、`stack`、`runId`、`ruleId`、`objectType`、`objectId`、`actionType`。
  3. 抽取公共日志 helper，避免多处实现不一致。

### 4.4 问题：迁移“部分生效”导致口径不一致

- 现象：`summary_scope` 列存在，但 `idx_summary_scope` 索引缺失；`048/050` 未在运行库生效。  
- 根因：迁移执行链条在不同环境落地不一致。  
- 解决方法：
  1. 建立“字段/索引存在性”核验清单（不是只看迁移文件）。  
  2. 缺口项执行定点 SQL 补齐。  
  3. 将校验 SQL 固化为发布门禁。

### 4.3 问题：用户误解“SQL 两条记录=重复执行”

- 现象：同一 `run_id` 出现两条摘要记录。
- 根因：`summary_scope` 设计导致同次执行产出两种摘要视角。
- 解决方法：
  1. 明确口径：`account`=账户级明细；`rollup`=执行汇总。  
  2. 在验证与文档中固化该规则，避免误判。
