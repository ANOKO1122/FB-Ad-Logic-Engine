## 📋 项目定位与目标

### 项目定位
**Facebook Ad-Intelligence Server（本地决策中枢）**  
基于 Vue 3 + Express.js + MySQL 的 Facebook Marketing API 监控与自动化规则系统。

### 核心目标
1. **实时监控**: 多层级（广告/广告组/广告系列）实时监控广告数据
2. **数据对齐**: 与 Facebook Ads Manager 数据口径完全一致
3. **多用户隔离**: 实现"谁投的广告谁看，谁设的规则谁管"
4. **自动规则**: 支持自定义规则，条件满足时自动执行操作
5. **安全认证**: JWT + HttpOnly Cookie 的认证体系

## 🐛 遇到的主要问题与解决方案

### 1. 成效数据7倍问题 ✅

**问题描述**:  
前端显示的"成效(购买)"是 Facebook Ads Manager 的 7 倍（如 Ads Manager 显示 1，前端显示 7）。

**根本原因**:  
后端提取购买数时使用了 `includes('purchase')`，导致多个包含 purchase 的 action_type 被累加：
- `offsite_conversion.fb_pixel_purchase`
- `purchase`
- `omni_purchase`
- 等等...

而 Facebook Ads Manager 通常只展示其中一个权威口径。

**解决方案**:  
- 修改 `extractPurchaseCount` 方法，改为精确匹配 `offsite_conversion.fb_pixel_purchase`
- 优先命中权威 action_type，不再累加所有变体
- 如果精确匹配失败，再尝试其他常见变体，但只取一个值

**相关代码**:  
- `server/index.js`: `extractPurchaseCount()`, `pixelActionTypes()`

---

### 2. 切换账户数据残留问题 ✅

**问题描述**:  
切换账户后，页面仍显示上一个账户的数据，需要手动刷新才能看到新账户数据。

**根本原因**:  
- 旧账户的监控 interval 没有正确清理
- ROI 轮询定时器没有停止
- 数据订阅回调没有取消

**解决方案**:  
- 引入 `currentMonitoringAccountId` 追踪当前正在监控的账户
- 在 `onAccountChange` 中调用 `stopMonitoringForAccount`，清理旧账户的所有异步任务：
  - 停止监控 interval
  - 取消数据订阅
  - 清除 ROI 轮询定时器
- 切换账户时立即拉取新账户的基础数据，提供即时反馈

**相关代码**:  
- `src/views/Monitor.vue`: `onAccountChange()`, `stopMonitoringForAccount()`, `startMonitoringForAccount()`

### 3. ROI 异步接口 500 错误 ✅

**问题描述**:  
- `/api/roi/result` 返回 500 Internal Server Error
- 前端一直轮询失败，数据永远加载不出来（5-6分钟一直转圈）
- 控制台报错：`Facebook API Error: (#100) Tried accessing nonexisting summary field (cost_per_unique_link_click)`

**根本原因**:  
- 异步任务请求了 `cost_per_unique_link_click` 字段
- 但在 report-run（异步报表）场景下，该字段变成 "summary field 不存在"
- Facebook API 直接返回 400，后端抛出异常，返回 500
- 前端轮询对异常静默忽略，导致一直重试

**解决方案**:  
- **异步任务不再请求该字段**：`startRoiAsyncJob` 使用降级字段集，避免生成"必定报错"的任务
- **自动降级机制**：`/{report_run_id}/insights` 拉取时，如果字段不支持，自动降级到"最稳字段集"
- **错误处理优化**：`/api/roi/result` 遇到字段错误时，自动降级而不是直接 500

**相关代码**:  
- `server/index.js`: `startRoiAsyncJob()`, `getReportRunInsightsAll()`, `isInvalidFieldError()`

---

### 4. 数据对齐问题 ✅

**问题描述**:  
- uCPC、CPA、ROAS 等字段与 Facebook Ads Manager 不一致
- 有些字段对得上，有些有误差
- ROAS 显示为百分比（252.97%），而不是比值（2.53）

**根本原因**:  
- 前端使用 `spend / 次数` 推导费用，与 Ads Manager 的官方字段口径不一致
- ROAS 计算为 `(purchase_value / spend) * 100`，显示为百分比

**解决方案**:  
- **后端优先返回官方字段**：
  - CPA: 从 `cost_per_action_type` 提取 purchase 对应项
  - uCPC: 优先 `cost_per_unique_link_click`，其次 `cost_per_unique_inline_link_click`
  - 加购费/结账费/支付费: 从 `cost_per_action_type` 提取对应项
- **前端优先展示官方字段**：如果后端返回了官方值，优先展示；缺失时才用 `spend/次数` 推导
- **ROAS 格式调整**：改为纯数字比值（`purchase_value / spend`），显示为 2.53 而非 252.97%

**相关代码**:  
- `server/index.js`: `getAdInsightsWithROI()`, `extractRoiFromInsights()`, `pickCostPerActionType()`
- `src/views/Monitor.vue`: 表格列显示逻辑

### 5. Vue key 重复警告 ✅

**问题描述**:  
控制台出现大量 "Duplicate keys found during update" 警告，如：
```
[Vue warn]: Duplicate keys found during update: "120239217887300388"
```

**根本原因**:  
- `getRowKey` 在不同层级可能返回相同的值（如 `ad_id` 和 `adset_id` 可能相同）
- 某些行的 ID 为空，导致多个空字符串作为 key
- 不同层级切换时，key 可能冲突

**解决方案**:  
- 在 key 前加上层级前缀（`ad_`、`adset_`、`campaign_`），确保不同层级即使 ID 相同也不会冲突
- 如果 ID 为空，使用层级前缀 + 索引作为备用
- `v-for` 添加 `index` 参数，确保唯一性

**相关代码**:  
- `src/views/Monitor.vue`: `getRowKey(row, index)`, `v-for="(r, index) in adRows"`

### 6. 前端白屏问题 ✅

**问题描述**:  
前端页面白屏，控制台报错：`ReferenceError: dataLevel is not defined`

**根本原因**:  
- `return` 语句中引用了 `dataLevel`，但在 `setup` 函数中没有定义
- 之前的 `search_replace` 可能没有正确应用

**解决方案**:  
- 在 `setup` 函数开头添加变量定义：
  ```javascript
  const dataLevel = ref('ad') // 层级：ad / adset / campaign
  const adStatusMap = ref({}) // 广告状态映射
  ```

**相关代码**:  
- `src/views/Monitor.vue`: `setup()` 函数开头

## 🔑 关键设计决策

### 1. 认证方式：JWT + HttpOnly Cookie
- **原因**: 安全性高，防止 XSS 攻击（JS 无法读取 Cookie）
- **实现**: 登录时设置 HttpOnly Cookie，后续请求自动携带

### 2. 数据隔离：后端过滤
- **原因**: 前端过滤不安全，容易被绕过
- **实现**: 所有数据接口根据 `req.user.owner_id` 和 `account_mappings` 过滤

### 3. 规则存储：localStorage（当前）
- **原因**: 快速实现，按用户隔离
- **未来**: 迁移到 MySQL，支持 7x24h 后台执行

### 4. ROI 加载：异步 + 轮询
- **原因**: 同步请求可能超时（Facebook API 限制）
- **实现**: 先返回基础数据，ROI 深度指标异步加载，前端轮询结果

### 5. 字段对齐：官方字段优先
- **原因**: 确保与 Ads Manager 数据一致
- **实现**: 优先使用 Facebook 官方 `cost_per_*` 字段，缺失时才推导

## 📚 学习要点

### 1. Facebook Marketing API
- **归因设置**: `use_account_attribution_setting` 确保与 Ads Manager 一致
- **层级参数**: `level=ad|adset|campaign` 控制数据聚合层级
- **异步报表**: 大数据量时使用 `async=true`，通过 `report_run_id` 轮询结果
- **字段兼容性**: 不同账户/版本对字段支持不同，需要降级机制

### 2. Vue 3 Composition API
- **响应式数据**: `ref()` 用于基本类型，`reactive()` 用于对象
- **生命周期**: `onMounted()`, `onUnmounted()` 处理组件挂载/卸载
- **key 唯一性**: `v-for` 的 `:key` 必须唯一，否则可能导致渲染问题

### 3. 异步任务管理
- **清理机制**: 切换账户/组件卸载时必须清理所有异步任务（interval、轮询、订阅）
- **状态追踪**: 使用 `currentMonitoringAccountId` 等变量追踪当前状态
- **错误处理**: 异步任务失败时要有降级方案，避免卡死

### 4. 数据精度保护
- **ID 处理**: Facebook 的 ID 是字符串，不能转为 Number（会溢出）
- **字段匹配**: 使用精确匹配优先，模糊匹配作为兜底
