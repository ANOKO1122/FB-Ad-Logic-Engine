> 说明：本文件用于承接 2026-03-31 的 M2 收口记录，补充 2026-04-01 对“钉钉门禁模式”行为的修正，并把当前项目的重要上下文、开关语义、验证方法、上线边界一次性写清，避免后续切窗口或上下文压缩时丢失关键事实。

## 一、这次为什么还要改

前一版已经做出了两个后端开关：

- `ENABLE_DINGTALK_AUTH`
- `ENABLE_DINGTALK_GATE`

但是前端的实际表现仍然是：

- 当 `ENABLE_DINGTALK_GATE=true` 时
- 登录页和注册页不会立刻跳钉钉
- 而是先显示一张“请先通过钉钉扫码验证”的引导卡片

这和当前产品目标不一致。

当前产品目标已经明确为：

- 只要门禁模式开启
- 用户访问登录页或注册页时
- 不应该再先看到传统账号密码表单
- 而应该直接进入钉钉扫码验证流程

也就是说，这次修正的不是“有没有钉钉按钮”，而是“登录入口的控制权是不是已经真正前置到钉钉”。

---

## 二、这次改动的核心思路

### 1. 从“引导式门禁”改成“自动跳转门禁”

原来更像是：

- 门口放了一个牌子
- 告诉你应该去钉钉扫码
- 但你仍然先进入了登录页面

现在改成：

- 前端启动后先请求 `/api/auth/options`
- 读取当前门禁配置
- 如果发现 `dingtalkGateEnabled=true` 且 `dingtalkEnabled=true`
- 则直接执行 `window.location.replace('/api/auth/dingtalk/login')`

这就相当于：

- 先问门卫“今天是不是只走钉钉门”
- 如果是，就立刻把人带到钉钉入口
- 而不是让他先站到账号密码柜台前

### 2. 为什么前端还需要先拉一次 `/api/auth/options`

因为 `Login.vue` 和 `Register.vue` 是静态前端页面，浏览器打开它们时，本地并不知道后端当前的门禁开关状态。

所以必须先问后端：

- 钉钉登录功能有没有开启
- 门禁模式有没有开启

然后前端才知道该：

- 显示传统登录页
- 显示普通“可选钉钉登录”
- 还是直接跳钉钉

### 3. 为什么要增加“加载态”

如果不加“正在检查登录方式”的加载态，会出现一个经典前端问题：

- 页面先按默认值渲染出用户名密码表单
- 几百毫秒后才拿到后端配置
- 然后再突然跳走

这会造成你之前看到的“明明想做门禁，结果用户还是先看到了登录页”。

所以这次修正的关键点之一，就是：

- 在拉取 `/api/auth/options` 期间
- 暂时不渲染传统表单
- 只显示一个过渡提示

这样视觉上就更接近“直接钉钉门禁”。

## 三、本次实际修改文件

### 1. `src/views/Login.vue`

本次做了三件事：

- 新增 `loadingAuthOptions`，在配置未加载完成前不展示传统登录表单
- 当 `dingtalkGateEnabled=true && dingtalkEnabled=true` 时，页面加载后立刻跳转到 `/api/auth/dingtalk/login`
- 当门禁开启但钉钉入口未开启时，明确展示配置错误提示，而不是继续放出账号密码入口

### 2. `src/views/Register.vue`

本次做了三件事：

- 新增 `loadingAuthOptions`，避免注册表单在门禁模式下先闪出来
- 当门禁开启且钉钉可用时，访问注册页会直接跳钉钉
- 只有门禁关闭时，才会继续加载负责人列表和公开注册表单

### 3. `.env.auth`

本次仅修正一个配置拼写错误：

- `ENABLE_DINGTALK_GATE=flase`
- 修正为 `ENABLE_DINGTALK_GATE=false`

这个修正非常关键，因为如果拼错，后端读取到的就不是一个明确的布尔开关值，验证结果会和你的预期不一致。

## 四、当前开关语义表（必须牢记）

### 情况 A：两个都关

配置：

- `ENABLE_DINGTALK_AUTH=false`
- `ENABLE_DINGTALK_GATE=false`

表现：

- 传统用户名密码登录正常可见
- 登录页不显示钉钉扫码入口
- `/api/auth/dingtalk/*` 路由返回 404

适用场景：

- 云端暂不上线钉钉能力
- 本地功能尚未全部开发完

### 情况 B：只开钉钉能力，不开门禁

配置：

- `ENABLE_DINGTALK_AUTH=true`
- `ENABLE_DINGTALK_GATE=false`

表现：

- 传统用户名密码登录仍然可见
- 登录页会额外显示“钉钉扫码登录”按钮
- 适合灰度联调、内部试跑

适用场景：

- 你还不想一刀切切到钉钉门禁
- 但要先把钉钉闭环跑通

### 情况 C：两个都开

配置：

- `ENABLE_DINGTALK_AUTH=true`
- `ENABLE_DINGTALK_GATE=true`

表现：

- 访问登录页时，前端先检查配置
- 然后自动跳转到钉钉扫码登录
- 访问注册页时，同样自动跳转到钉钉扫码登录
- 传统公开登录/注册接口在后端也会被拒绝

适用场景：

- 最终统一上线
- 对外彻底关闭公开账号密码入口

### 情况 D：门禁开了，但钉钉能力没开

配置：

- `ENABLE_DINGTALK_AUTH=false`
- `ENABLE_DINGTALK_GATE=true`

表现：

- 前端会提示“门禁模式已开启，但钉钉入口未开放”
- 后端公开登录/注册入口会被门禁策略限制
- 这属于错误配置，不应该长期存在

结论：

- 想要“直接跳钉钉”，必须是情况 C
- 只开 `ENABLE_DINGTALK_GATE` 而不开 `ENABLE_DINGTALK_AUTH`，不会得到正确的门禁体验

## 五、为什么这次没有去改 Nginx

这是一个架构边界问题，要讲清楚。

### 当前实现层

我们这次改的是：

- Vue 前端页面
- 也就是浏览器拿到前端应用以后，由前端代码决定是否立刻跳转到钉钉

### 为什么不是服务端 302 直接跳

因为现在的 `/login` 和 `/register` 本质上是：

- 前端路由
- 由静态页面承载
- 不是 Node 后端的服务端渲染页面

所以“打开页面瞬间就服务端 302 到钉钉”这件事，如果要做到极致，通常需要：

- Nginx 层做条件路由
- 或者服务端动态渲染接管登录入口

这次先不改 Nginx，有两个原因：

- 当前 M2 主线仍是 FB 钉钉登录闭环，不是网关重构
- 用前端首屏配置检查 + 立即跳转，已经足以满足现在的产品行为要求

后续如果你追求“连过渡页都不要看到”，那就是下一层架构优化，应该单独立项讨论，而不是混进这次 M2 收口里。

## 六、这次修改后的验证方法

### 验证前准备

先确认 `.env.auth` 里开关值正确。

如果你要验证“自动跳钉钉”，请设置为：

```env
ENABLE_DINGTALK_AUTH=true
ENABLE_DINGTALK_GATE=true
```

如果你暂时不想正式放开云端门禁，请保持：

```env
ENABLE_DINGTALK_AUTH=false
ENABLE_DINGTALK_GATE=false
```

### 云端生效步骤

如果你改的是前端页面：

```bash
cd /root/work/FB-Ad-Logic-Engine
npm run build
docker restart fb-ad-nginx
```

如果你改了 `.env.auth` 开关：

```bash
cd /root/work/FB-Ad-Logic-Engine
sudo systemctl restart fb-ad-logic-engine.service
```

注意：

- 前端代码变了，要重新 `build`
- 后端环境变量变了，要重启 `systemd`
- 这两类动作不是一个东西，少做一个都会出现“我明明改了但页面没变”的错觉

### 验证项 1：门禁关闭时

配置：

```env
ENABLE_DINGTALK_AUTH=false
ENABLE_DINGTALK_GATE=false
```

预期：

- 访问 `http://ad_tools.thinkpro.top/login`
- 可以看到传统用户名密码登录页
- 不自动跳钉钉

### 验证项 2：普通钉钉登录模式

配置：

```env
ENABLE_DINGTALK_AUTH=true
ENABLE_DINGTALK_GATE=false
```

预期：

- 登录页仍然存在
- 页面上出现“钉钉扫码登录”按钮
- 不会自动跳钉钉

### 验证项 3：门禁自动跳转模式

配置：

```env
ENABLE_DINGTALK_AUTH=true
ENABLE_DINGTALK_GATE=true
```

预期：

- 访问 `http://ad_tools.thinkpro.top/login`
- 页面不再展示传统账号密码表单
- 浏览器会自动跳到钉钉授权页

补充验证：

- 访问 `http://ad_tools.thinkpro.top/register`
- 同样应该自动跳钉钉

### 验证项 4：错误配置保护

配置：

```env
ENABLE_DINGTALK_AUTH=false
ENABLE_DINGTALK_GATE=true
```

预期：

- 登录页不应继续放出传统表单
- 页面会提示门禁已开但钉钉入口未开放

这个验证的目的不是正常上线，而是确保错误配置时不会出现“门禁想开没开成，结果账号密码入口又漏出来”的事故。

## 八、为什么这份文档重要

你前面已经明显遇到过几个非常典型的“多层系统调试错觉”：

- 你改了前端，但没重新构建
- 你改了环境变量，但没重启后端
- 你以为开关没生效，其实是拼写写成了 `flase`
- 你以为门禁没做，其实做的是“引导式门禁”，不是“自动跳转式门禁”

这些问题都不是技术太难，而是：

- 前端
- 后端
- 配置
- 部署

四层同时存在，任何一层没同步，现象都会骗人。

这份文档的作用，就是把这些“很容易搞混的层次”彻底拆开。
