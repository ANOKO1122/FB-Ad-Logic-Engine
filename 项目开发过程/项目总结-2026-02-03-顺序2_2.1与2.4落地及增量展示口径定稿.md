## 一、本窗口完成的工作

### 2. 2.4 手动同步结构实现与验收

- **实现**：`server/services/structureSyncService.js`（锁 + 冷却 + 分页拉取 + upsert）；`server/index.js` 增加 `getStructurePage` 的 `updated_time` 映射、`POST /api/sync/structure` 路由。
- **约束**：账户级锁（sync:structure:{accountId}）、冷却期 2 分钟（UNIX_TIMESTAMP(MAX(last_synced_at)) 判定，避免驱动时区误判）、AdsPolar 状态过滤（仅拉 6 类状态，剔除 DELETED/ARCHIVED）。
- **验收**：首次同步 200 + synced_count；2 分钟内再次请求 429 + 冷却日志；DB 中 structure_ads 条数与状态分布符合预期。

### 3. TASKS / DEV_PLAN 口径多轮补全（仅文档，无代码）

- **结构同步增量化**：首次全量、增量（updated_time 回看）、structure_sync_status 表（last_sync_updated_ts、last_full_count、has_full_synced 等）、回退全量判定、限流与并发。
- **选择器本地展示过滤**：默认 ACTIVE、include_paused=1 加 PAUSED、q 非空放宽到 6 类；本期选用 A（安全优先，q 非空仍排除 DELETED/ARCHIVED），验收不包含「能搜到已归档/删除」。
- **增量时间格式写死**：对外（FB filtering）ISO8601 UTC Z；对内（游标）Unix 秒（last_sync_updated_ts）；structure_ads.updated_time 仍为 ISO8601 字符串（与 014 一致），入库前标准化 UTC Z。
- **回退全量判定**：主规则为「增量 0 条 且（current_local_count==0 或（last_full_count>0 且 current_local_count < last_full_count×0.3）」；**last_full_count=0 时不做 ×0.3 比较**，只按 current_local_count==0 判定；不得仅因「增量返回 0 条」就回退。
- **5 分钟回看窗口算法**：cursorTs = last_sync_updated_ts；sinceTs = max(0, cursorTs - 300)；对外 sinceISO = sinceTs 转 ISO Z，用 updated_time > sinceISO 拉增量；入库 upsert；游标推进取 max(updated_time) 转 Unix 秒写回，0 条则游标不变；回看窗口只用于查询起点，不影响游标推进。

## 二、本窗口遇到的问题与解决方法

| 问题 | 解决方法 |
|------|----------|
| 冷却期不生效，连续两次同步都返回 200 | mysql2 返回的 last_synced_at 被按本地时区解析，elapsed 差约 8 小时。改为用 **UNIX_TIMESTAMP(MAX(last_synced_at))** 取 UTC 秒再与 Date.now() 比较，冷却 2 分钟正确触发。 |
| 拉取广告数量过多（含 DELETED/ARCHIVED） | 结构同步增加 **AdsPolar 状态过滤**：仅拉 ACTIVE/PAUSED/PENDING_REVIEW/DISAPPROVED/IN_PROCESS/WITH_ISSUES，不拉 DELETED/ARCHIVED。 |
| 回退全量判定「增量返回 0 条且上次全量有大量对象」过于激进 | 改为 **主规则**：增量 0 条 且（current_local_count==0 或（last_full_count>0 且 current_local_count < last_full_count×0.3））；**last_full_count=0 时**只按 current_local_count==0 判定；明确不得仅因「增量 0 条」就回退。 |
| 增量格式「二选一」导致实现歧义 | **写死**：对外 FB filtering 用 **ISO8601 UTC Z**；对内游标用 **Unix 秒**（last_sync_updated_ts）；structure_ads.updated_time 仍为 **ISO8601 字符串**（与 014 一致）；文档禁止「FB filtering 用 Unix 秒」。 |
| 回退判定依赖的 current_local_count / last_full_count 无表字段 | 在 structure_sync_status 表定义中**明确** last_sync_updated_ts、last_full_count、has_full_synced、last_full_success_at；current_local_count = COUNT(*) FROM structure_ads WHERE account_id=?。 |
| 选择器 A/B 同时存在导致落地不知选哪种 | **本期选用 A**（安全优先）；B 留待后续任务；验收不包含「能搜到已归档/删除广告」。 |
