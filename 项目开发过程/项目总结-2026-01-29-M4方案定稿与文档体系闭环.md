### 1. M4 动作执行层落地方案从无到有、多轮定稿

- **产出**：`docs/M4_动作执行层落地步骤.md`（完整步骤、硬口径、验收点、实现时注意）。
- **内容要点**：
  - 落地顺序：步骤 1+6（仲裁+摘要/冷却）→ 步骤 2（预算幂等）→ 步骤 3（Pre-Flight）→ 步骤 4（Smart Mute）→ 步骤 5（预算单位统一）。
  - 硬口径：run_id 采用**加列**；冷却采用**评估即冷却**；Pre-Flight **不新增**对每个广告的 GET FB，仅用本地快照 status + FB already-in-that-state 容错。
  - 仲裁：同一 account + 同一 runId 下所有可执行规则（跨 user）；每 ad 只执行一次动作；同优先级同动作类型去重时赢家为 ruleId 最小者；仲裁输出必须含 matchedAd；每 ad 固定一个 matchedAd 来源。
  - 权限：汇总前按用户权限过滤，无权规则不 evaluateRule、只写 no_permission 摘要。
  - 三条实现时注意：Pre-Flight 状态等价值（PAUSED/DISABLED/ARCHIVED 等）；预算幂等 actionSignature 含所有影响目标值字段；仲裁 matchedAd 取谁、执行层始终用同一份。

### 2. 文档多轮修正，消除歧义与冲突

- **动作优先级**：pause_ad 与 activate_ad 写死 tie-break（pause 优先于 activate），避免同一广告同时命中暂停/激活规则时结果随遍历顺序变化。
- **多动作规则**：每条规则内部只取「最狠一条 action」参与仲裁，仲裁输入压成「每规则每 ad 一条候选」。
- **仲裁作用域**：明确为「同一 account + 同一 runId 下所有可执行规则」，从根上避免跨 user 重复改同一广告。
- **仲裁输出**：补全 matchedAd（含 status、ad_set_id、mute_*），供 Pre-Flight / Smart Mute / 预算执行使用。
- **同动作去重赢家**：写死为 ruleId 最小者，审计/摘要稳定。
- **摘要模型**：同一规则可「部分 ad 赢、部分 ad 被压制」落一行，用 executed_count + skip_details.suppressed_for_ads 表达。
- **预算幂等**：禁止在 rule.actions[] 原对象上挂 _resolvedBudgetCents；用本 ad action 副本或上下文缓存（actionSignature 含全量影响字段）。
- **冷却口径**：二选一定死为「评估即冷却」；目标 #6 与步骤 1.4 引用一致。
- **Pre-Flight**：全文删除「轻量级 GET / 额外 GET 查询广告当前状态」的旧表述，统一为「不新增 GET、本地 status + FB 容错」。

## 二、遇到的问题与解决方法

| 问题 | 解决方法 |
|------|----------|
| 落地顺序与「6 条方法编号」混淆 | 统一表述为「按本文步骤编号：1+6→2→3→4→5」，并在开头注明对应内容（仲裁→预算幂等→Pre-Flight→Smart Mute→预算单位）。 |
| 仲裁只在 user 内做导致跨 user 重复改同一 ad | 明确仲裁作用域为「同一 account + 同一 runId 下所有可执行规则」；汇总前按用户权限过滤，无权规则不 evaluateRule、只写 no_permission。 |
| 仲裁输出缺 matchedAd，Pre-Flight/Smart Mute/预算无法落地 | 仲裁输出结构补为 `{ winnerRule, winnerAction, matchedAd, suppressedRules }`，且每 ad 固定一个 matchedAd 来源。 |
| pause_ad 与 activate_ad 同优先级导致结果不稳定 | 写死 tie-break：pause_ad=1、activate_ad=2，同优先级同动作类型去重时赢家为 ruleId 最小者。 |
| 多动作规则与「每 ad 只执行一次」冲突 | 约定每条规则内部只取「最狠一条 action」参与仲裁，仲裁前压成「每 (rule, ad) 一个 candidateAction」。 |
| _resolvedBudgetCents 挂在 rule.actions 导致串 ad | 禁止在原对象上挂载；改用本 ad 的 action 副本或上下文缓存（actionSignature 含 type、value、max_daily_budget 及护栏）。 |
| Pre-Flight 旧口径「轻量级 GET」与硬口径冲突 | 在 DEV_PLAN 6.3 和全文档中删除「调用一次轻量级 GET 查询广告当前状态」，统一为「不新增 GET、本地 status + FB 容错」。 |
| 其他历史文档仍含旧表述 | 约定「施工以 TASKS/DEV_PLAN 为准」，历史文档不参与施工，避免走偏；必要时可后续标注归档或按需修正。 |

## 三、项目进度与目标总结

### 当前阶段

- **Phase 3（规则重心下沉与 7x24h 自动化）** 进行中：

  - M3 规则配置层：基础功能已完成（约 80%）。
  - **M4 动作执行层**：方案与文档已定稿（约 0% 代码落地），下一窗口按 TASKS.md 第 3 节实施。
  - M5 反馈与监控层：部分完成（规则执行摘要、展示层权限、前端时区等约 30%）。

### M4 目标（6 条验收标准）

1. **按广告仲裁**：同一轮同一 ad_id 只执行一次动作；同狠度同动作去重；记录赢家 ruleId、被压制 ruleId 列表（写入摘要）。
2. **幂等**：预算动作重试不二次叠加；第一次算出 newBudgetCents 后，重试只 POST 同一值。
3. **Smart Mute**：执行链路能拿到 mute_until/mute_reason；now < mute_until 则跳过并写 skip_reason。
4. **Pre-Flight**：不额外 GET FB；用本地 ad_snapshots.status 判断「目标已达成则 skipped」；FB 返回 already-in-that-state 容错为 skipped/success。
5. **预算单位**：后端全用「分」(int)；前端展示除 100。
6. **冷却**：命中也要冷却；被压制规则写摘要 skip_reason=suppressed_by_priority；本项目采用评估即冷却。

## 四、下一窗口特别注意

- **run_id 加列迁移**：步骤 3.1 中「automation_logs 表增加 run_id 列」的迁移脚本必须在数据库侧**实际执行并生效**，否则后续按 run_id 做「同一 run 同一 ad 仅一条」验收会卡住。建议在完成迁移后立刻用 `SELECT run_id, ad_id, COUNT(*) ... GROUP BY run_id, ad_id` 做一次抽样校验。
- **权限过滤**：汇总「该 account 下所有 cooled-down 规则」时，必须先按用户权限过滤，只纳入对该 account 有访问权的用户的规则，与现有 cronService 的权限行为一致，避免越权数据读取。
