# 凌晨新广告盲视 — 根因与修复方案

## 现象

natie 账户 5/14 06:16 出现 34 条广告集中暂停。排查发现：凌晨 00:00-06:00 系统只同步到 10 条老广告，06:15 突然出现 46 条新广告，全部命中暂停规则。

## 根因

系统按账户本地时间分两条数据同步路径：

| 时段 | 函数 | 广告来源 |
|---|---|---|
| 06:00-18:00 | `syncAccountTodayStats` | Facebook 账户级 Insights（无 ad_id 限定） |
| 00:00-06:00 | `syncAccountSlidingWindow` | 数据库 `filterActiveAds`（查已知 spend>0 的 ad_id） |

`filterActiveAds` 从 `daily_stats` + `ad_snapshots` 查询：

```sql
SELECT DISTINCT ad_id FROM daily_stats WHERE spend>0 AND date>=...
UNION
SELECT DISTINCT ad_id FROM ad_snapshots WHERE spend>0 AND data_date>=...
```

然后 `fetchInsightsInBatches` 对每个已知 ad_id 逐个发请求：

```
/{adId}/insights?date_preset=today
```

新创建且首次产生花费的广告不在数据库任何表中 → `filterActiveAds` 查不到 → `fetchInsightsInBatches` 不会请求 → 凌晨全程盲视。

直到 06:00 切到白天路径，`syncAccountTodayStats` 发送账户级请求 `/{accountId}/insights?date_preset=today`，一次性拉回所有 spend>0 广告（含新广告）→ 集中命中规则 → 雪崩。

## 影响范围

- 全部 61 个活跃账户，不限于 natie
- 全部使用 today 时间窗口的规则，不限于测新广告暂停规则
- 仅在 00:00-06:00（账户本地时间）发生

## 分步修复计划

**文件**：`server/services/ingestorService.js`
**函数**：`syncAccountSlidingWindow`（第 940 行起）
**插入点**：当前第 1018 行后（`filterActiveAds` 完成、`resolveObjectsByIds` 之前）

```
1005: let targetAdIds = await filterActiveAds(...)          // ← DB 已知 ID
1018: logger.info(已查到 N 个活跃广告)                       // ← 插入点
1020: // 6. 获取广告状态
1024: resolveObjectsByIds(targetAdIds, ...)                  // ← 状态解析
1039: fetchInsightsInBatches(accountId, targetAdIds, ...)    // ← insights 拉取
```

---

### 步骤 1：新广告嗅探 + 错误隔离（核心修复）

在 `filterActiveAds()` 和 `resolveObjectsByIds()` 之间，插入账户级 Insights 嗅探。嗅探失败不中断主流程，回退到原有同步逻辑。

**插入位置**：第 1018 行后，第 1020 行前。

```javascript
// [修复] 凌晨滑动窗口无法发现新广告：补一个账户级 Insights 嗅探
// 仅在 DB 已知活跃广告数 > 0 时执行（无活跃广告时已在 1007 行提前 return）
let discoveredAdIds = []
try {
  const discovered = await facebookApi.getAdInsights(accountId, { preset: 'today' }, {
    level: 'ad',
    useAccountAttributionSetting: true,
    spendGreaterThanZero: true,
    signal
  })
  discoveredAdIds = [...new Set(
    (discovered || []).map(i => String(i.ad_id || '')).filter(Boolean)
  )]
  const newIds = discoveredAdIds.filter(id => !targetAdIds.includes(id))
  if (newIds.length > 0) {
    logger.info(`🔍 发现 ${newIds.length} 个新广告（DB 未记录），合并到同步列表`)
    targetAdIds = [...targetAdIds, ...newIds]
  }
} catch (err) {
  // 嗅探失败不中断主流程：已知广告的同步照常进行
  logger.warn(`⚠️ 新广告嗅探失败，跳过: ${err.message}`)
}
```

**设计要点**：
- `try/catch` 包裹整个嗅探调用。超时、FB 错误、signal abort 均被捕获，不会阻断后续的 `resolveObjectsByIds` 和 `fetchInsightsInBatches`
- `targetAdIds` 已声明为 `let`（原代码即如此），直接追加新 ID
- 嗅探返回的数据仅提取 ad_id，metrics 数据不复用（原因见步骤 3）

**验证**：凌晨任意 cron 周期，日志中出现 `🔍 发现 N 个新广告` → 表示生效。若出现 `⚠️ 新广告嗅探失败` → 表示降级成功，已有广告仍正常同步。

---

### 步骤 2：新广告 status 自动覆盖（无需额外代码）

`resolveObjectsByIds` 接收的是 `targetAdIds`（已包含追加的新 ID），新广告的状态元数据（name / effective_status / campaign_id / adset_id）在同一轮中被批量解析。

`fetchInsightsInBatches` 同样接收 `targetAdIds`，新广告的 insights 通过 Batch API 按 ad_id 拉取。

**无需额外代码改动。**

---

### 步骤 3（可选，后续优化）：消除已知广告数据重复拉取

步骤 1 中 `getAdInsights` 返回了所有 spend>0 广告的完整 metrics（含 spend、actions、purchase_value 等），但只提取了 ad_id。已知广告的数据随后又被 `fetchInsightsInBatches` 按 ad_id 重拉一遍。

**暂不实施的原因**：
1. `getAdInsights` 和 `fetchInsightsInBatches`（Batch API）返回的数据结构不完全一致 — 字段名、嵌套层级有差异，直接复用需要适配
2. 凌晨新广告数量通常较少（natie 案例中 46 条），重拉的 API 开销可接受
3. 改动范围会扩展到 `saveSnapshotsToDb` 的调用方式，风险高于收益

若后续观察到 API 配额瓶颈，可在此处做优化：过滤出 `newIds` 对应的 insights 数据直接走 `saveSnapshotsToDb`，将其从 `targetAdIds` 中移除避免 `fetchInsightsInBatches` 重拉。

## 副作用

| 副作用 | 程度 | 处理 |
|---|---|---|
| 每个账户多加 1 次 `getAdInsights`（凌晨约 24 次/账户） | 可控 | 61 账户 × 24 次 = 约 1464 次/晚 |
| 新广告凌晨即被规则评估（不再等到 06:00） | 预期行为 | 规则本就是全天评估的 |
| 新广告的 insights 被拉两次（嗅探 + Batch） | 小幅浪费 | 新广告数少，后续可优化（步骤 3） |
| 嗅探失败时回退原有逻辑 | 已处理 | try/catch 隔离 |
| 大账户分页延迟 | 已处理 | `getAdInsights` 自带分页循环，pLimit(6) 限制并发 |
| 首轮后嗅探持续空跑（新广告已入库 → filterActiveAds 已覆盖） | 小幅浪费 | 24 次中仅首轮有增量价值，后续可加连续空跑计数跳过，但性价比低 |

## 为什么不改白天路径

白天 `syncAccountTodayStats` 已使用 Insights First 策略（账户级查询无 ad_id 限定），天然覆盖新广告，无需改动。
