# 云端项目总结-2026-04-15-动态筛选执行路径 union 误判与 Git 收口

## 一、背景与目标

- **业务现象**：多条规则在界面配置了**动态筛选**（例如名称「包含」`常规-平稳版`、`测新`、`帖子`），但执行日志中出现对**名称明显不符**的广告执行 `PAUSE_AD` 等动作（典型：广告名为「平稳版」，不包含配置中的关键字）。
- **技术目标**：厘清「动态重算 / `rule_matched_objects` 快照」与「规则引擎按账户批处理时的目标解析、缓存评估」两条链路，定位**执行阶段**是否在未严格按快照过滤的情况下误评广告；在代码层关闭该路径，并补充单测与可检索日志。
- **协作目标**：云上服务器曾直接修改运行目录中的源码，需将修复**纳入 Git 远端**，使本地与其它环境 `git pull` 后与线上一致。

| 主题 | 说明 |

## 三、根因分析（机制层）

### 3.1 核心缺陷（H5）：`evaluateRuleWithCache` 在动态快照为空时误用 union 全量

- **路径**：`loadDataForAccount` 按账户一次拉取多条规则，将各规则的 `targetAdIds` 做 **union**，再按时间窗口去重查询指标，得到 `fullData`。
- **旧逻辑问题**：当某条规则 **`use_dynamic_scope=1`** 且从 `rule_matched_objects` 解析出的 **`targetIds` 为空** 时，`filterSet` 为 `null`，代码将 **`fullData` 原样**交给 `evaluateRuleWithData`，等价于用「本批其它规则贡献的广告 union」评估**本条**规则的条件。
- **后果**：名称类动态筛选被**事实绕过**——只要指标满足 IF，就可能对**不在本条规则快照内**的广告执行动作（例如「测新」规则误伤「平稳版」）。

### 3.2 次要缺陷（H1）：动态快照读库失败时回退「全账户广告」

- **路径**：`resolveTargetAdIdsForRule` 在 `use_dynamic_scope=1` 时优先 `SELECT rule_matched_objects`；**旧逻辑**在 `catch` 后继续走「无目标则拉全账户 `structure_ads`」分支。
- **后果**：快照不可读时，目标集合膨胀为全账户广告，同样会突破名称筛选语义。

### 3.3 与「重算 / LIKE」的区分

- **动态重算**（`calculateMatchedAdIdsForRule` + `buildStructureFilter`）对广告级名称条件使用 `name LIKE ?` 与参数 `%value%`，与产品「包含」一致；**问题主因不在 LIKE 拼错**，而在**执行调度层**对空快照与失败回退的处理。
- **数据侧**：若某 `ad_id` **不在**规则 A 的 `rule_matched_objects` 中，但日志仍显示规则 A 对其执行，则更支持 **H5 执行路径**而非「快照误插该行」。

### 3.4 SQL 校验时的常见误用（排障经验）

- 每条规则的 `scope_filters.value` **不同**（例如 778 为「测新」、770 为「帖子」）。审计「快照是否与名称条件矛盾」时，应使用 **该规则配置中的关键字** 构造 `NOT LIKE '%关键字%'`，**禁止**用 A 规则的关键字去审 B 规则的快照，否则会出现「合法行被当成异常」的假阳性（例如 770 下用 `%测新%` 会列出全部「帖子」广告）。

## 四、解决方法（代码与验证）

### 4.1 代码修改（落点：`server/services/ruleEngineDispatcher.js`）

1. **`evaluateRuleWithCache`**  
   - 当 **`use_dynamic_scope=1` 且 `targetIds.length === 0`**：不再使用 union 的 `fullData`，改为 **`evaluateRuleWithData(rule, [])`**。  
   - 若此时 `fullData.length > 0`，打 **`logger.warn`**，文案含固定关键字 **`DYNAMIC_SCOPE_EMPTY_SKIP_UNION`**，便于线上检索。

2. **`resolveTargetAdIdsForRule`**  
   - 动态快照 `SELECT` **抛错**时：**不再**回退到全账户广告；**fail-closed** 返回 **`{ targetAdIds: [], queryCount: 0 }`**，并打 warn 说明「本规则本轮跳过评估」。

### 4.2 单测（`server/tests/ruleEngineDispatcher.test.js`）

- 新增用例：**`use_dynamic_scope=1` 且快照目标为空时不得使用 union 全量数据**（构造 `loadResult` 中 `targetAdIdsByRuleId` 为空、`cache` 中非空，期望匹配数为 0）。

### 4.3 调试手段的演变

- 曾在 Debug 模式下使用向本机 ingest 的 **`fetch` 埋点**；生产环境通常无对应服务，**不宜长期保留**。  
- 收口后改为 **`logger.warn` + 固定关键字**，避免依赖 Cursor 调试通道。

### 4.4 数据库侧验证（MySQL，示例思路）

- **`rule_matched_objects`**：某 `rule_id` + `account_id` + `object_id` 是否存在（判断该规则快照是否包含该广告）。  
- **`structure_ads`**：该 `ad_id` 的 `name` 是否与 `scope_filters` 的「包含」语义一致。  
- **审计 SQL**：`rule_matched_objects` 与 `structure_ads` 按账户+广告 join，筛出「`name` 不满足 `NOT LIKE '%配置关键字%'`」应为 **Empty set**（配置为单一 `contains` 时近似适用）。

**本次会话中已核对的事实摘要**（以用户当时库表为准）：

- 规则 **778** + 广告 `120244987662560603`：快照中 **无** 该行，且 `structure_ads.name` 为「平稳版」——与「测新」不符；支持**执行路径**曾出错而非快照含该行。  
- 规则 **770**：快照 **8** 条；用「帖子」做 `NOT LIKE '%帖子%'` 审计为 **Empty set**，快照与名称条件一致。
