## 二、遇到的问题与解决方法

### 2.1 MySQL 预编译语句 LIMIT 占位符报错（Incorrect arguments to mysqld_stmt_execute）

**现象**：执行 `node server/scripts/run-history-cleanup.js` 或 04:30 定时任务时，报错 `Incorrect arguments to mysqld_stmt_execute`，堆栈指向 `pool.execute` 与 `runNightlyHistoryCleanup`。

**根因**：在 MySQL 预编译语句协议中，**LIMIT（及 OFFSET）不能作为绑定参数**（`?`），只能写在 SQL 文本里。代码中使用了 `DELETE FROM ... WHERE ... LIMIT ?` 并传入 `[retentionDays, HISTORY_CLEANUP_BATCH_SIZE]`，驱动/服务端不支持对 LIMIT 的占位符绑定。

**解决**：将 LIMIT 改为 SQL 内联常量（`HISTORY_CLEANUP_BATCH_SIZE` 为代码常量 10000，安全）。三处 DELETE 均改为形如：
- `DELETE FROM rule_matched_objects_history WHERE refreshed_at < NOW() - INTERVAL ? DAY LIMIT 10000`，参数仅 `[retentionMatched]`；
- 同理 structure_ads_history、rule_history 只保留一个天数占位符，LIMIT 写死 10000。

修改后脚本与 Cron 任务均正常执行。

### 2.2 单次清理脚本设置保留期 0 仍显示 30/60 天且删除为 0

**现象**：为测试「全删」设置环境变量 `HISTORY_RETENTION_DAYS_MATCHED=0`、`HISTORY_RETENTION_DAYS_ADS=0`、`HISTORY_RETENTION_DAYS_RULE=0` 后，脚本输出仍为「保留期: 30 天, 60 天」，且三张表删除行数均为 0。

**根因**：脚本与 cronService 中保留期取值使用了 `Number(process.env.XXX) || 30`（或 60）。在 JavaScript 中 `0` 为 falsy，`Number("0") || 30` 结果为 30，导致显式传入的 0 被当作「未设置」而回退到默认值；WHERE 条件仍是「超过 30/60 天」，当天数据不会被删。

**解决**：当前生产环境使用默认 30/60 天即可，无需传 0。若需「全删」测试，可临时改代码或后续改进：用「是否在 env 中出现」区分「未设置」与「显式 0」，例如 `process.env.XXX !== undefined && process.env.XXX !== '' ? Number(process.env.XXX) : 30`，这样传入 0 才会生效。

## 三、已完成的实现与验收

### 3.1 建表（步骤 1）

| 迁移 | 表名 | 用途 |
|------|------|------|
| 037 | rule_history | 规则配置变更历史（CREATE/UPDATE/DELETE/TOGGLE/SYSTEM_REFRESH） |
| 038 | rule_matched_objects_history | 动态快照变动（按规则+账户 diff，added_count/removed_count、大列表 checksum） |
| 039 | structure_ads_history | 广告 name/effective_status 变更（按账户 ad 维度） |

索引满足方案要求（如 rule_id+changed_at、rule_id+refreshed_at+account_id、account_id+ad_id+changed_at 等）。

### 3.2 rule_history 写入（步骤 2、3）

- **ruleHistoryService.js**：`buildRuleSnapshot(rule)`（仅配置字段）、`insertRuleHistory(opts)`，支持可选 `connection` 在事务内写入。
- **rulesService.js**：createRule/updateRule 成功后写 rule_history（CREATE/UPDATE/TOGGLE），deleteRule 改为同一事务内：SELECT 规则 → INSERT rule_history(DELETE) → DELETE rule_matched_objects → DELETE rules → commit。
- **routes/rules.js**：向 createRule/updateRule/toggleRule 传入 `req.user.owner_id` 作为 ownerId。
- **dynamicScopeService.js**：在批量/单条 UPDATE rules（dynamic_scope_status、target_ids/target_by_account）之后调用 `insertRuleHistory(SYSTEM_REFRESH, dynamic_scope_refresh)`。

验收：CREATE/UPDATE/TOGGLE/DELETE 及 SYSTEM_REFRESH 均有记录；按 rule_id、change_type 查询正常。

### 3.3 rule_matched_objects_history（步骤 4）

- **dynamicScopeService.js**：在 `refreshDynamicTargetsForRulesInAccount` 中，事务内、DELETE 前一次性 SELECT 该账户当前 rule_matched_objects 的 object_id 得到 oldSet；DELETE+INSERT 后按规则做 diff，仅当 added_count 或 removed_count > 0 时 INSERT rule_matched_objects_history；快照窄化：object_count≤500 存最多 200 个 ID，>500 存前 100 个 ID + MD5(object_ids) 的 checksum。

验收：连续两次相同刷新不产生新记录；有增/删时产生一条；大列表不存全量 JSON。

### 3.4 structure_ads_history（步骤 5）

- **structureSyncService.js**：引入队列 `structureAdsHistoryQueue`，上限 `STRUCTURE_ADS_HISTORY_QUEUE_MAX`（默认 5000）；`loadStructureAdsMapForAccount(connectionOrPool, accountId)` 一次性拉取 ad_id/name/effective_status；`pushStructureAdsHistory(record)` 满则丢弃并打 Warn；`flushHistoryQueue(timeoutMs)` 批量 INSERT structure_ads_history；每 5 秒定时 flush。
- 四处写 structure_ads 的路径均改为「先加载该账户 Map，循环内仅内存比较，有变更则 push」：doStructureAdsUpsertAndStatus、piggybackStructureFromToday、runPseudoIncrementForAccount、applyMergedFastSyncPayload 的 upsertAds（source 分别为 structure_sync、piggyback、pseudo_increment、track2_fast_sync）。
- **server.js**：优雅退出时在关闭 DB 前调用 `flushHistoryQueue(10000)`，再执行原有 processWriteQueue 等。

验收：未改名/改状态不产生 history；主同步不因 history 失败而失败；循环内无额外 SELECT；重启/部署前队列中未写数据被 flush 落库。

### 3.5 历史表定时清理（步骤 6）

- **cronService.js**：在「每日 04:00 热表清理」之后新增 **04:30** 任务（Cron：`30 4 * * *`）。`runNightlyHistoryCleanup(opts)` 对三张表分批 DELETE（每批 LIMIT 10000），批与批之间 sleep(500ms)，直到某批 affectedRows < 10000 结束该表；保留期由 opts 或环境变量传入（默认 matched 30 天、ads/rule 60 天）。
- 环境变量：`ENABLE_HISTORY_CLEANUP`（默认启用，设为 0 或 false 可关闭）、`HISTORY_RETENTION_DAYS_MATCHED`、`HISTORY_RETENTION_DAYS_ADS`、`HISTORY_RETENTION_DAYS_RULE`。
- **单次执行脚本**：`server/scripts/run-history-cleanup.js`，调用同一套 `runNightlyHistoryCleanup`，便于本机验证或手动跑一次清理。

验收：任务日志可见每表删除行数；超期数据被清除；不影响凌晨同步 IO（批间 sleep）。
