# 项目总结-2026-04-08-动态筛选适用账户并集修复与验证

## 一、项目目标与执行方案

- **业务目标**：动态筛选规则在配置了「多账户名单 `target_account_ids`」与「按户手动物体 `target_by_account`」双写时，**适用账户**应与产品预期一致：**凡出现在任一侧、且应参与该规则刷新路径的账户，都不应被静默丢弃**。
- **技术目标**：修正 `getRuleTargetAccountIds` 的取数口径，使「按户有 key 时仅返回 key、忽略名单」的旧逻辑改为 **并集**（`target_by_account` 中非空 key + `target_account_ids` + 主归属 `account_id`），并与 `fetchDynamicScopeRulesForAccount` 中的 `.includes(accountId)` 二次过滤对齐。
- **执行方案（唯一依据的 plan）**：**`.cursor/plans/动态筛选适用账户口径修复_落地方案.plan.md`**（《动态筛选适用账户口径修复（落地方案）》）。落地内容对应该 plan 的 **§3 推荐修复策略（代码优先）** 与 **§4 实施步骤 checklist**；仓库内其他泛化方案（如云本地协同总方案等）**不作为**本缺陷的执行依据。
- **代码落点**：`server/services/dynamicScopeService.js`（`getRuleTargetAccountIds`、`normalizeAccountId` 等）。

## 二、遇到的问题与解决方法

### 2.1 现象：仅在 `target_account_ids` 中的新户无动态快照

**现象**：规则 604～608、697 开启动态范围，`structure_ads` 中账户 `act_832525359903650` 存在名称含「测新」的广告，但 `rule_matched_objects` 中该账户与上述规则 **无记录**（`Empty set`）。同账户下规则 **762** 快照正常（例如 24 条）。

**根因**：`getRuleTargetAccountIds` 在 **`target_by_account` 存在至少一个非空数组的 key** 时，**只返回这些 key**，**不再合并** `target_account_ids`。而 `fetchDynamicScopeRulesForAccount` 在 SQL 查出候选规则后，又用 `getRuleTargetAccountIds(rule).includes(accountId)` 过滤。于是：**只出现在 `target_account_ids`、未作为 `target_by_account` key 出现的账户**（典型为新户）不会进入该账户维度的动态刷新 → `rule_matched_objects` 长期为空。这与「关广告 / spend=0」无关，是 **适用账户两套字段在刷新路径上口径不一致**。

**解决**：重写 `getRuleTargetAccountIds`：用 `Set` 收集 **(1)** `target_by_account` 中非空数组对应的账户 key；**(2)** `target_account_ids` 数组中的每一项；**(3)** 规则主归属 `account_id`；经 `normalizeAccountId` 去重后排序返回。保证与「并集」业务含义一致。

### 2.2 辅助：账户 ID 归一化

**说明**：若存在 `act_act_` 重复前缀等脏数据，需在归一化层收紧，避免 `includes` 比较失败。具体以 `dynamicScopeService.js` 中 `normalizeAccountId` 实现为准。

### 2.3 验证中的解读：`act_1477243063804898` 查询为空

**现象**：对 `act_1477243063804898` 执行 `rule_matched_objects` 中规则 604～608、697 的计数查询，得到 **Empty set**。

**说明**：这 **不能单独** 用来否定修复是否生效。可能原因包括：（1）该账户在部署/刷新后 **尚未触发** 针对这些规则的动态快照刷新；（2）该账户 **`structure_ads` 中无满足 `scope_filters`（如名称含「测新」）的广告**，快照本应为 0；（3）该账户 **未出现在** 对应规则的 `target_account_ids` 或适用集合中（需与 `rules` 表配置核对）。

**结论**：以 **已知命中问题场景的账户** `act_832525359903650` + 规则 604 的快照与结构侧「测新」广告 **一致** 为主验收依据（见下文第三节）。

## 三、验收结论（是否算验证通过）

**可以认定本次问题在目标场景下修复成功**，依据如下。

| 验证项 | 结果 | 说明 |
|--------|------|------|
| `act_832525359903650` + 规则 **604** | `rule_matched_objects` 与 `structure_ads` join 后 **5 条**，广告名均含「测新」 | 该户属于「仅在 `target_account_ids`、非 `target_by_account` key」类；修复前此处为空，修复后与结构数据一致 |
| `act_832525359903650` + 规则 **762** | **24** 条 | `target_by_account` 为空时历史路径本就正常，用于对照无回归 |
| `act_1477243063804898` + 604～697 | Empty | 见 2.3，不作为否定修复的单一证据 |

**总结**：核心回归点 **`act_832` + 604（及同批 605～608、697 在联调中的 cnt 与预期一致）** 已满足「并集后新户参与刷新、快照非空且与筛选条件一致」的预期，可收口本次缺陷修复。

## 四、项目过程简述

1. **问题定位**：通过规则元数据（`JSON_KEYS(target_by_account)` 与 `target_account_ids` 长度）、`rule_matched_objects` 与 `structure_ads` 对比，区分「执行轨道 / 花费」与「未进入刷新」两类原因，锁定 `getRuleTargetAccountIds` 与 `fetchDynamicScopeRulesForAccount` 的过滤逻辑。
2. **方案定稿**：在计划文档中明确并集语义、mermaid 因果图、不修的影响及上线验证 SQL 思路。
3. **实现与测试**：修改 `dynamicScopeService.js`；配套更新 `server/tests/dynamicScopeService.helpers.test.js`（并集期望、仅名单户、归一化等用例）；本地/CI 跑 vitest（见下节）。
4. **部署与数据验证**：部署后触发或等待动态刷新，用 MySQL 查询 `rule_matched_objects` 与 `structure_ads` 核对 `act_832` 等账户。
