## 二、遇到的问题与解决方法

| 问题 | 原因/现象 | 解决方法 |
|------|------------|----------|
| **1. 默认仍走 updated_time 增量** | 文档已禁止依赖 FB `/ads` 的 updated_time filtering（现网不可靠），但实现里仍优先尝试增量分支。 | 将增量分支改为「实验开关」默认关闭（`ENABLE_INCREMENTAL_EXPERIMENT = false`）；手动同步默认仅走全量分页；实验开启时仍可尝试增量，失败/回退走全量。 |
| **2. 同轮两次 resolve 违背口径** | Today 热同步已对 activeAdIds 调 resolveObjectsByIds（statusMap），若 Piggyback 再对同一批调一次则违背「结构写库的 resolve 同轮只执行一次」。 | Today 同步改为一次 resolve 并带结构字段（adset_id, campaign_id, updated_time），产出 `structurePayload` 并返回；Piggyback 优先用 structurePayload 写 structure_ads，仅对「缺口」（DB 缺失/空名且 payload 无）补查一次。 |
| **3. 伪增量只跑 activeAdIds>0 的账户** | 后置阶段用 `results.filter(r => r?.activeAdIds?.length > 0)`，spend=0 的账户不跑伪增量，漏掉「没花钱也要管状态」。 | 改为对所有账户跑伪增量：`results.filter(r => r?.accountId)`，传 `r.activeAdIds \|\| []`；`runPseudoIncrementForAccount` 内用 recentActiveIds 决定是否调 FB，空 activeAdIds 时 diffIds = recentActiveIds。 |
| **4. 2.4 冷却被 Piggyback 误触发** | 冷却基于 `MAX(structure_ads.last_synced_at)`，Piggyback/伪增量每轮 upsert 会更新该字段，用户刚点手动同步易被判冷却。 | 冷却改为基于 `structure_sync_status.last_success_at`（仅手动/每小时全量成功时更新），Piggyback/伪增量不再影响 2.4 冷却判定。 |
| **5. MySQL INTERVAL ? / LIMIT ? 报错** | 部分 MySQL/驱动对 `INTERVAL ? HOUR`、`LIMIT ?` 占位符不支持，导致伪增量 SQL 报错、任务不跑。 | 改为字面量：`DATE_SUB(UTC_TIMESTAMP(), INTERVAL 24 HOUR)`、`LIMIT 200`；后续用常量拼接（`\${RECENT_ACTIVE_HOURS}` 等）实现「改常量即生效」，占位符仅保留 accountId。 |
| **6. 同一批每 15 分钟重复 resolve** | recentActiveIds 按 `last_synced_at DESC LIMIT 200` 取「最新」的 200 条，upsert 后这些行 last_synced_at 更新为 NOW()，下轮仍被选中，导致 spend=0 账户也每轮打 200 次 resolve。 | 增加「1h 降频 + 最旧优先旋转」：WHERE 增加「至少 1h 未同步」条件（NULL 放行），ORDER BY last_synced_at ASC，只从满足条件的候选中取 200 条；同一批被 touched 后 1h 内不再进入候选。 |
| **7. NULL 行永远进不了候选** | recentActiveIds 的 WHERE 使用 `last_synced_at >= ... 24 HOUR`，NULL 被过滤掉，无法自愈历史/异常数据。 | WHERE 改为「NULL 放行」：(last_synced_at IS NULL OR last_synced_at >= 24h 内) 与 (last_synced_at IS NULL OR last_synced_at < 1h 前)；排序 last_synced_at ASC，NULL 优先被选中并补齐。 |
| **8. 常量未驱动 SQL** | RECENT_ACTIVE_HOURS / PSEUDO_INCREMENT_MIN_INTERVAL_HOURS / MAX_DIFF_IDS 未参与 SQL，改常量不生效。 | SQL 使用模板字符串注入常量（\${RECENT_ACTIVE_HOURS} 等），仅数值来自同文件常量，无用户输入，安全；占位符仍只用于 accountId。 |

## 三、本窗口完成内容（实现与文档）

### 实现

- **structureSyncService.js**：手动同步默认全量；Piggyback（`piggybackStructureFromToday`）；伪增量（`runPseudoIncrementForAccount`，24h 窗口 + 1h 降频 + NULL 放行 + 最旧优先）；每小时全量轮转（`runHourlyStructureFullRotation`）；2.4 冷却基于 `last_success_at`；验证日志（recentActiveIds/diffIds/touched/diffHead5）。
- **ingestorService.js**：`syncAccountTodayStats` 返回 `activeAdIds`、`structurePayload`，对 activeAdIds 仅一次 resolve（结构字段）；统一心跳内 Today 后调 Piggyback（同轮复用 facebookApi）；后置阶段对所有账户跑伪增量；账户任务返回 `activeAdIds`。
- **server/index.js**：`resolveObjectsByIds` 支持返回 adset_id、campaign_id、updated_time。
- **cronService.js**：注册每小时第 5 分钟执行 `runHourlyStructureFullRotation`。
- **rateLimitService.js**：`getLastUsageRate()`，供结构轮转「usage 高本小时跳过」判断。

### 文档

- **docs/顺序2_结构同步_验证与测试清单.md**：8 条用例（目的、步骤、预期日志、SQL、通过/失败判定、失败时 3 个可能原因与定位）；最小可执行命令（Windows）；检查点 SQL 速查；验收打勾表；**六、验证通过后：下一步与未完成项**。
- **docs/顺序2_伪增量与轮转_验证指南.md**（若存在）：分步验证说明。
