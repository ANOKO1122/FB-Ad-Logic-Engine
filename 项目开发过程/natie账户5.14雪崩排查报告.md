## 4. 为什么5/14 06:16集中执行

### 4.1 数据同步时间线（北京时间）

```
00:01  滑动窗口同步 → natie Today=0条     Facebook还没返回数据
00:16  滑动窗口同步 → natie Today=0条
00:31  滑动窗口同步 → natie Today=9条     开始有数据，但只有9条老广告
00:46  滑动窗口同步 → natie Today=10条    稳定在10条
01:01~06:00  每15分钟一次 → 始终10条       连续5小时无变化
06:15  今日数据同步 → natie Today=56条     ← 突增46条！
06:16  cron评估    → 命中规则2条 → 37个广告仲裁
06:16:30           → 34 PAUSE_AD + 2 SET_BUDGET + 1 skipped
```

### 4.2 精确切换点（EDT时区日志原文）

```
18:15:04  cron评估 → 命中规则0条 → 冷却跳过
18:15:37  【数据同步】拉取spend>0=56条，写入ad_snapshots=56  ← 分水岭
18:16:12  cron评估 → 命中规则2条（首次命中）
18:16:30  M4完成：仲裁37个广告
```

### 4.3 数据库交叉验证

查询 `ad_snapshots` 表中14条雪崩广告的最早记录：

```
synced_at = 2026-05-14 06:15:15（北京时间）
全部14条广告同一时刻首次写入
```

**06:15之前，这批广告在本地数据库中不存在。**

### 4.4 根因

1. 这批测新广告5/13创建，Facebook侧有冷启动延迟（审核+投放排队），到5/14清晨才陆续产生花费数据
2. 系统每15分钟同步一次Facebook数据，06:15的同步窗口首次拉到这46条广告的spend>0数据
3. 06:16的cron评估立即捕获，全部满足暂停条件后批量执行

**链路中没有系统延迟或故障。** 06:16就是这批广告数据到达本地后的第一个评估周期。
