## 二、遇到的问题与解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| **预算被错改成 50 美元（实际应为 30 美元）** | Facebook API 的 `daily_budget` 在 GET 时返回的已是「分」，我们误当美元再乘 100，导致 3000 分被当成 300000 分（3000 美元），后续 +10% 与上限 50 取小得到 50 美元下发。 | **server/index.js**：`getAdsetBudget` 不再乘 100，直接 `Math.round(Number(raw))` 返回；注释与文档标明 FB 读写均为「分」。单测与 M4_3.5 文档同步更新。 |
| **规则执行后无法再次运行** | 账户级锁未释放：上次执行在「规则限定目标广告」之后某步卡住（如等待 FB 接口），未返回导致锁一直占用；再次点击执行时 GET_LOCK 失败返回 409。 | 说明 5 分钟超时保险丝会强制释放锁；增加关键日志（规则数据查询、评估完成、匹配数、Pre-Flight、开始执行动作、获取广告组预算）便于下次定位卡点。 |
| **规则匹配 0 个广告（条件未满足）** | ① 库里 `purchase_value=0` 导致 roas=0，规则条件 roas>5 不满足；② 根因为 Route 1（getAdInsights）未请求 action_values 等（在另一窗口已修复）；③ 即使数据修好，规则阈值为 roas>5，实际 roas≈3.82 仍不满足。 | 另一窗口：Route 1 字段补齐。本窗口：给出 SQL 查 roas_computed、说明规则条件与数据口径；建议用户将条件改为如 roas>3 以匹配当前广告。 |
| **ROAS 为 NULL 时规则如何取值** | 规则引擎对 `adData.roas` 为 null 时按 0 处理；DB 未存 roas 时需用 purchase_value/spend 计算。 | 说明数据流（ruleDataService → calculateSingleDayMetrics）；给出「库内 roas 与按公式计算 ROAS」的 SQL；方案 A 在读侧统一「计算优先→API兜底→零兜底」（另一窗口）。 |
| **purchase_value 为 0 但 API 有金额** | 当日同步时 Route 1 未请求 action_values，或 FB 当日归因未返回；与用户用 Graph API Explorer 查「当天」数据对比后确认 API 有 79.24。 | 另一窗口：getAdInsights 补齐 action_values、purchase_roas、website_purchase_roas 等；本窗口：分析并给出「当天 API vs 库」对比表与验证步骤。 |
| **2.3.1 阶段 3 验收 2、阶段 4 未闭环** | 阶段 3 需验收「回填 OR 不丢失」；阶段 4 需确认「模板填充 whenLines+whenTimeWindow、无 v2 JSON 模式」。 | 用户执行验收 2（新建 OR 规则→保存→编辑确认第二行为 OR）；确认三模板均正确填充时间窗口与条件；TASKS 中阶段 3、阶段 4 勾选并备注。 |
