## 三、遇到的问题与解决方法

### 3.1 现象：本地与云端同时打开时，「本地像云上」、甚至数据库也像云上

**误解**：曾将本机 **Node 监听端口从 3001 改为 3101** 视为防串云主手段。

**根因澄清**：

- **浏览器请求打到哪**：主要由前端 **axios `baseURL`**（及是否误配 **`VITE_API_URL` 为线上绝对地址**）决定；与 Node 监听 3001 还是 3101 **无直接因果关系**。
- **数据来自哪张库**：由后端 **`.env` 中 `DB_HOST` / 库名等**决定；改 HTTP 端口**不会**自动切换数据库。
- **本机改 3101 的实际作用**：与 **`.env` 中 `PORT=3101`**、**Vite `proxy.target` 指向同一端口**对齐，避免代理指错进程；属于**本机工程习惯**，不是「防串云」的万能开关。

**解决思路**：前端默认 **`/api` + `.env.local` 不提交**；数据库开发期坚持连 **本机 MySQL**（除非刻意连云调试）。

---

### 3.2 现象：`git pull origin main` 报错 untracked files would be overwritten

**原因**：本机曾存在 **未跟踪** 的  
`.cursor/plans/终版云本地协同方案_6100b6d5.plan.md`、  
`项目开发过程/云端项目总结-2026-03-30-钉钉登录方案修正与执行基线更新.md`，  
而 **`origin/main` 已纳入同路径文件**，合并会覆盖本机副本，Git 中止以防丢内容。

**解决**：在 pull 前**删除或移走**上述未跟踪文件（用户选择「以仓库为准」则直接删除），再执行 `git pull origin main`。

**附带失败**：曾尝试 `Move-Item` 到 `D:\backup\...`，因目标目录不存在报 **「未能找到路径中的某个部分」**；需先 `New-Item -ItemType Directory` 建目录，或改用**删除后 pull**。

---

### 3.3 云端 `.env` 未配置 `VITE_API_URL` 是否有影响

**结论**：一般**无负面影响**。构建时若 `VITE_API_URL` 未注入，运行时代码走 **`|| '/api'`**，与同域 **`/api` → Nginx → Node** 一致。需避免的是在**构建环境**中误设 `VITE_API_URL=https://线上域名`。

---

### 3.4 PR 描述中需注意的表述

**`ENABLE_CRON`**：设计为**未配置时默认开启**（保障云端定时任务不因忘配而静默关闭）；**本地开发**在私有 `.env` 中设 **`ENABLE_CRON=false`**。PR 正文**不应**写成「建议在云上关闭 cron」类易误导运维的句子。

## 四、验收与自检建议

- **Git**：`main` 上 `git status` 仅余**有意保留的本机修改**（如 `vite.config.js`）或无改动。
- **本地运行**：`npm run dev` + 后端启动后，浏览器 Network 中 API 请求应为 **`http://localhost:<前端端口>/api/...`**，而非线上域名。
- **忽略生效**：`git status` **不出现** `.env.local`、`server/scripts/local/` 下文件。
