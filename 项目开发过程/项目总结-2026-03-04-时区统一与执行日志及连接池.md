## 一、本窗口遇到的问题与解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| 执行日志前端时间少 8 小时 | ① DB 写入时 Drizzle 使用独立 pool，未保证会话 UTC，导致存库时刻错误；② 读接口用 connection.js 的 pool（已 SET UTC），读出与写入基准不一致；③ mysql2 未设 `timezone` 时，Date 编解码可能按本地时区，API 返回的 ISO 与预期不符。 | ① Drizzle 改为复用 connection.js 的 pool（Single Source of Truth）；② connection.js 的 createPool 增加 `timezone: '+00:00'`；③ 执行日志 API 用 toUTCISO() 统一返回带 Z 的 UTC ISO；前端 parseUTC 对无 Z 字符串补 Z 后按 UTC 转北京。 |
| 冷却 debug 日志 lastAt 与本地时间差 8 小时 | last_executed_at 存 UTC，日志用 toISOString() 打出 UTC，与 Node 日志前缀（本地北京）混在一起难以对照。 | ruleExecutionStateService.isCooldownDue 的 debug 日志同时输出 lastAtUtc（ISO 带 Z）和 lastAtBj（北京时间 yyyy-MM-dd HH:mm:ss）。 |
| SQL 查询「北京时间」列快 8 小时 | 用户 MySQL 客户端会话为北京时区，SELECT triggered_at 已是北京时刻；再用 CONVERT_TZ(triggered_at,'+00:00','+08:00') 会误把该值当 UTC 再加 8。 | 文档明确：北京会话下直接把 triggered_at 当北京时间看，勿再做 CONVERT_TZ；仅当会话为 UTC 时才用 CONVERT_TZ 转北京。 |
| 间歇性 campaign_id 为 NULL 导致 CBO 预算失败 | ad_snapshots/daily_stats 的 ON DUPLICATE KEY UPDATE 允许新写入的 NULL 覆盖已有有效 campaign_id。 | ingestorService 中 ad_snapshots 与 daily_stats 的 UPDATE 子句对 ad_set_id、campaign_id 使用 COALESCE(VALUES(列), 列)，避免 NULL 覆盖。 |
| 冷却期内仍每分钟执行预算动作 | isCooldownDue 用 JS Date 与 DB 时间差算 diffMin，存在时区/基准不一致，导致 diffMin 偏大、始终 due=true；且调度层只写 budget_* 冷却键，过滤用 ad: 键导致查不到。 | ① isCooldownDue 改为在 MySQL 内用 TIMESTAMPDIFF(MINUTE, last_executed_at, UTC_TIMESTAMP()) 算 diffMin；② cronService 双写：除 cooldownKey 外再写一条 ad:${adId}，使广告级过滤生效。 |

### 7.1 目标与背景

- 在「预算单位统一为分」「按值增减」「set_budget 固定值」基础上，进一步收紧预算动作语义并补上 UI 防呆，避免：
  - 规则误配置导致「减预算带上限」「加预算带下限」「set_budget 同时有上下限」等脏数据落库；
  - 预算在多轮执行中超出预期上限/下限，或者在已经达成目标时仍然重复发 API。
- 目标是让「预算动作护栏 + 冷却 + Pre-Flight + 前端配置」形成一条完整、可测试的链路。

### 7.2 本次代码改动（概要）

- **校验层（templateValidator）**：
  - 为预算动作新增 `min_daily_budget` 支持，并写死组合口径：increase 仅允许 `max_daily_budget`，decrease 仅允许 `min_daily_budget`，set_budget 禁止上下限；同一动作同时配置 max/min 直接 400 拦截。
  - 上下限须为整数分且 `>=100`，否则视为无效配置；避免「小于 1 美元」或非整数分的护栏落库。
- **预算计算层（computeNewBudgetCentsOnce）**：
  - increase：若当前预算已高于有效上限（含 MIN_BUDGET_CENTS 归一），本轮保持不动；否则按步长增量后以上限封顶。
  - decrease：若当前预算在下限或以下，本轮保持不动；否则按步长减量后以 `min_daily_budget` 托底（同时不低于 `MIN_BUDGET_CENTS`）。
  - set_budget：只上调不下调——当前预算大于等于目标时直接返回 current，小于目标时设置为目标值（再套全局最小预算与可选上限）。
- **执行层 Pre-Flight**：在 ABO/CBO 分支中，对「当前=目标」的预算操作：
  - 区分三类跳过原因：`above_max_cap`（加预算超上限）、`below_min_floor`（减预算低于下限）、`budget_already_at_target`（单纯等于目标值）。
  - 审计日志与 API 响应中带上上述 `reason`，便于后续排查与教学演示。
- **单元测试**：
  - 扩展 `budgetUnitCents.test.js` / `templateValidator.test.js` 覆盖超上限不动、下限托底/不动、set_budget 仅上调、上下限口径错误等场景。
  - 新增 `budgetPreFlightReason.test.js` 验证三类跳过原因在执行层中的行为与日志输出。

#### 7.2.1 set_budget 固定值只上调不下调 bug 复盘与修复

- **现象**：在一条「设置预算为固定值」规则中，将目标设为 7 美元、实际广告组当前预算为 10 美元时，预期应命中「只上调不下调」而直接 skipped，但运行日志显示预算被下调到了 7 美元。
- **根因定位**：
  - 在 cron 预计算阶段，`resolveNewBudgetCentsForAction` 的 set_budget 分支为了避免额外 GET，直接以 `current=0` 调用 `computeNewBudgetCentsOnce(0, action)`，得到 `_resolvedBudgetCents = target`。
  - 执行层 ABO 分支原先逻辑是：若 `_resolvedBudgetCents` 存在且为整数，则直接当作 `newBudgetCents` 使用，而不再基于实时 `currentCents` 重新计算。
  - 这样一来，「当前 10 美元、目标 7 美元」场景中，执行层看到的是 `currentCents=1000`、`newBudgetCents=700`，`current!==new`，既没有触发 set_budget 的「只上调不下调」语义，也没有命中 `current===new` 的 Pre-Flight skipped 分支，最终真的对 FB 下发了 7 美元的修改。
- **修复策略（方案 A）**：
  - 保持调度层「预计算一次、少一次 GET」的总体思路不变，仅对 set_budget 在执行层 ABO 路径做精细化处理。
  - 在 `actionExecutorService.executeActionsForAd` 的 ABO 分支中，遇到 `action.type === 'set_budget'` 时**一律忽略 `_resolvedBudgetCents`**，改为基于实时 `currentCents` 调用 `computeNewBudgetCentsOnce(currentCents, action)`。
  - 这样在目标小于当前的场景下，`computeNewBudgetCentsOnce` 会按「只上调不下调」返回 current，随后 Pre-Flight 看到 `currentCents === newBudgetCents`，记为 `status=skipped, skip_reason=budget_already_at_target`，不再对 FB 发送修改请求，同时仍然通过冷却表占用本轮冷却窗口。

### 7.3 前端 RuleManager/AdminTemplates 精简（不改模板内容）

- **RuleManager.vue**：
  - 动作编辑区按类型显示对应字段：increase 仅有「预算上限（美元，可选）」；decrease 仅有「预算下限（美元，可选）」；set_budget 不显示上下限。
  - 新增 `onMinBudgetInput` 将下限从美元转换为分；`onActionTypeChange` 在动作切换时自动清理无关字段（防止旧值污染 payload）。
  - `saveRule` 时对上限/下限做前端兜底校验，并在 `actionsPayload` 中仅对 increase 透传 `max_daily_budget`、对 decrease 透传 `min_daily_budget`，set_budget 不再透传上下限字段。
- **AdminTemplates.vue**：
  - 与 RuleManager 保持同一 UI 与字段口径，只对预算护栏字段做「按动作类型归一 + 无效值清理」，**不修改任何模板文案与业务逻辑内容**。
  - 模板列表和详情本身（名称、描述、条件组合、动作语义）保持不变，只有在编辑并保存时，才会对不符合新口径的上下限字段做兼容性归一化。
