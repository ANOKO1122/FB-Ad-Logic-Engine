# 代码实现与原设计差异分析

> 对照时间：2026-05-14
> 对照范围：统一心跳（`unifiedHeartbeatSync`）内的数据同步、历史回填、冷数据归档机制

---

## 差异一：历史回填与热数据同步耦合

### 原设计

历史回填是独立任务，在统一心跳内按特定时间点触发。与每 15 分钟的热数据同步是两套独立逻辑，互不干扰。

### 实际代码

历史回填和热数据同步混在 `syncAccountSlidingWindow` 同一个函数里，通过 `day/night hour` 分区切换：

```js
// processHeartbeatAccount, 第 3323-3341 行
if (hour >= 6 && hour < 18) {
  daysBack = 0                           // 白天：只同步 today
} else {
  daysBack = 7 或 14                      // 夜间：today + 7/14 天回填，同一次调用
}
syncAccountSlidingWindow(accountId, ..., daysBack)  // 一个函数同时做 hot + backfill
```

`syncAccountSlidingWindow` 内部先拉 today（写 `ad_snapshots`），再拉历史按日数据（写 `daily_stats`），串行执行，不是两个独立任务。

**影响**：回填不是按 "特定时间点独立触发" 设计的，而是和夜间热数据同步绑死。白天每小时整点还有 3 天回补，也是同一个路径。

## 差异二：冷数据归档双轨并行，未按优先级调度

### 原设计

冷数据归档是特定时间点任务（如 12:00），在统一心跳内拥有最高优先级。

### 实际代码

存在 **两套归档机制**，并行运行，互不协调：

| 机制 | 触发方式 | 时间窗口 | 写入目标 |
|---|---|---|---|
| `archiveAllAccountsDailyStats` | 仅 `manualArchive()`（手动 API）、测试 | 06:00-06:09 窗口 | `daily_stats` |
| `checkAndExecuteArchive` → `executeArchive` → `archiveDailyStats` | 每 15 分钟心跳内，每个账户都跑 | `hour≥2` → 初归 (PENDING→ARCHIVED)，`hour≥12` → 深对 (ARCHIVED→FINALIZED) | `daily_stats` + `daily_archive_status` 状态表 |

**问题**：
1. **机制 1 从未被 cron 自动调用** — `archiveAllAccountsDailyStats` 只在手动触发的 API 和测试里调用，生产心跳没有调用它
2. **机制 2 每 15 分钟都在检查** — 不是 "某个特定时刻触发"，而是每个心跳周期都查 `hour≥2` 和 `hour≥12`
3. **两套机制写同一张表** (`daily_stats`)，没有互斥协调
4. **没有优先级系统** — 归档、回填、热数据全部在同一轮心跳里串行，不分先后

**影响**：归档没有独立调度，没有优先级保护，和热数据同步争夺同一轮心跳的执行时间。

---

## 差异三：热数据同步不是滑动窗口

### 原设计

每 15 分钟热数据同步使用滑动窗口（now - 2 小时），按广告的更新时间字段增量拉取。

### 实际代码

两条热数据路径都使用 `date_preset: 'today'` — **Facebook 的全天累计查询**，不是滑动窗口增量：

`syncAccountTodayStats`（白天路径）：

```
GET /{accountId}/insights?date_preset=today&filtering=[spend>0]
```

`syncAccountSlidingWindow` 内的 `fetchInsightsInBatches`（夜间路径）：

```
POST /v24.0/ (Batch)
  → {adId}/insights?date_preset=today
```

**没有 `time_range: { since: now-2h, until: now }`，没有 `updated_time` 过滤，没有增量拉取。** 每轮拉的是广告当日 0 点到现在的全部累计数据，然后 `ON DUPLICATE KEY UPDATE` 覆盖上一条快照。

代码里有 `time_range` 的 JSON 构造能力（`server/index.js` 第 1342 行），但没有被热数据同步路径使用。

**影响**：
- API 每次返回全量累计值，数据量大时存在浪费
- 与用户原设的 "2h 滑动窗口增量" 完全不符
- 如果不拉全量而用增量，Facebook API 配额消耗会大幅降低

---

## 差异四：Track2 增量同步仅覆盖结构，不覆盖指标

代码里唯一使用 `updated_time >= sinceSec` 增量的只有 Track2 快速结构同步（`:07` `:27` `:52` 触发），同步的是广告结构元数据（名称、状态、层级），不是 spend/actions 等指标数据。

这与用户原设的 "按广告更新字段收集指标数据" 不是同一个东西。

---

## 差异总览

| 设计意图 | 当前实现 | 差异性质 |
|---|---|---|
| 历史回填 = 独立时间点任务 | 混在 `syncAccountSlidingWindow`，和热数据同步同函数 | 耦合 |
| 冷归档 = 特定时间 + 最高优先级 | 双轨并行，每 15 分钟检查，无优先级 | 架构偏离 |
| 热数据同步 = now-2h 滑动窗口增量 | `date_preset=today` 全天全量 | 策略不同 |
| 广告按 updated_time 增量收集 | Track2 有此能力，但不用于指标数据 | 范围不符 |
