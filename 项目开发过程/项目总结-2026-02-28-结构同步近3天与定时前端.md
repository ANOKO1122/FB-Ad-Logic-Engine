## 一、本窗口遇到的问题与解决方法

### 1. getStructurePage 未映射 created_time 导致库内全为空

- **现象**：结构镜像三张表虽有 created_time 列（迁移 029），但同步后查库该字段全为空。
- **原因**：`server/index.js` 的 `getStructurePage` 在把 FB 返回的 `data.data` 映射为 items 时，没有包含 `created_time`，只传了 id、name、status、updated_time 等。
- **解决**：在 items 的 map 中增加 `created_time: x.created_time || null`，使同步写入时能落库。

### 2. 结构同步仍全量拉取，易限流且无法利用「近 3 天」语义

- **现象**：syncAccountStructureAds 对 campaigns/adsets/ads 仍为全量分页、filtering 为 null 或 baseFiltering，请求多、易触发限流。
- **原因**：历史实现以全量为主；方案文档已明确改为「只拉 updated_time >= since（3 天前）」。
- **解决**：按执行方案在 structureSyncService 中：计算 sinceSec（3 天前 Unix 秒）；campaigns/adsets/ads 三个 edge 均用 `filtering: [{ field: 'updated_time', operator: 'GREATER_THAN', value: sinceValue }]` 分页拉完再 upsert；ads 的 doStructureAdsUpsertAndStatus 传 isFullRun: false；移除实验性增量路径，统一走近 3 天路径。filtering value 优先 Unix 秒（String(sinceSec)），单 edge 报错时自动回退为 ISO8601。

### 3. 轮转命名与语义不一致

- **现象**：runHourlyStructureFullRotation 仍称「全量轮转」，实际已改为「近 3 天」拉取，易误解。
- **解决**：JSDoc 与日志统一为「结构轮转（近 3 天）」/「结构轮转-近3天」；冷却、usage 等提示也改为「结构同步（近3天）」口径。

### 4. 过滤窗口无可观测性

- **现象**：无法事后核对「本次同步用的 since 是多少」与同步结果是否一致。
- **解决**：structure_sync_status 增加 last_filter_since_sec（迁移 030）；doStructureAdsUpsertAndStatus 增加参数 filterSinceSec，同步成功时写入该列。

### 5. 多选账户时「刷新列表」会怎样

- **现象**：用户以为多选几个账户后点「刷新列表」会对每个选中账户都触发一次结构同步。
- **原因**：前端 refreshScopeItemsWithSync 仅在 `selectedAccountIds.length === 1` 时调用 syncStructure，多选时只执行 refreshScopeItems（查库刷新），不调后端同步。
- **解决**：产品上改为「多选禁止使用刷新列表、仅单选可用」：按钮 `:disabled="selectedAccountIds.length !== 1 || scopeLoading"`，并加 title 提示「仅在选择一个广告账户时可刷新列表（触发该账户结构同步）」。

### 6. 结构同步频率与每次账户数

- **需求**：定时由「每小时一次、每次 6 账户」改为「每 20 分钟 :05/:35/:55、每次 5 账户」。
- **解决**：cronService 中 cron 从 `'5 * * * *'` 改为 `'5,35,55 * * * *'`；runHourlyStructureFullRotation(api, { maxAccounts: 5 })；启动日志与注释同步更新。

### 7. TLS 连接错误日志（200 OK 后）

- **现象**：SOCKS5 + TLS 场景下，响应已 200 且 body 收完后出现「❌ TLS连接错误」，随后「JSON解析成功（由TLS错误回退路径完成）」。
- **结论**：业务成功；属连接关闭触发的 TLS 层噪音，已由回退路径正确处理，未影响同步结果。可后续将此类情况降为 debug 或单独区分，避免误读为失败。

## 二、项目进度与目标（本窗口相关）

### 项目目标（与方案文档一致）

- **结构同步任务改造**：少拉数据、少限流——campaigns/adsets/ads 只拉「近 3 天有更新」的数据；created_time 与 updated_time 一并落库；可观测 last_filter_since_sec；filtering value 优先 Unix 秒、单 edge 报错自动回退 ISO。
