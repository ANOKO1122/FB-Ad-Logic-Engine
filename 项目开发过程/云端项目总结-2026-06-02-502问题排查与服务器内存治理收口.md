## 三、根因分析

### 3.1 问题链路

```
用户浏览器 → fb-ad-nginx (端口 80) → proxy_pass http://172.17.0.1:3001 → Node.js
```

1. **前端静态页面**由 Nginx 直接 serve，能正常加载
2. 点击钉钉登录时，前端调用后端 API（如 `/api/auth/options`）
3. Nginx 尝试反代到 `172.17.0.1:3001`，但 **Node.js 后端已不可用**
4. Nginx 返回 **502 Bad Gateway**

### 3.2 为什么 Node.js 不可用

**直接原因：** 系统 OOM（Out Of Memory）

```
MySQL 长时间运行 → buffer pool 持续膨胀 → 占用 10.9GB RSS
         ↓
15GB 物理内存被吃光 → Linux OOM Killer 触发
         ↓
OOM Killer 选择杀掉 MySQL（最大内存占用者）
         ↓
MySQL 被 SIGKILL → 容器退出
         ↓
Node.js 连接 MySQL 失败（Connection Refused）
         ↓
连锁反应：API 请求无法处理 → 502
```

**间接原因：**
1. MySQL 容器**没有内存限制**，buffer pool 可以无限增长
2. Swap 仅 1GB，不足以提供缓冲
3. Node.js 也**没有堆内存限制**，异常情况下可能参与内存竞争

### 3.3 为什么重启后恢复

`sudo systemctl restart` 重新拉起 Node.js 进程，同时 MySQL 已被 Docker 自动重启，数据库恢复正常，服务链路恢复。但如果不加限制，MySQL 会再次膨胀，OOM 会重复发生。

### 3.4 为什么自动化规则昨晚正常运行

自动化规则是 Node 内部的 **cron 定时任务**，不走 Nginx。如果 Node 进程活着且数据库正常，cron 任务就能正常工作。OOM 发生在白天内存压力累积到临界点时，而非昨晚。

## 四、解决方案

### 4.1 MySQL 容器内存限制（Docker --memory）

**操作：**

```bash
docker update --memory=6g --memory-swap=6g mysql
```

**验证：**

```bash
docker inspect mysql --format '{{.HostConfig.Memory}}'
# 输出: 6442450944 (6GB)

docker stats mysql --no-stream
# MEM USAGE / LIMIT: 3.3GiB / 6GiB
```

**原理：**
- `--memory=6g`：容器最多使用 6GB 物理内存
- `--memory-swap=6g`：禁用 swap 扩展（防止内存+swap 超过 6GB）
- 超过限制时，kernel 发送 SIGKILL 杀死容器内进程，**不是卡死**
- Docker 有 `--restart` 策略，被杀后自动重新拉起
- 整个中断窗口约 **5~30 秒**

**补充说明：** 最初设了 `--memory=4g`，但当时 MySQL 已使用 3.4GB（占 85%），余量不足。调至 6GB 后余量 76%，安全。

### 4.2 Swap 扩容（swap file）

**操作：**

```bash
fallocate -l 8G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

**验证：**

```bash
free -h
# Swap: 9.0Gi (原 1Gi + 新增 8Gi)
```

**原理：**
- Swap 是磁盘空间模拟的内存，速度慢于物理内存（约 10000 倍）
- 作用是**应急缓冲**：内存不足时，把不常用的内存页换出到磁盘
- **不是加速，是减速保命**——慢比挂掉好
- 原 Swap 仅 1GB（已用 72%），扩容到 9GB 提供充分缓冲

**警告：** Swap 使用过多会导致系统性能断崖式下跌。如果发现 Swap 使用持续超过 50%，说明物理内存仍然不足，需要进一步治理。

### 4.3 Node.js 堆内存限制（--max-old-space-size）

**操作：**

```bash
# 编辑 service 文件
sudo sed -i 's|ExecStart=/usr/bin/node server/server.js|ExecStart=/usr/bin/node --max-old-space-size=1536 server/server.js|' /etc/systemd/system/fb-ad-logic-engine.service

# 重载并重启
sudo systemctl daemon-reload
sudo systemctl restart fb-ad-logic-engine.service
```

**验证：**

```bash
sudo systemctl status fb-ad-logic-engine.service --no-pager -l
# └─1304992 /usr/bin/node --max-old-space-size=1536 server/server.js
```

**原理：**
- `--max-old-space-size=1536`：V8 引擎的 JavaScript 堆最大 1.5GB
- 当前 Node 实际使用约 **262MB**，远低于限制，不影响正常运行
- 超限时：V8 先触发 GC → 仍超限则抛出 `RangeError` → `uncaughtException` 处理器捕获 → 优雅退出 → systemd 自动重启
- **相当于安全气囊**：平时不干扰，异常时保命

## 六、经验教训

### 6.1 对 Docker 容器的理解

- `docker update --memory=X` 是**热更新**，即时生效，不需要重启容器
- 超过内存限制时**不是卡死**，是 **OOM Kill 后 Docker 自动重启**
- 限制内存不是伤害服务，而是**隔离爆炸半径**——让 MySQL 只炸自己，不牵连其他服务

### 6.2 对 Swap 的理解

- Swap 是**硬盘模拟内存**，速度慢 10000 倍
- Swap 的作用是**延缓 OOM，不是避免 OOM**
- 治本还是管好内存使用

### 6.3 对 Node 内存管理的理解

- `--max-old-space-size` 限制 V8 堆，超限时优雅退出而不是拖垮系统
- 需要配合 systemd 的 `Restart=on-failure` 实现自愈
- 当前 Node 使用 100~260MB，远低于 1.5GB 上限

### 6.4 系统架构认知

- 一台 15GB 服务器上跑了 **3 套独立业务系统**（FB 广告 + 融合平台 + 工单系统）
- 每套系统有独立的数据库和 Nginx 入口
- 最大的内存消费者是 MySQL（Docker 容器），需要优先管控
