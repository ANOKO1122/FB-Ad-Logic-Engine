# 项目总结-2026-03-07-动态筛选多账户闭环与项目收尾

## 一、本窗口目标

- 完成动态筛选执行方案的端到端落地与验收，覆盖：
  - 动态范围快照（`rule_matched_objects`）；
  - 执行路径与快照一致性；
  - 多账户规则下 TriggerB/TriggerC/执行语义闭环；
  - 策略 B（`ERROR_OVERSIZE` 保留旧快照）；
  - 文档与测试补齐。

## 二、核心问题与根因

1. **策略 B 偏差**  
   - 问题：oversize 时快照被清空，与“保留上一版快照”不一致。  
   - 根因：写库采用“按账户先删后建”，导致 `ERROR_*` 规则也丢快照。

2. **单条执行口径不一致**  
   - 问题：`POST /api/rules/:id/execute` 仍会执行排除名单中的对象。  
   - 根因：单条执行走旧 `evaluateRule` 路径，没有统一走 Dispatcher 快照目标解析。

3. **多账户动态筛选语义不闭环**  
   - 问题：手动重算/保存后刷新仅覆盖主 `account_id`。  
   - 根因：TriggerB/TriggerC 触发逻辑按单账户实现，未按规则涉及账户展开。

4. **目标解析存在跨账户串值风险**  
   - 问题：`targetIds` 使用 `act_xxx:id` 时，动态计算可能误用其它账户 ID。  
   - 根因：手工目标展开未做当前账户 scoped 解析。

## 三、已完成改造

### 1) 动态快照与策略 B

- DynamicScope 刷新改为：仅 `NORMAL` 规则按 `rule_id + account_id` 原子替换快照。
- `ERROR_OVERSIZE/ERROR_FILTER_INVALID/ERROR_REFRESH_FAILED` 仅更新规则状态，不清理旧快照。

### 2) 执行路径统一

- 单条执行 `executeSingleRule` 统一改为 Dispatcher 评估链路（与调度一致）：
  - `loadDataForAccount` + `evaluateRuleWithCache`。
- `use_dynamic_scope=1` 时按 `rule_matched_objects(rule_id, account_id)` 读取目标，执行与快照一致。

### 3) 多账户动态筛选闭环

- 新增按规则维度刷新：
  - `refreshDynamicTargetsForRule(ruleId)`；
  - 规则涉及账户解析优先级：`target_by_account` 非空账户 > `target_account_ids` > `account_id`。
- TriggerB：
  - 规则保存后按规则涉及账户逐个防抖刷新。
- TriggerC：
  - 传 `ruleId` 自动刷新该规则全部目标账户；
  - 传 `accountId` 保持单账户兼容行为。
- 前端“重算动态范围”按钮改为按 `ruleId` 触发并展示多账户汇总结果。

### 4) 账户隔离与排除名单

- `targetIds=act_xxx:id` 在动态计算中按当前账户隔离解析，避免跨账户串值。
- 规则页新增排除名单配置与回显，执行语义严格按：
  - `finalAdIds = (dynamicMatched ∪ manualTarget) - excludeIds`。

### 5) 数据一致性与观测性

- 删除规则时事务内联动清理 `rule_matched_objects`，避免孤儿快照。
- 动态刷新日志增强为结构化统计：
  - `trigger/processed/normal/inserted/oversize/invalid/refresh_failed/duration_ms`。

## 六、经验沉淀（可复用）

1. 先定义不变式：`final = (动态 ∪ 手动) - 排除`，再改触发器与执行器。  
2. 先补测试再扩行为：先锁口径，避免“修一处坏一处”。  
3. 多账户系统必须账户隔离：刷新、快照、执行、日志都按账户维度对齐。  
4. 错误不等于清空：策略 B 证明“保留上一版可用结果”比“失败即清空”更稳健。
