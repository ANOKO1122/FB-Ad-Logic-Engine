#### 1.2 核心功能实现

**智能路由逻辑**：
- `selectDataSource(timeWindow, timezoneName)`：根据时间窗口选择数据源
  - `today` → `ad_snapshots`（热数据）
  - `yesterday` → `daily_stats`（冷数据，06:00 归档），06:00 前降级到 `ad_snapshots`
  - `last_3_days` / `last_7_days` / `lifetime` → `daily_stats`（多天聚合）

**06:00 边界降级策略**：
- `isBeforeArchiveTime(timezoneName)`：判断当前时间是否在 06:00 前（基于账户时区）
- 如果 `daily_stats` 未落盘，降级到 `ad_snapshots` 的"昨日最后快照"
- 误差约 0.7%-1%（可接受）

**动态聚合计算**：
- `aggregateMultiDayMetrics(dailyStatsArray)`：多天聚合计算（避免辛普森悖论）
  - 累加原始计数（分子和分母）
  - 动态重算单价/比率类指标（含除零保护）
  - 公式：ROAS = 总购买金额 / 总花费（不是每天 ROAS 的平均值）

**数据查询函数**：
- `queryAdSnapshots()`：从 `ad_snapshots` 表查询数据（热数据）
  - 支持 today 查询（最新快照）
  - 支持 yesterday 降级查询（最后快照）
  - 使用 ROW_NUMBER() 窗口函数或最大时间戳连接法（兼容非 MySQL 8.0）
- `queryDailyStats()`：从 `daily_stats` 表查询数据（冷数据）
  - 支持 yesterday 查询（单天）
  - 支持 last_3_days / last_7_days 查询（多天）

**时区处理**：
- `getAccountTimezone(accountId)`：从 `account_mappings` 表获取账户时区
- 所有时间计算都基于账户时区（不是服务器时区）

### 2. 阶段5：增强 RuleEngine

#### 2.1 核心功能实现

**从数据库查询数据（完全离线）**：
- 修改 `evaluateRule()` 方法：支持传入 `accountId`，从数据库查询数据
- 使用 `ruleDataService.queryRuleData()` 查询数据（不调用 Facebook API）
- 实现了向后兼容：如果传入 `insights` 数组，使用旧版本逻辑

**AND/OR 逻辑运算符**：
- `evaluateConditions(conditions, adData, logicOperator)`：评估所有条件
  - AND 逻辑：使用 `Array.every()`（所有条件都必须满足）
  - OR 逻辑：使用 `Array.some()`（至少一个条件满足）
- 规则配置中增加 `logic_operator` 字段（'AND' | 'OR'）

**目标筛选（target_level 和 target_ids）**：
- 支持 `target_level`：'ad' | 'adset' | 'campaign'
- 支持 `target_ids`：JSON 数组，存放目标 ID 列表
- 如果指定了 `target_ids`，只查询指定的广告
- 如果没有指定，查询账户下所有广告（从数据库查询）

**时间窗口支持**：
- `getTimeWindowFromConditions(conditions)`：从条件数组中提取时间窗口
- 支持 `time_window` 参数：'today' | 'yesterday' | 'last_3_days' | 'last_7_days' | 'lifetime'
- 查询时使用时间窗口参数，从数据库查询对应时间范围的数据

**增强指标获取**：
- `getMetricValue(metric, adData)`：支持新字段
  - 新字段：roas, cpa, purchases, purchase_value, ucpc, add_to_cart_cost, checkout_cost, payment_cost
  - 兼容旧字段：ctr, cpc, spend, conversions, clicks, impressions, cpm

**预算调整动作**：
- `executeActions(actions, matchedAd, isSimulation)`：使用 `ad_set_id` 执行预算调整
- `adjustBudget(adsetId, action)`：使用 `ad_set_id` 调整预算（不需要额外查询 Facebook API）
- 支持 Dry Run 模式（`is_simulation` 参数）

#### 2.2 向后兼容
- `evaluateRuleLegacy(rule, insights, ads)`：旧版本逻辑（使用 Facebook API 数据）
- `executeActionsLegacy(actions, ad, insight)`：旧版本动作执行
- 自动判断调用方式：如果第一个参数是字符串（accountId），使用新版本；如果是数组（insights），使用旧版本

## 🐛 遇到的问题与解决方法

### 问题1：时区不一致导致 today 查询返回空

**现象**：
- 查询 `today` 时返回空数组
- 查询 `yesterday` 时能查到数据
- 前端显示的是今天的数据

**原因分析**：
1. 数据同步时使用：`Asia/Shanghai` 时区（`ad_snapshots.timezone_name = 'Asia/Shanghai'`）
2. 账户配置时区：`UTC`（`account_mappings.timezone_name = 'UTC'`）
3. 查询时使用账户时区（UTC）计算"今天"，导致时间窗口不匹配

**具体数据**：
- 数据时间：`2026-01-19 16:55:52 GMT+0800`（中国时区）
- 转换为 UTC：`2026-01-19 08:55:52 UTC`
- Today 时间窗口（UTC）：`2026-01-20 00:00:00` 到 `2026-01-20 23:59:59`
- 结果：数据是昨天的（UTC），不在今天的窗口内

**诊断工具**：
- 创建 `test-check-data-timezone.js` 诊断脚本
- 显示数据库中实际的数据时间、时区计算、时间窗口匹配情况

**解决方案（待实现）**：
- 方案A：统一账户时区配置（将账户时区改为 `Asia/Shanghai`）
- 方案B：修改查询逻辑（使用数据的时区，而不是账户时区）
- 方案C：混合方案（数据时区优先，账户时区兜底）【推荐】

**状态**：已定位问题，待下一个窗口解决

---

### 问题2：测试脚本 SQL 查询错误（DISTINCT + ORDER BY）

**现象**：
- 测试脚本报错：`Expression #1 of ORDER BY clause is not in SELECT list`

**原因**：
- 使用 `SELECT DISTINCT ad_id, ad_name ORDER BY synced_at DESC`
- MySQL 不允许在 `DISTINCT` 查询中 `ORDER BY` 不在 `SELECT` 列表中的字段

**解决方法**：
- 改为使用子查询 + GROUP BY（兼容所有 MySQL 版本）
- 或使用 ROW_NUMBER() 窗口函数（MySQL 8.0+）

**状态**：✅ 已修复

---

### 问题3：测试脚本没有显示实际数据值

## 💡 经验总结

### 1. 时区处理的重要性
- 数据同步时区和账户时区必须一致，否则时间窗口查询会失败
- 建议：数据同步时记录时区，查询时使用数据的时区
