## 2. 验收项与留痕证据

### 2.1 验收项：新图床统一上传入口可达（主域名）

**验证命令（不带文件，验证“路由存在且到达网关”）**

```bash
curl -i -X POST "http://ad_tools.thinkpro.top/api/upload-image"
```

**预期现象**

- HTTP 状态码：400
- 响应体：JSON，包含“缺少 file 字段”的提示（说明请求到达融合平台 `api-gateway`，不是被入口层吃掉，也不是打到 FB 后端 404）

**实际现象（留痕）**

- `HTTP/1.1 400 Bad Request`
- `{"success":false,"error":"没有上传文件，请使用 form-data 并传入字段 file"}`

---

**验证命令（带文件，验证“真实上传闭环”）**

```bash
curl -i -X POST "http://ad_tools.thinkpro.top/api/upload-image" \
  -F "file=@/root/work/tuchuang-backend/test-assets/demo.svg;type=image/svg+xml"
```

**预期现象**

- HTTP 状态码：200
- JSON：`success:true`，并包含 `url`（OSS 外链）、`key` 等字段

**实际现象（留痕）**

- `HTTP/1.1 200 OK`
- 返回 JSON（示例）：
  - `url=https://my-project-hangzhou-imageup.oss-cn-hangzhou.aliyuncs.com/uploads/2026/04/02/1775103147000-demo.svg`
  - `key=uploads/2026/04/02/1775103147000-demo.svg`

### 2.2 验收项：旧接口已下线（避免串线）

**验证命令**

```bash
curl -i -X POST "http://ad_tools.thinkpro.top/api/upload-to-imgfi"
```

**预期现象**

- HTTP 状态码：404
- 内容包含：`Cannot POST /api/upload-to-imgfi`

**实际现象（留痕）**

- `HTTP/1.1 404 Not Found`
- `Cannot POST /api/upload-to-imgfi`

## 3. 常见失败形态（定位口诀）

- **404（Cannot POST /api/upload-image）**：大概率请求没被入口层分流到融合平台（入口层 Nginx 规则未生效/未重启/路径未覆盖）。
- **502 Bad Gateway**：大概率是入口层或 `ronghe-nginx` 上游指向失效（容器重建导致 upstream IP 变化，Nginx 仍缓存旧地址）。
- **200 但页面不显示图片**：大概率是 `/temp/` 或 `/uploads/` 未正确代理到融合平台，或浏览器缓存旧前端 bundle。

## 4. 回滚原则（只描述，不在此文执行）

- 回滚入口层分流：回退 `deploy/nginx-fb-ad-docker.conf` 对融合平台路径的改动，并重启 `fb-ad-nginx`。
- 回滚后必须复跑本文件的 2.1 与 2.2 验收命令，确认行为与预期一致。
