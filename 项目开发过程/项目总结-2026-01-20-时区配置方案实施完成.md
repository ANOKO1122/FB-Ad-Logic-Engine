### 1. 阶段A：快速止血（时区匹配 + 降级策略）

#### 1.1 修复时区参数问题

**问题**：
- `queryDailyStats` 在多天窗口查询时使用 `timezoneName`（账户时区）
- 但 `daily_stats` 表中的数据可能使用不同的时区（如 `Asia/Shanghai`）
- 导致 `AND timezone_name = ?` 过滤条件不匹配，返回空数组

**解决方案**：
- 修改 `queryRuleData`：多天窗口也使用 `data_timezone_used`（而不是 `timezoneName`）
- 修改 `queryDailyStats` 调用：传入 `dataTimezoneUsed` 而不是 `timezoneName`

**代码位置**：
- `server/services/ruleDataService.js` 第 192-202 行：数据时区选择逻辑
- `server/services/ruleDataService.js` 第 235-237 行：查询函数调用

#### 1.2 实现降级策略

**问题**：
- 当 `daily_stats` 查询结果为空时，多天窗口（`last_7_days`、`last_30_days`）返回空数组
- 用户无法看到今天的数据

**解决方案**：
- 在 `queryRuleData` 中实现降级逻辑：
  - 如果 `daily_stats` 查询结果为空，降级到 `ad_snapshots` 查询今天的数据
  - 添加 `HISTORY_EMPTY` warning，记录降级信息
  - 确保不返回空数组

**代码位置**：
- `server/services/ruleDataService.js` 第 247-260 行：降级策略实现

#### 1.3 返回值结构优化

**修改**：
- `queryRuleData` 现在返回 `{ data, warnings }` 而不是直接返回数组
- 更新所有调用方（`server/index.js` 中的两处）以适配新结构

**代码位置**：

### 2. 阶段B：合并查询（历史+今天）

#### 2.1 实现 `queryMultiDayWithToday` 函数

**功能**：
- 历史段（start → 昨天）：从 `daily_stats` 查询
- 今天段：从 `ad_snapshots` 查询（今天最后快照）
- 合并后调用 `aggregateMultiDayMetrics` 聚合

**核心逻辑**：
1. 先用 `data_timezone_used` 查询历史数据
2. 检查时区不匹配，如果检测到 `TIMEZONE_MISMATCH` 且第一次查询为空，二次尝试用数据实际时区查询
3. 查询今天数据（从 `ad_snapshots`）
4. 合并数据并聚合

**代码位置**：
- `server/services/ruleDataService.js` 第 509-615 行：`queryMultiDayWithToday` 函数

#### 2.2 完善 warnings 策略

**实现的 warnings**：
- `HISTORY_EMPTY`：历史数据为空，仅返回今天的数据
- `TIMEZONE_MISMATCH`：历史数据时区与查询时区不匹配，已使用数据时区查询
- `HISTORY_QUERY_FAILED`：查询历史数据失败
- `TODAY_QUERY_FAILED`：查询今天数据失败

**判断逻辑优化**：
- `HISTORY_EMPTY` 只有在确实没有数据时才记录（区分时区不匹配导致的假空）
- `TIMEZONE_MISMATCH` 只用于提示口径差异，不阻断数据返回

**代码位置**：
- `server/services/ruleDataService.js` 第 546-579 行：warnings 判断逻辑

#### 2.3 优化时区不匹配时的查询逻辑

**问题**：
- 当检测到 `TIMEZONE_MISMATCH` 时，如果使用查询时区（如 `UTC`）查询历史数据，会因为时区过滤导致空结果
- 但 `daily_stats` 中确实有数据（使用 `Asia/Shanghai` 时区）

**解决方案**：
- 先用 `data_timezone_used` 查询历史数据
- 如果检测到 `TIMEZONE_MISMATCH` 且第一次查询为空，二次尝试用数据实际时区（`timezones[0]`）查询
- 确保能查询到历史数据，而不是因为时区过滤导致假空

**代码位置**：
- `server/services/ruleDataService.js` 第 546-579 行：时区不匹配处理逻辑

## 🐛 遇到的问题和解决方法

### 问题 1：数据库会话时区问题

**问题描述**：
- 用户从 MySQL 命令行查询时，显示 `session_tz: SYSTEM`
- 但后端日志显示"✅ 数据库会话时区已设置为 UTC"

**原因分析**：
- MySQL 命令行是独立会话，不受 Node.js 应用影响
- Node.js 应用的连接池已正确设置会话时区为 UTC

**解决方法**：
- 创建 `test-verify-timezone.js` 验证脚本
- 确认 Node.js 应用的数据库连接确实设置为 UTC
- 说明 MySQL 命令行查询的是独立会话，不影响应用

**验证结果**：
- ✅ `test-verify-timezone.js` 显示 `session_tz: +00:00`
- ✅ 应用数据库连接正确设置为 UTC

### 问题 2：`getDataTimezone` SQL 错误

**问题描述**：
- `getDataTimezone` 函数使用 `SELECT DISTINCT timezone_name ORDER BY synced_at DESC` 导致 SQL 错误
- 错误信息：`Expression #1 of ORDER BY clause is not in SELECT list, references column 'fb_ad_brain.ad_snapshots.synced_at' which is not in SELECT list; this is incompatible with DISTINCT`

**原因分析**：
- MySQL 不允许在 `DISTINCT` 查询中使用不在 `SELECT` 列表中的 `ORDER BY` 列

**解决方法**：
- 重构 SQL 查询：使用子查询找到 `MAX(synced_at)`，然后连接获取对应的 `timezone_name`
- 避免 `DISTINCT` 和 `ORDER BY` 冲突

**代码位置**：
- `server/services/ruleDataService.js` 第 120-180 行：`getDataTimezone` 函数

**验证结果**：
- ✅ SQL 错误已解决
- ✅ 函数能正确获取数据时区

### 问题 4：`last_7_days` 和 `last_30_days` 天数计算错误

**问题描述**：
- `test-timewindow-extended.js` 显示：
  - `last_7_days` 返回 8 天（期望 7 天）
  - `last_30_days` 返回 31 天（期望 30 天）

**原因分析**：
- `calculateTimeWindow` 函数中，`now.minus({ days: X })` 计算错误
- 例如：`last_7_days` 使用 `minus({ days: 7 })`，实际包含了 8 天（今天 + 过去 7 天）

**解决方法**：
- 调整 `days` 参数：`minus({ days: X - 1 })`
  - `last_7_days`：`minus({ days: 6 })`（今天 + 过去 6 天 = 7 天）
  - `last_30_days`：`minus({ days: 29 })`（今天 + 过去 29 天 = 30 天）

**代码位置**：
- `server/utils/timeWindow.js` 第 55-77 行：时间窗口计算

**验证结果**：
- ✅ 所有时间窗口天数计算正确
- ✅ 跨时区测试通过
