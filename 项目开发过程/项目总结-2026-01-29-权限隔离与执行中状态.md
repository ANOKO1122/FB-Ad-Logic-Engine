## 二、本对话遇到的问题与解决方法

### 2.1 pytest 权限隔离测试一度大量失败

**现象**：  
- IDOR 类用例期望 403，实际得到 200（或 structure 得到 500）。  
- 单条执行：不存在的规则 404 时 `r.json()` 报 JSONDecodeError；执行自己规则得到 404。

**原因**：  
1. **认证优先顺序**：authJwt 原先先读 cookie、再读 Authorization。浏览器曾用 admin 登录后，请求带 Bearer staff_token 时，若同一域名下存在 cookie，服务端仍用 cookie 里的 admin token，导致以 admin 身份放行，返回 200。  
2. **404 响应体**：部分 404 返回 body 为空或非 JSON，测试里直接 `r.json()` 抛错。  
3. **规则主键字段**：GET /api/rules 返回的规则对象主键可能是 id/ruleId/rule_id，测试只取 `rules[0]["id"]` 可能取错导致请求到错误 ID 返回 404。

**解决**：  
- **authJwt.js**：改为优先读 `Authorization: Bearer`，再回退到 `req.cookies?.token`，保证 API/测试用 Bearer 时按正确用户校验。  
- **test_permission_isolation_api.py**：  
  - 404 用例先断言 `status_code == 404`，再在「有 body 且可解析」时解析并断言 error/code，避免对空 body 调 `r.json()`。  
  - 单条执行用例用 `first.get("id") or first.get("ruleId") or first.get("rule_id")` 取规则 ID，缺省时 skip。

**验证**：修改后 10 个用例全部通过。

---

### 2.2 getCronStatus 没有 isRunning 导致 409 不生效

**现象**：  
- 路由里用 `status.isRunning` 判断“是否正在执行”并返回 409，但 getCronStatus() 未返回 isRunning，导致永远为 undefined（false），409 几乎不会触发。  
- 用户可在规则执行中重复点「立即运行所有规则」/「运行此规则」，仅靠账户级锁兜底，产生大量“拿不到锁就跳过”的噪音。

**解决**：  
- 在 cronService 中引入**全局计数** `runningCount`（不再用布尔 isRunning 变量）：  
  - executeAllRules：入口 `runningCount++`，结尾 `try/catch/finally` 中 `finally { runningCount = Math.max(0, runningCount - 1) }`。  
  - executeSingleRule：入口 `runningCount++`，结尾 `finally { runningCount = Math.max(0, runningCount - 1) }`。  
- getCronStatus() 返回 `isRunning: runningCount > 0`。  
- executeAllRules 入口增加保护：若 `runningCount > 0` 直接 return（并打日志），由路由层返回 409。

**效果**：  
- 执行中再次触发会明确得到 409 + ALREADY_RUNNING。  
- 系统健康/cron 状态可正确展示“正在执行”。  
- 避免 executeAllRules 与 executeSingleRule 并发时，先结束的一方错误地把“执行中”清掉（用计数加减替代布尔置位，避免竞态）。

---

### 2.3 执行中状态：布尔 isRunning 的复位与竞态

**现象**：  
- 若 executeAllRules 只置 `isRunning = true` 而缺少 `finally { isRunning = false }`，会永久卡死（一直 409、一直显示执行中）。  
- 若 executeAllRules 与 executeSingleRule 共用同一布尔，当全量跑着时单条先结束并把 isRunning 置 false，会导致“实际仍在跑全量但状态被清掉”的竞态。

**解决**：  
- 采用**计数器** runningCount：  
  - 开始执行：`runningCount++`  
  - 结束执行：`runningCount = Math.max(0, runningCount - 1)`（防御多减）  
- 对外仍通过 getCronStatus() 返回布尔 `isRunning: runningCount > 0`。  
- 两处执行路径（executeAllRules、executeSingleRule）均严格在 finally 中减 1，确保任一路径结束都会正确复位，且多任务并发时不会互相清错状态。

---

### 2.4 语法错误 / 残留 finally 块（用户反馈）

**现象**：  
- 用户反馈文件中存在“一层 catch + finally (runningCount--) 之后又多了一段 } finally { isRunning = false } }”，且 isRunning 已改为 runningCount，会导致语法错误或 ReferenceError，后端起不来。

**处理**：  
- 在当前仓库版本中核对：executeAllRules 末尾仅有一层 `} catch (error) { ... } finally { runningCount = Math.max(0, runningCount - 1) }`，且全文无 `isRunning = false` 赋值。  
- 若本地仍有多余块，需删除多余 `} finally { isRunning = false }`，保证只保留一层 catch + 一层 finally（runningCount--）。
