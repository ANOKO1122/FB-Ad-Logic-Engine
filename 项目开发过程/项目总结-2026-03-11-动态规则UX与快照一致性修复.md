# 项目总结-2026-03-11-动态规则UX与快照一致性修复

## 一、本窗口目标

- 排查并修复「当前匹配 N 个对象」与「规则配置共 M 个」不一致的问题（N ≠ M）。
- 排查规则 603「仅 3 个账户配置却显示 95/113 个对象」的根因并修复。
- 确认方案 `.cursor/plans/动态规则_ux_与列表性能优化_c2f7ceda.plan.md` 的完成情况，并补充本窗口发现的修复项。

## 二、遇到的问题与根因

### 2.1 编辑弹窗「当前匹配」与「规则配置」数字不一致

**现象**：重算动态范围后（如 inserted=91），再点「编辑规则」，界面显示「当前匹配 91 个对象」但「规则配置共 6 个」。

**根因**：

- 「当前匹配」来自接口 `matched_count`，即 `rule_matched_objects` 的 `COUNT(*)`（实时/重算后的快照）。
- 「规则配置」来自表单 `ruleForm.targetIds.length`。打开编辑时表单用**列表接口返回的规则**初始化；拉取 GET /api/rules/:id 时只更新了 `matchedCount` 和 `invalidIds`，**没有用详情里的 target_ids 更新 targetIds**，因此配置数一直是列表时的旧值（如 6）。

### 2.2 2 秒防抖「自动应用」覆盖详情带回的 targetIds

**现象**：打开编辑后约 2 秒，「规则配置」从 91 又变回 6。

**根因**：监听「监控范围条件 / 已选账户 / 目标层级」的 watch 在 2 秒后调用 `applyScopeConditions()`，用预览或当前列表结果重写 `ruleForm.targetIds`，把详情接口刚带回来的数据覆盖掉。开启动态筛选时，真相来源应是规则详情/快照，而非自动应用结果。

### 2.3 规则目标账户缩小后 matched_count 虚高（规则 603）

**现象**：规则 603 的 `target_account_ids` 仅 3 个账户，但 `rule_matched_objects` 中该规则有 14 个账户、95 或 113 条；列表与详情显示「当前生效 113 个对象」，执行也按 113 个对象跑。

**根因**：用户曾将规则从多账户改为 3 个账户并保存。重算动态范围时，`refreshDynamicTargetsForRule` 只对**当前目标账户**（3 个）执行「删旧 + 插新」，**没有删除「已不在 target_account_ids」的账户**在 `rule_matched_objects` 里的旧行，导致 11～12 个账户的 106 条残留，matched_count 虚高。

## 三、已完成的解决与改动

### 3.1 前端：编辑时用详情同步「规则配置」并刷新展示

**文件**：`src/views/RuleManager.vue`

**改动**：

1. **抽取 `resolveSelectedTargetIds(rawIds)`**  
   将「targetIds → 可读标签 → resolvedSelectedMap」的逻辑抽成函数，供打开编辑与详情回调复用；`rawIds` 为空时清空 `resolvedSelectedMap`。

2. **打开编辑时**  
   先 `resolvedSelectedMap.value = new Map()`，再调用 `resolveSelectedTargetIds(ruleForm.targetIds)`（不再内联大段解析逻辑）。

3. **动态规则拉取 GET /api/rules/:id 后**  
   在已有 `matchedCount`、`invalidIds` 的基础上：  
   - 用 `data.rule.targetIds ?? data.rule.target_ids` 得到 `nextTargetIds`（兼容数组或 JSON 字符串）；  
   - 执行 `ruleForm.value.targetIds = nextTargetIds`；  
   - 调用 `resolveSelectedTargetIds(nextTargetIds)` 刷新已选对象标签。

**效果**：再次打开编辑时，「规则配置共 N 个」与「当前匹配 N 个」一致，标签展示正确。

### 3.2 后端：重算时清理「已不在目标账户」的快照行

**文件**：`server/services/dynamicScopeService.js`

**改动**：在 `refreshDynamicTargetsForRule` 中，在按 `accountIds` 循环刷新各账户快照**之后**、更新规则状态与回写 `target_ids` **之前**，增加：

- 若 `accountIds.length > 0`，执行  
  `DELETE FROM rule_matched_objects WHERE rule_id = ? AND account_id NOT IN (当前 accountIds 列表)`  
  使用参数化占位，避免 SQL 注入。

**效果**：目标账户从多改少后再重算，快照中只保留当前目标账户，matched_count 与配置一致，不再出现 113 vs 7 等差异。

### 3.3 临时数据修正（用户已执行）

对规则 603 执行 SQL，删除不在当前 3 个目标账户内的快照行（106 行），使该规则快照从 113 条变为 7 条，与界面一致。  
SQL 示例（账户 ID 需加单引号）：

```sql
DELETE FROM rule_matched_objects
WHERE rule_id = 603
  AND account_id NOT IN (
    'act_927139705822379',
    'act_3762284617405378',
    'act_1255287319167068'
  );
```

## 六、验收与后续建议

- **验收**：规则 603 目标账户改为 3 个并保存 → 重算动态范围 → 再打开编辑，应显示「当前匹配 7 个对象（规则配置共 7 个）」；用户确认「符合预期，没有问题了」。
- **可选**：若仍出现「打开编辑后约 2 秒配置数被改回较小值」，可在 watch 中增加「当 `ruleForm.useDynamicScope === true` 时不启动 2 秒防抖自动应用」，避免自动应用覆盖详情带回的 targetIds。
- **文档**：TASKS.md、DEV_PLAN.md 已更新本窗口总结与「动态规则 UX 与快照一致性」小节；本文件归档于 `项目开发过程/`。

## 七、本窗口小结

本窗口解决了三类问题：① 编辑弹窗「规则配置」与「当前匹配」不一致（详情未回写 targetIds）；② 2 秒防抖自动应用可能覆盖详情数据（根因已说明，可选在开启动态筛选时关闭自动应用）；③ 目标账户缩小后快照残留导致 matched_count 虚高（重算时按规则删除已不在目标账户的快照行）。方案「动态规则_ux_与列表性能优化」的优化 A/B 此前已落地，本窗口补充了编辑态与快照一致性的修复，并完成 TASKS/DEV_PLAN 与项目开发过程文档的更新。
