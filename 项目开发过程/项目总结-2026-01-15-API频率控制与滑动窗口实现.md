## 🐛 遇到的问题与解决方法

### 问题 1：测试用例失败 - `calculateSleepTime` 逻辑错误

**问题描述**：
- 测试期望 `estimated_time_to_regain_access: 1.5` 返回 1500ms，但实际返回 2000ms
- 测试期望 `estimated_time_to_regain_access: 8` 返回 10000ms，但实际返回 2000ms

**原因分析**：
- 当有 `estimated_time_to_regain_access` 时，代码先计算了休眠时间，但随后被使用率分档逻辑覆盖
- 使用率被固定推断为 85%，导致走 50-85% 的逻辑（2s）

**解决方法**：
- 修复逻辑：当有 `estimated_time_to_regain_access` 时，直接使用它作为休眠时间并提前返回
- 根据恢复时间推断使用率（用于告警），但不再根据使用率分档覆盖休眠时间

---

### 问题 2：其他测试失败 - 动态导入导致的问题

**问题描述**：
- `health.test.js` 和 `rules.test.js` 失败
- 错误：`TypeError: Cannot read properties of undefined (reading 'get')`

**原因分析**：
- 在 `makeRequest` 中使用了 `await import()` 动态导入
- 导致模块加载时序问题

**解决方法**：
- 将 `getCircuitBreakerStatus` 改为静态导入
- 在文件顶部统一导入所有需要的函数

---

### 问题 3：单位换算陷阱（来自 Gemini 3 AI 的建议）

**问题描述**：
- 代码假设 `estimated_time_to_regain_access` 的单位是秒
- 但 Facebook API 实际返回的单位是**分钟**

**潜在风险**：
- 如果 API 返回 1.5（代表 1.5 分钟 = 90秒），而代码把它当成秒处理，计算出 1500ms（1.5秒）
- 后果：系统休眠 1.5 秒后立即重试 → 再次触发限流 → 死循环

**解决方法**：
- 修复单位换算：默认使用分钟单位，`estimatedTime * 60 * 1000`（分钟 → 毫秒）
- 添加安全缓冲：在等待时间基础上增加 2000ms（2秒）缓冲
- 支持配置：通过 `options.estimatedTimeUnit` 可以配置单位（默认 'minutes'）

**关键代码**：
```javascript
if (estimatedTimeUnit === 'minutes') {
  exactWaitMs = Math.ceil(estimatedTime * 60 * 1000) // 分钟 -> 毫秒
} else {
  exactWaitMs = Math.ceil(estimatedTime * 1000) // 秒 -> 毫秒
}
const finalWaitMs = exactWaitMs + safetyBufferMs // 添加安全缓冲
```

### 问题 4：数据库字段名不匹配

**问题描述**：
- 测试失败：`Unknown column 'account_id' in 'field list'`
- 测试失败：`Unknown column 'timezone_name' in 'field list'`

**原因分析**：
- `account_mappings` 表的字段名是 `fb_account_id`，不是 `account_id`
- `account_mappings` 表中没有 `timezone_name` 字段
- 状态字段是 `is_active`，不是 `status`

**解决方法**：
- 修复 SQL 查询：使用 `fb_account_id as account_id`（使用别名保持代码兼容）
- 移除 `timezone_name` 字段查询，使用默认值 `'UTC'`
- 修复状态判断：使用 `is_active = 1` 而不是 `status = 'active'`

**修复位置**：
- `syncAllAccountsTodayStats()` 函数
- `archiveAllAccountsDailyStats()` 函数

## 💡 经验总结

### 1. 单位换算陷阱的重要性
- Facebook API 的 `estimated_time_to_regain_access` 单位是分钟，不是秒
- 如果误将分钟当作秒处理，会导致休眠时间严重不足，触发死循环限流
- **解决方案**：明确单位，添加安全缓冲，支持配置

### 2. 数据库字段名的一致性
- 实际表结构可能与设计文档不一致
- 需要根据实际表结构调整代码
- **解决方案**：先查询表结构，再编写代码

### 3. 测试驱动开发的重要性
- 通过测试发现了多个问题（逻辑错误、字段名不匹配等）
- 测试用例帮助验证修复是否正确
- **解决方案**：为关键功能编写测试用例

### 4. 安全缓冲的必要性
- 网络延迟、服务器时间偏差可能导致"卡点"唤醒
- 添加安全缓冲可以避免刚好撞在限流枪口上
- **解决方案**：在等待时间基础上增加 1-2 秒缓冲
