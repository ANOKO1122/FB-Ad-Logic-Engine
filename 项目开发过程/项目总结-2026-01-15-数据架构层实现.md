### 1. 数据库表结构设计与实现

#### 1.1 创建了三个核心数据表

**`ad_snapshots` 表（热数据：Today）**
- 存储今日广告快照，每 10 分钟更新一次
- 包含 10 个核心字段：状态、花费、成效(购买)、CPA、uCPC、加购费、结账费、支付费
- 特殊处理：CPC 和 ROAS **不入库**（当天数据从 API 获取，历史数据通过计算）

**`daily_stats` 表（冷数据：历史聚合）**
- 存储每日汇总数据，每天 06:00 将昨日数据落盘
- 用于历史数据查询和报表

**`automation_logs` 表（审计日志）**
- 记录每次规则执行的详细信息
- 用于审计和问题排查

#### 1.2 字段精简优化
- 删除了不需要的字段（`ctr`、`cpm`、`unique_clicks`、`add_to_cart`、`initiate_checkout`、`add_payment_info`）
- 只保留你需要的 10 个核心字段 + 必要的系统字段

### 2. Data Ingestor 服务实现

#### 2.1 核心函数实现

**`syncAccountTodayStats(accountId, ownerId, timezoneName)`**
- 同步单个账户的今日广告数据
- 流程：获取广告列表 → 获取广告状态 → 批量拉取数据 → 批量写入数据库

**`syncAllAccountsTodayStats()`**
- 同步所有账户的今日广告数据
- 从 `account_mappings` 表获取所有活跃账户，逐个同步

**`fetchInsightsInBatches(accountId, adIds, facebookApi)`**
- 使用 Facebook Batch API 批量拉取数据（20 个一组）
- 请求字段：`ad_id, ad_name, spend, cpc, roas, actions, action_values, cost_per_action_type, cost_per_unique_link_click, cost_per_unique_inline_link_click`
- 处理频率限制（批次之间休眠 500ms）

**`saveSnapshotsToDb(insights, accountId, ownerId, syncSessionId, syncedAt, timezoneName)`**
- 批量写入数据库
- 使用 `ON DUPLICATE KEY UPDATE` 更新已存在的记录
- 提取所有成本字段（CPA、uCPC、加购费、结账费、支付费）

#### 2.2 辅助函数实现

**`parseActions(actions)`**
- 解析 Facebook API 返回的 `actions` 字段
- 提取购买次数（精确匹配 `offsite_conversion.fb_pixel_purchase`）

**`pickCostPerActionType(costPerActionType, actionTypes)`**
- 从 `cost_per_action_type` 数组中提取指定 action_type 的成本
- 做好防御性编程（处理 Facebook API 的"不值不显"原则）

**`extractUcpc(insight)`**
- 提取 uCPC（独立单次链接点击费用）
- 优先使用 `cost_per_unique_link_click`，如果没有则使用 `cost_per_unique_inline_link_click`

**`generateSyncSessionId()`**
- 生成唯一的同步会话ID（格式：`sync_20260109_143025_abc123`）

**`chunkArray(array, size)`**
- 将数组按指定大小切分（用于 Batch API 的 20 个一组切分）

### 3. 数据字段提取逻辑

#### 3.1 10 个核心字段的数据来源

| 字段 | 数据来源 | 提取方式 |
|------|---------|---------|
| 状态 | `/ads` API | `effective_status` 或 `status` |
| 花费 | `insights` API | `spend` 字段 |
| 成效(购买) | `insights` API | 从 `actions` 中提取 `offsite_conversion.fb_pixel_purchase` |
| CPA | `insights` API | 从 `cost_per_action_type` 中提取 `purchase` 的成本 |
| uCPC | `insights` API | `cost_per_unique_link_click` 或 `cost_per_unique_inline_link_click` |
| 加购费 | `insights` API | 从 `cost_per_action_type` 中提取 `add_to_cart` 的成本 |
| 结账费 | `insights` API | 从 `cost_per_action_type` 中提取 `initiate_checkout` 的成本 |
| 支付费 | `insights` API | 从 `cost_per_action_type` 中提取 `add_payment_info` 的成本 |
| CPC | `insights` API | **不入库**，当天数据从 API 获取 |
| ROAS | `insights` API | **不入库**，当天数据从 API 获取或从 `action_values` 计算 |

#### 3.2 CPC 和 ROAS 的特殊处理逻辑

**CPC（单次点击成本）**
- **当天数据**：直接从 Facebook API 的 `cpc` 字段获取（不存储）
- **历史数据**：通过计算得出 `CPC = 花费 / 点击量`
- **原因**：看过去一周的数据时，需要计算那一周的总花费除以总点击量，才能得出正确的 CPC

**ROAS（广告支出回报率）**
- **当天数据**：直接从 Facebook API 的 `roas` 字段获取，或从 `action_values` 计算 `ROAS = 购买总金额 / 花费`（不存储）
- **历史数据**：通过计算得出 `ROAS = 购买总金额 / 花费`
- **原因**：看过去一周的数据时，需要计算那一周的总购买金额除以总花费，才能得出正确的 ROAS

### 4. 防御性编程实现

#### 4.1 Facebook API 的"不值不显"原则
- **问题**：如果某个指标的值是 0，API 会直接把这个字段藏起来（不返回），而不是返回 0
- **解决方案**：所有字段提取都做好空值判断，如果字段不存在，返回 `null`（而不是 0）

#### 4.2 代码示例

```javascript
// ❌ 错误写法：如果今天没单，这行代码会报错导致页面白屏
const cpa = data.insights[0].cost_per_action_type.find(x => x.action_type === 'purchase').value;

// ✅ 正确写法：先找，找不到就给默认值 null
const cpaObj = data.insights[0].cost_per_action_type?.find(x => x.action_type === 'purchase');
const cpa = cpaObj ? parseFloat(cpaObj.value) : null;
```

## 🐛 遇到的问题与解决方法

### 问题 1：`ls` 命令在 Windows 上不可用

**问题描述**：
- 用户在 Windows 上执行 `ls` 命令报错：`'ls' 不是内部或外部命令`

**原因分析**：
- `ls` 是 Unix/Linux/Mac 的命令，Windows CMD 默认不支持
- Windows 使用 `dir` 来列出文件

**解决方法**：
- 使用 Windows 原生命令：`dir`（CMD）或 `ls`（PowerShell）
- 或者安装 Git Bash 使用 Unix 命令

**学习要点**：
- 理解跨平台命令差异
- 掌握 Windows 常用命令（`dir`、`type`、`cd` 等）

---

### 问题 2：测试脚本中 `parseActions` 返回值不匹配

**问题描述**：
- 测试脚本期望 `parseActions` 返回 `{ purchases: 5, addToCart: 10 }`
- 但实际实现只返回购买次数（`number`）

**原因分析**：
- 我们只需要购买次数，不需要加购次数（加购费直接从 `cost_per_action_type` 获取）

**解决方法**：
- 更新测试脚本，修改断言逻辑
- 删除 `addToCart` 相关的测试代码

---

### 问题 3：CPC 和 ROAS 的存储策略

**问题描述**：
- 最初设计将 CPC 和 ROAS 存储到数据库
- 但用户指出：看过去一周的数据时，需要计算那一周的总花费除以总点击量，才能得出正确的 CPC

**原因分析**：
- CPC 和 ROAS 是比值，不是累加值
- 如果存储每天的 CPC/ROAS，无法准确反映一周的 CPC/ROAS

**解决方法**：
- **CPC 和 ROAS 不入库**
- 当天数据：直接从 API 获取（不存储）
- 历史数据：通过计算得出（查询时实时计算）

---

### 问题 4：Facebook API 的"不值不显"原则

**问题描述**：
- 如果某个指标的值是 0，API 会直接把这个字段藏起来（不返回）
- 直接访问可能不存在的字段会导致代码报错

**解决方法**：
- 所有字段提取都做好防御性编程
- 使用可选链（`?.`）和默认值
- 如果字段不存在，返回 `null`（而不是 0）

**代码示例**：
```javascript
// ✅ 正确写法
const cpa = pickCostPerActionType(costPerActionType, ['purchase'])
// 如果不存在，返回 null（而不是 0）

// ❌ 错误写法
const cpa = costPerActionType.find(x => x.action_type === 'purchase').value
// 如果字段不存在，会报错
```

## 🔑 关键技术点

### 1. Facebook Batch API 使用
- 每次最多 20 个子请求
- 使用 `batch` 字段（JSON 字符串）发送请求
- 响应是数组，每个元素对应一个子请求的响应

### 2. 防御性编程
- 处理 Facebook API 的"不值不显"原则
- 所有字段提取都做好空值判断
- 使用可选链（`?.`）和默认值

### 3. 数据精度保护
- 所有 ID 字段使用 `VARCHAR(50)`，避免使用 `BIGINT`
- 使用 `String()` 确保 ID 是字符串类型

### 4. 批量数据库写入
- 使用 `ON DUPLICATE KEY UPDATE` 更新已存在的记录
- 使用原生 SQL 批量插入（Drizzle 不直接支持 ON DUPLICATE KEY UPDATE）

## 💡 经验总结

### 1. 需求确认的重要性
- 在实现前，先确认需要收集哪些字段
- 避免实现后才发现字段不需要或缺失

- Windows 和 Linux/Mac 的命令差异
- 路径分隔符的差异（`/` vs `\`）
- 使用 Node.js 的 `path` 模块处理路径
