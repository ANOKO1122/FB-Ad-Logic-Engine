## 二、这次会话中遇到的主要问题与最终结论

### 1. 问题一：最初误把钉钉能力理解为“可以直接被 oauth2-proxy 当 OIDC Provider 使用”

#### 现象

- 先尝试了 `oauth2-proxy + provider=oidc`。
- 又尝试了手动指定 `login-url`、`redeem-url`、`profile-url`。
- 最终容器启动阶段持续失败。

#### 关键报错

- 第一轮报错：
  - `cookie_secret must be 16, 24, or 32 bytes ... but is 44 bytes`
- 第二轮报错：
  - `missing required setting: issuer-url`
  - `missing required setting: jwks-url`

#### 根因

- `oauth2-proxy` 当前版本在 `provider=oidc` 模式下，仍然要求标准 OIDC 的 `issuer` / `jwks` 验签链路。
- 钉钉网页登录第三方网站提供的是 OAuth2 页面授权能力，不是一个能被当前 `oauth2-proxy` 平滑消费的标准 OIDC provider。
- 同时，官方 `oauth2-proxy` provider 列表里也没有 `dingtalk` 原生 provider。

#### 最终结论

- **放弃“oauth2-proxy 直连钉钉”作为本项目 M2 主路线。**
- 该路线不是简单配置问题，而是协议与工具能力不匹配。

---

### 2. 问题二：`OAUTH2_PROXY_COOKIE_SECRET` 一开始填法错误

#### 现象

- 使用 `openssl rand -base64 32` 生成了一个 44 字符串。
- `oauth2-proxy` 报错长度不合法。

#### 根因

- 该值被当成普通字符串处理时，长度变成 44 字节，不满足 16/24/32 bytes 的要求。

#### 解决

- 重新改用符合长度要求的值（例如十六进制字符串方案）。

#### 备注

- 这个问题虽然修掉了，但后续 `issuer/jwks` 问题仍然存在，所以它不是最终阻塞点，只是第一个表层错误。

---

### 3. 问题三：启用了 `auth_request` 但认证服务未正常启动，导致主页直接 500

#### 现象

- 访问 `http://ad_tools.thinkpro.top/rules` 时浏览器出现 `500 Internal Server Error`。

#### 根因

- Nginx 已经启用了认证子请求。
- 但 `oauth2-proxy` 未成功启动。
- 所以 Nginx 对认证上游探测失败，最终给浏览器返回 500。

#### 解决

- 及时回退/停止继续推进这条错误路径。
- 恢复线上可用状态后，再重新讨论正确实现路线。

#### 验证结果

- 最终 `curl -I http://ad_tools.thinkpro.top/rules` 已恢复为 `200 OK`。
- 说明主系统当前已恢复可访问，不再处于被错误认证配置锁死的状态。

---

### 4. 问题四：对钉钉文档的理解曾混淆了三类场景

#### 混淆的三个文档/能力

- 企业内部应用免登（更偏钉钉生态内应用/H5微应用）
- 网页应用（H5微应用）内嵌二维码登录
- 实现网页方式登录应用（登录第三方网站）

#### 最终判断

- 对我们当前场景最匹配的是：
  - **网页登录第三方网站**
- 这是因为我们的目标是：
  - 用户在普通浏览器访问 `http://ad_tools.thinkpro.top/`
  - 被引导到钉钉授权/扫码
  - 完成身份确认后回到自己的系统
  - 然后复用本系统的会话继续访问 `/rules` 与 `/ronghe/`

#### 补充结论

- “方式二：内嵌二维码 `DTFrameLogin`” **不是不能做**，但更适合作为体验升级。
- 第一版更稳的落地路径应是：
  - **方式一：钉钉提供的页面登录授权**

---

### 5. 问题五：是否应该把钉钉登录能力做成一个独立保安层

#### 讨论过的方向

- `oauth2-proxy`
- 新增独立 `auth-service`
- 直接并入现有 FB 后端登录体系

#### 最终结论

- 当前最优路线不是再引入一个新的中间层。
- **直接并入现有 FB 后端认证体系更合理。**

#### 原因

- FB 代码里已经有完整的：
  - `/api/auth/login`
  - `/api/me`
  - JWT 签发
  - `token` cookie
  - `requireAuth / requireActive / requireAdmin`
- 所以现在不是“从零新建登录系统”，而是“在现有登录系统里增加钉钉登录入口”。

## 三、今天形成的正式方案结论

### 正式采用的 M2 修正版路线

#### 核心路线

- **方式一：FB 后端直连钉钉 OAuth2 页面授权**
- **复用现有 JWT / `token` cookie / `/api/me` 体系**

#### 第一版最小闭环

- 新增 `GET /api/auth/dingtalk/login`
- 新增 `GET /api/auth/dingtalk/callback`
- 浏览器访问登录入口后由后端 302 到钉钉授权页
- 钉钉回调带回 `code`
- 后端使用 `code` 换用户 token、拉用户信息
- 根据“钉钉身份 -> 本地用户”映射，找到本地 `users.id`
- 复用现有 `signToken()` 与 `token` cookie 建立登录态
- 浏览器后续访问 `/api/me`、`/rules`、`/dashboard`、`/ronghe/` 时继续沿用现有登录判定

#### 不再采用

- `oauth2-proxy` 作为钉钉登录主实现
- `X-Forwarded-User` 作为当前唯一长期身份方案
- 将“扫码通过”直接等价成“系统管理员身份”

- 用户访问 `http://ad_tools.thinkpro.top/`

- 钉钉负责“身份确认”

- 删除用户、管理员页、规则操作依旧由本地系统后端做最终兜底

### 目标四：最终形成“云端地基稳定，本地业务开发高效，云端发布顺滑”的协同节奏

- 云端先把入口、认证、路径打稳
- 本地再做业务功能与体验优化
- 最后统一回云端发布、验收
