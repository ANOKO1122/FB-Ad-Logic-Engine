#+ 项目总结 - 2026-03-07：Track2 极速结构同步与全局限流

> 本窗口聚焦：在现有「顺序2 伪增量 + 每小时结构轮转」基础上，增加一条 **Track2 极速结构同步轨道**，配合 **全局 FB 请求限流与优先级队列**，把结构可见时效压到约 20 分钟，同时保证 Today 热同步、规则执行等 P0 链路不被结构任务饿死或挤爆配额。并将 Fast Sync 与规则执行前的「脏结构刷新」打通，形成完整的 Track2 → Dirty Flag → 规则前刷新闭环。

## 一、本窗口遇到的问题与解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| Track2 试运行时结构同步易被 FB 限流（400 + `User request limit reached`） | 在开启 Track2 的早期验证中，同一账户在短时间内连续发出多条结构请求（Batch + 单 edge 回补），在代理链路抖动的前提下更容易触发 FB 的「用户级限流」(code=17, subcode=2446079)。原有实现对 400 的处理不够精细，adsets/ads 失败时容易产生「campaigns 成功、其它边失败」的半成功状态。 | 在 `FacebookMarketingAPI.unifiedStructureBatch` 中对 Batch 子响应增加细粒度错误解析（message/code/subcode），并为每个 edge 实现「单 edge 回补 + ISO 回退」：先按 Unix 秒过滤失败，再自动改用 ISO8601 `updated_time > sinceISO` 重拉第一页；如果多次重试后仍报限流，则将该 edge 的 `error` 明确写入结果，由上层 Track2 判定为关键 edge 失败，整账户 Fast Sync 返回 `ok=false, reason='edge_failed'`，避免假成功。 |
| 单账户 Track2 失败时仍有写库风险，容易产生「只铺了 campaigns 的假成功」 | 早期版本的 Fast Sync 主要围绕 unified batch 的成功路径编写，对「批次里某个 edge 失败」的场景只是在日志中给出 warning，`fastSyncStructureForAccount` 仍可能继续写库。 | 在 `collectFastSyncDataForAccount` 中，将 `adsets/ads` 明确视为关键 edge：一轮 unified batch 完成后，如果 `batchResult.adsets.error` 或 `batchResult.ads.error` 非空，则立即释放账户锁并返回 `{ ok:false, reason:'edge_failed', edge_errors:[...] }`，上层 Cron 统计时也会把该账户标记为失败；只有三边都没有错误时才继续进行软分页补页和数据返回。`fastSyncStructureForAccount` 同样复用这一判断逻辑。 |
| Track2 与 Today/Track1 并发时存在「谁先占满配额」与「止损规则被饿死」的风险 | 之前的限流方案主要依赖：① 每任务内部的 `p-limit` 控制账户并发；② 响应头 `x-business-use-case-usage` 的动态休眠。但是不同任务之间（Today 热同步、冷归档、结构轮转、Track2、规则动作）缺乏统一的「全局闸门」，一旦多个任务叠加，很容易出现某一类任务占满所有并发名额，导致规则动作（尤其止损类）排队过久甚至被客户端超时掐断。 | 在 `server/index.js` 的 `FacebookMarketingAPI.makeRequest` 外层引入全局请求队列 `enqueueFacebookRequest({ priority, label, queueTimeoutMs }, taskFn)`：所有对 Facebook 的请求（Today/Track1/Track2/规则动作等）都包装为任务入队；并通过环境变量 `FB_GLOBAL_REQUEST_CONCURRENCY` 和 `FB_GLOBAL_QUEUE_TIMEOUT_MS` 控制发往 FB 的总并发数和排队超时时间。队列内部按优先级 `action > today > track2 > cold` 和 FIFO 顺序调度任务，确保规则动作始终优先于结构同步/冷任务；排队超过 queueTimeout 的请求会返回明确错误，避免客户端自发 timeout 且难以定位。 |
| 结构同步窗口容易「卡在原地」反复拉旧数据 | 传统结构轮转只维护了一个「近 3 天窗口」的 since 水位，且主要围绕 Track1 每小时轮转，新增的 Track2 若仅依赖同一个水位，极端情况下（例如连续空结果、限流重试）容易导致窗口停滞，重复拉取已经确认为空的时间段。 | 新增独立的 Fast Sync 水位：通过迁移 032 在 `structure_sync_status` 中增加 `last_fast_sync_ts` 和 `last_fast_filter_since_sec`，并在 Track2 成功时无论是否有数据命中，都用本轮使用的 `sinceSec` 推进 `last_fast_filter_since_sec`，用完成时间写入 `last_fast_sync_ts`。下次 Track2 计算窗口时使用 `sinceSec = max(last_fast_sync_sec - bufferSec, now - 3d)`，其中 buffer 默认为 4 小时（`TRACK2_FAST_SYNC_BUFFER_SEC=14400`），既保证了「有安全缓冲」又不会一直停留在过去很久的时间点。 |
| 规则执行前无法感知「结构是否刚被快速更新」，可能在旧结构上做决策 | 原有链路中，结构同步与规则执行之间仅通过「近 3 天结构轮转 + 手动刷新」松散关联，缺少一个明确的「脏结构标记」机制来驱动规则在必要时刷新结构。 | 通过迁移 033 在 `structure_sync_status` 中新增 `fast_dirty`、`fast_dirty_marked_at`、`fast_dirty_cleared_at` 列：Track2 合并 upsert 完成后，对于结构有变更的账户，将 `fast_dirty` 置 1 并记录标记时间；规则调度入口 `executeRulesForAccount(fromScheduler=true)` 在获取账户锁后调用 `refreshStructureIfDirtyBeforeRules(lockedAccountId)`，若 dirty=1 则按最近 Fast Sync 水位触发一次快速结构刷新（内部复用 unified batch + 软分页 + 写库），成功后清空 dirty 并记录 cleared 时间，失败则保持 dirty 并写告警日志——从而把 Track2 与规则执行前的「脏结构刷新」串成闭环。 |
| SOCKS5/TLS 代理场景下日志中频繁出现 `❌ TLS连接错误`，干扰观察 Track2 行为 | 在自实现 SOCKS5 隧道 + TLS 解析器中，响应体已经完整收完并成功 JSON 解析后，底层连接在 FIN/close 过程中仍可能触发 `tlsSocket` 的 error 事件，导致日志中出现 `❌ TLS连接错误`，但实际上业务已经拿到数据。之前没有很好地区分「可恢复噪声」与「真实失败」，易造成误判。 | 在 `server/socks5.js` 中为 TLS 错误增加回退路径：当 `tlsSocket.on('error')` 触发时，若此前已经收到完整的 HTTP 头和响应体，则优先尝试对已缓存的 body 做 JSON 解析，成功则打印 `✅ JSON解析成功（由TLS错误回退路径完成）` 并视为这次请求成功，仅把 TLS 错误记录为监控事件；只有在 body 不完整或解析失败的情况下才真正将错误向上抛出。通过 Track2 实际运行日志可以看到「200 OK → TLS 错误 → JSON 解析成功（回退）」的完整链路，此类情况不会影响 Fast Sync 成功率。 |

## 二、项目目标与当前进度（本窗口相关）

- **整体目标**：
  - 将结构数据的可见时效从「依赖每小时轮转 + 手动刷新」提升到 **P95 约 20 分钟** 级别；
  - 在此基础上保持配额安全：结构同步必须让路 Today 热同步与规则执行，且不能显著增加 `User request limit reached` 或高 usage 触发熔断的风险；
  - 将结构同步结果与规则执行前的数据新鲜度串联起来，通过 Dirty Flag 与「规则前刷新」保证规则尽量跑在最新结构上。

- **本窗口完成的关键工作**：
  1. **统一 Batch 内核（unifiedStructureBatch）**：在 `FacebookMarketingAPI` 中实现 campaigns/adsets/ads 三边统一 Batch 拉取函数，对 FB Batch 子响应做细粒度错误解析与单 edge 回退，形成所有结构同步路径（Track1/Track2/手动同步）的共同核心。
  2. **Track2 Fast Sync 账户级链路**：在 `structureSyncService.js` 中实现 `fastSyncStructureForAccount` 与 `collectFastSyncDataForAccount`，统一处理单账户的 Batch 拉取、软分页补页、关键 edge 校验与数据返回。
  3. **Track2 Cron + 全账户合并写库**：在 `cronService.js` 中增加 Track2 Fast Sync 定时任务（默认每小时 :07/:27/:52，白名单账户、可配置并发与窗口 buffer），一轮内对所有目标账户调用 collect，再通过 `applyMergedFastSyncPayload` 将结果合并去重后对三张结构表进行分块批量 upsert，并按账户标记 fast_dirty。
  4. **Fast Sync 专用水位与 Dirty Flag 生命周期**：通过迁移 032/033 为 `structure_sync_status` 增加 Fast Sync 水位与 fast_dirty 系列字段，在 Track2 和规则执行入口之间建立「结构是否需要刷新」的桥梁。
  5. **全局 FB 请求限流与优先级队列**：在 `makeRequest` 之上引入全局请求队列与并发闸门，所有任务统一通过一个带优先级的队列发请求，避免结构任务与规则动作互相抢占配额。

- **与既有设计的关系**：
  - Track2 并没有替代原来的「顺序2：伪增量 + 每小时结构轮转 + 手动刷新」，而是在其之上增加了一条「更频繁、轻量」的结构同步轨道：
    - **Track2**：更高频率（3 次/小时）、更小窗口（上次 Fast Sync 水位减 4 小时缓冲，且不超过近 3 天）、Batch + 软分页 + 合并 upsert，主要职责是尽快把近期结构变化（新建/改名/状态变化）补进本地镜像表；
    - **Track1**：保持「近 3 天」语义的每小时轮转，全量兜底与校准，确保即便 Track2 某轮失败或因 usage/熔断跳过，也能在下一轮恢复全局一致性；
    - **手动刷新**：仍然是最高优先级的运营兜底手段，配合 Track2/Track1 一起构成完整的结构同步方案。

## 三、本次修改的主要代码与行为摘要

### 3.1 统一三边 Batch 内核（unifiedStructureBatch）

- 文件：`server/index.js`
- 关键点：
  - 新增 `FacebookMarketingAPI.unifiedStructureBatch(accountId, { sinceSec, limit })`：内部构造一个包含三条子请求的 Batch（campaigns/adsets/ads），fields 明确包含 id/name/status/effective_status/updated_time/created_time 及层级 ID（adset_id/campaign_id）。
  - 对响应体：
    - 移除早期「直接把 body 当 JSON 用」的做法，统一使用 `JSON.parse(item.body)` 处理双重转义；
    - 若子响应 code≠200 或 body 中含 error，则构造带 message/code/subcode 的错误字符串写入 result[edge].error，并打印 `[UnifiedBatch] ... 非成功响应 code=...` 日志；
    - 对 body.error 存在但 HTTP 层 200 的情况，打印 `[UnifiedBatch] ... API错误: ...` 日志并跳过该 edge；
    - 所有 edge 完成后，对存在 error 的 edge 依次调用 `getStructurePage` 做「单 edge 回补」，先用 Unix 秒 filtering 失败再自动切换到 ISO8601 过滤。

### 3.2 Track2 Fast Sync 单账户链路

- 文件：`server/services/structureSyncService.js`
- 关键点：
  - `collectFastSyncDataForAccount(accountId, facebookApi, opts)`：
    - 使用 `GET_LOCK('structure_fast:' + accountId, 0)` 做账户级 MySQL 锁，防止与手动刷新或其他结构任务并发写冲突；
    - 计算 `sinceSec`（默认 3 天前，可由 Cron 传入具体窗口）与 `limit`（默认 500）；
    - 调用 unified batch 得到三边首批数据，并构建 edgeState 记录每个 edge 的 `after` 游标和已补页数；
    - 将 `adsets/ads` 标记为关键 edge，若其中任一 edge 在 Batch 解析后仍有 error，则释放锁并返回 `{ ok:false, reason:'edge_failed', edge_errors:[...]} `，避免写库；
    - 对有 `after` 的 edge 调用 `getStructurePage` 做软分页补页，直到 `items.length=0 且 after=null` 或达到 `maxSoftPagesPerEdge`（默认 20，上限 100）；到达上限时打印 `[FastSync-Collect] ... 软分页达到上限 ...`。
  - `fastSyncStructureForAccount`：在早期验证中作为单账户快速落库入口，现在主要被 Track1/手动路径复用（Track2 合并写库主要使用 collect + applyMerged）。

### 3.3 Track2 Cron + 多账户合并写库

- 文件：`server/services/cronService.js`
- 关键点：
  - 新增 Track2 Fast Sync 定时任务：
    - Cron 为 `7,27,52 * * * *`（每小时 3 次，特意避开 :45 的 Today 热同步）；
    - 账户列表来源：`account_mappings` 中 is_active=1 的账户按 fb_account_id 排序，再按环境变量 `TRACK2_FAST_SYNC_ACCOUNT_IDS` 白名单过滤；
    - 并发控制：通过 `p-limit(TRACK2_FAST_SYNC_CONCURRENCY)`（默认 1～3）控制单轮中并发处理的账户数；
    - 自适应窗口：对每个账户读取 `structure_sync_status` 中 `COALESCE(UNIX_TIMESTAMP(last_fast_sync_ts), UNIX_TIMESTAMP(last_success_at)) AS last_fast_sync_sec`，计算 `sinceSec = max(last_fast_sync_sec - bufferSec, nowSec - 3d)`，buffer 由 `TRACK2_FAST_SYNC_BUFFER_SEC` 控制（当前为 14400 秒，日志中 `lastFastSyncSec - sinceSec = 14400` 反推验证通过）；若查询失败则退回 `nowSec - 3d`。
  - 对每个账户：
    - 若 Token 熔断器 `getCircuitBreakerStatus().isLocked` 为 true 或 `getLastUsageRate() >= TRACK2_FAST_SYNC_USAGE_SKIP_THRESHOLD`，直接在任务入口打印 `[Track2 FastSync] 跳过：Token 熔断中 / API 使用率 ...` 并跳过本轮；
    - 否则调用 `collectFastSyncDataForAccount` 并记录 `{ accountId, ok, reason, synced_ads }`。
  - 轮次结束后：
    - 对 `ok=true` 的账户收集 `collected` payload 列表，并传入 `applyMergedFastSyncPayload` 做全账户合并 upsert；
    - `applyMergedFastSyncPayload` 返回 `{ accounts, campaigns, adsets, ads, dirtyMarked }`，Cron 记录一条 `[Track2 FastSync] mergedUpsert 完成：accounts=... campaigns=... adsets=... ads=... dirtyMarked=...` 日志；
    - 最终打印 `[Track2 FastSync] 完成：success=X/Y, lock_busy=Z, synced_ads=T` 统计本轮汇总结果。

### 3.4 Fast Sync 水位与 Dirty Flag

- 文件：`server/db/migrations/032_add_fast_sync_columns_to_structure_sync_status.sql`、`033_add_fast_dirty_columns_to_structure_sync_status.sql`、`server/services/structureSyncService.js`、`server/services/cronService.js`
- 关键点：
  - 迁移 032：为 `structure_sync_status` 新增 `last_fast_sync_ts`（TIMESTAMP）和 `last_fast_filter_since_sec`（BIGINT）列，分别记录最近一次 Fast Sync 完成时间与该轮使用的 sinceSec。
  - 迁移 033：新增 `fast_dirty`（TINYINT）、`fast_dirty_marked_at`、`fast_dirty_cleared_at`，用于表示「结构是否在近期 Fast Sync 中发生变更」及其标记/清理时间。
  - 在 `applyMergedFastSyncPayload` 中：对每个参与 Fast Sync 的账户根据合并后的结构数据是否有变动决定是否将 `fast_dirty` 置 1 并写入 marked_at；调用者（Track2 Cron）根据结果在日志中打印 dirtyMarked 数量。
  - 在 `executeRulesForAccount(fromScheduler=true)` 中：在加账户锁后调用 `refreshStructureIfDirtyBeforeRules(lockedAccountId)`：
    - 若 `fast_dirty=1`，则按最近 Fast Sync 水位触发一次快速结构刷新（内部调用 unified batch + 软分页 + 分块 upsert）；成功后将 `fast_dirty` 置 0、写入 cleared_at；
    - 若刷新失败，则保留 dirty 并写 warn 日志，但不会阻断本轮规则执行，仅作为「后续再尝试刷新」的信号。

### 3.5 全局 FB 请求限流与优先级队列

- 文件：`server/index.js`（`FacebookMarketingAPI.makeRequest` 与 `enqueueFacebookRequest` 相关实现）、`server/utils/tlsErrorMonitor.js`（统计用）
- 关键点：
  - 环境变量：
    - `FB_GLOBAL_REQUEST_CONCURRENCY`：全局并发上限（建议 10，可按实际代理/网络情况调整）；
    - `FB_GLOBAL_QUEUE_TIMEOUT_MS`：任务在队列中的最长等待时间，超过即返回队列超时错误；
    - 所有对 Facebook 的调用会在 options 中传入 `requestPriority`（枚举：`action` / `today` / `track2` / `cold`）与 `requestLabel`（如 `today_sync`、`track2_batch`、`pause_ad` 等）。
  - 队列行为：
    - 内部维护一个按优先级 + FIFO 的任务队列，以及固定数目的「并发槽位」；
    - 当有空闲槽位时，从队列头部取出最高优先级任务执行，执行完成释放槽位；
    - 若某任务在队列中等待时间超过 `queueTimeoutMs`，则直接以错误结束（不会再尝试发送请求），防止客户端自身 HTTP 超时造成难以诊断的问题。
  - 日志与观测：
    - 对 `requestPriority='action'` 的任务，在实际执行时必打 `[FBQueue] 开始执行 priority=action label=... wait_ms=...`，方便从日志中确认关键止损/扩量请求是否被及时发出；
    - 对 Track2/冷任务，仅在等待时间较长或发生错误时输出 summary 日志，避免噪音淹没关键信息。

## 四、验证方法与结果概览

1. **统一 Batch 内核与软分页**：
   - 使用脚本 `server/scripts/test-unified-structure-batch.js` 对某测试账户调用 `unifiedStructureBatch(accountId, { sinceSec: now-3d, limit:100 })`，确认三个 edge 均返回 code=200、fields 完整且 `paging.after` 存在；
   - 使用 `test-structure-page-after.js` 对 ads 的 `after` 再拉一页，验证「遇到 data.length=0 且 after=null 即停止」的软分页语义；
   - Track2 实际运行日志中可以看到：第一轮 Batch 返回 after，随后针对有 after 的 edge 调用 `GET /act_xxx/campaigns?...after=...` / `GET /act_xxx/ads?...after=...` 等补页请求，并在补页返回数据为空且 after=null 时停止。

2. **Fast Sync 水位与 4 小时缓冲窗口**：
   - 在结构同步状态表中手动设置某账户的 `last_fast_sync_ts`，观察下一轮 Track2 日志中打印的 `lastFastSyncSec` 与 `sinceSec` 差值是否为 14400 秒（4 小时）；
   - 日志示例：`lastFastSyncSec=1772785622 bufferSec=14400 sinceSec=1772771222`，差值 14400=4×3600，可反推出窗口公式 `sinceSec = max(lastFastSyncSec - 14400, nowSec - 3d)` 已生效。

3. **限流与失败短路**：
   - 在高压场景下（如代理表现不稳定或手动模拟较高 usage）观察 Track2 日志：
     - 若触发 FB 「User request limit reached」错误，Batch 日志会出现 `[UnifiedBatch] ... edge=adsets/ads 非成功响应 code=400 error=User request limit reached code=17 subcode=2446079`，随后 `collectFastSyncDataForAccount` 会打印 `[FastSync-Collect] ... 关键edge失败，返回失败 ...` 并使本账户 `ok=false`；
     - 若最近 `getLastUsageRate()` 超过 `TRACK2_FAST_SYNC_USAGE_SKIP_THRESHOLD`，则本轮 Track2 会直接输出 `[Track2 FastSync] 跳过：API 使用率 ... >= 阈值`，不再发送结构请求，保护配额。

4. **Dirty Flag 与规则前刷新**：
   - 在 Track2 成功同步结构后，检查 `structure_sync_status.fast_dirty` 对应账户是否被置为 1，且 `fast_dirty_marked_at` 记录了时间；
   - 在规则每分钟 Cron 正常运行一段时间后，通过日志确认：当某账户 dirty=1 时，`executeRulesForAccount(fromScheduler=true)` 会先打印一条「脏结构刷新」相关日志（实现中可见），刷新成功后 fast_dirty 被清 0、`fast_dirty_cleared_at` 更新；
   - 若刷新过程中出现错误，则 dirty 标记不会被清除，下次规则执行前仍会尝试刷新。

5. **全局队列优先级与止损不饿死**：
   - 在 Track2 与 Today 热同步同时进行时，手动触发一条止损规则（例如 Pause 某广告），观察日志：
     - 应该可以看到该规则对应的 FB 请求很快打出 `[FBQueue] 开始执行 priority=action label=pause_ad ...`，且 wait_ms 较小；
     - 即使此时 Track2 正在发送结构请求，action 仍然能插队优先获得并发槽位。
