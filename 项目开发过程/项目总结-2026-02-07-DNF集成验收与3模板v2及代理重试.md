## 二、遇到的问题与解决方法

| 问题 | 原因/现象 | 解决方法 |
|------|------------|----------|
| **1. DNF 验证需手把手操作** | 用户需按文档逐条构造请求、替换 accountId/token/ruleId，易出错。 | 新增 `docs/Thunder_Client_DNF验证集合.json`（预填 accountId、token），配套 `docs/DNF验证_手把手步骤.md`，支持导入即用、按序执行。 |
| **2. 请求 Body JSON 格式错误** | Thunder Client 导入后 raw 显示为字符串或含多余字符，导致 INVALID_JSON。 | 提供可直接复制的 JSON 供用户粘贴到 Body；并在手把手步骤中说明「清空后粘贴、检查英文引号」。 |
| **3. 代理 TLS 提前断开** | SOCKS5 握手成功，但 TLS 握手前 socket 被代理断开：`Client network socket disconnected before secure TLS connection was established`、`收到空响应`。 | 在 `makeRequest()` SOCKS 分支增加瞬态错误重试：仅对上述两类错误 sleep 200–500ms 后重试 1 次；重试失败再走 socks-proxy-agent 兜底。 |
| **4. 重试条件过宽** | 初版将 ECONNRESET、socket hang up 也纳入重试，可能误重试「请求已发送但响应丢失」的场景。 | 收窄为只重试「TLS 未建立前」的高置信错误：`Client network socket disconnected before secure TLS connection was established`、`收到空响应`。 |
| **5. 重试日志与实测不一致** | 打印 jitter 后又调用 `sleepWithJitter(200,500)`，日志显示与实际等待不一致。 | 改为 `const jitter = 200 + Math.floor(Math.random() * 301)`，用同一 jitter 做 `setTimeout`，日志=实际。 |
| **6. 代理并发压力** | 多账户同步时并发 8 易触发代理瞬时断连。 | `CONCURRENT_LIMIT` 从 8 降到 6（ingestorService.js），降低代理瞬时压力。 |
| **7. 模板仍为 v1** | 点模板后填充 v1 数组，创建规则仍为 v1，无法利用 v2 DNF 能力。 | `applyTemplate()` 改为：`isV2EditMode = true`、`conditionsV2Json = JSON.stringify({ version: 2, groups: [...] }, null, 2)`，三个模板（stop_loss、scale_up、zero_click）均输出 v2 结构。 |
| **8. 模板切换后未进 JSON 模式** | 若只改 conditions 而不切 isV2EditMode，前端仍显示 v1 表单。 | 每次 applyTemplate 首先设置 `isV2EditMode.value = true`，确保点模板后立即进入 JSON 模式。 |

## 三、本窗口完成内容（实现与文档）

### 实现

- **RuleManager.vue**：`applyTemplate(type)` 对 stop_loss、scale_up、zero_click 设置 `isV2EditMode=true`、`conditionsV2Json` 为 v2 结构；清空 `ruleForm.conditions`、`logicOperator='AND'`；保留原 actions。

- **ingestorService.js**：`CONCURRENT_LIMIT` 从 8 调整为 6。

### 文档

- **docs/Thunder_Client_DNF验证集合.json**：预填 accountId、token 的 Thunder Client 集合，含 6 个请求（验证 3/4、创建 1/2、执行 1/2）。

- **docs/2.3_条件组_DNF_验证教学.md**：补充 Thunder Client 集合引用说明。
