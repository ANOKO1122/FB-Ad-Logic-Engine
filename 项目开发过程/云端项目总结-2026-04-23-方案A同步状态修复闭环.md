## 一、本窗口项目开发目标

本窗口围绕 `方案A同步状态修复` 执行，目标分成两层。

### 1. 稳定性目标

解决统一心跳在多账户并发场景下的“单账户拖死整轮”问题，避免以下现象继续出现：

- 某个账户任务卡住，整轮心跳长时间不收尾
- `heartbeatRunning` 无法释放，后续任务持续命中 `already_running`
- 进程虽然没死，但实际已经“假活着”，`systemd Restart=on-failure` 无法介入

### 2. 可观测性目标

把统一心跳的账户级最后状态写入 `structure_sync_status`，让系统状态页真正基于“业务心跳事实状态”判断，而不是继续依赖：

- `ad_snapshots.MAX(synced_at)`

目标是让页面能明确区分：

- 成功且有数据
- 成功但无数据
- 同步失败
- 从未尝试

## 二、本窗口实际执行过程

这次执行严格按原方案顺序推进，没有改方案方向，只是在 `Milestone 0` 与 `0.5` 之间补了一个必要的 `0.1` 层，用来解决“表层超时结束了，但底层任务并没有真正停下”的工程问题。

### 阶段 0：单账户超时 + Promise.allSettled

目标：先止血，保证整轮能收口。

做法：

- 从 `unifiedHeartbeatSync()` 中抽出账户处理函数 `processHeartbeatAccount()`
- 新增公共工具 `server/utils/withTimeout.js`
- 用账户级超时包装代替无限等待
- 外层从 `Promise.all` 改为 `Promise.allSettled`

效果：

- 单账户超时不再拖死整轮
- `finally` 有机会执行，锁和 `heartbeatRunning` 可以释放

### 阶段 0.1：超时中断 + drain + 升级失败

目标：让超时不只是“上层不等了”，而是“底层尽量停下来”。

做法：

- 每个账户任务独立 `AbortController`
- `signal` 从统一心跳一路向下透传到 Facebook API 请求
- 超时触发时主动 `abort()`
- 增加 drain/grace period，确认任务是否真正撤场
- 若宽限期后仍未撤场，则升级为进程级失败

效果：

- 防止“账面结束、实体未停”
- 避免下一轮心跳和僵尸任务重叠执行

### 阶段 0.5：watchdog + process.exit(1) 自愈

目标：解决“整轮假活着但没明显抛错”的场景。

做法：

- 引入 `heartbeatStartedAt`、`heartbeatLastProgressAt`
- 在关键阶段打点 `markHeartbeatProgress()`
- 加入 watchdog 定时检查“是否长时间无进展”
- 命中阈值时执行 `process.exit(1)`，交给 `systemd` 自愈

效果：

- 让 `Restart=on-failure` 真正参与恢复
- 避免手工清旗子带来的并发重叠风险

### 阶段 1：表结构与状态契约

目标：把业务心跳状态字段从结构同步状态里彻底解耦。

做法：

  - `server/db/migrations/044_add_heartbeat_status_columns_to_structure_sync_status.sql`

新增字段：

- `last_heartbeat_data_update_at`

结果码：

### 阶段 2：统一心跳写状态

目标：让统一心跳结果对象真正写回数据库。

做法：

- 在每账户主同步开始前写 `last_heartbeat_attempt_at`

- 失败时统一写 `FAILED`

效果：

- `structure_sync_status` 现在可以准确表达每个账户这轮业务心跳的最终状态

## 三、本窗口遇到的问题与解决方法

### 问题 1：表层超时解决不了底层残留任务

**现象**

- 只做 `Promise.allSettled + Promise.race(timeout, task)` 后，整轮能结束
- 但超时任务的底层 API 请求还在继续执行

**风险**

- 并发槽位账面释放，实际后台任务仍在跑
- 下一轮心跳可能和上一轮残留任务重叠

**解决方法**

- 引入 `AbortController`
- 将 `signal` 透传到主同步链路和 Facebook API 请求
- 超时后 `abort()` + drain
- 仍未撤场则升级为进程级失败

### 问题 2：watchdog 极端压测时触发太快，容易误判

**现象**

- 使用 `HEARTBEAT_WATCHDOG_TIMEOUT_MS=1` 时，几乎在 `accounts_loaded` 后立即退出

**解释**

- 这是故意压测 watchdog 的极端验证，不是业务逻辑错误

**解决方法**

- 明确区分“压力验证”和“正常生产阈值”
- 只把这类验证作为“自愈链路是否打通”的证明

### 问题 3：`success:false` 被误当成同步成功

**现象**

- 某些同步分支会返回 `success:false` 但不抛异常
- 原逻辑只要没抛错就会把 `synced = true`

**解决方法**

- 统一以 `syncResult.success === true` 作为成功判定
- `success:false` 统一映射为 `FAILED`
- 给账户结果对象补上 `resultCode / errorMessage / durationMs`

### 问题 4：重试结果对象不完整

**现象**

- 原逻辑重试成功后只回填 `synced = true`
- 没有同步回填 `dataUpdated / syncError / resultCode / durationMs`

**解决方法**

- 用完整 `retryResult` 覆盖原账户结果
- 保留 `archived/finalized/activeAdIds` 的合理合并

### 问题 5：页面看起来“全都一样”，但数据库已有状态

**现象**

- 页面表头已更新
- 但每条记录都显示：
  - 从未尝试
  - 暂无成功记录
  - 暂无数据更新
  - 暂无结果

**根因**

- 前端已经更新
- 但后端服务没有重启，`/api/system/health` 仍跑旧代码

**解决方法**

- 重启后端服务
- 再刷新页面，页面即按新字段正常显示

这个问题说明：**前端资源更新不等于 Node 服务已经切换到新接口口径。**
