#### 2. 高频检查 + 账户本地时区窗口

**问题：**
- 原实现使用固定服务器时间 `06:00`（`cron: 0 6 * * *`）
- 所有账户都在同一时间归档，不考虑账户时区
- 多时区账户无法正确归档

**解决方案：**
- ✅ 将固定 `06:00` 改为每 10 分钟检查一次（`*/10 * * * *`）
- ✅ 遍历所有账户，检查账户本地时区是否在 06:00-06:09 窗口
- ✅ 使用 Luxon 将当前时刻转换到账户时区判断
- ✅ 目标日期 = 账户本地时区的"昨日自然日"

**核心代码：**
```javascript
// 判断账户本地时区是否在 06:00-06:09 窗口
const now = DateTime.now()
const localTime = now.setZone(timezoneName)
const hour = localTime.hour
const minute = localTime.minute
shouldArchive = (hour === 6 && minute >= 0 && minute <= 9)
```

**效果：**
- ✅ 支持多时区账户（不同账户在不同时间归档）
- ✅ 避免所有账户同时归档造成峰值
- ✅ 平滑负载分布

---

#### 3. 幂等与并发控制

**问题：**
- 多实例运行时可能重复归档
- 同一账户同一日期可能被多次归档

**解决方案：**
- ✅ COUNT(*) 检查：归档前检查是否已归档
- ✅ DB 锁：使用 `GET_LOCK('archive:{account_id}:{date}', 0)` 防止并发
- ✅ 唯一索引：`uk_account_ad_date(account_id, ad_id, date)` 确保数据库层幂等
- ✅ ON DUPLICATE KEY UPDATE：插入时自动更新已存在记录

**三重保护：**
1. COUNT(*) 快速跳过（性能优化）
2. GET_LOCK 防止多实例并发（分布式锁）
3. 唯一索引 + ON DUPLICATE KEY UPDATE（数据库层幂等）

### 阶段C：时区自动更新修复

#### 1. 问题发现

**问题：**
- 用户通过 Facebook API 获取账户信息，发现时区是 `Asia/Shanghai`
- 但 `ad_snapshots` 表中最新数据时区是 `UTC`
- `account_mappings.timezone_name` 也是 `UTC`

**根本原因：**
- `syncAllAccountsTodayStats` 从 `account_mappings` 读取时区（可能是旧的 `UTC`）
- 传入 `syncAccountTodayStats` 时，如果时区不为 `null`，不会从 Facebook API 获取最新时区
- 导致即使 Facebook API 返回 `Asia/Shanghai`，也不会使用

---

#### 2. 修复方案

**解决方案：**
- ✅ 修改 `syncAccountTodayStats`：无论是否传入 `timezoneName`，都从 Facebook API 获取最新时区
- ✅ 如果 API 返回的时区与数据库不一致，自动更新 `account_mappings.timezone_name`
- ✅ 修改 `syncAccountSlidingWindow`：同样从 API 获取并更新时区

**核心代码：**
```javascript
// 无论是否传入 timezoneName，都从 Facebook API 获取最新时区
const apiTimezone = await facebookApi.getAccountTimezone(accountId)

// 如果时区不一致，更新数据库
if (timezoneName && timezoneName !== apiTimezone) {
  await pool.execute(
    `UPDATE account_mappings SET timezone_name = ? WHERE fb_account_id = ?`,
    [apiTimezone, accountId]
  )
}

// 使用从 API 获取的时区（最准确）
timezoneName = apiTimezone
```

**效果：**
- ✅ 每次同步时自动从 Facebook API 获取最新时区
- ✅ 自动更新数据库配置
- ✅ 确保数据一致性

### 阶段D：滑动窗口语义污染修复

#### 1. 问题发现

**问题：**
- `syncAccountSlidingWindow` 拉取 `today` + `last_7d` 数据
- `last_7d` 是过去7天的聚合值（多天合计）
- 这些数据被写入 `ad_snapshots` 表
- 当查询 `today` 时，可能会把 `last_7d` 的聚合值当作"今天的数据"
- **语义污染**：今天没有花费，但过去7天有花费，查询 today 时可能误判

**示例：**
- 某广告今天没有花费，但过去 7 天合计花费 300
- 滑动窗口把 `last_7d` 聚合写入 `ad_snapshots` 后
- `today` 查询会捡到这条快照
- 规则"今天花费>0"可能误判为真
- 这与"今天的真实值"为 0 的业务语义相悖

---

#### 2. 修复方案

**核心改进：严格按自然日语义**

1. **`ad_snapshots` 只存"今天"的实时快照**
   - 只使用 `date_preset=today` 的数据
   - 不混入 `last_7d` 聚合值

2. **`daily_stats` 存"按自然日"的历史归档**
   - 使用 `time_increment=1` + `since/until` 拉取按日数据
   - 对每个"广告-自然日"更新 `daily_stats`
   - 把迟到的归因补齐到正确日期

3. **查询路由保持不变但更干净**
   - `today` → `ad_snapshots`（最后快照，只包含 today 数据）
   - `yesterday` → `daily_stats`，06:00 前空则降级到 `ad_snapshots` 的"昨日最后快照"
   - `last_X_days` → `daily_stats`（历史段）+ `ad_snapshots`（今天段）合并后动态聚合

---

#### 3. 实现细节

**新增函数：**
1. `fetchInsightsByDay`：使用 `time_increment=1` 拉取按日数据
2. `updateDailyStatsFromInsights`：将按日数据更新到 `daily_stats` 表

**重构函数：**
1. `syncAccountSlidingWindow`：严格按自然日语义
   - `today` 数据 → `ad_snapshots`
   - 过去 N 天的按日数据 → `daily_stats`

2. `syncAllAccountsSlidingWindow`：添加分布式锁

**优化建议实施：**
1. ✅ 在 ON DUPLICATE KEY UPDATE 中补充 `timezone_name` 更新
2. ✅ 修正定时任务日志的统计口径（使用 `todayCount` 和 `dailyStatsCount`）
3. ✅ 确保日期字段一致性（使用 `date_start` 作为自然日）

## 🎓 关键知识点总结

### 1. 时区处理策略

**数据时区优先：**
- `today` 查询：优先使用 `ad_snapshots` 最新快照的 `timezone_name`
- `last_X_days` 查询：使用 `daily_stats.timezone_name` 或 `account_mappings.timezone_name`

**自动更新机制：**
- 每次数据同步时自动从 Facebook API 获取最新时区
- 如果时区不一致，自动更新 `account_mappings.timezone_name`
- 历史时区可通过工具脚本统一刷新

---

### 2. 滑动窗口语义

**严格按自然日语义：**
- `ad_snapshots` 只存"今天"的实时快照（`date_preset=today`）
- `daily_stats` 存"按自然日"的历史归档（`time_increment=1`）
- 迟到归因修复到正确的自然日

**查询路由：**
- `today` → `ad_snapshots`（只包含 today 数据）
- `yesterday` → `daily_stats`（06:00 前降级到 ad_snapshots）
- `last_X_days` → `daily_stats`（历史）+ `ad_snapshots`（今天）合并后聚合

---

### 3. 幂等与并发控制

**三重保护：**
1. COUNT(*) 检查：快速跳过已归档记录
2. GET_LOCK：防止多实例并发归档
3. 唯一索引 + ON DUPLICATE KEY UPDATE：数据库层幂等

---

### 4. 定时任务配置

**当前配置：**
- 规则执行：每 15 分钟（离线查询模式）
- 数据同步：每 10 分钟（自动更新时区）
- 滑动窗口同步：每 30 分钟（修复归因延迟）
- 冷数据归档：每 10 分钟（高频检查 + 账户本地时区）
