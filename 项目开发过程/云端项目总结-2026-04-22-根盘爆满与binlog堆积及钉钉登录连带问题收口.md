# 云端项目总结-2026-04-22-根盘爆满与 binlog 堆积及钉钉登录连带问题收口

> 说明：本文件记录 2026-04-22 在云端环境中，针对「网站极卡、钉钉扫码长时间转圈后报错」等问题完成的排查结论、根因定位、已执行处置与后续运维要点，便于日后回溯与新人上手。

---

## 一、本窗口要解决的问题（现象层）

1. **钉钉扫码登录**：长时间刷新后失败，浏览器侧出现 `DINGTALK_CALLBACK_ERROR` 等报错；日志中曾出现 `DINGTALK_STATE_MISMATCH`、`AUTH_REJECT_STATE_MISMATCH` 等。
2. **整站体验**：无痕窗口打开也很慢，系统整体响应差。
3. **日志与数据库**：出现 `写入 auth_login_audits 失败 The table 'auth_login_audits' is full`、`Lock wait timeout exceeded` 等。

---

## 二、初步结论（业务层误解澄清）

- **钉钉 AppKey/AppSecret 并非「静态过期 token」**：用户态与 app 态 token 均在运行时向钉钉接口换取（见 `server/services/dingtalkAuthService.js`），不应把登录失败简单归因于「钉钉 token 过期」。
- **`state` 不匹配**仍可能由 Cookie 策略、域名/协议不一致、重复扫码等引起；历史上 HTTP 环境下 `Secure` Cookie 导致不回传 `state` 的案例见 `云端项目总结-2026-03-31-M2钉钉登录闭环实施与验证收口.md`。
- 本窗口最终确认：**磁盘与 MySQL binlog 失控是主因**，在此之上叠加数据库锁等待、接口超时，会表现为「整站卡 + 回调 500」。

---

## 三、根因（基础设施层）

| 层级 | 结论 |
|------|------|
| 宿主机 `/` | 使用率曾达 **100%**，几乎无可用空间；伴随 swap 吃紧、IO 与系统调度恶化。 |
| Docker 卷 `ronghe-platform_mysql_data` | 体积极大；拆解后主要为 **MySQL binlog 文件**（`binlog.000xxx`）连续堆积，量级约 **150GB+**，而业务库 `fb_ad_brain` 实际数据体量远小于此。 |
| `auth_login_audits is full` | **并非表行数撑爆**：在磁盘满、InnoDB 无法扩展空间时，写入失败也会被报成「表满」类错误。 |
| `rule_history` 体量 | 曾观测到千万级行、约 **800MB** 量级；与「规则引擎高频写审计」一致；**定时清理任务默认仍生效**，但若磁盘满，分批 DELETE 可能无法顺利完成，形成积压。 |

**一句话**：根分区被 binlog 占满 → MySQL 与应用层连锁异常 → 前端体感「极卡」、OAuth 回调易超时或 500。

---

## 四、已采取的解决方法（按执行顺序）

### 1. 释放非业务数据空间（低风险）

- `docker builder prune -af`：清理 Docker 构建缓存（本环境约释放 **1.3GB** 量级，以实际输出为准）。
- `truncate -s 0` 清空各容器 `*-json.log`：降低 `/var/lib/docker/containers` 下 JSON 日志占用。

### 2. MySQL binlog 紧急瘦身（高收益）

- 在确认当前正在写入的 binlog 文件名后（本环境当时为 `binlog.000140`），执行：
  - `PURGE BINARY LOGS TO 'binlog.000140';`
- 效果：删除该序号之前的旧 binlog 文件，**根分区可用空间从「满盘」恢复到健康区间**（本环境观测为 binlog 文件数从约 140 个降至 2 个，数据卷从约 155GB 级降至约数 GB 级）。

### 3. binlog 自动过期策略（持久化）

- 使用 MySQL 8.0 的 **`SET PERSIST binlog_expire_logs_seconds = <秒>`**，写入数据目录下的 `mysqld-auto.cnf`，**容器重启后仍保留**，无需在 compose 中额外挂载 `my.cnf`（当前 `ronghe-platform/docker-compose.yml` 中 MySQL 仅挂数据卷）。
- 本窗口内策略调整脉络（便于审计）：
  - 先曾将过期设为 **21 天**（`1814400` 秒）以平衡恢复窗口与磁盘。
  - 后续按运维偏好改为 **1 天**（`86400` 秒）；验证命令输出中 `binlog_expire_logs_seconds` 已为 `86400` 即表示生效。
- **若需改为 3 天**：`259200` 秒，即 `SET PERSIST binlog_expire_logs_seconds = 259200;`。

### 4. 应用侧历史表与杂项

- **`rule_history` 等历史表自动清理**：已在 `server/services/cronService.js` 中注册 **每日 04:30** 任务；默认 `rule_history` 保留 **60 天**（可用 `HISTORY_RETENTION_DAYS_RULE` 等环境变量覆盖）；总开关 `ENABLE_HISTORY_CLEANUP` 设为 `0`/`false` 可关闭。单次补跑：`node server/scripts/run-history-cleanup.js`。
- **单次脚本执行结果（用户本机留痕）**：曾删除 `rule_matched_objects_history` 331 条，`structure_ads_history` / `rule_history` 为 0（说明当时无超过保留期的行，属正常）。
- **`oauth2-proxy` 容器**：曾处于反复重启状态；与当前钉钉登录实现路径无强依赖，用户侧已执行 `docker stop` + `docker rm` 去除干扰。

## 五、原理备忘：binlog 是什么？

- **Binary Log（二进制日志）**：MySQL 按顺序记录数据变更（复制、部分恢复场景依赖）。**只追加**，若不做过期与清理，在高写入业务下会迅速占满磁盘。
- **`binlog_expire_logs_seconds`**：控制「超过多少秒的 binlog 可被自动清理」；与业务表 `rule_history` 的保留策略是两套机制，勿混为一谈。

---

## 六、验证建议（本窗口收口检查清单）

1. `df -h /`：根分区 **Use%** 应远离 100%，并保留合理余量。
2. `docker exec mysql mysql -uroot -p... -e "SHOW MASTER STATUS; SHOW VARIABLES LIKE 'binlog_expire_logs_seconds';"`：确认当前 binlog 文件名与过期秒数符合预期。
3. 业务侧：**钉钉扫码登录**与关键写库路径在磁盘恢复后应明显恢复；若仍有 `state` 类错误，再回到 Cookie/域名/HTTPS 与 `.env.auth` 专项排查。
4. 定期关注：`docker system df`、`/var/lib/docker/volumes/*/ _data` 下 binlog 文件数量与单文件大小趋势。

## 七、后续优化方向（非本窗口必做）

- 若**无复制链路与长窗口按时间点恢复**需求，可继续缩短 binlog 保留或评估是否必须开启 binlog（需与 DBA/架构确认，避免影响既定备份与演练流程）。
- 对 `rule_history` 写入频率与保留天数做容量规划：在「审计需要」与「磁盘与清理任务耗时」之间取折中。
- 监控与告警：根分区使用率、MySQL 数据卷大小、binlog 总大小、容器日志体积。
