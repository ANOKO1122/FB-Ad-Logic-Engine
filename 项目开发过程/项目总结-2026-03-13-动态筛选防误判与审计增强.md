## 二、遇到的问题与解决方法

### 2.1 迁移 040 重复执行报错 Duplicate column name 'manual_count'

**现象**：执行 `ALTER TABLE rule_matched_objects_history ADD COLUMN manual_count ... ADD COLUMN dynamic_count ...` 时，报错 `ERROR 1060 (42S21): Duplicate column name 'manual_count'`。

**根因**：表内已存在 `manual_count` 列（可能此前执行过 040 或曾手动加列），再次执行同一 ALTER 导致重复列名。

**解决**：将迁移 040 改为**幂等**脚本：通过存储过程先查 `INFORMATION_SCHEMA.COLUMNS` 判断列是否存在，仅不存在时才执行 `ADD COLUMN`。这样在新环境首次执行会加列，在已加列的环境再次执行不会报错。

### 2.2 重算后历史表 manual_count / dynamic_count 仍为 NULL

**现象**：点击「重算动态范围」后，查询 `rule_matched_objects_history` 最新几行，`manual_count`、`dynamic_count` 均为 NULL；表结构已含两列。

**根因**：处理「重算」请求的 Node 进程仍是**改代码前的旧进程**，未加载到 `calculateMatchedAdIdsForRule` 返回 `dynamicCount`/`manualCount` 及 INSERT 传入两列的改动。

**解决**：重启后端服务（`npm run dev:server`），确保进程加载最新代码后再次点击重算；新插入的历史行中 `manual_count`、`dynamic_count` 正常有值。

### 2.3 同一规则、同一账户出现两条接近时间的历史记录

**现象**：同一 `rule_id`、`account_id` 在极短时间内出现两条 `refreshed_at` 相差约 40ms 的历史记录。

**说明**：非 Bug。历史表按「每次有变动的刷新」写一条；短时间内的两次刷新会产生两条记录。SQL 显示一条 `trigger_type = 'manual'`（用户点击重算）、一条 `trigger_type = 'rule_saved'`（保存规则后的 TriggerB 防抖刷新），符合「保存 + 重算」或「重算被触发两次」的预期。
