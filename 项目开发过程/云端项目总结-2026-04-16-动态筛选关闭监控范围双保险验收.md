# 云端项目总结-2026-04-16-动态筛选关闭监控范围双保险验收

## 一、背景与目标

- **问题背景**：规则编辑页允许在“动态筛选关闭”状态下继续填写监控范围条件，导致配置语义冲突，存在误关风险。
- **目标口径**：
  - 动态筛选关闭时，不允许使用监控范围条件。
  - 前端与后端均需做防护，避免旧前端或脚本绕过。

---

## 二、本次遇到的问题

### 2.1 前端改完后页面看起来“没生效”

- 现象：代码已改，但页面仍可编辑监控范围条件。
- 根因：仅重启了 `fb-ad-logic-engine.service`（后端），未重建前端静态资源与刷新 Nginx 容器。
- 处理：
  - 执行 `npm run build` 生成新 `dist`。
  - 重启 `fb-ad-nginx` 使其加载新静态包。
  - 浏览器执行强制刷新（Ctrl+F5）清缓存。

### 2.2 交互口径调整：从“可选择清空”改为“强制清空”

- 初版实现：关闭动态筛选时弹确认，可取消清空。
- 用户最终口径：只要关闭动态筛选，监控范围条件必须立刻清空，不允许保留历史内容。
- 处理：前端改为“关闭即清空”，并保持禁用态与提示文案。

### 2.3 抓包验证时看错请求

- 现象：Network 中看到 `GET /api/rules`，误以为未提交保存参数。
- 根因：查看的是列表加载请求，不是保存请求。
- 处理：
  - 指定查看 `POST /api/rules` 或 `PUT /api/rules/:id` 的 `Request Payload`。
  - 验证关键字段 `useDynamicScope`、`scopeFilters`。

### 2.4 后端校验验证时先命中账户校验

- 现象：第一次 curl 返回 `ACCOUNT_NOT_FOUND`。
- 根因：使用了占位账户 ID（`act_xxx`），在动态筛选校验前即被账户校验拦截。
- 处理：改为真实账户 ID 后，成功命中目标错误码验证路径。

## 三、实施内容与落点

### 3.1 前端（M1）

- 文件：`src/views/RuleManager.vue`
- 已完成：
  - `useDynamicScope=false` 时监控范围条件整块禁用（下拉/输入/添加/删除均不可操作）。
  - 增加提示文案：未开启动态筛选时条件不生效。
  - 关闭动态筛选时强制清空监控范围条件（不再保留取消分支）。
  - 保存时在关闭动态状态提交 `scopeFilters=null`。

### 3.2 后端（M2）

- 文件：`server/routes/rules.js`
- 已完成：
  - 新增组合校验：`useDynamicScope=false` 且 `scopeFilters` 非空时拒绝保存。
  - 固定返回：`400 + DYNAMIC_SCOPE_DISABLED_SCOPE_FILTERS_FORBIDDEN`。
  - 允许 `scopeFilters=null` 作为“清空条件”的合法输入。

### 3.3 历史治理策略（M3）

- 已提供只读审计 SQL（不自动修复）。
- 业务决策：历史规则不做批量清洗，交由运营在后续使用和编辑中逐步修正。
