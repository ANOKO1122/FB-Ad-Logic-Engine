## 一、本窗口遇到的问题与解决方法

| 问题 | 解决方法 |
|------|----------|
| ROAS 口径不统一 | 用户要求规则执行条件只依据我们通过原始数据计算得出的数值。在 `ruleDataService.calculateSingleDayMetrics` 中移除对 `dailyStats.roas`（DB/FB）的兜底，单日 ROAS 仅用 `purchase_value/spend`：spend=0→null，有花费有转化→value/spend，有花费无转化→0。 |
| 验证 SQL 需具体账户 | 提供了针对 act_927139705822379 的单日/多日 ROAS SQL、时区查询，以及针对 act_1501999950830277 的规则执行记录 SQL。 |
| MySQL 子查询 LIMIT 报错 | 错误 1235：IN (SELECT ... LIMIT 1) 不被支持。去掉子查询中的 LIMIT，改用 IN (SELECT ad_id FROM ad_snapshots WHERE account_id = 'act_xxx') 等。 |
| 验证结果确认 | 用户执行 SQL 后确认：单日/多日 our_roas 与手算一致；规则 461 在预算已达目标后持续 skipped（Pre-Flight 生效），无多余 API 调用。 |
| **echo noop / Shell 命令** | 开发过程中若出现 `echo noop`、`echo skipping-shell` 等 Shell 输出，为工具误调用产生的空操作（仅打一行字就结束），未连 MySQL、未改任何代码或文件。.cursorrules 约定 SQL 与执行命令由用户自行运行，代码修改仅通过文件读写工具完成。 |

## 二、项目进度与目标（本窗口相关）

- **目标**：规则条件中的 ROAS 只依赖自有数据（purchase_value/spend），与多日聚合口径一致，避免 FB roas 为空或口径差异导致误判。
- **本窗口完成**：ROAS 单日口径改为「一律本地计算」并落地；提供并修正验证 SQL；确认单日/多日 ROAS 与 Pre-Flight 行为正确；**并在 TASKS.md / DEV_PLAN.md 中补充 budget-cooldown-and-prefight 方案与双写细节**。
- **此前同对话窗口已完成**：**预算冷却与 Pre-Flight 重构**（依据 `.cursor/plans/budget-cooldown-and-prefight-refactor_e251157f.plan.md`）：
  - 数据层：迁移 031，`rule_ad_execution_state` 主键改为 (rule_id, scope_key)，ad_id 保留诊断。
  - ruleExecutionStateService：loadRuleAdExecutionState、isCooldownDue、upsertRuleAdExecutionStateBatch。
  - actionExecutorService：pause/activate 用 cooldownKey=ad:；预算 ABO/CBO 用 budget_adset:/budget_campaign:；当前预算=目标预算时 Pre-Flight skipped 不调 FB；results 返回 cooldownKey。
  - **cronService 双写**：修改前只按 cooldownKey 写入（预算时为 budget_*），调度层过滤按 "ad:"+adId 查冷却，查不到故冷却期内每分钟仍进执行层。修改后除写 cooldownKey 外，若 cooldownKey≠"ad:"+adId 则再写一条 ad: 冷却，使执行频率（execution_interval_minutes）对预算动作真正生效。
  - 验收：冷却表可见同一 rule_id 下 budget_* 与 ad: 两条；15 分钟内 Cron 不再重复执行。

## 四、验证方法与结果

- **单日 ROAS**：`ad_snapshots` + ROW_NUMBER 取每广告当日最新快照，`our_roas = CASE WHEN spend=0 THEN NULL WHEN purchase_value>0 THEN purchase_value/spend ELSE 0 END`。用户用 act_927139705822379 验证，结果与手算一致。
- **多日 ROAS**：`daily_stats` 按 ad_id 聚合，`our_roas_3d = total_purchase_value/total_spend`（同上 CASE）。验证通过。
- **规则执行记录**：`automation_logs` 按 ad_id IN (子查询无 LIMIT) 或 rule_id IN 查询；act_1501999950830277 规则 461 显示 success 后连续 skipped，符合 Pre-Flight 行为。
