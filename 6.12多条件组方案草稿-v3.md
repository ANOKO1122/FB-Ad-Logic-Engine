# 多条件组方案草稿（v3）

> 状态：方向已确认
> 审查日期：2026-06-12  
> 核心原则：**即使多条件组功能出错，现有单条件组规则必须能正常运行（向后兼容 100%）**
> 附加原则：**需保证当改造进行中时，多条件规则与原有规则共存，当前系统及所有现存单条件组规则不受任何影响**

**已确认：**
- 组间连接符：灵活 AND/OR
- 条件组上限：不限制
- 同指标不同时间窗口：会用到（如组1 spend(today)，组2 spend(last_7d)）
- 前端 UI：平铺 + 分隔线
- 多条件组执行日志：可复用当前逻辑（explanation JSON 灵活扩展）

---

## 一、需求分析

### 1.1 现状回顾：以规则 969 为例

规则 969（「新发广告·暂停·严格版」）在数据库中落库的 `conditions` 为 v2 DNF 结构：

```
version: 2
groups (共 10 个):
  Group 0: AND { spend>1, link_clicks=0, add_to_cart=0, checkout=0, purchases=0 }
  Group 1: AND { spend>2, 0<link_clicks≤1, add_to_cart=0, checkout=0, purchases=0 }
  Group 2: AND { spend>4, add_to_cart=0, checkout=0, purchases=0 }
  Group 3: AND { spend>8, 0<add_to_cart≤1, checkout=0, purchases=0 }
  Group 4: AND { spend>8, checkout=0, purchases=0 }
  Group 5: AND { spend>16, checkout≤1, purchases=0 }
  Group 6: AND { spend>27, purchases=0 }
  Group 7: AND { spend≥1, cpm≥100, add_to_cart=0, checkout=0, purchases=0 }
  Group 8: AND { spend>10, cpc>1, 加购费>4, 结账费>8, purchases=0, ... }

全部 group 共用 time_window: "lifetime"
评估规则：组内 AND，组间 OR（硬编码）
```

**关键发现：** 当前系统已经支持丰富的组内 AND/OR 逻辑（前端条件行间 AND → 同 DNF group，OR → 新 DNF group），但所有 group 只能共用一个 time_window。规则 969 用 10 个 group + lifetime 表达了「不同花费门槛 + 不同指标表现下触发暂停」。

**多条件组的价值：** 可以用不同时间维度做判断：
- 条件组1 (today)：花费>5 AND purchases=0 → 当天表现差就停
- 条件组2 (last_7_days)：花费>30 AND purchases≤1 → 一周表现差也停

两个条件组用 OR 连接 → 今天差或长期差任一满足即触发，比只用 lifetime 更精准。

### 1.2 目标

```
条件组1 [时间窗口: 今天]      花费>20 AND ROAS<0.5  OR  CPA>10
  AND
条件组2 [时间窗口: 近7天]     购买次数<3
  OR
条件组3 [时间窗口: 昨天]      CPA>10 AND 花费>5
```

**最终语义：** `[组1] AND [组2] OR [组3]`，每个条件组内部复用现有 AND/OR（v2 DNF 子组）。

## 二、v3 数据格式

### 2.1 结构设计：v3 = v2 DNF 的外层包装

```json
{
  "version": 3,
  "condition_groups": [
    {
      "join": null,
      "time_window": "today",
      "custom_range": null,
      "groups": [
        {
          "operator": "AND",
          "conditions": [
            {"metric": "spend", "operator": "gt", "value": 20},
            {"metric": "roas", "operator": "lt", "value": 0.5}
          ]
        },
        {
          "operator": "AND",
          "conditions": [
            {"metric": "cpa", "operator": "gt", "value": 10}
          ]
        }
      ]
    },
    {
      "join": "AND",
      "time_window": "last_7_days",
      "custom_range": null,
      "groups": [
        {
          "operator": "AND",
          "conditions": [
            {"metric": "purchases", "operator": "lt", "value": 3}
          ]
        }
      ]
    },
    {
      "join": "OR",
      "time_window": "yesterday",
      "custom_range": null,
      "groups": [
        {
          "operator": "AND",
          "conditions": [
            {"metric": "cpa", "operator": "gt", "value": 10},
            {"metric": "spend", "operator": "gt", "value": 5}
          ]
        }
      ]
    }
  ]
}
```

**层级关系：**
```
condition_groups[N]           ← 条件组数组（不限制数量）
  ├─ join                     ← 与前一组的连接符（首组 null，后续 "AND"|"OR"）
  ├─ time_window              ← 该组独立的时间窗口
  ├─ custom_range             ← 仅 time_window=custom_range 时有效
  └─ groups[M]                ← 内嵌 v2 DNF 子组
       ├─ operator: "AND"     ← 子组内条件固定 AND
       └─ conditions[]        ← {metric, operator, value}（不含 time_window）
```

### 2.2 与 v2 的差异

| 维度 | v2（当前） | v3（新增） |
|---|---|---|
| 版本 | `version: 2` | `version: 3` |
| 时间窗口 | 每 condition 冗余携带（全局一致） | condition_group 级别，各组独立 |
| condition 字段 | `{metric, operator, value, time_window}` | `{metric, operator, value}`（tw 提升到组级） |
| 组间连接 | 硬编码 OR | 每组 join 字段（null/"AND"/"OR"），与条件行间连接语义一致 |
| 内层 groups | `groups[]`（顶层数组） | `condition_groups[].groups[]`（嵌套在条件组内） |
| `validateTimeWindowConsistency` | 强制所有 condition tw 一致 | 跳过（不校验跨组） |

### 2.3 关键约定

- condition 不带 time_window → 和 v2 的 condition 结构不冲突，不会混淆
- v3 中的 `groups` 与 v2 中的 `groups` 语义完全相同（组内 AND，子组间 OR）
- 在前端保存时：单条件组 → 落 v2 格式（兼容）；多条件组 → 落 v3 格式

## 三、评估逻辑设计

### 3.1 核心思路：逐组独立评估，不合并数据

**不做跨 time_window 的数据合并**。原因：如果组1用 spend(today)，组2也用了 spend(last_7d)，合并后同名指标互相覆盖，评估出错。

**改为**：每个条件组用自己的 time_window 原始数据行独立评估，得出该组的布尔结果，最后按组的 `join` 字段累积合并。**与条件行间 AND/OR 的语义完全一致**——首组结果直接作为累积值，后续组按自己的 `join` 与前序累积结果合并。

```
evaluateRuleV3(rule, dataByTimeWindow):

  allAdIds = dataByTimeWindow 所有数据源的 ad_id 并集

  for adId in allAdIds:
    accumulatedResult = null    // 累积结果
    
    for (i, conditionGroup) in rule.conditions.condition_groups:
      twKey = conditionGroup.time_window + customRangeKey
      data = dataByTimeWindow.get(twKey) || []
      adRow = data.find(row => row.ad_id === adId)
      
      groupPassed = adRow
        ? evaluateV3ConditionGroup(conditionGroup, adRow)
        : false
      
      if (i === 0):
        accumulatedResult = groupPassed       // 首组：直接赋值
      else if conditionGroup.join === "AND":
        accumulatedResult = accumulatedResult && groupPassed
      else:  // "OR"
        accumulatedResult = accumulatedResult || groupPassed
    
    if accumulatedResult:
      matchedAds.push(...)
```

### 3.2 内嵌 v2 评估复用

**核心发现：v3 条件组的 `groups` 字段与 v2 的 `groups` 语义完全相同（组内 AND，子组间 OR）。** 因此不需要重写评估逻辑，直接复用 RuleEngine 已有的 `evaluateConditions()` 方法即可：

```js
// v3 条件组评估：直接复用 RuleEngine.evaluateConditions()
// conditionGroup.groups 就是 v2 DNF 的 groups 数组，语义完全一致
function evaluateV3ConditionGroup(conditionGroup, adRow) {
  // evaluateConditions 内部已做 normalizeConditionsToV2 → 组内AND组间OR
  return this.evaluateConditions(
    { version: 2, groups: conditionGroup.groups },
    adRow,
    'AND'
  )
}
```

**同时 `isConditionTraceMatched()` 也可直接复用**——v3 子组的 conditionTrace 与 v2 完全同构（都用 `groupIndex` 分组），不改一行即可判定子组是否通过：

```js
// v3 条件下，子组 trace 判定复用现有逻辑
const subTrace = this.buildConditionTrace(
  { version: 2, groups: conditionGroup.groups },
  adRow,
  'AND'
)
const groupPassed = this.isConditionTraceMatched(subTrace)
```

调用链不经过 `normalizeConditionsToV2()`（v3 直接构造 v2 格式传入），全程使用现有的 `evaluateCondition()`、`getMetricValue()`、`evaluateConditions()`、`isConditionTraceMatched()`。

### 3.3 关键优势

1. **零合并歧义**：每个组用独立数据源，不存在指标覆盖
2. **复用现有代码（高复用度 ~80%）**：
   - `evaluateCondition()` / `getMetricValue()` → 完全不动
   - `evaluateConditions()` → 直接用于 v3 子组评估（不重写）
   - `isConditionTraceMatched()` → 直接用于 v3 子组判定（不重写）
   - `buildConditionTrace()` → v3 重载可复用 v2 内层逻辑，逐 conditionGroup 生成 trace + 加外层标记
   - `evaluateRuleWithData()` 中 matchedAds 构建逻辑 → v1/v2/v3 完全一致（P0 已提取 `_buildMatchedAd()` 公共函数）
3. **审计清晰**：`buildConditionTrace()` 自然加上 `conditionGroupIndex` 和 `time_window` 标记
4. **性能无退化**：每个 time_window 只查一次 DB（跨规则去重），查询次数 = 去重后的 time_window 种类数

> 🆕 **审查补充**：`evaluateRule()`（`server/index.js:2147`）有独立的手动执行路径，不走 Dispatcher。v3 改造后在入口处增加显式检测——若 `conditions.version === 3` 则**抛错提示使用 Dispatcher 路径**，而不是静默走单 tw 查询。

> 🆕 **审查补充**：`getCustomRangeFromConditions()`（`server/index.js:2477`）内部调 `normalizeConditionsToV2`，v3 下会返回 null。v3 适配方案：Dispatcher 中按 `condition_groups[i].custom_range` 直接读取，不通过此函数；或在该函数中增加 v3 分支返回 `Map<conditionGroupIndex, customRange>`。

## 四、数据加载改动

### 4.1 Dispatcher 当前模型

```js
// 当前：每条规则 1 个 cacheKey
cacheKey = `ad:today`              // targetLevel:timeWindow
cacheKeysByRule = Map<ruleId, cacheKey>   // 1 对 1

evaluateRuleWithCache() {
  const cacheKey = cacheKeysByRule.get(rule.id)
  const data = cache.get(cacheKey) || []   // 取 1 份数据
  return ruleEngine.evaluateRuleWithData(rule, data)
}
```

### 4.2 Dispatcher v3 扩展

> 🆕 **审查修正**：原方案"统一 `cacheKeysByRule` 为数组"会破坏 `evaluateRuleWithCache()` 中 `cacheKey` 的读取（`dispatcher.js:260` 期待 string）。**改用独立 Map 隔离**，v1/v2 路径完全不碰。

**新增 `cacheKeysByRuleV3` Map（仅 v3 规则使用）：**

```js
// loadDataForAccount 中：v1/v2 保持原样，v3 规则写入独立 Map
const cacheKeysByRule = new Map()      // ruleId -> string  (v1/v2 不变)
const cacheKeysByRuleV3 = new Map()    // ruleId -> string[] (v3 only，新增)

// v3 规则 → 多个 key
const keys = []
for (const cg of rule.conditions.condition_groups) {
  const cacheKey = `${targetLevel}:${cg.time_window}` + customRangeStr
  keys.push(cacheKey)
}
cacheKeysByRuleV3.set(ruleId, keys)    // ← 写入独立 Map，不污染 v1/v2

// evaluateRuleWithCache 中：按 version 分发
const cacheKeyV3 = loadResult.cacheKeysByRuleV3?.get(rule.id)  // v3 专属
if (cacheKeyV3 != null) {
  // v3 路径：多 tw 数据源，逐 key 过滤 targetIds
  // ...
}
const cacheKey = loadResult.cacheKeysByRule?.get(rule.id)      // v1/v2 原逻辑
// ...
```

### 4.3 数据查询量估算

| 场景 | DB 查询增量 |
|---|---|
| 同账户内所有规则都是单条件组 | 0（与现在完全一样） |
| 1 条 v3 规则，N 组不同 tw | +(N-1) 次查询 |

**`flattenV3Conditions()` 展平约定**（用于卡片展示 + 折叠/展开逻辑）：
```js
// 输入：condition_groups[]（v3 外层数组）
// 输出：扁平 condition[]，每条附加：
//   _conditionGroupIndex: 1-based 条件组索引
//   _groupIndex: 1-based 子组索引（同 v2）
//   _timeWindow: 该条件所属的 time_window（字符串，如 "today"）
function flattenV3Conditions(conditionGroups) {
  const list = []
  for (let cgIdx = 0; cgIdx < conditionGroups.length; cgIdx++) {
    const cg = conditionGroups[cgIdx]
    for (let gIdx = 0; gIdx < (cg.groups || []).length; gIdx++) {
      for (const c of cg.groups[gIdx].conditions || []) {
        list.push({
          ...c,
          _conditionGroupIndex: cgIdx + 1,
          _groupIndex: gIdx + 1,
          _timeWindow: cg.time_window || 'today'
        })
      }
    }
  }
  return list
}
```

**卡片展示适配**（现有 v2 函数扩展为感知 `_conditionGroupIndex`）：
```js
// visibleConditionsForCard: v3 折叠态只显示第一个条件组的条件

  const conds = r.conditions || []
  if (isRuleCardExpanded(r.id)) return conds
  if (r.conditionsVersion === 3) return conds.filter(c => (c._conditionGroupIndex || 1) === 1)
  if (r.conditionsVersion === 2) return conds.filter(c => (c._groupIndex || 1) === 1)
  return conds
}

// extraCondGroupsCount: v3 需要同时统计条件组数和子组数
const extraCondGroupsCount = (r) => {
  if (r.conditionsVersion === 3) {
    let maxCg = 1, maxG = 1
    for (const c of r.conditions) {
      maxCg = Math.max(maxCg, c._conditionGroupIndex || 1)
      maxG = Math.max(maxG, c._groupIndex || 1)
    }
    // 有多条件组时优先显示条件组数量
    if (maxCg > 1) return maxCg - 1  // "还有 X 组条件"
    return Math.max(0, maxG - 1)      // 单组内还有 X 组子条件
  }
  // v2 原逻辑
  if (r.conditionsVersion !== 2 || !r.conditions?.length) return 0
  let maxG = 1
  for (const c of r.conditions) {
    const g = c._groupIndex || 1
    if (g > maxG) maxG = g
  }
  return Math.max(0, maxG - 1)
}
```

### 6.4 新增/删除条件组逻辑

- 「添加条件组」新增到末尾，默认 time_window=today，1 条空条件行
- 「删除条件组」至少保留 1 组（= 当前行为），最后 1 组删除按钮置灰
- 组间连接符下拉在 2 组以上时出现在两组之间
- 不限制条件组数量，但建议前端在组数过多时显示滚动区域

### 8.1 当前执行日志机制

每次规则执行命中后，`cronService.js` 中的 `writeBatchStatusAuditLog()` 会调用 `buildAutomationLogExplanation()` 构建可解释审计 JSON，写入 `automation_logs.explanation` 字段：

### 8.2 v3 需要扩展的字段

| 现有字段 | v3 扩展 | 说明 |
|---|---|---|
| `window` (单对象) | 保留在 `conditionTrace[].time_window` 各条目中 | 多 tw 时每个条件属于不同窗口，不需要顶层 `window` |
| `conditionTrace[].groupIndex` | 增加 `conditionGroupIndex` | 标记条件属于哪个条件组 |
| `conditionTrace[].time_window` | 新增字段 | 标记该条件使用的具体时间窗口 |
| `logic.logicOperator` | 保留（向后兼容） | v3 中可能无意义，但保留不破坏 |
| `logic` 对象 | 增加 `groupResults[]`（每组的 join 和结果） | 展示每个条件组的布尔评估结果 + 连接方式 |

v3 的 explanation JSON 示例：
```json

  "target": { ... },
  "logic": {
    "logicOperator": "AND",
    "groupResults": [
      { "conditionGroupIndex": 0, "join": null, "time_window": "today", "passed": true },
      { "conditionGroupIndex": 1, "join": "AND", "time_window": "last_7_days", "passed": false },

      "metric": "purchases", "operator": "lt", "threshold": 3, "actual": 5, "passed": false }
  ]
}
```

### 8.3 复用程度评估

**可直接复用的：**
- `automation_logs` 表结构 → `explanation` 是 JSON 列，扩展字段不破坏旧记录
- `buildAutomationLogExplanation()` → v1/v2 路径完全不动
- `writeBatchStatusAuditLog()` → 只负责写入，不关心 explanation 内部结构
- `evaluateCondition()` / `getMetricValue()` → 条件级别评估不变

**需要新增的（v3 专属，对 v1/v2 零影响）：**
- `buildConditionTrace()` 增加 v3 版本，每条 trace 带上 `conditionGroupIndex` 和 `time_window`
- `buildAutomationLogExplanation()` 新增 v3 参数：`conditionGroups` 信息（各组 join + time_window）
- 返回的 explanation JSON 中 `logic` 扩展 `groupResults`（每组 join + 结果 + time_window）

> 🆕 **审查修正**：当前 `logic.matched` 使用 `conditionTrace.every(c => c?.passed !== false)`（`automationLogExplanation.js:70`），这在 v2 多组 DNF 场景下语义不正确（组间 OR 被误判为全局 AND）。v3 改造时一并修正：`logic.matched` 改为使用实际评估的 `groupResults` 级联结果。

**结论：执行日志逻辑可复用，改动是纯扩展（新增 v3 分支），v1/v2 日志完全不受影响。**
