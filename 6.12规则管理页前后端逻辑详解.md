## 一、系统架构总览

规则管理系统的核心职责是：**让运营人员定义「当广告指标满足什么条件时，自动执行什么操作」**。整个系统分为三个层次：

1. **前端 Vue 3 页面** — 规则 CRUD 界面，编辑条件/动作/对象
2. **后端 REST API** — 鉴权、校验、落库、触发动态刷新
3. **定时调度 + 执行引擎** — 按频率拉取数据、评估条件、执行动作

| `dynamic_scope_error_msg` | VARCHAR(255) | 动态筛选错误信息 |

| `execution_time_windows` | JSON | 允许执行时间段（北京时间），如 `[{start:"09:00:00", end:"18:00:00"}]` |
| `last_executed_at` | TIMESTAMP | 上次执行时间 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 更新时间 |

### 2.2 动态范围快照表 `rule_matched_objects`

| 字段 | 类型 | 含义 |
|---|---|---|
| `id` | INT PK | 自增主键 |
| `rule_id` | INT NOT NULL | 规则 ID |
| `account_id` | VARCHAR(50) | 账户 ID |
| `object_id` | VARCHAR(50) | 对象 ID（ad_id / adset_id / campaign_id） |
| `object_type` | VARCHAR(16) | 对象类型：ad / adset / campaign |
| `created_at` | TIMESTAMP | 创建时间 |

唯一键升级为 `(rule_id, account_id, object_type, object_id)`（M2 typed snapshot）。

### 2.3 动态范围历史表 `rule_matched_objects_history`

存储每次动态刷新时的 `manual_count`、`dynamic_count`、`refreshed_at`、`trigger_type`，用于前端展示「动态 X / 手动 Y」和趋势。

### 2.4 执行状态表 `rule_ad_execution_state`

| 字段 | 类型 | 含义 |
|---|---|---|
| `rule_id` | INT PK | 规则 ID |
| `scope_key` | VARCHAR(100) PK | 冷却键：`ad:xxx` / `status_ad:xxx` / `budget_adset:xxx` / `budget_campaign:xxx` |
| `last_executed_at` | TIMESTAMP | 上次执行时间 |
| `last_status` | ENUM | success / fail / suppressed / outside_window |
| `ad_id` | VARCHAR(50) | 广告 ID（兼容字段） |

主键为 `(rule_id, scope_key)`，冷却期由 `TIMESTAMPDIFF(MINUTE, last_executed_at, UTC_TIMESTAMP()) >= interval` 判定。

### 2.5 执行摘要表 `rule_execution_summaries`

记录每次规则评估的摘要：`run_id`、`rule_id`、`account_id`、`matched_count`、`executed_count`、`failed_count`、`skipped_count`、`status`、`skip_reason`、`skip_details`、`duration_ms`。

### 2.6 审计日志表 `automation_logs`

记录每次动作执行的详细信息：`run_id`、`ad_id`、`object_type`、`object_id`、`metrics_snapshot`、`explanation`（可解释审计 JSON）、`action_type`、`action_payload`、`api_request`、`api_response`、`status`（success/fail/skipped）、`is_simulation`。

### 2.7 规则变更历史表 `rule_history`

记录每次人为操作（CREATE/UPDATE/TOGGLE/DELETE）的规则配置快照（仅 SNAPSHOT_KEYS 定义的 18 个配置字段，严禁运行时统计数据）。

### 2.8 数据源表

| 表 | 用途 |
|---|---|
| `ad_snapshots` | 热数据：今日广告快照，每 10 分钟更新 |
| `daily_stats` | 冷数据：每日汇总，每天 06:00 归档 |
| `daily_archive_status` | 归档状态注册表 |
| `structure_ads` | 广告结构镜像（ad_id → adset_id → campaign_id 映射） |
| `structure_adsets` | 广告组结构镜像 |
| `structure_campaigns` | 广告系列结构镜像 |
| `structure_sync_status` | 结构同步状态（含 `fast_dirty` 标记） |

| `users` | 用户表（旧表，只读） |
| `owners` | 负责人表（旧表，只读） |
| `rule_templates` | 规则模板（铺底用） |
| `scheduled_tasks` | 定时任务表（独立调度系统） |

## 三、前端逻辑详解

### 3.1 规则列表页 (`RuleManager.vue`)

#### 3.1.1 页面布局

- **顶部免责 Banner**：提醒用户在 Facebook 后台手动开启广告前先暂停相关规则
- **头部操作栏**：
  - 负责人筛选（仅管理员可见，多选下拉 + "管理员创建" 选项）
  - 显示/隐藏已关闭规则切换
  - 全部折叠/全部展开按钮
  - 批量添加账户模式入口
  - 新建规则按钮
- **规则卡片网格**：每个规则一张卡片，包含：
  - 规则名称（v2 DNF 规则有 "DNF/多组" 标签）
  - 启用/禁用开关
  - 作用账户、负责人、动态筛选状态
  - IF 条件（折叠态仅显示第一组）
  - THEN 动作
  - 执行频率、执行时间段
  - 操作按钮：重算动态范围、运行此规则、编辑、删除
- **空状态**：暂无一键生成模板规则按钮（仅非管理员且规则为空时显示）

#### 3.1.2 新建/编辑规则模态框 — 五大区块

**区块 1：基本信息**
- 规则名称（文本输入）
- 模拟运行开关 (Dry Run)：勾选后仅写日志，不调用 Facebook API

**区块 2：作用对象 (Where)**
- 广告账户多选：下拉添加、全选、清空、刷新列表
- 目标层级：广告 (ad) / 广告组 (adset) / 广告系列 (campaign)
- 动态筛选开关：
  - **开启动态筛选** → 填写监控范围条件，系统自动计算匹配对象 → 显示当前匹配数，目标列表不可手动勾选
  - **关闭动态筛选** → 手动搜索/勾选目标对象，支持全选/清空/加载更多
- 监控范围条件（动态筛选开启时有效，最多 3 行，AND 关系）：
  - 名称：包含 / 不包含
  - 状态：等于 / 不等于（投放中/已暂停/全部）
  - 创建时间段：近24/48/72小时 / 自定义小时内 / XX小时以前 / 介于 XX~YY 小时之间
- 排除名单：独立搜索区，支持多选对象从最终范围中剔除
- 预览动态范围按钮：调用 `/api/rules/preview-dynamic-scope` 获取与规则执行一致的 ID 列表

**区块 3：触发条件 (When)**
- 快速应用模板按钮（如「空耗关停」「ROAS 止损」等）
- 比值类指标口径提示：CPC/CPA/ROAS 等需要分母 > 0 才有意义，自动检查并提示
- 全局时间窗口（所有条件共用）：
  - `today` — 今天
  - `yesterday` — 昨天
  - `last_3_days` — 近3天（含今天）
  - `last_3_days_excluding_today` — 近3天（不含今天）
  - `last_5_days` — 近5天
  - `last_5_days_excluding_today` — 近5天（不含今天）
  - `last_7_days` — 近7天
  - `last_7_days_excluding_today` — 近7天（不含今天）
  - `lifetime` — 至今为止
  - `custom_range` — 自定义范围（since/until 日期选择器）
- 线性条件列表：
  - 每行：逻辑连接符(AND/OR) + 指标 + 运算符(>/</≥/≤/=) + 阈值
  - 可选指标：花费、ROAS、CPA、CPC、CPM、加购费、结账费、支付费、购买次数、链接点击、加购次数、结账次数、添加支付信息次数、（仅 ad 层级）多天购买次数平均数
  - 可动态增删行

**区块 4：执行动作 (Then)**
- 动作类型：
  - `pause_ad` — 暂停目标
  - `activate_ad` — 启用目标
  - `increase_budget` — 增加预算（百分比或固定美元，可选上限）
  - `decrease_budget` — 减少预算（百分比或固定美元，可选下限）
  - `set_budget` — 设置预算为固定值
  - `set_dynamic_budget` — 设置动态预算值（公式：指标 × 倍率，可选上下限，可配置「当前更高时不调低」「当前更低时不调高」）
- CBO 提示：若目标层级非 ad，增减预算影响整个系列
- 预算护栏：上下限单位为「美分」（100 美分 = 1 美元），前端美元输入 ÷100 转换

**区块 5：执行频率与执行时间**
- 执行频率：15分钟 / 30分钟 / 1小时 / 3小时 / 6小时 / 自定义小时数
- 允许执行时间（北京时间）：
  - 全天
  - 指定时段（支持跨日，如 20:00–05:00 表示当晚到次日凌晨）
  - 使用 `TimePicker24` 组件

#### 3.1.3 核心交互流程

**保存规则数据流：**
1. 前端校验（名称、账户、动作、条件、时间窗口等）
2. 将 `whenLines` 转为 v2 DNF 结构（`linesToV2Groups`）
3. 构建 `target_by_account`（复合 ID `act_xxx:id`）
4. 构建 `scopeFilters`（监控范围条件序列化）
5. 构建 `excludeIds`（排除名单序列化）
6. `POST /api/rules`（新建）或 `PUT /api/rules/:id`（编辑）
7. 后端保存后异步触发动态快照刷新（Trigger B）
8. 重新加载规则列表

**手动执行规则：**
1. 调用 `facebookApi.executeRuleById(ruleId)`
2. 后端走 `POST /api/rules/execute` → `cronService.executeSingleRule`
3. 前端展示结果：命中数、执行成功数、失败数、Run ID

**批量添加账户：**
1. 勾选多条规则 → 进入批量模式 → 打开批量弹窗
2. 选择要追加的广告账户
3. `POST /api/rules/batch/add-accounts`（best-effort 逐条处理）
4. 展示汇总结果：成功/未变/失败/待结构同步

### 3.2 审计页面 (`AdminRuleHistory.vue`)

- 筛选条件：规则 ID、操作人、变更类型（CREATE/UPDATE/TOGGLE/DELETE）、日期范围
- 日期预设：今天、昨天、最近7天、30天、3个月
- 列表展示：时间、规则 ID、类型标签、规则名、操作人
- 变更详情弹窗：双栏对照（修改前/修改后），字段级差异展示，支持：
  - 扁平字段差异（flat）
  - 排除名单层级差异（exclude）
  - 按账户分组目标差异（target_by_account）

### 4.5 关键校验逻辑

**创建规则时的校验链：**
1. `ruleName`、`conditions`、`actions` 必填
2. `accountId` 必填（防止反向索引退化）
3. `validateActions(actions, targetLevel)` — 动作合法性
4. `validateConditionsStructure(conditions)` — 条件结构（v1/v2）
5. `validateTimeWindowConsistency(normalizedForTw)` — 时间窗口一致性
6. `validateDynamicScopePayload(...)` — 动态筛选参数
7. `assertManualScopeHasTargets(...)` — 关闭动态时至少有一个手动目标
8. 权限校验：普通用户 accountId 须属于其负责人且激活；管理员 accountId 须在 account_mappings 中存在且激活
9. 多账户权限校验：targetAccounts 中每个账户均需权限校验

**更新规则时的额外校验：**
1. 白名单字段过滤（只允许更新 18 个指定字段）
2. 若更新 `accountId`，重新做权限校验
3. `mergeRuleForEnableCheck` → `assertRuleReadyToEnable`：若最终 enabled=true，校验最小配置完整性（账户绑定、条件、动作、动态筛选功能开关等）

## 五、规则条件判断详解

### 5.1 条件数据模型

规则条件支持两种格式：

**v1 格式（线性数组 + logicOperator）：**
```json
{
  "conditions": [
    {"metric": "spend", "operator": "gt", "value": 20, "time_window": "today"},
    {"metric": "roas", "operator": "lt", "value": 0.5, "time_window": "today"}
  ],
  "logicOperator": "AND"
}
```

**v2 格式（DNF：OR of AND Groups）：**
```json
{
  "version": 2,
  "groups": [
    {
      "operator": "AND",
      "conditions": [
        {"metric": "spend", "operator": "gt", "value": 20, "time_window": "today"},
        {"metric": "cpa", "operator": "gt", "value": 5, "time_window": "today"}
      ]
    },
    {
      "operator": "AND",
      "conditions": [
        {"metric": "purchases", "operator": "lt", "value": 1, "time_window": "last_3_days"}
      ]
    }
  ]
}
```

**评估语义：**
- 组内 AND：组内所有条件满足 → 该组为 true
- 组间 OR：任意一组为 true → 整体满足
- 同规则内所有条件必须使用相同的 `time_window`

### 5.2 条件评估核心逻辑 (`RuleEngine`)

`RuleEngine` 类位于 `server/index.js`，核心方法：

1. **`evaluateRule(rule, accountId)`**：完整评估流程（含 DB 查询）
   - 获取账户时区
   - 解析目标广告 ID（从 `target_by_account` 或 `targetIds`）
   - 通过 `queryRuleData` 获取指标数据
   - 逐广告调用 `buildConditionTrace` + `isConditionTraceMatched`

2. **`evaluateRuleWithData(rule, dataArray)`**：使用预加载数据评估（供 Dispatcher 批量优化）

3. **`buildConditionTrace(conditions, adData, logicOp)`**：构建条件命中链路
   - 遍历所有条件，计算每个条件的实际值和判定结果
   - 返回 `[{metric, operator, threshold, actualValue, met, timeWindow, customRange}]` 数组

4. **`isConditionTraceMatched(trace)`**：DNF 判定
   - 若 v1 AND：所有条件 met 均为 true
   - 若 v1 OR：任一条件 met 为 true
   - 若 v2 DNF：(组内 AND) OR (组内 AND) ...

### 5.3 指标取值来源

指标数据由 `ruleDataService.queryRuleData` 提供处理后的数据行，每行包含以下字段：

| 字段 | 来源 |
|---|---|
| `ad_id`, `ad_name`, `status` | ad_snapshots / daily_stats |
| `spend` | 花费（美元，decimal） |
| `purchases` | 购买次数 |
| `cpc` | 单次链接点击费用 |
| `ucpc` | 独立单次链接点击费用 |
| `roas` | 广告支出回报率（本地 purchase_value/spend 计算） |
| `cpa` | 单次购买成本 |
| `add_to_cart_cost`, `checkout_cost`, `payment_cost` | 各阶段单次成本 |
| `link_clicks`, `unique_link_clicks` | 点击相关计数 |
| `add_to_cart_count`, `initiate_checkout_count`, `add_payment_info_count` | 各阶段事件计数 |
| `purchases_avg_after_create` | 多天购买次数平均数（按首次花费日期计算有效天数） |
| `object_type`, `object_id`, `object_name` | 目标对象类型/ID/名称（M5 通用字段） |

### 5.4 评估结果 (matchedAds)

每条匹配的广告包含：`ad_id`、`ad_name`、`ad_set_id`、`campaign_id`、所有指标、`conditionTrace`（条件命中链路）、`objectType`、`objectId`、`objectName`。这些数据直接传递给 `actionExecutorService` 执行动作。

## 六、时间窗口详解

### 6.1 时间窗口类型

定义在 `server/utils/timeWindow.js` 的 `calculateTimeWindow` 函数中，基于账户时区（Luxon）计算：

| 窗口类型 | 计算逻辑 | 数据源 |
|---|---|---|
| `today` | 当天 00:00:00 ~ 23:59:59.999 | ad_snapshots（热数据） |
| `yesterday` | 昨天 00:00:00 ~ 23:59:59.999 | daily_stats（06:00 前降级 ad_snapshots） |
| `last_3_days` | 2天前 00:00 ~ 今天 23:59（含今天，共3天） | daily_stats |
| `last_3_days_excluding_today` | 3天前 00:00 ~ 昨天 23:59 | daily_stats |
| `last_5_days` | 4天前 00:00 ~ 今天 23:59（含今天，共5天） | daily_stats |
| `last_5_days_excluding_today` | 5天前 00:00 ~ 昨天 23:59 | daily_stats |
| `last_7_days` | 6天前 00:00 ~ 今天 23:59（含今天，共7天） | daily_stats |
| `last_7_days_excluding_today` | 7天前 00:00 ~ 昨天 23:59 | daily_stats |
| `last_30_days` | 29天前 00:00 ~ 今天 23:59 | daily_stats |
| `lifetime` | epoch ~ 今天 23:59（实际起点按广告 created_time 覆盖） | daily_stats |
| `custom_range` | since ~ until（验证：since ≤ until，跨度 ≤ 365 天） | daily_stats（≤1天时 ad_snapshots） |

### 6.2 智能数据源路由 (`ruleDataService.js`)

`selectDataSource(timeWindow, timezoneName)` 函数根据时间窗口智能选择数据源：

- **Today** → `ad_snapshots`（热数据，实时更新，毫秒级响应）
- **Yesterday** → `daily_stats`（冷数据），但如果当前时间在账户时区的 06:00 之前（数据尚未归档），降级到 `ad_snapshots` 查昨日最后快照
- **多天窗口** → `daily_stats`（需要动态聚合重算比率类指标，避免辛普森悖论）
- **lifetime** → `daily_stats` + 按广告 `created_time` 覆盖起点
- **custom_range** → 跨度 ≤ 1 天用 `ad_snapshots`，大于 1 天用 `daily_stats`

### 6.3 时间窗口一致性校验

在 `conditionsValidator.js` 的 `validateTimeWindowConsistency` 中，同规则内所有条件（包括跨 DNF 组）的 `time_window` 必须一致，`custom_range` 的 since/until 也必须一致。这是全局时间窗口的设计约束。

## 七、对象选择与动态范围详解

### 7.1 目标对象选择路径

规则的作用对象可以由三种方式确定，最终取**并集减去排除**：

1. **手动选择** → `target_ids`（复合 ID 数组） + `target_by_account`（按账户分组）
2. **动态筛选** → `rule_matched_objects` 表（由后台定时/事件触发刷新）
3. **排除名单** → `exclude_ids`（分 ad/adset/campaign 三层）

**最终公式：** `(手动 ∪ 动态) − 排除`

### 7.2 动态筛选完整流程

#### 7.2.1 监控范围条件 (`scope_filters`)

前端三个监控范围控件序列化为：
```json
{
  "level": "ad",
  "conditions": [
    {"field": "name", "operator": "contains", "value": "优化师"},
    {"field": "effective_status", "operator": "in", "value": ["ACTIVE"]},
    {"field": "created_time", "operator": "within_hours", "value": 72}
  ]
}
```

支持的操作符：
- `name`：`contains` / `not_contains`
- `effective_status`：`in` / `not_in`（值：`["ACTIVE"]` / `["PAUSED"]` / `["ACTIVE","PAUSED"]`）
- `created_time`：`within_hours`（N小时内）/ `older_than_hours`（N小时以前）/ `between_hours`（介于X~Y小时之间）

#### 7.2.2 快照刷新 (`dynamicScopeService.js`)

核心函数 `refreshDynamicTargetsForAccount(accountId, rule)` 的执行步骤：

1. 从 `scope_filters` 构建 SQL WHERE 条件，查 `structure_ads` / `structure_adsets` / `structure_campaigns`
2. 展开手动目标 ID：非 ad 层级时通过 `structure_ads` 将 campaign_id/adset_id 展开为 ad_id，去重合并
3. 展开排除 ID：同理将 campaign/adset 层级的排除展开为 ad 级别
4. **最终 Ad IDs = (动态匹配 ads) ∪ (手动 ads) − (排除 ads)**
5. 按规则 `targetLevel` 反向映射为对应层级的目标对象 ID
6. 检查上限：超过 `max_dynamic_matches` → 状态标记为 `ERROR_OVERSIZE`，**不写新快照**（保留上一版，fail-safe）
7. 按 chunk（默认 300 条/批）REPLACE INTO `rule_matched_objects`
8. 更新 `rules.dynamic_scope_status`、`dynamic_scope_updated_at`
9. 写入 `rule_matched_objects_history`（含 manual_count、dynamic_count）

#### 7.2.3 触发时机 (Trigger B)

- **规则保存后**：`scheduleDynamicScopeRefreshForRule(ruleId, {trigger: 'rule_saved'})`
- **手动点击「重算动态范围」**：调用 `POST /api/rules/refresh-dynamic-scope`
- **批量添加账户后**：结构数据新鲜时异步刷新
- **结构同步后**：通过 dirty 标记间接触发（Cron 扫描时 `refreshStructureIfDirtyBeforeRules`）

防抖机制：同一规则 ID 的刷新请求在 `TRIGGER_B_DEBOUNCE_MS`（默认 15 秒）内合并。

### 7.3 对象展开（层级翻译）

当 `targetLevel` 为 `adset` 或 `campaign` 时，通过 `structure_ads` 表将上层对象 ID 展开为子广告 ID：
- `adset` → `SELECT DISTINCT ad_id FROM structure_ads WHERE adset_id IN (...)`
- `campaign` → `SELECT DISTINCT ad_id FROM structure_ads WHERE campaign_id IN (...)`

这确保了即使规则作用于广告组或广告系列层级，最终的数据查询和条件评估仍然在广告粒度进行（V2 环境变量控制可选同层聚合评估）。

## 八、规则执行全链路

### 8.1 定时调度 (`cronService.js`)

**主循环（AdsPolar 流水线架构）：**

1. **数据同步**：
   - Today 快照（每 10 分钟）
   - 滑动窗口归档（每 15 分钟）
   - 结构全量轮转（每小时）+ Fast Sync（事件驱动）
   - 统一心跳同步

2. **Dirty 预检查**：若 `structure_sync_status.fast_dirty=1`，先在规则评估前执行一次 fast sync

3. **规则执行** (`executeRulesForAllAccounts`)：
   - 查询所有 `enabled=true` 的规则及其关联账户
   - 逐账户并发执行（`p-limit` 控制并发）

4. **执行时间窗检查** (`isInExecutionWindow`)：
   - 基于北京时间（Asia/Shanghai）
   - 支持跨日窗口（如 20:00–次日 05:00）
   - 空/NULL/非数组视为全天允许

5. **账户级 MySQL 锁** (`GET_LOCK`)：
   - 锁名：`rule:account:{accountId}`
   - 0 秒等待（拿不到立即跳过）
   - 5 分钟超时保险丝（`connection.destroy()` 强制释放）

### 8.2 单账户规则执行流程

```
executeRulesForAccount(accountId)
  │
  ├─ 1. 获取该账户下所有 enabled 规则
  │
  ├─ 2. RuleEngineDispatcher.loadDataForAccount(accountId, rules)
  │     ├─ 收集所有规则的 (timeWindow, customRange) 去重
  │     ├─ 按去重后的缓存键分别调用 queryRuleData / queryRuleDataByLevel
  │     ├─ 解析每条规则的目标对象 ID（动态快照 或 手动配置）
  │     └─ 返回 { timezoneName, cache, targetObjectIdsByRuleId, targetAdIdsByRuleId, cacheKeysByRule }
  │
  ├─ 3. 逐规则评估：evaluateRuleWithCache(ruleEngine, rule, loadResult)
  │     ├─ 从缓存取数据 → 按目标 ID 过滤 → 调用 evaluateRuleWithData
  │     └─ 返回 matchedAds []
  │
  ├─ 4. 冷却期检查：loadRuleAdExecutionState + isCooldownDue
  │     ├─ 冷却键 scope_key = ad:{adId} | status_ad:{adId} | budget_adset:{adsetId} | budget_campaign:{campaignId}
  │     ├─ 冷却时长 = rule.executionIntervalMinutes
  │     └─ 跳过仍在冷却期的匹配对象
  │
  ├─ 5. 动作执行：executeActionsForRule(rule, accountId, matchedAds, runId)
  │     ├─ Pre-Flight 状态刷新（批量拉取 FB effective_status）
  │     ├─ 按规则 actions 数组选出「最狠的一条」(pickSingleCandidateAction)
  │     ├─ 执行动作（调用 Facebook Marketing API）
  │     ├─ 预算护栏：最小 1 美元（100 美分），set_dynamic_budget 上下限
  │     ├─ 写入 automation_logs（含 explanation 可解释审计）
  │     └─ 更新冷却状态 upsertRuleAdExecutionStateBatch
  │
  └─ 6. 写入执行摘要 insertRuleExecutionSummary
```

### 8.3 动作优先级与仲裁

定义在 `actionPriority.js`，数字越小越优先：

| 动作 | 优先级 |
|---|---|
| `pause_ad` | 1（最高） |
| `activate_ad` | 2 |
| `set_dynamic_budget` | 2.5 |
| `decrease_budget` | 3 |
| `increase_budget` | 4 |
| `set_budget` | 5（最低） |

**仲裁规则：**
- 每条规则对每个广告只出一个候选动作（最优先的）
- 同广告多规则命中时，同优先级以 ruleId 最小者为赢家
- 状态动作（pause/activate）按 `targetLevel` 解释执行意图：
  - `targetLevel=adset` 且动作为 `pause_ad` → 实际暂停广告组（`pauseAdset`）
  - `targetLevel=campaign` → 实际暂停/启用广告系列

### 8.4 冷却期机制

冷却期以 `(rule_id, scope_key)` 为粒度：
- `ad:123` — 广告级冷却
- `status_ad:123` — 状态动作冷却
- `budget_adset:456` — 广告组预算冷却
- `budget_campaign:789` — 广告系列预算冷却

冷却时长 = `rules.execution_interval_minutes`（默认 15 分钟）。在 MySQL 层用 `TIMESTAMPDIFF(MINUTE, last_executed_at, UTC_TIMESTAMP()) >= interval` 判定，避免 JS/DB 时区不一致。

`outside_window` 和 `suppressed` 状态不计入冷却表（即不阻止后续执行）。

## 九、权限与安全模型

### 9.1 用户角色

- **普通用户** (`owner` 角色)：只能操作其 `owner_id` 下的规则和账户
- **管理员** (`admin` / `super_admin`)：可查看/操作所有规则，可按负责人筛选

### 9.2 规则列表权限

- 非管理员：按 `users.owner_id` 过滤（同负责人下所有用户创建的规则共享）
- 管理员：可按 `ownerIds` 筛选特定负责人，或 `includeNoOwner` 包含管理员创建的规则

### 9.3 账户权限

- 普通用户创建/更新规则时，`accountId` 必须属于其负责人且在 `account_mappings` 中 `is_active=1`
- 管理员：账户必须在 `account_mappings` 中存在且激活
- 批量添加账户时逐账户校验

### 9.4 规则启用「开闸」校验 (`ruleEnableGateService.js`)

启用规则前（enabled=true），必须满足：
1. 条件结构合法
2. 时间窗口一致
3. 动作数组非空且合法
4. 至少绑定一个有效广告账户
5. 每个账户均有权限访问
6. 若开启动态筛选：系统功能开关 `ENABLE_DYNAMIC_SCOPE` 必须为 true，`scope_filters` 必须有效

## 十一、关键设计决策与约定

1. **DNF 条件格式**：V2 支持 «组内 AND，组间 OR»，前端保持线性编辑体验（linesToV2Groups / v2ToLines 互转）
2. **全局时间窗口**：同规则内所有条件必须使用相同的 `time_window`，`custom_range` 的 since/until 也必须一致
3. **target_by_account**：以 `target_ids` 为唯一真理来源，`syncTargetByAccount()` 在每次保存时强制从 `target_ids` 生成并覆盖 `target_by_account`
4. **复合 ID 归一化**：`parseCompositeId` 反复剥离 `act_` 前缀防止 `act_act_xxx` 脏数据
5. **动态筛选 fail-safe**：快照为空时不回退到全量广告（fail-closed），`ERROR_OVERSIZE` 时保留上一版快照
6. **预算护栏**：底层存储单位为「美分」，前端展示为「美元」；最小预算 100 美分 = 1 美元
7. **审计完整性**：每次 CRUD 操作写入 `rule_history`（配置快照），每次动作执行写入 `automation_logs`（含可解释审计 JSON）
8. **AdsPolar 流水线**：数据同步 → 结构同步 → 规则执行形成链式反应，不依赖固定 cron 表达式
