# outside_window 冷却死锁 Bug — 修复方案

> 日期：2026-05-16 | Bug ID：规则1179排查 | 严重程度：🔴 P0（85% 作用域永久死锁）

---

## 一、Bug 一句话描述

`outside_window` 状态写入 `rule_ad_execution_state` 时**错误地更新了 `last_executed_at`**，重置了 24h 冷却时钟，导致作用域一旦首次评估落在执行窗口外就永久死锁。

---

## 二、修复策略

### 核心原则

> **"跳过"不计入冷却，只有真正执行过（success / fail）才计入冷却。**

`outside_window` 和 `suppressed` 的含义是"这次没执行"，既然没执行，就不应该启动冷却倒计时。下次 cron 周期照常重试。

### 实现方式

**`outside_window` 和 `suppressed` 状态根本不写入 `rule_ad_execution_state` 表。** 这样冷却表里没有记录 → `loadRuleAdExecutionState` 查不到 → 冷却判断为"从未执行过，放行" → 每分钟重试直到进入窗口真正执行。

这是最干净的实现：不加占位值、不改 SQL、不绕弯——从源头不写入。

### 为什么 `last_status` 字段可以丢弃？

全系统搜索确认：**`last_status` 字段只被写入，从未被任何代码读取用于决策。**

```
读取 last_executed_at 的地方（参与冷却判断）：
  ✅ loadRuleAdExecutionState()  — SELECT scope_key, last_executed_at
  ✅ isCooldownDue()             — SELECT last_executed_at, TIMESTAMPDIFF(...)

读取 last_status 的地方（参与逻辑判断）：
  ❌ 无。没有任何代码用 last_status 做 if/switch 判断。
```

`last_status` 纯粹是人工诊断用的冗余字段。"跳过类"状态的诊断信息已经在 `rule_execution_summaries` 表中有完整记录（`skip_reason = 'outside_execution_window'`）。

## 三、修复范围：只需改 1 个文件、1 个函数、加 2 行

```
server/services/ruleExecutionStateService.js → upsertRuleAdExecutionStateBatch()
                                                    ↑
                                              唯一的写入入口
```

**为什么只改一个函数就覆盖所有规则？**

- `rule_ad_execution_state` 表的**所有** INSERT/UPDATE 只经过这一个函数
- 不管是规则 1179 还是其他任何规则，不管执行窗口怎么配置，都走这条路
- 这里修好 = 所有规则的冷却机制都修好

```
全系统调用链：
  cronService.executeRulesForAccount()
    ├─ 读: loadRuleAdExecutionState()        → SELECT last_executed_at
    ├─ 执行窗口检查 → outside_window 判定
    └─ 写: upsertRuleAdExecutionStateBatch()  → INSERT/UPDATE  ← 只改这里！
  
  actionExecutorService
    └─ 读: isCooldownDue()                   → SELECT last_executed_at
```

## 四、具体代码修改

### 修改位置

`server/services/ruleExecutionStateService.js` 第 100-120 行

### 修改前

```js
export async function upsertRuleAdExecutionStateBatch(entries) {
  if (!entries?.length) return
  const validStatus = ['success', 'fail', 'suppressed', 'outside_window']
  try {
    for (const { ruleId, scopeKey, lastStatus } of entries) {
      const status = validStatus.includes(lastStatus) ? lastStatus : null
      let adIdVal = ''
      if (scopeKey.startsWith('status_ad:')) {
        adIdVal = scopeKey.slice('status_ad:'.length)
      } else if (scopeKey.startsWith('ad:')) {
        adIdVal = scopeKey.slice(3)
      }
      await pool.execute(
        `INSERT INTO rule_ad_execution_state (rule_id, scope_key, last_executed_at, last_status, ad_id)
         VALUES (?, ?, UTC_TIMESTAMP(), ?, ?)
         ON DUPLICATE KEY UPDATE last_executed_at = VALUES(last_executed_at), last_status = VALUES(last_status)`,
        [ruleId, scopeKey, status, adIdVal]
      )
    }
  } catch (err) {
    logger.warn('   ⚠️  upsertRuleAdExecutionStateBatch 失败:', err.message)
  }
}
```

### 修改后

```js
export async function upsertRuleAdExecutionStateBatch(entries) {
  if (!entries?.length) return
  const validStatus = ['success', 'fail', 'suppressed', 'outside_window']
  try {
    for (const { ruleId, scopeKey, lastStatus } of entries) {
      // outside_window / suppressed 不计入冷却，不写入此表
      // 冷却表只记录真正执行过（success/fail）的状态
      if (lastStatus === 'outside_window' || lastStatus === 'suppressed') continue

      const status = validStatus.includes(lastStatus) ? lastStatus : null
      let adIdVal = ''
      if (scopeKey.startsWith('status_ad:')) {
        adIdVal = scopeKey.slice('status_ad:'.length)
      } else if (scopeKey.startsWith('ad:')) {
        adIdVal = scopeKey.slice(3)
      }
      await pool.execute(
        `INSERT INTO rule_ad_execution_state (rule_id, scope_key, last_executed_at, last_status, ad_id)
         VALUES (?, ?, UTC_TIMESTAMP(), ?, ?)
         ON DUPLICATE KEY UPDATE last_executed_at = VALUES(last_executed_at), last_status = VALUES(last_status)`,
        [ruleId, scopeKey, status, adIdVal]
      )
    }
  } catch (err) {
    logger.warn('   ⚠️  upsertRuleAdExecutionStateBatch 失败:', err.message)
  }
}
```

**只加了两行**（第 6-7 行的 `if continue`），其余代码一行不动。

### 修改前后对比

| 状态 | 修改前 | 修改后 |
|------|--------|--------|
| `outside_window` | 写入冷却表 → 重置 `last_executed_at` → 死锁 | **跳过，不写入** → 无记录 → 每分钟重试 |
| `suppressed` | 写入冷却表 → 重置 `last_executed_at` | **跳过，不写入** |
| `success` | 正常写入冷却表 | **不变** |
| `fail` | 正常写入冷却表 | **不变** |

## 五、会不会影响别的规则？

### 5.1 对所有规则的影响

**会。所有配置了执行窗口的规则都会受益。**

这是修复了一个**通用 Bug**，不是规则 1179 的特例。任何规则只要同时满足：
- 配置了执行时间段（`execution_time_windows` 非空）
- 冷却间隔 > 0

都有相同的死锁风险。规则 1179 只是最典型（窗口仅 20 分钟 + 冷却 24 小时 → 死锁概率极高）。

### 5.2 对各功能模块的影响

| 模块 | 是否受影响 | 说明 |
|------|:--:|------|
| **冷却机制** | ✅ 修好 | outside_window/suppressed 不再参与冷却计时 |
| **success/fail 冷却** | ❌ 不变 | 真正执行过的照常冷却，24h 等逻辑完全不动 |
| **执行窗口检查** | ❌ 不变 | `isInExecutionWindow()` 不读冷却表 |
| **规则条件评估** | ❌ 不变 | 条件计算不依赖冷却表 |
| **数据同步** | ❌ 不变 | 完全不相关 |
| **手动执行（executeSingleRule）** | ❌ 不变 | 不经过此表 |
| **仲裁逻辑（arbitration）** | ❌ 不变 | 不读冷却表 |
| **前端展示** | ❌ 不变 | 前端不读此表 |
| **API 接口** | ❌ 不变 | 没有接口暴露此表 |
| **`rule_execution_summaries`** | ❌ 不变 | 该表独立记录，不受影响 |

### 5.3 对已有 `outside_window` 记录的影响

数据库中规则 1179 下 18 条 `last_status = 'outside_window'` 的历史记录：

- **不会被自动删除**，会留在表里
- **不再被更新**（因为不再写入 outside_window 状态）
- **不影响冷却判断**（因为 `loadRuleAdExecutionState` 会读到它们的 `last_executed_at`，这些时间都在 16:33~20:58 BJ，距现在不到 24h → 冷却未到期 → 仍会被过滤）

所以上线后需要执行一次清理 SQL（见第六节），让这些历史卡死记录不再阻塞冷却判断。

## 六、修复后的行为推演

```
Day 1 16:33  首次匹配 → 条件满足 → 冷却表无记录 → 通过冷却
             → 执行窗口检查 → 不在 00:00~00:20 → outside_window
             → 不写冷却表 ← 改动点
             
Day 1 16:34  再次匹配 → 冷却表仍无记录 → 通过冷却
             → 执行窗口检查 → 不在窗口 → outside_window → 不写表
             
...每分钟重复，直到...

Day 2 00:00  匹配 → 冷却表无记录 → 通过冷却
             → 执行窗口检查 → 在窗口内！→ 执行动作 ✅
             → success → 写入冷却表 → last_executed_at = 00:00

Day 2 00:01  匹配 → 冷却检查: 1min < 1440min → 跳过 ✅ 正常冷却生效

Day 3 00:00  冷却到期 → 在窗口内 → 成功 ✅
```

**冷却机制完整保留，只是"跳过"不再重置计时器。**

## 七、上线步骤

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 修改 `ruleExecutionStateService.js` | 加 2 行 `if continue` |
| 2 | 重启服务 | `pm2 restart fb-ad-server` |
| 3 | **立即执行清理 SQL** | 清除历史卡死记录（见下方） |
| 4 | 观察今晚 00:00~00:20 | 确认所有作用域恢复 |

### 上线时必须执行的 SQL

```sql
-- 删除所有 outside_window 和 suppressed 的历史记录
-- 这些记录不再被更新，留在表里会阻塞冷却判断
DELETE FROM rule_ad_execution_state 
WHERE last_status IN ('outside_window', 'suppressed');

-- 验证：规则 1179 应该只剩下 success 和 fail
SELECT last_status, COUNT(*) 
FROM rule_ad_execution_state 
WHERE rule_id = 1179 
GROUP BY last_status;
-- 期望：只剩 success(2) + fail(1)，共 3 条
```

> ⚠️ 这条 DELETE 必须在代码部署后执行，否则旧的 cron 还会重新写入。

### 回滚方案

如果修复引入问题：

```bash
git checkout server/services/ruleExecutionStateService.js
pm2 restart fb-ad-server
```

不需要回滚数据（`last_executed_at` 语义未变）。
