# 时区问题说明

## 🔍 问题现象

**用户发现**：
- 规则执行时间：`2026/1/27 15:53:05`（本地时间，UTC+8）
- 数据库存储时间：`2026-01-27 07:53:05`（UTC 时间）
- 时间差：8小时

---

## ✅ 这是正常行为

### 时区转换说明

1. **执行时间（本地时间）**：`15:53:05`（UTC+8，北京时间）
2. **存储到数据库（UTC）**：`07:53:05`（UTC，比北京时间少8小时）
3. **查询显示（UTC）**：`07:53:05`（UTC）

**这是正确的行为！** 数据库统一使用 UTC 时间存储，这是最佳实践。

---

## 📊 时区处理原则

根据代码注释（`server/db/connection.js`）：

```
时区处理原则：
- 存储：统一使用 UTC（+00:00）
- 规则判断：后端转成广告账户时区
- 日志显示：前端转成浏览器本地时区
```

### 1. 存储层（数据库）

- **数据库会话时区**：UTC（`SET time_zone = '+00:00'`）
- **字段类型**：`TIMESTAMP`（会根据会话时区自动转换）
- **存储值**：UTC 时间

### 2. 应用层（后端）

- **冷却期计算**：使用 UTC 时间（正确）
- **规则判断**：转换为广告账户时区

### 3. 展示层（前端）

- **日志显示**：转换为浏览器本地时区

## 🔍 验证冷却期计算

### 冷却期计算逻辑

```javascript
function isRuleCooledDown(rule) {
  if (!rule.lastExecutedAt) return true
  
  const now = new Date()  // 当前 UTC 时间
  const lastExec = new Date(rule.lastExecutedAt)  // 上次执行时间（UTC）
  const diffMinutes = (now - lastExec) / (1000 * 60)
  
  // AdsPolar 模式：15分钟冷却期
  return diffMinutes >= 15
}
```

**说明**：
- `new Date()` 和 `new Date(rule.lastExecutedAt)` 都是 UTC 时间
- 冷却期计算使用 UTC 时间，这是**正确的**

## 📝 查询时显示本地时间

如果需要在查询时显示本地时间，可以使用 MySQL 的时区转换函数：

```sql
-- 显示 UTC 时间（默认）
SELECT last_executed_at FROM rules WHERE id = 138;

-- 显示本地时间（UTC+8）
SELECT 
  last_executed_at as utc_time,
  CONVERT_TZ(last_executed_at, '+00:00', '+08:00') as local_time
FROM rules 
WHERE id = 138;
```
