# 时区字段用途说明

> 目的：解释 `ad_snapshots.timezone_name` 和 `account_mappings.timezone_name` 的不同用途

---

## 📊 两个时区字段的用途

### 1. `ad_snapshots.timezone_name`（数据时区）

**用途**：给**规则执行**用的，用于确定数据是在哪个时区下同步的

**作用**：
- **数据时区优先策略**：查询 `today` 时，优先使用 `ad_snapshots` 最新快照的 `timezone_name`
- **确保时区一致性**：确保"写入时怎么切日 → 查询时就怎么切日"
- **避免时区不匹配**：不会因为配置变更导致"今天查不到"

**使用场景**：
- 规则执行时查询 `today` 数据
- 确定数据的自然日边界（在哪个时区下切日）

**不是用于**：
- ❌ 前端显示（前端应该从账户配置或用户设置获取时区）
- ❌ 归档检查（归档检查使用 `account_mappings.timezone_name`）

---

### 2. `account_mappings.timezone_name`（账户时区）

**用途**：给**归档检查**、**规则执行**、**前端显示**用的

**作用**：
- **归档检查**：判断账户本地时区是否在 06:00-06:09 窗口
- **规则执行**：作为默认时区（如果数据没有时区）
- **前端显示**：可以作为前端显示的时区配置

**使用场景**：
- 归档检查时判断账户本地时区
- 规则执行时计算时间窗口（如果数据没有时区）
- 前端显示时区信息（可选）

---

## 🔍 你发现的问题分析

### 问题：`ad_snapshots` 表中有两种时区

```
UTC: 1670 条记录，最新同步时间：2026-01-21 14:44:34（最新）
Asia/Shanghai: 25 条记录，最新同步时间：2026-01-19 16:55:52（较旧）
```

### 问题说明

1. **历史数据**（1月19日）使用 `Asia/Shanghai` 时区同步
2. **最新数据**（1月21日）使用 `UTC` 时区同步
3. **可能原因**：
   - 数据同步时，Facebook API 返回的时区变化了
   - 或者数据同步时没有正确获取时区

### 影响

**规则执行时**：
- 查询 `today` 数据时，会使用最新的时区（`UTC`）
- 如果账户实际时区是 `Asia/Shanghai`，会导致时区不匹配
- 可能影响"今天"数据的查询结果

---

## 💡 解决方案

### 方案 1：统一时区（推荐）

**目标**：确保 `ad_snapshots.timezone_name` 和 `account_mappings.timezone_name` 一致

**步骤**：
1. 从 Facebook API 获取账户真实时区
2. 更新 `account_mappings.timezone_name`
3. 后续数据同步时，使用 `account_mappings.timezone_name` 写入 `ad_snapshots.timezone_name`

### 方案 2：清理历史数据（可选）

如果历史数据不重要，可以清理旧时区的数据：

```sql
-- 删除旧时区的数据（谨慎操作！）
DELETE FROM ad_snapshots 
WHERE account_id = 'act_927139705822379' 
  AND timezone_name = 'Asia/Shanghai';
```

**注意**：这个操作会删除数据，请谨慎使用！

---

## 🎯 总结

### `ad_snapshots.timezone_name` 的用途
- ✅ **给规则执行用的**：确定数据是在哪个时区下同步的
- ✅ **确保时区一致性**：写入时和查询时使用相同的时区
- ❌ **不是给前端显示用的**：前端应该从账户配置获取时区

### `account_mappings.timezone_name` 的用途
- ✅ **给归档检查用的**：判断账户本地时区
- ✅ **给规则执行用的**：作为默认时区
- ✅ **给前端显示用的**：可以作为前端显示的时区配置

---

**建议**：先执行方案 2 的 SQL 更新 `account_mappings.timezone_name`，然后修改代码确保后续同步时自动更新。
