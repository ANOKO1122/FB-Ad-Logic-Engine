## ⏳ 待验证项目

### 1. 归档功能验证（动作A - 使用 data_date）

**目的**：验证归档查询是否真的使用 `data_date` 而不是 `synced_at`

**验证步骤**：
```bash
# 手动触发归档
node -e "import('./server/services/cronService.js').then(m => m.manualArchive())"
```

**验证 SQL**：
```sql
-- 检查归档后的 daily_stats 表
SELECT 
  account_id,
  date,
  COUNT(DISTINCT ad_id) as ad_count,
  SUM(spend) as total_spend
FROM daily_stats
WHERE date = DATE_SUB(CURDATE(), INTERVAL 1 DAY)
GROUP BY account_id, date
ORDER BY account_id
LIMIT 20;

-- 对比 ad_snapshots 中昨天的数据（使用 data_date）
SELECT 
  account_id,
  data_date,
  COUNT(DISTINCT ad_id) as ad_count,
  SUM(spend) as total_spend
FROM ad_snapshots
WHERE data_date = DATE_SUB(CURDATE(), INTERVAL 1 DAY)
GROUP BY account_id, data_date
ORDER BY account_id
LIMIT 20;
```

**预期结果**：
- 归档应该能正常执行
- `daily_stats` 中的数据应该与 `ad_snapshots` 中 `data_date = 昨天` 的数据一致

### 2. 优雅退出测试

**目的**：验证进程崩溃时，写入队列中的数据不会丢失

**验证步骤**：
1. 启动后端服务
2. 手动触发一次同步任务（让写入队列中有数据）
3. 在数据写入过程中，强制终止进程（Ctrl+C 或 kill）
4. 检查日志，看是否有 "优雅退出" 的日志
5. 重新启动服务，检查是否有数据丢失

**预期结果**：
- 应该看到 `🛑 收到 SIGINT 信号，开始优雅退出...` 的日志
- 应该看到 `💾 [队列写入器] 紧急模式：强制写入剩余数据...` 的日志
- 重新启动后，数据应该完整（因为队列中的数据已被写入）

**预期结果**：
- 单个任务失败应该被捕获并记录日志
