# 优化建议实施总结

## ✅ 已采纳并实施的建议

### 1. 历史长尾时区刷新（一次性任务）

**问题：**
- 当前的 `timezone_name` 更新只会覆盖滑动窗口范围内（默认近 7 天）
- 更早的自然日如果历史上曾以旧时区写入，仍可能触发"时区不匹配"的降级与 warnings

**解决方案：**
- ✅ 创建了 `refreshAccountTimezoneHistory` 函数
- ✅ 创建了 `refresh-timezone-history.js` 工具脚本

**使用方法：**
```bash
# 刷新指定账户
node refresh-timezone-history.js act_927139705822379

# 刷新所有活跃账户
node refresh-timezone-history.js
```

**功能：**
- 统一刷新 `daily_stats` 表中账户的历史时区
- 将不一致的 `timezone_name` 更新为账户当前时区
- 支持单个账户或批量刷新

---

### 2. 配额与成本优化

**问题：**
- 按日拉取 `time_increment=1` 数据量更大
- 如果账户/广告量增长明显，API 配额可能紧张

**解决方案：**
- ✅ 创建了 `filterActiveAds` 函数
- ✅ 在 `syncAccountSlidingWindow` 中添加了 `optimizeQuota` 参数
- ✅ 在 `syncAllAccountsSlidingWindow` 中添加了 `optimizeQuota` 参数

**功能：**
- 筛选近 N 天内有花费或转化的广告
- 只对活跃广告拉取按日数据，降低无效批次
- 保留"账户间 1 秒休眠"（已在代码中实现）

**使用方式：**
```javascript
// 启用配额优化（只拉取有数据的广告）
await syncAccountSlidingWindow(accountId, ownerId, timezoneName, 7, true)

// 或批量同步时启用
await syncAllAccountsSlidingWindow(7, true)
```

**优化效果：**
- 如果活跃广告数量 < 总广告数量的 50%，只拉取活跃广告
- 如果大部分广告都有数据，使用全部广告（避免过度筛选）

## 📊 实施细节

### 1. `filterActiveAds` 函数

**位置：** `server/services/ingestorService.js` 第 400-440 行

**逻辑：**
1. 查询 `daily_stats` 和 `ad_snapshots` 表
2. 找出近 N 天内有花费、转化或点击的广告
3. 如果筛选后的数量 < 总数量的 50%，返回筛选后的列表
4. 否则返回原列表（避免过度筛选）

**SQL 查询：**
```sql
SELECT DISTINCT ad_id
FROM (
  SELECT ad_id FROM daily_stats
  WHERE account_id = ? 
    AND date >= ? 
    AND date <= ?
    AND (spend > 0 OR purchases > 0 OR link_clicks > 0)
  
  UNION
  
  SELECT ad_id FROM ad_snapshots
  WHERE account_id = ?
    AND DATE(synced_at) >= ?
    AND DATE(synced_at) <= ?
    AND (spend > 0 OR purchases > 0 OR link_clicks > 0)
) AS active_ads
```

### 2. `refreshAccountTimezoneHistory` 函数

**位置：** `server/services/ingestorService.js` 第 455-509 行

**逻辑：**

3. 批量更新 `daily_stats.timezone_name`
4. 返回更新结果

**SQL 更新：**
```sql
UPDATE daily_stats
SET timezone_name = ?, updated_at = NOW()

```

## 🎯 使用建议

### 历史时区刷新
