## 🔑 重要知识点回顾

### 1. 数据字段处理
- **10 个核心字段**：状态、花费、成效(购买)、CPA、ROAS、CPC、uCPC、加购费、结账费、支付费
- **字段获取方式**：
  - 可直接获取：`spend`、`cpc`、`status`（从不同 API）
  - 需要提取：`purchases`、`ucpc`、`add_to_cart_cost` 等（从数组/对象中提取）
  - 必须计算：`cpa`（spend / purchases）、`roas`（从 `purchase_roas` 提取或计算）
- **重要发现**：`roas` 不是有效字段，应使用 `purchase_roas`（数组格式）
- **防御性编程**：处理 Facebook API 的"不值不显"原则（字段值为 0 时不返回）

### 2. 规则数据查询服务（新增）
- **智能路由**：根据时间窗口选择数据源（today → ad_snapshots，历史 → daily_stats）
- **06:00 边界降级**：如果 daily_stats 未落盘，降级到 ad_snapshots 的"昨日最后快照"
- **动态聚合计算**：多天窗口必须用"总分子/总分母"重算，避免辛普森悖论
- **时区处理**：统一使用账户时区计算时间窗口（⚠️ 当前存在时区不一致问题）

### 3. 规则引擎增强（新增）
- **离线查询**：从数据库查询数据，不调用 Facebook API
- **AND/OR 逻辑**：支持 `logic_operator` 字段（'AND' | 'OR'）
- **目标筛选**：支持 `target_level` 和 `target_ids` 筛选
- **时间窗口**：支持 `time_window` 参数（today/yesterday/last_3_days 等）
- **向后兼容**：保持旧版本 API 兼容，使用参数类型判断调用方式

### 2. Facebook Batch API
- 每次最多 20 个子请求
- 使用 `batch` 字段（JSON 字符串）发送请求
- 响应是数组，每个元素对应一个子请求的响应

### 3. 数据库表结构
- `ad_snapshots`：热数据（Today），28 个字段（包含 10 个核心字段 + 7 个原始计数字段 + ad_set_id）
- `daily_stats`：冷数据（历史聚合），包含 10 个核心字段的单日值
- `automation_logs`：审计日志
- **重要**：`ad_snapshots` 表已添加 `cpc` 和 `roas` 字段（迁移脚本 004）
