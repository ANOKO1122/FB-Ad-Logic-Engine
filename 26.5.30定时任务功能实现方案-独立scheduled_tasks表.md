# 定时任务功能实现方案（方案二：独立 scheduled_tasks 表）

> 版本：v1.6（set_budget 可上可下）  
> 日期：2026-05-30  
> 状态：方案设计阶段  

### v1.6 本次变更

`set_budget` 语义从「仅上调」改为「真正的设置预算（可上可下）」。定时任务在 `scheduledTaskService.js` 内直接计算目标美分，**不经过** `computeNewBudgetCentsOnce`（该函数对 `set_budget` 限定为仅上调，是为现有条件规则设计的保守策略，不适用于定时任务场景）。

| 变更点 | 旧行为 | 新行为 |
|---|---|---|
| `set_budget` | 当前预算≥目标 → 跳过不下调 | 直接设为目标值，无论当前更高或更低 |
| 前端文案 |「上调预算到」+ ⚠️仅上调 |「设置预算到」 |
| 计算路径 | 走 `computeNewBudgetCentsOnce`（仅上调） | 在 `executeAction` 内直接 `Math.round(value * 100)` + clamp `$1~$FB上限` |

### v1.1 审查修复清单

| 问题编号 | 严重级别 | 问题描述 | 修复方式 |
|---|---|---|---|
| #1 | 🔴 严重 | campaign 级预算调用了 getAdsetBudget | `executeAction` 按 targetLevel 分流：campaign→`getCampaignBudget`/`updateCampaignBudget` |
| #2 | 🔴 严重 | 失败任务每分钟无限重试 | 新增 `retry_count`/`max_retries` 字段 + 指数退避（5分→10分→20分）+ 超限自动禁用 |
| #3 | 🔴 严重 | node-cron 不防并发重叠 | Cron 入口增加 `scheduledTaskRunning` 互斥锁 |
| #4 | 🔴 严重 | set_budget 仅上调不下调，与用户直觉不符 | **v1.6 已彻底修复**：定时任务不走 `computeNewBudgetCentsOnce`，直接计算目标美分，可上可下 |
| #5 | 🟡 中等 | preflightMode 参数未写入 DB | `writeAuditLog` 中按 `isSimulation` 映射为 `'preflight'` |
| #6 | 🟡 中等 | 失败时 lastStatus 未更新 | `updateTaskAfterFailure` 中更新 `lastStatus='fail'` 或 `'stale'` |
| #7 | 🟡 中等 | cron 类型缺少 calcNextExecuteAt | 引入 `cron-parser`，在 switch 中增加 `cron` case |
| #8 | 🟡 中等 | ad 级预算找不到 adset 时会无限重试 | 结合问题#2：先 skip → 重试退避 → 最终禁用 |
| #11 | 🟢 轻微 | once 任务过期容差未在查询中体现 | 查询条件增加 `next_execute_at >= NOW() - INTERVAL 2 MINUTE` |
| #12 | 🟢 轻微 | 缺少最低预算校验 | API 层校验：预算结果 ≥ 100 cents（$1.00） |

## 一、方案选型结论

经过对比分析，**方案二优于方案一**，原因：

| 维度 | 方案一（扩展 rules） | 方案二（独立表） |
|---|---|---|
| 对现有功能影响 | 改动 6 个现有文件，包括最核心的 `executeRulesForAccount` | **零影响**——不改任何现有文件 |
| 对系统性能影响 | scheduled 与 condition 共享账户锁，全量拉取规则 | **隔离**——独立 Cron 不竞争锁，精确索引查询 |
| 回滚安全性 | 需回滚 rules 表 + 6 文件代码 | **删除新表 + 新文件即可** |
| 代码清晰度 | 规则表承载两种完全不同语义 | **职责分离**，各走各路 |

---

## 二、核心设计原则：最大化复用现有接口

**方案二不重写任何执行逻辑**。它只解决"存什么时间执行 + 到时间没到"的问题，实际的 **暂停/激活/调预算** 全部调用现有系统接口。

```
scheduled_tasks 表          ← 新增：存"什么时间做什么"
     │
     ▼
scheduledTaskService.js     ← 新增：时间判断 + 调度
     │
     ├─ 读取账户列表 ──▶ accountSyncService.getAccountsFromDatabase()
     ├─ 读取广告结构 ──▶ structureSyncService.listStructureObjectsFromDb()
     ├─ 执行暂停/激活 ──▶ FacebookMarketingAPI.pauseAd/activateAd (index.js)
     ├─ 执行调整预算 ──▶ FacebookMarketingAPI.updateAdsetBudget (index.js)
     │                   actionExecutorService.computeNewBudgetCentsOnce()
     └─ 写审计日志   ──▶ automation_logs 表（复用 Drizzle insert）
```

## 三、数据模型设计

### 3.1 新表 `scheduled_tasks`

```sql
CREATE TABLE scheduled_tasks (
  id              INT AUTO_INCREMENT PRIMARY KEY,
  -- 用户归属
  user_id         INT NOT NULL,
  owner_id        INT NOT NULL DEFAULT 0,

  -- 调度配置
  schedule_type   VARCHAR(16) NOT NULL,       -- once / daily / weekly / cron
  schedule_at     VARCHAR(32) NULL,             -- 格式见 3.2；cron 类型可为 NULL
  schedule_cron   VARCHAR(64) NULL,            -- cron 表达式（仅 cron 类型）
  schedule_timezone VARCHAR(50) NULL,          -- 时区，NULL=跟随账户时区
  next_execute_at DATETIME NULL,               -- 下次执行时间（UTC），由系统计算维护

  -- 目标对象
  account_id      VARCHAR(50) NOT NULL,        -- 广告账户 ID（act_xxx）
  target_level    VARCHAR(16) NOT NULL DEFAULT 'ad',  -- ad / adset / campaign
  target_id       VARCHAR(50) NOT NULL,        -- 目标对象 ID

  -- 动作配置
  action_type     VARCHAR(32) NOT NULL,        -- pause_ad / activate_ad / pause_adset / activate_adset / pause_campaign / activate_campaign / set_budget / increase_budget / decrease_budget（五轮审查#2.1：与现有规则系统枚举统一）
  action_params   JSON NOT NULL,               -- 动作参数（见 3.3）

  -- 状态控制
  enabled         TINYINT(1) NOT NULL DEFAULT 1,
  is_simulation   TINYINT(1) NOT NULL DEFAULT 0,  -- Dry Run 模式
  auto_disable    TINYINT(1) NOT NULL DEFAULT 1,  -- once 类型执行后自动 enabled=false

  -- 执行追踪
  last_executed_at DATETIME NULL,
  last_status     VARCHAR(16) NULL,            -- success / fail / skipped

  -- 重试控制
  retry_count     INT NOT NULL DEFAULT 0,      -- 累计重试次数
  max_retries     INT NOT NULL DEFAULT 3,      -- 最大重试次数，超出后自动禁用

  -- 并发控制（五轮审查#2.3）
  version         INT NOT NULL DEFAULT 0,       -- 乐观锁版本号

  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  INDEX idx_next_execute (enabled, next_execute_at),
  INDEX idx_account (account_id),
  INDEX idx_user (user_id),
  INDEX idx_owner (owner_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

|---|---|---|---|
| `once` | `YYYY-MM-DD HH:mm` | `2026-06-03 14:00` | 具体日期时间，执行后自动禁用 |
| `daily` | `HH:mm` | `08:00` | 每天这个时刻 |
| `weekly` | `W1,W2,W3\|HH:mm` | `1,3,5\|09:00` | 周一三五 09:00（W1=周一 ... W7=周日） |
| `cron` | — | — | 使用 `schedule_cron` 字段，`schedule_at` 可为 NULL |

## 四、执行流程设计

### 4.1 整体流程

```
每分钟 Cron (* * * * *)  ← 新增第 8 条，与现有 7 条并列
  │
  └─ executeDueScheduledTasks()
       │
       ├─ 1. SELECT * FROM scheduled_tasks
       │      WHERE enabled = 1
       │        AND next_execute_at >= UTC_TIMESTAMP() - INTERVAL 2 MINUTE   ← 2分钟容差窗口
       │        AND next_execute_at <= UTC_TIMESTAMP()
       │      LIMIT 100
       │
       ├─ 2. 遍历每条到期任务
       │    │
       │    ├─ 查 account_mappings 确认账户 active
       │    ├─ 查 structure_ads 确认目标对象存在
       │    ├─ 创建 FacebookMarketingAPI 实例
       │    │
       │    ├─ action_type === 'pause' ?
       │    │   ├─ targetLevel=ad      → api.pauseAd(targetId)
       │    │   ├─ targetLevel=adset   → api.pauseAdset(targetId)
       │    │   └─ targetLevel=campaign→ api.pauseCampaign(targetId)
       │    │
       │    ├─ action_type === 'activate' ?
       │    │   └─ 同理调用 api.activateAd / activateAdset / activateCampaign
       │    │
       │    ├─ action_type === 'set_budget' / 'increase_budget' / 'decrease_budget' ?
       │    │   ├─ 调用 computeNewBudgetCentsOnce() 计算目标预算（复用现有）
       │    │   ├─ 按 targetLevel 分别获取当前预算：
       │    │   │   ├─ ad     → 查 structure_ads 获取 adset_id → api.getAdsetBudget(adsetId)
       │    │   │   ├─ adset  → api.getAdsetBudget(targetId)
       │    │   │   └─ campaign→ api.getCampaignBudget(targetId)   ← 独立方法
       │    │   ├─ 按 targetLevel 分别更新预算：
       │    │   │   ├─ ad/adset → api.updateAdsetBudget(id, newCents)
       │    │   │   └─ campaign → api.updateCampaignBudget(id, newCents)
       │    │
       │    └─ 写入 automation_logs 审计记录
       │
       └─ 3. 更新状态
            ├─ 成功：更新 next_execute_at + lastStatus='success' + retryCount=0
            ├─ 失败：指数退避重试；超 maxRetries → 自动禁用 + lastStatus='stale'
            └─ 跳过：更新 lastStatus='skipped'
```

### 4.2 核心函数 `executeDueScheduledTasks`

```js
// server/services/scheduledTaskService.js

// 二次审查 R2#2：修正 logger 导入路径（所有 service 统一使用 ../utils/logger.js）
import logger from '../utils/logger.js'
import { db } from '../db/drizzle.js'
import { scheduledTasks, automationLogs } from '../db/schema.js'
import { and, eq, lte, gte } from 'drizzle-orm'
import pool from '../db/connection.js'
import { FacebookMarketingAPI } from '../index.js'
import { computeNewBudgetCentsOnce } from './actionExecutorService.js'
import { DateTime } from 'luxon'

// 二次审查 R2#4：互斥锁只在此文件定义一次，导出检查函数供 cron 处调用
let _scheduledTaskRunning = false
export function isScheduledTaskRunning() { return _scheduledTaskRunning }

export async function executeDueScheduledTasks() {
  // 二次审查 R2#4：入口处检查互斥锁
  if (_scheduledTaskRunning) {
    return { executed: 0, skipped: 0, errors: 0, skippedDueToLock: true }
  }
  _scheduledTaskRunning = true

  try {
    const token = process.env.FACEBOOK_ACCESS_TOKEN
  if (!token) {
    logger.warn('[ScheduledTask] 跳过：FACEBOOK_ACCESS_TOKEN 缺失')
    return { executed: 0, skipped: 0, errors: 0 }
  }

  // 1. 查询到期的任务（审查问题11：2 分钟容差窗口，避免执行过期太久的 once 任务）
  const toleranceWindow = new Date(Date.now() - 2 * 60 * 1000)
  const dueTasks = await db
    .select()
    .from(scheduledTasks)
    .where(and(
      eq(scheduledTasks.enabled, true),
      gte(scheduledTasks.nextExecuteAt, toleranceWindow),
      lte(scheduledTasks.nextExecuteAt, new Date())
    ))
    .limit(100)

  if (dueTasks.length === 0) return { executed: 0, skipped: 0, errors: 0 }

  // 四轮审查#5：达到上限时告警，避免静默积压
  if (dueTasks.length >= 100) {
    logger.warn('[ScheduledTask Cron] 到期任务数达到 LIMIT 上限 100，可能存在积压')
  }

  const api = new FacebookMarketingAPI(token)
  let executed = 0, skipped = 0, errors = 0

  for (const task of dueTasks) {
    try {
      // 2. 校验账户是否 active
      const [mappingRows] = await pool.execute(
        'SELECT 1 FROM account_mappings WHERE fb_account_id = ? AND is_active = 1 LIMIT 1',
        [task.accountId]
      )
      if (mappingRows.length === 0) {
        logger.info(`[ScheduledTask #${task.id}] 账户 ${task.accountId} 未激活，跳过`)
        skipped++
        await updateTaskAfterSkip(task)
        continue
      }

      // 3. 确定执行目标（审查问题1：按 targetLevel 分流预算操作）
      const targetId = task.targetId
      const isBudgetAction = ['set_budget', 'increase_budget', 'decrease_budget'].includes(task.actionType)
      let budgetObjectId = targetId  // 预算操作的真实对象 ID（adset或campaign）

      if (task.targetLevel === 'ad' && isBudgetAction) {
        // ad 级预算：需查它所属的 adset_id
        const [adRows] = await pool.execute(
          'SELECT adset_id FROM structure_ads WHERE account_id = ? AND ad_id = ? LIMIT 1',
          [task.accountId, targetId]
        )
        if (adRows.length === 0) {
          logger.warn(`[ScheduledTask #${task.id}] 找不到广告 ${targetId}，跳过`)
          skipped++
          await updateTaskAfterSkip(task)
          continue
        }
        budgetObjectId = adRows[0].adset_id
      }

      // 4. 执行动作
      if (task.isSimulation) {
        logger.info(`[ScheduledTask #${task.id}] Dry Run: ${task.actionType} on ${targetId}`)
        executed++
        // 四轮审查#2：先更新状态，后写日志（状态是主数据，日志是追溯数据）
        await updateTaskAfterExecute(task)
        await writeAuditLog(task, 'success', 'preflight', null)
        continue
      }

      await executeAction(api, task, targetId, budgetObjectId)
      executed++
      // 五轮审查#2.2：状态更新 + 日志写入在同一事务中，防止不一致
      await db.transaction(async (tx) => {
        await updateTaskAfterExecute(task)
        await writeAuditLog(task, 'success', null, null)
      })

    } catch (err) {
      logger.error(`[ScheduledTask #${task.id}] 执行失败:`, err.message)
      errors++
      // 四轮审查#2：失败时先更新状态（退避重试），后写日志
      await updateTaskAfterFailure(task)
      await writeAuditLog(task, 'fail', null, err.message)
  }

  return { executed, skipped, errors }
  } finally {
    // 二次审查 R2#4：确保锁一定会释放
    _scheduledTaskRunning = false
  }
}

/**
 * 执行单个动作（审查问题1：按 targetLevel 分别调用正确的 FB API）
 */
async function executeAction(api, task, targetId, budgetObjectId) {
  const { actionType, targetLevel } = task
  const params = typeof task.actionParams === 'string'
    ? JSON.parse(task.actionParams)
    : (task.actionParams || {})

  switch (actionType) {
    case 'pause_ad':
      if (targetLevel === 'adset') return api.pauseAdset(targetId)
      if (targetLevel === 'campaign') return api.pauseCampaign(targetId)
      return api.pauseAd(targetId)

    case 'activate_ad':
      if (targetLevel === 'adset') return api.activateAdset(targetId)
      if (targetLevel === 'campaign') return api.activateCampaign(targetId)
      return api.activateAd(targetId)

    case 'set_budget': {
      // v1.6：set_budget 在定时任务中的语义是"精确设置，可上可下"
      // 不经过 computeNewBudgetCentsOnce（该函数对 set_budget 限制为仅上调）
      const targetUsd = Number(params?.value)
      if (!Number.isFinite(targetUsd) || targetUsd <= 0) {
        logger.warn(`[ScheduledTask #${task.id}] set_budget value 无效: ${params?.value}`)
        return
      }
      let newCents = Math.round(targetUsd * 100)
      newCents = Math.max(newCents, 100) // 最低 $1.00
      if (params?.max_daily_budget != null) {
        const cap = Math.round(Number(params.max_daily_budget))
        if (cap > 0) newCents = Math.min(newCents, cap)
      }
      if (targetLevel === 'campaign') {
        return api.updateCampaignBudget(budgetObjectId, newCents)
      }
      return api.updateAdsetBudget(budgetObjectId, newCents)
    }

    case 'increase_budget':
    case 'decrease_budget': {
      // increase/decrease 仍走 computeNewBudgetCentsOnce（按百分比或美元增减）
      let currentCents
      if (targetLevel === 'campaign') {
        currentCents = await api.getCampaignBudget(budgetObjectId)
      } else {
        currentCents = await api.getAdsetBudget(budgetObjectId)
      }
      const action = { ...params, type: actionType }
      const newCents = computeNewBudgetCentsOnce(currentCents, action)
      if (newCents === currentCents) {
        logger.info(
          `[ScheduledTask #${task.id}] 预算无需调整 ` +
          `（当前=${currentCents}分，目标=${newCents}分，action=${actionType}），跳过API调用`
        )
        return
      }
      if (targetLevel === 'campaign') {
        return api.updateCampaignBudget(budgetObjectId, newCents)
      }
      return api.updateAdsetBudget(budgetObjectId, newCents)
    }

    default:
      throw new Error(`不支持的动作类型: ${actionType}`)
  }
}

/** 计算下次执行时间并更新 */
async function updateTaskAfterExecute(task) {
  const nextAt = await computeNextExecuteAt(task)  // M1: async 函数需 await
  let newEnabled = task.enabled
  if (task.scheduleType === 'once' && task.autoDisable) newEnabled = false

  await db.update(scheduledTasks)
    .set({
      lastExecutedAt: new Date(),
      lastStatus: 'success',
      nextExecuteAt: nextAt,
      enabled: newEnabled,
      retryCount: 0  // 成功后重置重试计数
    })
    .where(eq(scheduledTasks.id, task.id))
}

/**
 * 失败后更新：指数退避重试（审查问题2）
 * 第1次失败：5分钟后重试
 * 第2次失败：10分钟后重试
 * 第3次失败：20分钟后重试（超过 maxRetries → 自动禁用）
 */
async function updateTaskAfterFailure(task) {
  const newRetryCount = (task.retryCount || 0) + 1
  const maxRetries = task.maxRetries || 3
  const now = DateTime.utc()

  if (newRetryCount >= maxRetries) {
    // 超过重试上限，自动禁用
    await db.update(scheduledTasks)
      .set({
        lastStatus: 'stale',
        retryCount: newRetryCount,
        enabled: false,
        lastExecutedAt: now.toJSDate()
      })
      .where(eq(scheduledTasks.id, task.id))
    logger.warn(
      `[ScheduledTask #${task.id}] 已重试 ${newRetryCount}/${maxRetries} 次，自动禁用。` +
      `请检查任务配置或目标对象状态。`
    )
  } else {
    // 指数退避：5分 × 2^(retryCount-1)
    const backoffMinutes = 5 * Math.pow(2, newRetryCount - 1)
    const nextRetry = now.plus({ minutes: backoffMinutes }).toJSDate()
    await db.update(scheduledTasks)
      .set({
        lastStatus: 'fail',
        retryCount: newRetryCount,
        nextExecuteAt: nextRetry,
        lastExecutedAt: now.toJSDate()
      })
      .where(eq(scheduledTasks.id, task.id))
    logger.info(
      `[ScheduledTask #${task.id}] 第 ${newRetryCount}/${maxRetries} 次失败，` +
      `${backoffMinutes} 分钟后重试`
    )
  }
}

/** 跳过时更新：重新计算 next_execute_at 前推到下一周期，避免死循环（R2#3） */
async function updateTaskAfterSkip(task) {
  const nextAt = await computeNextExecuteAt(task)  // M1: async 函数需 await
  await db.update(scheduledTasks)
    .set({
      lastStatus: 'skipped',
      nextExecuteAt: nextAt  // 前推到下一周期，防止每分钟重复 skip
    })
    .where(eq(scheduledTasks.id, task.id))
}

/**
 * 从任务配置计算下次执行时间（复用于 updateTaskAfterExecute 和 updateTaskAfterSkip）
 * M1 修复：ESM 模块使用 await import() 而非 require()
 */
async function computeNextExecuteAt(task) {
  const zone = task.scheduleTimezone || 'UTC'
  const now = DateTime.now().setZone(zone)

  switch (task.scheduleType) {
    case 'once':
      return null  // 一次性任务无需 next
    case 'daily': {
      const [hh, mm] = (task.scheduleAt || '00:00').split(':').map(Number)
      return now.plus({ days: 1 }).set({ hour: hh, minute: mm, second: 0, millisecond: 0 }).toUTC().toJSDate()
    }
    case 'weekly': {
      const [weekDays, timeStr] = (task.scheduleAt || '').split('|')
      const allowed = weekDays.split(',').map(Number)
      const [hh, mm] = timeStr.split(':').map(Number)
      for (let d = 1; d <= 7; d++) {
        const candidate = now.plus({ days: d })
        if (allowed.includes(candidate.weekday)) {
          return candidate.set({ hour: hh, minute: mm, second: 0, millisecond: 0 }).toUTC().toJSDate()
        }
      }
      return null
    }
    case 'cron': {
      try {
        // M1 修复：ESM 模块必须使用 await import，不可用 require
        const { default: cronParser } = await import('cron-parser')
        const interval = cronParser.parseExpression(task.scheduleCron || '* * * * *', { tz: zone })
        return interval.next().toDate()
      } catch {
        return null
      }
    }
    default:
      return null
  }
}

/**
 * 写入审计日志
 * S1 修复：非 ad 级任务从 structure_ads 获取关联 ad_id，填充 auditAdId
 */
async function writeAuditLog(task, status, preflightMode, errorMessage) {
  // S1：非 ad 级操作从 structure_ads 获取一个关联广告 ID，用于审计日志追溯
  let auditAdId = ''
  if (task.targetLevel === 'ad') {
    auditAdId = String(task.targetId)
  } else {
    try {
      const filterCol = task.targetLevel === 'adset' ? 'adset_id' : 'campaign_id'
      const [rows] = await pool.execute(
        `SELECT ad_id FROM structure_ads WHERE account_id = ? AND ${filterCol} = ? LIMIT 1`,
        [task.accountId, task.targetId]
      )
      auditAdId = String(rows?.[0]?.ad_id || task.targetId)
    } catch {
      auditAdId = String(task.targetId)  // 优雅降级：用 targetId 兜底
    }
  }

  try {
    await db.insert(automationLogs).values({
      accountId: String(task.accountId),
      adId: auditAdId,
      objectType: task.targetLevel,
      objectId: String(task.targetId),
      actionType: String(task.actionType).toUpperCase(),
      actionPayload: typeof task.actionParams === 'string'
        ? JSON.parse(task.actionParams)
        : (task.actionParams || {}),
      explanation: JSON.stringify({
        trigger_type: 'scheduled',
        schedule_type: task.scheduleType,
        schedule_at: task.scheduleAt,
        task_id: task.id
      }),
      preflightMode: preflightMode || (task.isSimulation ? 'preflight' : null),
      status,
      errorMessage,
      isSimulation: task.isSimulation || false,
      triggeredAt: new Date(),
      ownerId: task.ownerId || 0
    })
  } catch (e) {
    logger.error(`[ScheduledTask #${task.id}] 审计日志写入失败:`, e.message)
  }
}
```

### 7.4 `calcNextExecuteAt` 函数实现

服务端在创建/更新时自动计算 `next_execute_at`（UTC）：

```js
/**
 * 二次审查 R2#5：若 schedule_timezone 为 NULL，回退到账户时区（查询 account_mappings）
 */
async function calcNextExecuteAt(scheduleType, scheduleAt, scheduleCron, scheduleTimezone, accountId) {
  // R2#5：NULL 时回退到账户时区，而非硬编码 UTC
  let zone = scheduleTimezone
  if (!zone) {
    try {
      const [rows] = await pool.execute(
        'SELECT timezone_name FROM account_mappings WHERE fb_account_id = ? LIMIT 1',
        [accountId]
      )
      zone = rows?.[0]?.timezone_name || 'UTC'
    } catch {
      zone = 'UTC'
    }
  }
  const now = DateTime.now().setZone(zone)

  switch (scheduleType) {
    case 'once': {
      const [datePart, timePart] = scheduleAt.split(' ')
      return DateTime.fromISO(`${datePart}T${timePart}:00`, { zone }).toUTC().toJSDate()
    }
    case 'daily': {
      const [hh, mm] = scheduleAt.split(':').map(Number)
      let target = now.set({ hour: hh, minute: mm, second: 0, millisecond: 0 })
      if (target <= now) target = target.plus({ days: 1 })
      return target.toUTC().toJSDate()
    }
    case 'weekly': {
      const [weekDays, timeStr] = scheduleAt.split('|')
      const allowed = weekDays.split(',').map(Number)
      const [hh, mm] = timeStr.split(':').map(Number)
      for (let d = 0; d <= 7; d++) {
        const candidate = now.plus({ days: d }).set({ hour: hh, minute: mm, second: 0, millisecond: 0 })
        if (allowed.includes(candidate.weekday) && candidate > now) {
          return candidate.toUTC().toJSDate()
        }
      }
      return null
    }
    case 'cron': {
      // 审查问题7：cron 类型使用 cron-parser 计算 next
      try {
        const { default: cronParser } = await import('cron-parser')
        const interval = cronParser.parseExpression(scheduleCron || '* * * * *', { tz: zone })
        return interval.next().toDate()
      } catch (e) {
        logger.error(`[ScheduledTask] cron 表达式解析失败: ${scheduleCron}`, e.message)
        return null
      }
    }
    default:
      return null
  }
}
```

### 与条件规则的交互约定（四轮审查#3）

定时任务与条件规则**独立运作**，不互斥、不互锁：

| 场景 | 行为 |
|---|---|
| 定时任务 pause + 条件规则 activate | 可能产生"乒乓效应"——两个系统独立判断，轮流操作同一对象 |
| 定时任务刚执行完 set_budget | 条件规则下一分钟评估时用的仍是旧指标值（定时任务不更新 `ad_snapshots`） |
| 冲突规避 | **用户自行协调**，不在同一对象上设置冲突的定时任务和条件规则 |

所有定时任务的审计日志均标记 `explanation.trigger_type = 'scheduled'`，前端日志页可据此区分操作来源。

### 清理策略（R2#7）

`once` 类型任务执行后 `enabled=false`，会累积在表中。建议定期清理：

```sql
-- 清理 90 天前已禁用的 once 类型定时任务
DELETE FROM scheduled_tasks
WHERE enabled = 0
  AND schedule_type = 'once'
  AND updated_at < NOW() - INTERVAL 90 DAY;
```

可加入现有的历史表清理 Cron（`cron.schedule('30 4 * * *', ...)`）中一并执行。具体做法：在 `server/services/cronService.js` 的 `runNightlyHistoryCleanup()` 函数中追加此 DELETE 语句（五轮审查#3.2）。

## 十三、风险与边界

| 风险 | 应对 |
|---|---|
| 服务器宕机错过 once 任务 | `next_execute_at <= NOW()` + 2 分钟容差窗口兜底——重启后首次 Cron 扫描会补偿执行 |
| once 任务已过期（超过 `schedule_at` 2分钟以上） | 查询条件 `next_execute_at >= NOW() - INTERVAL 2 MINUTE` 排除，不会执行 |
| daily 任务在 DST（夏令时）切换日 | 使用 `schedule_timezone` 时区计算，Luxon 自动处理 DST |
| 同一任务被并发执行 | Cron 回调入口有互斥锁（`scheduledTaskRunning` 标志），防止重叠；SQL UPDATE 是原子的 |
| 任务持续失败打爆 API 配额 | 指数退避重试（5分→10分→20分），超 `maxRetries` 自动禁用，不会无限重试 |
| 广告已被删除 | 执行前查 `structure_ads` 确认存在，不存在则跳过 + 重试退避 → 最终禁用 |
| 调预算时广告组不存在 | 同理，跳过 + 重试退避 → 最终禁用 |
| campaign 级预算操作调错 API | 已按 `targetLevel` 分流：campaign→`getCampaignBudget`/`updateCampaignBudget`，ad/adset→`getAdsetBudget`/`updateAdsetBudget` |
