## 八、🔍 根因代码定位

### 泄漏源头：`server/db/connection.js` — 自定义 pool.execute() 封装

**文件路径**：`/root/work/FB-Ad-Logic-Engine/server/db/connection.js`

#### 问题代码（第 51-61 行）：

```javascript
// ❌ 此封装是 PS 泄漏的根因
const originalExecute = rawPool.execute.bind(rawPool)
rawPool.execute = async function(sql, params) {
  const connection = await rawPool.getConnection()
  try {
    await setConnectionTimezone(connection)
    // 执行 SQL（使用连接的 execute 方法）
    return await connection.execute(sql, params)  // ⚠️ 问题在这里
  } finally {
    connection.release()
  }
}
```

#### 泄漏机制分析：

```
每次 pool.execute(sql, params) 调用：
┌─────────────────────────────────────────────────┐
│ 1. rawPool.getConnection()  获取池中连接         │
│ 2. connection.execute("SET time_zone...")        │
│    └→ 创建 PS #1 (SET time_zone)                │
│ 3. connection.execute(sql, params)               │
│    └→ 创建 PS #2 (实际业务SQL)                   │
│ 4. connection.release()  归还连接到池            │
│    └→ ⚠️ PS #1 和 PS #2 保留在 MySQL 服务端！    │
└─────────────────────────────────────────────────┘

连接被池复用 → 新 SQL → 又创建新 PS → 永久累积 → 92,647 个 PS
```

#### 为什么会泄漏？三层原因：

| 层 | 问题 |
|----|------|
| **mysql2 行为** | `connection.execute()` 创建的 prepared statement 缓存在连接对象上，`release()` 时不会自动 DEALLOCATE |
| **自定义封装** | 绕过 mysql2 原生 `pool.execute()` 的 PS 缓存管理机制（原生有 `maxPreparedStatements` 限制） |
| **动态 SQL** | 大量查询使用 `ad_id IN (?,?,...)` 动态占位符，不同数量的占位符 = 不同的 PS，成倍放大泄漏 |
| **连接池复用** | 25 个连接永久存活，6 天内从未关闭，PS 无限累积 |

#### 泄漏量级估算：

```
25 个连接 × ~3,700 个不同SQL变体/连接 ≈ 92,500 个 PS
每个 PS 约 60 KB → 5.38 GB 内存
```

#### 涉及的全部调用方（40+ 文件，99+ 处调用）：

| 文件 | PS 数量贡献 | 典型模式 |
|------|------------|---------|
| `server/services/structureSyncService.js` | 🔴 高 | 多次 `getConnection()` + 动态 IN 子句 |
| `server/services/ingestorService.js` | 🔴 高 | 大量动态 INSERT/批量写入 |
| `server/services/rulesService.js` | 🔴 高 | 频繁规则匹配查询 |
| `server/routes/rules.js` | 🟠 中 | GET/POST 规则 CRUD |
| `server/routes/admin.js` | 🟠 中 | 事务内多查询 |
| `server/middleware/authJwt.js` | 🟡 低 | 每次请求鉴权 |
| `server/utils/accountAccess.js` | 🟡 低 | 权限检查 |
| `server/services/dynamicScopeService.js` | 🟠 中 | 动态范围查询 |

相同封装也影响 `pool.query()`（第 68-76 行）和 `pool.getConnection()`（第 82-86 行）。

### 🔧 短期修复（代码层面，1-3 天内）

#### 3. 🔴 修复 `server/db/connection.js` 的 Prepared Statement 泄漏（核心）

**方案 A（推荐，改动最小）**：恢复 mysql2 原生 `pool.execute()` + 连接时设置时区

```javascript
// server/db/connection.js — 修复后的代码

import mysql from 'mysql2/promise'

const pool = mysql.createPool({
  host: process.env.DB_HOST || 'localhost',
  port: parseInt(process.env.DB_PORT || '3306'),
  user: process.env.DB_USER || 'root',
  password: process.env.DB_PASS || '',
  database: process.env.DB_NAME || 'fb_ad_brain',
  waitForConnections: true,
  connectionLimit: 25,
  queueLimit: 0,
  enableKeepAlive: true,
  keepAliveInitialDelay: 0,
  timezone: '+00:00',
  // ✅ 关键：限制每个连接的 PS 缓存数量，超出自动 deallocate 最旧的
  maxPreparedStatements: 100,  // 每个连接最多 100 个 PS
})

// ✅ 使用 pool.on('connection') 事件在连接创建时设置时区
// 这比每次 execute 都 SET time_zone 高效，且不会创建额外 PS
pool.on('connection', async (connection) => {
  try {
    await connection.execute("SET time_zone = '+00:00'")
  } catch (error) {
    logger.warn('设置数据库会话时区失败', { message: error.message })
  }
})

// ✅ 直接导出原生 pool，无需自定义封装
// pool.execute() 由 mysql2 内部管理 PS 缓存和自动清理
export default pool
```

**方案 B（更彻底，消除所有 PS 开销）**：使用 `pool.query()` 代替 `pool.execute()`

```javascript
// query() 不使用服务器端 prepared statement
// 改用客户端参数转义，无 PS 累积风险
// 缺点：轻微性能差异（通常可忽略）
```

**方案 C（紧急 hotfix，无需改代码）**：添加 `maxPreparedStatements` + 定期重置连接

```javascript
// 在现有 connection.js 的 rawPool 配置中添加：
const rawPool = mysql.createPool({
  // ... 现有配置 ...
  maxPreparedStatements: 50,  // ✅ 加这一行即可大幅缓解
})
```
