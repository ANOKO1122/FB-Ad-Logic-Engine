FB Ad-Intelligence Server 深度优化详案 (V2.0 极致落地版)
一、 数据架构层：基础稳固与极致频率控制
1. 数据搬运工 (Data Ingestor) 的 20-Batch 聚合引擎
分块并行策略：使用 lodash.chunk 将所有待同步广告 ID 按 20 个一组分块。
Batch 构造逻辑：按照 FB Batch API 规范，在一个 POST 请求中封装 20 个子请求。
字段定义：显式包含 spend, cpc, ctr, roas, actions, purchase, add_to_cart 等，时间范围锁定为 today。
归因窗口强制对齐 (Attribution Alignment)：请求中硬编码 use_account_attribution_setting=true，确保本地数据与 FB 后台 100% 像素级对齐，消除决策误差。
原子化同步会话 (Atomic Sync)：
在 ad_snapshots 增加 sync_session_id (时间戳)。
双缓冲逻辑：规则引擎只处理上一个“完整批次”的数据，防止处理到一半时，前 10 个账户是新的，后 20 个是旧的。
2. API 频率管理与自愈机制
弹性休眠 (Header 解析)：
解析 x-business-use-case-usage 字符串并 JSON.parse。
动态调整延迟：
负载 < 50%：休眠 500ms（全速冲刺）。
负载 50%-85%：休眠 2s（标准节奏）。
负载 > 85%：休眠 10s（熔断模式）并推送机器人。
竞速熔断 (Promise Race)：封装 fetchWithTimeout，设定 45 秒硬性超时。防止单账户网络 Pending 导致整个 Node 进程卡死，赋予系统“自愈”能力。
Token 智能熔断：全局维护 failureCount，连续 3 次返回 190 错误（Token 失效）立即进入 isSystemLocked 状态，停止所有搬运任务并报警。
3. 数据精准度保护
ID 精度保护：
DB 层：所有 ID 字段（Account/Ad/Rule）数据库设为 VARCHAR(50)，严禁使用 BIGINT。
传输层：后端引入 json-bigint，强制将 API 返回的长数字解析为 String 类型，杜绝尾数变 000 的精度丢失问题。
滑动窗口数据回溯 (Rolling Window Sync)：
逻辑：同步范围动态设定为 Today + Past 7 Days。
意义：利用 ON DUPLICATE KEY UPDATE 覆盖更新过去一周数据，补齐因 FB 归因延迟导致“迟到”的购买转化数。
热/冷数据分离策略 (Hot/Cold Data ETL)：
热数据 (Hot Data)：每 10 分钟轮询一次“今天 (Today)”的数据。这是规则引擎触发的唯一依据。
冷数据 (Cold Data)：每天早上 06:00 (账户当地时间) 将昨日数据最终快照汇总至 daily_stats 历史表，历史报表优先读取此表，不再消耗 API。
二、 规则配置层：颗粒度、时区与 SOP
1. 多层级精细指派
级联选择流：前端级联：账户 -> 广告系列 -> 广告组 -> 广告，将选中 ID 存入 target_ids。
运行时状态双重保险：扫描时强制附加逻辑：只计算 effective_status === 'ACTIVE' 的对象。即使规则绑定了 10 个广告，若运营在 FB 后台手动关了 3 个，引擎会自动跳过，不报错误。
2. 复合逻辑与时区对齐
时区处理 (Luxon 规范)：
存储账户 timezone_name (如 Asia/Shanghai)。
禁止使用 new Date()，所有“今日”数据判定必须基于 Luxon 计算的账户当地自然日。
复合判断 (AND/OR)：支持逻辑运算符配置。代码层利用 .every() 或 .some() 对存储在 JSON 中的多条件数组进行逻辑评估。
灵活时间窗口：支持 Today, Yesterday, Last 3 Days, 累计至今。
3. SOP 模板化与免责声明
JSON 模板：预设行业成熟的止损（Spend > 20 & Conv < 1）和扩量模板，运营一键填空。
前端免责提示：在页面显眼处提醒：“手动在 FB 后台开启已被系统关闭的广告前，请先在本系统暂停规则，否则会被再次关闭。”
三、 动作执行层：冲突仲裁与安全护栏
1. 动作冲突仲裁与优先级 (Conflict Resolution)
权重排序策略：
Level 1 (最高优先级)：PAUSE (暂停), ARCHIVE (归档), DELETE (删除)。
Level 2：DECREASE_BUDGET (减预算), DECREASE_BID (降出价)。
Level 3：INCREASE_BUDGET (加预算), INCREASE_BID (加出价)。
竞争择优：按 ad_id 分组，同一广告在同一扫描周期触发多条规则时，仅保留优先级最高的一个，直接丢弃低优先级动作。
2. 智能手动挂起与恢复 (Smart Mute)
数据结构：ad_snapshots 增加 mute_until (时间戳) 和 mute_reason 字段。
自动化干预检测：Ingestor 若发现广告状态变为开启，自动设置默认 2 小时优待期。
UI 交互：当运营在监控大屏点击“开启”时，弹窗询问挂起时长（1h、4h、次日 0 点、永久）。
引擎检查：扫描判定前执行 if (now < mute_until) return skip;。
3. 预算护栏与执行保护
预算安全护栏：加预算动作强制要求 max_daily_budget。新预算若超标，强制取最大值。
飞行前检查 (Pre-Flight Check)：
执行 PAUSE 动作前，先轻量级 GET 一次广告当前 status。
意义：若运营已手动关闭，则记录为“跳过”，避免 API 产生 Redundant Update 报错。
异常处理机制：包含除零保护 (clicks === 0)、空值检查、动作队列支持 3 次指数退避重试。
四、 反馈与监控层：透明度与 IM 管理
1. 详细执行审计
审计日志 (automation_logs)：记录触发时间、广告名、负责人、触发时刻的瞬时快照数据（例如：当时消耗 $25.3, ROI 为 0.8）、API 执行原文。
Dry Run (模拟运行)：支持 is_simulation 模式。判定成功后仅记日志并通知，不下发 API 指令。
2. IM 机器人通知与去噪 (Debouncing)
报警消息聚合：建立 60 秒内存缓冲区。
聚合逻辑：当 60 秒内有多个规则触发时，汇总为一条“聚合简报”发送。
通知内容：对接钉钉/飞书 Webhook，消息包含广告名、操作动作、负责人，利用关键词实现 @负责人 提醒。
3. 系统健康看板
实时监控指标：
各账户上次同步成功的时间。
规则引擎上一轮全量扫描的总耗时。
当前动作队列中积压的任务数。
开发者意义：快速发现局域网断网、代理失效或 Token 异常导致的“静默失败”。
