# 超时保险丝与异步任务说明

## 🔍 问题现象

**用户观察**：
- 执行时间：`16:02:16`（强制模式）
- 26个账户在5分钟后触发超时保险丝（`🚨 规则执行超时 (5m)，强制断开连接以释放锁！`）
- 但是后续又突然弹出很多日志（规则评估的输出）

---

## 📊 时间线分析

### 执行流程

1. **16:02:16** - 开始执行强制模式
   - 36个账户并发获取锁成功
   - 开始执行规则评估

2. **16:07:16**（5分钟后）- 超时保险丝触发
   - 26个账户超时，强制断开连接
   - 释放数据库锁

3. **16:07:16 之后** - 异步任务继续执行
   - 规则评估的异步任务还在继续执行
   - 输出日志（`🎯 规则限定目标广告` 等）

---

## 🔍 原因分析

### 超时保险丝的工作原理

```javascript
// 设置5分钟超时保险丝
const timeoutHandle = setTimeout(() => {
  if (connection) {
    console.error(`🚨 [${accountId}] 规则执行超时 (5m)，强制断开连接以释放锁！`)
    connection.destroy() // 💥 物理切断 TCP 连接 -> MySQL 自动释放锁
  }
}, RULE_LOCK_TIMEOUT_MS)  // 5分钟

// 执行业务逻辑
await ruleLogicFn(accountId)  // 规则评估（可能很慢）
```

### 问题根源

1. **超时保险丝触发**：
   - `connection.destroy()` 强制断开数据库连接
   - MySQL 自动释放锁（`GET_LOCK` 在连接断开时自动释放）

2. **异步任务继续执行**：
   - `ruleLogicFn(accountId)` 中的规则评估是异步操作
   - 即使数据库连接断开，**已经开始的异步任务可能还在继续执行**
   - 这些任务不依赖于数据库连接（它们可能在做数据库查询、计算等）

3. **后续日志输出**：
   - 规则评估完成后，输出日志（`🎯 规则限定目标广告` 等）
   - 这些日志是之前执行的延续

## ⚠️ 性能问题

### 核心问题

**规则评估太慢了**，导致：
- 5分钟内无法完成
- 触发超时保险丝
- 但异步任务还在继续执行

### 可能的原因

1. **数据库查询慢**：
   - 查询 `ad_snapshots` 表可能很慢
   - 特别是查询"规则限定目标广告"时

2. **规则评估逻辑复杂**：
   - 规则评估可能涉及大量计算
   - 每个账户都要评估规则

3. **并发压力**：
   - 36个账户并发执行
   - 数据库压力大

## 🎯 解决方案

### 方案1：优化规则评估性能（推荐）

1. **优化数据库查询**：
   - 添加索引
   - 优化 SQL 查询
   - 减少查询次数

2. **优化规则评估逻辑**：
   - 减少不必要的计算
   - 使用缓存
   - 批量处理

### 方案2：增加超时时间（临时方案）

如果规则评估确实需要较长时间，可以适当增加超时时间：

```javascript
const RULE_LOCK_TIMEOUT_MS = 10 * 60 * 1000  // 改为10分钟
```

### 方案3：优化并发控制

减少并发数，避免数据库压力过大：

```javascript
// 使用 p-limit 限制并发数
const limit = pLimit(10)  // 限制为10个并发
```
