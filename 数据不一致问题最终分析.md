# 数据不一致问题最终分析

## 🔍 验证结果分析

### 日志解读

日志中的 `30/19 条` 格式表示：`${archivedCount}/${expectedCount} 条`
- `archivedCount`：`daily_stats` 中已归档的记录数
- `expectedCount`：`ad_snapshots` 中期望归档的记录数（当前查询到的）

**示例**：
- `act_1311074497415630`: "30/19 条" → daily_stats有30条，ad_snapshots当前只有19条
- `act_1501999950830277`: "8/1 条" → daily_stats有8条，ad_snapshots当前只有1条
- `act_927139705822379`: "28/17 条" → daily_stats有28条，ad_snapshots当前只有17条

---

## ✅ 结论：这是正常的！

### 为什么数据不一致是正常的？

#### 1. **归档逻辑是正确的**
- 归档时取的是"最后快照"（`ORDER BY synced_at DESC`）
- `daily_stats` 中的数据是归档时的完整数据（30条、8条、28条）
- 这是**正确的**，因为归档是"最后快照"，不是实时数据

#### 2. **ad_snapshots 数据被覆盖是正常的**
- `ad_snapshots` 使用 `ON DUPLICATE KEY UPDATE`
- 唯一索引是 `uk_account_ad_date(account_id, ad_id, data_date)`
- 当今天同步数据时，如果 `data_date` 仍然是昨天（跨时区或同步时间问题），会**更新**昨天的记录
- 更新时会更新 `synced_at`，但 `data_date` 保持不变
- 这导致 `ad_snapshots` 中昨天的数据不完整（只有部分广告的数据）

#### 3. **系统设计是正确的**
- `ad_snapshots` 是**热数据表**，主要用于查询 Today 的数据
- `daily_stats` 是**冷数据表**，用于查询历史数据
- 历史数据应该从 `daily_stats` 查询，而不是 `ad_snapshots`
