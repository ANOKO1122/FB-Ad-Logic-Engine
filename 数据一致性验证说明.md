# 数据一致性验证说明

## 📋 验证目的

验证数据库热数据表和冷数据表的数据是否与 Facebook API 一致：
- **热数据（Today）**：允许15分钟误差
- **冷数据（Yesterday）**：必须完全一致

---

## 🚀 运行验证脚本

```bash
node server/scripts/verify-data-consistency.js
```

---

## 📊 验证逻辑

### 1. 热数据验证（Today）

**数据源对比**：
- **Facebook API**：查询 `today` 的 insights，过滤 `spend > 0`
- **数据库**：查询 `ad_snapshots` 表中 `data_date = 今天` 且 `spend > 0` 的数据

**验证标准**：
- ✅ **广告数量**：允许差异（15分钟内会同步）
- ✅ **Spend 金额**：允许 $0.01 的误差（浮点数精度）

**判断逻辑**：
- 如果广告数量差异 ≤ 1%（热数据差异），认为一致
- 如果 Spend 差异 ≤ $0.01，认为一致

---

### 2. 冷数据验证（Yesterday）

**数据源对比**：
- **Facebook API**：查询指定日期（昨天）的 insights，过滤 `spend > 0`
- **数据库**：查询 `daily_stats` 表中 `date = 昨天` 且 `spend > 0` 的数据

**验证标准**：
- ❌ **广告数量**：必须完全一致（不允许差异）
- ❌ **Spend 金额**：必须完全一致（只允许 $0.01 的浮点数精度误差）

**判断逻辑**：
- 如果广告数量完全一致，且 Spend 差异 ≤ $0.01，认为一致
- 否则认为不一致

## 🔍 验证结果解读

### 热数据不一致的可能原因

1. **15分钟误差范围内**（正常）：
   - 统一心跳每15分钟执行一次
   - 如果验证脚本在两次心跳之间执行，可能会有差异
   - 这是正常的，属于"热数据差异"

2. **超过15分钟误差**（需要关注）：
   - 同步任务可能失败
   - 需要检查同步日志

### 冷数据不一致的可能原因

1. **归档逻辑问题**：
   - 归档时可能遗漏了部分广告
   - 需要检查归档日志

2. **归档时机问题**：
   - 归档可能在数据更新之前执行
   - 需要检查归档窗口

3. **数据覆盖问题**：
   - `ad_snapshots` 中的数据可能被覆盖
   - 归档时查询的数据不完整

## ✅ 验证通过标准

### 热数据（Today）
- ✅ 广告数量差异 ≤ 1%（或差异在15分钟内会同步）
- ✅ Spend 差异 ≤ $0.01

### 冷数据（Yesterday）
- ✅ 广告数量完全一致
- ✅ Spend 差异 ≤ $0.01

---

## 🛠️ 如果验证失败

### 热数据不一致

1. **检查同步任务**：
   - 查看最近一次同步的日志
   - 确认是否有同步失败

2. **等待15分钟**：
   - 如果是15分钟误差范围内，等待下次同步后再验证

3. **手动触发同步**：
   ```bash
   node -e "import('./server/services/cronService.js').then(m => m.manualUnifiedHeartbeat())"
   ```

### 冷数据不一致

1. **检查归档日志**：
   - 查看最近一次归档的日志
   - 确认是否有归档失败

2. **重新归档**：
   ```bash
   node -e "import('./server/services/cronService.js').then(m => m.manualArchive())"
   ```

3. **检查归档逻辑**：
   - 确认归档查询是否使用 `data_date`
   - 确认归档时机是否正确

---

## 📝 注意事项

1. **时区问题**：
   - 验证脚本使用 UTC 时区计算日期
   - 账户时区可能不同，需要注意时区转换

2. **API 限制**：
   - Facebook API 有频率限制
   - 如果验证失败，可能是 API 限流导致

3. **数据延迟**：
   - Facebook API 的数据可能有延迟
   - 冷数据验证时，昨天的数据可能还在更新
