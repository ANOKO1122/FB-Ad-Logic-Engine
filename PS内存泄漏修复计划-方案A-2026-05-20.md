# Prepared Statement 内存泄漏修复计划

**日期**：2026-05-20  
**修复方案**：方案 A — 恢复 mysql2 原生 pool 方法 + `pool.on('connection')` 事件  
**关联诊断**：[服务器内存诊断报告-2026-05-20.md](./服务器内存诊断报告-2026-05-20.md)

---

## 一、问题回顾

### 现象
- MySQL 进程占用 **10.4 GB** 内存（正常运行仅需 ~500 MB）
- **92,647 个** Prepared Statement 未释放，占用 **5.38 GB**
- Swap 完全耗尽，服务器面临 OOM 风险

### 根因
`server/db/connection.js` 中自定义封装了 `pool.execute()`，每次调用手动 `getConnection()` → `connection.execute()` → `release()`，**绕过了 mysql2 原生的 PS 缓存管理机制**，导致 Prepared Statement 在 MySQL 服务端无限累积。

```
自定义封装:  每次查询 = 新建 PS → release 不清理 → 累积 92,647 个
原生 pool:   每次查询 = 查 PS 缓存 → 命中则复用 → 超限自动淘汰 → 最多 100×25=2,500 个
```

## 三、代码变更明细

### 文件：`server/db/connection.js`

#### 变更 1：连接池配置 — 添加 `maxPreparedStatements`

```diff
 const rawPool = mysql.createPool({
   host: process.env.DB_HOST || 'localhost',
   port: parseInt(process.env.DB_PORT || '3306'),
   user: process.env.DB_USER || 'root',
   password: process.env.DB_PASS || '',
   database: process.env.DB_NAME || 'fb_ad_brain',
   waitForConnections: true,
-  connectionLimit: 25,  // 从 10 增加到 25，支持多人并发
+  connectionLimit: 25,
   queueLimit: 0,
   enableKeepAlive: true,
   keepAliveInitialDelay: 0,
-  // 关键：告诉 mysql2 在 JS Date <-> 字符串 编解码时使用 UTC，
-  // 与我们在会话里 SET time_zone = '+00:00' 的策略保持一致
-  timezone: '+00:00'
+  timezone: '+00:00',
+  // 限制每个连接的 Prepared Statement 缓存数量
+  // 超出后 mysql2 自动 deallocate 最旧的 PS，防止 MySQL 服务端 PS 无限累积
+  maxPreparedStatements: 100
 })
```

> **说明**：`maxPreparedStatements` 是 mysql2 内置选项，限制每个连接最多缓存 100 个 PS。超出时自动淘汰最久未使用的。25 连接 × 100 = 最多 2,500 个 PS，约占 150 MB，完全可控。

---

#### 变更 2：添加 `pool.on('connection')` 事件 — 连接创建时设置时区

```diff
+// 新连接建立时自动设置会话时区为 UTC
+// 使用 pool.on('connection') 事件，仅在连接创建时触发一次
+// 比每次查询都 SET time_zone 更高效
+rawPool.on('connection', async (connection) => {
+  try {
+    await connection.execute("SET time_zone = '+00:00'")
+  } catch (error) {
+    logger.warn('设置新连接会话时区失败', { message: error.message })
+  }
+})
```

> **说明**：`pool.on('connection')` 在 mysql2 创建新连接时触发。连接归还池后复用时不触发。由于 MySQL 会话级 `SET time_zone` 在连接存活期间持续有效，无需每次查询前重复设置。配合池配置中已有的 `timezone: '+00:00'`（处理 JS Date 编解码），时区处理双重保障。

#### 变更 5：保留 `pool.getConnection` wrapper（不变）

```javascript
// ✅ 保留，理由：
// 1. 事务代码依赖此封装自动设置时区
// 2. pool.on('connection') 只对新连接触发，复用连接不会再次设置
// 3. getConnection 封装确保每次手动获取的连接时区正确
const originalGetConnection = rawPool.getConnection.bind(rawPool)
rawPool.getConnection = async function() {
  const connection = await originalGetConnection()
  await setConnectionTimezone(connection)
  return connection
}
```

## 五、时区处理验证

修复后时区设置的 **三重保障**：

```
第 1 层: pool 配置 timezone: '+00:00'
         └→ mysql2 驱动层：JS Date ↔ 字符串 编解码使用 UTC

第 2 层: pool.on('connection') → SET time_zone = '+00:00'
         └→ MySQL 服务端：NOW()、CURDATE() 等函数返回 UTC

第 3 层: getConnection wrapper → SET time_zone = '+00:00'（兜底）
         └→ 手动获取连接时确保时区正确（理论上第2层已覆盖）
```

## 附录：修复前后对比

### 修复前（`pool.execute()` 调用流程）

```
pool.execute(sql, params)
  └→ wrapper 拦截
       └→ rawPool.getConnection()        ← 从池取连接
            └→ SET time_zone = '+00:00'   ← 创建 PS #1（每次查询）
                 └→ connection.execute(sql) ← 创建 PS #2（累积泄漏）
                      └→ connection.release() ← PS 未清理，归还池
                           └→ 下次取值时 PS 仍在 → 新 SQL 创建新 PS → 无限累积
```

### 修复后（`pool.execute()` 调用流程）

```
pool.execute(sql, params)
  └→ mysql2 原生处理
       └→ 从池取连接（时区已由 on('connection') 设置）
            └→ 查询 PS 缓存
                 ├→ 命中 → 复用已有 PS ✅
                 └→ 未命中 → 创建新 PS
                      ├→ 缓存未满 → 缓存
                      └→ 缓存已满 → 淘汰最旧 PS → 缓存新 PS
                           └→ DEALLOCATE PREPARE 旧 PS ✅
       └→ 自动归还连接到池
```

**关键差异**：修复后 PS 数量有硬上限（100/连接），且旧 PS 会被主动 DEALLOCATE。
