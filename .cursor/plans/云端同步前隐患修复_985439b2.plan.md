# 云端同步前隐患修复执行方案

## 里程碑1：修复动态快照 fail-closed 分支崩溃

### [目标]

修复 `server/services/ruleEngineDispatcher.js` 中 `resolvedObjectType` 作用域错误，确保动态范围快照查询异常时按 fail-closed 返回空目标，而不是二次抛 `ReferenceError`。

### [具体问题]

在 `resolveTargetAdIdsForRule()` 中，`resolvedObjectType` 定义在 `try` 内，但 `catch` 中返回对象继续引用它。

```53:88:server/services/ruleEngineDispatcher.js
      const resolvedObjectType = (ruleTargetLevel || 'ad').toLowerCase()
      const [rows] = await pool.execute(
        `SELECT object_id
         FROM rule_matched_objects
...
    } catch (err) {
      ...
      return { targetLevel: resolvedObjectType, targetObjectIds: [], targetAdIds: [], queryCount: 0 }
    }
```

### [影响]

- 正常路径不受影响。
- 一旦 `rule_matched_objects` 查询异常，原本应跳过本规则本轮评估。
- 实际会触发 `ReferenceError: resolvedObjectType is not defined`，导致外层调度异常，破坏 fail-closed 语义。

### [解决方法]

1. 将 `resolvedObjectType` 提升到 `try` 外层，在 `if (useDynamicScope)` 之前或内部 `try` 之前定义。
2. `catch` 继续返回 `{ targetLevel: resolvedObjectType, targetObjectIds: [], targetAdIds: [], queryCount: 0 }`。
3. 保持动态范围读取失败时不回退到全账户广告，避免误扩大执行范围。

### [验证方法]

1. 新增/扩展 `server/tests/ruleEngineDispatcher.test.js`：

   - 模拟 `pool.execute()` 查询 `rule_matched_objects` 抛错。
   - 断言 `loadDataForAccount()` 不抛异常。
   - 断言该规则 `targetObjectIds` / `targetAdIds` 为空。

2. 跑测试：

   - `npm run test -- server/tests/ruleEngineDispatcher.test.js`

### [预期结果]

- 动态快照查询失败时，规则本轮安全跳过。
- 日志能看到快照读取失败原因。
- 不再出现 `ReferenceError`。

## 里程碑2：修复 campaign/adset 审计旧字段污染

### [目标]

避免 `campaign/adset` 同层动作审计中，旧兼容字段 `automation_logs.ad_id` 被写成字符串 `'null'`。

### [具体问题]

`queryRuleDataByLevel()` 聚合对象会返回 `ad_id: null`，这是正确的，因为目标对象已经是 `campaign/adset`。

```1169:1187:server/services/ruleDataService.js
    return {
      account_id: accountId,
      object_type: validLevel,
      object_id: objectId,
...
      ad_id: null,
      ad_name: null,
```

但 `actionExecutorService.js` 主写入路径中使用 `String(matchedAd.ad_id)`：

```868:873:server/services/actionExecutorService.js
      const insertResult = await db.insert(automationLogs).values({
        runId: runId || null,
        accountId: String(accountId),
        adId: String(matchedAd.ad_id),
```

### [影响]

- 新字段 `object_type/object_id` 是正确的，主功能不坏。
- 旧字段 `ad_id` 可能写成 `'null'`，后续按旧字段查询或统计时会产生误导。
- 多账户、多层级动作审计时，数据清洁度下降。

### [解决方法]

1. 将主路径改为安全兜底：`adId: String(matchedAd?.ad_id || '')`。
2. 同步检查 `adName` 与 Batch 路径，确保主路径、单规则 Batch、调度 Batch 口径一致。
3. 保持新字段 `objectType/objectId/objectName` 作为真实目标对象来源。

### [验证方法]

1. 新增/扩展 `server/tests/actionExecutor.auditLogging.test.js`：

   - 构造 `targetLevel='campaign'` 且 `matchedAd.ad_id=null`。
   - 断言插入 `automationLogs` 时 `adId` 为 `''`，不是 `'null'`。
   - 断言 `objectType='campaign'`、`objectId` 为真实 campaign id。

2. 跑测试：

   - `npm run test -- server/tests/actionExecutor.auditLogging.test.js`

### [预期结果]

- campaign/adset 审计日志旧字段不再出现字符串 `'null'`。
- 新字段继续准确记录真实操作对象。

## 里程碑3：补齐迁移幂等与半生效防护

### [目标]

避免云端执行迁移时出现“字段已存在导致脚本中断，后续索引未创建”的半生效状态。

### [具体问题]

本地已经验证过：`049` 曾出现 `summary_scope` 列存在，但 `idx_summary_scope` 索引不存在。原因是迁移不是幂等的：

```10:16:server/db/migrations/049_add_summary_scope_to_rule_execution_summaries.sql
ALTER TABLE rule_execution_summaries
  ADD COLUMN summary_scope VARCHAR(20) DEFAULT NULL ...
  AFTER status;

ALTER TABLE rule_execution_summaries
  ADD INDEX idx_summary_scope (summary_scope);
```

如果云端已有 `summary_scope` 列但没有索引，再执行该脚本会在 `ADD COLUMN` 报重复列，导致 `ADD INDEX` 不执行。

### [影响]

- 功能层可能可用。
- 查询统计性能和环境一致性不稳定。
- 云端部署时容易出现“执行一半”的迁移状态。

### [解决方法]

1. 新增一条幂等迁移，例如 `051_ensure_summary_scope_index.sql`：

   - 检查 `summary_scope` 列是否存在，不存在则补列。
   - 检查 `idx_summary_scope` 是否存在，不存在则补索引。

2. 不依赖重复执行 `049` 来补救。
3. 云端发布时优先执行幂等补丁迁移。

### [验证方法]

在本地与云端分别执行：

```sql
SELECT COLUMN_NAME
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'rule_execution_summaries'
  AND COLUMN_NAME = 'summary_scope';

SELECT INDEX_NAME
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'rule_execution_summaries'
  AND INDEX_NAME = 'idx_summary_scope';
```

### [预期结果]

- `summary_scope` 列存在。
- `idx_summary_scope` 索引存在。
- 迁移重复执行不会中断发布。

## 里程碑4：扩展 schema guard 覆盖范围

### [目标]

让启动自检不仅发现 `automation_logs` 缺列，也能发现 `summary_scope`、关键索引等迁移遗漏。

### [具体问题]

当前 `server/services/dbSchemaGuardService.js` 只检查 `automation_logs`：

```3:10:server/services/dbSchemaGuardService.js
export const REQUIRED_AUTOMATION_LOG_COLUMNS = [
  'run_id',
  'object_type',
  'object_id',
  'object_name',
  'preflight_mode',
  'explanation'
]
```

### [影响]

- 可以发现审计日志缺列。
- 发现不了 `rule_execution_summaries.idx_summary_scope` 缺失。
- 发现不了同层聚合索引缺失。

### [解决方法]

1. 将 `dbSchemaGuardService.js` 从单表检查扩展为统一 schema gate。
2. 增加检查项：

   - `automation_logs` 必需列。
   - `rule_execution_summaries.summary_scope`。
   - `rule_execution_summaries.idx_summary_scope`。
   - `rule_matched_objects.uq_rule_account_type_object` 与 `idx_account_rule_type`。
   - 同层聚合索引：`idx_daily_account_date_adset`、`idx_daily_account_date_campaign`、`idx_snap_account_data_date_adset`、`idx_snap_account_data_date_campaign`。

3. 启动时只告警，不阻断服务，避免云端临时 schema 漂移导致服务无法启动。

### [验证方法]

1. 扩展 `server/tests/dbSchemaGuardService.test.js`：

   - 模拟列缺失。
   - 模拟索引缺失。
   - 断言日志中输出 `missingColumns` 与 `missingIndexes`。

2. 跑测试：

   - `npm run test -- server/tests/dbSchemaGuardService.test.js`

### [预期结果]

- 服务启动时能明确提示缺失字段/索引。
- 云端同步后能第一时间发现迁移遗漏。

## 里程碑5：云端同步前迁移矩阵检查

### [目标]

在云端同步代码前，先确认云库与本地库的结构差异，避免上线后才发现缺字段或缺索引。

### [具体问题]

本地已出现 `048/050` 未生效、`049` 半生效的历史状态。云端项目尚未同步本次代码，云库更可能处于旧结构。

### [解决方法]

云端上线前执行迁移矩阵 SQL，判断 `045~051` 是否齐全：

```sql
-- 045 typed snapshot 索引
SELECT TABLE_NAME, INDEX_NAME
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'rule_matched_objects'
  AND INDEX_NAME IN ('uq_rule_account_type_object', 'idx_account_rule_type');

-- 046 同层聚合索引
SELECT TABLE_NAME, INDEX_NAME
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = DATABASE()
  AND (
    (TABLE_NAME = 'daily_stats' AND INDEX_NAME IN ('idx_daily_account_date_adset', 'idx_daily_account_date_campaign'))
    OR
    (TABLE_NAME = 'ad_snapshots' AND INDEX_NAME IN ('idx_snap_account_data_date_adset', 'idx_snap_account_data_date_campaign'))
  );

-- 047/048/050 automation_logs 字段
SELECT COLUMN_NAME
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'automation_logs'
  AND COLUMN_NAME IN ('object_type', 'object_id', 'object_name', 'preflight_mode', 'explanation');

-- 049/051 summary_scope 字段与索引
SELECT COLUMN_NAME
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'rule_execution_summaries'
  AND COLUMN_NAME = 'summary_scope';

SELECT INDEX_NAME
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'rule_execution_summaries'
  AND INDEX_NAME = 'idx_summary_scope';
```

### [验证方法]

- 缺字段：执行对应幂等迁移补字段。
- 缺索引：执行对应幂等迁移补索引。
- 补完后重跑矩阵 SQL。

### [预期结果]

- 云库结构与本地代码合同一致。
- 代码同步后不会因缺字段导致审计写入失败。
