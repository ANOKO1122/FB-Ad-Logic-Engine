# 审计日志缺列与链路日志补强执行方案

## 里程碑1：锁定根因并完成止血

### [目标]

确认当前问题是 **`automation_logs` 表缺少 `explanation` 列**，并把“动作已执行成功但审计日志入库失败”的现象转为可验证、可修复的数据库问题。

### [具体改动方案]

1. 以数据库现状为准，先做根因确认：

   - `SHOW CREATE TABLE automation_logs;`
   - `SELECT COLUMN_NAME FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 'automation_logs' AND COLUMN_NAME = 'explanation';`

2. 以代码合同为准，确认当前所有审计写入路径都在写 `explanation`：

   - [D:/projects/FB-Ad-Logic-Engine/server/services/actionExecutorService.js](D:/projects/FB-Ad-Logic-Engine/server/services/actionExecutorService.js)
   - [D:/projects/FB-Ad-Logic-Engine/server/services/cronService.js](D:/projects/FB-Ad-Logic-Engine/server/services/cronService.js)
   - [D:/projects/FB-Ad-Logic-Engine/server/db/schema.js](D:/projects/FB-Ad-Logic-Engine/server/db/schema.js)

3. 以迁移文件为准，确认应补执行的 DDL 为：

   - [D:/projects/FB-Ad-Logic-Engine/server/db/migrations/048_add_explanation_to_automation_logs.sql](D:/projects/FB-Ad-Logic-Engine/server/db/migrations/048_add_explanation_to_automation_logs.sql)

4. 先按 **运行库补齐 `048`** 作为止血动作，恢复 `automation_logs` 可写能力。

### [关键证据]

- 数据库查询返回 `Empty set`，说明当前运行库中 **确实不存在** `automation_logs.explanation`。
- 代码插入对象中明确包含 `explanation`，例如：
```810:833:server/services/actionExecutorService.js
      await db.insert(automationLogs).values({
        runId: runId || null,
        accountId: String(accountId),
        adId: String(matchedAd.ad_id),
        ...
        metricsSnapshot: metricsSnapshot,
        explanation: explanationPayload,
        actionType: action.type.toUpperCase(),
        ...
        triggeredAt: now
      })
```


### [验证/验收标准]

- `INFORMATION_SCHEMA.COLUMNS` 能查到 `explanation`。
- 再次执行 `rule_id=1202` 后，`automation_logs` 出现对应 `run_id` 的新记录。
- 前端“执行日志”页面能看到本次执行结果。

## 里程碑2：修复迁移链与环境一致性风险

### [目标]

让 `automation_logs` 的结构定义、迁移文件、实际数据库三者一致，解决“某些环境能写、某些环境缺列”的漂移问题。

### [具体改动方案]

1. 复核 `automation_logs` 相关迁移链：

   - [D:/projects/FB-Ad-Logic-Engine/server/db/migrations/013_add_run_id_to_automation_logs.sql](D:/projects/FB-Ad-Logic-Engine/server/db/migrations/013_add_run_id_to_automation_logs.sql)
   - [D:/projects/FB-Ad-Logic-Engine/server/db/migrations/016_automation_logs_triggered_at_timestamp.sql](D:/projects/FB-Ad-Logic-Engine/server/db/migrations/016_automation_logs_triggered_at_timestamp.sql)
   - [D:/projects/FB-Ad-Logic-Engine/server/db/migrations/047_add_generic_object_fields_to_automation_logs.sql](D:/projects/FB-Ad-Logic-Engine/server/db/migrations/047_add_generic_object_fields_to_automation_logs.sql)
   - [D:/projects/FB-Ad-Logic-Engine/server/db/migrations/048_add_explanation_to_automation_logs.sql](D:/projects/FB-Ad-Logic-Engine/server/db/migrations/048_add_explanation_to_automation_logs.sql)

2. 对照 [D:/projects/FB-Ad-Logic-Engine/server/db/schema.js](D:/projects/FB-Ad-Logic-Engine/server/db/schema.js) 中 `automationLogs` 定义，列出“运行库必须具备”的最小字段集合：

   - `run_id`
   - `object_type`
   - `object_id`
   - `object_name`
   - `preflight_mode`
   - `explanation`

3. 规划一个 **环境一致性自检**：

   - 启动时检查 `automation_logs` 是否具备以上关键列。
   - 若缺列，启动日志中输出：`tableName`、`missingColumns`、`expectedMigration`。

4. 规划一个 **发布前校验 SQL**，作为 Runbook 固定步骤，避免再次漏跑 migration。

### [设计意图]

当前问题不是单点代码 bug，而是 **代码合同升级了，数据库合同没有同步升级**。因此修复不能只停留在“手工跑一次 SQL”，还要补一层环境一致性防线。

### [验证/验收标准]

- 新环境或历史环境都能通过统一 SQL 校验关键列。
- 服务启动时若数据库缺列，会有明确告警，而不是等到用户执行规则时才暴露。

## 里程碑3：补齐三条审计写入链路的结构化错误日志

### [目标]

让后续任何 `automation_logs` 写入失败都能在 **一次日志查看** 中定位为字段缺失、长度超限、JSON 非法、约束失败或连接异常，避免再次出现“只看到写入失败，没有具体原因”。

### [具体改动方案]

1. 统一改造以下三条写入路径的 `catch`：

   - [D:/projects/FB-Ad-Logic-Engine/server/services/actionExecutorService.js](D:/projects/FB-Ad-Logic-Engine/server/services/actionExecutorService.js)
     - `executeActionsForAd()` 主写入路径
     - `executeActionsForRule()` 内部 `insertBatchAuditLog()`
   - [D:/projects/FB-Ad-Logic-Engine/server/services/cronService.js](D:/projects/FB-Ad-Logic-Engine/server/services/cronService.js)
     - `writeBatchStatusAuditLog()`

2. 将现有“只传 `error.message` 字符串”的写法改为 **结构化对象日志**，至少带上：

   - `message`
   - `operation: 'automation_logs.insert'`
   - `runId`
   - `ruleId`
   - `accountId`
   - `ownerId`
   - `objectType`
   - `objectId`
   - `adId`
   - `actionType`
   - `auditStatus`
   - `errorMessage`
   - `code`
   - `errno`
   - `sqlState`
   - `sqlMessage`
   - `stack`

3. 建议抽一个公共序列化函数，例如 `serializeDbError(err)` / `logAuditInsertFailure(context, err)`，供上述三条路径复用，保证字段名统一。
4. 避免直接记录超大 JSON，`actionPayload`、`metricsSnapshot`、`explanation` 只记录：

   - `payloadKeys`
   - `objectCounts`
   - `traceSize`
   - 必要时加截断后的摘要字符串

### [关键证据]

当前日志丢失细节的原因，在于调用方把异常当成普通字符串传入：

```834:836:server/services/actionExecutorService.js
    } catch (logError) {
      logger.error(`    ⚠️  写入审计日志失败:`, logError.message)
    }
```

而 [D:/projects/FB-Ad-Logic-Engine/server/utils/logger.js](D:/projects/FB-Ad-Logic-Engine/server/utils/logger.js) 的 `consoleFormat` 只会序列化 `meta` 对象；传字符串不会保留 `code/sqlMessage/stack`。

### [验证/验收标准]

- 故意制造一次写库失败时，日志中能直接看到 `sqlMessage` 或明确的缺列/约束原因。
- `actionExecutorService` 与 `cronService` 的错误日志字段名保持一致。

## 里程碑4：补齐动作请求链路日志，区分“动作失败”与“审计失败”

### [目标]

把请求链路拆成两个可观测阶段：

1. `Facebook API` 动作执行阶段
2. `automation_logs` 审计落库阶段

避免未来再出现“FB 动作已成功，但页面无日志”时只能靠人工猜测。

### [具体改动方案]

1. 在动作执行成功后、审计写入前，记录一个轻量成功事件：

   - `operation: 'fb_action.execute.success'`
   - `runId`
   - `ruleId`
   - `accountId`
   - `objectType`
   - `objectId`
   - `actionType`
   - `httpStatus`

2. 在动作执行失败时，补齐结构化字段：

   - `fbErrorCode`
   - `fbErrorSubcode`
   - `fbErrorType`
   - `httpStatus`
   - `apiPath`
   - `requestMethod`
   - `responseBodyPreview`

3. 在审计落库成功时，记录一个轻量事件：

   - `operation: 'automation_logs.insert.success'`
   - `runId`
   - `ruleId`
   - `objectType`
   - `objectId`
   - `automationLogId`（若可取）

4. 最终形成清晰链路：

   - `规则命中`
   - `FB 动作调用`
   - `FB 返回`
   - `审计写库`
   - `摘要写入`

### [设计意图]

当前链路里，`rule_execution_summaries` 已经证明“规则命中并执行过”，但 `automation_logs` 缺失会让前端误以为没执行。补链路日志后，可以快速分辨：

- 是 `FB` 没执行成功
- 还是 `FB` 成功了但审计没落库

### [验证/验收标准]

- 任一手动执行都有完整链路日志可追。
- 用户再次遇到“后台状态变了但日志页空白”时，运维能 1 次检索确定断点位置。

### [目标]

   - `SELECT COLUMN_NAME, COLUMN_TYPE FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 'automation_logs';`

2. 功能验证：

   - 前端再次执行 `rule_id=1202`
   - 检查 `rule_execution_summaries` 是否仍为 `account + rollup` 两条
   - 检查 `automation_logs` 是否新增对应 `run_id` 记录

3. 链路验证：

   - 控制台日志中同时看到：`fb_action.execute.success` 与 `automation_logs.insert.success`
   - 若故意制造错误，则看到完整 `sqlMessage/code/stack`

4. 发布清单：

   - 先补数据库 migration
   - 再发后端代码
   - 再跑一次手动执行回归
   - 最后更新 Runbook 与巡检 SQL

### [验证 SQL]

```sql
-- 1) explanation 列是否存在
SELECT COLUMN_NAME
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'automation_logs'
  AND COLUMN_NAME = 'explanation';

-- 2) automation_logs 关键列是否齐全
SELECT COUNT(*) AS ok
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'automation_logs'
  AND COLUMN_NAME IN (
    'run_id',
    'object_type', 'object_id', 'object_name', 'preflight_mode',
    'explanation'
  );

-- 3) 某次运行是否成功写入审计日志
SELECT id, run_id, rule_id, account_id, action_type, status, object_type, object_id, created_at
FROM automation_logs
WHERE run_id = ?
ORDER BY id DESC;

-- 4) 同一次运行的摘要记录
SELECT id, run_id, rule_id, status, matched_count, executed_count, failed_count, summary_scope, created_at
FROM rule_execution_summaries
WHERE run_id = ?
ORDER BY id DESC;
```

## 里程碑6：测试补强

### [目标]

让这次问题变成一个可自动回归的场景，而不是只靠人工肉眼排查。

### [具体改动方案]

1. 为 `automation_logs` 写入增加测试覆盖：

   - 缺列场景下，断言不会中断主执行链，但会输出结构化错误日志。
   - 正常场景下，断言 `explanation` 字段可成功写入。

2. 覆盖三条路径：

   - `executeActionsForAd`
   - `executeActionsForRule` 的 Batch 分支
   - `cronService.writeBatchStatusAuditLog`

3. 对启动自检增加最小测试：

   - 模拟缺列时输出明确告警
   - 列齐全时安静通过

### [验证/验收标准]

- 缺列问题可通过自动化测试稳定复现。
- 结构化错误日志字段有断言，避免以后又退化成“只有一条 message”。

## 最终交付结果

1. **数据库层**：`automation_logs` 与代码定义一致，`explanation` 不再缺失。
2. **执行层**：动作成功后，审计日志可稳定落库，前端“执行日志”恢复可见。
3. **可观测性层**：任何后续写库失败都能直接看到 `sqlMessage/code/stack`，不再需要二次猜测。
4. **流程层**：通过 migration 校验、自检、Runbook 与测试，降低环境漂移再次导致线上黑盒故障的概率。
