# 同层闭环规则执行完整修复方案

## 一、整体目标

### 1.1 目标架构

- `结构表` 只负责对象包含关系：[`server/services/structureSyncService.js`](server/services/structureSyncService.js)、[`server/db/migrations/023_create_structure_campaigns.sql`](server/db/migrations/023_create_structure_campaigns.sql)、[`server/db/migrations/024_create_structure_adsets.sql`](server/db/migrations/024_create_structure_adsets.sql)、[`server/db/migrations/014_create_structure_ads.sql`](server/db/migrations/014_create_structure_ads.sql)
- `热数据表` 只负责广告级事实指标：[`server/services/ingestorService.js`](server/services/ingestorService.js)、[`server/services/ruleDataService.js`](server/services/ruleDataService.js)
- `动态范围快照` 只负责记录规则当前作用对象：[`server/services/dynamicScopeService.js`](server/services/dynamicScopeService.js)
- `规则执行调度` 只负责按 `targetLevel` 选择正确的数据查询与评估路径：[`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js)、[`server/index.js`](server/index.js)
- `动作执行` 永远按 `targetLevel` 落到正确对象层：[`server/services/actionExecutorService.js`](server/services/actionExecutorService.js)
- `摘要表` 永远清晰区分“账户级明细摘要”和“汇总摘要”：[`server/services/ruleExecutionSummaryService.js`](server/services/ruleExecutionSummaryService.js)、[`server/services/cronService.js`](server/services/cronService.js)

### 1.2 最终执行语义

- `ad` 规则：直接使用 `ad_snapshots` 的广告级数据评估。
- `adset` 规则：通过 `structure_ads` 找到子广告，再按 `ad_set_id` 聚合后评估。
- `campaign` 规则：通过 `structure_ads` 找到子广告，再按 `campaign_id` 聚合后评估。
- `automation_logs` 只记录“真正产生动作的审计”。
- `rule_execution_summaries` 记录“执行事实”，包括 `matched`、`skipped`、`no_match`、`error`。

## 二、当前问题与根因

### 2.1 当前暴露的问题

- `campaign` 动态规则点击“运行此规则”后显示已执行，但 `matched_count=0`，`automation_logs` 为空，用户感知为“没反应”。
- `campaign` 静态规则（`use_dynamic_scope=0`，手动选择广告系列）在 `RULE_LEVEL_EXECUTION_V2=true` 且热数据已存在时，点击“运行此规则”仍然 `no_match`。
- `rule_execution_summaries` 在手动单账户执行时会写出两条语义接近的摘要，统计口径混乱。
- 前端规则页只基于 `automation_logs` 形成执行感知，`no_match` 完全不可见。
- 结构增量同步是近 3 天窗口，可能出现 `structure_ads` 已有子广告、但 `structure_campaigns` / `structure_adsets` 暂未补齐的情况。

### 2.2 已定位的根因链

- 动态范围快照按 `targetLevel` 写入 typed snapshot（`campaign` / `adset` / `ad`），见 [`server/services/dynamicScopeService.js`](server/services/dynamicScopeService.js)。
- 调度器在 `RULE_LEVEL_EXECUTION_V2=0` 时，会把规则有效层级强制压回 `ad`，见 [`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js)。
- 但动态范围分支对 `campaign/adset` 只返回 `targetObjectIds`，**没有再展开成 `targetAdIds`**，导致在 `V2=0` 下评估空集，最终 `no_match`。
- 对于静态目标规则（`use_dynamic_scope=0`）的 `campaign/adset + today` 场景，[`server/services/ruleDataService.js`](server/services/ruleDataService.js) 中的 `queryLevelAggregateRows()` 与 `queryLevelChildrenRows()` 在 `includeHistory=false && includeToday=true` 时，仍无条件以 `params = [accountId]` 起始，导致 today 分支 SQL 占位参数整体错位。
- 参数错位后，`ad_snapshots` 的 `account_id / data_date / campaign_id` 过滤条件绑定错误，`queryRuleDataByLevel()` 实际抛错；但 [`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js) 当前 catch 后直接 `cache.set(cacheKey, [])`，最终把“查询失败”伪装成“0 命中”。
- 当前日志 `logger.warn(\`[Dispatcher] queryRuleData(...)\`, err.message)` 只打印到前缀，未把 SQL/stack 结构化带出，增加了定位成本。
- 手动执行摘要路径在 [`server/services/cronService.js`](server/services/cronService.js) 中先写 `perAccountSummaries`，再写一条聚合摘要；单账户场景也如此，造成重复写入。
- 前端规则页只读取 `POST /api/rules/:id/execute` 返回的计数，没有解释 `status`，也没有读取摘要接口，见 [`src/views/RuleManager.vue`](src/views/RuleManager.vue)；日志页只读 `automation_logs`，见 [`src/views/Logs.vue`](src/views/Logs.vue)。

## 三、修复总策略

### 3.1 分期策略

1. **阶段A：止血恢复可用性**

- 固定 `RULE_LEVEL_EXECUTION_V2=true`
- 验证当前 `campaign` / `adset` 规则恢复命中

2. **阶段B：根修执行链路**

- 修 `ruleEngineDispatcher` 的动态范围回退逻辑
- 修 `queryRuleDataByLevel()` 在 `today` 场景下的 SQL 参数对齐问题
- 让 `V2=0` 场景也不再静默 `no_match`
- 让 `use_dynamic_scope=0 + targetLevel=campaign/adset + time_window=today` 不再把“查询失败”伪装成“0 命中”

3. **阶段C：引入 `summary_scope` 并修摘要模型**

- 区分 `account` 与 `rollup`
- 修统计翻倍与重复感知

4. **阶段D：前端执行反馈与可观测性**

- 让“已执行但 0 命中”成为显式可见结果

5. **阶段E：结构自愈、回归测试、运维闭环**

- 把这套思路固化为长期稳定机制

## 四、详细执行步骤

## 里程碑1：止血恢复可用性

### [目标]

先恢复 `campaign/adset` 同层规则的正确执行，避免业务继续被 `no_match` 阻塞；其中既包括动态范围规则，也包括手动选择目标的静态规则。

### [具体改动方案]

- 在运行环境中固定：
- `.env`：`RULE_LEVEL_EXECUTION_V2=true`
- 重启后端服务，使 [`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js) 走 `queryRuleDataByLevel()` 路径。
- 使用当前已定位的规则与账户做验收：
- `account_id = act_882617628155388`
- `rule_id = 1202`
- `campaign_id = 120245313451340015`

### [验证/验收标准]

- `rule_execution_summaries.matched_count > 0`
- `status` 不再为 `no_match`
- 若动作是模拟或安全动作，前端可见到明确反馈，或数据库可见到新增摘要/日志。

## 里程碑2：根修动态范围与层级执行兼容缺陷

### [目标]

让系统无论 `RULE_LEVEL_EXECUTION_V2` 开关如何，`campaign/adset` 规则都不会因为“目标展开缺失”或“today 同层聚合 SQL 参数错位”而静默落成 `no_match`。

### [具体改动方案]

- 修改 [`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js)：
- 在 `resolveTargetAdIdsForRule()` 中，若 `useDynamicScope=1` 且 `resolvedObjectType` 为 `campaign` / `adset`：
- 保持 `targetObjectIds` 返回原始快照对象 ID。
- 同时补一层通过 `structure_ads` 展开得到 `targetAdIds`。
- 保持两条语义并存：
- `V2=1`：优先用 `queryRuleDataByLevel()` 做同层聚合评估。
- `V2=0`：若仍保留兼容分支，则用上一步展开出的 `targetAdIds` 执行 ad 级评估；至少不能再静默空集。
- 若团队接受更强约束，可在后续阶段直接废弃 `V2=0` 的非 ad 层回退分支。
- 修改 [`server/services/ruleDataService.js`](server/services/ruleDataService.js)：
- 修正 `queryLevelAggregateRows()` 的参数组装顺序：
- `includeHistory=false` 时，today 分支不得继承多余的前置 `accountId`
- 仅在拼接 `historySql` 时写入 history 对应参数
- 仅在拼接 `todaySql` 时写入 today 对应参数
- 修正 `queryLevelChildrenRows()` 的同类问题，保证 children trace 查询与 aggregate 查询参数顺序一致。
- 补充日志：
- 调度器 catch 时至少打印 `err.message`、`err.code`、`err.sqlMessage` 或结构化错误对象
- 避免再次出现“日志看起来像 no_match，实际是 SQL 执行失败”的误判
- 补充本次复现场景作为固定验收样例：
- `rule_id = 1202`
- `account_id = act_882617628155388`
- `use_dynamic_scope = 0`
- `target_level = campaign`
- 手动选择 `campaign_id = 120245313451340015`
- 条件：`spend > 0`，`time_window = today`

### [关键文件]

- [`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js)
- [`server/services/dynamicScopeService.js`](server/services/dynamicScopeService.js)
- [`server/services/ruleDataService.js`](server/services/ruleDataService.js)
- [`server/utils/logger.js`](server/utils/logger.js)

### [验证/验收标准]

- `use_dynamic_scope=1 + targetLevel=campaign + RULE_LEVEL_EXECUTION_V2=0` 不再因为空目标集而 `no_match`
- `use_dynamic_scope=1 + targetLevel=campaign + RULE_LEVEL_EXECUTION_V2=1` 继续按 `queryRuleDataByLevel()` 命中
- `use_dynamic_scope=1` 且快照为空时仍保持 fail-closed，不误用 union 全量数据
- `use_dynamic_scope=0 + targetLevel=campaign + time_window=today + RULE_LEVEL_EXECUTION_V2=true` 在热数据存在时能正常命中，不再因为 SQL 参数错位落成 `no_match`
- 日志中可以区分“真正 0 命中”和“数据查询失败”，至少能看到可检索的错误字段

## 里程碑3：引入 `summary_scope`，完整修复摘要模型

### [目标]

让 `rule_execution_summaries` 能明确区分“账户级执行事实”和“汇总级执行事实”，消除单账户双写歧义与统计翻倍问题。

### [具体改动方案]

- 新增 migration：
- 位置：[`server/db/migrations/`](server/db/migrations/)
- 新增字段：`summary_scope`
- 推荐枚举：
- `account`：每个账户一条明细摘要
- `rollup`：手动执行后生成的汇总摘要
- 修改 [`server/services/ruleExecutionSummaryService.js`](server/services/ruleExecutionSummaryService.js)：
- `insertRuleExecutionSummary()` 增加 `summary_scope`
- 修改 [`server/services/cronService.js`](server/services/cronService.js)：
- `writeSummariesAfterArbitration()` 写入 `account`
- `executeSingleRule()`：
- `perAccountSummaries` 写入 `account`
- 聚合摘要写入 `rollup`
- 修改 [`server/routes/system.js`](server/routes/system.js)：
- `GET /api/rule-execution-summaries` 支持读取与筛选 `summary_scope`
- `GET /api/rule-execution-summaries/stats` 明确统计口径：
- 默认只统计 `account`
- 或显式支持 `summary_scope` 过滤
- 历史数据处理策略：
- `account_id='multi'` 的老数据可直接回填 `rollup`
- 旧的单账户重复汇总行无法百分百无损区分，采用“新数据完全正确、旧数据保守兼容”策略；如业务必须清理历史，可追加一次性修复脚本。

### [关键文件]

## 里程碑4：前端把“无命中”做成可见结果

### [关键文件]

### 8.2 调度器错误日志可观测性不足（已修复）

- 问题：`queryRuleData(...)` 失败时日志在运行态可能只剩前缀，难以区分 `no_match` 与 SQL 失败。
- 解决：

- 同时将摘要写入失败日志改为对象输出，避免字符串参数拆分污染日志。
- 关键文件：
- [`server/services/ruleEngineDispatcher.js`](server/services/ruleEngineDispatcher.js)
- [`server/services/ruleExecutionSummaryService.js`](server/services/ruleExecutionSummaryService.js)

### 8.3 摘要查询路由异常作用域风险（已修复）

- 问题：`/api/rule-execution-summaries` 路由 `catch` 中引用 `whereClause/params` 存在作用域风险，可能掩盖原始异常。
- 解决：
- 将 `whereClause`、`params` 提升到 `try` 外层定义，保证 `catch` 访问安全。
- 关键文件：
