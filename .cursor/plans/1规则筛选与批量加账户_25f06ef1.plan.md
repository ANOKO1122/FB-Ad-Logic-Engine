# 规则筛选与批量加账户执行方案

## 目标

本次改动分成两块，但它们都围绕同一个业务本质：让“规则”和“账户”的绑定关系更可运营。

- 需求一：管理员能在负责人筛选里看到并筛选“负责人为无”的规则，也就是管理员自己创建、`users.owner_id IS NULL` 的那批规则。
- 需求二：规则管理页支持手动勾选多个规则卡片，然后把新接入系统的广告账户批量追加到这些规则上，让这些规则尽快对新账户生效。

从架构上看，这不是“重做一套规则系统”，而是在现有单条规则保存链路外面，再包一层更高效的批量操作壳。这样风险最小，也最符合你当前项目的演进方式。

## 现状依据

已确认的现有实现如下：

- 前端负责人筛选已经有多选 UI 骨架，但只支持真实负责人 ID，不支持“无”这一类虚拟维度：[/root/work/FB-Ad-Logic-Engine/src/views/RuleManager.vue](/root/work/FB-Ad-Logic-Engine/src/views/RuleManager.vue)
- 规则列表接口当前只接收 `ownerIds`，并在服务层按 `users.owner_id IN (...)` 过滤，所以 `NULL` 永远不会命中：[/root/work/FB-Ad-Logic-Engine/server/routes/rules.js](/root/work/FB-Ad-Logic-Engine/server/routes/rules.js)、[/root/work/FB-Ad-Logic-Engine/server/services/rulesService.js](/root/work/FB-Ad-Logic-Engine/server/services/rulesService.js)
- 单条规则保存时，前端会提交 `targetAccounts`、`targetIds`、`target_by_account`；后端 PUT 路由已有账户权限校验；服务层会把 `target_ids` 归一化并同步 `target_by_account`：[/root/work/FB-Ad-Logic-Engine/src/views/RuleManager.vue](/root/work/FB-Ad-Logic-Engine/src/views/RuleManager.vue)、[/root/work/FB-Ad-Logic-Engine/server/routes/rules.js](/root/work/FB-Ad-Logic-Engine/server/routes/rules.js)、[/root/work/FB-Ad-Logic-Engine/server/services/rulesService.js](/root/work/FB-Ad-Logic-Engine/server/services/rulesService.js)
- 可选广告账户列表已经有统一来源：前端 `facebookApi.getAdAccounts()` 调 `/api/accounts`，后端会按当前用户权限返回可见账户：[/root/work/FB-Ad-Logic-Engine/src/services/facebookApi.js](/root/work/FB-Ad-Logic-Engine/src/services/facebookApi.js)、[/root/work/FB-Ad-Logic-Engine/server/index.js](/root/work/FB-Ad-Logic-Engine/server/index.js)

关键约束也已经确认：

- 批量加账户是“手动点选哪些规则就改哪些规则”，不是整页自动全选模式。
- 批量接口采用 `best-effort`：请求体合法时，单条规则失败不回滚其它已成功规则。
- 动态筛选规则在批量加账户后，允许触发“本地异步 refresh”；这个 refresh 只会重算 `rule_matched_objects`，不会触发轨道二，也不会额外调用 Facebook API。
- 若目标账户结构数据不新鲜，不承诺“立刻生效”，而是返回 `pending_data` 提示，等待统一心跳/结构同步后生效。
- 已存在账户时静默合并，并返回每条规则本次真正新增的 `added` 数量。
- `targetLevel` 不同可以一起批量处理，因为本次追加的是账户维度，不改层级。

## 设计原则

1. 尽量复用现有单条规则更新模型，不重造字段语义。
2. 把“批量”放在接口层和页面交互层，规则本身的数据结构不做大改。
3. 权限检查以后端为准，前端只做体验优化，不能把安全寄托在前端按钮禁用上。
4. 对用户来说，“新增账户”应该是追加而不是替换；对系统来说，必须保证去重、可审计、可回滚定位。

## 硬合同（先定死）

### 合同 1：批量接口语义

- `best-effort = 请求体合法后，进入逐规则编排；单条规则失败不回滚其它已成功规则`
- 整体请求只在下列场景返回 4xx：
  - `ruleIds/accountIds` 非法
  - 当前用户对传入 `accountIds` 中任一账户无权访问
- 进入逐规则编排后，接口固定返回 `200`，并在 `results[]` 中返回每条规则的真实结果

### 合同 2：动态规则生效语义

- 静态规则：
  - 规则保存成功后立即生效，因为执行账户集合直接读取 `targetAccountIds/target_by_account/accountId`
- 动态规则：
  - 若目标账户结构数据新鲜，批量接口在保存成功后调用本地异步 `scheduleDynamicScopeRefreshForRule(...)`
  - 该 refresh 仅重算 `rule_matched_objects` 和规则动态状态，不触发轨道二，不调用 Facebook API
- 结构数据不新鲜：
  - 规则绑定先保存成功
  - 响应中返回 `pendingDataAccounts`
  - 文案明确提示“账户已追加，待结构同步后生效”

### 合同 3：结构数据新鲜度

- 新鲜度判断字段：`structure_sync_status.last_heartbeat_data_update_at`
- 默认阈值：`30` 分钟
- 判定结果：
  - `fresh`：允许本地异步 refresh
  - `stale/missing`：不自动 refresh，不触发轨道二，返回 `pending_data`

### 合同 4：结果返回结构

- `summary.requested` = 请求中的规则数
- `summary.updated` = 至少新增了 `1` 个账户的规则数
- `summary.unchanged` = `added=0` 的规则数
- `summary.failed` = 逐规则执行失败数
- `summary.pendingDataRules` = 因结构数据不新鲜而未触发本地 refresh 的规则数
- `results[]` 至少返回：
  - `ruleId`
  - `status=updated|unchanged|forbidden|not_found|failed`
  - `added`
  - `skipped`
  - `pendingDataAccounts`
  - `code`

## Milestone 1：管理员“负责人无”筛选

### 为什么这么做

把“负责人无”当成一个独立筛选维度，而不是硬塞进真实负责人 ID 列表，是因为它在数据库里对应的是 `NULL`，底层语义和正常 `owner_id=123` 不一样。数据库里 `NULL` 不能靠 `IN (...)` 命中，所以必须单独建表达式。

### 参数契约

建议新增一个显式参数，而不是偷塞 `0` 到 `ownerIds`。

- `ownerIds=1,2,3`：筛真实负责人
- `includeNoOwner=1`：包含 `users.owner_id IS NULL` 的规则
- 二者都不传：全部规则
- 二者同时传：`owner_id IN (...) OR owner_id IS NULL`

这样做的好处是：语义清楚、以后看日志一眼能懂、避免把 `0` 这种哑值扩散到多处逻辑。

- [/root/work/FB-Ad-Logic-Engine/server/routes/rules.js](/root/work/FB-Ad-Logic-Engine/server/routes/rules.js)
- [/root/work/FB-Ad-Logic-Engine/server/services/rulesService.js](/root/work/FB-Ad-Logic-Engine/server/services/rulesService.js)

### 前端改动点

- 在现有负责人下拉中增加一个固定项：`无（管理员创建）`
- 新增独立状态，例如 `includeNoOwner`
- `ownerFilterDisplayValue` 文案拼接时支持展示“无”
- `loadRules()` 组装 URL 时追加 `includeNoOwner=1`
- “清除筛选(全选)”时同时清空 `selectedOwnerIds` 和 `includeNoOwner`

### 后端改动点

- 路由层解析 `includeNoOwner`
- service 层把管理员过滤条件改成三分支：
  - 只选负责人：`inArray(users.ownerId, ownerIds)`
  - 只选无：`isNull(users.ownerId)`
  - 两者都选：`or(inArray(...), isNull(...))`
- 非管理员忽略该参数，避免被手工调用绕过

2. 只选“无”，应只看到管理员创建、卡片显示负责人为“未分配/无”的规则。
3. 同时选某负责人和“无”，应看到两类规则并集。
4. 普通用户登录时，不应出现这个筛选项。

## Milestone 3：批量添加账户弹窗

### 为什么复用现有账户来源

新增账户最终还是“用户有权限访问的广告账户”。既然现有创建/编辑规则时已经有 `/api/accounts` 这条成熟链路，就不要再另开一套账户选择来源，否则会出现“弹窗能选，但编辑规则页不能选”的口径分裂。

### 交互定义

- 弹窗内展示当前用户可选的广告账户列表
- 支持多选账户
- 点击确认后，把这些账户追加到已勾选规则
- 弹窗内应展示一句风险提示：这是“追加”不是“替换”

### 数据语义

这一步追加的是 `targetAccountIds`，不是替换整条规则。

推荐合并规则：

- 新 `targetAccountIds = 去重(old.targetAccountIds + 新选账户)`
- `accountId` 保持原值不变，只在原规则缺失主账户时，用合并后数组第一项兜底
- `targetIds`、`target_by_account` 保持原值不清空

这样做的原因是：

- 你的业务目标是让规则对新账户“起作用”，底层判定账户适用性时，系统已经会读取 `targetAccountIds` / `target_by_account` / `accountId` 的并集口径
- 如果我们贸然改 `targetIds` 或清空 `target_by_account`，反而可能破坏已有精细对象绑定

这一点可以从执行口径里看出来：[/root/work/FB-Ad-Logic-Engine/server/services/ruleEnableGateService.js](/root/work/FB-Ad-Logic-Engine/server/services/ruleEnableGateService.js) 与动态范围服务都优先基于 `target_by_account`、其次 `targetAccountIds`、最后 `accountId` 取账户集合。

### 生效时点文案

- 静态规则：
  - 成功追加后即可进入执行账户集合，属于“保存成功即生效”
- 动态规则：
  - 若账户结构数据新鲜，显示“已追加，后台正在刷新动态范围”
  - 若账户结构数据不新鲜，显示“已追加，待结构同步后生效”
- 弹窗说明必须避免使用绝对表述“全部规则立刻生效”

### 本阶段验证

1. 打开弹窗时能加载当前用户可见账户。
2. 多选几个账户后，确认按钮可用。
3. 没选规则或没选账户时，按钮禁用并有明确提示。
4. 选中包含动态规则的多条规则时，弹窗说明能区分“后台刷新”和“待同步生效”两种口径。

## Milestone 4：后端批量追加账户接口

### 接口建议

新增接口：`POST /api/rules/batch/add-accounts`

请求体建议：

```json
{
  "ruleIds": [101, 102, 103],
  "accountIds": ["act_1", "act_2"]
}
```

响应体建议：

```json
{
  "message": "批量添加完成",
  "summary": {
    "requested": 3,
    "updated": 2,
    "unchanged": 1,
    "failed": 0,
    "pendingDataRules": 1
  },
  "results": [
    {
      "ruleId": 101,
      "status": "updated",
      "added": 2,
      "skipped": 0,
      "pendingDataAccounts": [],
      "code": "OK"
    },
    {
      "ruleId": 102,
      "status": "unchanged",
      "added": 0,
      "skipped": 2,
      "pendingDataAccounts": ["act_2"],
      "code": "PENDING_DATA"
    }
  ]
}
```

### 为什么单独建接口，而不是前端循环调 3 次 PUT

因为前端循环调用会有三个问题：

- 错误处理分散，用户只会看到一串 toast，不知道哪条规则成功哪条失败
- 多条请求并发时，审计和日志不容易归总
- 后端无法统一输出 `added/skipped` 统计

所以更稳的做法是：后端提供一个“批量编排接口”，内部逐条复用既有规则更新能力。

### 后端处理流程

建议放在：[/root/work/FB-Ad-Logic-Engine/server/routes/rules.js](/root/work/FB-Ad-Logic-Engine/server/routes/rules.js)

处理步骤：

1. 校验 `ruleIds`、`accountIds` 为非空数组并做去重
2. 预校验每个 `accountId` 当前用户是否有权访问

   - 普通用户：`account_mappings.owner_id = req.user.owner_id AND is_active = 1`
   - 管理员：`account_mappings.is_active = 1`

3. 一次性读取 `structure_sync_status.last_heartbeat_data_update_at`，计算每个账户的 `fresh|stale|missing`
4. 逐条读取规则：复用 `rulesService.getRuleById(...)` 做规则权限校验
5. 对每条规则计算：

   - `oldAccounts`
   - `mergedAccounts`
   - `added = mergedAccounts.length - oldAccounts.length`
   - `skipped = 请求账户数 - added`

   - `pendingDataAccounts = mergedAccounts 中 freshness != fresh 的账户`

6. 调用 `rulesService.updateRule(...)` 更新 `targetAccountIds`；如主账户缺失再补 `accountId`
7. 若该规则 `use_dynamic_scope=1` 且 `pendingDataAccounts.length === 0`，调用 `scheduleDynamicScopeRefreshForRule(ruleId, { trigger: 'batch_add_accounts' })`
8. 汇总结果并返回给前端

### 为什么这里允许本地异步 refresh

- 现有 `scheduleDynamicScopeRefreshForRule(...)` 只会基于本地结构表重算 `rule_matched_objects`
- 它不会走 `refreshStructureIfDirtyBeforeRules(...)`
- 它不会触发轨道二 `fastSyncStructureForAccount(...)`
- 因此它属于“本地数据库侧 refresh”，不是“外部 API 侧同步”

### 为什么不在这里触发轨道二

- 批量加账户是高频运营动作，不能把一次界面操作放大成不确定时长的外部 fan-out 请求
- 轨道二本质是“拉结构数据”，职责属于统一心跳/结构同步，不属于规则绑定保存链路
- 若绑定保存链路直接触发轨道二，接口时延、失败面、重试复杂度都会显著上升

### 为什么不直接批量 SQL 更新

因为规则更新不是只改一列这么简单：

- 现有 `updateRule()` 会写 `rule_history`
- 会经过启用前最小配置校验
- 未来如果规则更新链路再加审计/事件，总批量接口也能自然继承

所以批量接口更适合做“循环编排器”，而不是绕过 service 层直接 SQL 改库。

### 特别注意

当前 `updateRule()` 只有在传入 `targetIds` 时才会触发 `syncTargetByAccount()`，因此这次批量追加如果只改 `targetAccountIds`，不会误动已有 `target_by_account`，这正好符合本次“只追加账户、不改对象明细”的目标。

### 错误处理合同

- 整体 `400`：
  - `ruleIds/accountIds` 非法
  - 账户数组为空
- 整体 `403`：
  - 传入账户中存在当前用户无权访问的账户
- 整体 `200`：
  - 请求体合法且账户权限通过后，进入逐规则编排
  - 单条规则失败写入 `results[]`，不回滚其它已成功条目

### 本阶段验证

1. 普通用户传入不属于自己负责人的账户，应返回 `ACCOUNT_FORBIDDEN`。
2. 批量传入 3 条规则和 2 个账户，响应中每条规则都有 `added/skipped`。
3. 对已包含该账户的规则重复追加，`added=0`，但整体请求成功。
4. 动态规则仅触发本地异步 refresh，不触发轨道二，不额外调用 Facebook API。
5. 结构数据不新鲜时，响应中返回 `pendingDataAccounts`，且不触发本地 refresh。

## Milestone 5：前后端联调与结果反馈

### 前端联调点

- 弹窗确认后调用批量接口
- 成功后关闭弹窗、清空勾选、重新 `loadRules()`
- 用汇总提示用户：成功多少条、每条新增多少账户
- 不主动调用手动 `refreshDynamicScopeForRule`
- 根据接口返回结果展示两类提示：
  - `updated + 后台刷新中`
  - `updated + 待结构同步后生效`

### 用户体验建议

结果提示可分两层：

- 第一层：总提示，例如“已为 5 条规则追加账户”
- 第二层：明细，例如“规则 A 新增 2 个，规则 B 新增 0 个（已存在）”
- 若存在 `pendingDataAccounts`，额外提示“部分动态规则待结构同步后生效”

如果页面空间紧张，可以只在弹窗底部展示一次明细结果，不必给每张卡片单独打标。

### 本阶段验证

1. 批量追加后刷新列表，规则卡片账户信息应已更新。
2. 关闭批量模式再进入，不应残留上一次勾选状态。
3. 动态筛选开启的规则在批量追加后，若结构数据新鲜，应看到“后台刷新中”或稍后更新后的匹配状态。
4. 结构数据不新鲜时，不应出现“已立刻生效”的误导提示。

## 关键风险与应对

- 风险 1：管理员“无”筛选和现有 `ownerIds` 逻辑耦合，容易把“未传”和“只选无”混淆。
  - 应对：单独引入 `includeNoOwner`，不复用 `ownerIds` 哑值。
- 风险 2：批量追加如果误覆盖 `targetIds` / `target_by_account`，会破坏已有对象级规则范围。
  - 应对：本次仅合并 `targetAccountIds`，不主动清空对象维度字段。
- 风险 3：前端多选状态和筛选条件切换耦合，可能误把旧列表选择带到新列表。
  - 应对：列表重载、筛选切换、退出批量模式时统一清空 `selectedRuleIds`。
- 风险 4：普通用户批量修改的权限边界和现有单条编辑口径不一致。
  - 应对：严格复用 `getRuleById()` / `updateRule()` 当前共享负责人维度的鉴权逻辑，不另写一套。
- 风险 5：动态规则在结构数据陈旧时被错误宣传为“已立刻生效”。
  - 应对：引入 `pending_data` 合同，前后端统一提示“待结构同步后生效”。
- 风险 6：一次批量操作被误扩展为轨道二外部同步，导致接口时延和失败面骤增。
  - 应对：明确只允许本地异步 refresh，不允许从批量接口直接触发轨道二。

补充：

改动 1：全选/取消全选按钮

原理

这是一个状态驱动的二元切换。核心关系是：

判断条件：selectedRuleIds.length === rules.length（当前已选数量是否等于列表总数）

全选：把当前 rules 数组里每条规则的 id 全部塞进 selectedRuleIds

取消全选：直接把 selectedRuleIds 重置为空数组 []

为什么用 rules（当前列表）而不是全部规则？因为用户看到的是经过筛选后的列表——如果用户先按"负责人无"筛出了 3 条规则，点"全选"就应该只选这 3 条，而不是后台数据库里的几十条。这和批量操作的语义一致：你操作的永远是"当前列表里能看到的规则"。

rules 是 Vue 的响应式数组，筛选条件一变它就会自动更新，所以按钮文案和全选范围都能跟着联动，不需要额外监听。

改动 2：点击整张卡片即选中

原理

这是一个事件委托 + 交互元素排除的模式。

给 .rule-card 这个顶层容器绑了 @click，然后用 event.target.closest() 做了一件事：判断用户点击的位置是否落在交互元素上。排除列表是：

button, input, select, textarea, label, a

工作流程：

用户在卡片上某处点击

onCardClick 被触发

先判断 bulkMode，非批量模式直接 return——不影响正常浏览

调用 event.target.closest('button, input, ...') 向上查找，如果命中说明点在交互元素上，return 跳过
没命中，说明点在了卡片的空白区域（标题、描述文本等），触发 toggleRuleSelection
