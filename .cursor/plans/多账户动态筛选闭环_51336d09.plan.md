# 多账户动态筛选闭环改造计划

## 目标与口径

- 规则多选账户时，动态筛选刷新应覆盖该规则全部目标账户（你已确认）。
- `max_dynamic_matches` 采用按账户判定（你已确认），即每个账户独立 `NORMAL/ERROR_*`。
- 执行层继续按 `rule_matched_objects(rule_id, account_id)` 读取，做到每账户快照与执行一致。

## 变更范围

- 后端动态刷新核心：[d:\projects\FB-Ad-Logic-Engine\server\services\dynamicScopeService.js](d:\projects\FB-Ad-Logic-Engine\server\services\dynamicScopeService.js)
- 规则 API 触发逻辑（TriggerB/TriggerC）：[d:\projects\FB-Ad-Logic-Engine\server\routes\rules.js](d:\projects\FB-Ad-Logic-Engine\server\routes\rules.js)
- 执行分发读取（校对无需大改）：[d:\projects\FB-Ad-Logic-Engine\server\services\ruleEngineDispatcher.js](d:\projects\FB-Ad-Logic-Engine\server\services\ruleEngineDispatcher.js)
- 前端按钮调用参数（若需最小调整）：[d:\projects\FB-Ad-Logic-Engine\src\views\RuleManager.vue](d:\projects\FB-Ad-Logic-Engine\src\views\RuleManager.vue)
- 测试补齐：[d:\projects\FB-Ad-Logic-Engine\server\tests\ruleEngineDispatcher.test.js](d:\projects\FB-Ad-Logic-Engine\server\tests\ruleEngineDispatcher.test.js)、[d:\projects\FB-Ad-Logic-Engine\server\tests\rules.test.js](d:\projects\FB-Ad-Logic-Engine\server\tests\rules.test.js)

## 实施步骤

1. 在 `dynamicScopeService` 抽出“规则涉及账户”解析函数（优先 `target_by_account` 非空账户，其次 `target_account_ids`，最后回退 `account_id`）。
2. 将刷新入口从“按单账户查规则”扩展为“按规则+账户集”可复用流程：

- 保留现有 `refreshDynamicTargetsForAccount(accountId)`（供 TriggerA）；
- 新增 `refreshDynamicTargetsForRule(ruleId)`，循环刷新该规则全部目标账户；
- 仍按账户内事务写入，状态按账户判定。

3. 修正手工目标展开逻辑，支持 `targetIds` 的 `act_xxx:id` 多账户格式，确保当前账户只使用本账户目标，避免跨账户串值。
4. 调整 TriggerB：规则保存后不再只防抖主 `accountId`，而是对该规则全部目标账户逐个防抖刷新。
5. 调整 TriggerC：当传 `ruleId` 时默认刷新该规则全部目标账户；传 `accountId` 时维持单账户刷新（兼容旧用法）。
6. 校对 Dispatcher：确认多账户执行时每个账户都只读取该账户快照（当前已具备），仅补测试覆盖。
7. 增加自动化测试：

- 多账户规则下，`ruleId` 触发手动重算会覆盖多个账户；
- `max_dynamic_matches` 按账户独立生效；
- `targetIds=act:id` 情况下当前账户解析正确。

## 验证清单（你手工执行）

- 场景A：多账户规则（A/B）保存后，日志应出现两个账户的 TriggerB 刷新记录。
- 场景B：TriggerC 仅传 `ruleId`，返回应包含多账户刷新结果汇总。
- 场景C：A 账户超限、B 账户正常时：A 状态 `ERROR_OVERSIZE`，B 状态 `NORMAL` 且快照可执行。
- 场景D：单条执行分别在 A/B 账户，仅命中各自快照，不跨账户。

## 风险与兼容

- 维持 TriggerA 账户级入口不变，避免影响现有每分钟调度路径。
- TriggerC 保持 `accountId` 入参兼容，新增 `ruleId->多账户` 语义。
- 先补测试再扩展行为，降低回归风险。
