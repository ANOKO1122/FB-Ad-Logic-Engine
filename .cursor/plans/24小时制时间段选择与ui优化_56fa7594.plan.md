# 24 小时制执行时间段选择与 UI 优化方案

## 一、现状与问题

- **控件**：[RuleManager.vue](src/views/RuleManager.vue) 第 655–656 行使用两个 `<input type="time">`，绑定 `executionTimeStart` / `executionTimeEnd`（格式 `"HH:mm"`）。  
- **问题 1**：原生 `type="time"` 在多数浏览器/系统下会显示 12 小时制 + AM/PM，无法通过 HTML 强制 24 小时。  
- **问题 2**：当前「时间段」一行内同时有「时间段」标签、两个时间输入、「—」和整段说明文案，布局拥挤，且原生时间控件样式与项目内 `.select` / `.input-number` 不统一。

数据流保持不变：保存时仍为 `executionTimeWindows: [{ start: "HH:mm:00", end: "HH:mm:00" }] `或 `null`（全天）；回显仍从 `win[0].start/end `取前 5 位得到 `"HH:mm"`。

---

## 二、目标

1. **取消 AM/PM**：时间选择仅展示 0–23 时、0–59 分，无 12 小时制与 AM/PM。  
2. **统一 24 小时制**：交互与展示均为 24 小时，与下方说明文案一致。  
3. **优化 UI**：布局清晰、与现有表单风格一致（`.section-block`、`.form-row`、`.label`、`.select`），说明文案单独成行不挤在同一行。

---

## 三、实现思路

### 3.1 控件形态（二选一，推荐 A）

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **A. 时/分下拉** | 每个时间点用两个 `<select>`：时 00–23，分 00–59（或 00/15/30/45） | 无 AM/PM、与现有 `.select` 一致、易做无障碍 | 点击两次选择 |
| **B. 时/分数字框** | 每个时间点用两个 `<input type="number">`，min/max 限制 | 与 `.input-number` 一致、可键盘输入 | 需校验范围与补零 |

**推荐 A**：下拉明确是 24 小时，且项目内已有大量 `.select`，风格统一；分钟可用 00–59 或步长 5/15，按产品偏好定。

### 3.2 布局与结构

- **「允许执行时间」**：保持现有「全天 / 指定时间段」单选不变。  
- **指定时间段时**：  
- 不再使用单行「时间段」+ 两个 time 输入 + 长文案。  
- 改为独立小区块（例如类名 `.execution-time-range`）：  
- 第一行：**开始时间** [时] [分]（下拉或数字框，24 小时）  
- 第二行：**结束时间** [时] [分]  
- 第三行：说明文案（24 小时制、跨日语义）使用 `.muted`，单独一行或放在区块底部。  
- 视觉上可与「执行频率」那一段的 `.form-row` 对齐，必要时给「开始/结束」加 `.label`，与现有 `.form-row .label` 一致。

### 3.3 数据与绑定

- 继续使用 `executionTimeStart`、`executionTimeEnd` 两个 ref，值为 `"HH:mm"` 字符串。  
- **若用下拉**：  
- 用计算属性或方法从 `"HH:mm"` 拆出 `startHour`/`startMin`/`endHour`/`endMin`（数字）。  
- 用两个 `<select>` 绑定小时/分钟，`@change` 时拼回 `"HH:mm"` 写回 `executionTimeStart`/`executionTimeEnd`（注意补零，如 `9` → `"09"`）。  
- 保存逻辑不变：`(executionTimeStart.value || '09:00').slice(0, 5) + ':00'` 与 end 同理；回显逻辑也不变（`win[0].start/end` slice 0,5）。

### 3.4 是否抽组件

- **方案 1（推荐）**：在 [src/components](src) 下新增 `TimePicker24.vue`（若尚无 `components` 目录则新建）。  
- Props：`modelValue: String`（`"HH:mm"`）。  
- Emit：`update:modelValue`。  
- 内部为两个 select（时/分），24 小时制；样式使用与 RuleManager 一致的 class（或 scoped + 同一套变量）。  
- RuleManager 中「开始时间」「结束时间」各用一个 `<TimePicker24 v-model="executionTimeStart" />`、`v-model="executionTimeEnd" />`，模板更简洁且可复用。  
- **方案 2**：不抽组件，在 RuleManager 内直接写两对「时+分」下拉及拼/拆逻辑，少一层抽象。

## 四、实施步骤（建议顺序）

1. **数据结构与兼容**  

- 确认并保持：`executionTimeStart`/`executionTimeEnd` 为 `"HH:mm"`；保存/回显继续用现有 slice 与 `executionTimeWindows` 结构。无需改后端或 API。

2. **24 小时选择控件**  

- 若选方案 A（推荐）：  
- 新增 `src/components/TimePicker24.vue`，内部为「时」select（0–23）、「分」select（0–59，或 0/15/30/45）。  
- 从 `modelValue` 解析出 hour/min，选择变化时拼成 `"HH:mm"` 并 `emit('update:modelValue', value)`。  
- 若选方案 B：在 RuleManager 内实现两对 number 输入，带 min/max 与补零后拼成 `"HH:mm"`。

3. **RuleManager 模板与布局**  

- 在「5. 执行频率与执行时间」的 section-block 内，当 `!executionTimeAllDay` 时：  
- 移除原来的单行「时间段」+ 两个 `<input type="time">` + 同一行说明文案。  
- 改为 `.execution-time-range` 区块：  
- 第一行：标签「开始时间」+ 开始时间的 24 小时选择器（组件或两列下拉）。  
- 第二行：标签「结束时间」+ 结束时间的 24 小时选择器。  
- 第三行：当前「可只填一个时段；24 小时制。结束时间早于…」整段移至此处，单独一行，class 可用 `muted` 或现有说明样式。

4. **样式**  

- 为 `.execution-time-range` 增加简单布局（如每行 `display: flex; align-items: center; gap: 8px;`，标签固定宽度或与其它 `.form-row .label` 一致）。  
- 时/分下拉使用现有 `.select`（或与 `.input-number` 同宽度的 class），保证与「执行频率」等表单项对齐；不引入新 UI 库。

5. **回显与边界**  

- 打开编辑时，仍用 `win[0].start/end `的 `slice(0,5)` 赋给 `executionTimeStart`/`executionTimeEnd`；若后端给的是 `"HH:mm:00"`，解析时只取前 5 位即可。  
- 新建规则默认值保持 `'09:00'` / `'18:00'`；重置表单时同样保持。

6. **验收**  

- 选择「指定时间段」后，界面仅出现 0–23 时、0–59 分（或设定的分钟步长），无 AM/PM。  
- 选择 20:00–05:00 保存后，列表与编辑回显正确，后端收到的仍为 `start/end` 带 `:00` 的格式。  
- 说明文案单独成行，整体与「执行频率」区块风格一致。

## 六、风险与注意点

- **不引入新依赖**：使用原生 `<select>` 或 `<input type="number">`，不新增 UI 库。  
- **格式**：拼 `"HH:mm"` 时注意个位数补零（如 `9` → `"09"`），避免保存成 `"9:0"` 导致后端或展示异常。  
- **无障碍**：若用 select，为「开始时间」「结束时间」及时/分加合适 `label` 或 `aria-label`，便于读屏与键盘操作。

按上述步骤实现后，即可取消 AM/PM、统一 24 小时制，并完成执行时间段选择的前端 UI 优化。
