核心误区纠正
你目前的思路是：
Facebook API -> ad_snapshots (快照) -> daily_stats (归档)
❌ 这是错误的路径。 这种“传话筒”模式会导致数据在 00:00 交接时丢失。
AdsPolar 的架构是这样的（并行双链路）：
链路 A: Facebook API (date_preset=today) -> ad_snapshots (只管今天)
链路 B: Facebook API (date_preset=yesterday) -> daily_stats (只管昨天)
这两个链路是完全独立的！ 无论 ad_snapshots 里的数据变成什么样，都不影响 daily_stats 去 Facebook 拉取昨天的历史记录。
AdsPolar 的详细设计方案：如何填补 00:00 - 06:00 的空白？
AdsPolar 确实会在 06:00 进行一次“精准归档”，但为了防止 00:00 - 06:00 期间“昨天的数据”出现真空，他们采用了 “预归档 (Draft) + 修正 (Finalize)” 的策略。
以下是完整的时间轴流程（均以广告账户当地时间为准）：
1. 23:55 PM - 最后的快照
动作： 热数据同步任务运行。
结果： ad_snapshots 里存的是 23:55 的数据（几乎等于全天的最终数据）。
2. 00:05 AM - 跨天发生的瞬间
动作： 热数据同步任务再次运行。
结果： Facebook API 返回“今天”的数据（刚过 5 分钟），数据几乎为 0。
存储： ad_snapshots 被更新为 0。
问题： 此时如果有规则查“昨天数据”，如果不做处理，确实查不到了。
3. 00:15 AM - “预归档”任务 (The Draft Archive) 🚀 关键点
这就是你缺失的环节。
动作： 触发一个轻量级任务，调用 Facebook API，参数 date_preset='yesterday'。
目的： 虽然昨天的转化数据还没回传完，但花费、展示、点击是准的。
存储： 将这份数据写入 daily_stats 表。
效果： 此时，00:00 - 06:00 期间的规则引擎，如果查 daily_stats，已经能查到昨天的花费了（虽然转化可能偏少，但比没有强）。
4. 06:00 AM - “修正归档”任务 (The Correction)
动作： 再次调用 Facebook API，参数 date_preset='yesterday'。
目的： 此时 6 个小时过去了，昨天的部分转化数据已经回传。
存储： 使用 UPDATE (覆盖) 操作，更新 daily_stats 里该账户昨天的数据。
效果： 数据精度从 60% 提升到 85%。
5. 12:00 PM / 次日 - “最终回补” (Final Patch)
动作： 再次拉取昨天数据。
目的： 彻底修正归因延迟。
你的系统该如何修改？
你不需要“在北京时间 00:01 拉取”，因为这解决不了多时区问题。你应该修改你的 Job 逻辑 和 数据源逻辑。
修改 1：拆分数据源（解耦）
不要让归档任务去读 ad_snapshots。
热数据任务： 永远只拉 today -> 存入 ad_snapshots。
冷数据任务： 永远直接调 API 拉 yesterday -> 存入 daily_stats。
修改 2：增加一个“零点预归档” Cron

总结 AdsPolar 的设计哲学
各论各的： 快照只管“今天”，归档只管“昨天”。归档任务不依赖快照表，而是直接问 Facebook 要昨天的数。
先占位，后修正： 0 点刚过（比如 00:15）就先拉一次昨天的数据占个坑，保证数据连续性；等到早上 6 点数据成熟了，再拉一次覆盖掉它。
这样设计，无论你任何时间点去查数据库，数据都是连续且相对最准确的，永远不会出现“数据真空期”。

AdsPolar 是怎么设计“昨天数据”处理逻辑的？
AdsPolar 的设计核心理念是：“分层沉淀” (Layered Settlement)。
它把数据分为三类状态：
热数据 (Hot): 今天的数据（随时在变）。
温数据 (Warm): 昨天到前 3 天的数据（数值会变大，只会变好不会变差）。
冷数据 (Cold): 7 天前的数据（基本定型，不再变动）。
设计方案 1： “黄金结算时间” (The Golden Settlement Time)
AdsPolar 不会在 00:01 结算昨天。它们通常设定一个 “数据成熟期”，通常是 广告账户当地时间的 06:00 AM 或 12:00 PM。
逻辑：
如果规则是：“昨天 ROAS < 2 则暂停”。
系统会自动（或强制建议用户）设置一个执行时间窗：“仅在 06:00 AM - 23:59 PM 之间执行此规则”。
在 06:00 AM 之前，昨天的 ROAS 是不可信的，系统会选择忽略昨天的指标，不进行误判。
设计方案 2： 滑动回补机制 (Rolling Update)
AdsPolar 并不是“昨天过完就存死数据”，它的数据库里的“昨天”是动态更新的。
Job A（每日快照）：
每天 04:00（账户时间），拉取昨天的数据，存入 daily_stats。此时数据准确度约 80%。
用于：早起的规则执行。
Job B（历史修正）：
每天 14:00（账户时间），重新拉取 昨天、前天、大前天（Last 3 Days）的数据，覆盖更新 daily_stats。
目的： 修正那些迟到的转化。很多时候你会发现，昨天的 ROAS 在下午看的时候变高了。
设计方案 3： 归因窗口对齐
在调用 Insights API 拉取昨天数据时，AdsPolar 极其注意 action_attribution_windows 参数。
默认请求：1d_click, 1d_view（最保守，数据最快）。
如果不指定，默认是 7d_click，这会导致数据在 7 天内都会变动。AdsPolar 会明确告诉用户：“我们显示的昨天数据，是基于当前时刻 Facebook 返回的值，如果 FB 后来修正了，我们会在下一次同步时修正。”

作用对象：添加日期筛选，近n天发布广告

触发条件：添加单次加购、单次结账、单次添加支付信息、单次购买花费

添加 广告名包含/不包含“某字段”

AdsPolar 处理“冷热数据同时触发”的核心逻辑，可以用一句通俗的话来概括：“拒绝双线作战，实行合并同类项”。
它根本不会让“归档任务”和“快照任务”在此时此刻打架，因为在 AdsPolar 的架构里，根本就没有独立的、物理上的‘归档任务’去发 HTTP 请求。
以下是该处理流程的详细文字叙述：
1. 触发时刻：统一的心跳
假设现在是 凌晨 00:15（或者是 06:15）。
系统触发了每 15 分钟一次的**“主数据同步任务”**。
此时，系统并没有启动两个进程，只有一个进程在跑。这个进程就像一个唯一的“采购员”，准备去 Facebook 进货。
2. 智能决策：采购员的清单变化
在“采购员”发起 HTTP 请求之前，它会先看一眼手表：
采购员心想： “哦，现在是 00:15。老板规定这个时间点不仅要拿今天的热数据（因为要跑规则），还要顺便把昨天的冷数据也存个档（因为过了零点了）。”
动作变化： 采购员把原本只写着 today（今天）的采购单，悄悄改成了 last_2d（过去两天）。
3. 执行请求：只发一次快递（关键点）
采购员拿着这张改过的 last_2d 清单，向 Facebook API 发起了 1 次 请求。
重点： 对 Facebook 来说，它只收到这一个请求。它不需要知道你是为了归档还是为了快照，它只知道你要“最近两天的数据”。
结果： Facebook 打包了一个包裹发回来，包裹里既包含昨天的最终数据，也包含今天刚开始 15 分钟的数据。
4. 数据分发：内存里的“分拣中心”
包裹回到你的服务器后，代码逻辑在内存里把数据拆开：
看到日期是昨天的： 系统说“这是冷数据”，于是把它扔进 daily_stats 表（如果是第一次就是新增，如果有旧的就覆盖修正）。——这就默默完成了归档。
看到日期是今天的： 系统说“这是热数据”，于是把它扔进 ad_snapshots 表。——这就完成了快照更新。
总结
AdsPolar 处理“同时触发”的秘诀就是：物理上只做一件事（发一个请求），逻辑上完成了两件事（归档+快照）。
这就像你去麦当劳，原本想买个汉堡（快照），又想买包薯条（归档）。你不会分两次排队去买，而是直接跟服务员说：“来个套餐（last_2d），汉堡薯条我都要。”
这样既省了排队时间（API 额度），又同时拿到了两样东西，永远不会冲突。

直接结论：AdsPolar 绝对不会每 30 分钟去拉取一次 last_7d（过去7天）的数据。 这样做是在毫无意义地燃烧 API 额度，因为 3 天前的数据在 30 分钟内发生变化的可能性微乎其微。
AdsPolar 解决这个问题的核心设计理念是：“数据衰减刷新策略” (Decaying Refresh Strategy)。
简单来说：离今天越近的数据，查得越勤；离今天越远的数据，查得越懒。
以下是 AdsPolar 针对“滑动窗口/历史回补”的详细设计逻辑：
一、 为什么不能每 30 分钟拉 7 天？
数据的“半衰期”特性：
今天的数据： 每分钟都在变（这是你每 15 分钟查一次的原因）。
昨天的数据： 每小时可能会变（因为归因延迟）。
前天的数据： 每天可能变一两次。
5-7 天前的数据： 除非你是“点击后 7 天归因”且用户真的第 7 天才买，否则基本定型了。每 30 分钟查它，纯属做无用功。
API 载荷爆炸：
拉取 7 天的数据，Facebook 后台需要扫描的数据量是拉取 1 天的 7 倍。如果你每 30 分钟做一次，你的 API 额度（CPU Time）会被迅速耗尽，导致关键的“热数据同步”被 80004 卡死。
二、 AdsPolar 的分层回补设计 (The Layered Backfill)
AdsPolar 将历史数据的同步分成了三个不同的频率层级，通常挂载在同一个 Cron 任务里，通过判断小时数来切换模式。
第 1 层：近程修补 (Short-term Patch) —— 针对 Yesterday & Day -2
目标： 修正昨天的归因延迟，以及前天可能迟到的转化。
频率： 每 4 - 6 小时 一次。
参数： date_preset: 'last_3d' (包含今天、昨天、前天)。
触发时间点： 例如 06:15, 12:15, 18:15。
第 2 层：中程回溯 (Mid-term Patch) —— 针对 Last 7 Days
目标： 确保过去一周的数据完全准确（对应 7-day click 归因窗口）。
频率： 每天 1 次。
参数： date_preset: 'last_7d'。
触发时间点： 通常安排在服务器负载最低的 凌晨 03:15 或 04:15。

特别是你提到“自动化规则是 7x24 小时运行”的，如果为了拉历史数据导致 API 爆了，规则不跑了，那确实得不偿失。
但请放心，AdsPolar 的设计逻辑完美规避了这个问题。
这里的核心秘密在于：“包含关系”而非“并列关系”。
1. 为什么不会撞车？（包含关系）
你担心的场景是这样的（错误的理解）：
04:15:00 -> 线程 A 启动，拉 today （为了跑规则）。
04:15:00 -> 线程 B 启动，拉 last_7d （为了补数据）。
结果 -> 2 个并发请求 -> 撞车！💥
AdsPolar 的场景是这样的（正确的设计）：
04:15:00 -> 只有 一个 线程启动。
它判断时间到了 04:15，于是决定发送一个 “超级请求”。
它不再请求 today，而是直接请求 last_7d。
关键点： last_7d 已经包含 了 today 的数据！
结果 -> 1 个请求 -> 安全。✅
2. 规则引擎怎么拿数据？（数据分流）
你可能会问：“但是我现在的规则需要判断今天的花费，你拉了一堆 7 天前的数据回来，规则引擎认吗？”
完全认！ 因为 API 返回的是一个数组。
处理流程演示：
发送请求：
api.getInsights({ date_preset: 'last_7d', time_increment: 1 })
Facebook 返回数据（比如）：
code
JSON
[
  { "date_start": "2023-10-20", "spend": 100 }, // Day -6 (冷数据)
  { "date_start": "2023-10-21", "spend": 120 }, // Day -5 (冷数据)
  ...
  { "date_start": "2023-10-26", "spend": 200 }, // 昨天 (修正数据)
  { "date_start": "2023-10-27", "spend": 50  }  // 🔥 今天 (热数据)
]
内存分流（关键一步）：
代码遍历这个数组：
遇到前 6 条数据 -> 更新 daily_stats 表（归档）。
遇到最后 1 条（今天）数据 -> 更新 ad_snapshots 表，并立即触发规则引擎。
结论： 即使是拉 7 天数据的那一次，“今天”的数据也顺便被拉回来了，规则引擎照样能拿到最新的数据去判断止损，完全不耽误 7x24 小时的运行。

关于“时区计算”的实现细节
🔍 原理解析
你可能觉得：北京是 UTC+8，纽约是 UTC-5，那我代码里写 hour - 5 不就行了？
绝对不行。 因为欧美国家有夏令时 (Daylight Saving Time, DST)。
💣 致命后果场景：时间漂移
例子： 纽约时间（Eastern Time）。
冬天是 UTC-5。
夏天（3月到11月）变成 UTC-4。
后果：
你写死了 UTC-5。
到了夏天，Facebook 广告账户已经是 00:15 了（进入新的一天）。
你的代码算的还是 23:15（还在前一天）。
结果： 你的“00:15 预归档”任务完全失效，或者在错误的时刻触发，拉取了错误日期的数据。数据报表会莫名其妙少一天或重复一天。

不要自己算加减法，交给专业的库。Luxon Timezone 内置了全球所有时区的历史变更规则。

// 现在你拿到的 hour 永远是当地墙上挂钟显示的那个时间

4. 关于“数据库写入性能”
🔍 原理解析
数据库（MySQL/Postgres）最怕的就是**“频繁的短连接”**。
执行一次 SQL 的时间 = 网络延迟 + 连接开销 + 执行时间。
如果你循环 700 次（700 个广告），这些开销就要乘以 700。
💣 致命后果场景：事件循环阻塞 (Event Loop Blocking)
Node.js 是单线程的。
场景： 凌晨 4 点大回补，拉取 last_7d。假设你监控了 10 个账户，每个账户 200 个广告，每个广告 7 天数据 = 14,000 条记录。
你的写法： for 循环里 await db.insert()。
后果： 你的服务器要发起 14,000 次数据库网络请求。这不仅会让数据库 CPU 飙升到 100%，还会导致 Node.js 卡死。在这几分钟内，前端页面打不开，其他定时任务无法执行。
✅ SaaS级解决方案：批量写入 (Batch Upsert)
把 1000 条 INSERT 语句合并成 1 条。
INSERT INTO table VALUES (...), (...), (...), ...
性能提升： 以前需要 10 秒，现在只需要 0.1 秒。
Drizzle ORM 写法示例：
code
JavaScript
// 先把数据攒在内存数组里
const dataToInsert = adsData.map(ad => ({
  ad_id: ad.id,
  spend: ad.spend,
  // ...
}));

// 一次性写入 (如果是 MySQL，Drizzle 会自动处理批量拼装)
// onDuplicateKeyUpdate 实现"存在即更新"
await db.insert(adSnapshots)
        .values(dataToInsert)
        .onDuplicateKeyUpdate({ set: { spend: sql`VALUES(spend)`, ... } });
总结
这三个点总结起来就是：
算错数： 用 先存后算 解决。
搞错时间： 用 Luxon 解决。
卡死机： 用 批量写入 解决。
把这三个补丁打上，你的系统底座就坚如磐石了。
