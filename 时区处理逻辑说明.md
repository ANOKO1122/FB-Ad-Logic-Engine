# 时区处理逻辑说明

> 目的：解释服务器时间、账户时区、前端显示的时区处理逻辑

---

## 📅 服务器时间说明

### 服务器时间的来源

**服务器时间**是从日志中看到的：
```
⏰ 当前服务器时间: 2026-01-21T06:24:49.091Z
```

这个时间是通过 `DateTime.now().toUTC().toISO()` 获取的：
- `DateTime.now()`：获取当前系统时间
- `.toUTC()`：转换为 UTC 时区
- `.toISO()`：转换为 ISO 8601 格式（如 `2026-01-21T06:24:49.091Z`）

**注意**：
- `Z` 表示 UTC 时区（零时区）
- 你的服务器本地时间可能是 `UTC+8`（北京时间），但转换为 UTC 后就是 `06:24:49`

## 🌍 账户时区说明

### 账户时区应该从哪里获取？

**正确流程**：
1. **数据同步时**：从 Facebook API 获取账户时区
   - API 调用：`/{accountId}?fields=timezone_name`
   - 返回时区：如 `Asia/Shanghai`、`America/Los_Angeles`、`America/New_York` 等
2. **写入数据库**：应该更新 `account_mappings.timezone_name` 字段
3. **后续使用**：从 `account_mappings.timezone_name` 读取

### 为什么所有账户时区都是 UTC？

**可能的原因**：
1. **`account_mappings.timezone_name` 字段没有更新**
   - 数据同步时虽然从 Facebook API 获取了时区
   - 但可能没有更新到 `account_mappings` 表
   - 所以使用了默认值 `UTC`

2. **数据同步时只写入了 `ad_snapshots.timezone_name`**
   - 时区写入了 `ad_snapshots` 表（用于数据查询）
   - 但没有更新 `account_mappings` 表（用于归档检查）

### 如何修复？

**方案 1：在数据同步时更新 `account_mappings.timezone_name`**

在 `syncAccountTodayStats` 函数中，获取时区后更新 `account_mappings` 表：

```javascript
// 获取时区后，更新 account_mappings 表
if (timezoneName) {
  await pool.execute(
    `UPDATE account_mappings 
     SET timezone_name = ? 
     WHERE fb_account_id = ?`,
    [timezoneName, accountId]
  )
}
```

**方案 2：在归档检查时从 Facebook API 获取时区**

如果 `account_mappings.timezone_name` 为 NULL，从 Facebook API 获取并更新。

## 🖥️ 前端时区显示说明

### 前端应该如何处理时区？

**后端返回的数据**：
- 数据库会话时区：UTC
- 时间字段：UTC 格式（如 `2026-01-21T06:24:49.091Z`）

**前端显示时**：
- 应该使用 Luxon 将时间转换为用户本地时区（如 `Asia/Shanghai`，即 `UTC+8`）
- 转换方式：`DateTime.fromISO(utcTime).setZone('Asia/Shanghai')`

**示例代码**：
```javascript
import { DateTime } from 'luxon'

// 后端返回的 UTC 时间
const utcTime = '2026-01-21T06:24:49.091Z'

// 转换为北京时间（UTC+8）
const beijingTime = DateTime.fromISO(utcTime).setZone('Asia/Shanghai')
console.log(beijingTime.toFormat('yyyy-MM-dd HH:mm:ss'))  // 2026-01-21 14:24:49
```

**时区来源**：
- 可以从账户配置获取（`account_mappings.timezone_name`）
- 或者从用户设置获取（用户偏好时区）
- 或者使用浏览器时区（`DateTime.now().zoneName`）

### 服务器时间
- 来源：`DateTime.now().toUTC().toISO()`

- 你的服务器本地时间可能是 `UTC+8`，但转换为 UTC 后就是 `06:24:49`

- **修复**：在数据同步时更新 `account_mappings.timezone_name`

### 前端显示
- **后端返回**：UTC 时区的时间
- **前端转换**：使用 Luxon 转换为用户本地时区（如 `Asia/Shanghai`，即 `UTC+8`）
