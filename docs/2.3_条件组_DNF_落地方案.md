# 2.3 条件组方案（DNF：OR of AND Groups）落地方案

> 更新：2026-02-06 - 整合规划修正与拍板决策

## 一、方案确认

**方向**：用「条件组（OR of AND groups）」表达括号，运营不需理解优先级。

- 规则由多个「条件组」组成
- 组内固定 AND（都满足）
- 组与组之间固定 OR（满足任意一组）
- UI 用「组」表达括号，天然无歧义

**示例**：`(spend > 0.8 AND link_clicks = 0) OR (cpc > 0.8)` → 组1 AND 组2 OR

---

## 二、拍板决策

| 决策 | 结论 | 理由 |
|------|------|------|
| 1. 新建规则一律 v2，旧规则兼容 | **同意** | 新建走 v2，旧规则评估时 v1→v2 归一化；无需全量迁移，历史规则照跑 |
| 2. 阶段 1 后先补单测再动前端 | **同意** | 先锁定 (A AND B) OR C 语义，避免后续 UI 改动破坏后端逻辑 |
| 3. 模板策略 | **先单组 P0，组合模板 P1** | 单组：空耗/贵点击/空转化；多场景组合止损（3 个 OR 组）放 P1 |

## 三、规划修正（落地必守）

### 1. rules API：conditions 校验

**现状**：`POST /api/rules`、`PUT /api/rules/:id` 强校验 `Array.isArray(conditions)`，v2 对象会被 400。

**修正**：允许 v1 array 或 v2 object，并做最小结构校验。

```javascript
// 校验逻辑（示意）
const isV1 = Array.isArray(conditions) && conditions.length > 0
const isV2 = conditions?.version === 2 && Array.isArray(conditions?.groups) && conditions.groups.length > 0
if (!isV1 && !isV2) return res.status(400).json({ error: 'conditions 格式无效' })
```

- v1：`conditions` 为数组，结构沿用现有校验
- v2：`conditions` 为对象，`version === 2` 且 `groups` 为非空数组
- 不符合任一格式 → 400

### 2. v1→v2 归一化（语义正确）

**错误做法**：把 v1 的 `logicOperator` 塞进单组的 `operator`，会导致 v1 OR 被错误当成 AND。

**正确做法**：

| v1 结构 | 转成 v2 |
|---------|---------|
| logicOperator = AND | 1 个组，组内多个条件 AND |
| logicOperator = OR | 多个组，每个组 1 条条件（等价 OR） |

```javascript
// 示意
if (v1) {
  if (logicOperator === 'AND') {
    groups = [{ operator: 'AND', conditions: [...conditions] }]
  } else {
    groups = conditions.map(c => ({ operator: 'AND', conditions: [c] }))
  }
}
```

### 3. time_window 约束（阶段 1）

**现状**：RuleEngine 只取「第一个条件的 time_window」查库（`getTimeWindowFromConditions`）。

**风险**：v2 各组条件若自由选 time_window，实际只生效第一条，易误配。

**阶段 1 约束**：同一条规则（跨所有组）必须使用同一 time_window / custom_range。

- 前端：保存前校验，混用则拒绝并提示
- 后端：可选做二次校验（或仅前端约束）

### 4. 前端 normalize/load/edit 兼容 v2

**现状**：`normalizeJsonArray(conditions)` 只处理数组，v2 对象会被当作非数组处理，可能变成 `[]`，UI 显示为空。

**修正**：load/edit 时区分 v1 / v2：

```javascript
// 示意
function normalizeConditions(raw) {
  if (!raw) return { version: 1, groups: [{ operator: 'AND', conditions: [] }] }
  if (raw.version === 2 && Array.isArray(raw.groups)) return raw
  if (Array.isArray(raw)) return v1ToV2(raw, logicOperator)  // 转成 v2 再渲染
  return { version: 1, groups: [{ operator: 'AND', conditions: [] }] }
}
```

- 展示：统一用 v2 结构渲染（v1 先转 v2）
- 保存：新规则一律写 v2；旧规则编辑后也写 v2（可选）

## 四、数据结构约定

### v1（旧）

```json
{
  "conditions": [
    { "metric": "spend", "operator": "gt", "value": 20, "time_window": "today" },
    { "metric": "link_clicks", "operator": "eq", "value": 0, "time_window": "today" }
  ],
  "logicOperator": "AND"
}
```

### v2（新）

```json
{
  "version": 2,
  "groups": [
    {
      "operator": "AND",
      "conditions": [
        { "metric": "spend", "operator": "gt", "value": 0.8, "time_window": "today" },
        { "metric": "link_clicks", "operator": "eq", "value": 0, "time_window": "today" }
      ]
    },
    {
      "operator": "AND",
      "conditions": [
        { "metric": "cpc", "operator": "gt", "value": 0.8, "time_window": "today" }
      ]
    }
  ]
}
```

## 五、实施顺序（含修正点）

| 阶段 | 内容 | 注意 |
|------|------|------|
| 1 | 后端 v2 评估逻辑 + v1→v2 归一化（v1 OR → 多单条件组） | 归一化语义必须正确 |
| 2 | 后端 `getTimeWindowFromConditions` / `getCustomRangeFromConditions` 支持 v2 | 从所有组收集 time_window |
| 3 | rules API 校验：允许 v1 array / v2 object，最小结构校验 | 否则 v2 会被 400 |
| 4 | **单测**：`(spend>0.8 AND link_clicks=0) OR (cpc>0.8)` 语义锁定 | 在动前端前完成 |
| 5 | 前端 `normalizeConditions` 兼容 v2，load/edit 不丢条件 | 否则列表/编辑显示为空 |
| 6 | 前端条件组 UI（组卡片、组间 OR、组内 AND） | 含 time_window 同规则校验 |
| 7 | 模板改为填充 v2（空耗/贵点击/空转化，单组 P0） | 组合模板 P1 |
| 8 | 验收 | 覆盖 v1 旧规则 + v2 新规则 |

---

## 六、验收要点

- v1 旧规则：评估结果与改前一致
- v2 新规则：`(A AND B) OR C` 按 DNF 语义正确命中
- 单测：锁定核心表达式语义
- 前端：v1/v2 规则都能正确展示和编辑，无丢条件
- time_window：同规则内一致，混用则保存前拦截
