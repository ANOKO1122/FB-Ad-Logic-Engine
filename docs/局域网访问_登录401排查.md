# 局域网访问登录 401 排查指南

## 现象

- 「别人」通过局域网 IP（如 `http://192.168.0.67:3000`）能打开页面
- 登录后出现 `GET /api/me 401 (Unauthorized)`
- 本机 `localhost` 访问时登录正常

## 原因分析

### 1. Cookie 与代理

- 前端：Vite 在 `http://192.168.0.67:3000`（或 `localhost:3000`）
- 后端：Express 在 `localhost:3001`
- `/api` 由 Vite 代理到后端

通过代理时，可能出现：

- `Set-Cookie` 的 `domain` 与浏览器访问的地址不一致
- 后端通过代理收到 `Host: localhost:3001`，Cookie 可能被当成 `localhost` 域名
- 在别的机器上访问 `192.168.0.67` 时，不会带 `localhost` 的 Cookie，导致 401

### 2. 访问方式不一致

- 你：`http://localhost:3000` → Cookie 域是 localhost → 正常
- 别人：`http://192.168.0.67:3000` → 需要 Cookie 域为 `192.168.0.67` → 可能不一致

### 3. 其他可能

- 登录接口本身返回非 2xx（如密码错误），未设置 Cookie
- `secure: true` 时，HTTP 下 Cookie 不会发送
- 浏览器隐私/无痕模式、扩展拦截 Cookie

---

## 解决方案

### 方案 A：统一用同一地址访问（快速验证）

所有人都用同一地址访问，避免 localhost 和 IP 混用：

- 统一用 `http://192.168.0.67:3000`，或
- 统一用 `http://localhost:3000`

**验证**：别人用 `http://192.168.0.67:3000` 登录，看是否仍 401。

### 方案 B：后端按请求 Host 设置 Cookie（推荐）

让后端根据实际请求的 Host 来设置 Cookie，使其匹配访问地址。

在 `server/routes/auth.js` 的登录成功分支中修改 `res.cookie`，例如：

```javascript
const cookieOpts = {
  httpOnly: true,
  secure: process.env.NODE_ENV === 'production',
  sameSite: 'lax',
  maxAge: 7 * 24 * 60 * 60 * 1000
}
// 代理场景下使用原始 Host，确保局域网 IP 访问时 Cookie 域正确
const host = req.get('x-forwarded-host') || req.get('host') || 'localhost:3001'
if (host.includes('192.168.') || host.includes('localhost')) {
  cookieOpts.domain = host.split(':')[0]  // 仅用主机名，不用端口
}
res.cookie('token', token, cookieOpts)
```

注意：`domain` 的用法要符合 RFC，并兼顾 localhost / 192.168.x.x。

### 方案 C：Vite 代理重写 Set-Cookie

在 `vite.config.js` 中通过 `proxyRes` 重写 `Set-Cookie`，使 Cookie 域与访问地址一致：

```javascript
proxy: {
  '/api': {
    target: 'http://localhost:3001',
    changeOrigin: true,
    configure: (proxy) => {
      proxy.on('proxyRes', (proxyRes, req) => {
        const cookies = proxyRes.headers['set-cookie']
        if (cookies) {
          // 移除 domain，让浏览器使用请求的 host（如 192.168.0.67）
          proxyRes.headers['set-cookie'] = cookies.map(c =>
            c.replace(/;\s*domain=[^;]+/gi, '')
          )
        }
      })
    }
  }
}
```

## 排查步骤

### 1. 确认登录是否成功

在浏览器开发者工具：

1. 打开 Network
2. 输入用户名密码，点击登录
3. 找到 `login` 请求，查看：
   - **Status**：应为 200，若是 401 表示账号/密码错误
   - **Response Headers**：是否有 `Set-Cookie: token=...`

若没有 `Set-Cookie`，说明登录失败或后端未返回 Cookie。

### 2. 确认 Cookie 是否被保存

1. Application → Cookies → 选中 `http://192.168.0.67:3000`（或你访问的地址）
2. 查看是否存在 `token` Cookie

若无，说明 Cookie 未被浏览器接受（域/路径/secure 等不匹配）。

### 3. 确认 /api/me 是否带上 Cookie

1. 在 Network 中找到 `me` 请求
2. 查看 Request Headers 中是否有 `Cookie: token=...`

若无 Cookie，说明请求未附带凭证。

## 快速验证命令

在**别人**的电脑上，用 curl 模拟登录并获取 /api/me：

```powershell
# 1. 登录，保存 Cookie 到文件
curl -X POST "http://192.168.0.67:3000/api/auth/login" ^
  -H "Content-Type: application/json" ^
  -d "{\"username\":\"你的用户名\",\"password\":\"你的密码\"}" ^
  -c cookies.txt -v

# 2. 使用 Cookie 访问 /api/me
curl "http://192.168.0.67:3000/api/me" -b cookies.txt -v
```

若第 1 步返回 200 且响应中有 `Set-Cookie`，第 2 步应返回用户信息。若第 2 步仍 401，多半是 Cookie 未被正确携带或校验失败。

---

## 建议

1. 先按 **方案 A** 统一访问地址，确认是否与 localhost/IP 混用有关。
2. 若仍 401，按上述排查步骤检查：登录是否成功、Cookie 是否写入、/api/me 是否带 Cookie。
3. 若确认为 Cookie 域问题，再采用 **方案 B** 或 **方案 C** 修改配置。
