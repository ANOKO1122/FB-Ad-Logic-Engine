# 顺序2：结构表 structure_ads 验证与测试清单

伪增量 + Piggyback + 每小时全量轮转 + 手动同步 — 可执行验证清单。  
每条含：**目的、操作步骤、预期日志/字段、需查 SQL、通过/失败判定、失败时 3 个可能原因与定位**。

> **手把手教学**：若使用 **Thunder Client** 按步骤跑完 8 项验证，请直接看 **[顺序2_结构同步_ThunderClient验证教学.md](./顺序2_结构同步_ThunderClient验证教学.md)**，内含每步的请求配置、Headers/Body、预期响应、日志关键字与 SQL。
>
> **说明**：当前代码库仅有单账户接口 `POST /api/sync/structure`，无 `batch`。若后续增加多账户/全选接口，可复用「单账户」的预期与 SQL，按账户逐个校验。

## 一、最小可执行验证命令（Windows）

### 环境假设
- 服务地址：`http://localhost:3000`（按实际改）
- 已登录拿到 JWT：`$token` 或直接替换为你的 Bearer token
- 单账户 ID：`act_xxx`（替换为真实账户）

### PowerShell 示例

```powershell
# 1) 手动同步结构（单账户）
$token = "你的JWT"
$accountId = "act_xxx"
Invoke-RestMethod -Uri "http://localhost:3000/api/sync/structure" -Method POST `
  -Headers @{ "Authorization" = "Bearer $token"; "Content-Type" = "application/json" } `
  -Body (@{ account_id = $accountId } | ConvertTo-Json)

# 2) 触发统一心跳（跑伪增量 + Piggyback）
node server/scripts/manual-unified-heartbeat.js
```

### curl（Windows CMD / Git Bash）

```bash
# 手动同步结构（单账户）
curl -X POST "http://localhost:3000/api/sync/structure" ^
  -H "Authorization: Bearer 你的JWT" ^
  -H "Content-Type: application/json" ^
  -d "{\"account_id\":\"act_xxx\"}"

# 统一心跳
node server/scripts/manual-unified-heartbeat.js
```

## 二、各检查点数据库查询（通用）

### structure_ads

```sql
-- 某账户下广告条数、含 NULL 自愈候选
SELECT account_id, COUNT(*) AS cnt,
       SUM(CASE WHEN last_synced_at IS NULL AND effective_status = 'ACTIVE' THEN 1 ELSE 0 END) AS null_active_count
FROM structure_ads
WHERE account_id = 'act_xxx'
GROUP BY account_id;

-- 指定 ad_id 的状态与 last_synced_at（用于状态差异、NULL 自愈）
SELECT ad_id, name, effective_status, status, configured_status, last_synced_at
FROM structure_ads
WHERE account_id = 'act_xxx' AND ad_id IN ('ad_id_1', 'ad_id_2');
```

### structure_sync_status

```sql
-- 某账户全量同步状态（手动同步/轮转后必看）
SELECT account_id, has_full_synced, last_full_count, last_full_success_at, last_success_at
FROM structure_sync_status
WHERE account_id = 'act_xxx';
```

### 1. 手动同步全量

| 项目 | 内容 |
|------|------|
| **目的** | 确认 2.4 仅走全量、只刷新指定账户，且状态表正确更新。 |
| **操作步骤** | 1. 选一个已绑定账户 `act_xxx`。<br>2. 调用 `POST /api/sync/structure`，body `{"account_id":"act_xxx"}`（或 query `account_id=act_xxx`）。<br>3. 看响应与服务器日志；可选：再调一次（2 分钟内）验证冷却。 |
| **预期日志关键字** | `[2.4] 冷却检查` → `✅ [2.4] 结构同步完成 account=act_xxx mode=full synced=N duration_ms=...`<br>**不能出现**：`mode=incremental` 且当轮 synced 接近该账户全表行数（即“incremental 但 touched=全表”）。 |
| **需查 SQL** | `SELECT account_id, has_full_synced, last_full_count, last_full_success_at, last_success_at FROM structure_sync_status WHERE account_id = 'act_xxx';` |
| **通过标准** | ① 仅该账户被同步（日志仅该 account）。② 日志为 `mode=full`。③ 无“incremental 却 touched≈全表”。④ `structure_sync_status` 该行 `has_full_synced=1`，`last_full_count` 为正数，`last_full_success_at`/`last_success_at` 为刚执行时间。 |
| **失败判定** | 出现 mode=incremental 且 synced≈全表；或 status 表未更新/全为 0。 |
| **可能原因 1** | 实验开关打开，走了增量分支 → `structureSyncService.js` 中 `ENABLE_INCREMENTAL_EXPERIMENT` 或 `opts.useIncrementalExperiment` 为 true。 |
| **可能原因 2** | status 写入条件错误 → `structureSyncService.js` 中 `syncAccountStructureAds` 的 `isFullRun`/`fullCount`/`upsertStatusSql` 逻辑。 |
| **可能原因 3** | 路由或 token 导致未真正调用同步 → `server/index.js` 中 `POST /api/sync/structure` 与 `assertAccountAccess`。 |

### 2. 伪增量差异触达

| 项目 | 内容 |
|------|------|
| **目的** | 确认伪增量只对差异集合调 FB，且 touched 规模合理。 |
| **操作步骤** | 1. 执行一次统一心跳：`node server/scripts/manual-unified-heartbeat.js`。<br>2. 在日志中搜索 `[伪增量]`，按账户查看。 |
| **预期日志字段** | 每账户一行：<br>`[伪增量] account=act_xxx recentActiveIds=N diffIds=M touched=T diffHead5=[...]`<br>或 `diffIds=0 touched=0 diffHead5=[]`。 |
| **需查 SQL** | 可选：`SELECT COUNT(*) FROM structure_ads WHERE account_id = 'act_xxx';` 作全表行数对比。 |
| **通过标准** | ① 每个有 `accountId` 的账户都有一条 `[伪增量]` 日志。<br>② 有 diffIds 时：`touched <= diffIds`，且 touched 远小于该账户 `structure_ads` 全表行数。<br>③ `diffIds=0` 时无后续 FB resolve 日志（不因伪增量再打一批 resolve）。 |
| **失败判定** | touched 接近或等于全表；或 diffIds=0 仍出现大量 resolve 调用。 |
| **可能原因 1** | recentActiveIds 取数范围过大或未降频 → `structureSyncService.js` 中 `runPseudoIncrementForAccount` 的 WHERE（24h/1h/NULL 条件）。 |
| **可能原因 2** | activeAdIds 未正确传入，diffIds 被算大 → `ingestorService.js` 中后置阶段 `r.activeAdIds || []` 及 `runPseudoIncrementForAccount` 调用。 |
| **可能原因 3** | 其他逻辑对同一批 id 重复调 resolve → 搜索 `resolveObjectsByIds` 调用点，确认伪增量仅对 diffIds 调一次。 |

### 3. 状态差异检测生效

| 项目 | 内容 |
|------|------|
| **目的** | 在 FB 后台改广告状态后，下一轮伪增量能更新本地 `structure_ads`。 |
| **操作步骤** | 1. 记下某账户下一条 ACTIVE 广告的 `ad_id`。<br>2. 在 FB 广告后台将该广告暂停（或启用）。<br>3. 触发一次统一心跳，等待伪增量跑完。<br>4. 查库核对该行的 `effective_status`（及 status/configured_status）。 |
| **预期日志** | 该账户 `[伪增量]` 中 `diffIds>=1`、`touched>=1`，且 `diffHead5` 包含该 ad_id（若在当轮前 5 个）。 |
| **需查 SQL** | `SELECT ad_id, effective_status, status, configured_status, last_synced_at FROM structure_ads WHERE account_id = 'act_xxx' AND ad_id = '刚操作的那条ad_id';` |
| **通过标准** | 该行 `effective_status`（或 status/configured_status）与 FB 后台一致（如 PAUSED/ACTIVE）。 |
| **失败判定** | 库内状态未变或与 FB 不一致。 |
| **可能原因 1** | 该 ad 不在当轮 recentActiveIds（如被 1h 降频排除）→ 多等一轮或看 WHERE 条件。 |
| **可能原因 2** | FB 返回字段未写入 → `structureSyncService.js` 中 `UPSERT_STRUCTURE_SQL` 与 `runPseudoIncrementForAccount` 内 upsert 字段。 |
| **可能原因 3** | resolve 返回结构与映射错误 → `server/index.js` 中 `resolveObjectsByIds` 返回的 `effective_status`/status 等。 |

### 4. 1h 降频 + 旋转

| 项目 | 内容 |
|------|------|
| **目的** | 同一账户 1 小时内不重复触达刚 touched 的那批；多轮 diffHead5 会轮转。 |
| **操作步骤** | 1. 触发一次统一心跳，记下某账户的 `[伪增量] ... diffHead5=[...]`。<br>2. 几分钟内再触发一次统一心跳，再记该账户的 `diffHead5`。<br>3. 可选：再触发 1～2 次，看多轮 diffHead5 是否变化。 |
| **预期日志** | 第二次：要么 `diffIds=0 diffHead5=[]`，要么 `diffHead5` 与第一次**不同**。多轮之间 `diffHead5` 有变化。 |
| **需查 SQL** | 无需必查；可选查 `last_synced_at` 分布：<br>`SELECT ad_id, last_synced_at FROM structure_ads WHERE account_id = 'act_xxx' AND effective_status = 'ACTIVE' ORDER BY last_synced_at DESC LIMIT 10;` |
| **通过标准** | 第二次不出现与第一次完全相同的 diffHead5 且 touched>0；多轮有轮转。 |
| **失败判定** | 连续两轮同一批 id、同一批 diffHead5 且 touched>0。 |
| **可能原因 1** | 1h 条件未生效（仍用 last_synced_at DESC 或未加 1h 条件）→ `structureSyncService.js` 中 WHERE 与 ORDER BY。 |
| **可能原因 2** | 候选集过小，轮转不明显 → 同一账户 ACTIVE 数少，可多跑几轮或换账户观察。 |
| **可能原因 3** | 时钟/时区导致间隔判断错误 → 确认 DB 与应用使用 UTC（如 `UTC_TIMESTAMP()`）。 |

### 5. NULL 自愈

| 项目 | 内容 |
|------|------|
| **目的** | `last_synced_at IS NULL` 的 ACTIVE 行能被伪增量选中并补齐 `last_synced_at`。 |
| **操作步骤** | 1. 造数：`UPDATE structure_ads SET last_synced_at = NULL WHERE account_id = 'act_xxx' AND effective_status = 'ACTIVE' LIMIT 1;`（记下 ad_id）。<br>2. 触发一次统一心跳。<br>3. 查该行 `last_synced_at`；再触发一次，确认该 ad 不在新一批 diffHead5（或 diffIds=0）。 |
| **预期日志** | 第一次：该账户 `recentActiveIds>=1`、`diffIds>=1`、`touched>=1`，且 diffHead5 含该 ad_id（若在前 5）。第二次：该 ad 不再出现在 diffHead5 或 diffIds=0。 |
| **需查 SQL** | `SELECT ad_id, last_synced_at FROM structure_ads WHERE account_id = 'act_xxx' AND ad_id = '造数的那条ad_id';` |
| **通过标准** | 第一次后该行 `last_synced_at` 非 NULL；第二次伪增量不再次选中该条。 |
| **失败判定** | 更新后仍为 NULL；或下一轮仍被选中且 touched。 |
| **可能原因 1** | WHERE 未对 NULL 放行 → `structureSyncService.js` 中 `(last_synced_at IS NULL OR ...)` 两处条件。 |
| **可能原因 2** | UPSERT 未更新 `last_synced_at` → `UPSERT_STRUCTURE_SQL` 中 `last_synced_at = NOW()`。 |
| **可能原因 3** | 该 ad 在 activeAdIds 中导致被 diffIds 排除 → 仅当该账户当轮有 spend 时可能；看 diffIds 计算逻辑。 |

### 6. Piggyback 与伪增量不重复 resolve

| 项目 | 内容 |
|------|------|
| **目的** | Today 对 activeAdIds 只做一次 resolve；Piggyback 优先复用，缺口补查很少。 |
| **操作步骤** | 1. 触发一次统一心跳（保证有账户走 Today 分支）。<br>2. 在日志中搜：Today 的「同轮仅此一次 resolve」、Piggyback 的「补齐 structure_ads: 复用=X, 缺口补查=Y」。 |
| **预期日志** | Today：`批量解析活跃广告元数据（…同轮仅此一次 resolve）`。<br>Piggyback：`[Piggyback] account=act_xxx 补齐 structure_ads: 复用=X, 缺口补查=Y`，Y 通常为 0 或很小。 |
| **需查 SQL** | 无需必查。 |
| **通过标准** | 有「同轮仅此一次 resolve」；Piggyback 缺口补查为 0 或个位数。 |
| **失败判定** | 无上述 Today 日志；或 Piggyback 每轮缺口补查都很大。 |
| **可能原因 1** | Today 未返回 structurePayload 或未传 facebookApi → `ingestorService.js` 中 `syncAccountTodayStats` 返回值与调用处传参。 |
| **可能原因 2** | Piggyback 未优先用 structurePayload，直接对大量 id 调 resolve → `structureSyncService.js` 中 `piggybackStructureFromToday` 的 needUpdate/复用逻辑。 |
| **可能原因 3** | 多个地方对同一批 activeAdIds 调 resolve → 搜索 `resolveObjectsByIds`，确认仅 Today 一次 + Piggyback 仅缺口。 |

### 7. 每小时全量轮转

| 项目 | 内容 |
|------|------|
| **目的** | 整点后 5 分钟有轮转执行或明确跳过；优先级正确；usage 高/熔断时本小时跳过。 |
| **操作步骤** | 1. 等到某小时的第 5 分钟（如 14:05），看日志。<br>2. 期望看到：`[结构轮转] 本小时处理 N 个账户（并发 1）` 或 `本小时跳过: ...`。<br>3. 可选：在 usage 高或熔断时再等下一个整点后 5 分钟，看是否跳过。 |
| **预期日志** | 正常：`[结构轮转] 本小时处理 6 个账户（并发 1）`，随后有 `✅ [2.4] 结构同步完成`（若干条）。<br>跳过：`[结构轮转] 本小时跳过：Token 熔断` 或 `本小时跳过：API 使用率 85% >= 85%`。 |
| **需查 SQL** | 轮转前：`SELECT account_id, has_full_synced, last_full_count FROM structure_sync_status ORDER BY has_full_synced, last_full_count;`<br>轮转后：同上，确认未 full 的账户先被更新。 |
| **通过标准** | ① 整点后 5 分钟有轮转日志或跳过日志。<br>② 被处理的账户符合优先级（has_full_synced=0 或 last_full_count=0 优先）。<br>③ usage≥85 或熔断时出现「本小时跳过」。 |
| **失败判定** | 无轮转日志；或顺序与优先级不符；或高 usage 时仍跑满。 |
| **可能原因 1** | Cron 未注册或未到点 → `cronService.js` 中 `5 * * * *` 与 `runHourlyStructureFullRotation`。 |
| **可能原因 2** | 优先级 SQL 与预期不符 → `structureSyncService.js` 中 `runHourlyStructureFullRotation` 的 ORDER BY。 |
| **可能原因 3** | usage/熔断未写入或未读 → `rateLimitService.js` 的 `getLastUsageRate`/`lastKnownUsageRate` 与 `getCircuitBreakerStatus()`。 |

### 8. 权限与安全（手动同步）

| 项目 | 内容 |
|------|------|
| **目的** | 非负责人/非管理员不能同步他人账户；越权请求返回 403，且不触发实际同步。 |
| **操作步骤** | 1. 用**非管理员**且**非该账户负责人**的账号登录，拿到 JWT。<br>2. 调用 `POST /api/sync/structure`，body 里填**他人**的 `account_id`。<br>3. 看响应状态码与 body；看服务端日志是否有 `syncAccountStructureAds` 或 FB 请求。 |
| **预期日志/响应** | HTTP 403；body 含 `error`/`ACCOUNT_FORBIDDEN` 等；服务端**无** `[2.4] 结构同步完成` 或 FB 拉取日志。 |
| **需查 SQL** | 可选：确认该账户在 `account_mappings` 中不属于当前用户的 `owner_id`。 |
| **通过标准** | 403，无实际同步与 FB 调用。 |
| **失败判定** | 返回 200 或执行了同步。 |
| **可能原因 1** | 未做账户权限校验 → `server/index.js` 中 `POST /api/sync/structure` 是否在调用 `syncAccountStructureAds` 前有 `assertAccountAccess(req, res, accountId)`。 |
| **可能原因 2** | assertAccountAccess 逻辑错误 → `server/utils/accountAccess.js` 中 admin/owner 判断与 SQL。 |
| **可能原因 3** | 前端/调用方传错 account_id（如传成自己的）→ 用他人 account_id 直接调接口复现。 |
