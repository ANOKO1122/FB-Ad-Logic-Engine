# 结构同步（近 3 天）改造 — 测试与验证指南

本文说明如何测试和验证「结构同步任务改造」及三项补充采纳（轮转命名、last_filter_since_sec、过滤 value 自动回退）是否正确生效。

---

## 一、前置条件

1. **执行迁移 030**（若尚未执行）  
   ```bash
   mysql -u 用户 -p 数据库 < server/db/migrations/030_add_last_filter_since_sec_to_structure_sync_status.sql
   ```  
   验证：`DESCRIBE structure_sync_status;` 应看到 `last_filter_since_sec` 列。

2. **环境**：本地或测试环境，能调用 Facebook Marketing API（有效 Token、账户有 campaigns/adsets/ads）。

---

## 二、验证项

### 1. created_time 落库

- **操作**：执行一次结构同步（见下方「触发方式」），然后查库。
- **SQL 示例**：  
  ```sql
  SELECT account_id, campaign_id, name, created_time, updated_time, last_synced_at
  FROM structure_campaigns ORDER BY last_synced_at DESC LIMIT 5;

  SELECT account_id, adset_id, name, created_time, updated_time, last_synced_at
  FROM structure_adsets ORDER BY last_synced_at DESC LIMIT 5;

  SELECT account_id, ad_id, name, created_time, updated_time, last_synced_at
  FROM structure_ads ORDER BY last_synced_at DESC LIMIT 5;
  ```
- **预期**：近 3 天内有更新或新建的对象被同步后，`created_time`、`updated_time` 有值（ISO 或合理时间字符串），不再全为空。

### 2. 近 3 天过滤生效（拉取量减少）

- **操作**：看日志中本次同步的 campaigns/adsets/ads 数量。
- **预期**：  
  - 日志出现 `[2.4] 结构同步完成 account=act_xxx 近3天 campaigns=N adsets=M ads=K`。  
  - N、M、K 应为「近 3 天有更新」的子集，通常比全量少（除非账户本身近期改动很多）。

### 3. 轮转命名与日志口径一致

- **操作**：  
  - 由**定时**触发一次结构轮转（等 Cron 或手动调 `runHourlyStructureFullRotation`），看日志。  
  - 或手动调用 `POST /api/sync/structure`（若已实现）触发单账户同步，看日志。
- **预期**：  
  - 轮转相关日志含「结构轮转-近3天」或「近3天」，不再出现「全量轮转」等易误解为「全量拉取」的措辞。  
  - 单账户同步日志含「近3天」或「结构同步（近3天）」。

### 4. last_filter_since_sec 可观测

- **操作**：执行一次结构同步后查表。  
  ```sql
  SELECT account_id, last_success_at, last_filter_since_sec,
         FROM_UNIXTIME(last_filter_since_sec) AS since_iso
  FROM structure_sync_status
  WHERE account_id = 'act_你的账户ID';
  ```
- **预期**：  
  - `last_filter_since_sec` 有值，为约「当前时间减 3 天」的 Unix 秒。  
  - `since_iso` 约为 3 天前的 UTC 时间，与本次同步使用的过滤起点一致。

### 5. 过滤 value 自动回退（可选，需可制造报错时再测）

- **场景**：若某 edge 使用 Unix 秒时 FB 报错（如格式不支持），代码应自动用 ISO8601 重试。
- **操作**：难以在正常环境稳定复现时，可临时在 `fetchAllPagesForEdge` 内对指定 edge 第一次请求抛错，观察日志是否出现「使用 Unix 秒报错，回退 ISO8601」且第二次请求成功。
- **预期**：  
  - 仅该 edge 回退为 ISO，其余 edge 仍优先 Unix 秒。  
  - 同步最终成功，无需改代码。

---

## 三、触发方式

- **手动单账户**：若项目提供 `POST /api/sync/structure`（带 accountId 或从会话取），调用一次即可触发 `syncAccountStructureAds`。
- **定时轮转**：等待 Cron（例如每小时第 5 分钟）执行 `runHourlyStructureFullRotation`，或在本机临时调用该函数（需传入 facebookApi 实例）。
- **脚本**：若有类似 `server/scripts/xxx.js` 里调用 `syncAccountStructureAds(accountId, facebookApi)`，可直接运行脚本触发。

---

## 四、回归注意

- **补数**：某账户三张结构表为空时，方案不做「首次全量」，需用现有补数脚本（如 `backfill-adsets-act_xxx.js`）先铺表，再依赖近 3 天同步维持更新。
- **冷却**：同一账户 2 分钟内重复触发结构同步会返回 `cooldown`，属预期。
- **选择器**：规则管理里选择广告/广告组/广告系列时仍从 structure_* 查库，行为不变；仅同步阶段改为「近 3 天」拉取。

按上述步骤验证通过即可认为本次改造与三项补充采纳工作正常。
