## 小白向：你查到的 MySQL 时区结果代表什么？

你在 MySQL 客户端里执行：

```sql
SELECT
  @@system_time_zone,
  @@global.time_zone,
  @@session.time_zone,
  NOW()           AS now_session,
  UTC_TIMESTAMP() AS now_utc;
```

**结果解读：**

| 列 | 你看到的值 | 含义（通俗说） |
|----|------------|----------------|
| `@@system_time_zone` | 乱码（实为中国时区） | 你这台电脑/服务器用的时区 |
| `@@global.time_zone` | SYSTEM | MySQL 用“系统时区” |
| `@@session.time_zone` | SYSTEM | 当前这次连接用的也是系统时区（= 北京） |
| `NOW()` (now_session) | 2026-03-04 13:52:38 | 你本地时间（北京时间） |
| `UTC_TIMESTAMP()` (now_utc) | 2026-03-04 05:52:38 | 世界标准时间（比北京晚 8 小时） |

所以：**13:52 和 05:52 差 8 小时是正常的**，一个北京、一个 UTC。

**重要：**  
- 你用 **MySQL 客户端**（命令行/Workbench）连库时，默认就是 SYSTEM（北京），所以 `NOW()` 一直是北京时间。  
- 我们的 **Node 后端**在连数据库时，已经每次都会执行 `SET time_zone = '+00:00'`，所以**后端读写数据库时用的是 UTC**，你不需要改 MySQL 服务器配置也能让“新写入的数据按 UTC 存”。

**接下来你可以怎么做（三选一，按需）：**

1. **什么都不改**  
   后端照常跑，时区方案按计划做（DB 存 UTC、前端显示北京）。你查表时若想直接看北京时间：
   - **若你当前 MySQL 会话是北京时区**（`@@session.time_zone = SYSTEM` 且本机在中国）：`SELECT` 出来的 `triggered_at` **已经是北京时间**，直接当“北京时间”看即可，**不要再做** `CONVERT_TZ(triggered_at, '+00:00', '+08:00')`，否则会多加 8 小时。
   - **若你当前会话是 UTC**（执行过 `SET time_zone = '+00:00'`）：可用下面查询看到北京时刻：
   ```sql
   SELECT id, rule_id, triggered_at,
          CONVERT_TZ(triggered_at, '+00:00', '+08:00') AS 北京时间
   FROM automation_logs ORDER BY id DESC LIMIT 10;
   ```

2. **只在“当前这次”MySQL 会话里用 UTC 看**  
   每次打开 MySQL 客户端后，先执行一句：
   ```sql
   SET time_zone = '+00:00';
   ```
   之后再查表，`NOW()` 和表里存的 UTC 时间就一致了（只对当前窗口有效，关掉重连又会变回 SYSTEM）。

3. **让 MySQL 客户端默认就用 UTC（可选）**  
   在 MySQL 配置文件（如 Windows 的 `my.ini`）里 `[mysqld]` 下加一行 `default_time_zone = '+00:00'`，重启 MySQL；这样所有新连接默认都是 UTC（包括你手动打开的客户端）。  
   若你不熟配置文件位置，用上面 1 或 2 即可，不必强求。

- **历史数据（可选）**

**你遇到的两种情况说明：**

- **「数据库中北京时间快了 8 小时」**

- **「截图里执行日志前端时间少了 8 小时」**
