## 一、现状核对清单

### 1. spend>0 过滤 — 后端位置

| 数据路径 | 文件 | 函数/位置 | 说明 |
|----------|------|-----------|------|
| API 嗅探（单账户 Today 全量） | `server/services/ingestorService.js` | 约 **310–315 行**，`syncAccountTodayStats` 内 | `insights = insights.filter(insight => { const spend = parseFloat(insight.spend \|\| 0); return spend > 0 })`；过滤后无数据则提前 return |
| 滑动窗口 Today 写入 | `server/services/ingestorService.js` | 约 **893–902 行**，`syncAccountTodayStatsWithSlidingWindow` 内 | `filteredTodayInsights = todayInsights.filter(... spend > 0)`；日志：`Today 过滤结果: ${todayInsights.length} → ${filteredTodayInsights.length} (只保留 spend>0)` |
| 队列写入器（统一落盘） | `server/services/ingestorService.js` | 约 **1617–1637 行**，队列消费/写入逻辑内 | 以当前仓库实际代码为准：可能是严格 `spend > 0`，也可能是宽松条件 `spend>0 \|\| impressions>0 \|\| linkClicks>0 \|\| hasPurchases`（允许 spend=0 但有曝光/点击/购买的数据进热表）。日志：`🧹 [队列写入器] 数据清洗: ...` |

**细节风险点（验收时必读）**  
- 队列写入器（L1617–L1637）**不保证**在所有版本/分支中都是「严格 spend>0」：有的实现是 **spend>0 \|\| impressions>0 \|\| linkClicks>0 \|\| hasPurchases**，即可能允许 spend=0 但有曝光/点击/购买的数据写入热表。  
- today 两条主链路（Insights First 的 today 同步、Sliding Window 的 today 写入前）在上游都做了 spend>0 过滤，所以通常不会把 spend=0 送进队列；但**口径验收必须用数据库结果说话**，不能只看某一段代码或注释。  
- **结论**：以 **SQL 1**（today 是否存在 spend=0，期望 0 条）为最终验收标准；若 SQL 1 通过，即认定「today 只写 spend>0」成立，与队列写入器当前是严格还是宽松条件无关。

## 二、验证用 SQL（2～3 条）

在 MySQL 中执行以下语句时，请把 `CURDATE()` 换成你实际关心的「今日」日期（若库是 UTC 或与业务「今日」不一致，可改为 `'YYYY-MM-DD'` 或按账户时区分天查）。

---

### SQL 1：today 是否存在 spend=0（期望：0 条）

用于验收：**DB 中不存在 spend=0 的 today 快照**。

```sql
-- 检查 ad_snapshots 中「数据日期 = 今天」且 spend=0 或 NULL 的记录数（期望 0）
SELECT COUNT(*) AS spend_zero_today_count
FROM ad_snapshots
WHERE data_date = CURDATE()
  AND (COALESCE(spend, 0) = 0 OR spend IS NULL);
```

- **通过**：`spend_zero_today_count = 0`  
- **不通过**：若 > 0，说明至少有一条 today 快照是 spend=0，需回头检查三条过滤路径是否都生效、是否有历史脏数据未清理。

---

### SQL 2：status 取值分布（佐证与 FB 一致）

用于佐证：**status 列存的是 ACTIVE/PAUSED 等与 FB 一致的取值**（代码上来自 effective_status）。

```sql
-- 查看 ad_snapshots 中 today 数据的 status 分布（应与 FB 广告后台一致）
SELECT status, COUNT(*) AS cnt
FROM ad_snapshots
WHERE data_date = CURDATE()
GROUP BY status
ORDER BY cnt DESC;
```

- **通过**：出现 `ACTIVE`、`PAUSED`、`ARCHIVED`、`DELETED` 等 FB 常见 effective_status 值，且无异常占位符（如全为 NULL 或未知字符串）。  
- **不通过**：若大量 NULL 或与 FB 后台明显不一致，需检查 statusMap 是否覆盖到所有写入路径、FB 接口是否返回了 effective_status。

---

### SQL 3（可选）：各账户 today 条数与 spend>0 占比

用于多账户环境下快速看「每个账户 today 是否全是 spend>0」。

```sql
-- 按账户看 today 条数及 spend>0 占比（期望 spend_gt_zero_pct = 100）
SELECT
  account_id,
  COUNT(*) AS total,
  SUM(CASE WHEN COALESCE(spend, 0) > 0 THEN 1 ELSE 0 END) AS spend_gt_zero,
  ROUND(100.0 * SUM(CASE WHEN COALESCE(spend, 0) > 0 THEN 1 ELSE 0 END) / COUNT(*), 2) AS spend_gt_zero_pct
FROM ad_snapshots
WHERE data_date = CURDATE()
GROUP BY account_id;
```

- **通过**：每个账户 `spend_gt_zero_pct = 100`。  
- **不通过**：若某账户 < 100，说明该账户存在 spend=0 的 today 记录，需结合 SQL 1 排查。

## 三、日志：过滤前后数量如何看

以下均为**应用日志**（控制台或 Winston 等），需在「触发一次 Today 同步」之后查看。

| 场景 | 日志关键字 / 内容 | 含义 |
|------|------------------|------|
| 滑动窗口 Today 写入 | `Today 过滤结果:` 或 `只保留 spend > 0` | 格式：`Today 过滤结果: ${todayInsights.length} → ${filteredTodayInsights.length} (只保留 spend>0)`，即过滤前 → 过滤后条数 |
| 队列写入器统一清洗 | `[队列写入器] 数据清洗:` | 格式：`数据清洗: ${insights.length} → ${cleanInsights.length} (过滤 ${filteredCount} 条 spend=0 的广告)` |
| API 嗅探无 spend>0 | `没有 spend>0 的广告（API 查询结果）` | 该账户当次 API 结果里没有 spend>0，跳过同步 |

验证时：**跑一轮 Today 同步 → 在日志中搜上述关键字 → 确认有「过滤前 → 过滤后」且过滤后仅 spend>0 再写入**即可。
