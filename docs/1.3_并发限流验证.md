# TASKS §1.3 并发+限流验证

## 一、验证目标

在「并发度 = 6 + Header 自适应休眠策略」下：
1. 整体同步时间明显优于串行
2. 不会出现严重限流或大量 429 错误

## 二、自动化验证（无 FB 依赖）

### 2.1 单测

```bash
npm test -- --run server/tests/concurrentLimit.test.js
```

**预期**：36 任务、并发度 6 时，加速比 ≥ 2x，最大并发数 ≤ 6。

### 2.2 验证脚本

```bash
node server/scripts/verify-concurrent-limit.js
```

**预期**：输出 `✅ 加速比 X.XXx >= 2x`、`✅ 最大并发数 6 <= 6`，exit 0。

## 三、真实环境验证（需 36 账户）

当 `account_mappings` 中有约 36 个活跃账户时：

1. 手动触发统一心跳：`node -e "import('./server/services/cronService.js').then(m=>m.manualUnifiedHeartbeat())"`
2. 观察日志：
   - `使用受控并发模式（并发度 = 6）`
   - 整体耗时（应明显优于串行理论值 36 × 单账户耗时）
3. 检查 429 错误：`grep -i 429 logs/combined-*.log` 或应用日志中检索
4. **验收**：无大量 429，同步完成时间可接受

## 四、相关代码

- `server/services/ingestorService.js`：`CONCURRENT_LIMIT=6`、`accountTaskLimiter`
- `server/services/rateLimitService.js`：`parseUsageHeader`、`sleepBasedOnUsage`
