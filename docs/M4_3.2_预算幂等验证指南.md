# M4 步骤 3.2 预算幂等 — 详细验证指南

按下面顺序做，每步都有「预期结果」和「怎么看算通过」。

---

## 一、验证 1：纯函数 `computeNewBudgetCentsOnce`（不碰数据库、不碰 FB）

**目的**：确认「给当前预算 + 动作」算出的新预算（美分）正确，且护栏生效。

### 方式 A：用项目自带 Node 跑小脚本（推荐）

1. 在项目根目录新建或使用下面的脚本。

2. 创建文件 `server/tests/verify-budget-once.js`（见下方「脚本内容」）。

3. 在项目根目录执行：

   ```bash
   node server/tests/verify-budget-once.js
   ```

4. **预期**：终端输出里能看到类似：

   ```
   [1] 1000 分 +10% => 期望 1100, 实际 1100  通过
   [2] 1000 分 -10% => 期望 900, 实际 900   通过
   [3] 1000 分 -99% 下限护栏 => 期望 100, 实际 100  通过
   [4] 500 分 上限 300 分 => 期望 300, 实际 300  通过
   [5] 未知 type 走减少分支(默认10%) => 期望 900, 实际 900  通过
   全部通过
   ```

   若全部为「通过」且无报错，则验证 1 通过。

### 方式 B：不用脚本，用 Node REPL 手跑

1. 在项目根目录执行：

   ```bash
   node
   ```

2. 在 REPL 里粘贴（注意本项目是 ES Module，需用动态 import）：

   ```js
   import('./server/services/actionExecutorService.js').then((m) => {
     const { computeNewBudgetCentsOnce } = m
     // 1) 1000 分增加 10% => 1100
     console.log('1100?', computeNewBudgetCentsOnce(1000, { type: 'increase_budget', value: 10 }))
     // 2) 1000 分减少 10% => 900
     console.log('900?', computeNewBudgetCentsOnce(1000, { type: 'decrease_budget', value: 10 }))
     // 3) 下限护栏：减少 99% 不低于 100 分
     console.log('100?', computeNewBudgetCentsOnce(1000, { type: 'decrease_budget', value: 99 }))
     // 4) 上限护栏：max_daily_budget 300 分
     console.log('300?', computeNewBudgetCentsOnce(500, { type: 'increase_budget', value: 50, max_daily_budget: 300 }))
   })
   ```

3. **预期**：依次输出 `1100`、`900`、`100`、`300`。若一致，则验证 1 通过。

## 二、验证 2：执行链路里「未传 _resolvedBudgetCents」时会 GET 一次并算出新预算

**目的**：确认走预算分支时，在没有预算缓存的情况下会打一次「当前预算」日志并计算新预算。

### 前提

- 有一条**启用的**规则，且该规则的 **actions** 里包含 `increase_budget` 或 `decrease_budget`（且该规则能匹配到至少一个广告，否则不会执行到预算逻辑）。
- 若没有这样的规则，需要先在规则管理里创建/编辑一条，例如 actions 设为：`[{ "type": "increase_budget", "value": 10 }]`，并保证规则能命中某个有 `ad_set_id` 的广告（否则会因缺 adset_id 报错，不验证幂等）。

### 步骤

1. **启动后端**（若未启动）：

   ```bash
   npm run dev
   # 或
   node server/index.js
   ```

2. **触发一次规则执行**（任选其一）：
   - 前端：规则管理页点击「立即运行所有规则」；
   - 或调用接口：  
     `POST /api/rules/execute-all`  
     （需要带登录态或 Bearer Token，见项目 curl 文档）。

3. **看后端终端日志**，找到本次 run 里**预算相关**的那几行，应类似：

   ```
   🔧 执行: 增加广告组 xxx 预算 10%
      当前预算: x.xx 美元 (xxx 分)
      新预算: xxx 分 (增加 10%)
   ✅ 成功增加预算: xxx 分
   ```

4. **判定**：
   - 若规则命中了广告且动作是预算类，且日志里出现**一次**「当前预算: … 美元」和「新预算: … 分」，则说明「未传 _resolvedBudgetCents 时会 GET 一次并计算」的逻辑正确，验证 2 通过。
   - 若规则没有命中任何广告、或命中的广告没有 ad_set_id，则不会进预算分支，不会看到上述日志，需换一条能命中且带 ad_set_id 的预算规则再试。

## 三、验证 3：传入 _resolvedBudgetCents 时不再 GET、直接用该值（幂等路径）

**目的**：确认当调用方在 action 上挂了「已算好的预算（分）」时，执行器不会再去调 getAdsetBudget，只打「使用已算好的预算」并 POST。

### 方式 0：用 Vitest 或 pytest 跑单测（推荐）

- **Vitest**（在项目根目录执行）：
  ```bash
  npm test -- --run server/tests/actionExecutorBudgetIdempotent.test.js
  ```
  通过即表示：未传 _resolvedBudgetCents 时 getAdsetBudget 被调 1 次；传入 _resolvedBudgetCents 时 getAdsetBudget 不被调、只 POST 该值。

- **pytest**（在项目根目录执行）：
  ```bash
  pytest test_m4_budget_idempotent.py -v
  ```
  该用例内部会调上述 Vitest 单测并检查退出码，通过即验证 3 通过。

### 方式 A：临时改 cronService 传 _resolvedBudgetCents（仅验证用）

1. 打开 `server/services/cronService.js`，找到「执行每个广告」的那段，即调用 `executeActionsForAd` 的地方（传入 `actionsOverride: [meta.winnerAction]` 的那里）。

2. **临时**把「只传 winnerAction」改成「先算好 newBudgetCents 再传副本」：
   - 若当前动作是 `increase_budget` 或 `decrease_budget`，可先调用 `getAdsetBudget`（或用固定值如 1000 美分）算出 `newBudgetCents`（用 `computeNewBudgetCentsOnce(1000, meta.winnerAction)` 即可），
   - 然后传 `actionsOverride: [{ ...meta.winnerAction, _resolvedBudgetCents: newBudgetCents }]`。

3. 再触发一次规则执行，看日志应出现：
   - **不再**出现「当前预算: … 美元」；
   - 出现「使用已算好的预算（幂等）: xxx 分」。

4. 验证完后**务必把这段临时改动还原**，恢复为只传 `[meta.winnerAction]`。

### 方式 B：写一个只测 actionExecutorService 的小测试（推荐，可保留）

在 `server/tests/` 下加一个测试文件，用 Vitest  mock `api.getAdsetBudget` / `api.updateAdsetBudget`：

- 第一次调用 `executeActionsForAd` 时**不**传 `_resolvedBudgetCents`：断言 `getAdsetBudget` 被调用了 1 次，`updateAdsetBudget` 被调用 1 次，且传入的预算值等于 `computeNewBudgetCentsOnce(当前预算美分, action)`。
- 第二次用**同一 rule、同一 matchedAd**，但 `actionsOverride: [{ ...action, _resolvedBudgetCents: 1234 }]`：断言 `getAdsetBudget` **没有被再调用**，`updateAdsetBudget` 被调用 1 次且参数为 `12.34`（1234 分 = 12.34 美元）。

这样就能稳定验证「传了 _resolvedBudgetCents 就不再 GET、只 POST 该值」。

## 四、验证 4：没有在 rule.actions 原对象上挂 _resolvedBudgetCents（代码审查）

**目的**：确保幂等是通过「调用方传 action 副本 + _resolvedBudgetCents」实现的，而不是改库里的 rule 或内存里的 rule.actions。

1. 打开 `server/services/actionExecutorService.js`，搜索 `_resolvedBudgetCents`。

2. **预期**：
   - 只有**读取** `action._resolvedBudgetCents`（例如 `if (action._resolvedBudgetCents != null ...)`），
   - **没有任何**对 `action._resolvedBudgetCents = ...` 或 `rule.actions[...]._resolvedBudgetCents = ...` 的赋值。

3. 再打开 `server/services/cronService.js`，搜索 `_resolvedBudgetCents`。

4. **预期**：当前实现里 cron 没有给 `rule.actions` 或共享的 action 对象挂 `_resolvedBudgetCents`（只有将来若做「重试时传副本」时，会构造新对象 `{ ...winnerAction, _resolvedBudgetCents }` 传下去）。

若以上都满足，则验证 4 通过。

---

## 五、验收结论怎么填

- **验证 1**：纯函数结果与预期一致（含上下限护栏） → 计算逻辑正确。
- **验证 2**：未传 _resolvedBudgetCents 时，日志里有一次「当前预算」+「新预算」且执行成功 → 正常路径正确。
- **验证 3**：传了 _resolvedBudgetCents 时不再出现「当前预算」、只出现「使用已算好的预算（幂等）」→ 幂等路径正确（或用单测等价验证）。
- **验证 4**：代码里没有在 rule.actions 上写 _resolvedBudgetCents → 不会串 ad。

全部通过即可认为 3.2 预算幂等验收完成。
