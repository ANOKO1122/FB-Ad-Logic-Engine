监控范围条件「创建时间段」— 执行方案
================================
一、目标与口径
位置：在「监控范围条件」的字段下拉中增加第三项「创建时间段」，与「名称」「状态」并列（同一行：字段 | 操作符 | 值）。
语义：从当前时间往前 N 小时内创建的对象才符合条件。
例如当前为 12:00，选择「近 24 小时内」，则仅「昨天 12:00 至今天 12:00 之间创建」的对象计入。
行为：与「名称」「状态」完全一致：
后端 SQL 在结构表上按条件过滤，只返回匹配项；
单账户/多账户、首页/加载更多都带同一组过滤参数；
多账户时，每个账户各自按条件查库，再在后端合并结果。
适用范围：广告 / 广告组 / 广告系列 三个层级（structure_ads / structure_adsets / structure_campaigns 均已具备 created_time 并随结构同步落库）。
行之间关系：所有监控范围行（名称 / 状态 / 创建时间段）之间统一为 AND 关系：对象必须同时满足每一行条件。
二、前端改造（RuleManager.vue）
2.1 字段下拉增加「创建时间段」
位置：监控范围条件部分，字段下拉：
<select v-model="row.field" class="select scope-field-select" @change="onScopeConditionFieldChange(row)">  <option value="name">名称</option>  <option value="status">状态</option>  <!-- 新增 -->  <option value="created_within">创建时间段</option></select>
效果：字段下拉展开时为「名称 / 状态 / 创建时间段」三项，与截图交互一致。
2.2 操作符（中间一列）
规则：当 row.field === 'created_within' 时，操作符固定为「包含」（语义为「在…时间内」）。
实现建议：
在 onScopeConditionFieldChange(row) 中，当切换为 created_within 时，强制设置 row.operator = 'include'。
模板中对 row.field === 'created_within' 时：
可以只渲染一个「包含」的 <option>，或
直接显示只读文本「包含」，保持三列表头对齐但不允许修改。
2.3 值（第三列）：时间范围 + 自定义小时
当 row.field === 'created_within' 时，第三列控件变为「预设时间段 + 自定义小时」：
下拉枚举（存到 row.value）：
近 24 小时内（value: '24'）
近 48 小时内（value: '48'）
近 72 小时内（value: '72'）
自定义小时内（value: 'custom'）
当 row.value === 'custom' 时，显示一个数字输入框，例如：
v-model.number="row.customHours"
占位："请输入小时数"
建议范围：1–720（最多 30 天，可按项目需要调整）。
校验逻辑（scopeConditionRowError(row)）：
当 row.field === 'created_within' 时：
若 row.value 为 '24' | '48' | '72' → 合法；
若 row.value === 'custom'，则要求 row.customHours 为合法正整数且在允许区间内，否则返回错误提示「请选择时间范围或填写小时数」；
其它情况视为未填写完整。
2.4 多行 AND 关系（支持多条创建时间段）
多个监控范围行之间整体为 AND：对象必须同时满足每一行。
「创建时间段」字段允许多行，语义为连续 AND：
多个时间段 AND 在一起，等价于「取所有时间段中最短的时间窗口」。
为了避免 SQL 过于复杂，前端在计算时可将所有 field === 'created_within' 的行汇总为一个有效小时数：
    const createdRows = completed.filter(r => r.field === 'created_within')    // 提取每一行的 hours（24/48/72 或 customHours）    const hoursList = createdRows      .map(r => resolveHoursFromRow(r)) // 返回数字或 null      .filter(h => Number.isFinite(h) && h > 0)    const effectiveCreatedWithinHours = hoursList.length ? Math.min(...hoursList) : null
最终传给后端的 scope_created_within_hours 使用 effectiveCreatedWithinHours，从语义上等价于「所有时间段的交集」，符合 AND。
2.5 应用监控范围条件（含创建时间段）
在 applyScopeConditions 中：
从 scopeConditionRows 中筛选已完成的行（无校验错误）。
从中：
抽取名称条件（field === 'name'）→ nameIncludeKeyword / nameExcludeKeyword；
抽取状态条件（field === 'status'）→ scopeStatus 与 scope_status_exclude；
抽取所有创建时间段条件（field === 'created_within'），用上一节的逻辑计算 effectiveCreatedWithinHours。
新增一个 ref：scopeCreatedWithinHours（数字或 null）：
若 effectiveCreatedWithinHours 存在，则 scopeCreatedWithinHours.value = effectiveCreatedWithinHours；
否则 scopeCreatedWithinHours.value = null。
needFetchAll 的逻辑与现有保持一致：只要存在任意名称/状态/创建时间段条件（即 completed.length > 0），则判定为需要带过滤重新拉列表。
在 needFetchAll 分支中：
将 scopeSearch 设置为「名称 包含」行的关键字（若有）；
将 scopeNameExclude 设置为「名称 不包含」行的关键字（若有）；
状态条件设置 scopeStatusFilter / scopeStatusExcludeFilter；
若存在状态条件，则强制 scopeIncludePaused.value = true；
新增：不论是否有状态条件，只要有创建时间段条件，也应参与 needFetchAll。
2.6 列表请求：始终带上创建时间段参数
在 refreshScopeItems 中（单账户与多账户两个分支）调用 facebookApi.getStructureObjects / getStructureObjectsMulti 时，params 增加：
scope_created_within_hours: scopeCreatedWithinHours.value ?? undefined
在 loadMoreScopeItems 中，同样增加上述参数，确保分页时过滤条件一致：
scope_created_within_hours: scopeCreatedWithinHours.value ?? undefined
2.7 当前列表匹配与自动勾选
后端已按 scope_created_within_hours（及名称/状态）过滤，列表接口返回的就是满足所有监控范围条件的子集。
applyScopeConditionsToCurrentList 中：
仍使用已有的 match(item) 对名称和状态做前端侧校验（保持一致性）；
创建时间段可以不再二次校验（已由 SQL 过滤），也可以按需加上（需要接口返回 created_time）。
匹配成功的 scopeItems 仍按现有逻辑全部勾选：
单账户：targetIds = items.map(it => it.id)。
多账户：targetIds = items.map(it => \\${it.account_id}:\${it.id}\)。
当有更多页时，点击「加载更多」会继续带着相同的过滤条件（包括创建时间段），新拉回的项目也按相同规则自动勾选。
2.8 清空 / 重置 / 回显
清空监控范围条件或切换目标层级、账户时：
保持现有行为：
当没有「名称 包含」条件时清空 scopeSearch；
清空 scopeStatusFilter / scopeStatusExcludeFilter / scopeNameExclude；
新增：同时清空 scopeCreatedWithinHours（设为 null），防止旧的时间段绑在新条件上。
回显规则（编辑已有规则）时：
若规则中保存了 field: 'created_within' 的行：
重建对应的 scopeConditionRows；
将 scopeCreatedWithinHours 设置为所有创建时间段行中「最短窗口」的小时数（与 AND 语义一致）。
2.9 自动勾选与层级 / 账户变更
保持现有「自动抖动 + 自动应用」机制：
监控范围条件 / 已选广告账户 / 目标层级 三者任一变更时，已有的 watch 会在 2 秒防抖后调用 applyScopeConditions。
创建时间段行参与「完整条件」的判定：
只要存在至少一行包含创建时间段且校验通过，就视为「有完整条件」；
改变账户或目标层级时，会重新走一次「带创建时间段的列表请求 → 自动勾选」，与名称/状态保持一致。
三、前端 API 层（facebookApi.js）
getStructureObjects(level, accountId, opts) 增加参数：
options 增加属性：scope_created_within_hours（数字或 null）；
params 增加：scope_created_within_hours: scope_created_within_hours ?? undefined。
getStructureObjectsMulti(type, accountIds, opts) 同样增加：
options 增加 scope_created_within_hours；
params 中带上 scope_created_within_hours。
仅当 scope_created_within_hours 有合法数值时才传，保持对旧调用的兼容。
四、后端改造
4.1 structureSyncService.js：三类结构列表支持创建时间段过滤
以 listStructureAdsFromDb 为例，其它两个方法逻辑相同。
读取参数
从 opts 中读取 scope_created_within_hours：
解析为整数 hours；
仅当 Number.isFinite(hours) && hours > 0 时认为有效。
构建 WHERE 条件
已有条件：
account_id = ?
effective_status IN (...)
可选：AND name LIKE '%q%'
可选：AND (name NOT LIKE '%name_exclude%' OR name IS NULL)
分页：AND id > ?
新增创建时间段条件：
结构表中 created_time 为 VARCHAR 存放 ISO8601（写库时已归一化为 UTC，例如 2026-02-28T03:15:00.000Z）。
SQL 条件：
       AND created_time IS NOT NULL       AND created_time >= (UTC_TIMESTAMP() - INTERVAL ? HOUR)
或在需要时使用 CAST(created_time AS DATETIME) / CONVERT_TZ 等方式做时间比较（视当前 MySQL 对 ISO 字符串的比较行为而定，但推荐在迁移时已保证 ISO 形式可与 DATETIME 直接比较）。
对应在 params 中追加 hours。
listStructureCampaignsFromDb / listStructureAdsetsFromDb
与广告完全一致，只是表与字段换成 structure_campaigns / structure_adsets。
同样增加 created_time 过滤。
listStructureObjectsFromDb
在 listOpts 中增加 scope_created_within_hours: opts.scope_created_within_hours；
下游调用 listStructureCampaignsFromDb / listStructureAdsetsFromDb / listStructureAdsFromDb 时一并透传。
4.2 server/index.js 路由：透传 scope_created_within_hours
涉及的结构列表接口：
GET /api/structure/:level（ads / campaigns / adsets）
GET /api/structure/objects
GET /api/structure/objects/multi
在各路由中：
从 req.query.scope_created_within_hours 读取原始值；
解析为整数 hours（例如 parseInt）；
若 hours 非数字或 ≤ 0，则视为未设置；
否则传入对应的 list 方法 opts，例如：
    const scopeCreatedWithinHoursRaw = req.query.scope_created_within_hours    const scopeCreatedWithinHours = scopeCreatedWithinHoursRaw != null      ? parseInt(scopeCreatedWithinHoursRaw, 10)      : undefined    const opts = {      q, limit, after, include_paused,      scope_status, scope_status_exclude, name_exclude,      scope_created_within_hours: Number.isFinite(scopeCreatedWithinHours) && scopeCreatedWithinHours > 0        ? scopeCreatedWithinHours        : undefined    }
4.3 时间滑动窗口与时区口径
创建时间段的窗口基于 数据库当前时间 UTC_TIMESTAMP()，每次调用列表接口都会重新计算：
当日 2 月 28 日 12:00，选择「近 24 小时内」→ SQL 条件为 created_time >= (UTC_TIMESTAMP() - INTERVAL 24 HOUR)，窗口为 [2 月 27 日 12:00, 2 月 28 日 12:00]。
次日 2 月 29 日 12:00 再次应用同一规则时，窗口自动前滚为 [2 月 28 日 12:00, 2 月 29 日 12:00]，不会固定在首次应用的自然日上。
写库时 created_time 已统一归一化为 UTC ISO8601；查询时用 UTC_TIMESTAMP() 保证「过去 N 小时」在不同机器与时区下语义一致。
将来若需要在 UI 上用北京时间展示「创建时间段」，可在读出后使用 CONVERT_TZ 或前端格式化，不影响本期实现的过滤逻辑。
五、与结构同步「近 3 天」的关系
结构同步任务本身（syncAccountStructureAds）已按「当前时间 − 3 天」只拉近 3 天有更新的对象，并写入 created_time 与 updated_time，同时在 structure_sync_status.last_filter_since_sec 中记录本轮起点。
该行为也是滑动窗口：每天每轮都会重新计算「当前时间 − 3 天」，与创建时间段的「当前时间 − N 小时」一致，都会自动前滚。
「创建时间段」监控范围是在「已同步的近 3 天结构数据」之上，再做一层「近 N 小时内创建」过滤：
不改变同步行为，只是在已有的结构镜像数据中精细筛出最近创建的子集；
若某对象创建于 4 天前，即便本轮同步因「近 3 天」策略未包含它，其创建时间也自然不会落入「近 N 小时」，符合预期。
