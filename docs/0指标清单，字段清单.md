# 指标字段矩阵（前端可选指标 ↔ DB 字段 ↔ 计算公式 ↔ 写入链路）

目的：避免出现“前端能选但后端算不出来/永远为 null”的指标；并提供一套固定的新增指标流程（**先确认数据源与字段** → **再上前端** → **最后补验证**）。

本表覆盖当前 `src/views/RuleManager.vue`「触发条件（When）」下拉里的指标。

---

## 1. 数据分层与口径（先统一脑图）

- **`ad_snapshots`（热表）**：用于 today / 真空期兜底（未归档时的 yesterday），字段以“分子/分母 + 原始计数”为主。
- **`daily_stats`（冷表）**：用于按日与多日窗口（yesterday / last_3_days 等），同样保存原始计数，便于读侧重算比值/单次成本。
- **读侧统一计算**（推荐做法）：在 `server/services/ruleDataService.js` 中用原始计数字段重算比值/单次成本，分母为 0 时返回 `null`（规则不会误触发）。

---

## 2. 字段矩阵（你前端能选的每一项）

说明：
- **DB 列**：`ad_snapshots` / `daily_stats` 是否有该列（或等价列）  
- **计算**：是否由读侧计算（推荐），以及公式  
- **依赖字段**：计算必须的分子/分母字段（是否在迁移 003 中已添加）  
- **写入链路**：哪些任务会把依赖字段写进 DB（today 同步 / 归档 / 回补）

| 前端 metric | 中文名（前端展示） | DB 列（ad_snapshots / daily_stats） | 计算（读侧口径） | 依赖字段（分子/分母） | 写入链路覆盖 |
|---|---|---|---|---|---|
| `spend` | 花费 | ✅/✅ | 直接读（数值） | 无 | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `purchases` | 购买次数 | ✅/✅ | 直接读（数值） | actions → purchases（写侧提取） | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `link_clicks` | 链接点击 | ✅/✅（迁移 003） | 直接读（数值） | insights.inline_link_clicks / actions fallback | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `add_to_cart_count` | 加购次数 | ✅/✅（迁移 003） | 直接读（数值） | actions（add_to_cart） | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `initiate_checkout_count` | 结账次数 | ✅/✅（迁移 003） | 直接读（数值） | actions（initiate_checkout） | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `add_payment_info_count` | 添加支付信息次数 | ✅/✅（迁移 003） | 直接读（数值） | actions（add_payment_info） | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `cpc` | CPC（花费/链接点击） | ✅/（daily_stats 可有或可算） | \(cpc = spend / link\_clicks\)，`link_clicks=0 → null` | `spend`, `link_clicks` | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `cpa` | 单次购买花费（CPA） | ✅/✅ | \(cpa = spend / purchases\)，`purchases=0 → null` | `spend`, `purchases` | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `roas` | ROAS | ✅/✅（或 API 兜底） | \(roas = purchase\_value / spend\)，`spend=0 → null`（API roas 可作兜底） | `purchase_value`, `spend` | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `add_to_cart_cost` | 单次加购花费 | ✅/✅ | \(= spend / add\_to\_cart\_count\)，分母 0 → null | `spend`, `add_to_cart_count` | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `checkout_cost` | 单次结账花费 | ✅/✅ | \(= spend / initiate\_checkout\_count\)，分母 0 → null | `spend`, `initiate_checkout_count` | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |
| `payment_cost` | 单次添加支付信息花费 | ✅/✅ | \(= spend / add\_payment\_info\_count\)，分母 0 → null | `spend`, `add_payment_info_count` | Today 同步 ✅ / 归档 ✅ / 回补 ✅ |

**证据位置（代码）**：
- 写入 `ad_snapshots`：`server/services/ingestorService.js` 的 `INSERT INTO ad_snapshots (...)` 明确包含上述 cost/count 字段
- 读侧查询：`server/services/ruleDataService.js` 查询 `ad_snapshots`/`daily_stats` 时选取了 `add_to_cart_count/initiate_checkout_count/add_payment_info_count` 与 `add_to_cart_cost/checkout_cost/payment_cost`
- 读侧计算：`server/services/ruleDataService.js` 的 `calculateSingleDayMetrics` 与 `aggregateMultiDayMetrics` 对 cost/比值做分母检查并重算
- DB 迁移：`server/db/migrations/003_add_raw_count_fields_and_indexes.sql` 为 `ad_snapshots` 与 `daily_stats` 添加了三类事件计数与点击/转化金额字段

## 3. 规则配置建议（避免“永远不触发”）

比值/单次成本类指标在分母为 0 时会是 `null`，比较结果为 false（不会触发），因此建议：

- 用 `cpc` / `add_to_cart_cost` / `checkout_cost` / `payment_cost` / `cpa` / `roas` 时，**配一个分母保护条件**：
  - `link_clicks > 0`
  - `add_to_cart_count > 0`
  - `initiate_checkout_count > 0`
  - `add_payment_info_count > 0`
  - `purchases > 0`
  - `spend > 0`

前端已做提示：当你选择上述指标但缺少分母条件时，会在 When 区块顶部显示中文提示。

## 4. 自查 SQL（你要“铁证”时用）

### 4.1 检查列是否存在

```sql
DESCRIBE ad_snapshots;
DESCRIBE daily_stats;
```

### 4.2 抽样检查最近 1 天这些分母计数字段是否有写入

```sql
SELECT
  data_date,
  COUNT(*) cnt,
  SUM(link_clicks) sum_link_clicks,
  SUM(add_to_cart_count) sum_atc,
  SUM(initiate_checkout_count) sum_checkout,
  SUM(add_payment_info_count) sum_payment
FROM ad_snapshots
WHERE synced_at >= NOW() - INTERVAL 1 DAY
GROUP BY data_date
ORDER BY data_date DESC;
```

### 4.3 抽样检查 yesterday 的 daily_stats 是否已落盘（若未落盘则看真空期兜底）

```sql
SELECT
  date,
  COUNT(*) cnt,
  SUM(link_clicks) sum_link_clicks,
  SUM(add_to_cart_count) sum_atc,
  SUM(initiate_checkout_count) sum_checkout,
  SUM(add_payment_info_count) sum_payment
FROM daily_stats
WHERE date >= CURDATE() - INTERVAL 2 DAY
GROUP BY date
ORDER BY date DESC;
```

## 5. 新增指标的标准流程（以后按这个走）

1. **定义指标**：中文名、metric key（前端/后端一致）
2. **确认数据源**：FB Insights 哪个字段/哪个 actions key
3. **确认入库字段**：ad_snapshots / daily_stats 是否已有列（没有则迁移）
4. **写侧采集**：Today 同步 / 归档 / 回补写入该字段（至少写原始计数）
5. **读侧口径**：单日 + 多日聚合公式、分母为 0 返回 null
6. **前端上线**：下拉选项 + label + 分母提示
7. **验收**：SQL 抽样 + 单测/脚本验证
