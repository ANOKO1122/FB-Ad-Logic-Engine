# 阶段五与阶段六 — 知识点与教学说明

本文档按 `.cursorrules` 的「教学三部曲」补全阶段五（移除 Smart Mute 执行路径）和阶段六（移除 execute-all）的概念、实操与验证，方便你理解「是什么、为什么、怎么做、怎么验」。

---

# 阶段五：移除 Smart Mute 执行路径

## 第一步：拆解概念（Conceptual Breakdown）

### 1) 一句话比喻

**Smart Mute** 像给某条广告贴了一张「请勿打扰」便签，到期之前规则引擎不会对这条广告执行任何动作（暂停、改预算等）；到期或没有便签就照常执行。

### 2) 为什么要学 / 为什么要「移除执行路径」

- **产品决策**：本方案用「执行频率 + 执行时间段」来控频和控时，不再用「单广告挂起」来跳过执行；逻辑更统一，运维更简单。
- **工程意义**：学会「移除功能但保留数据」的常见做法——**执行路径**（代码逻辑）和**数据与脚本**（字段、清理脚本）可以分开处理，避免破坏历史数据或将来要用的能力。
- **面试可问**：「你们如何下线一个老功能？」→ 答：区分「行为下线」（不再根据 mute 跳过执行）和「数据保留」（表字段、管理脚本保留，用于排查或将来只读展示）。

### 3) 面试怎么问 + 如何答

| 问法 | 回答要点 |
|------|----------|
| Smart Mute 是什么？ | 按广告维度的「挂起」：`mute_until` 未过期则跳过执行，避免误动正在人工调优的广告。 |
| 为什么移除执行路径还保留字段？ | 历史数据可能已有 mute 信息；保留字段便于排查「当时为什么没动」；脚本可清 mute，便于运维；将来若只做「展示挂起状态」不必改表结构。 |
| 执行路径移除具体改了什么？ | 在 `actionExecutorService` 里去掉「若 mute_until 未过期则直接 return 并记 muted」的分支；单条规则路径不再因全 mute 设置 skipReason='muted'。 |

### 4) 本节学习目标（可检验）

学完阶段五你应能：

- 说清 Smart Mute 的语义和「移除执行路径」的含义。
- 在代码里指出：哪里曾经根据 `mute_until` 跳过执行，现在已删除。
- 说明为什么保留 `mute_until`/`mute_reason` 和 `clear-ad-mute.js`。
- 跑通 `server/tests/smartMute.test.js`，理解测试现在验证的是「历史逻辑 + 摘要格式兼容」。

### 5) 最小可行方案（MVP）

本次只做「从执行链路里拿掉 mute 判断」，不删表字段、不删脚本、不改写入 mute 的数据层；摘要层仍兼容 `skip_reason='muted'`（兼容历史或将来只读）。

## 第二步：安全实操（我们改了什么）

### 变更点概览

| 位置 | 改动类型 | 说明 |
|------|----------|------|
| `server/services/actionExecutorService.js` | 删除一段逻辑 | 删除「若 `mute_until` 未过期则 return 并 push actionType:'muted'」的整块代码。 |
| 同上 | 删除一段逻辑 | 删除「若单条规则下全部因 mute 跳过则设置 skipReason/skipDetails」的块（因执行层已不再返回 muted，该分支永不会触发）。 |
| `server/tests/smartMute.test.js` | 文档化 + 删用例 | 注明 Smart Mute 已从执行路径移除；删除「模拟 cron 里 mute 导致跳过」的集成风用例；保留 checkMuteStatus 纯函数与 skip_details 格式用例。 |

### 安全与规范对照

- **不删字段、不删脚本**：`ad_snapshots.mute_until` / `mute_reason`、`clear-ad-mute.js` 保留，符合「只改行为、保留数据与工具」。
- **无敏感信息**：未新增 Token/密码；日志仅保留原有级别。
- **可回滚**：若需恢复「按 mute 跳过」，只需把删除的 if 块重新加回执行入口。

### 代码在干什么（简要）

- **执行入口**：原先在真正执行动作前，若 `matchedAd.mute_until` 存在且大于当前时间，就 push 一个 `{ actionType: 'muted', ... }` 并 return，不调 API。删除后，不再做这一判断，直接进入 Dry Run / 真实执行分支。
- **单条规则汇总**：原先若所有广告都因 mute 被跳过，会设置 `stats.skipReason = 'muted'` 和 skipDetails；因执行层不再返回 muted，这段逻辑永远不成立，故删除以避免误导。

## 第三步：验证与排故（Test & Debug）

### 1) 如何验证跑通

- **单元测试**：在项目根目录执行  
  `npm test -- --run server/tests/smartMute.test.js`  
  应全部通过（checkMuteStatus 历史逻辑 + skip_details 格式）。
- **行为验证**：对一条带 `mute_until` 未来的广告跑单条规则执行或等 Cron；该广告不应再因 mute 被跳过，应按条件正常执行（或按其他逻辑跳过）。

### 2) 最小自动化测试

现有 `smartMute.test.js` 即最小测试集：纯函数 + 摘要格式，不依赖真实 DB/API。你可以在该文件里加一条注释用例，例如「执行路径不再检查 mute，此用例仅保留字段格式」。

### 3) 可观测性

- 执行路径中已无「因 mute 跳过」的日志；若需确认某次未执行是否曾与 mute 有关，可查 DB 中该广告的 `mute_until`/`mute_reason`（保留字段的目的之一）。

### 4) 最容易出的 3 个 Bug + 如何定位

| Bug | 可能原因 | 如何定位 |
|-----|----------|----------|
| 以为某广告「应该被 mute 跳过」却执行了 | 执行路径已移除 mute 判断，属预期行为 | 确认方案为「移除执行路径」；若需保留 mute 行为需恢复判断逻辑。 |
| 历史摘要里 skip_reason 曾为 muted，现在没有了 | 新执行不再产生 muted 跳过 | 正常；旧摘要仍可查；摘要层仍兼容写入 muted。 |
| 测试报错或断言过时 | 测试仍断言「mute 导致跳过」 | 检查 smartMute.test.js 是否已改为「历史逻辑 + 格式」而非「当前执行行为」。 |

# 阶段六：移除 execute-all 与「立即运行所有规则」

## 第一步：拆解概念（Conceptual Breakdown）

### 1) 一句话比喻

**「立即运行所有规则」** 像一键把当前账户下所有启用的规则立刻跑一遍；**移除** 后，规则只会在「每分钟 Cron」自动跑，或你在某一条规则上点「运行此规则」单独跑这一条。

### 2) 为什么要学 / 为什么要移除

- **产品与运维**：方案约定规则执行只由「每分钟 Cron」驱动，避免「心跳拉数 + 立刻跑全量规则」带来的重复执行和时序混乱；保留「单条执行」满足手动验证与调试。
- **工程意义**：学会「下线一个 API + 前端能力」的完整动作——路由、前端调用、废弃接口的提示文案要一起改，避免留死链接或误导用户。
- **面试可问**：「你们如何下线一个接口？」→ 答：移除或 410 路由、删前端调用、废弃接口返回明确迁移指引（例如指向单条执行接口）。

### 3) 面试怎么问 + 如何答

| 问法 | 回答要点 |
|------|----------|
| execute-all 是什么？ | 原「立即运行所有规则」接口：一次请求对某账户跑遍所有启用规则；从 DB 读数据不直接调 FB API，避免限流。 |
| 为什么移除？ | 规则执行统一由每分钟 Cron 驱动，不再提供「全量立即跑」入口，避免与 Cron 重复、时序难控。 |
| 单条执行保留了什么？ | 保留 `POST /api/rules/:id/execute`：只跑指定规则，忽略冷却，用于调试和即时验证。 |

### 4) 本节学习目标（可检验）

学完阶段六你应能：

- 说清「立即运行所有规则」与「单条规则执行」的区别，以及为何只保留后者。
- 在代码里指出：execute-all 路由已删、前端已无调用、废弃接口返回什么。
- 会用「运行此规则」或 Cron 验证规则执行，而不再依赖「运行所有规则」。

### 5) 最小可行方案（MVP）

本次只做：删 execute-all 路由、删前端 `executeAllRulesOffline`、旧接口返回「用单条执行」的提示；不改 Cron、不改单条执行接口。

---

## 第二步：安全实操（我们改了什么）

### 变更点概览

| 位置 | 改动类型 | 说明 |
|------|----------|------|
| `server/routes/rules.js` | 删除路由 | 删除 `POST /rules/execute-all` 整段；该路径不再存在，请求会 404。 |
| `server/index.js` | 改文案 | 废弃接口 `POST /api/execute-rules` 的 410 文案改为「规则由 Cron 执行，单条请用 POST /api/rules/:id/execute」。 |
| `src/services/facebookApi.js` | 删除方法 | 删除 `executeAllRulesOffline()`，避免调用已不存在的 execute-all。 |
| `server/routes/admin.js` | 无改动 | `POST /api/admin/cron/execute` 仍返回 410，文案已是「已移除立即运行所有规则」。 |
| `server/services/cronService.js` | 无改动 | `executeAllRules` 仍为只打日志的 stub；`runningCount` 仍供单条执行与 Cron 共用。 |
| 测试脚本 | 文案 | `test-e2e-verify-*.js` 中「立即运行所有规则」改为「运行此规则」或「等 Cron」。 |

### 安全与规范对照

- **无破坏性删库/删表**：仅删路由与前端方法，不动数据。
- **兼容旧客户端**：若仍有老端调用 execute-all，会得到 404；旧接口 `/api/execute-rules` 仍返回 410 并提示新用法。

### 代码在干什么（简要）

- **删路由**：Express 不再注册 `POST /rules/execute-all`，对应 handler 消失，请求落到 404。
- **删前端方法**：`facebookApi.js` 不再暴露 `executeAllRulesOffline()`，UI 上若有残留按钮会因无方法而报错或需一并移除（当前仓库规则页只有「运行此规则」）。
- **改废弃提示**：调用 `/api/execute-rules` 的用户会看到「用单条执行」的明确指引，而不是「请用 execute-all」。

## 第三步：验证与排故（Test & Debug）

### 1) 如何验证跑通

- **接口**：  
  - `POST /api/rules/execute-all`（若仍能拼出路径）应 404。  
  - `POST /api/execute-rules` 应 410，body 中含 `new_endpoint: 'POST /api/rules/:id/execute'` 或等价提示。  
  - `POST /api/rules/:id/execute` 仍 200（或按业务返回），单条执行正常。
- **前端**：规则管理页只有「运行此规则」，没有「立即运行所有规则」；点击「运行此规则」能触发单条执行并看到结果。

### 2) 最小自动化测试

可手测：用 curl/Postman 调 `POST /api/rules/execute-all` 看 404；调 `POST /api/execute-rules` 看 410 与文案。若有 E2E，可把「立即运行所有规则」的步骤改为「运行此规则」或「等待 Cron」。

### 3) 可观测性

- 若有人误调 execute-all：服务端 404 无 body 或由全局 404 处理返回；查访问日志可见请求路径。
- 废弃接口被调：现有 `logger.warn` 会打「废弃接口被调用」，便于统计迁移进度。

### 4) 最容易出的 3 个 Bug + 如何定位

| Bug | 可能原因 | 如何定位 |
|-----|----------|----------|
| 前端报错「executeAllRulesOffline is not a function」 | 某处仍引用已删方法 | 全局搜 `executeAllRulesOffline`，删除调用或改为「运行此规则」流程。 |
| 期望「一键跑全量」却找不到入口 | 功能已下线 | 确认需求：若确需全量，只能通过 Cron 或自行用脚本轮询单条执行。 |
| 旧书签/脚本调 execute-all 得到 404 | 路由已删 | 正常；将调用方改为单条执行或依赖 Cron。 |

# 小结：两阶段在「做项目」上的共通点

1. **先概念再动手**：先搞清楚「Smart Mute / execute-all 是什么、为什么要改」，再改代码，避免改错范围。
2. **区分「行为」与「数据」**：阶段五改的是「是否根据 mute 跳过」，没动表结构；阶段六改的是「是否提供全量执行入口」，没动 Cron 和单条执行。
3. **每阶段都要可验证**：阶段五用单元测试 + 执行行为观察；阶段六用接口状态 + 前端操作，确认「该没有的没有了、该保留的还能用」。
4. **文档与注释**：在代码里注明「已移除」「保留用于…」，方便以后自己和同事排查、面试时也能说清设计理由。

若你愿意，下次新阶段我们可以严格按「先讲概念 → 再列变更与安全清单 → 你确认后再改 → 改完一起列验证步骤」来做，这样更符合 .cursorrules 的导师型要求。
