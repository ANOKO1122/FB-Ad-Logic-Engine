# 2.3.1 阶段 3：保存与回填 v2 验收步骤（手把手）

> 保存用 `linesToV2Groups` 输出 v2；编辑回填用 `v2ToLines` / `v1ToLines`。验收：保存后每条 condition 同 time_window；回填 OR 不丢失。

---

## 一、验收 1：保存输出 v2，且每条 condition 同 time_window

### 1.1 在页面上操作

1. 打开前端：**规则管理**。
2. 点击 **「新建规则」**。
3. 填写：
   - **规则名称**：随便填，例如 `阶段3验收-AND`。
   - **选择广告账户**：选一个你有权限的账户。
   - **触发条件**：
     - 时间窗口选 **「今天」**。
     - 第一行：指标选 **花费**，运算符 **大于(>)**，数值 **10**。
     - 点击 **「+ 添加条件」**，第二行左侧选 **AND**，指标 **ROAS**，运算符 **小于(<)**，数值 **0.5**。
   - **执行动作**：保留默认或选「暂停广告」。
   - **作用对象**：可先不选或选一个测试广告（否则可能无法保存，视你项目校验而定）。
4. 点击 **「保存规则」**，确认保存成功（弹窗关闭、列表里出现新规则）。

### 1.2 用 SQL 查看刚保存的 conditions

在 MySQL 里执行（把规则名换成你写的，或直接用最后一条）：

```sql
-- 按规则名查（把 '阶段3验收-AND' 换成你填的规则名）
SELECT id, rule_name, conditions
FROM rules
WHERE rule_name = '阶段3验收-AND'
LIMIT 1;

-- 或者直接看最新一条
SELECT id, rule_name, conditions
FROM rules
ORDER BY id DESC
LIMIT 1;
```

### 1.3 预期结果（验收 1 通过标准）

- `conditions` 是一个 JSON，形如：
  - 顶层有 `"version": 2` 和 `"groups"`。
  - `groups` 里至少有一个对象，里面有 `"operator": "AND"` 和 `"conditions": [ ... ]`。
  - 每个 condition 里都有 **相同的** `"time_window": "today"`（不能有的有、有的没有）。
- 示例（仅作参考，不必完全一致）：

```json
{
  "version": 2,
  "groups": [
    {
      "operator": "AND",
      "conditions": [
        { "metric": "spend", "operator": "gt", "value": 10, "time_window": "today" },
        { "metric": "roas", "operator": "lt", "value": 0.5, "time_window": "today" }
      ]
    }
  ]
}
```

若你用的是「自定义范围」并选了起止日期，则每条 condition 里还应有相同的 `"custom_range": { "since": "...", "until": "..." }`。

## 二、验收 2：回填 OR 不丢失

### 2.1 新建一条带 OR 的规则

1. **规则管理** → **新建规则**。
2. 规则名称例如：`阶段3验收-OR`。
3. 选好账户后，**触发条件**：
   - 时间窗口：**今天**。
   - 第一行：**花费** **大于** **10**（左侧没有 AND/OR 下拉是正常的）。
   - 点 **「+ 添加条件」**，在**第二行左侧下拉选「OR」**（不要选 AND），然后指标 **ROAS**，**小于** **0.5**。
4. 执行动作、作用对象按需填，点 **保存规则**。

### 2.2 用 SQL 确认保存成两个 group（OR 对应新组）

```sql
SELECT id, rule_name, conditions
FROM rules
WHERE rule_name = '阶段3验收-OR'
LIMIT 1;
```

**预期**：`conditions.groups` 有 **2 个**元素：第一个 group 里是「花费>10」，第二个 group 里是「ROAS<0.5」。例如：

```json
{
  "version": 2,
  "groups": [
    { "operator": "AND", "conditions": [ { "metric": "spend", "operator": "gt", "value": 10, "time_window": "today" } ] },
    { "operator": "AND", "conditions": [ { "metric": "roas", "operator": "lt", "value": 0.5, "time_window": "today" } ] }
  ]
}
```

### 2.3 回填验收：编辑这条规则，看 OR 是否还在

1. 在规则列表找到 **「阶段3验收-OR」**，点击 **「编辑」**。
2. 看弹窗里的**触发条件**：
   - **第一行**：花费 > 10，左侧**没有** AND/OR 下拉（或显示为“首行”）。
   - **第二行**：左侧下拉必须是 **OR**（不是 AND），且条件为 ROAS < 0.5。
3. 若第二行显示的是 **OR** 且条件一致，则 **验收 2 通过**。

## 三、验收 3：v1 旧规则回填（可选）

仅当你库里还有**旧版 v1 规则**（conditions 是数组、没有 `version: 2`）时才需要验。

### 3.1 查是否存在 v1 规则

```sql
-- v1 的 conditions 是数组，v2 是对象；JSON_TYPE 可区分
SELECT id, rule_name, JSON_TYPE(conditions) AS cond_type, conditions
FROM rules
ORDER BY id DESC
LIMIT 5;
```

若某条 `cond_type` 为 **ARRAY**，或你看到 `conditions` 是 `[{ "metric": ... }, ...]` 这种数组，就是 v1。

### 3.2 操作

1. 规则列表里点该 v1 规则的 **「编辑」**。
2. **预期**：弹窗正常打开，触发条件以线性列表展示，每条一行，且时间窗口、指标/运算符/数值正确，没有报错或空白。  
若满足，则 **验收 3 通过**。

---

## 四、用接口验证（不用 SQL 时）

若你更习惯用接口，可以用列表接口拿到规则后再看 `conditions`：

```bash
# 需已登录或带 cookie / token
curl -s -b "你的登录 cookie" "http://localhost:5173/api/rules" | jq '.rules[] | {id, rule_name: .ruleName, conditions}'
```

或只取最新一条：

```bash
curl -s -b "你的登录 cookie" "http://localhost:5173/api/rules" | jq '.rules[0] | {id, rule_name: .ruleName, conditions}'
```

根据返回的 `conditions` 按上面「预期结果」判断即可。

## 五、通过标准汇总

| 验收项 | 你要做的 | 通过标准 |
|--------|----------|----------|
| **验收 1** | 新建 AND 规则 → 保存 → 用 SQL 或接口查该条 `conditions` | 是 v2（有 version:2 和 groups），且每条 condition 都有相同 time_window |
| **验收 2** | 新建 OR 规则 → 保存 → 再点编辑 | 编辑弹窗里第二行左侧显示 **OR**，且条件正确 |
| **验收 3** | 若有 v1 规则，点编辑 | 能正常打开并回填，条件与时间窗口正确 |

三项都满足后，在 **TASKS.md** 的 2.3.1 阶段 3 打勾即可。
