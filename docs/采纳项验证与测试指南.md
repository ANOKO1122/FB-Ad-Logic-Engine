# 采纳项验证与测试指南

> 本文档教你怎么验证「状态不等于、账户数上限、p-limit 并发、多账户加载更多、前端状态行」等采纳项是否成功。  
> 每步会说明：**在测什么**、**怎么做**、**成功长什么样**、**失败时怎么排查**。

---

## 零、验证前准备（必做）

### 0.1 后端要跑起来

- 在项目根目录执行：`npm run dev:server`（或你平时启动后端的命令）。
- 看到类似 “Server listening on port 3001” 即表示后端已启动；端口以你项目配置为准（如 3000）。

### 0.2 拿到登录 Token（用于带认证的接口）

- **方式 A**：用前端登录  
  - 打开前端页面（如 `http://localhost:5173`），登录后打开浏览器开发者工具 → **Application**（或 **存储**）→ **Local Storage**，找到存 token 的 key（如 `token` 或 `auth_token`），复制 value（一长串 JWT）。  
- **方式 B**：用登录接口  
  - 若你有登录接口（如 `POST /api/auth/login`），用 Postman/Thunder Client 发请求，响应里会带 token，复制下来。  
- 后面所有「需要认证」的请求，在 **Header** 里加：  
  `Authorization: Bearer <你复制的 token>`  
  （注意：`Bearer` 后面有一个空格，再跟 token。）

### 0.3 准备两个「你有权限」的广告账户 ID

- 从你前端的账户列表里抄两个 `act_xxxxx`，例如：`act_1905531400268801` 和 `act_1501999950830277`。  
- 单账户验证时用一个即可；多账户验证时用两个（或你实际有的多个）。

## 一、验证 scope_status_exclude（状态不等于）— 后端 API

**在测什么**：后端是否根据 `scope_status_exclude` 参数排除指定状态，返回的列表里是否没有这些状态。

### 1.1 单账户：排除 PAUSED（只应看到 ACTIVE 等）

- **请求**：GET，URL 形如（把 `act_xxx` 换成你有权限的账户 ID）：  
  `http://localhost:3001/api/structure/ads?account_id=act_xxx&scope_status_exclude=PAUSED&q=&limit=50`  
- **Header**：`Authorization: Bearer <你的 token>`

**成功标准**：

- **Status**：`200`  
- **Response** 里 `items` 数组中，**任意一条的 `effective_status` 都不能是 `PAUSED`**（可以有 ACTIVE、PENDING_REVIEW 等）。  
- 若你账户下本来就有 PAUSED 的广告，不传 `scope_status_exclude` 时能查到，传了 `scope_status_exclude=PAUSED` 后列表中不应再出现 PAUSED。

**失败时**：若 items 里还有 `effective_status: 'PAUSED'`，说明后端没有正确应用 scope_status_exclude，需检查 `structureSyncService.js` 里 list 方法是否做了「排除」逻辑，以及路由是否把 `scope_status_exclude` 传给了 list。

### 1.2 多账户：排除 ACTIVE 和 PAUSED

- **请求**：GET，URL 形如（把两个 act 换成你有权限的）：  
  `http://localhost:3001/api/structure/objects/multi?account_ids=act_a,act_b&type=ad&scope_status_exclude=ACTIVE,PAUSED&limit=50`  
- **Header**：同上。

**成功标准**：

- **Status**：`200`  
- **Response** 里 `items` 中，**没有一条的 `effective_status` 是 ACTIVE 或 PAUSED**（只会是 PENDING_REVIEW、DISAPPROVED 等其它状态）。  
- 若两个账户下全是 ACTIVE/PAUSED，则 items 可能为空，这也是「排除生效」的表现。

## 二、验证账户数上限（20/50）与 TOO_MANY_ACCOUNTS

**在测什么**：非管理员一次传超过 20 个账户是否被拒绝并返回 400 + code；管理员传 25 个是否允许。

### 2.1 非管理员：传 25 个 account_ids 应 400

- 用**非 admin 账号**登录，拿到该账号的 token。  
- **请求**：GET  
  `http://localhost:3001/api/structure/objects/multi?account_ids=act_1,act_2,act_3,...,act_25&type=ad&limit=50`  
  （这里要凑满 25 个不同的 act_xxx；若你账户不够，可重复写同一个 ID 多份，因为后端会去重后数 allowedIds 的数量——但更准确的是用 25 个不同账户 ID。）

**成功标准**：

- **Status**：`400`  
- **Response** 里有 `"code": "TOO_MANY_ACCOUNTS"`，且 `error` 文案里提到「最多选择 20 个广告账户」（或类似）。

**失败时**：若返回 200，说明上限校验没生效，检查 `server/index.js` 里 multi 路由在得到 `allowedIds` 后是否有 `allowedIds.length > maxAccounts` 的判断并返回 400。

### 2.2 管理员：传 25 个 account_ids 应 200

- 用 **admin 账号**登录，拿 token。  
- 同上，传 25 个 account_ids（同上，尽量 25 个不同账户）。

**成功标准**：

- **Status**：`200`  
- 有 `items` 和 `meta`，说明管理员放宽到 50 生效（25 < 50）。

## 三、验证 p-limit(4) 首轮并发（看日志）

**在测什么**：多账户首轮是否用并发（约 4 个一批）而不是纯串行，总耗时是否比「串行」短。

### 3.1 怎么测

- 选 5～10 个有权限的 account_ids（不要太多，避免请求过慢）。  
- 发一条 **首轮** multi 请求（**不带** `after` 参数）：  
  `GET http://localhost:3001/api/structure/objects/multi?account_ids=act_1,act_2,...,act_8&type=ad&limit=50`  
- 同时看**后端终端/日志**：  
  - 若打了多条类似 `[2.2] GET /api/structure/objects/multi` 的日志，且时间戳很接近（几乎同时），说明是多请求并发；  
  - 总耗时应明显小于「8 次串行查库」的时间（具体取决于 DB 和网络，这里主要看「有没有并发」）。

**成功标准**：  
- 首轮一次请求返回 200，且日志里能看到多账户在短时间内被处理（不是严格的「一条一条间隔几秒」）。  
- 若你愿意，可以临时在代码里打日志：每个账户 list 开始/结束各打一条，看是否「最多 4 个同时在跑」。

## 四、验证多账户「加载更多」与 after 游标

**在测什么**：首轮返回的 `paging.after` 是否有值；用这个 after 再请求是否拿到下一页；直到没有更多页时 after 为空。

### 4.1 首轮拿到 after

- 发：  
  `GET http://localhost:3001/api/structure/objects/multi?account_ids=act_a,act_b&type=ad&limit=10`  
  （limit 小一点，方便有多页。）

**成功标准**：  
- `paging.after` 存在且为一串 base64 字符串（若两个账户下广告不少，一般会有下一页）。  
- `meta.per_account_after` 里能看到每个账户的 after 状态。

### 4.2 带 after 请求第二页

- 把上一步响应里的 `paging.after` 复制出来，作为 query 参数再发一条：  
  `GET http://localhost:3001/api/structure/objects/multi?account_ids=act_a,act_b&type=ad&limit=10&after=<粘贴 after 值>`  

**成功标准**：  
- Status 200，返回的 `items` 是**新的一页**（和第一页不重复或仅有少量重叠属正常，因为按账户+游标合并）；  
- 若还有更多页，新的 `paging.after` 仍有值；若没有，`paging.after` 为 null 或不存在。

## 五、前端验证：状态「不等于」+ 应用监控范围

**在测什么**：前端选择「状态 不等于 已暂停」并应用监控范围后，列表是否只显示非 PAUSED 的广告，且请求里带了 `scope_status_exclude`。

### 5.1 操作步骤

1. 打开规则管理页面，进入「新建规则」或「编辑规则」。  
2. 在「监控范围」区域：  
   - 添加一行条件：**字段**选「状态」，**运算符**选「**不等于**」，**值**选「已暂停」（或「所有投放中」等，看你们下拉项）。  
3. 选择**一个**你有权限的广告账户，点击「加载列表」或等列表加载出来。  
4. 点击「**应用监控范围**」按钮。

### 5.2 成功标准

- 列表**只显示**「状态 不等于 已暂停」的广告（即没有 effective_status 为 PAUSED 的项）；若选的是「不等于 所有投放中」，则列表里不应出现 ACTIVE。  
- 打开浏览器开发者工具 → **Network**，找到请求 `/api/structure/ads` 或 `/api/structure/objects` 或 `/api/structure/objects/multi`，点开看 **Query String** 或 **Payload**：  
  - 应能看到 `scope_status_exclude=PAUSED`（或 ACTIVE，取决于你选的值）。  
- 再点「加载更多」：请求里同样应带 `scope_status_exclude=PAUSED`，且新加载的列表仍然没有已暂停的广告。

**失败时**：  
- 若列表里还有已暂停：先确认后端 1.1/1.2 已通过，再查前端是否把「不等于」映射成了 `scopeStatusExcludeFilter` 并在 refresh/loadMore 里带上。  
- 若请求里没有 scope_status_exclude：查 applyScopeConditions 里 statusRow.operator === 'not_equals' 时是否写了 scopeStatusExcludeFilter，以及 getStructureObjects/getStructureObjectsMulti 的 params 里是否包含 scope_status_exclude。

## 六、验收结果怎么算「通过」

建议你按下面打勾，全部通过即本次采纳项验证成功：

| 序号 | 项 | 通过标准 |
|------|----|----------|
| 1 | scope_status_exclude 单账户 | 1.1 请求返回 200，items 中无 PAUSED |
| 2 | scope_status_exclude 多账户 | 1.2 请求返回 200，items 中无 ACTIVE/PAUSED |
| 3 | 账户数上限（非 admin） | 2.1 传 25 个 account_ids → 400，code: TOO_MANY_ACCOUNTS |
| 4 | 账户数上限（admin） | 2.2 传 25 个 account_ids → 200 |
| 5 | p-limit 首轮 | 3.1 多账户首轮日志/耗时体现并发 |
| 6 | 多账户加载更多 | 4.1 首轮有 after；4.2 带 after 能拿到下一页 |
| 7 | 前端状态不等于 | 5.1–5.2 应用后列表正确，且请求带 scope_status_exclude |

## 七、常用工具与快捷方式

- **Thunder Client（VS Code 插件）**：在 VS Code 里新建 Request，选 GET，URL 填上面的地址，Headers 里加 `Authorization: Bearer <token>`，点 Send。可把「采纳项验证」建一个集合，每条请求保存好，方便以后回归。  
- **浏览器 Network**：前端验证时一定要看请求的 Query 里是否带了 `scope_status_exclude`，避免只凭「列表看起来对」就放过（有可能是缓存或单账户默认行为）。  
- **后端日志**：验证 p-limit 和 multi 时，边发请求边看终端输出，能更快确认行为是否符合预期。
