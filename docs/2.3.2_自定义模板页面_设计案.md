# 2.3.2 自定义模板页面 — 设计案（定稿）

> 目标：管理员可在模板管理页面增删改模板，普通用户在规则创建时只读使用。

---

## 一、目标与权限

| 角色 | 模板管理页面 | 创建/编辑/删除模板 | 规则管理中使用模板 |
|------|--------------|--------------------|--------------------|
| 管理员 | ✅ 可见 | ✅ 允许 | ✅ 允许 |
| 普通用户 | ❌ 不可见 | ❌ 无 | ✅ 只读使用 |

- **前端**：`meta.requiresAdmin` 控制导航与页面可见性
- **后端**：所有 `/api/admin/templates` 的 CRUD 必须强制校验 `role === 'admin'`，防止越权（403）

---

## 二、数据模型

### 2.1 表：`rule_templates`

迁移脚本：`server/db/migrations/018_create_rule_templates.sql`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT AUTO_INCREMENT PK | 主键 |
| name | VARCHAR(100) NOT NULL | 模板展示名 |
| slug | VARCHAR(50) NOT NULL UNIQUE | 唯一标识，新建可填、更新不可改 |
| description | VARCHAR(500) | 简要说明 |
| when_lines | JSON NOT NULL | 条件行 `[{join,metric,operator,value},...]` |
| when_time_window | VARCHAR(50) NOT NULL | `today\|yesterday\|last_3_days\|lifetime\|custom_range` |
| when_custom_range | JSON | `custom_range` 时的 `{since,until}` |
| actions | JSON NOT NULL | 动作 `[{type,value,max_daily_budget},...]` |
| sort_order | INT DEFAULT 0 | 排序，小的在前 |
| is_active | TINYINT(1) DEFAULT 1 | 软删除/禁用（0=已删除或禁用） |
| created_by | INT | 创建人 user_id |
| updated_by | INT | 最后更新人 user_id |
| created_at | TIMESTAMP | |
| updated_at | TIMESTAMP | |

**索引**：`uk_slug`、`idx_sort_order`、`idx_is_active`

**字符集**：utf8mb4 / utf8mb4_0900_ai_ci（与仓库一致）

### 2.2 与 RuleManager 结构对齐

- `when_lines` ↔ `src/utils/conditionsTransform.js` 的 `linesToV2Groups` / `v2ToLines`
- `actions`：`pause_ad | activate_ad | increase_budget | decrease_budget`，含 `value`、`max_daily_budget`（分）

### 2.3 规则溯源（可选扩展）

- 若需溯源，可在 `rules` 表增加 `source_template_slug` 或 `source_template_id`，保存规则时写入
- 当前设计：规则保存的是快照，删除模板不影响已有规则

## 三、服务端校验

### 3.1 when_time_window

- 值域：`today | yesterday | last_3_days | lifetime | custom_range`
- 当为 `custom_range` 时：`when_custom_range` 必须包含 `since`、`until`，且 `since <= until`

### 3.2 actions

- `type`：仅允许 `pause_ad | activate_ad | increase_budget | decrease_budget`
- `value`：非负；预算类动作时必填且 1～100（百分比）
- `max_daily_budget`：仅增/减预算时有效，单位「分」，非负；可做平台上限二次校验

### 3.3 when_lines

- 结构：`[{ join, metric, operator, value }]`，首行 `join` 为 null，后续为 `AND` 或 `OR`
- metric：与规则引擎支持的指标一致（spend, roas, cpa, cpc, purchases, link_clicks 等）

### 3.4 slug

- 新建：必填，唯一约束
- 更新：**不可改**，保持引用稳定
- **软删除与 slug**：slug 全局唯一，软删除后仍占用。如需复用 slug，应使用「恢复」接口而非新建同 slug 的模板。

## 四、接口设计

### 4.1 管理员接口（需 admin 鉴权）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/admin/templates | 列表；`include_inactive`、`query`、`limit`(1-100 默认 20)、`offset`(0-100000) |
| POST | /api/admin/templates | 新建；校验 slug 唯一、actions/when_lines 合法性 |
| PUT | /api/admin/templates/:id | 更新；slug 不可改；**乐观并发**：body.updated_at 不匹配返回 409 |
| DELETE | /api/admin/templates/:id | **软删除**：`is_active=0` |
| POST | /api/admin/templates/:id/restore | **恢复**：`is_active=1`，用于复用已删模板的 slug |

### 4.2 普通用户接口（只读）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/templates | 仅返回 `is_active=1`，字段：id、name、slug、description、when_lines、when_time_window、when_custom_range、actions、sort_order |

### 4.3 错误码

| 场景 | 状态码 | 说明 |
|------|--------|------|
| slug 重复 | 409 | DUPLICATE_SLUG |
| 乐观并发冲突（updated_at 不匹配） | 409 | CONFLICT |
| 缺少 updated_at | 400 | MISSING_VERSION |
| 无效参数 | 400 | INVALID_PARAMS |
| 越权（非 admin 访问 admin 接口） | 403 | Forbidden |
| 资源不存在 | 404 | Not Found |

## 五、前端设计

### 5.1 模板管理页 `/admin/templates`

- **路由**：`/admin/templates`，`meta: { requiresAdmin: true }`
- **侧边栏**：在管理区增加「模板管理」（仅 admin 可见）
- **布局**：
  - 标题 + 「新建模板」
  - 卡片列表：名称、slug、描述、条件摘要、动作摘要、排序、启用状态
  - 操作：编辑、删除（软删除）、恢复（仅已删除卡片）
  - 可选：拖拽排序（批量更新 sort_order）

### 5.2 模板编辑弹窗

- **新建**：name、slug（必填）、description、sort_order、条件（线性编辑）、动作
- **编辑**：slug 只读；其余可改；保存时需传 `updated_at` 以支持乐观并发
- **条件编辑**：复用 RuleManager 线性列表 UI + `conditionsTransform.js`（linesToV2Groups / v2ToLines）
- **动作编辑**：与 RuleManager 区块 4 一致

- **max_daily_budget**：输入单位为「美元」，保存时转为「分」存储；不填表示不限制

### 5.3 规则管理页改造

- 进入新建/编辑规则时：`GET /api/templates` 拉取模板列表
- 动态渲染模板按钮（或下拉/卡片），点击后填充 whenLines、whenTimeWindow、whenCustomRange、actions
- 无模板时：不显示模板区，或保留 3 个内置默认模板兜底（可选）

## 六、落地步骤

1. **DB**：执行 `018_create_rule_templates.sql`
2. **后端**：
   - 新增 `server/routes/admin.js` 下的 `/api/admin/templates` CRUD
   - 新增 `GET /api/templates`（或挂到 rules 路由）
   - 校验 when_lines、actions、when_time_window；slug 冲突 409
   - 鉴权：admin 路由强制 role=admin
3. **前端**：
   - 新增 `src/views/AdminTemplates.vue`（`/admin/templates`）
   - 编辑弹窗复用 RuleManager 的线性条件 + 动作编辑逻辑
   - 路由 + 侧栏（仅 admin）
   - RuleManager：从 `/api/templates` 加载模板并应用
4. **数据迁移**：执行 `019_seed_rule_templates.sql` 插入 3 个预设模板（止损、扩量、空耗）
5. **验收**：按「八、验收」执行

## 七、边界与风险

| 点 | 处理 |
|----|------|
| slug 冲突 | 唯一约束 + 409；软删除后 slug 仍占用，复用需「恢复」 |
| 删除模板 | 软删除，已有规则不受影响 |
| 并发编辑 | PUT 乐观并发（updated_at），冲突 409 |
| limit/offset | parseInt + clamp（limit 1-100 默认 20，offset 0-100000），防注入与大页 |
| 空模板列表 | 不显示模板区或内置兜底 |
| 结构变更 | JSON 存 when_lines/actions，前后端 schema 校验兼容 |

---

## 八、验收

1. **管理员**：进入模板管理，增删改模板；规则管理中可选到最新模板
2. **普通用户**：看不到模板管理；规则管理仍可使用模板
3. **数据一致性**：应用模板后保存规则，conditions/actions 与模板一致
4. **越权**：普通用户调用 `/api/admin/templates` POST 返回 403
5. **软删除**：删除后 `/api/templates` 不返回，已有规则不受影响

---

## 九、测试建议

- **单测**：slug 唯一、actions/when_lines 合法性、403 鉴权
- **集成**：RuleManager 应用模板 → 保存规则 → 校验 conditions/actions
- **回归**：模板更新不影响已存在规则（快照）
