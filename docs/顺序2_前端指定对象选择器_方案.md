# 顺序 2：前端指定对象选择器 — 方案（目标 → 步骤 → 风险 → 验证）

> 边界：只做「默认只拉/只展示 ACTIVE/PAUSED，可切 ALL」；不做任何列表字段增强（保持 ad_id + ad_name）。  
> 你先确认本方案后再动手；采用 B 方式：直接给变更步骤与代码片段。

---

## 一、目标

- **默认**：创建/编辑规则时，选择指定对象（广告/广告组/广告系列）的列表**只拉取、只展示** `effective_status` 为 **ACTIVE** 与 **PAUSED** 的对象。
- **可切换**：用户可切换为 **ALL**，此时拉取并展示所有状态（含 ARCHIVED、DELETED 等）。
- **列表展示**：保持现状，仅 **ad_id + ad_name**（或对应层级的 id + name），不做「身份证信息」等字段增强。

---

## 二、现状与接口

| 项目 | 现状 |
|------|------|
| 前端「作用对象」列表 | `RuleManager.vue` 通过 `facebookApi.getStructureObjects(levelKey, accountId, { q, limit, after })` 拉取；无 status 相关参数。 |
| 后端接口 | `GET /api/structure/:level`（`server/index.js` 约 L1931）；已有 **filtering**：当前写死为 `effective_status IN ['ACTIVE','PAUSED','PENDING_REVIEW','DISAPPROVED','IN_PROCESS','WITH_ISSUES']`（即排除已删除/归档，但包含审核中等）。 |
| 后端能力 | `FacebookMarketingAPI.getStructurePage` 已支持传入 `filtering`；路由侧可根据查询参数决定 filtering 内容。 |

结论：**后端已有 filtering 能力，只需增加一个查询参数（如 `status_filter`）控制「默认 ACTIVE+PAUSED」还是「ALL」；前端增加一个状态切换控件，并把该参数传给接口即可。无需新增接口，只需扩参 + 前端 UI。**

---

## 三、方案步骤与改动点

### 3.1 后端：增加 `status_filter` 参数（仅改一处）

**文件**：`server/index.js`  
**位置**：`GET /api/structure/:level` 路由内，约 L1952–1971（构建 `filtering` 与调用 `getStructurePage` 的那段）。

**逻辑**：

- 从 `req.query` 读取 `status_filter`（可选）：`active_paused` | `all`。
- **默认**：未传或为 `active_paused` 时，`effective_status` 只保留 **ACTIVE、PAUSED**（与 TASKS 2.3 一致）。
- **ALL**：`status_filter=all` 时，**不**向 `filtering` 添加 `effective_status` 条件（仅保留已有的 `name` 关键词过滤，若有 `q`）。

**代码片段（替换现有 filtering 构建）**：

```javascript
// 从查询参数读取状态筛选：active_paused（默认）| all
const statusFilter = String(req.query.status_filter || 'active_paused').toLowerCase()
const filtering = []

if (statusFilter !== 'all') {
  // 默认：只展示 ACTIVE、PAUSED（与 TASKS 2.3 一致）
  filtering.push({
    field: 'effective_status',
    operator: 'IN',
    value: ['ACTIVE', 'PAUSED']
  })
}

const keyword = String(q || '').trim()
if (keyword) {
  filtering.push({ field: 'name', operator: 'CONTAIN', value: keyword })
}

const page = await api.getStructurePage(String(account_id), edge, {
  fields: fieldsByEdge[edge],
  limit,
  after: after ? String(after) : null,
  filtering
})
```

**说明**：原逻辑是 6 种状态；这里改为默认仅 2 种（ACTIVE、PAUSED）；`all` 时不加 effective_status，由 FB 返回全部状态。

### 3.2 前端 API：传 `status_filter`

**文件**：`src/services/facebookApi.js`  
**方法**：`getStructureObjects(level, accountId, { q, limit, after } = {})`

- 增加可选参数 **`statusFilter`**（建议默认 `'active_paused'`）。
- 请求时增加查询参数 **`status_filter`**，值与 `statusFilter` 一致。

**代码片段**：

```javascript
async getStructureObjects(level, accountId, { q = '', limit = 50, after = null, statusFilter = 'active_paused' } = {}) {
  const resp = await apiClient.get(`/structure/${level}`, {
    params: {
      account_id: accountId,
      q: q || undefined,
      limit,
      after: after || undefined,
      status_filter: statusFilter || undefined  // active_paused | all
    }
  })
  return resp.data || { items: [], paging: null }
}
```

### 3.3 前端 RuleManager：状态筛选控件 + 传参

**文件**：`src/views/RuleManager.vue`

**改动点 1：状态筛选 UI（在「选择目标对象」区块内）**

- 在「选择目标对象（可多选）」上方或搜索框旁，增加**状态筛选**：单选框或下拉，选项为「仅 ACTIVE/PAUSED（默认）」和「全部 (ALL)」。
- 绑定一个 ref，例如 `scopeStatusFilter`，取值 `'active_paused'` | `'all'`。

**建议位置**：在「目标层级」和「选择目标对象（可多选）」之间，或紧贴搜索框同一行右侧。

**模板片段示例**（插入到「选择目标对象（可多选）」的 label 下方、搜索框上方或同排）：

```html
<div class="form-group">
  <label>状态筛选</label>
  <div class="radio-group">
    <label class="radio-label">
      <input type="radio" v-model="scopeStatusFilter" value="active_paused">
      仅 ACTIVE/PAUSED（默认）
    </label>
    <label class="radio-label">
      <input type="radio" v-model="scopeStatusFilter" value="all">
      全部 (ALL)
    </label>
  </div>
</div>
```

**改动点 2：ref 与默认值**

- 在 setup 里增加：`const scopeStatusFilter = ref('active_paused')`。
- 切换为「全部」时如需自动刷新列表，可对 `scopeStatusFilter` 做 `watch`，在变化时调用 `refreshScopeItems()`。

**改动点 3：调用接口时传入 status_filter**

- **refreshScopeItems**：调用 `facebookApi.getStructureObjects(..., { q, limit, after, statusFilter: scopeStatusFilter.value })`。
- **loadMoreScopeItems**：同样传入 `statusFilter: scopeStatusFilter.value`，保证「加载更多」与当前筛选一致。

**refreshScopeItems 修改片段**（仅展示关键调用）：

```javascript
const results = await Promise.all([
  facebookApi.getStructureObjects(levelKey, selectedAccountId.value, {
    q: keyword,
    limit: 50,
    after: null,
    statusFilter: scopeStatusFilter.value   // 新增
  })
])
```

**loadMoreScopeItems 修改片段**：

```javascript
const resp = await facebookApi.getStructureObjects(levelKey, selectedAccountId.value, {
  q: String(scopeSearch.value || '').trim(),
  limit: 50,
  after: scopePagingAfter.value,
  statusFilter: scopeStatusFilter.value    // 新增
})
```

**可选**：`watch(scopeStatusFilter, () => refreshScopeItems())`，这样切换「全部」时自动刷新列表，无需用户再点「刷新列表」。

## 四、涉及文件与接口汇总

| 类型 | 文件 | 改动摘要 |
|------|------|----------|
| 后端 | `server/index.js` | GET `/api/structure/:level` 增加 `status_filter` 查询参数；默认 `active_paused` 时 filtering 仅 ACTIVE+PAUSED，`all` 时不加 effective_status。 |
| 前端 API | `src/services/facebookApi.js` | `getStructureObjects` 增加参数 `statusFilter`，请求时带 `status_filter`。 |
| 前端页面 | `src/views/RuleManager.vue` | 增加状态筛选控件（ACTIVE/PAUSED vs ALL）、`scopeStatusFilter` ref、在 `refreshScopeItems` 与 `loadMoreScopeItems` 中传入 `statusFilter`；可选 watch 切换时自动刷新。 |

**后端是否加 filtering 参数**：已支持；本次只是在现有 filtering 上根据 `status_filter` 决定「是否加 effective_status、以及加哪些值」，无需新能力，仅扩参。

---

## 五、风险点

1. **Facebook API 行为**：当 `filtering=[]` 或仅含 name 时，部分账号/层级可能返回量很大或超时；若遇限流，可考虑在 `all` 时仍传一档「常见多状态」白名单（ACTIVE/PAUSED/ARCHIVED/DELETED/…）以略微收窄，而不是完全无状态条件。  
2. **分页与切换**：用户先「仅 ACTIVE/PAUSED」选了一页，再切到「全部」会重新拉第一页，已选 ID 仍会保留（现有逻辑已支持「已选但不在当前列表」的展示）。  
3. **列表字段**：严格不增加列，仅 id + name，与需求一致。

---

## 六、验证方式

1. **默认行为**：打开规则编辑/新建 → 选账户 → 选层级「广告」→ 不切换状态筛选 → 点击「刷新列表」。  
   - **预期**：列表中仅出现 ACTIVE、PAUSED 的广告（必要时在 FB 后台对照若干条 id 的 effective_status）。  
2. **切换 ALL**：将状态筛选改为「全部 (ALL)」→ 再点「刷新列表」（或若做了 watch 则自动刷新）。  
   - **预期**：列表中可出现 ARCHIVED、DELETED、CAMPAIGN_PAUSED 等其它状态；列表列仍仅为 id + name。  
3. **加载更多**：在默认 ACTIVE/PAUSED 下加载多页，再切 ALL 再加载更多，确认分页与筛选一致、无串参。  
4. **接口直测**（可选）：  
   - `GET /api/structure/ads?account_id=act_xxx&status_filter=active_paused` → 返回项均为 ACTIVE/PAUSED。  
   - `GET /api/structure/ads?account_id=act_xxx&status_filter=all` → 返回项含其它状态。

---

## 七、小结

- **目标**：默认只拉/只展示 ACTIVE/PAUSED，可切 ALL；列表保持 ad_id + ad_name。  
- **实现**：后端 `status_filter` 控制 filtering；前端状态筛选控件 + 两处请求带 `statusFilter`。  
- **风险**：FB 在 `all` 时可能返回量大；分页与切换已选 ID 保留。  
- **验收**：默认仅 ACTIVE/PAUSED、切 ALL 可见其它状态、列表字段不变。
