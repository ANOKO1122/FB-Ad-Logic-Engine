# 结构镜像：三层查库 — 验收与排故（教学版）

按 .cursorrules 教学三部曲：拆解概念 → 安全实操 → 验证与排故。

---

## 一、拆解概念

### 1) 一句话比喻

**结构镜像表**就像「本地通讯录」：FB API 是「运营商通讯录」，每次查列表都要打电话；本地建了 campaigns/adsets/ads 三个表后，查列表就是翻本地通讯录，0 次电话。

### 2) 为什么要学（测开视角）

- **面试高频**：为什么 mock 时要和实现「参数对齐」？为什么接口增加参数后单测会悄悄挂？
- **可维护性**：单测的断言要跟随接口契约；接口变了，断言必须同步更新。
- **外企 WLB**：自动化测试能在改完代码后立刻告诉你「哪里对不上」，避免上线后才发现。

### 3) 面试怎么问 + 回答要点

| 问法 | 回答要点 |
|------|----------|
| 单测为什么突然挂了？ | 接口签名变了（比如 updateAdsetBudget 增加了 isDaily 参数），断言仍用旧参数，expect().toHaveBeenCalledWith(a, b) 与实际 (a, b, c) 不匹配。 |
| 如何避免这种问题？ | ① 接口变更时顺带改单测；② 用类型/契约测试（如 Zod、OpenAPI）约束参数；③ 回归跑全量测试。 |
| mock 和 stub 区别？ | mock 会验证「调用次数、参数」；stub 只提供返回值，不关心怎么被调用。 |

### 4) 本节学习目标

- 能看懂「expected X, received X+1 参数」类失败，并定位到接口/实现变更。
- 能按接口契约更新单测断言。
- 能区分「与本次改动相关」和「与本次改动无关」的失败。

### 2.2 单测失败根因（与三层查库无关）

| 失败用例 | 期望 | 实际 | 根因 |
|----------|------|------|------|
| actionExecutorBudgetIdempotent.test.js | `updateAdsetBudget(adsetId, 900)` 两参数 | `updateAdsetBudget(adsetId, 900, true)` 三参数 | actionExecutorService 在 ABO/CBO 改造时增加了 `isDaily` 参数；单测断言未同步 |

**重要**：该失败与 structure_campaigns/adsets 改动**无关**，是历史遗留的「接口与断言不同步」。

### 2.3 修复内容

- **文件**：`server/tests/actionExecutorBudgetIdempotent.test.js`
- **修改**：`toHaveBeenCalledWith(adsetId, 900)` → `toHaveBeenCalledWith(adsetId, 900, true)`
- **原因**：`updateAdsetBudget(adsetId, newBudgetCents, isDaily)` 第三参数 `isDaily` 在 ABO 且 daily_budget>0 时为 `true`，mock 断言需包含此参数。

## 四、最容易出的 3 个 Bug + 快速定位

| Bug | 现象 | 快速定位 |
|-----|------|----------|
| 1. 接口参数变更未同步断言 | 单测 expect(A,B) 实际 (A,B,C) | 看报错 "expected ... to be called with [A,B]"，Received 多了参数，去实现里搜调用处，把新参数补进断言。 |
| 2.  migration 未执行，表不存在 | 列表接口 500 或 "Unknown table" | `SHOW TABLES LIKE 'structure_%'`；缺哪个表就执行对应迁移。 |
| 3. campaigns/adsets 列表为空 | 返回 items: [] | 表是新建的，需要先执行 `POST /api/sync/structure` 拉取 FB 数据并 upsert。 |
