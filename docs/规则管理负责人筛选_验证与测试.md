# 规则管理页「负责人筛选」— 验证与测试说明

本文说明如何验证和测试规则页的**负责人筛选**功能（方案 B：联表一次查询）。

---

## 一、功能要点（回顾）

- **谁可见**：仅**管理员**登录后，规则列表页顶部会出现「负责人」多选下拉；普通用户不显示。
- **语义**：不选 = 查全部负责人的规则；选若干负责人 = 只显示这些负责人下属用户创建的规则。
- **数据来源**：后端 `GET /api/rules` 通过 `rules → users → owners` 联表一次查出，每条规则带 `ownerId`、`ownerName`；管理员选负责人时请求带 `?ownerIds=1,2,3`。

---

### 1. 规则 API 测试（含 targetIds 复合 ID 修复）

负责人筛选**未单独写用例**，但依赖的规则列表接口在现有测试中已覆盖。请先跑通规则测试，确保无回归：

```bash
npm test -- --run server/tests/rules.test.js
```

**预期**：18 个用例全部通过（包含 GET /api/rules、创建/更新规则的 targetIds 复合 ID 格式）。

### 2. 可选：为负责人筛选加接口测试

若希望显式覆盖「带 ownerIds 的 GET /api/rules」和「返回含 ownerId/ownerName」：

- 使用**管理员 token** 请求 `GET /api/rules`，断言 `res.body.rules` 中每条规则存在 `ownerId`、`ownerName`（可为 null）。
- 再请求 `GET /api/rules?ownerIds=1`（1 为真实存在的 owner id），断言返回的规则对应的 `ownerId` 均为 1。

测试库中需有：至少一个 owner、两个用户（分别绑定不同 owner）、每人至少一条规则，再用 admin 查全部与按 ownerIds 过滤。

## 三、手工验证（推荐步骤）

### 前置条件

- 本地或测试环境：后端已启动（如 `npm run dev:server`），前端已启动（如 `npm run dev`）。
- 数据库中有至少 **2 个负责人**（`owners` 表）、每个负责人下至少 **1 个用户**（`users.owner_id`）、每个用户至少 **1 条规则**（便于区分「全部」与「按负责人筛选」）。

### 步骤 1：管理员看全部规则并确认负责人信息

1. 使用**管理员账号**登录。
2. 进入「规则管理」页。
3. **预期**：
   - 页面顶部有「负责人」多选下拉（不选时提示「不选=全部」）。
   - 规则卡片上每张有「负责人：xxx」或「负责人：未分配」。
   - 列表接口返回的每条规则应带 `ownerId`、`ownerName`（可在浏览器 Network 里看 `GET /api/rules` 的响应）。

### 步骤 2：按负责人筛选

1. 在「负责人」下拉里**只选一个负责人**（如「张三」）。
2. **预期**：列表刷新，只显示该负责人下属用户创建的规则；卡片上的「负责人」均为该负责人。
3. 再**多选两个负责人**。
4. **预期**：列表只显示这两个负责人下的规则。
5. **清空选择**（取消勾选所有负责人）。
6. **预期**：列表恢复为「全部规则」，与步骤 1 一致。

### 步骤 3：普通用户无筛选、无负责人列

1. 退出管理员，用**普通用户**（staff，绑定某一 owner）登录。
2. 进入「规则管理」页。
3. **预期**：
   - **不显示**「负责人」筛选下拉。
   - 规则卡片上**不显示**「负责人：xxx」一行。
   - 列表仅包含该用户自己的规则（与是否传 ownerIds 无关，后端会忽略 ownerIds）。

### 步骤 4：接口直测（可选）

用 curl 或 Postman（需先拿到 token）：

```bash
# 管理员 token，不按负责人过滤
curl -s -H "Cookie: token=YOUR_ADMIN_TOKEN" "http://localhost:3000/api/rules" | jq '.rules[0] | {id, ruleName, ownerId, ownerName}'

# 管理员 token，只查 owner_id=1 的规则
curl -s -H "Cookie: token=YOUR_ADMIN_TOKEN" "http://localhost:3000/api/rules?ownerIds=1" | jq '.rules | length, .rules[0] | {ownerId, ownerName}'
```

- 第一条：应返回第一条规则的 `ownerId`、`ownerName`（可能为 null）。
- 第二条：返回的规则条数应 ≤ 总规则数，且每条 `ownerId` 应为 1。

## 四、常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 管理员看不到「负责人」下拉 | 首屏 `GET /api/rules` 未返回或未用 `data.isAdmin` 更新前端状态 | 看 Network 里 `/api/rules` 的 `isAdmin` 是否为 true；刷新页面再试。 |
| 选了负责人后列表为空 | 该负责人下没有用户或没有规则 | 在 DB 检查 `users.owner_id`、`rules.user_id`，确认有数据。 |
| 卡片上负责人为「未分配」 | 对应用户 `users.owner_id` 为 NULL 或关联不到 owners | 检查 `users.owner_id` 与 `owners.id` 是否一致。 |
| 普通用户看到负责人筛选 | 前端未按 `isAdminFromRules` 隐藏 | 确认仅当 `data.isAdmin === true` 时显示筛选与卡片负责人行。 |

## 五、小结

- **自动化**：跑 `npm test -- --run server/tests/rules.test.js` 通过即可认为规则接口与 targetIds 行为正常。
- **手工**：管理员登录 → 看全部规则与负责人列 → 按负责人多选/清空 → 普通用户确认无筛选无负责人列。
- **接口**：用 admin token 调 `GET /api/rules` 与 `GET /api/rules?ownerIds=1` 检查返回结构与过滤结果。

按上述步骤验证通过后，即可认为「规则页面筛选负责人」功能工作正常。
