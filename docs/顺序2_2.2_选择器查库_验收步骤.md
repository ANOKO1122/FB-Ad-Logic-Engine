# 顺序2 阶段 2.2：选择器查库 — 验收步骤

> **约定**：验收由你在本地执行；服务端不发起任何 FB 列表请求。

---

## 一、实现摘要

- **接口**：`GET /api/structure/ads`（即 `GET /api/structure/:level` 当 `level=ads`）
- **行为**：只查 `structure_ads` 表，支持 `q`、`limit`、`after`、`include_paused`；**0 次 FB 列表调用**。
- **权限**：沿用 `assertAccountAccess`，非 admin 查他人 account 仍 403。
- **过滤口径**：
  - `q` 为空：只返回 ACTIVE；`include_paused=1` 时再加 PAUSED。
  - `q` 非空：6 类状态（ACTIVE/PAUSED/PENDING_REVIEW/DISAPPROVED/IN_PROCESS/WITH_ISSUES），仍排除 DELETED/ARCHIVED。
- **分页**：`paging.after` 为当前页最后一条的 `structure_ads.id`（稳定游标），下一页传 `after=该值`。

---

## 二、验收步骤（请你本地执行）

### 1. 准备数据

- 对至少一个账户执行过一次「强制同步结构」：`POST /api/sync/structure`，body 或 query 带 `account_id=act_xxx`，确保 `structure_ads` 有数据。

### 2. 默认短列表（仅 ACTIVE）

- 请求：`GET /api/structure/ads?account_id=act_xxx&limit=20`（不传 `q`、不传 `include_paused`）
- 期望：返回的 `items` 中每条 `effective_status` 均为 `ACTIVE`；数量 ≤ 20；`meta.source === 'structure_ads'`。

### 3. 包含暂停（include_paused=1）

- 请求：`GET /api/structure/ads?account_id=act_xxx&limit=50&include_paused=1`
- 期望：返回中可出现 `ACTIVE` 与 `PAUSED`，无 DELETED/ARCHIVED。

### 4. 搜索（q 非空）

- 请求：`GET /api/structure/ads?account_id=act_xxx&q=某关键词&limit=20`
- 期望：仅名称包含关键词的广告；状态为 6 类之一（仍无 DELETED/ARCHIVED）。

### 5. 翻页不卡

- 请求第一页：`GET /api/structure/ads?account_id=act_xxx&limit=10`
- 取响应 `paging.after`，请求第二页：`GET /api/structure/ads?account_id=act_xxx&limit=10&after=<paging.after>`
- 期望：第二页与第一页无重复、顺序稳定；响应时间正常（无长时间等待）。

### 6. 权限：非 admin 查他人 account 仍 403

- 使用**非 admin** 账号登录，请求他人账户：`GET /api/structure/ads?account_id=他人act_xxx`
- 期望：HTTP 403，且响应体含 `code: 'ACCOUNT_FORBIDDEN'` 或等价说明。

### 7. 服务端日志证明 0 FB 列表请求

- 在请求 2.2 接口时观察服务端控制台/日志。
- 期望：出现 `[2.2] GET /api/structure/ads from structure_ads only, no FB request`，且该请求路径下**无** `getStructurePage`、`makeRequest` 等 FB 列表相关调用日志（即无向 graph.facebook.com 请求 `/ads` 的日志）。
