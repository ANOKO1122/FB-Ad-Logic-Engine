## 一、需修正 3 点 — 当前代码状态与结论

### 1. 多账户分页游标与 meta

**Reviewer 说法**：接口仍返回 paging.after=undefined，meta 只有 requested/allowed；与说明不一致。

**当前代码核实**（`server/index.js`）：

- **带 after 分支**：已解析 `{ i, a, pas }`，对第 `i` 个账户用 `a` 拉下一页；下一游标规则已实现（当前账户有下一页则用其 after；否则从 i+1 起找第一个 `pas[j]!=null`，构造 `{ i:j, a:pas[j], pas }`；否则 nextAfter=undefined）。meta 已含 `denied_accounts`。
- **首轮**：已计算 `perAccountAfter` 数组，`hasMore` 时生成并返回 `paging.after`（base64 的 `{ i, a, pas }`），meta 已有 `denied_accounts`、`per_account_limit`、`per_account_after`。

**结论**：游标与 denied_accounts 已实现；reviewer 可能基于旧版本。  
**已做的小完善**：将 `meta.per_account_after` 从「与 allowedIds 同序的数组」改为「`{ [account_id]: after|null }` 对象」，便于联调与排查；带 after 的响应也统一返回 `per_account_after` 对象。

### 2. ruleAppliesToAccount 空数组口径

**Reviewer 说法**：存在「ids != null 就返回 true」分支，空数组仍被判定适用。

**当前代码核实**（`cronService.js` L305–314）：

```javascript
if (targetByAccount && typeof targetByAccount === 'object' && accountId in targetByAccount) {
  const ids = targetByAccount[accountId]
  return Array.isArray(ids) && ids.length > 0
}
```

**结论**：`target_by_account[accountId]` 仅在 `Array.isArray(ids) && ids.length > 0` 时返回 true；空数组 `[]` 会返回 false，不会产生空评估与加锁。无需再改逻辑。  
（若 reviewer 指的是 `target_account_ids` 分支：该分支表示「规则适用账户列表」，只要 accountId 在列表中即适用，与「该账户下有无目标 id」是不同语义，保持现状即可。）

---

### 3. refreshScopeItems 的「按当前列表清理已选」

**Reviewer 说法**：条件仍为「仅当 !scopeIncludePaused 时清理」。

**当前代码核实**（`RuleManager.vue` L1165–1170）：

```javascript
if (scopeItems.value.length > 0) {
  const multi = selectedAccountIds.value.length > 1
  const allowedSet = new Set(scopeItems.value.map(it => multi ? `${it.account_id || ''}:${it.id}` : String(it.id)))
  ruleForm.value.targetIds = ruleForm.value.targetIds.filter(id => allowedSet.has(String(id)))
}
```

**结论**：已是「只要列表有数据就清理」，未再判断 `scopeIncludePaused`；多账户用 act:id、单账户用 id。与说明一致，无需修改。

## 五、快速验收清单

1. **multi 首轮**  
   - GET `/api/structure/objects/multi?account_ids=act_1,act_2&type=ad&limit=50`  
   - 检查：`meta.denied_accounts`、`meta.per_account_after` 为 `{ act_xxx: after|null }`、有更多页时 `paging.after` 为 base64 字符串、`paging.has_more` 与 `meta.has_more` 一致。

2. **multi 带 after**  
   - 用上一步的 `paging.after` 再请求，同一 URL 加 `after=...`  
   - 检查：返回下一页数据、无重复；`meta.per_account_after` 仍为对象；若再无更多页则 `paging.after` 为 undefined、`has_more` 为 false。

3. **ruleAppliesToAccount**  
   - 将某条规则的 `target_by_account` 中某账户改为 `[]`，触发执行或单条执行；该账户不应出现「无匹配」评估日志（且不应加锁）。

4. **refreshScopeItems 清理**  
   - 多账户下勾选若干项，切换「包含暂停」或切换层级/账户，再刷新列表；检查已选仅保留当前列表中的 id（多账户为 act:id）。

5. **单条执行摘要**  
   - 单条规则多账户执行后，查 `rule_execution_summaries`：同一 run_id 下应有多条 account_id 为具体账户的记录，以及一条 account_id='multi' 的汇总（若当前实现如此）。

6. **scope_status 与 include_paused**  
   - 请求带 `scope_status=active_only` 时，结果应仅 ACTIVE，与 include_paused 无关；未传 scope_status 时行为按 include_paused 与 q 不变。
