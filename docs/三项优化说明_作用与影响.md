# 三项优化说明：作用、优缺点与改前改后影响

> 针对「多账户 p-limit + account_ids 上限」「name_exclude NOT LIKE 大表/索引」「智能挂起/防恢复守护/再次关闭延迟」的详细说明。

---

## 一、多账户 p-limit + account_ids 上限 ≤20

### 是做什么的

- **account_ids 上限 ≤20**：限制单次请求里 `account_ids` 参数最多能传多少个账户（例如 20 个）。超过则直接 400，不执行查询。
- **p-limit(4~6)**：多账户**首轮**（无 after）时，不是「for 循环里逐个 await 每个账户」，而是用并发库（如 p-limit）限制**同时**在跑的账户查询数，例如最多 4～6 个并行，其余排队；全部完成后再合并结果、生成游标。

### 作用

- **上限**：防止有人一次传几十上百个账户，单次请求把 DB 连接/CPU 打满，影响其他接口；同时限制单次响应体大小和超时风险。
- **p-limit**：首轮多账户时，从「完全串行」变成「有限并行」，首屏延迟降低（例如 10 个账户从 10 次串行变成约 2～3 批），同时用并发度上限控制 DB 压力，避免 N 个账户同时查库。

### 改前 vs 改后

| 维度 | 改前（当前） | 改后 |
|------|--------------|------|
| 请求参数 | 可传任意多个 account_ids | 超过 20 个返回 400，并提示「最多 20 个账户」 |
| 首轮多账户执行方式 | for 循环，一个账户查完再查下一个（串行） | 用 p-limit(4) 等限制并发，最多 4 个账户同时查库，其余排队 |
| 首屏延迟 | 账户越多，总耗时越线性增加 | 账户多时总耗时明显缩短（约 1/并发度） |
| DB 压力 | 同一时刻只有 1 个账户在查 | 同一时刻最多 4～6 个账户在查，需确保连接池/索引能承受 |

### 优缺点

- **优点**：上限防滥用、控风险；p-limit 提升多账户首屏速度且可控。
- **缺点**：上限会拒绝「一次查 21 个账户」的合法需求，需前端或调用方拆成多次请求；并发会短时增加 DB 负载，需观察连接数与慢查询。

### 建议

- **account_ids 上限**：建议做，实现简单（校验 `allowedIds.length <= 20`），对正常使用影响小。
- **p-limit**：可选，在首轮 multi 的 for 循环外包一层 `pLimit(4, allowedIds.map(...))`，并观察 DB 与超时率。

## 二、name_exclude NOT LIKE 大表/索引

### 是做什么的

- 当前「名称不包含」在后端是用 SQL 的 **`name NOT LIKE '%关键词%'`** 实现的。
- 在 MySQL 里，**前导通配符**（`%xxx` 或 `%xxx%`）一般**无法**用普通 B-tree 索引做 range 扫描，优化器常退化为**全表扫描**（或大范围扫描）。表越大、调用越频繁，耗时和锁竞争越高。

### 作用

- **说明现状**：让团队知道「name_exclude 在大表上是贵操作」，避免误以为加个 name 索引就能解决。
- **可选优化**：
  - **先包含再排除**：若条件里同时有「名称包含 A」和「名称不包含 B」，可先用 `name LIKE '%A%'`（若 A 足够具体可考虑前缀索引）缩小候选集，再在应用层或小结果集上做 NOT LIKE 过滤，减少大表全表扫。
  - **局部/前缀索引**：对 name 建前缀索引（如 `name(50)`）对 `LIKE 'xxx%'` 有用，对 `%xxx%` / `NOT LIKE '%xxx%'` 帮助有限，只能作为辅助（排序、其他条件先缩小集合后再用 name 过滤时可能用到）。

### 改前 vs 改后

| 维度 | 改前（当前） | 改后（若做优化） |
|------|--------------|------------------|
| SQL | 直接 `WHERE ... AND (name NOT LIKE '%关键词%' OR name IS NULL)` | 方案 A：不变，仅文档注明风险；方案 B：先查「包含」候选再在应用层或子查询中排除 |
| 索引利用 | NOT LIKE 难以用上 name 索引，大表易全表扫 | 若先「包含」再排除，包含部分有可能用上前缀或全文等，整体扫描行数减少 |
| 语义 | 一致 | 一致（最终仍是「包含 AND 不包含」） |

### 优缺点

- **优点**：文档注明后，排查慢查询时有据可依；若做「先包含再排除」，大表下可显著减少扫描行数。
- **缺点**：方案 B 实现更复杂（两段查询或子查询），需保证语义一致；前缀索引对 `%xxx%` 帮助有限。

### 建议

- **当前**：保持现有 SQL，在方案文档中已注明「NOT LIKE 可能全表扫、大表/高频 name_exclude 时注意」即可。
- **后续**：若监控到 name_exclude 查询慢，再考虑「先包含再排除」或配合其他条件缩小集合。

## 三、智能挂起 / 防恢复守护 / 再次关闭延迟

### 分别是做什么的

- **智能挂起（已有）**：即当前 **Smart Mute**。给广告设 `mute_until`（和可选 `mute_reason`），在未过期前规则执行会跳过该广告（不执行 pause 等动作），并记 skip_reason=muted。用于「运营手动开启广告后，一段时间内不要再自动关掉」。
- **防恢复守护（建议里的新能力）**：指在规则之外，再跑一个**定时/事件任务**，专门盯着「最近被系统 pause 掉的广告」：若发现它们在 FB 侧又被人工改回 ACTIVE，就再次执行 pause 或记录，使「系统关掉的广告」不会悄悄被人工恢复而长期跑下去。名字里的「守护」强调这是一种**持续兜底**，而不是单次执行。
- **再次关闭延迟（3 小时等）**：指「人工开启后，系统**隔多久**再次把它关掉」。若没有防恢复守护，只能依赖**下一轮规则扫描**（例如 15 分钟一次）再次命中并执行 pause，所以看起来会有「延迟」；若 mute_until 设得较长（如 3 小时），则这 3 小时内规则根本不会动它，所以也会表现为「人工开启后 3 小时才再次被关」。

### 作用

- **智能挂起**：避免「刚手动开启又被规则立刻关掉」的糟糕体验，给运营一个时间窗口。
- **防恢复守护**：在「规则按周期跑」之外，增加一层对「被关广告是否被恢复」的监控与再次关闭，缩短「人工恢复 → 再次被关」的间隔，或至少可观测。
- **再次关闭延迟**：由「规则执行周期 + mute_until + 是否有防恢复任务」共同决定；优化方向是缩短从「人工恢复」到「再次被关」的时间（例如加防恢复任务、或缩短 mute 默认时长）。

### 改前 vs 改后

| 维度 | 改前（当前） | 改后（若做防恢复守护） |
|------|--------------|--------------------------|
| 规则执行 | 有 mute_until 则跳过该广告 | 不变 |
| 人工恢复后的处理 | 仅依赖下一轮规则扫描（如 15 分钟）再次命中并 pause；若在 mute 期内则不会动 | 增加独立任务：定期/事件驱动检查「最近被系统 pause 的广告」在 FB 是否又变 ACTIVE，若是则再次 pause 或记录 |
| 「再次关闭」延迟 | 至少一个规则周期（如 15 分钟），若在 mute 期则更长（如 3 小时） | 可由防恢复任务周期决定（如 5 分钟扫一次），不受规则周期和 mute 影响（若设计成绕过 mute） |

### 优缺点

- **防恢复守护**  
  - **优点**：人工恢复后能更快再次被关或可观测，策略更可控。  
  - **缺点**：多一套任务与逻辑，需要明确与 mute、规则执行的关系（是否绕过 mute、是否重复写日志等），可能增加 API 调用（查状态、发 pause）。
- **再次关闭延迟**  
  - 当前「延迟」主要来自：规则周期、mute_until 时长、是否有防恢复任务。  
  - 若发现「人工开启后 3 小时才再次关闭」，可先查：该广告 mute_until 是否设了 3 小时；规则是否 15 分钟才跑一次；today 快照是否只写 spend>0 导致该广告未进评估集合。

### 建议

- **智能挂起**：已实现，保持现状。
- **防恢复守护**：作为**独立需求**排期更合适，与现有 mute、规则执行解耦，避免混在一起难以维护；实现前需定好：触发频率、是否绕过 mute、是否限流/配额。
- **再次关闭延迟**：先通过「mute_until + 规则周期 + 快照口径」排查，再决定是否上防恢复任务或调整 mute 默认值。

## 四、三者区别小结

| 项目 | 主要面向 | 改动的层次 | 典型改动 |
|------|----------|------------|----------|
| **多账户 p-limit + 上限** | 接口安全与多账户首屏性能 | 请求校验 + 首轮执行方式 | 限制 account_ids 个数；首轮用 p-limit 并发查库 |
| **name_exclude NOT LIKE** | 大表查询性能与索引 | SQL/查询策略 + 文档 | 文档注明风险；可选「先包含再排除」或索引优化 |
| **智能挂起/防恢复/延迟** | 运营体验与「被关广告是否被恢复」的策略 | 产品策略 + 任务设计 | 已有 mute；可选新增防恢复守护任务；排查延迟根因 |

- **多账户**：偏「接口与资源控制 + 性能」，改的是**怎么调 DB、允许多少账户**。
- **name_exclude**：偏「单次查询成本与索引」，改的是**怎么写 WHERE、是否拆成多步查询**。
- **智能挂起/防恢复/延迟**：偏「行为策略与任务形态」，改的是**何时跳过执行、是否额外盯住被恢复的广告**。
