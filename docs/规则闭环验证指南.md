# 规则闭环验证指南

完成时区修复、口径统一后，需做最后两项闭环验证。

---

## 一、Dry Run 闭环（指标快照一致性）

**目的**：确认 `automation_logs.metrics_snapshot` 中的 spend/link_clicks/cpc 满足 `cpc = spend / link_clicks`（与规则引擎口径一致）。

### 步骤

**1. 取最新一条有 metrics_snapshot 的 automation_logs**

```sql
SELECT 
  id,
  run_id,
  ad_id,
  metrics_snapshot,
  triggered_at
FROM automation_logs
WHERE metrics_snapshot IS NOT NULL
  AND JSON_LENGTH(metrics_snapshot) > 0
ORDER BY triggered_at DESC
LIMIT 1;
```

**2. 用 JSON 函数提取并验证 cpc = spend / link_clicks**

```sql
SELECT 
  id,
  ad_id,
  run_id,
  JSON_UNQUOTE(JSON_EXTRACT(metrics_snapshot, '$.spend'))        AS spend,
  JSON_UNQUOTE(JSON_EXTRACT(metrics_snapshot, '$.link_clicks'))  AS link_clicks,
  JSON_UNQUOTE(JSON_EXTRACT(metrics_snapshot, '$.cpc'))          AS cpc,
  CASE 
    WHEN CAST(JSON_UNQUOTE(JSON_EXTRACT(metrics_snapshot, '$.link_clicks')) AS UNSIGNED) > 0 
    THEN CAST(JSON_UNQUOTE(JSON_EXTRACT(metrics_snapshot, '$.spend')) AS DECIMAL(12,4)) 
         / CAST(JSON_UNQUOTE(JSON_EXTRACT(metrics_snapshot, '$.link_clicks')) AS UNSIGNED)
    ELSE NULL 
  END AS computed_cpc
FROM automation_logs
WHERE metrics_snapshot IS NOT NULL
  AND JSON_LENGTH(metrics_snapshot) > 0
ORDER BY triggered_at DESC
LIMIT 5;
```

**3. 与数据源对照（规则 today 用 ad_snapshots）**

```sql
-- 用上一步查到的 ad_id 替换 120240693013880501
SELECT ad_id, spend, link_clicks, unique_link_clicks,
       ROUND(spend / NULLIF(link_clicks, 0), 4) AS computed_cpc
FROM ad_snapshots
WHERE account_id = 'act_1155891202142879'
  AND ad_id = '120240693013880501'
  AND data_date = CURDATE();
```

**4. 判定**

- snapshot 的 cpc 与 computed_cpc 差异 < 0.01：通过
- snapshot 的 spend/link_clicks 与 ad_snapshots 一致：通过

## 二、真实执行闭环（非 simulation）

**目的**：确认真实执行时 API 调用成功，且 Ads Manager 侧确有变更。

### 步骤

**1. 准备规则**

- 新建规则或把现有 Dry Run 规则改为 `is_simulation = 0`
- 动作选「风险最低」的一种，例如：
  - `decrease_budget`，value 设很小（如 1 分）
  - 或 `pause_ad` 针对一条测试广告（可手动恢复）

**2. 执行规则**

- 在 Rule Manager 手动执行该规则
- 或调用 API：`POST /api/rules/:id/execute`

**3. 检查 automation_logs**

```sql
SELECT 
  run_id,
  ad_id,
  action_type,
  status,
  error_message,
  api_request,
  api_response,
  triggered_at
FROM automation_logs
WHERE run_id = '你刚执行的 run_id'
ORDER BY triggered_at DESC;
```

- `status = 'success'`：执行成功
- `api_response` 含成功字段（如 Facebook 返回的 id）：API 成功

**4. 对账 Ads Manager**

- 登录 Ads Manager，找到对应广告
- 确认预算/状态与规则动作一致
- 若为 pause，确认广告已暂停

**5. 冷却/去重验证**

- 再次执行同一规则（或等冷却期过后再执行）
- 预期：
  - 有冷却期：第二次不重复执行
  - 有 mute：挂起期间不执行
  - 无冷却期：第二次会再次执行，但不应出现重复动作（如多次 pause 同一条广告）

## 三、快速检查 SQL（可选）

```sql
-- 最近 20 个 run 的 triggered_at 与 evaluated_at 差值（应接近 0）
SELECT
  s.run_id,
  s.evaluated_at,
  a.triggered_at,
  TIMESTAMPDIFF(SECOND, a.triggered_at, s.evaluated_at) AS diff_sec
FROM rule_execution_summaries s
JOIN automation_logs a ON a.run_id = s.run_id
ORDER BY s.evaluated_at DESC
LIMIT 20;
```
