结构同步任务改造 — 执行方案（定稿）
一、目标
少拉数据、少限流：不再全量拉 campaigns / adsets / ads，只拉「近 3 天有更新」的数据。
数据不丢、可更新：
近 3 天新创建的 → 会进来（创建时 updated_time = created_time）。
很久以前创建、近 3 天改过名/改过状态的 → 也会进来，按 id 更新覆盖。
顺带把 created_time 落库：在同一条请求里用 fields 带上 created_time 和 updated_time，一次拿到两个字段并写入三张结构表；created_time 来自 Meta，准确可靠。
不改动：热数据、冷数据、归档、Piggyback、每小时轮转的调用方式；只改「拉列表 + 写三张结构表」这一段。
二、核心逻辑
项目	说明
拉取条件	每个 edge 只发 1 条请求：filtering: updated_time >= sinceISO（sinceISO = 当前时间 − 3 天，ISO 格式）。
为何一条就够	在 Meta 里 updated_time ≥ created_time；「近 3 天创建」的集合是「近 3 天更新」的子集，只按 updated_time 筛即可同时覆盖「新建」和「最近修改」。
字段	同一请求的 fields 中包含 created_time 和 updated_time，一次响应里两个字段都有；created_time 为真实创建时间。
落库	按 id（campaign_id / adset_id / ad_id）做 upsert：有则更新（name、status、created_time、updated_time 等），无则插入。
首次全量	不做；默认三张表已有数据；某账户某表缺数时用现有补数脚本（如 backfill-adsets-act_927139705822379.js）单独补。
三、需要改动的代码
1. server/index.js — getStructurePage
位置：返回的 items 的映射（约 119–128 行）。
改动：在映射中增加 created_time，例如：
created_time: x.created_time || null
这样结构同步拿到的 created_time 才会被写入 DB；当前未映射导致库里 created_time 全为空。
2. server/services/structureSyncService.js — syncAccountStructureAds
计算时间
在拉取前算一次：
sinceSec = Math.floor((Date.now() - 3 * 24 * 60 * 60 * 1000) / 1000)（3 天前的 Unix 秒）。
filtering 的 value：
优先用 Unix 时间戳（秒）：value: String(sinceSec)；若某 edge 报错或文档要求，再改为 ISO：value: new Date(sinceSec * 1000).toISOString()。
这样和「官方推荐 Unix 时间戳」一致，也保留 ISO 作为备选。
campaigns
删除：当前「全量分页、filtering: null」的 do-while。
改为：
单一路径：getStructurePage(accountId, 'campaigns', { fields: CAMPAIGNS_FIELDS, limit: 100, after, filtering: [{ field: 'updated_time', operator: 'GREATER_THAN', value: sinceISO }] })；
分页拉完（按 after 循环直到无下一页）；
调用现有 doStructureCampaignsUpsert(lockConnection, accountId, allCampaigns)。
注意：campaigns 此前因 effective_status filtering 易触发 FB #100，这里仅用 updated_time，不再加 status 类筛选；若后续需按状态过滤，在应用层做。
adsets
删除：当前「全量分页、filtering: null」的 do-while。
改为：
同上，edge 改为 'adsets'，fields 用 ADSETS_FIELDS；
filtering 仅：updated_time >= sinceISO；
分页拉完后 doStructureAdsetsUpsert(lockConnection, accountId, allAdsets)。
ads
删除：当前「全量分页 + baseFiltering」的 do-while；可选：同时移除或保留实验性「增量」路径（建议移除，统一走下面逻辑）。
改为：
单一路径：getStructurePage(accountId, 'ads', { fields: ADS_FIELDS, limit: 100, after, filtering: [{ field: 'updated_time', operator: 'GREATER_THAN', value: sinceISO }] })；
分页拉完；
调用现有 doStructureAdsUpsertAndStatus(lockConnection, accountId, allItems, cursorTs, isFullRun)。
isFullRun：因不再做全量，这里可传 false；如需保留 last_full_count / has_full_synced 的语义，可按现有规则传（例如仍传 false，仅更新 last_success_at 等）。
常量
确认 CAMPAIGNS_FIELDS、ADSETS_FIELDS、ADS_FIELDS 已包含 created_time 和 updated_time（当前代码里已有，无需再改）。
3. 其它文件
不改：cronService（结构轮转仍调用 syncAccountStructureAds）、ingestorService（Piggyback、热数据等）、冷数据/归档逻辑。
补数：某账户某表为空时，继续用现有脚本（如 backfill-adsets-act_927139705822379.js）或同类脚本补全，不在此次「结构同步任务」内做首次全量。
四、数据流小结
每小时（或手动）触发结构同步 → 调用 syncAccountStructureAds。
计算 sinceISO = 当前时间 − 3 天（ISO）。
对 campaigns：带 updated_time >= sinceISO 的 filtering、fields 含 created_time/updated_time，分页拉完 → upsert structure_campaigns。
对 adsets：同上 → upsert structure_adsets。
对 ads：同上 → upsert structure_ads，并更新 structure_sync_status（按现有 doStructureAdsUpsertAndStatus 逻辑）。
每次拉到的对象都会带上 created_time 和 updated_time，getStructurePage 映射 created_time 后，三张表的 upsert 会把 created_time 一起写入，之后新同步到的行都会有准确的创建时间。
五、验证
接口：在 Graph API Explorer 对 campaigns / adsets / ads 各测一次：
filtering: [{"field":"updated_time","operator":"GREATER_THAN","value":"<sinceISO>"}]，
fields 含 created_time,updated_time，确认返回条目的两个时间都有且符合「近 3 天」预期。
DB：
执行一次结构同步（或等一次定时）。
用你之前的 SQL 查三张表：
各账户 total / with_created_time；
抽样看 created_time、updated_time 是否有值且合理。
行为：
近 3 天新建或修改过的 campaign/adset/ad 应出现在对应表中，且名称/状态与 Meta 一致；
仅更新、不新建的旧对象应被更新而非重复插入（按 id upsert）。
六、风险与注意
filtering value 格式：Meta 文档推荐对 updated_time 使用 Unix 时间戳（秒）；实现时优先用秒级时间戳，若某 edge 报错再尝试 ISO 字符串。

campaigns/adsets 的 filtering：此前未对这两类用 filtering（因 effective_status 易触发 #100）；本次仅用 updated_time，一般更安全。若上线后某 edge 报错，再考虑该 edge 回退为 filtering: null（全量）或换日期格式。
sinceISO 格式：使用标准 ISO 字符串（如 2026-02-24T00:00:00.000Z），与 FB 文档一致，避免时区/格式导致筛选异常。
空表账户：不在此方案中做首次全量，缺数继续用单独脚本补。
按上述执行即可得到：只发一条请求 per edge、拿到创建时间与更新时间、创建时间准确落库、请求数减半的最终方案。

七、补充采纳（轮转命名、可观测、格式回退）
- **轮转命名与语义一致**：`runHourlyStructureFullRotation` 仍保留函数名（兼容调用方），但 JSDoc 与日志统一为「结构轮转（近 3 天）」/「结构轮转-近3天」，避免与「全量」混淆。
- **记录过滤基准（可观测性）**：`structure_sync_status` 增加 `last_filter_since_sec`（BIGINT，上次使用的 since Unix 秒），便于核对过滤窗口与同步结果。迁移：`030_add_last_filter_since_sec_to_structure_sync_status.sql`。
- **过滤 value 格式回退**：实现时每个 edge（campaigns / adsets / ads）先使用 `value: String(sinceSec)`；若该 edge 请求报错，自动切换为 `value: new Date(sinceSec * 1000).toISOString()` 再拉一次，无需手工改代码。
