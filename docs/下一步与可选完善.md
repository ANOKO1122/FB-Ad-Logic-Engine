## 二、还可完善什么（可选）

### 1. 多账户「应用监控范围」拉全量时的请求量

- **现象**：点「应用监控范围」后，前端会 `while (scopePagingAfter)` 循环调用「加载更多」，直到没有下一页；多账户、每页 500 条时，可能连续发出很多次带 `after` 的 multi 请求（你提供的日志里同一秒内大量 `(after)` 即为此）。
- **是否问题**：当前是按设计「拉全量匹配项并全部勾选」，功能正确；只是请求次数多、耗时会较长。
- **可选优化**：  
  - 前端：加「加载中」提示或进度（如「已加载 N 条，正在加载更多…」），避免用户以为卡死；  
  - 或：提供「只加载前 M 页」选项，不全量拉完；  
  - 或：后端在合适时机构建「匹配 id 列表」的轻量接口，前端只拉 id 再按需分页展示（需产品/技术权衡）。

### 2. 测试覆盖率阶段目标

- TASKS §5：覆盖率分阶段 60% → 80%。  
- 可在不赶功能时，针对新增/改动的模块（如 structureSyncService 的 scope_status_exclude、index.js multi 路由）补单测，逐步提高覆盖率。

### 3. 回归清单（下次大改前建议跑一遍）

- 监控范围：状态等于/不等于、名称包含/不包含、单账户与多账户、加载更多与游标。
- 多账户：账户数上限（非 admin 20、admin 50）、TOO_MANY_ACCOUNTS、首轮 p-limit 行为。
- 规则执行：多账户 target_by_account、单条执行每账户摘要、ruleAppliesToAccount 空数组不评估。
