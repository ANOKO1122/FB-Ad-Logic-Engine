# 顺序2：伪增量 + Piggyback + 轮转 验证指南

本文档教你**亲手**按顺序验证/测试，不依赖自动化脚本。每步包含：做什么、看哪里、怎么算通过。

---

## 前置准备

- 本地或测试环境已启动服务（`npm run dev` 或 `node server/server.js`）。
- 有至少 1 个已绑定的广告账户（`account_mappings` 有 `is_active=1`）。
- 能访问：控制台日志、MySQL（或你的 DB 客户端）、接口（如 Postman/curl）。

---

## 1. 伪增量「降频生效」

**目的**：同一账户 1 小时内连续跑两次伪增量，第二次不应再触达**刚 touched 的那批**（看 `diffHead5` 是否变化或为空）。

**步骤**：

1. 选一个账户 `account_id`（如 `act_xxx`），记下。
2. **第一次伪增量**：触发一次统一心跳（见下方「如何触发统一心跳」），在日志里找到该账户的伪增量行，例如：
   ```text
   [伪增量] account=act_xxx recentActiveIds=50 diffIds=20 touched=20 diffHead5=["123","456","789","101","102"]
   ```
   记下这一条的 `diffHead5`（前 5 个 ad_id）。
3. **立刻再触发一次统一心跳**（或等几分钟再触发一次即可，不必等 15 分钟）。
4. 在日志里再找该账户的伪增量行。

**通过标准**：

- 第二次的 `diffHead5` **要么和第一次不同**，**要么** `diffIds=0`（即 `diffHead5=[]`）。
- 若第二次仍出现和第一次**完全相同的** `diffHead5` 且 `touched > 0`，说明 1h 降频未生效，需排查。

**如何触发统一心跳**（任选其一）：

- **推荐**：在项目根目录执行脚本（会跑一次完整统一心跳）：
  ```bash
  node server/scripts/manual-unified-heartbeat.js
  ```
- 若你有「手动触发统一心跳」的 HTTP 接口，调一次即可。
- 或等 15 分钟定时跑一次，然后看该时间点附近的日志。

---

## 2. 伪增量「旋转生效」

**目的**：多跑几轮，同一账户的 `diffHead5` 应在轮转（不是每轮都同一批）。

**步骤**：

1. 对同一账户，连续触发 3～4 次统一心跳（可每隔几分钟触发一次，或等 15 分钟 × 3～4 轮）。
2. 每次在日志里记下该账户的 `[伪增量] ... diffHead5=...`。
3. 对比这几条的 `diffHead5`。

**通过标准**：

- 多轮之间 `diffHead5` 有变化（不同轮出现不同的 id 组合），或出现 `diffIds=0`。
- 若每轮都是**完全相同**的 5 个 id 且一直 `touched > 0`，说明旋转未生效，需排查。

---

## 3. NULL 自愈

**目的**：存在 `last_synced_at IS NULL` 且 `effective_status='ACTIVE'` 的行时，跑一次伪增量后应被 upsert 成 `last_synced_at=NOW()`，且之后 1 小时内不再被选入候选。

**步骤**：

1. **造数据**（在测试/本地库执行，注意替换 `act_xxx`）：
   ```sql
   -- 查看是否已有 NULL 的 ACTIVE 行（可选）
   SELECT ad_id, name, effective_status, last_synced_at
   FROM structure_ads
   WHERE account_id = 'act_xxx' AND effective_status = 'ACTIVE' AND last_synced_at IS NULL;

   -- 若没有，可手动插一条或把某条改成 NULL（仅测试用）
   UPDATE structure_ads
   SET last_synced_at = NULL
   WHERE account_id = 'act_xxx' AND effective_status = 'ACTIVE'
   LIMIT 1;
   ```
2. 触发一次统一心跳，在日志里找到该账户的伪增量：应看到 `recentActiveIds>=1`、`diffIds>=1`、`touched>=1`，且 `diffHead5` 里包含你改的那条 ad_id（若只有这一条 NULL）。
3. **查库**：
   ```sql
   SELECT ad_id, last_synced_at
   FROM structure_ads
   WHERE account_id = 'act_xxx' AND ad_id = '你改的那条ad_id';
   ```
   `last_synced_at` 应为非 NULL（刚写入的 NOW()）。
4. **立刻再触发一次**统一心跳，再看该账户的伪增量日志：`diffHead5` 里**不应再出现**这条 ad_id（或整轮 `diffIds=0`），因为已满足「至少 1h 未同步」的相反条件。

**通过标准**：NULL 行被更新为有 `last_synced_at`，且下一轮伪增量不再把该条放进候选（降频生效）。

## 4. spend=0 账户也会跑（但不乱打 FB）

**目的**：统一心跳后置阶段对所有账户跑伪增量；spend=0 的账户应看到日志，且多为 `diffIds=0 touched=0`，不会大量调用 resolveObjectsByIds。

**步骤**：

1. 触发一次统一心跳。
2. 在日志里搜索 `[伪增量]`，统计：
   - 有多少条（应对应「有 accountId 的账户数」）。
   - 其中多少条是 `diffIds=0 touched=0`。

**通过标准**：

- 每个账户都有一条伪增量日志（证明「所有账户都跑」）。
- 多数账户（尤其 spend=0）为 `diffIds=0 touched=0`，整体没有成批的 resolve 调用（可再搜 `resolveObjectsByIds` 或 FB 请求相关日志做感性判断）。

---

## 5. Piggyback 同轮只补缺口

**目的**：Today 同步只做一次 resolve（拿 structurePayload），Piggyback 用该结果写库，缺口补查应很少。

**步骤**：

1. 触发一次统一心跳（尽量在「白天」时段或保证该账户走 Today 分支）。
2. 在日志里找：
   - Today 同步：应出现类似「批量解析活跃广告元数据（…同轮仅此一次 resolve）」。
   - Piggyback：应出现 `[Piggyback] account=... 补齐 structure_ads: 复用=X, 缺口补查=Y`。

**通过标准**：

- 有「同轮仅此一次 resolve」的日志。
- Piggyback 的「缺口补查」通常为 0 或很小；若每轮都很大，需排查是否 structurePayload 未正确传递或查缺逻辑有误。

---

## 6. 手动全量同步（2.4）冷却不被 Piggyback 误伤

**目的**：冷却基于 `structure_sync_status.last_success_at`，Piggyback/伪增量更新 `structure_ads.last_synced_at` 不应导致「刚点手动同步就冷却」。

**步骤**：

1. 让系统跑一轮统一心跳（Piggyback/伪增量会更新部分 `structure_ads.last_synced_at`）。
2. **立刻**调用手动同步结构接口（注意替换 `act_xxx` 和你的 token）：
   ```bash
   curl -X POST "http://localhost:3000/api/sync/structure" \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer 你的token" \
     -d '{"account_id":"act_xxx"}'
   ```
3. 看响应和日志：
   - 若返回成功且日志有「结构同步完成」类输出，说明**没有**被误判冷却，通过。
   - 若返回 `reason: 'cooldown'` 且你确认**没有**在 2 分钟内做过一次**真正的**手动全量同步，则说明冷却仍可能被误触发，需排查（确认冷却是否已改为查 `structure_sync_status.last_success_at`）。

**通过标准**：在仅跑过 Piggyback/伪增量的前提下，立刻调 2.4 不会被冷却挡住。

---

## 7. 每小时结构全量轮转不抢 P0

**目的**：整点后 5 分钟有轮转日志；且在 usage 高或熔断时本小时应跳过（观测即可）。

**步骤**：

1. **正常情况**：等到某小时的第 5 分钟（如 14:05），看日志是否有 `[结构轮转]` 相关输出（例如「本小时处理 N 个账户」或「本小时跳过: ...」）。
2. **跳过情况**（可选）：若你方便制造高 usage 或熔断（例如限流、Token 失效），再等到下一个整点后 5 分钟，看是否出现「本小时跳过: usage_high」或「本小时跳过: circuit_breaker」。

**通过标准**：轮转任务在整点后 5 分钟有执行或明确跳过；且不会在 P0 主心跳关键路径上抢资源（主观感受：主心跳耗时/成功率正常）。
