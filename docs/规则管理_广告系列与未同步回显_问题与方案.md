# 规则管理：广告系列层级支持 + 编辑回显「未同步」问题

## 一、问题与根因

### 问题 1：广告系列也需要像广告组/广告一样支持增减预算、暂停等操作

**现象**：规则目标层级选「广告系列 (Campaign)」、选好系列并执行规则时，日志出现：
- `❌ 查询 campaign 下的广告ID失败`
- `📋 规则 "系列test" 匹配 0 个广告`

**根因**：  
后端在解析「按 campaign 筛选」时，用 `ad_snapshots` 表按 `campaign_id` 查所属广告：

- `server/index.js`（约 1514–1522 行）、`server/services/ruleEngineDispatcher.js`（约 26–29 行）使用：
  - `filterColumn = ruleTargetLevel === 'adset' ? 'ad_set_id' : 'campaign_id'`
  - `SELECT DISTINCT ad_id FROM ad_snapshots WHERE account_id = ? AND ${filterColumn} IN (...)`
- 但 **`ad_snapshots` 表当前只有 `ad_set_id`，没有 `campaign_id` 列**（003 迁移只加了 `ad_set_id`），因此 SQL 报错（如 Unknown column 'campaign_id'），导致查不到任何广告、匹配 0 条。

结论：要支持「按广告系列」筛选并执行规则，必须先让 `ad_snapshots` 支持 `campaign_id`（加列 + 写入 + 查询）。

---

### 问题 2：编辑规则时已选对象显示「id (未同步)」，但下面列表里同一条有名称

**现象**：选好目标对象并执行规则成功，再点「编辑规则」打开弹窗，已选里显示 `120236941795330501 (未同步)`，而下方选择列表里同一条有名称显示。

**根因**：  
回显只依赖「结构表」`structure_ads`，没有用「快照表」兜底。

- 编辑时前端调用 `GET /api/structure/resolve?ids=...` → 后端 `resolveStructureAdsByIds` 只查 **`structure_ads`**（按 `ad_id`）。
- 若该广告已在 Today 同步进 **`ad_snapshots`**（有 `ad_name`），但还没进 **`structure_ads`**（未跑结构同步/伪增量/Piggyback 等），则 resolve 返回 `{ id, missing: true }`，前端就显示「(未同步)」。
- 下方列表可能来自「当前账户下的 scope 列表」（如 `GET /api/structure/ads` 或带账户的拉取），若该接口能拿到该条（例如从 FB 或别的来源），就会显示名称，造成「上面未同步、下面有名称」的不一致。

结论：resolve 在 `structure_ads` 查不到时，应用 `ad_snapshots` 最新快照做**仅回显用**的兜底（取 `ad_name`），这样只要该广告在系统里有过同步，编辑时就能显示名称，而不是一律标「未同步」。

---

## 二、解决思路与方案

### 问题 1：广告系列层级支持（按选中广告系列操作）

**目标**：  
规则目标层级选「广告系列」时，能正确解析出该系列下的广告，并对其执行与「广告/广告组」相同的动作（如增减预算、暂停广告等）。  
（执行层仍是「按广告」：先 campaign → 得到 ad 列表，再对每个 ad 做 pause_ad / 对所属 adset 做 increase_budget 等，无需改动作语义。）

**思路**：  
1. **库表**：给 `ad_snapshots` 增加 `campaign_id` 列（与现有 `ad_set_id` 一致，VARCHAR(50)，可 NULL）。  
2. **写入**：在写入 `ad_snapshots` 的地方（如 ingestorService 的 Today 同步、以及所有生成 snapshot 行的地方），从 insights/结构数据里带出 `campaign_id` 并写入。  
   - Insights 拉数时已含 `campaign_id`（如 `getAdInsights` 的 fullFields），在组 snapshot 行时把 `insight.campaign_id` 写入即可。  
3. **查询**：保持现有逻辑——`server/index.js` 与 `ruleEngineDispatcher.js` 里按 `ruleTargetLevel === 'campaign'` 用 `campaign_id IN (targetIds)` 查 `ad_snapshots` 得到 `targetAdIds`，后续评估与执行不变。  
4. **可选**：若存在 `daily_stats` 等也参与「按 campaign 查 ad」的路径，需一并确认是否有 `campaign_id`，没有则同思路加列+写入。

**变更点（概要）**：

| 位置 | 变更内容 |
|------|----------|
| `server/db/migrations/` | 新增迁移：`ALTER TABLE ad_snapshots ADD COLUMN campaign_id VARCHAR(50) DEFAULT NULL`（及注释）；如需一致，可同时为 `daily_stats` 加 `campaign_id`） |
| `server/services/ingestorService.js` | 构建 snapshot 行时增加 `campaignId: insight.campaign_id ? String(insight.campaign_id) : null`；INSERT ad_snapshots 的列与 VALUES、ON DUPLICATE KEY UPDATE 中增加 `campaign_id` |
| 其他写 ad_snapshots 的路径 | 若有其他地方写 ad_snapshots，同样加上 `campaign_id` 的读写 |

**风险点**：  
- 迁移需在无破坏性变更前提下执行（仅加列，默认 NULL）。  
- 历史数据 `campaign_id` 为 NULL，只有新同步的数据才有值；规则「按 campaign 筛选」时只命中有新快照的 campaign，属预期。

**验证步骤**：  
1. 执行迁移后，`DESCRIBE ad_snapshots` 含 `campaign_id`。  
2. 触发一次 Today 同步，查 `ad_snapshots` 中最新几条，确认 `campaign_id` 有值。  
3. 创建一条目标层级为「广告系列」、选择某系列、条件简单的规则，执行后日志不再出现「查询 campaign 下的广告ID失败」，且「匹配 N 个广告」N>0（若该系列下确有广告）。  
4. 单测：可对 `resolveTargetAdIdsForRule` 或等价的「按 campaign 解析 targetAdIds」加用例，mock DB 返回含 `campaign_id` 的 ad_snapshots，断言得到正确 ad_id 列表。

### 问题 2：编辑规则时「未同步」回显优化

**目标**：  
当已选对象在 `structure_ads` 中不存在、但在 `ad_snapshots` 中有最新记录时，回显名称而不是「id (未同步)」。

**思路**：  
- **后端**：在 `resolveStructureAdsByIds`（如 `server/services/structureSyncService.js`）中，对每个请求的 id：  
  1. 先查 `structure_ads`（现有逻辑）。  
  2. 若未命中，再按 `ad_id` 在 `ad_snapshots` 中取该广告**最新一条**（如按 `synced_at DESC` 或按 `data_date + synced_at` 取最新），若存在则返回 `{ id, name: row.ad_name || id, missing: false }`（不设 missing: true）。  
  3. 若两处都没有，再返回 `{ id, missing: true }`。  
- **前端**：无需改逻辑；只要后端对「仅有 ad_snapshots 无 structure_ads」的 id 不再返回 `missing: true`，并带上 `name`，前端就会显示名称而非「(未同步)」。

**变更点（概要）**：

| 位置 | 变更内容 |
|------|----------|
| `server/services/structureSyncService.js` | 在 `resolveStructureAdsByIds` 中，对 `structure_ads` 未命中的 id 列表，再查一次 `ad_snapshots`（按 ad_id 取最新一条，如 GROUP BY ad_id 取 MAX(synced_at) 或等效），合并进返回列表：有 name 则 `{ id, name, missing: false }`，避免误标未同步 |

**风险点**：  
- 仅用于展示，不改变规则执行逻辑；`ad_snapshots` 为热数据，量可控，按 ad_id 取最新一条成本低。  
- 若希望严格区分「仅快照有/结构表无」，可在返回对象中增加可选字段（如 `source: 'structure_ads' | 'ad_snapshots'`），便于以后前端区分展示（当前可不做）。

**验证步骤**：  
1. 构造场景：某 ad 在 `ad_snapshots` 有记录（含 ad_name），在 `structure_ads` 无记录（或先删掉该条）。  
2. 调用 `GET /api/structure/resolve?ids=该ad_id`，应返回 `{ id, name: '...', missing: false }` 或等价，且无 `missing: true`。  
3. 前端：规则保存该 ad 为目标，再编辑规则，已选区域应显示名称而非「id (未同步)」。  
4. 单测：对 `resolveStructureAdsByIds` 写用例，mock structure_ads 空、ad_snapshots 有该 id 的 name，断言返回带 name、无 missing。

## 五、与现有设计的对齐

- **目标层级**：规则层已有 `targetLevel: 'ad' | 'adset' | 'campaign'` 与 `targetIds`，无需改接口。  
- **动作语义**：继续按「广告」粒度执行（pause_ad、activate_ad、increase_budget 等），campaign 只用于筛选出「这些广告」，不新增 campaign 级 API 动作。  
- **Resolve 语义**：仍以「从结构/快照解析出可展示的 id→name」为主；用 ad_snapshots 兜底仅影响回显，不改变「未同步」在业务上的含义（仍可通过「刷新列表」把 structure_ads 补齐）。

## 七、Step 4 验证与 Step 5 UI 提示

### Step 4：如何验证/测试

以下按「数据 → 规则匹配 → 执行层 ABO/CBO」顺序验证。

**1. 数据层（campaign_id 已写入）**

- 执行迁移 020 后，`DESCRIBE ad_snapshots` 含 `campaign_id`。
- 触发一次 Today 同步，查最新快照：
  ```sql
  SELECT ad_id, campaign_id, ad_set_id, data_date, synced_at
  FROM ad_snapshots
  WHERE account_id = 'act_你的账户ID'
  ORDER BY synced_at DESC LIMIT 20;
  ```
  确认 `campaign_id` 有值（非 NULL）。
- 若规则按某 campaign 筛选，可再查该系列下是否有广告：
  ```sql
  SELECT DISTINCT ad_id FROM ad_snapshots
  WHERE account_id = 'act_xxx' AND campaign_id = '你的campaign_id' AND data_date = CURDATE();
  ```
  应返回至少 1 条（若该系列下确有广告）。

**2. 规则匹配与日志**

- 创建/使用一条**目标层级为「广告系列」**的规则，选择有广告的系列，执行该规则。
- 看后端日志应出现类似：
  - `规则限定目标广告系列: 1 个，包含广告: N 个`（N ≥ 1）
  - 不再出现「查询 campaign 下的广告ID失败」或「匹配 0 个广告」（在数据正确的前提下）。

**3. 执行层 ABO vs CBO**

- **ABO（广告组有预算）**：规则作用在「有 daily_budget 或 lifetime_budget 的广告组」上时，执行增减预算后日志应出现「广告组」相关成功信息（如「成功增加/减少预算 (广告组)」）。
- **CBO（系列预算）**：规则作用在「无广告组预算、仅系列预算」的广告上时，执行增减预算后日志应出现「广告系列(CBO)」相关成功信息（如「成功增加/减少预算 (广告系列(CBO))」）。
- 单测已覆盖：`server/tests/actionExecutorBudgetIdempotent.test.js`、`preFlight.test.js` 中 mock 了 `getAdsetBudgetDetail`，可运行 `npm test` 确认通过。

**4. 可选：执行结果接口**

若后续需要在前端展示「本次是否为 CBO」，可让单条规则执行接口在响应中返回 `budget_target: '广告组' | '广告系列(CBO)'`（与 actionExecutorService 中 `targetLabel` 一致），前端即可在结果里展示「本次已按广告系列(CBO)调整预算」。

### Step 5：CBO UI 提示（已做）

在规则编辑弹窗的 **「4. 执行动作 (Then)」** 区块中，当存在「增加预算」或「减少预算」时，展示一条说明：

- **文案**：若规则作用对象属于 **CBO（优势+系列预算）**，执行增减预算时将调整整个广告系列预算，影响该系列下所有广告组。
- **实现**：`RuleManager.vue` 中增加 `hasBudgetAction` 计算属性，在存在 `increase_budget` / `decrease_budget` 时显示上述提示框（样式为左侧蓝色竖条的信息框）。
- **目的**：用户在配置预算类动作时即可知悉 CBO 下的行为，避免误以为只改单个广告组。
