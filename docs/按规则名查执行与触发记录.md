# 按规则名查执行与触发记录

用于确认某条规则（如「2.2测新test cpc」）是否有**触发记录**（被评估/被跳过）和**实际执行记录**（动作成功写入 automation_logs）。

**本窗口相关**：规则评估时 CPC/ROAS 等**一律用本地计算**（分母为 0 则 0，不用存库的 Facebook API 值）；前端展示仍可直接用 API 数据。接口已支持按 `rule_name` 筛选（`/api/rule-execution-summaries`、`/api/automation-logs`）。验证方式见第 5 节。

## 1. 用接口查（推荐）

需**管理员**登录后调用。

### 触发/评估记录（规则执行摘要）

- **接口**：`GET /api/rule-execution-summaries?rule_name=2.2测新test cpc`
- **说明**：返回该规则的所有评估记录（含 `status`：matched / no_match / skipped / error；若被权限过滤会有一条 `skip_reason=no_permission`）。
- **有记录**：说明规则被触发过（可能被跳过，未真正执行）。
- **无记录**：说明该规则从未被纳入过某次运行的评估（例如账户未同步、或规则不在该账户下）。

### 实际执行记录（审计日志）

- **接口**：`GET /api/automation-logs?rule_name=2.2测新test cpc`
- **说明**：返回该规则触发的**实际动作**记录（成功/失败/跳过）。非 admin 默认只看到 success。
- **有记录**：说明规则曾真正执行过动作（如暂停广告）。
- **无记录**：说明没有以该规则执行过动作（可能从未匹配、或被权限/冷却期跳过）。

## 2. 用 SQL 直接查

在数据库客户端执行（将 `'2.2测新test cpc'` 换成实际规则名）：

```sql
-- 规则是否存在
SELECT id, rule_name, account_id, user_id, enabled, last_executed_at
FROM rules
WHERE rule_name = '2.2测新test cpc';

-- 触发/评估记录（rule_execution_summaries）
SELECT id, run_id, rule_id, rule_name, account_id, status, skip_reason, evaluated_at
FROM rule_execution_summaries
WHERE rule_name = '2.2测新test cpc'
ORDER BY evaluated_at DESC
LIMIT 20;

-- 实际执行记录（automation_logs）
SELECT id, account_id, ad_id, rule_id, rule_name, action_type, status, triggered_at
FROM automation_logs
WHERE rule_name = '2.2测新test cpc'
ORDER BY triggered_at DESC
LIMIT 20;
```

## 3. 如何解读

| 情况 | 说明 |
|------|------|
| `rule_execution_summaries` 有记录，`status=skipped` 且 `skip_reason=no_permission` | 规则被触发过，但因**权限**未执行（常见于 staff 创建的规则，cron 按 account_mappings 校验时该用户对该账户无权限）。 |
| `rule_execution_summaries` 有记录，`status=matched` 或 有 `executed_count` | 规则被评估且匹配/执行过，可再查 `automation_logs` 看具体动作。 |
| `automation_logs` 有该 `rule_name` 且 `status=success` | 该规则曾**成功执行**过动作（如暂停广告）。 |
| 两张表都没有该规则名 | 规则可能刚创建、所属账户尚未同步触发执行、或规则名称不一致（注意空格与大小写）。 |

## 4. 进一步确认：规则 no_match 时看快照里的 CPC/spend/link_clicks

当规则一直是 **no_match**（如 CPC>0.8 从未执行）时，可查**规则评估用到的数据**：该账户、目标广告在 **ad_snapshots** 里「今天」**最新一条快照**的 `cpc`、`spend`、`link_clicks`，确认是否都 ≤0.8 或为 null。

### 4.1 先查出规则绑定的账户和目标广告 ID

```sql
SELECT id, rule_name, account_id, user_id, target_level, target_ids
FROM rules
WHERE rule_name = '2.2测新test cpc';
```

记下 `account_id` 和 `target_ids`（JSON 数组里的 ad_id）。

### 4.2 查这 3 条广告在「今天」最新快照的 CPC/spend/link_clicks

把下面 SQL 里的 **账户 ID** 和 **3 个 ad_id** 换成上一步的结果（规则 161 对应：`act_1155891202142879`，3 个广告 `120241071446680501, 120241071446690501, 120241071446700501`）。

```sql
-- 规则 161「2.2测新test cpc」：账户 act_1155891202142879，3 个目标广告
-- 取「每个广告在今日内的最新一条快照」，看 cpc / spend / link_clicks
SELECT
  s.account_id,
  s.ad_id,
  s.ad_name,
  s.spend,
  s.link_clicks,
  s.cpc,
  s.synced_at,
  (CASE
    WHEN s.link_clicks > 0 THEN ROUND(s.spend / s.link_clicks, 4)
    ELSE NULL
  END) AS cpc_calc
FROM ad_snapshots s
INNER JOIN (
  SELECT ad_id, MAX(synced_at) AS last_synced
  FROM ad_snapshots
  WHERE account_id = 'act_1155891202142879'
    AND ad_id IN ('120241071446680501', '120241071446690501', '120241071446700501')
    AND synced_at >= CURDATE()
    AND synced_at < CURDATE() + INTERVAL 1 DAY
  GROUP BY ad_id
) t ON s.account_id = 'act_1155891202142879' AND s.ad_id = t.ad_id AND s.synced_at = t.last_synced
WHERE s.account_id = 'act_1155891202142879'
  AND s.ad_id IN ('120241071446680501', '120241071446690501', '120241071446700501')
ORDER BY s.ad_id;
```

**说明**：

- `synced_at >= CURDATE()` 且 `< 次日`：只取**当天**的快照（与 MySQL 会话时区一致；若库是 UTC，请把 `CURDATE()` 改成账户时区「今天」对应的 UTC 范围，或先查 `SELECT @@session.time_zone;` 确认）。
- `cpc`：库里存的 CPC（API 或写入时 spend/link_clicks）。
- `cpc_calc`：用当前快照的 `spend/link_clicks` 现场算一遍，便于和规则逻辑对照。
- 若某条广告 **link_clicks=0** 或 **cpc 为 null**，规则里会当 **0** 用，**0 > 0.8** 为 false，不会匹配。

### 4.3 若「今天」没有快照：看最近任意时间的最新快照

若 4.2 查不到行，说明今天还没有这 3 条广告的快照，可看**最近一次**快照（不限制日期）：

```sql
SELECT
  s.account_id,
  s.ad_id,
  s.ad_name,
  s.spend,
  s.link_clicks,
  s.cpc,
  s.synced_at,
  (CASE WHEN s.link_clicks > 0 THEN ROUND(s.spend / s.link_clicks, 4) ELSE NULL END) AS cpc_calc
FROM ad_snapshots s
INNER JOIN (
  SELECT ad_id, MAX(synced_at) AS last_synced
  FROM ad_snapshots
  WHERE account_id = 'act_1155891202142879'
    AND ad_id IN ('120241071446680501', '120241071446690501', '120241071446700501')
  GROUP BY ad_id
) t ON s.account_id = 'act_1155891202142879' AND s.ad_id = t.ad_id AND s.synced_at = t.last_synced
WHERE s.account_id = 'act_1155891202142879'
ORDER BY s.ad_id;
```

看这里的 **cpc**（或 **cpc_calc**）是否都 ≤0.8 或为 null，即可确认「规则为何一直是 no_match、未执行」。

## 5. 验证「规则判断一律用本地计算」修改成功

规则评估时 CPC/ROAS 等已改为**只用本地计算**（分母为 0 则 0，不再用存库的 Facebook API 值）。可按下面两种方式验证。

### 5.1 跑单测（推荐）

在项目根目录执行：

```bash
npm test -- server/tests/ruleDataService.test.js
```

预期：

- `应该从数据库获取账户时区，如果不存在则返回 UTC` 通过；
- `link_clicks=0 时 cpc 应为 0，不使用存库的 API 值` 通过（存库 cpc=1.5 时仍返回 0）；
- `有 link_clicks 时 cpc 应为 spend/link_clicks` 通过（按 2/5=0.4 计算）。

全部通过即表示「规则判断用本地计算」逻辑已生效。

### 5.2 手动验证（可选）

1. 在库里给某条 ad_snapshots 写一条「今日」快照：`link_clicks=0`、`cpc=1.5`（模拟 API 曾给过 1.5）。
2. 触发该账户的规则执行（等一次热数据同步触发，或调用手动执行接口）。
3. 若规则条件为 CPC > 0.8：修改前可能匹配（用存库 1.5）；修改后应 **no_match**（规则侧 cpc=0）。
4. 查 `rule_execution_summaries` 该规则最近一条：`status` 应为 `no_match`，说明判断用的是 0 而不是 1.5。
