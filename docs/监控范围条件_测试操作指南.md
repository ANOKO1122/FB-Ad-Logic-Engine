# 监控范围条件 · 测试操作指南

本文档对应《监控范围条件_后端SQL过滤与优先展示方案.md》第七章的测试清单，按项说明**具体怎么测**。

---

## 前置准备

1. **启动服务**  
   - 后端：`npm run dev:server`（默认端口 3001）  
   - 前端：按项目习惯启动（如 `npm run dev`）

2. **认证**  
   - 登录后拿到 JWT，或从浏览器开发者工具 → Application → Local Storage 里复制 token。  
   - 请求时在 **Headers** 加：`Authorization: Bearer <你的token>`

3. **工具**  
   - 后端接口：Thunder Client / Postman / curl 均可（Thunder Client 用法见《Thunder_Client_multi接口验证步骤.md》）  
   - 数量核对：任意能连库的 SQL 客户端（如 DBeaver、Navicat），执行 `server/db/migrations/verify_scope_conditions_count.sql` 中的语句

4. **账户**  
   - 单账户测试：准备一个 `account_id`（如 `act_1905531400268801`）  
   - 多账户测试：准备两个及以上有权限的 `account_id`，逗号拼接

### 为什么 Thunder Client 里广告条数很少？

**原因**：接口默认**只返回「投放中」**（`effective_status = ACTIVE`）的广告，不传任何状态参数时不会带已暂停的广告，所以条数少。

**解决办法**：在 Query 里增加下面**任一**参数即可拿到「投放中 + 已暂停」的广告：

| parameter       | value                | 说明           |
|-----------------|----------------------|----------------|
| `include_paused` | `1`                  | ACTIVE + PAUSED |
| `scope_status`   | `active_and_paused`  | 同上（监控范围条件用） |

**一键本地验证**：项目里提供了脚本，用不同参数请求 multi 并自动翻页汇总条数，可直接对比「仅 ACTIVE」与「ACTIVE+PAUSED」的数量：

```bash
# 先启动后端：npm run dev:server
# 再在项目根目录执行（把 <JWT> 和四个账户 ID 换成你的）：
node server/scripts/test-multi-scope.js "<你的JWT>" "act_1,act_2,act_3,act_4"
```

或用环境变量：`set JWT_TOKEN=xxx`、`set ACCOUNT_IDS=act_1,act_2,act_3,act_4` 后执行 `node server/scripts/test-multi-scope.js`。脚本会打印三组数量（默认 / include_paused=1 / scope_status=active_and_paused），便于确认加上参数后条数是否增加。

## 一、7.1 后端测试

### 1.1 multi 不带 after

**目的**：确认 multi 首页 200，有更多时返回分页信息。

**步骤**：

1. **GET**  
   `http://localhost:3001/api/structure/objects/multi`
2. **Query 参数**：

   | parameter   | value |
   |------------|-------|
   | account_ids | act_账户1,act_账户2 |
   | type       | ad    |
   | limit      | 50    |

3. **Headers**：`Authorization: Bearer <token>`

**检查**：

- **Status**：`200`
- **Response**：
  - 有 `items` 数组
  - 若总条数 > 50：应有 `paging.after`（一串游标）、`has_more: true`
  - 若总条数 ≤ 50：可无 `paging` 或 `has_more: false`

---

### 1.2 multi 带 after（游标分页、不重复）

**目的**：验证用上一页的 `after` 请求下一页，且两页数据不重复。

**步骤**：

1. 先按 **1.1** 发一笔，记下返回的 `paging.after`（例如 `a:123456|b:789012`）。
2. 再发一笔，**Query 里增加**：

   | parameter | value        |
   |-----------|--------------|
   | after     | （粘贴上一步的 after） |

   其他参数（account_ids、type、limit）与第一步一致。

**检查**：

- **Status**：`200`
- **items** 与第一页的 **id** 无重复（游标修复后不应出现「第二页又出现第一页的 id」）
- 若还有更多，会再返回新的 `paging.after`

---

### 1.3 multi + scope 参数（结果条数与 SQL 一致）

**目的**：带 `scope_status` 或 `name_exclude` 时，接口返回条数与数据库 COUNT 一致。

**步骤**：

1. **先查库**：执行 `server/db/migrations/verify_scope_conditions_count.sql` 中你需要的语句，例如：
   - 广告、状态=所有投放中：记下 `ad_active_count`
   - 广告组、状态=所有投放中且名称包含 0224：记下 `adset_active_name0224_count`

2. **再请求**：
   - **GET**  
     `http://localhost:3001/api/structure/objects/multi`
   - **Query** 示例（广告、仅 ACTIVE）：

   | parameter    | value        |
   |-------------|--------------|
   | account_ids | act_1,act_2  |
   | type        | ad           |
   | limit       | 500          |
   | scope_status | active_only  |

   或带名称不包含：`name_exclude=xxx`（无更多页时，一次返回总数即可对比）

3. **若结果超过一页**：需用「加载更多」逻辑把所有页请求完，或写脚本循环带 `after` 请求，把 `items.length` 累加得到总条数。

**检查**：

- 总条数 = SQL 中对应 COUNT（文档提到：修复游标后广告「所有投放中」应为 94 而非 134，可与 `ad_active_count` 对比）

---

### 1.4 单账户 + scope

**目的**：单账户接口带 `scope_status`、`name_exclude` 时过滤正确。

**步骤**：

1. **GET**  
   `http://localhost:3001/api/structure/objects?account_id=act_xxx&type=ad&limit=100&scope_status=active_only`
   或按层级用：  
   `http://localhost:3001/api/structure/ads?account_id=act_xxx&limit=100&scope_status=active_only`

2. 再测名称不包含：  
   `...&name_exclude=关键词`

**检查**：

- 返回的每条 `effective_status` 均为 `ACTIVE`（当 scope_status=active_only 时）
- 返回的每条 `name` 均不包含所填关键词（当 name_exclude=关键词 时）

## 二、7.2 前端测试（规则弹窗 → 作用对象 → 监控范围条件）

在「规则管理」里打开一条规则（或新建），进入「作用对象」，使用「监控范围条件」进行以下操作。

### 2.1 单账户 + 名称包含

- 选**一个账户**，层级选广告/广告组/广告系列之一。
- 加条件：**名称 包含** 某关键词（如 `0224`）。
- 点击「应用监控范围条件」或等价操作。

**检查**：

- 列表只展示名称包含该关键词的项。
- 超过一页时出现「加载更多」，点击能加载下一页。
- 「已勾选 N 个」与后端该账户、该条件、该层级返回的总数一致（可用 1.4 的接口带 `q=0224` 自己请求对比）。

---

### 2.2 单账户 + 名称不包含

- 单账户，加条件：**名称 不包含** 某关键词。
- 应用条件。

**检查**：

- 列表里**只展示不包含**该关键词的项（后端 SQL 过滤，不是前端大列表过滤）。
- 对这些项**全部勾选**，勾选数 = 后端带 `name_exclude=关键词` 返回的总数。

---

### 2.3 多账户 + 名称包含

- 选**多个账户**，加条件：**名称 包含** 某关键词。
- 应用条件。

**检查**：

- 会自动循环「加载更多」直到没有更多（或你手动点完）。
- 最终「已勾选 N 个」= 各账户在该条件下 SQL 合计（可用 `verify_scope_conditions_count.sql` 中对应语句或 1.3 的方式核对）。

---

### 2.4 多账户 + 名称不包含

- 多账户，加条件：**名称 不包含** 某关键词。
- 应用条件。

**检查**：

- 只展示不包含关键词的项，勾选数与多账户 SQL 合计一致（同上，用 SQL 或 multi 带 `name_exclude` 核对）。

---

### 2.5 状态条件

- 分别测试三种状态：
  - **所有投放中**
  - **所有已暂停**
  - **投放中+已暂停**
- 单账户、多账户都可测；广告/广告组/广告系列各测一次更稳。

**检查**：

- 勾选与选项一致（例如选「所有投放中」则只勾选 ACTIVE 的）。
- **多账户时**：「已勾选 N 个」= SQL 对应 COUNT（用 `verify_scope_conditions_count.sql` 或等价 SQL）。

---

## 三、7.3 数量核对（可选）

- 打开 `server/db/migrations/verify_scope_conditions_count.sql`。
- 按你当前「所有账户」范围确认：  
  - 若用 `account_mappings` 的 is_active=1，直接用文件里的语句即可。  
  - 若用全表，用文件末尾注释里的三条 COUNT（按需取消注释）。
- 在界面应用相同条件（如广告层级、状态=所有投放中），看「已勾选 N 个对象」。
- **对比**：N 应等于 SQL 查出的 `ad_active_count`（文档注明：修复游标后应为 94 而非 134）。
- 广告组/广告系列同理：按「状态=所有投放中 AND 名称包含 0224」等条件与 SQL 结果一致。

---

## 四、7.4 自动化测试（可选）

- **multi 游标**：单元/集成测试里构造「带 after 且某账户 pas[j]==null」的场景，断言不会返回 `a: null` 的游标，且下一页与上一页不重复。
- **参数透传**：测试 `scope_status`、`name_exclude` 在 `listStructureObjectsFromDb` 及各路由（`/api/structure/objects`、`/api/structure/objects/multi`、`/api/structure/:level`）中是否正确传到 list 方法并在 SQL 中生效（可 mock 数据库或集成测试查库 COUNT）。
