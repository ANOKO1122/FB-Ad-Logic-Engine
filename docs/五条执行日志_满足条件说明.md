# 五条执行日志：满足什么条件才执行了关闭动作

规则 **0226测试 (id=400)** 的触发条件是 **6 组 DNF**：组内全部 AND，组间 OR（**满足任意一组**就会执行「暂停广告」）。

---

## 一、6 组条件分别是什么（简要）

| 组 | 条件（组内全部要满足） |
|----|------------------------|
| **组1** | 花费 > 1 **且** 链接点击 = 0 **且** 购买次数 = 0 **且** 加购次数 = 0 **且** 结账次数 = 0 |
| **组2** | 花费 > 2 **且** 链接点击 ≤ 1 **且** 购买次数 = 0 **且** 加购次数 = 0 **且** 结账次数 = 0 |
| **组3** | 花费 > 4 **且** 购买次数 = 0 **且** 加购次数 = 0 **且** 结账次数 = 0 |
| **组4** | 花费 > 8 **且** 加购次数 ≤ 1 **且** 结账次数 = 0 **且** 购买次数 = 0 |
| **组5** | 花费 > 8 **且** 结账次数 = 0 **且** 购买次数 = 0 |
| **组6** | 花费 > 16 **且** 结账次数 ≤ 1 **且** 购买次数 = 0 |

快照里没有「加购次数、结账次数」时，可先按 0 假设；**要严格复核需用下面 SQL 拉出真实数据**。

---

## 二、用 SQL 拉出加购/结账真实数据后再评估

执行日志里的 `metrics_snapshot` 未包含加购次数、结账次数，需要从 **ad_snapshots** 取**当日该广告最新快照**的真实值再判断规则是否应触发。

在 **`server/db/migrations/five_execution_logs_conditions.sql`** 中已增加 **第 4 段 SQL**：按「执行日期 + 广告」关联当日该广告最新一条 ad_snapshots，得到 `add_to_cart_count`、`initiate_checkout_count`、`add_payment_info_count`。

**操作步骤：**

1. 在库里执行该文件中的 **第 4 段**（五条日志 + 加购/结账真实数据）。
2. 把查询结果里的 **spend、link_clicks、purchases、add_to_cart_count、initiate_checkout_count** 代入下面复核逻辑。
3. 对每一行用「复核判定」判断：**应触发** 且 **实际已触发** → 规则执行正确；**不应触发** 却 **实际触发了** → 异常。

**复核判定（6 组 DNF，满足任一组即应触发）：**

- **组1**：spend>1 且 link_clicks=0 且 purchases=0 且 add_to_cart_count=0 且 initiate_checkout_count=0  
- **组2**：spend>2 且 link_clicks≤1 且 purchases=0 且 add_to_cart_count=0 且 initiate_checkout_count=0  
- **组3**：spend>4 且 purchases=0 且 add_to_cart_count=0 且 initiate_checkout_count=0  
- **组4**：spend>8 且 add_to_cart_count≤1 且 initiate_checkout_count=0 且 purchases=0  
- **组5**：spend>8 且 initiate_checkout_count=0 且 purchases=0  
- **组6**：spend>16 且 initiate_checkout_count≤1 且 purchases=0  

若某条日志代入后**不满足任何一组**，但实际又执行了 PAUSE_AD，则说明当时评估或数据口径可能有问题，需再查当时快照与规则条件。若**每条都至少满足一组**，则规则当时执行正确。

---

## 五、一句话理解

- **组2**：有一点点击（≤1）但没转化 → 花了一点钱、几乎没效果 → 关掉。  
- **组3**：花了超过 4 块、仍然 0 购买（且 0 加购、0 结账）→ 继续烧没转化 → 关掉。  

所以这 5 条执行日志，都是「**满足组2 或 组3 的 DNF 条件**」后才触发了关闭动作。
