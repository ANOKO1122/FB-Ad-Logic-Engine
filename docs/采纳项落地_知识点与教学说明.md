# 采纳项落地 — 知识点与教学说明

> 本文档补足「先讲清概念再动手」的部分：说明每一项在解决什么问题、涉及哪些知识点、代码是如何实现的，方便你系统学习并在后续维护/扩展时举一反三。

## 一、状态「不等于」与 scope_status_exclude

### 1.1 业务上要解决什么问题？

- **需求**：监控范围里除了「状态 等于 所有投放中/已暂停/两者」之外，还要支持「状态 **不等于** 某一种」，例如「排除已暂停」，只对 ACTIVE 的广告生效。
- **若不做**：只能选「等于」；想排除某些状态时只能拉全量再在前端过滤，数据量大、请求多、体验差。

### 1.2 设计思路（等于 vs 不等于）

- **等于**：后端只查「在这些状态里的」→ 用 `WHERE effective_status IN ('ACTIVE', 'PAUSED')` 等，对应参数 `scope_status=active_and_paused`。
- **不等于**：后端要查「在某个状态集合里，但**排除**若干状态」。  
  做法：先按现有逻辑得到「基础状态集合」`statusList`（由 scope_status 或 include_paused/q 决定），再从里面**去掉**要排除的状态，得到最终的 `statusList`；若去掉后为空，直接返回空结果。  
  对应参数：`scope_status_exclude=PAUSED` 或 `ACTIVE,PAUSED`（逗号分隔，和后端枚举一致）。

所以：**等于** 只改「查哪些状态」；**不等于** 在「查哪些状态」的基础上再做一次「排除」。两者互斥：前端要么传 `scope_status`，要么传 `scope_status_exclude`。

### 1.3 后端实现要点（你改了什么）

1. **三层 list**（`listStructureAdsFromDb` / Campaigns / Adsets）  
   - 在已有「根据 scope_status / include_paused / q 得到 `statusList`」之后，增加：  
     - 读 `opts.scope_status_exclude`，按逗号 split、trim、去空，得到 `excludeSet`。  
     - `statusList = statusList.filter(s => !excludeSet.has(s))`。  
     - 若 `statusList.length === 0`，直接 `return { items: [], paging: { after: null } }`，避免拼一个空的 `IN ()` 给数据库。  
   - 这样 SQL 仍然是 `WHERE effective_status IN (?, ?, ...)`，只是参与的状态列表被「排除」过了。

2. **路由透传**  
   - 从 `req.query.scope_status_exclude` 读字符串，原样传给 `listStructureObjectsFromDb` 的 opts，再通过 listOpts 传给三个 list。  
   - 单账户（`/structure/objects`、`/structure/:level`）和多账户（`/structure/objects/multi` 的首轮与带 after 分支）都要传，否则前端传了参数也不生效。

### 1.4 前端实现要点（你改了什么）

1. **多一个「状态排除」的 ref**  
   - `scopeStatusFilter`：状态**等于**时用，值是 `active_only` / `paused_only` / `active_and_paused`（和接口约定一致）。  
   - `scopeStatusExcludeFilter`：状态**不等于**时用，值是后端枚举字符串 `ACTIVE` / `PAUSED` / `ACTIVE,PAUSED`（因为后端 scope_status_exclude 是逗号分隔的状态码）。

2. **应用条件时的映射**  
   - 在 `applyScopeConditions` 里，找到「状态」行 `statusRow`（field === 'status' 且有 value）。  
   - 若 `statusRow.operator === 'not_equals'`：  
     - 把 row.value（前端用的 active_only / paused_only / active_and_paused）转成后端枚举：  
       - active_only → `'ACTIVE'`，paused_only → `'PAUSED'`，active_and_paused → `'ACTIVE,PAUSED'`。  
     - 赋给 `scopeStatusExcludeFilter`，并清空 `scopeStatusFilter`。  
   - 否则（等于）：  
     - 赋给 `scopeStatusFilter`，清空 `scopeStatusExcludeFilter`。  
   - 这样一次只会带「等于」或「不等于」之一给后端，互斥清晰。

3. **请求时带上参数**  
   - 所有调用「结构列表」的地方（refreshScopeItems 单账户/多账户、loadMoreScopeItems 单账户/多账户）在传 `scope_status` 的同时，传 `scope_status_exclude: scopeStatusExcludeFilter.value || undefined`。  
   - API 层（`getStructureObjects`、`getStructureObjectsMulti`）在 params 里加上 `scope_status_exclude`，后端就能收到并参与过滤。

**你要记住的**：  
- 等于 = 指定「只查这些状态」；不等于 = 在「基础状态集合」上「排除这些状态」。  
- 前端「状态」行的 operator 决定用 scope_status 还是 scope_status_exclude；后端只认两个参数，不关心你 UI 上是等于还是不等于。

## 二、多账户账户数上限（20/50）与 TOO_MANY_ACCOUNTS

### 2.1 为什么要限制单次请求的账户数？

- **multi 接口**一次可以传多个 `account_ids`，后端会对每个账户查库并合并结果。  
- 若不做限制：  
  - 有人传 100 个账户 → 单次请求里要查 100 次库（或并发 4 个也要很多轮），响应慢、数据库压力大、容易超时。  
  - 前端「全选 36 个账户」再点加载，容易误操作成「一次拉太多」，体验和稳定性都差。

所以需要**业务上限**：单次请求最多允许多少个账户。

### 2.2 为什么是 20 和 50？管理员为什么能 50？

- **20**：对普通用户，单次选 20 个账户已经能覆盖大部分场景，同时把单次请求的负载控制在可接受范围（例如首轮 4 并发 × 5 批 = 20）。  
- **50**：管理员可能需要做「大盘巡检」「批量排查」，允许一次多选一些账户，但依然有上限，避免单次请求拖垮服务。  
- 具体数字是产品/运维的折中，代码里用常量便于以后调整（如改成 10/30）。

### 2.3 实现要点

- 在 multi 路由里，**已经得到 `allowedIds`（有权限的账户列表）之后**、**还没进「带 after 游标」或「首轮并发」之前**做校验：  
  - `maxAccounts = isAdmin ? 50 : 20`（isAdmin 来自 `req.user && req.user.role === 'admin'`）。  
  - 若 `allowedIds.length > maxAccounts`，直接 `res.status(400).json({ error: '...', code: 'TOO_MANY_ACCOUNTS' })`。  
- 这样：  
  - 前端可以根据 `code === 'TOO_MANY_ACCOUNTS'` 提示「单次最多选 20/50 个账户」；  
  - 后端不执行超量查询，保护数据库和接口稳定性。

**你要记住的**：  
- 上限是「单次请求」的账户数，不是系统总账户数。  
- 先做权限过滤得到 `allowedIds`，再按身份（是否 admin）决定上限并校验，逻辑清晰、易维护。

## 三、multi 首轮 p-limit(4) 并发

### 3.1 什么是「并发控制」？为什么不能全串行也不能全并行？

- **全串行**：一个账户查完再查下一个。账户多时总耗时会线性增长，首屏等待很久。  
- **全并行**：几十个账户同时发几十个查库请求，数据库连接数、CPU、内存容易打满，反而变慢或超时。  
- **受控并发**：同一时刻只允许 N 个请求在跑（例如 4），这 4 个里有一个完成就立刻补下一个，直到全部完成。这样既比串行快，又不会一次性压垮数据库。

### 3.2 什么是 p-limit？为什么用 4？

- **p-limit** 是 Node 里常用的「并发限制器」：你给它一个数字 N，它返回一个函数 `limit(fn)`，保证「同时执行的 `fn` 最多 N 个」。  
- 用法：`const pl = pLimit(4)`，然后 `Promise.all(tasks.map(t => pl(() => doWork(t))))`，这样最多 4 个 `doWork` 同时在跑。  
- **为什么是 4**：  
  - 首轮要查的账户数可能不少（例如 20），若并发太大，数据库压力大；太小又接近串行。  
  - 4 是经验值：既能明显缩短首屏时间，又不会把 DB 打满；项目里已有「账户级并发 6」用于别的任务，这里首轮 4 专门针对 multi 首屏，可以按后续监控再调。

### 3.3 实现要点：顺序不能乱

- 首轮要按 `allowedIds` 的**顺序**合并结果：  
  - 每个账户对应一个 `perAccountAfter[i]`，后面「加载更多」的游标是按「第几个账户、该账户的 after」来的；  
  - 若顺序错乱，游标会对不上，出现重复或漏数据。  
- 做法：  
  - 用 `Promise.all(allowedIds.map((accountId, idx) => pl(() => listStructureObjectsFromDb(...).then(result => ({ accountId, result })))))`。  
  - `Promise.all` 会按**传入的数组顺序**返回结果，所以 `results` 的顺序就是 `allowedIds` 的顺序；  
  - 再按 `results` 顺序依次 push 到 `perAccountAfter` 和合并 `allItems`，这样和原来的「for 循环顺序」一致，游标逻辑不用改。

**你要记住的**：  
- 并发控制 = 限制「同时进行」的任务数，既快又稳。  
- p-limit 负责「最多 N 个同时执行」；Promise.all 负责「结果顺序与输入顺序一致」。  
- multi 首轮顺序必须和 `allowedIds` 一致，否则游标会错。

## 四、前端「状态」行的等于/不等于与两个 ref

### 4.1 为什么需要两个 ref？

- 后端有两个互斥参数：  
  - `scope_status`：状态**等于**（active_only / paused_only / active_and_paused）；  
  - `scope_status_exclude`：状态**不等于**（逗号分隔的状态码 ACTIVE / PAUSED / ACTIVE,PAUSED）。  
- 前端同一时刻只会用其中一个：  
  - 用户选「等于 所有投放中」→ 只传 scope_status=active_only；  
  - 用户选「不等于 所有投放中」→ 只传 scope_status_exclude=ACTIVE，不传 scope_status。  
- 用两个 ref 可以：  
  - 在「应用条件」时根据 operator 写其中一个、清空另一个；  
  - 在请求时两个都传（另一个为 undefined 或空），后端会忽略空的那个。

### 4.2 前端 value 和后端枚举的对应关系

- 前端下拉的 value（给用户看的）：`active_only` / `paused_only` / `active_and_paused`。  
- 传给后端的：  
  - **scope_status**：直接传这三个字符串之一（后端已经认识）。  
  - **scope_status_exclude**：要传「状态码」而不是「语义名」，因为后端 SQL 是 `effective_status IN (...)`，库里存的是 ACTIVE/PAUSED 等。  
  - 映射：active_only → `'ACTIVE'`，paused_only → `'PAUSED'`，active_and_paused → `'ACTIVE,PAUSED'`。

**你要记住的**：  
- 等于 → scope_status（前端 value 直接传）；不等于 → scope_status_exclude（前端 value 转成后端状态码再传）。  
- 应用条件时根据 operator 二选一写 ref，请求时两个都带，后端按「有值的那个」生效。

## 五、WHERE 顺序与「名称类条件较贵」

### 5.1 为什么 WHERE 顺序重要？（索引与过滤顺序）

- 数据库用索引可以快速定位「满足某几个条件」的行，然后再对这批行做其它条件判断。  
- 我们结构表上有 `(account_id, effective_status)` 的联合索引，所以：  
  - 先写 `WHERE account_id = ? AND effective_status IN (?, ...)`，数据库可以**先用索引**缩小到「某账户 + 某几个状态」的行，可能只有几百几千行。  
  - 再在这批行上做 `name LIKE '%x%'` 或 `name NOT LIKE '%x%'`，虽然不能再用 name 的索引，但需要扫描的行数已经少了很多。  
- 若把 name 条件写在前面、或打乱顺序，优化器可能选不到「先按 account_id+effective_status 缩小」的执行计划，就会扫更多行，变慢。

### 5.2 为什么说「名称包含/不包含」较贵？

- **B-Tree 索引**适合：等值、范围、前缀匹配（如 `name LIKE 'abc%'`）。  
- **`LIKE '%关键词%'`**（前后都有通配符）：无法用 B-Tree 按顺序定位，只能在一批行上**逐行**做字符串匹配。  
- **NOT LIKE**：同样无法用索引直接表达，且优化器有时更难优化，往往变成「先取出一批行再过滤」。  
- 所以：在「已经用 account_id + effective_status 缩小过的集合」上做 name 条件，是当前表结构下比较合理的做法；若表很大，name 条件仍然会在这批行上花不少时间，因此叫「较贵」。

### 5.3 代码里做了什么？文档里补了什么？

- **代码**：在三个 list 方法里，在「构建 WHERE」的那段（`const params = [accountId]` 和 `let sql = ...`）**上方**加了一行注释，说明：顺序是先 account_id + effective_status（走索引），再 name LIKE/NOT LIKE（大表较贵），便于大表性能。  
- **文档**：在《监控范围条件_后端SQL过滤与优先展示方案》里补了一句「较贵原因」：名称包含/不包含依赖字符串匹配，无法用 B-Tree 索引直接定位，只能在已筛选行上逐行判断；若出现慢查询可考虑两段查询或应用层过滤。

**你要记住的**：  
- WHERE 顺序影响能否用好索引；我们约定「先等值/IN，再 LIKE/NOT LIKE」。  
- 名称类条件在大表上较贵是「索引特性」决定的，不是写错 SQL；注释和文档是为了以后排查慢查询时有据可依。

## 六、整体串起来：一次「应用监控范围（状态不等于）」发生了什么？

1. 用户在前端选「状态 不等于 已暂停」，点「应用监控范围」。  
2. `applyScopeConditions` 找到状态行，operator 是 `not_equals`，value 是 `paused_only`。  
3. 前端把 `scopeStatusExcludeFilter` 设为 `'PAUSED'`，`scopeStatusFilter` 清空。  
4. 调用 `refreshScopeItems` → 请求 `/structure/objects` 或 `/structure/objects/multi`，带 `scope_status_exclude=PAUSED`（不传 scope_status 或传空）。  
5. 后端在 list 里先算 statusList（例如默认 ACTIVE+PAUSED），再排除 PAUSED，得到 `['ACTIVE']`，SQL 变成 `WHERE ... AND effective_status IN ('ACTIVE')`，只返回 ACTIVE 的广告。  
6. 前端拿到列表，展示并勾选；若有多页，加载更多时继续带 `scope_status_exclude=PAUSED`，行为一致。

这样从前端「状态行」到后端「SQL 过滤」的整条链路就通了；你只要记住「等于用 scope_status，不等于用 scope_status_exclude + 状态码映射」，就能复用到其它接口或条件上。

## 七、若你以后要自己改或扩展，可以怎么想？

1. **加新的「范围条件」**（例如按广告系列名排除）：  
   - 后端：在 list 的 opts 里加参数，在 WHERE 里加对应条件，注意顺序（先 account_id/status，再 name/其它）。  
   - 前端：加 ref，在 applyScopeConditions 里根据条件行写 ref，在请求里带对应 query 参数。  
2. **改 multi 上限或并发数**：  
   - 改常量即可（MULTI_ACCOUNT_IDS_MAX、pLimit(4) 的 4），注意文档和错误提示一起改。  
3. **排查「状态不等于不生效」**：  
   - 看请求 query 里有没有 `scope_status_exclude`；  
   - 看后端 list 里是否在 statusList 计算后做了 exclude 过滤；  
   - 看前端 statusRow.operator 是否为 not_equals 以及 exclude 的 ref 是否被正确赋值。
