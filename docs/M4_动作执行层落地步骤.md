# M4 动作执行层落地步骤（具体改哪些文件/函数 + 验收点）

> 约束：Pre-Flight 不新增「对每个广告额外 GET FB」。**落地顺序按本文步骤编号**：步骤 1+6 → 步骤 2 → 步骤 3 → 步骤 4 → 步骤 5（即：先仲裁+摘要/冷却，再预算幂等，再 Pre-Flight，再 Smart Mute，最后预算单位统一）。

---

## 目标与验收标准（6 条）

| # | 目标 | 验收方式 |
|---|------|----------|
| 1 | 按广告仲裁：同一轮同一 ad_id 只执行一次动作；同狠度同动作去重；记录赢家 ruleId、被压制 ruleId 列表（写入摘要） | 同一 run_id 下同一 ad_id 在 automation_logs 仅一条「成功/失败」记录；摘要表有 winner_rule_id / suppressed_rule_ids 或等价 skip_details |
| 2 | 幂等：预算动作重试不二次叠加；第一次算出 newBudget（分）后，重试只 POST 同一 newBudget，不再 GET/重算 | 重试逻辑仅传已算好的 newBudgetCents，不再次调用 getAdsetBudget |
| 3 | Smart Mute：执行链路能拿到 mute_until/mute_reason；now < mute_until 则跳过并写 skip_reason | 日志/摘要中可见 mute_until、skip_reason（如 muted） |
| 4 | Pre-Flight：不额外 GET FB；用本地 ad_snapshots.status 判断「目标已达成则 skipped」；FB 返回 already-in-that-state 也当作 skipped/success | 无新增 GET 调用；automation_logs 出现 status=skipped + 原因；FB 报错码/文案容错 |
| 5 | 预算单位：后端全用「分」(int)；仅前端展示 /100；max_daily_budget 等统一单位 | 代码与 DB/API 中预算均为整数「分」；文档注明前端展示除 100 |
| 6 | 冷却期：命中也要冷却；被压制规则也写摘要 skip_reason=suppressed_by_priority，避免刷屏 | 被压制规则有摘要行；冷却策略见本文「冷却口径」小节（本项目采用评估即冷却，与实现一致） |

## 开工前口径（易踩坑，动代码前必读）

以下 3 点若不在开工前定死，实现到一半会出现「按文档做不通」或行为随遍历顺序变化。

### 口径 1：动作优先级 tie-break（pause vs activate 必须写死）

文档里若把 `pause_ad` 和 `activate_ad` 都标为 Level 1 且「同层按先到先得」，同一广告同时命中「暂停规则」和「激活规则」时，结果会随规则遍历顺序变化，仲裁就失去确定性。

**约定**：同一优先级内写死 tie-break —— **pause_ad 优先于 activate_ad**（按狠度：暂停 > 激活）。实现时 Level 1 内部再细分为数字子级即可，例如 `pause_ad: 1`、`activate_ad: 2`，比较时「数字小的优先」。若未来要更细粒度，再引入 `rules.priority` 字段参与比较。

### 口径 2：多动作规则在仲裁前压缩成「每规则每 ad 一条候选」

当前执行器会对一个广告把 `rule.actions` 全部执行并写多条审计日志；仲裁设计是「每个 ad 只执行一次动作」。因此**仲裁的输入不能是「整条 rule 的 actions 数组」**，否则会破功。

**约定**：**每条规则内部也只取「最狠的一条 action」参与仲裁**（按 ACTION_PRIORITY 选优先级数字最小的那条；同优先级取配置顺序第一条），其余动作不参与仲裁、不执行。即：仲裁前先把「(rule, matchedAd)」压成「(rule, matchedAd, singleCandidateAction)」，再按 ad_id 做跨规则仲裁。这样「每 ad 只执行一次动作」与「规则可配置多动作」兼容：展示上规则仍可配多动作，但本轮只执行该规则被选中的那一条。

### 口径 3：automation_logs 与 run_id 的落地方式须提前定

当前 `automation_logs` 表结构**没有 run_id 字段**。文档验收里写了「同一 run_id 下同一 ad_id 仅一条」，若不做落地方案，无法严格验收。

**二选一（开工前定好）**：

- **推荐**：给 `automation_logs` 增加 `run_id` 列（需迁移）。执行动作时传入并写入当前 runId；验收用 `SELECT run_id, ad_id, COUNT(*) FROM automation_logs WHERE run_id = ? GROUP BY run_id, ad_id` 断言每对 (run_id, ad_id) 最多 1 条。
- **不迁表**：把 runId 写入 `action_payload` 或 `metrics_snapshot` 的 JSON，验收时用「同一分钟内 triggered_at + ad_id」去重，或用 `JSON_EXTRACT(metrics_snapshot, '$.run_id')` 查询（可读性与索引不如加列）。

**本项目采用**：**run_id 选「加列」**。M4 核心目标是「按 ad 仲裁 + 可验收」，日志侧没有 run_id 会让验收和排障都打折扣；建议给 `automation_logs` 增加 `run_id` 列并做迁移，验收与后续排查均按 run_id 进行。

## 冷却口径（与步骤 1+6 一起落地）

当前代码是**只要规则在本轮被跑过就更新 lastExecutedAt**（无论 matched / no_match），即「评估即冷却」。

**约定（二选一，开工前定死）**：

- **评估即冷却**：只要规则在本轮被评估（无论 matched / no_match），都更新 `last_executed_at`。规则每 15 分钟最多被评估一次，与现有 cronService 行为一致；实现简单。
- **命中即冷却**：仅当规则本轮至少有一个 ad 命中（赢家或被压制）时才更新 `last_executed_at`；no_match 不更新，下一轮仍会再评估，反应更频繁，但评估次数可能增加。

**本项目采用**：**冷却选「评估即冷却」**。更符合当前同步节奏与「系统稳定性优先」阶段。等 M4 稳住后，若确实觉得「响应慢」影响业务，再针对少数「高危止损规则」考虑引入「更短冷却 / 单独优先级」等机制，而不是全局切到命中即冷却。

文档与实现须统一采用其一；若与当前 cronService 不一致，需在步骤 1 中一并改掉。

## 实现时注意（易踩坑，做的时候别漏）

以下三点不改文档也能做，但实现时容易踩坑，建议心里有数。

### 注意 1：Pre-Flight 的「状态等价值」别只认 ACTIVE/PAUSED

FB/快照里状态不只有这两个，常见还有 `DISABLED`、`ARCHIVED` 等。若只判断 `=== 'PAUSED'`/`'ACTIVE'`，会出现：误执行（已是不可投放/已归档仍去打 activate/pause）、误跳过（等价于已关闭但不是字面 PAUSED 没 skip，重复发请求）。

**实现建议**：用「目标已达成」集合判断——  
- **pause_ad**：`PAUSED`、`DISABLED`、`ARCHIVED` 等均视作「无需再 pause」，直接 skipped。  
- **activate_ad**：仅对「确实可激活」的状态尝试激活；遇到 `ARCHIVED` 等直接 skipped 并写原因，避免无意义重试。  

结果：更少 FB 调用、更少失败、更干净的 automation_logs。

### 注意 2：预算幂等缓存的 key（actionSignature）要包含所有影响目标值的字段

若采用「上下文缓存」方案，key 为 `(runId, adsetId, actionSignature)`。**actionSignature 必须把会影响 newBudgetCents 的输入都纳入**：至少 `type`、`value`、`max_daily_budget`（如有），以及未来可能加的护栏参数。若只含 type+value，会误复用：两次动作参数不同（如上限不同），目标值不同，缓存却返回旧值，导致预算被调错。  
**实现建议**：同 key ⇒ 目标值必然一致；signature 包含所有护栏相关字段。

### 注意 3：仲裁输出里的 matchedAd 取谁的、如何保持一致

同一 ad_id 会被多条规则命中，每条规则都会返回一个 matchedAd（结构应一致，但可能因演进出现字段缺失/命名差异）。若仲裁或执行时混用（例如 winnerRule 用 A 的 matchedAd、预算动作用 B 的），会出现缺 `ad_set_id`/`status`/`mute_*` 导致失败或误执行。

**实现建议**：仲裁阶段对**每个 ad_id 固定选一个 matchedAd 来源**（例如第一次见到的，或 winnerRule 对应的那份），后续执行层**始终用这一份**，不在不同规则间混用。这样 status / ad_set_id / mute_* 的依赖稳定可控。

## 步骤 1：按广告仲裁 + 摘要/冷却（1+6）

### 1.1 仲裁作用域（必读）

当前实现是「同一 account 锁内按 user 分组、逐 user 跑规则」。若仲裁只在**单个 user 的 userRules** 内做，同一轮同一 ad_id 仍可能被**不同用户**的规则各执行一次，与目标 #1（同一轮同一 ad_id 只执行一次）不一致。

**约定**：仲裁作用域至少覆盖**同一 account + 同一 runId 下所有可执行规则**（不按 user 分组后再各自仲裁）。即：在拿到账户锁后，先**汇总该账户下所有 cooled-down 规则**（跨 user/owner），全量评估得到「所有 (rule, matchedAds)」，再**按 ad_id 做一次仲裁**，得到「每 ad 一个赢家动作」后再执行。通常按 account 维度汇总更合理；若按 owner 汇总，需在文档中写明「同一 account 下多 owner 时如何合并」。

**权限与越权防护**：汇总前必须先**按用户权限过滤**：仅把「对该 account 有访问权」的用户的规则纳入汇总；无权用户的规则**不参与汇总、不调用 evaluateRule**（避免用离线数据把该账户广告指标暴露给无权用户），只按现有逻辑写一条 `no_permission` 摘要并跳过。这样仲裁提升到「跨 user」后也不会引入新的越权数据读取风险。

### 1.2 改动的文件与函数

| 文件 | 改动类型 | 说明 |
|------|----------|------|
| `server/services/cronService.js` | 重构 | `executeRulesForAccount` 内「按 user 循环 → 按规则评估 → 执行」改为「先汇总该 account 下所有可执行规则 → 全量评估 → 按 ad_id 仲裁（跨 user）→ 再执行」 |
| `server/services/cronService.js` | 新增 | 函数 `collectAllMatchesForAccount(ruleEngine, allRulesForAccount, lockedAccountId)`：返回 `Array<{ rule, matchedAds }>`（allRulesForAccount 为该 account 下所有 cooled-down 规则，不按 user 切） |
| `server/services/cronService.js` | 新增 | 函数 `arbitrateByAdId(matchesPerRule)`：输入见下，输出 `Map<ad_id, { winnerRule, winnerAction, matchedAd, suppressedRules }>`（**必须含 matchedAd**，见 1.3） |
| `server/services/cronService.js` | 新增 | 函数 `writeSummariesAfterArbitration(...)`：按规则写摘要，支持「同一规则部分 ad 赢、部分 ad 被压制」落一行（见 1.4） |
| `server/services/cronService.js` | 修改 | 冷却更新策略与口径一致（见本文「冷却口径」小节） |
| `server/services/actionExecutorService.js` | 新增/修改 | 支持「单动作执行」：`executeActionsForAd` 增加可选参数 `actionsOverride?: Array<action>`，若传则只执行该数组，否则用 `rule.actions`；执行时使用的 `matchedAd` 必须来自仲裁输出的 `matchedAd`（含 status、ad_set_id、mute_*） |
| `server/services/ruleExecutionSummaryService.js` | 不改表结构 | 摘要仍用现有字段；`skip_reason` 新增枚举值 `suppressed_by_priority`；赢家/被压制信息放进 `skip_details`；支持同一规则一行内既有 executed_count 又有 skip_details.suppressed_for_ads（见 1.4） |

### 1.3 关键数据结构

**`arbitrateByAdId` 输入：**

- `matchesPerRule`: `Array<{ rule: { id, ruleName, actions, ... }, matchedAds: Array<matchedAd> }>`
- 每条 rule 的 `actions` 已存在；匹配到的广告来自 `evaluateRule(rule, accountId)`。

**优先级常量（建议放 `server/services/actionExecutorService.js` 或 `server/utils/actionPriority.js`）：**

- 数字越小越优先；**Level 1 内 tie-break 写死：pause_ad 优先于 activate_ad**（见上文「开工前口径 1」）。

```js
const ACTION_PRIORITY = {
  pause_ad: 1,       // Level 1，最狠
  activate_ad: 2,    // Level 1，同层但次于 pause（tie-break 写死）
  decrease_budget: 3,
  increase_budget: 4
}
function getActionPriority(type) { return ACTION_PRIORITY[type] ?? 99 }
```

**仲裁输入（每规则每 ad 只一条候选动作）**：

- 见「开工前口径 2」：**每条规则内部也只取「最狠的一条 action」**参与仲裁（按 ACTION_PRIORITY 取数字最小的；同优先级取配置顺序第一条），其余动作不参与、不执行。
- 即：`matchesPerRule` 在进 `arbitrateByAdId` 前，先压成「每 (rule, ad) 一个 candidateAction」，再按 ad_id 分组做跨规则比较。

**仲裁输出（按 ad_id）：**

- `Map<ad_id, { winnerRule, winnerAction, matchedAd, suppressedRules: Array<{ ruleId, ruleName }> }>`
- **必须包含 matchedAd**：该 ad 的完整快照（含 `status`、`ad_set_id`、`mute_until`、`mute_reason` 及指标），供执行层做 Pre-Flight、Smart Mute、预算调用；不能只传 winnerRule/winnerAction，否则无法判断「目标已达成」和 mute。**每个 ad_id 固定选一个 matchedAd 来源**（如第一次见到的或 winnerRule 对应的那份），执行层始终用同一份，不在不同规则间混用（见「实现时注意 3」）。
- 同一 ad_id 下：先按「动作类型优先级」选最高（数字最小）；**同优先级同动作类型去重时，赢家写死为 ruleId 最小者**（不按遍历顺序，保证审计/摘要稳定）。

**摘要 skip_details 约定与「部分赢家/部分被压制」：**

- `rule_execution_summaries` 是**每条规则一行**。同一规则在同一 runId 下可能**部分 ad 赢、部分 ad 被压制**，仍只写一行：用 `executed_count` / `failed_count` 表赢的 ad 数，用 `skip_details.suppressed_for_ads` 表被压制的 ad 列表（`[{ ad_id, winner_rule_id }, ...]`）。不按「整条规则赢/整条规则输」简化。
- 赢家规则（有至少一个 ad 执行）：`status = 'matched'`，`skip_details` 可为 `null` 或含 `winner_for_ads`、`suppressed_rule_ids_per_ad`（可选）。
- 被压制规则（所有命中 ad 均被压制）：`status = 'skipped'`，`skip_reason = 'suppressed_by_priority'`，`skip_details = { suppressed_for_ads: [ { ad_id, winner_rule_id }, ... ] }`。

### 1.4 验收方式

1. **日志**：同一 run 内对同一 ad_id 只出现一次「执行: 暂停/预算…」类日志。
2. **DB（与「开工前口径 3」一致）**：若已给 `automation_logs` 加 `run_id` 列，则 `SELECT run_id, ad_id, COUNT(*) FROM automation_logs WHERE run_id = ? GROUP BY run_id, ad_id` 每对 (run_id, ad_id) 最多 1 条。若未加列，则用同一分钟内 `triggered_at` + `ad_id` 去重，或从 `action_payload`/`metrics_snapshot` 的 JSON 中取 run_id 再聚合。
3. **摘要**：`SELECT * FROM rule_execution_summaries WHERE run_id = ? AND skip_reason = 'suppressed_by_priority'` 能查到被压制规则的行；该行 `skip_details` 含 `suppressed_for_ads` 且含 `winner_rule_id`。
4. **冷却**：与本文「冷却口径」小节一致（见上）。

## 步骤 2：预算动作幂等（重试不二次叠加）

### 2.1 改动的文件与函数

| 文件 | 改动类型 | 说明 |
|------|----------|------|
| `server/services/actionExecutorService.js` | 修改 | 预算分支：若调用方传入「已算好的 newBudgetCents」，则不再调用 `api.getAdsetBudget`，直接用该值 POST。**不得在 rule.actions[] 原对象上挂 _resolvedBudgetCents**（见 2.2） |
| `server/services/actionExecutorService.js` | 新增 | 函数 `computeNewBudgetCentsOnce(api, adsetId, action, currentBudgetCents)`：纯函数，根据 currentBudgetCents 和 action 算 newBudgetCents（分），并应用 MIN/MAX 护栏；返回 `newBudgetCents`（仅计算，不写库） |

### 2.2 关键数据结构与「_resolvedBudgetCents 不能挂在 rule.actions 上」

- **禁止**：把 `_resolvedBudgetCents` 挂在 `rule.actions[]` 的**原对象**上。`rule.actions` 是规则级配置，会被同一条规则命中的**多个广告**复用，写入原对象会串到别的 ad（严重错误）。
- **正确做法二选一**：
  - **推荐**：执行时传入的「单条 action」使用**本 ad 的 action 副本**（如 `{ ...winnerAction, _resolvedBudgetCents }`），仅在该次 `executeActionsForAd` 调用内有效，不写回 rule。
  - 或：用**上下文缓存**，key 为 `(runId, adsetId, actionSignature)`，value 为已算好的 newBudgetCents；重试时从缓存读，不写回 action 对象。**actionSignature 必须包含所有影响 newBudgetCents 的字段**（type、value、max_daily_budget 及护栏相关），否则会误复用（见「实现时注意 2」）。
- 预算护栏：`MIN_BUDGET_CENTS = 100`，`max_daily_budget` 单位约定为「分」（步骤 5 统一）。

### 2.3 验收方式

1. **单测/手测**：Mock getAdsetBudget 第一次返回 1000、第二次抛错；重试后应只看到一次 getAdsetBudget 调用，两次 updateAdsetBudget(1000)（或等价 POST）。
2. **日志**：重试时日志中不应出现「当前预算: xxx」的第二次 GET（仅第一次有）。

## 步骤 3：Pre-Flight（本地 status + FB already-state 容错）

### 3.1 改动的文件与函数

| 文件 | 改动类型 | 说明 |
|------|----------|------|
| `server/services/actionExecutorService.js` | 修改 | 执行 pause_ad 前：若 `matchedAd.status` 属于「目标已达成」集合（见下），直接记 status=skipped、不调 api.pauseAd。**别只认 PAUSED**：建议含 PAUSED/DISABLED/ARCHIVED 等（见「实现时注意 1」） |
| `server/services/actionExecutorService.js` | 修改 | 执行 activate_ad 前：仅对「可激活」状态尝试激活；若为 ARCHIVED 等不可激活状态，直接 skipped 并写原因。**别只认 ACTIVE**（见「实现时注意 1」） |
| `server/services/actionExecutorService.js` | 修改 | 执行 pause_ad/activate_ad 时：catch FB 返回错误，若为「already in that state」类（如文案含 "already" 或错误码为平台约定），置 status=skipped 或 success，并写 apiResponse 说明 |
| `server/index.js`（RuleEngine 构建 matchedAd） | 修改 | 确保 `matchedAd` 含 `status`（来自 ruleDataService 的 ad_snapshots.status） |
| `server/services/ruleDataService.js` | 已包含 | queryAdSnapshots 已选 status；evaluateRule 里 matchedAds 需把 adData.status 带到 matchedAd（若当前未带则补上） |

### 3.2 关键数据结构

- `matchedAd.status`：字符串。Pre-Flight 建议用「目标已达成」集合：`pause_ad` 时若 status 属于 `PAUSED`/`DISABLED`/`ARCHIVED` 等则 skip；`activate_ad` 时仅对可激活状态尝试，ARCHIVED 等直接 skip（详见「实现时注意 1」）。
- FB 错误容错：在 catch 里解析 `error.message` 或 `error.response?.body`，匹配 "already" / "duplicate" / 具体错误码（需查 FB 文档），则视为 skipped/success。

### 3.3 验收方式

1. **本地 Pre-Flight**：把某条广告在 ad_snapshots 的 status 改为 PAUSED，触发「暂停」规则；日志与 automation_logs 应为 skipped、无 pauseAd 调用。
2. **FB 容错**：若无法造「already」错误，可在单测里 mock api.pauseAd 抛错 "Ad is already paused"，断言 automation_logs 中该条为 skipped 或 success。

## 步骤 4：Smart Mute 数据链路（mute_until/mute_reason 能拿到并跳过）

### 4.1 改动的文件与函数

| 文件 | 改动类型 | 说明 |
|------|----------|------|
| `server/services/ruleDataService.js` | 修改 | queryAdSnapshots 两处 SELECT 增加 `s.mute_until, s.mute_reason`；返回的 row 映射里增加 `mute_until`, `mute_reason` |
| `server/index.js` | 修改 | RuleEngine.evaluateRule 里构建 matchedAd 时，把 `adData.mute_until`, `adData.mute_reason` 写入 matchedAd |
| `server/services/cronService.js` | 修改 | 在「按 ad_id 执行」前：若 `matchedAd.mute_until != null && new Date() < new Date(matchedAd.mute_until)`，则该 ad 不进入执行列表，并为该 ad 上所有命中规则写摘要或合并写一条 skip_reason=muted（见下） |
| `server/services/ruleExecutionSummaryService.js` | 约定 | skip_reason 新增 `muted`；skip_details 可含 `{ mute_until, mute_reason, ad_ids: [...] }` |

Smart Mute 与仲裁的关系：先仲裁得到「每个 ad_id 的赢家动作」；再过滤掉 mute_until 未过的 ad；只对未 mute 的 ad 执行；被 mute 的 ad 若有多条规则命中，可给每条规则写一条摘要 skip_reason=muted，或按 ad 聚合写一条（产品可选）。

### 4.2 关键数据结构

- `matchedAd.mute_until`: Date 或 ISO 字符串；`matchedAd.mute_reason`: string。
- 执行前过滤：`now < mute_until` 则跳过该广告所有动作，并写摘要（skip_reason=muted）。

### 4.3 验收方式

1. **DB**：`UPDATE ad_snapshots SET mute_until = DATE_ADD(NOW(), INTERVAL 1 HOUR), mute_reason = 'test' WHERE ad_id = ?`，触发规则；该 ad 不应有 automation_logs 执行记录；摘要中有 skip_reason=muted。
2. **日志**：出现「广告 xxx 处于 mute 期，跳过」类日志。

## 步骤 5：预算单位统一为「分」

### 5.1 改动的文件与函数

| 文件 | 改动类型 | 说明 |
|------|----------|------|
| `server/services/actionExecutorService.js` | 修改 | 所有预算变量与 MIN/MAX 护栏均为「分」；getAdsetBudget / updateAdsetBudget 的入参和返回值约定为「分」（若当前是美元则在此模块内 *100 再算，或改 API 封装层） |
| `server/db/schema.js` | 可选 | 若有 max_daily_budget 等字段，注明单位为「分」 |
| 规则配置/API | 约定 | rules.actions[].max_daily_budget 单位为「分」；前端展示时 /100 显示为「元」或美元 |
| 文档 | 新增 | 在 DEV_PLAN 或本说明中注明：后端预算一律「分」(int)，前端展示除 100 |

### 5.2 关键数据结构

- 内部：`currentBudgetCents`, `newBudgetCents`, `MIN_BUDGET_CENTS = 100`, `max_daily_budget` 存/传均为整数分。
- 与 FB API：若 FB 接口用美元，在封装层（如 FacebookMarketingAPI）做一次 *100 或 /100，对上游统一暴露「分」。

### 5.3 验收方式

1. **代码**：grep 预算相关变量，无「美元」或 0.01 等小数逻辑（或仅在与 FB 边界处换算）。
2. **接口/配置**：max_daily_budget 为整数；前端展示文档注明「除以 100 显示」。
