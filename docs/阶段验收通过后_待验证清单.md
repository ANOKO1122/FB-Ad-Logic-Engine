## 1. CBO + Dry Run：set_budget 的 newBudgetCents 非 0 且与计算一致

**目的**：确认 CBO 分支下 Dry Run 时，set_budget 会算出真实目标值并写入审计，不出现「目标=0 分」。

**步骤**：

1. 选一条 **只有系列预算、广告组无预算** 的广告（CBO），建一条规则：
   - 条件：spend > 0（today），能命中该广告即可
   - 动作：**设置预算为固定值**，填 **30** 美元（即 3000 分）
   - **勾选「仅模拟运行 (Dry Run)”**
2. 保存后点 **「运行此规则」**。
3. 打开 **后端控制台日志**，应看到类似：
   - `[Dry Run] 设置广告系列(CBO) xxx 预算 $30 → 3000 分`（**不应**是「不拉取、不下发」且 newBudgetCents=0）
4. **查库验证**：
   ```sql
   SELECT id, action_type, is_simulation, api_request, api_response, triggered_at
   FROM automation_logs
   WHERE action_type = 'SET_BUDGET' AND is_simulation = 1
   ORDER BY id DESC LIMIT 1;
   ```
5. **预期**：
   - `api_request` 的 JSON 里 `newBudgetCents` = **3000**（不为 0）
   - `api_response` 的 JSON 里 `newBudgetCents` = **3000**
   - 与 `computeNewBudgetCentsOnce(0, { type:'set_budget', value:30 })` 结果一致（3000）

**通过标准**：日志与 automation_logs 中 CBO + Dry Run + set_budget 的 newBudgetCents 均为 3000，无 0。

- [ ] 1.1 控制台日志出现「→ 3000 分」
- [ ] 1.2 automation_logs 最新一条 SET_BUDGET 的 api_request/api_response.newBudgetCents = 3000

## 2. ABO + 真实执行：increase/decrease（percent/usd）与 set_budget 各跑一条，FB 与日志一致

**目的**：确认真实下发时，ABO 广告组预算在 FB 后台真实变化，且 automation_logs 记录与之一致。

**前置**：选一条 **广告组有日预算** 的广告（ABO），并记下当前 FB 后台该广告组日预算（美元）。

### 2.1 increase_budget（percent 或 usd）真实执行

1. 建规则：条件能命中该广告；动作 **增加预算**（如 10% 或 +$5）；**不勾选** Dry Run。
2. 运行此规则。
3. **FB 后台**：该广告组日预算应增加（与 10% 或 +$5 一致）。
4. **查库**：
   ```sql
   SELECT id, action_type, is_simulation, status, api_request, api_response
   FROM automation_logs
   WHERE action_type IN ('INCREASE_BUDGET') AND is_simulation = 0
   ORDER BY id DESC LIMIT 1;
   ```
5. **预期**：`api_request` / `api_response` 中 `newBudgetCents` 与 FB 后台新预算（美元×100）一致，`budgetTarget` 为「广告组」。

- [ ] 2.1 FB 后台预算已增加
- [ ] 2.2 automation_logs 中 newBudgetCents、budgetTarget 正确

### 2.2 decrease_budget（percent 或 usd）真实执行

1. 建规则：动作 **减少预算**（如 10% 或 -$5）；不勾选 Dry Run。
2. 运行此规则。
3. **FB 后台**：该广告组日预算应减少；若低于 1 美元，应显示为 1 美元（MIN 护栏）。
4. **查库**：同上，`action_type = 'DECREASE_BUDGET'`，核对 newBudgetCents 与 FB 一致。

- [ ] 2.3 FB 后台预算已减少（且不低于 1 美元）
- [ ] 2.4 automation_logs 中 newBudgetCents、budgetTarget 正确

### 2.3 set_budget 真实执行

1. 建规则：动作 **设置预算为固定值**（如 $25）；不勾选 Dry Run。
2. 运行此规则。
3. **FB 后台**：该广告组日预算应变为 25 美元。
4. **查库**：`action_type = 'SET_BUDGET'`，`newBudgetCents` = 2500，`budgetTarget` =「广告组」。

- [ ] 2.5 FB 后台预算已变为设定值
- [ ] 2.6 automation_logs 中 newBudgetCents=2500、budgetTarget 正确

## 3. 上限/下限组合验证

**目的**：确认 capEffective、MIN 护栏在边界下的行为。

### 3.1 cap=0 等效为 MIN=100（执行层 capEffective）

**方式一（推荐）**：用单测已覆盖，可只做结果确认：
- `computeNewBudgetCentsOnce(0, { type:'set_budget', value:30, max_daily_budget:0 })` → 100
- `computeNewBudgetCentsOnce(1000, { type:'increase_budget', value:10, max_daily_budget:0 })` → 100  

**方式二（手工）**：若需真实请求验证，需绕过前端（如直接调 API 传 `max_daily_budget: 0`），执行后查 automation_logs 或 FB 后台，新预算应为 1 美元（100 分）。  
（前端已禁止传 0，正常保存不会出现 cap=0。）

- [ ] 3.1 单测中 cap=0 两条用例通过（或手工验证执行结果为 100 分/1 美元）

### 3.2 cap=1 美元（100 分）

1. 建规则：动作 **增加预算** 或 **设置预算**，预算上限填 **1** 美元。
2. 保存应成功（校验 mdb>=100 通过）。
3. 运行（Dry Run 或真实）：结果应为 **100 分**（1 美元）；api_request/api_response 中 `newBudgetCents` = 100。

- [ ] 3.2 预算上限 1 美元保存成功，执行结果 newBudgetCents=100

### 3.3 decrease 导致低于 MIN 时护栏生效

1. 选当前广告组预算较低的广告（或先设成 2 美元），建规则：动作 **减少预算** 如 -$5 或 -50%。

3. **预期**：计算结果若 < 100 分，最终应为 **100 分**（1 美元）；日志与 automation_logs 中 `newBudgetCents` = 100。

- [ ] 3.3 decrease 后 newBudgetCents 不低于 100
