# 顺序2 阶段 2.4：手动同步结构（强制同步该账户 ads 到 structure_ads）— 落地方案

> **约定**：本方案给出目标、变更点、接口设计与验证步骤；**具体代码与 SQL 由你在本地实现/执行**，Cursor 不代为运行。先方案再动手，每阶段验收后再进下一步。

---

## 一、本次目标

- **交付物**：新增后端接口「强制同步该账户的广告结构」：从 FB 拉取该账户下所有 ads（分页），写入 `structure_ads`（upsert），供后续 2.2 选择器查库使用。
- **约束**：**账户级锁**（同一账户同时只允许一个同步请求）+ **冷却期 2 分钟**（同一账户 2 分钟内不可重复触发），避免狂点与 API 滥用。
- **验收**：FB 新建/改名广告后，调用一次该接口，再查 `structure_ads` 或后续 2.2 选择器接口能搜到最新数据；未到冷却期再次调用返回 429 或明确提示冷却中。

---

## 二、接口设计

| 项目 | 说明 |
|------|------|
| 方法/路径 | `POST /api/sync/structure`（或 `POST /api/structure/sync`，与现有 `/api/structure/:level` 不冲突即可） |
| 鉴权 | `requireAuth` + `requireActive`；body 或 query 传 `account_id`，且需通过 `assertAccountAccess(req, res, account_id)`（非 admin 只能同步自己有权限的账户） |
| 请求体 | `{ "account_id": "act_xxx" }` 或 query `?account_id=act_xxx` |
| 成功响应 | `200`，body 如 `{ "ok": true, "account_id": "act_xxx", "synced_count": 123, "duration_ms": 1500 }` |
| 冷却中 | `429 Too Many Requests` 或 `200` 且 `{ "ok": false, "reason": "cooldown", "retry_after_sec": 90 }`，便于前端提示 |
| 锁占用 | 同一账户另一请求正在同步时，返回 `409 Conflict` 或 429，body 注明 `reason: "lock_busy"` |

## 四、核心实现要点

### 4.1 锁（与现有 ingestorService/cronService 一致）

- 使用 **同一连接**：`const conn = await pool.getConnection()`，在该连接上 `SELECT GET_LOCK('sync:structure:' + accountId, 0) AS acquired`，在 `finally` 中 `RELEASE_LOCK` 并 `conn.release()`。
- 锁名建议：`sync:structure:{accountId}`，避免与 `sync:today_stats` 等冲突。

### 4.2 冷却期

- 条件：`SELECT MAX(last_synced_at) AS last FROM structure_ads WHERE account_id = ?`
- 若 `last` 非 NULL 且 `(Date.now() - new Date(last).getTime()) < 120_000`（2 分钟），则返回冷却中，并可选在响应中带 `retry_after_sec`。

### 4.3 Upsert 语句示例（供你本地执行/实现时参考）

```sql
INSERT INTO structure_ads
  (account_id, ad_id, adset_id, campaign_id, name, effective_status, status, configured_status, updated_time, last_synced_at)
VALUES
  (?, ?, ?, ?, ?, ?, ?, ?, ?, NOW())
ON DUPLICATE KEY UPDATE
  adset_id = VALUES(adset_id),
  campaign_id = VALUES(campaign_id),
  name = VALUES(name),
  effective_status = VALUES(effective_status),
  status = VALUES(status),
  configured_status = VALUES(configured_status),
  updated_time = VALUES(updated_time),
  last_synced_at = NOW();
```

- 参数顺序与 `structure_ads` 列一致；`ad_id` 来自 FB 的 `id`。

### 4.4 FB 返回的 updated_time

- Graph API 的 ad 对象有 `updated_time`（ISO8601 字符串），可直接存入 `structure_ads.updated_time`（VARCHAR(80)）。若某条无该字段则写 NULL。

## 五、风险点与注意事项

1. **Token 与限流**：同步过程会连续调用 FB，建议每页之间保留短暂间隔（如 200–500ms），避免触发限流；错误处理与现有 `makeRequest` 一致（超时、重试、错误分级）。
2. **大账户**：广告数量很大时，全量分页可能较久，可考虑响应先返回 202 Accepted + 后台异步执行；当前最小闭环可采用同步执行，接口超时时间略放宽（如 60s），并在文档中说明「大账户可能较慢」。
3. **权限**：必须通过 `assertAccountAccess`，禁止用他人 `account_id` 同步。

## 六、验证步骤（请你亲自执行）

1. **实现并启动服务**：按上述变更点实现后，启动后端，确保无语法/启动错误。
2. **调用接口**：  
   `POST /api/sync/structure`，body `{ "account_id": "act_你的测试账户" }`，带鉴权 header。  
   - 期望：200，`synced_count` > 0，`structure_ads` 中该账户有对应行，`last_synced_at` 为刚才时间。
3. **冷却期**：2 分钟内再次用同一 `account_id` 调用。  
   - 期望：429 或 200 且 `reason: "cooldown"`，且未再次拉取 FB（可看日志或 DB 中 `last_synced_at` 未更新）。
4. **数据一致性**：在 FB 后台改某广告名称或新建一条广告，再次调用同步（等冷却过后），查 DB 或 2.2 列表接口（若已接）能见到最新名称/新广告。

## 七、与 2.2 的衔接

- 2.4 完成后，`structure_ads` 可由「手动刷新」填充；2.2 将 `GET /api/structure/ads` 改为查 `structure_ads`（limit + after 游标、默认过滤 DELETED/ARCHIVED）后，前端「刷新列表」调用本接口再拉列表即可得到最新数据。
- 后续可再做「定时同步结构」（如 30–60min 一次），与 TASKS 中「元数据同步（定时）」对齐。
