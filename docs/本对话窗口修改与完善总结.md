## 一、整体需求与背景

- **需求**：监控范围条件（状态=所有投放中/已暂停/投放中+已暂停，名称包含/不包含）通过**后端 SQL 过滤**只查匹配项，仅展示匹配项并全部勾选；多账户时拉全量匹配项，不重复、不漏数。
- **问题**：multi 接口带 `after` 换到下一账户时，若下一账户只有一页（`pas[i+1]==null`），仍返回游标 `{ i: i+1, a: null, pas }`，导致下一轮对该账户发 after=null、再拉第一页，产生重复（如界面 134 条、SQL 94 条）。
- **修复**：换到下一账户时只对「还有下一页」的账户（`pas[j]!=null`）返回 nextAfter；若后续均无下一页则不再返回 after。

以下为与该需求和修复相关的**全部代码与文档**，按文件列出。

### 2. `server/index.js`

**归属**：路由透传 scope 参数为**总结前已实现**；**multi 游标修复**为**本窗口修改**。

**修改点概览**：

| 位置 | 内容 | 归属 |
|------|------|------|
| **GET /api/structure/objects**（约 2202–2229 行） | 从 `req.query` 读取 `scope_status`、`name_exclude`，传入 `listStructureObjectsFromDb(..., { ..., scope_status, name_exclude })`。 | 总结前已实现 |
| **GET /api/structure/objects/multi**（约 2234–2348 行） | ① 从 query 读取 `scope_status`、`name_exclude`（约 2249–2250 行）；② **首轮**（无 after）：循环每账户调用 `listStructureObjectsFromDb(..., { ..., scope_status, name_exclude })`，合并去重，构建 `perAccountAfter`，若有更多则 `nextAfter = { i: firstI, a: perAccountAfter[firstI], pas }`（约 2325–2331 行）；③ **带 after**：解析 cursor 后对当前账户调用 `listStructureObjectsFromDb(..., { ..., scope_status, name_exclude })`（约 2285 行）；④ **带 after 时 nextAfter 计算**（约 2282–2308 行）：当前账户有下一页则 `nextAfter = { i, a: result.paging.after, pas }`；**当前账户无下一页且存在下一账户时**：原逻辑为固定 `nextAfter = { i: i+1, a: nextPas[i+1], pas }`（nextPas[i+1] 可能为 null）→ **本窗口改为**：从 j=i+1 起找第一个 `nextPas[j]!=null` 的 j，若存在则 `nextAfter = { i: j, a: nextPas[j], pas }`，否则 `nextAfter = undefined`。 | ①–③ 总结前；④ 本窗口 |
| **GET /api/structure/:level**（约 2372–2407 行） | 从 query 读取 `scope_status`、`name_exclude`，放入 `opts`，传入 `listStructureAdsFromDb` / campaigns / adsets（约 2386–2388、2394 等）。 | 总结前已实现 |

**本窗口具体改动（仅一处）**：  
在 multi 的「带 after」分支中，将「else if (i + 1 < allowedIds.length)」内的两行：

- `const nextA = (i + 1 < nextPas.length && nextPas[i + 1] != null) ? nextPas[i + 1] : null`  
- `nextAfter = Buffer.from(JSON.stringify({ i: i + 1, a: nextA, pas: nextPas })).toString('base64')`

替换为：循环查找第一个 `nextPas[j]!=null` 的 j；若 `nextI>=0` 则 `nextAfter = { i: nextI, a: nextPas[nextI], pas: nextPas }` 并 base64，否则 `nextAfter = undefined`。

| 位置 | 内容 |
|------|------|
| **ref 定义**（约 534–535 行） | `scopeStatusFilter = ref('')`（'' \| active_only \| paused_only \| active_and_paused）；`scopeNameExclude = ref('')`（名称不包含关键词）。 |

| **loadMoreScopeItems**（约 1180–1229 行） | 单账户/多账户请求均带上 `scope_status: scopeStatusFilter.value \|\| undefined`、`name_exclude: scopeNameExclude.value \|\| undefined`；多账户加载更多后若非「正在应用条件」则对当前列表执行 `applyScopeConditionsToCurrentList`。 |

| **applyScopeConditionsToCurrentList**（约 1395–1434 行） | 按当前 `scopeItems` 与已完成条件行做 match，写回 `ruleForm.targetIds`，并设置 `scopeApplyMessage`（已勾选 N 个 / 当前已加载条数等）。 |

**说明**：不再依赖「先拉全量再前端过滤」；有名称/状态条件时一律带 scope 参数请求后端，并循环加载更多直到无 after，实现全量匹配项并全部勾选。
