# 2.3 条件组 DNF — 验证教学（最小集成验收）

配合《2.3_条件组_DNF_落地方案.md》，按 **4 步** 完成最小集成验收。  
优先用**前端**，部分场景需 **Thunder Client**（v2 / 非法 payload）。

**Thunder Client 集合**：`docs/Thunder_Client_DNF验证集合.json`  
导入后可依次执行验证 3 / 4 / 1 / 2，记得先登录并替换 `act_xxx`、`ruleId`。

---

## 前置准备（必做）

| 项目 | 说明 |
|------|------|
| 后端 | `npm run dev:server` 或 `node server/server.js`，端口 3000 |
| 前端 | `npm run dev`，确保能访问规则管理页 |
| 登录 | 用有权限的用户登录（管理员或账户负责人） |
| 数据 | 规则作用的账户下，`ad_snapshots` 表有数据；否则 `matched_count` 恒为 0 |

**检查 ad_snapshots 是否有数据：**

```sql
SELECT account_id, COUNT(*) AS cnt
FROM ad_snapshots
GROUP BY account_id
HAVING cnt > 0
LIMIT 5;
```

若为空，先跑数据同步（拉广告快照），否则验证 1、2 的 `matched_count` 可能为 0。

## 验证 1：v1 OR 规则（cron / Drizzle 字段名兼容）

**目标**：确认 `rule.logicOperator`（驼峰）在真实执行链路生效，v1 OR 不被误算成 AND。

### 方式 A：前端（推荐）

1. 打开规则管理页 → **新建规则**
2. **基本信息**：规则名称填 `验证1-v1 OR`
3. **作用对象**：选一个有 `ad_snapshots` 的账户；目标层级选「广告」，选若干广告或「全选」
4. **触发条件**：
   - 逻辑关系选 **「满足任一条件 (OR)」**
   - 条件 1：`花费 > 999999`，时间窗口「今天」（几乎不可能满足）
   - 条件 2：`CPC > 0.01`，时间窗口「今天」（容易满足）
5. **执行操作**：选 `pause_ad` 或 `activate_ad`
6. **模拟运行**：建议先开 Dry Run
7. 保存 → 在规则卡片上点 **「运行此规则」**
8. **验收**：
   - 弹窗或提示中有 `matched_count >= 1`
   - 若 `matched_count = 0`，检查账户下是否有 `ad_snapshots`，或把条件 2 改成 `CPC > 0` 再试

### 方式 B：Thunder Client

1. **登录**：`POST {{baseUrl}}/api/auth/login`  
   Body: `{"username":"你的用户名","password":"你的密码"}`  
   记录响应里的 `token`

2. **创建 v1 OR 规则**：`POST {{baseUrl}}/api/rules`  
   Headers: `Authorization: Bearer {{token}}`, `Content-Type: application/json`  
   Body（把 `act_xxx` 换成你有权限的账户 ID）：

   ```json
   {
     "ruleName": "验证1-v1 OR",
     "accountId": "act_xxx",
     "conditions": [
       { "metric": "spend", "operator": "gt", "value": 999999, "time_window": "today" },
       { "metric": "cpc", "operator": "gt", "value": 0.01, "time_window": "today" }
     ],
     "logicOperator": "OR",
     "actions": [{ "type": "pause_ad" }]
   }
   ```

3. **执行**：`POST {{baseUrl}}/api/rules/{{ruleId}}/execute`  
   Headers: `Authorization: Bearer {{token}}`  
   把 `{{ruleId}}` 换成上一步返回的 `rule.id`

4. **验收**：响应中 `matched_count >= 1`

## 验证 2：v2 DNF 规则（(A AND B) OR (C)）

**目标**：确认 DNF 语义，任意一组满足即命中；两组都不满足则不命中。

### 说明

前端当前仅支持 v1 数组，v2 需用 Thunder Client 构造 payload。

### Thunder Client 步骤

1. **创建 v2 DNF 规则**：`POST {{baseUrl}}/api/rules`  
   Headers: `Authorization: Bearer {{token}}`, `Content-Type: application/json`  
   Body（把 `act_xxx` 换成你有权限的账户 ID）：

   ```json
   {
     "ruleName": "验证2-v2 DNF",
     "accountId": "act_xxx",
     "conditions": {
       "version": 2,
       "groups": [
         {
           "operator": "AND",
           "conditions": [
             { "metric": "spend", "operator": "gt", "value": 0.8, "time_window": "today" },
             { "metric": "link_clicks", "operator": "eq", "value": 0, "time_window": "today" }
           ]
         },
         {
           "operator": "AND",
           "conditions": [
             { "metric": "cpc", "operator": "gt", "value": 0.8, "time_window": "today" }
           ]
         }
       ]
     },
     "actions": [{ "type": "pause_ad" }]
   }
   ```

   - 组 1：`spend > 0.8 AND link_clicks = 0`（空耗）
   - 组 2：`cpc > 0.8`

2. **执行**：`POST {{baseUrl}}/api/rules/{{ruleId}}/execute`

3. **验收**：
   - 若账户内有 `spend>0.8 且 link_clicks=0` 或 `cpc>0.8` 的广告 → `matched_count >= 1`
   - 若都没有 → `matched_count = 0`（两组都不满足则不命中）

## 验证 3：API 拦截 v1 time_window 不一致（400）

**目标**：同一规则内 time_window 必须一致，否则 POST/PUT 返回 400，`code=INCONSISTENT_TIME_WINDOW`。

### 方式 A：前端

1. 新建规则
2. **触发条件**：
   - 条件 1：`花费 > 0`，时间窗口选 **「今天」**
   - 条件 2：`CPC > 0`，时间窗口选 **「近3天」**
3. 保存
4. **验收**：应提示 400，错误信息含「time_window 须一致」或 `INCONSISTENT_TIME_WINDOW`

### 方式 B：Thunder Client

`POST {{baseUrl}}/api/rules`  
Body:

```json
{
  "ruleName": "验证3-time_window不一致",
  "accountId": "act_xxx",
  "conditions": [
    { "metric": "spend", "operator": "gt", "value": 0, "time_window": "today" },
    { "metric": "cpc", "operator": "gt", "value": 0, "time_window": "last_3_days" }
  ],
  "logicOperator": "AND",
  "actions": [{ "type": "pause_ad" }]
}
```

**验收**：Status 400，Body 含 `code: "INCONSISTENT_TIME_WINDOW"` 或 `error` 含「time_window 须一致」。

## 验证 4：API 拦截 v2 空 conditions（400）

**目标**：v2 的 `group.conditions` 不能为空数组，否则 400，`code=INVALID_CONDITIONS`。

### Thunder Client（前端不支持 v2）

`POST {{baseUrl}}/api/rules`  
Body:

```json
{
  "ruleName": "验证4-v2空组",
  "accountId": "act_xxx",
  "conditions": {
    "version": 2,
    "groups": [
      { "operator": "AND", "conditions": [] }
    ]
  },
  "actions": [{ "type": "pause_ad" }]
}
```

**验收**：Status 400，Body 含 `code: "INVALID_CONDITIONS"` 或 `error` 含「非空 conditions」。

## 验证顺序建议

| 顺序 | 验证项 | 工具 | 耗时 |
|------|--------|------|------|
| 1 | v1 time_window 不一致（400） | 前端 / Thunder | ~1 分钟 |
| 2 | v2 空 conditions（400） | Thunder | ~1 分钟 |
| 3 | v1 OR 规则 | 前端 | ~3 分钟 |
| 4 | v2 DNF 规则 | Thunder | ~3 分钟 |

先做 3、4 可快速确认拦截逻辑；再做 1、2 验证执行链路。

---

## 常见问题

| 问题 | 可能原因 | 处理 |
|------|----------|------|
| `matched_count` 恒为 0 | 账户下无 `ad_snapshots` | 先做数据同步，或换有数据的账户 |
| 执行返回 409 | 有其他规则正在执行 | 等几秒再试 |
| 401 / 403 | 未登录或无权访问该账户 | 检查 token、账户归属 |
| Thunder 提示 CORS | 开发环境一般已处理 | 确认后端 `cors` 配置包含前端 origin |
