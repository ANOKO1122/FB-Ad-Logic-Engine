# 系统时区逻辑说明

## 1. 关于「Etc/GMT-8」——没有错

你在 Node 里看到的：

```text
Intl.DateTimeFormat().resolvedOptions().timeZone  →  'Etc/GMT-8'
new Date().toString()                             →  'Wed Jan 28 2026 18:11:38 GMT+0800 (GMT+08:00)'
```

**结论：进程时区没有错，Etc/GMT-8 就是 UTC+8（中国）。**

原因：在 IANA 时区库里，**Etc 命名空间里符号是反的**：

| IANA 名称   | 实际含义 | 常见地区   |
|------------|----------|------------|
| Etc/GMT+8  | **UTC-8** | 美西冬令时 |
| Etc/GMT-8  | **UTC+8** | 中国       |
| Etc/GMT-1  | UTC+1     | 西欧冬令时 |

所以 `Etc/GMT-8` = UTC+8 = 中国，和 `GMT+0800` 一致，**不存在“进程时区错误”**。

---

## 2. 系统时区原则（三句话）

- **存**：数据库里统一用 **UTC**（`time_zone = '+00:00'`）。
- **算**：规则、时间窗口、数据日期等，按 **账户时区**（Luxon + IANA，如 `Asia/Shanghai`）算。
- **看**：前端展示统一转换成 **UTC+8（北京时区 / Asia/Shanghai）**。

---

## 3. 各层时区怎么用

### 3.1 数据库层（存 UTC）

| 文件 / 位置 | 作用 |
|-------------|------|
| `server/db/connection.js` | 每次执行前 `SET time_zone = '+00:00'`，会话用 UTC。 |
| `server/db/drizzle.js` | 连接配置 `timezone: '+00:00'`，建连后也设 UTC。 |

效果：

- `synced_at`、`triggered_at`、`evaluated_at`、`created_at` 等 **写入/读出都按 UTC**。
- 不做“按本地时区存”，避免歧义。

### 3.2 规则与数据查询（Luxon + 账户时区）

| 文件 | 作用 |
|------|------|
| `server/utils/timeWindow.js` | 用 Luxon `DateTime.now().setZone(timezoneName)` 算「今天 / 昨天 / 过去 N 天」的起止时间；`timezoneName` 来自 **账户配置**（如 `Asia/Shanghai`）。 |
| `server/services/ruleDataService.js` | 取账户时区 → 调用 `calculateTimeWindow(timeWindow, timezoneName)` → 得到在 **账户时区** 下的自然日窗口，再查 DB。 |
| `server/services/ingestorService.js` | 同步时用 `DateTime.now().setZone(timezoneName)` 算“今天”的日期范围；`data_date`、按日归档等都按 **账户时区**。 |

要点：

- **不用** Node 的“系统时区”或 `Etc/GMT-8` 来算业务日期。
- **只用** 从配置/DB 来的 `timezoneName`（如 `Asia/Shanghai`）传给 Luxon，所以即使用户在别的时区，规则和同步逻辑也按账户时区算，**不会因为进程是 Etc/GMT-8 而错**。

### 3.3 账户时区从哪里来

| 来源 | 说明 |
|------|------|
| `account_mappings.timezone_name` | 主来源，可从 FB API 拉取并更新（如 `Asia/Shanghai`）。 |
| 规则表 `rules.timezone_name` | 规则级时区，用于该规则的时间窗口计算。 |
| 默认 | 未配置时用 `'UTC'`。 |

### 3.4 规则执行摘要时间（evaluated_at）

| 文件 | 作用 |
|------|------|
| `server/services/ruleExecutionSummaryService.js` | 写入 `evaluated_at` 时用 **`new Date()` 的 UTC 分量** 拼成 `YYYY-MM-DD HH:mm:ss` 字符串再写入，不依赖进程时区，保证库里是 **UTC**。 |

### 3.5 前端展示时区（统一 UTC+8）

| 位置 | 要求 |
|------|------|
| 前端所有时间展示 | **统一转换成 UTC+8（北京时区 / Asia/Shanghai）**，不管用户浏览器在哪个时区。 |
| 实现方式 | 前端收到 UTC 时间后，用 `luxon` 或 `dayjs` 等库，统一转成 `Asia/Shanghai` 时区显示。 |
| 控制台/日志 | 后端控制台和日志可以按系统本地时区显示（用于开发调试）。 |

**示例**（前端代码）：
```javascript
// 后端返回的 UTC 时间：'2026-01-29 02:02:15'
// 前端统一转成 UTC+8 显示
import { DateTime } from 'luxon'
const utcTime = '2026-01-29 02:02:15'
const beijingTime = DateTime.fromSQL(utcTime, { zone: 'UTC' })
  .setZone('Asia/Shanghai')
  .toFormat('yyyy-MM-dd HH:mm:ss')
// 显示：'2026-01-29 10:02:15'（UTC+8）
```

## 4. 数据流简图

```text
┌─────────────────────────────────────────────────────────────────┐
│  前端                                                           │
│  - 展示统一转换成 UTC+8（北京时区 / Asia/Shanghai）              │
│  - 控制台/日志：按系统本地时区显示（如 GMT+0800 / Etc/GMT-8）     │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  后端（Node）                                                     │
│  - 存库时间：一律 UTC（connection/drizzle 设 time_zone='+00:00'）│
│  - 规则/同步：Luxon + 账户 timezoneName（Asia/Shanghai 等）       │
│  - 不依赖 Intl 返回的 Etc/GMT-8 做业务计算                        │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  MySQL                                                           │
│  - 会话 time_zone = '+00:00'                                      │
│  - TIMESTAMP 存、取都是 UTC                                        │
└─────────────────────────────────────────────────────────────────┘
```

## 5. 小结

- **Etc/GMT-8**：就是 UTC+8，进程时区没错。
- **系统时区逻辑**：  
  - 存：UTC；  
  - 算：Luxon + 账户/规则时区（IANA 名）；  
  - 看：前端统一 UTC+8（北京时区），控制台/日志用系统本地时区。
- **没有用“进程时区”做业务计算**，只用在控制台/日志的本地时间显示；业务日期、规则执行摘要时间都按 UTC 存、按账户时区或显式 UTC 处理，因此**系统时区逻辑是正确的**。
