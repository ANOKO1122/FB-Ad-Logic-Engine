## 三、具体修改清单

### A. 状态「不等于」scope_status_exclude

#### A.1 后端：structureSyncService.js 三层 list 支持排除状态

- **文件**：`server/services/structureSyncService.js`
- **方法**：`listStructureAdsFromDb`、`listStructureCampaignsFromDb`、`listStructureAdsetsFromDb`
- **逻辑**：
  - 在「根据 scope_status / include_paused / q 得到 statusList」之后，读取 `opts.scope_status_exclude`（可选，逗号分隔，如 `'PAUSED'` 或 `'ACTIVE,PAUSED'`）。
  - 解析为数组并去空：`excludeList = (opts.scope_status_exclude || '').split(',').map(s => s.trim()).filter(Boolean)`，`excludeSet = new Set(excludeList)`。
  - `statusList = statusList.filter(s => !excludeSet.has(s))`。
  - 若 `statusList.length === 0`，直接返回 `{ items: [], paging: { after: null } }`。
  - JSDoc 的 `@param opts` 中增加 `scope_status_exclude` 说明。
- **listStructureObjectsFromDb**：将 `opts.scope_status_exclude` 传入三个 list 的 opts。

#### A.2 后端：路由透传 scope_status_exclude

- **文件**：`server/index.js`
- **位置**：
  1. **GET /api/structure/objects**（单账户，约 L2208 附近）：从 `req.query.scope_status_exclude` 读取，传入 `listStructureObjectsFromDb` 的 opts。
  2. **GET /api/structure/objects/multi**（约 L2248、L2285、L2335）：从 `req.query.scope_status_exclude` 读取，在「无 after」首轮与「带 after」分支中均传入 `listStructureObjectsFromDb`。
  3. **GET /api/structure/:level**（约 L2393）：从 `req.query.scope_status_exclude` 读取，并入 opts 传给下层 list。

#### A.3 前端：状态行「不等于」映射与传参

- **文件**：`src/views/RuleManager.vue`
- **新增**：`scopeStatusExcludeFilter`（ref，字符串，如 `''` 或 `'ACTIVE'` 或 `'ACTIVE,PAUSED'`）。
- **applyScopeConditions**：当监控范围「状态」行的 operator 为「不等于」时：
  - 根据 row.value：`active_only` → `'ACTIVE'`，`paused_only` → `'PAUSED'`，`active_and_paused` → `'ACTIVE,PAUSED'`，写入 `scopeStatusExcludeFilter`，并清空 `scopeStatusFilter`（等于与不等于互斥）。
  - 当 operator 为「等于」时，维持现有 `scopeStatusFilter` 逻辑，清空 `scopeStatusExcludeFilter`。
- **refreshScopeItems / loadMoreScopeItems**：在调用 `getStructureObjects` / `getStructureObjectsMulti` 时，增加参数 `scope_status_exclude: scopeStatusExcludeFilter.value || undefined`（单账户与多账户两处都要传）。

### B. 多账户账户数上限（管理员放宽）

- **文件**：`server/index.js`
- **位置**：GET /api/structure/objects/multi，在得到 `allowedIds` 之后、`if (afterRaw)` 之前。
- **逻辑**：
  - 定义常量 `MULTI_ACCOUNT_IDS_MAX = 20`，`MULTI_ACCOUNT_IDS_MAX_ADMIN = 50`。
  - `const isAdmin = req.user && req.user.role === 'admin'`。
  - `const maxAccounts = isAdmin ? MULTI_ACCOUNT_IDS_MAX_ADMIN : MULTI_ACCOUNT_IDS_MAX`。
  - 若 `allowedIds.length > maxAccounts`，则 `return res.status(400).json({ error: '单次请求最多选择 ' + maxAccounts + ' 个广告账户', code: 'TOO_MANY_ACCOUNTS' })`。

### C. multi 首轮 p-limit(4) 并发

- **文件**：`server/index.js`
- **位置**：GET /api/structure/objects/multi，首轮分支（无 `afterRaw`），当前 `for (const accountId of allowedIds)` 循环。
- **逻辑**：
  - 顶部或该路由内引入：`import pLimit from 'p-limit'`（若为 CJS 则 `const pLimit = (await import('p-limit')).default`）；`const pl = pLimit(4)`。
  - 使用 `Promise.all(allowedIds.map((accountId, idx) => pl(() => listStructureObjectsFromDb(accountId, { type, q, limit: perAccountLimit, after: null, include_paused: includePausedEffective, scope_status, name_exclude, scope_status_exclude }))))`，得到按 idx 顺序的 `results`。
  - 按 `results[idx]` 顺序：`perAccountAfter[idx] = results[idx].paging?.after ?? null`，合并 `allItems`（去重、slice(0, limit)），生成 `nextAfter` 与 `perAccountAfterMeta` 的逻辑与当前一致（第一个有下一页的账户的游标）。
- **依赖**：package.json 已有 p-limit；index 若为 ESM 用 import，CJS 用 require 或 dynamic import。

| 验收项 | 操作 / 预期 |
