## 二、同步任务与请求字段

### 1. 今日快照（ad_snapshots）— 主同步

- **入口**：`ingestorService.js` 中拉取「today」insights 并写入 ad_snapshots 的流程（如 `fetchInsightsInBatches(..., 'today')` → `saveSnapshotsToDb` / 队列写入）。
- **请求字段**（约 1475 行）：
  ```text
  ad_id, ad_name, adset_id, spend, actions, action_values, unique_actions,
  cost_per_action_type, cost_per_unique_link_click, cost_per_unique_inline_link_click,
  inline_link_clicks, unique_inline_link_clicks, purchase_roas, website_purchase_roas
  ```
- **关键**：已包含 **`actions`**。加购/结账/添加支付信息次数都是从 **`actions`** 里用 `extractActionCount` 解析得到：
  - 加购次数：`extractActionCount(actions, ['offsite_conversion.fb_pixel_add_to_cart', 'add_to_cart'])`
  - 结账次数：`extractActionCount(actions, ['offsite_conversion.fb_pixel_initiate_checkout', 'initiate_checkout'])`
  - 添加支付信息次数：`extractActionCount(actions, ['offsite_conversion.fb_pixel_add_payment_info', 'add_payment_info'])`
- **写入**：`saveSnapshotsToDbInternal`（约 1804–1831 行）INSERT/UPDATE ad_snapshots 时包含 `add_to_cart_count`, `initiate_checkout_count`, `add_payment_info_count`。

### 2. 按日历史（daily_stats）— 归档

- **入口**：`archiveDailyStats` 等：从 ad_snapshots（或 API 按日）取数写入 daily_stats。
- **来源**：读 ad_snapshots 时已含 `add_to_cart_count`, `initiate_checkout_count`, `add_payment_info_count`（约 2203–2205、2315–2317、2331、2344–2346 行），写入 daily_stats 时一并写入这三列。

### 3. 规则数据（ruleDataService）

- **读**：从 ad_snapshots / daily_stats 查数时 SELECT 了这三列（如 ruleDataService.js 362–364、416–418、475–477、760–762、804–806 等），并暴露给规则引擎。

## 三、为何查出来全是 0？

可能两种情况：

1. **API 未返回这些事件**  
   Facebook Insights 的 `actions` 里只有账户实际发生的转化。若账户没有配置对应像素事件、或该广告在该日没有加购/结账/添加支付信息，则不会有对应 `action_type`，`extractActionCount` 会返回 0，写入库也是 0。  
   → **属于“有收集、但数据为 0”**。

2. **JOIN 未命中快照**  
   若「执行日志」与「ad_snapshots」的日期口径不一致（例如 triggered_at 用 UTC、data_date 用账户时区），LEFT JOIN 可能匹配不到行，用 COALESCE(..., 0) 后也会显示为 0。  
   → 可通过下面验证 SQL 看是否有该日、该广告的快照行。

## 四、验证 SQL（是否真的在收这 3 个次数）

在库里执行下面语句，看**该账户**是否出现过**非 0** 的加购/结账/添加支付信息次数：

```sql
-- 该账户下，是否有任意一条快照的加购/结账/添加支付信息次数非 0
SELECT
  COUNT(*) AS total_rows,
  SUM(CASE WHEN COALESCE(add_to_cart_count, 0) > 0 THEN 1 ELSE 0 END) AS rows_with_add_to_cart,
  SUM(CASE WHEN COALESCE(initiate_checkout_count, 0) > 0 THEN 1 ELSE 0 END) AS rows_with_checkout,
  SUM(CASE WHEN COALESCE(add_payment_info_count, 0) > 0 THEN 1 ELSE 0 END) AS rows_with_payment_info
FROM ad_snapshots
WHERE account_id = 'act_1155891202142879'
  AND data_date >= '2026-02-01';
```

- 若 `rows_with_add_to_cart` / `rows_with_checkout` / `rows_with_payment_info` 有大于 0：说明**有在收集且曾出现过非 0**，那 5 条日志为 0 多半是那几条广告当天确实没有这些转化。
- 若全部为 0：可能是该账户近期没有任何加购/结账/添加支付信息事件（像素或投放导致），而不是“没收集”。

再查那 5 个广告在 2026-02-27 是否有快照（确认 JOIN 是否有命中）：

```sql
SELECT ad_id, data_date, add_to_cart_count, initiate_checkout_count, add_payment_info_count, synced_at
FROM ad_snapshots
WHERE account_id = 'act_1155891202142879'
  AND ad_id IN ('120242100873060501','120242100873100501','120242100873070501','120242100873090501','120242100873080501')
  AND data_date = '2026-02-27'
ORDER BY ad_id, synced_at DESC;
```

若这里没有行，说明「执行日志」那条 SQL 的 JOIN 条件（如 `data_date = DATE(triggered_at)`）与 ad_snapshots 的 data_date 口径不一致，需要按实际时区或存储方式调整 JOIN。

## 五、结论

- **12 个指标在同步链路上都有请求和写入**：今日快照请求了 `actions` 等字段，并解析出 add_to_cart_count、initiate_checkout_count、add_payment_info_count 写入 ad_snapshots，再被 daily_stats 和 ruleDataService 使用。
- 若库里这三项多为 0，优先考虑：**Facebook 侧未返回对应事件**（无像素/无转化），而不是“没收集”。用上面验证 SQL 看全账户是否有非 0 行、以及那 5 个广告 2 月 27 日是否有快照，即可进一步确认。
