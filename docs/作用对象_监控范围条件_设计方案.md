# 作用对象 · 监控范围条件 — 设计方案

> 在「2. 作用对象 (Where)」中新增「监控范围条件」区块：多行动态条件行（名称 包含/不包含、状态 等于），AND 连接，支持添加/删除与数量上限；配置完成后根据所有 AND 条件自动勾选下方「选择目标对象」列表中匹配的项。  
> 本文档为设计方案，不涉及直接改代码；实现前需评审确认。

---

## 一、目标与范围

- **位置**：规则编辑弹窗 → **2. 作用对象 (Where)** 内，插入在「目标层级」与「选择目标对象（可多选）」之间。
- **作用**：用户通过多行「字段 + 操作符 + 值」配置**监控范围条件**（仅用于筛选要勾选的对象，不写入规则触发条件）；系统根据**所有已填写条件的 AND 结果**，自动在下方「选择目标对象」列表中勾选匹配的广告/广告组/广告系列。
- **与现有模块的关系**：
  - **目标层级**：不变；列表仍按「广告 / 广告组 / 广告系列」加载。
  - **选择目标对象**：列表由「搜索名称或ID」「状态：仅启用/包含暂停」加载；**应用监控范围条件时**，会按条件中的「名称」关键词自动拉取**全量匹配结果**（自动多次「加载更多」直到拿完），列表展示全部匹配项并自动勾选；用户之后可再搜索、手动追加勾选。
  - **3. 触发条件 (When)**：不变；仍为 whenLines（花费、ROAS 等监控指标），与本「监控范围条件」无关。

## 二、数据结构

### 2.1 条件行对象 (ConditionRow)

建议使用**可扩展**结构，便于后续增加字段（如广告创建时间）：

```ts
// 单行条件
interface ScopeConditionRow {
  id: string              // 唯一键，用于 :key 与删除，如 nanoid 或 'scope-' + index
  field: 'name' | 'status' // 字段：名称 | 状态（后续可扩展 'created_time' 等）
  operator: string        // 操作符，见下表
  value: string           // 值：名称时为文本；状态时为选项 key
}
```

### 2.2 字段 × 操作符 × 值 矩阵

| 字段   | 操作符（展示名） | 值类型     | 值选项 / 说明 |
|--------|------------------|------------|----------------|
| 名称   | 包含             | 文本输入   | 占位符：请输入名称 (例: 优化师_项目) |
| 名称   | 不包含           | 文本输入   | 同上 |
| 状态   | 等于             | 下拉选择   | 所有投放中、所有已暂停、所有投放中和已暂停 |

**操作符与值约定（实现用）：**

- 名称：`operator in ['include','exclude']`，`value` 为字符串（trim 后参与匹配，建议不区分大小写）。
- 状态：`operator === 'equals'`，`value` 为以下之一：
  - `active_only` → 所有投放中（effective_status === 'ACTIVE'）
  - `paused_only` → 所有已暂停（effective_status === 'PAUSED'）
  - `active_and_paused` → 所有投放中和已暂停（ACTIVE 或 PAUSED）

### 2.3 存储与持久化

- **仅前端状态**：`scopeConditionRows: Ref<ScopeConditionRow[]>`，**不写入规则 API**（不新增后端字段）。  
- **保存规则时**：只提交现有的 `targetIds` / `target_by_account`；勾选结果已体现在 `targetIds` 中，无需单独存「条件行」。
- **打开编辑**：不回填「监控范围条件」行（或可选：本地记住上次使用的行数/默认一行，不按规则反序列化）。

## 三、UI 结构

### 3.1 区块标题与顺序

在「2. 作用对象 (Where)」内顺序为：

1. 广告账户（可多选）  
2. 目标层级  
3. **监控范围条件（新增）** — 见下  
4. 选择目标对象（可多选）— 保持现有

### 3.2 监控范围条件 UI

- **标题**：如「监控范围条件（可选）」+ 短说明：「根据下方条件自动勾选匹配的对象；不填则需手动勾选。」
- **左侧逻辑连接**：
  - **第一行**：左侧不显示逻辑符（占位保持对齐，如空 span 或占位 class）。
  - **第二行起**：每行左侧显示 **「AND」** 标识（纯文字或带浅色竖线，与「3. 触发条件」的 when-line 风格一致）。
- **条件行（Row）**：每行由三部分 + 右侧操作组成，**Flex 布局**，统一 gap（如 8px）、行内 padding，避免拥挤：
  - **字段选择（Select）**：下拉，选项「名称」「状态」。
  - **操作符（Select）**：根据字段变化；名称 →「包含」「不包含」；状态 →「等于」。
  - **值**：名称 → 文本输入框（placeholder：请输入名称 (例: 优化师_项目)）；状态 → 下拉（所有投放中 / 所有已暂停 / 所有投放中和已暂停）。
  - **右侧**：「×」按钮，删除当前行。
- **删除规则**：仅剩一行时，不显示「×」或点击删除后保留一行并重置为该行默认状态（字段默认「名称」、操作符默认「包含」、值为空）。
- **添加**：底部绿色文字链接「添加监控范围 (n/M)」，如 `(1/3)`、`(2/3)`。  
  - 点击后新增一行默认条件（field: 'name', operator: 'include', value: ''），新行左侧为 AND。  
  - **数量上限 M**：可配置（如 3）；达到 M 时禁用「添加」链接并可选灰显。
- **校验与错误提示**：
  - 若某行「值」为空（名称未输入、或状态未选择），在该**行下方**显示红色文案「数据不能为空」。
  - 可对整行或仅对「值」控件加红色描边（与现有「3. 触发条件」错误样式一致）。

### 3.3 布局与扩展性

- 使用 **Flex**：行内 `display: flex; align-items: center; gap: 8px;`；行与行之间统一 `margin-bottom` 或 `gap`。
- **字段选择**：用数组配置 `FIELD_OPTIONS = [{ value:'name', label:'名称' }, { value:'status', label:'状态' }]`，后续加「广告创建时间」等只需扩展该数组与对应的 operator/value 配置，便于维护。

## 四、联动自动勾选逻辑（含「按条件拉全量并展示」）

### 4.1 期望效果（产品口径）

- 用户添加监控条件，例如「名称 包含 0224」；实际有 60 个广告符合条件，但接口一次只返回 50 个（需点「加载更多」才再拿 50 个）。
- **期望**：应用条件后，列表**展示全部 60 个**匹配项，并**自动勾选这 60 个**；即系统要按条件**自动拉全量**（自动多次「加载更多」直到拿完），再在列表里展示并勾选。
- 之后用户再在搜索框输入别的关键词、找到某个广告并**手动勾选 1 个**，则最终勾选数为 **61**（60 来自条件 + 1 手动添加）。即：应用条件时用匹配结果**替换**当前勾选；之后的手动勾选/取消为**在现有勾选基础上增减**。

### 4.2 触发时机

- 当用户**完成任意一行的配置**（字段、操作符、值均有效）并触发「应用」时（如点击「应用条件」按钮，或失焦/防抖后自动应用），执行「按条件拉全量 → 展示 → 自动勾选」流程。  
- 建议在以下时机触发（可配合防抖）：
  - 某行 `field` / `operator` / `value` 变更后用户确认（如 blur 或按钮）；
  - 删除行后若仍存在有效行，可重新应用。
- **不**在「列表数据变化」时自动重算，避免与用户后续「搜索 + 手动添加」冲突；应用条件 = 明确一次「用条件结果替换勾选并拉全量展示」。

### 4.3 按条件拉全量并展示（实现要点）

- **名称类条件驱动拉取**：若存在已填写的「名称 包含 / 不包含 X」，用 **X 作为搜索关键词** 请求列表（即请求时 `q = X`），与当前「状态：仅启用/包含暂停」一致。
- **自动拉全量**：首请求 `limit=50`（或后端允许的单页上限）；若返回 `paging.after`，则**自动循环请求「加载更多」**（同一 `q`、同一 `include_paused`），将每页结果追加到同一列表，直到无 `after`。最终得到「所有名称匹配的项」（如 60 条）。
- **列表展示**：该列表直接作为「选择目标对象」的当前数据源展示（即此时列表 = 按条件拉到的全量），用户能看到 60 条并全部被勾选。
- **状态类条件**：在拉到的全量上做**前端过滤**。例如还有「状态 等于 所有投放中」，则只保留 `effective_status === 'ACTIVE'` 的项，再写入勾选并展示（若 60 条里 55 条 ACTIVE，则展示 55 条、勾选 55 条）。若只有「名称 包含 0224」无状态条件，则 60 条全部展示并勾选。
- **勾选结果**：将 `ruleForm.targetIds` **替换**为上述匹配结果的 id 集合；id 格式与现有一致（多账户用 `scopeItemValue(item)`，单账户用 `item.id`）。之后用户再搜索、手动勾选 1 个，targetIds 会多 1 个 → 共 61 个。

### 4.4 匹配规则（与现有数据一致）

- **名称 + 包含**：`item.name` 不区分大小写包含 `value`（trim）；用于驱动接口 `q` 时，用 `value` 作为 `q` 拉全量。
- **名称 + 不包含**：`item.name` 不区分大小写**不**包含 `value`（trim）。注意：后端若不支持「不包含」搜索，则需用「包含」拉回后再在前端过滤掉名称包含 value 的项，或后端扩展支持。
- **状态 + 等于**：  
  - `active_only` → `item.effective_status === 'ACTIVE'`  
  - `paused_only` → `item.effective_status === 'PAUSED'`  
  - `active_and_paused` → `['ACTIVE','PAUSED'].includes(item.effective_status)`  
  仅在**前端**对已拉取列表过滤，不改变请求参数。
- **多行**：仅**已填写完整**的行参与；多行之间为 **AND**。若没有任何一行完整，不执行拉全量也不改勾选。

### 4.5 与「状态：仅启用/包含暂停」的关系

- 列表请求的 **include_paused** 仍由「状态：仅启用 / 包含暂停」决定（只拉 ACTIVE 或 ACTIVE+PAUSED）。
- 「监控范围条件」里的**状态 等于**在拉到的全量上再做一次前端过滤，缩小最终展示与勾选集合。

## 五、技术实现要点（供实现参考）

### 5.1 与现有 RuleManager 的对接

- **插入位置**：在 `RuleManager.vue` 的「目标层级」与「选择目标对象」之间新增一块模板，绑定 `scopeConditionRows` 与新增方法（添加行、删除行、校验、**应用条件**）。
- **应用条件**：需实现「按条件拉全量」：用名称条件的 value 作为 `q` 调用现有列表接口，循环带 `after` 的「加载更多」直到无下一页，将结果合并进 `scopeItems`（或临时列表再赋给 scopeItems），再按状态等条件前端过滤后写回 `ruleForm.targetIds`。可选提供「应用条件」按钮，或失焦/防抖后自动应用。
- **复用**：`scopeItems`、`scopeItemValue(item)`、`ruleForm.targetIds`、现有 `refreshScopeItems` / `loadMoreScopeItems` 的请求封装均可复用；新增「拉全量循环」与「条件匹配过滤」逻辑。
- **样式**：与「3. 触发条件」的 when-line 行距、错误提示风格保持一致，便于视觉统一。

### 5.2 可配置上限

- 建议将最大行数提为常量，如 `SCOPE_CONDITION_MAX_ROWS = 3`，便于后续改为 6 或配置化。

### 5.3 校验与禁用保存（可选）

- 若存在**未填写完整的行**（值为空），可在保存规则时提示「请完善监控范围条件或删除空行」，或仅提示不阻断；建议首版不阻断保存，仅做提示。

## 六、边界与验收要点

| 场景 | 预期 |
|------|------|
| 第一行左侧 | 不显示逻辑符 |
| 第二行及以后左侧 | 显示「AND」 |
| 名称 + 包含/不包含 | 值为文本输入；空则红字「数据不能为空」 |
| 状态 + 等于 | 值为下拉（所有投放中/所有已暂停/所有投放中和已暂停）；未选则「数据不能为空」 |
| 添加监控范围 (n/M) | 点击新增一行；达到 M 禁用添加 |
| 仅剩一行时删除 | 不显示 × 或删除后保留一行并重置 |
| 自动勾选 | 仅已加载列表中，满足所有已填写行的 AND 条件的项被勾选；id 格式与多账户一致 |
| 列表为空或未选账户 | 不报错；自动勾选结果为空 |

## 七、小结

- **新增**：「2. 作用对象」内「监控范围条件」区块，多行 [字段、操作符、值]，AND 连接，添加 (n/M)、删除、校验。
- **字段**：名称（包含/不包含 + 文本）、状态（等于 + 投放中/已暂停/投放中+已暂停）；结构可扩展。
- **联动**：根据所有已填写条件的 AND 结果，对当前 scopeItems 过滤后写回 `ruleForm.targetIds`，实现自动勾选。
- **不落库**：条件行仅前端状态；规则保存仍只提交 targetIds/target_by_account。

实现前请确认：  
1）是否采用「仅对当前已加载 scopeItems 生效」；  
2）最大行数 M 是否定为 3；  
3）存在空行时保存规则是否仅提示不阻断。确认后即可按此方案在 RuleManager 内实现并做一次联调与验收。
