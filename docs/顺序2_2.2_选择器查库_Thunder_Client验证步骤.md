# 顺序2 阶段 2.2：选择器查库 — Thunder Client 详细验证步骤

本文档按《顺序2_2.2_选择器查库_验收步骤.md》逐条说明如何用 Thunder Client 在本地完成验收。

**约定**：所有请求都需登录。在 Thunder Client 的 Headers 里加：`Authorization: Bearer <你的token>`（先通过 `POST /api/auth/login` 获取 token）。服务端端口以你实际为准（默认 3001），下文用 `http://localhost:3001` 示例。

---

## 一、前置：准备数据与 Token

### 1.1 确保 structure_ads 有数据

- 若尚未做过「强制同步结构」，先发一次：
  - **POST** `http://localhost:3001/api/sync/structure`
  - Body (JSON)：`{ "account_id": "act_你的账户" }`
  - 期望 200，`synced_count > 0`
- 记下你要用的 **account_id**（如 `act_647601474804558`），后面所有请求都用同一个账户（除「步骤 6 权限」外）。

### 1.2 获取登录 Token

- **POST** `http://localhost:3001/api/auth/login`
- Body (JSON)：`{ "username": "你的用户名", "password": "你的密码" }`
- 在响应里复制 `token` 整段，后续每个请求在 **Headers** 里加：  
  `Authorization` = `Bearer <粘贴的token>`

### 1.3 准备「他人账户」用于步骤 6

- 需要另一个 **act_xxx**，且当前登录用户（非 admin）**没有**该账户权限（不在 `account_mappings` 里或 `owner_id` 不等于当前用户）。
- 若没有，步骤 6 可暂时跳过或换一个无权限的 act_xxx 试。

---

## 二、步骤 2：默认短列表（仅 ACTIVE）

**目的**：不传 `q`、不传 `include_paused` 时，只返回 ACTIVE，且来自 structure_ads。

### Thunder Client 设置

- **Method**：`GET`
- **URL**：  
  `http://localhost:3001/api/structure/ads?account_id=act_你的账户&limit=20`
- **Headers**：  
  - `Authorization`: `Bearer <你的token>`
- **Body**：无（GET 无 body）

### 发送后检查

1. **Status** 为 **200**。
2. **Response** 为 JSON，结构类似：
   ```json
   {
     "items": [ { "id": "...", "name": "...", "effective_status": "ACTIVE", ... } ],
     "paging": { "after": "123", "next": null },
     "meta": { "level": "ads", "limit": 20, "q": null, "source": "structure_ads" }
   }
   ```
3. **验收点**：
   - `meta.source === 'structure_ads'`。
   - `items` 里每条的 `effective_status` **均为** `ACTIVE`。
   - `items.length` ≤ 20。

---

## 三、步骤 3：包含暂停（include_paused=1）

**目的**：加上 `include_paused=1` 后，可出现 ACTIVE 与 PAUSED，且无 DELETED/ARCHIVED。

### Thunder Client 设置

- **Method**：`GET`
- **URL**：  
  `http://localhost:3001/api/structure/ads?account_id=act_你的账户&limit=50&include_paused=1`
- **Headers**：  
  - `Authorization`: `Bearer <你的token>`

### 发送后检查

1. **Status** 为 **200**。
2. **验收点**：
   - `items` 中只出现 `ACTIVE` 和/或 `PAUSED`（可有 0 条 PAUSED，取决于库里是否有）。
   - 不应出现 `DELETED`、`ARCHIVED`。
   - `meta.source === 'structure_ads'`。

---

## 四、步骤 4：搜索（q 非空）

**目的**：传关键词 `q` 时，只返回名称包含该关键词的广告，且状态为 6 类之一（无 DELETED/ARCHIVED）。

### Thunder Client 设置

- **Method**：`GET`
- **URL**：  
  `http://localhost:3001/api/structure/ads?account_id=act_你的账户&q=某关键词&limit=20`
- 把 **某关键词** 换成你库里真实存在的广告名称的一部分（例如某个广告名里的一个词），否则可能得到空列表（仍应为 200）。
- **Headers**：  
  - `Authorization`: `Bearer <你的token>`

### 发送后检查

1. **Status** 为 **200**。
2. **验收点**：
   - 每条 `items[].name` 应**包含**你传的 `q` 关键词（不区分大小写由 DB LIKE 决定，一般为区分）。
   - 每条 `effective_status` 只能是：`ACTIVE`、`PAUSED`、`PENDING_REVIEW`、`DISAPPROVED`、`IN_PROCESS`、`WITH_ISSUES` 之一，**不能**出现 `DELETED`、`ARCHIVED`。
   - `meta.q` 应为传入的关键词。

---

## 五、步骤 5：翻页不卡

**目的**：第一页用 `limit=10`，用返回的 `paging.after` 请求第二页，两页无重复、顺序稳定、响应正常。

### 5.1 请求第一页

- **Method**：`GET`
- **URL**：  
  `http://localhost:3001/api/structure/ads?account_id=act_你的账户&limit=10`
- **Headers**：`Authorization`: `Bearer <你的token>`

**检查**：Status 200，记下：
- `items` 中每条广告的 `id`（即 ad_id），例如 10 条。
- `paging.after` 的值（一个数字字符串，对应 `structure_ads.id`），例如 `"456"`。

### 5.2 请求第二页

- **Method**：`GET`
- **URL**：  
  `http://localhost:3001/api/structure/ads?account_id=act_你的账户&limit=10&after=<paging.after>`
- 把 `<paging.after>` 换成上一步记下的值，例如：  
  `http://localhost:3001/api/structure/ads?account_id=act_你的账户&limit=10&after=456`
- **Headers**：同上。

**检查**：
1. Status 200，响应时间正常（无长时间卡住）。
2. 第二页的 `items` 与第一页的 `items` **无重复**（比较两边的 `id`）。
3. 顺序稳定：第二页第一条的 `id` 应排在第一页最后一条之后（按 `structure_ads.id` 升序）。

---

## 六、步骤 6：权限 — 非 admin 查他人 account 应 403

**目的**：用**非 admin** 账号登录，请求**他人**的 `account_id`，应返回 403 且带 `code: 'ACCOUNT_FORBIDDEN'`。

### 6.1 使用非 admin 账号

- 用另一个账号登录（或你当前就是非 admin）：  
  **POST** `/api/auth/login`，Body 里用该账号的 username/password，拿到 **非 admin** 的 token。

### 6.2 请求他人账户

- **Method**：`GET`
- **URL**：  
  `http://localhost:3001/api/structure/ads?account_id=他人act_xxx&limit=20`
- 把 **他人act_xxx** 换成一个当前用户**没有权限**的广告账户 ID（不在该用户的 account_mappings 里）。
- **Headers**：  
  - `Authorization`: `Bearer <非admin的token>`

### 发送后检查

1. **Status** 为 **403**。
2. **Response** 中应包含 `code: 'ACCOUNT_FORBIDDEN'`（以及 `error` 说明，如「无权访问该广告账户」）。

---

## 七、步骤 7：服务端日志证明 0 FB 列表请求

**目的**：确认 2.2 接口只查 structure_ads，没有向 FB 发列表请求。

### 重要：必须「隔离」验证

终端里若出现 `graph.facebook.com`、`GET .../act_xxx/ads`、SOCKS 连接等，**不一定**是本次 `GET /api/structure/ads` 触发的。  
其他请求也会打 FB，例如：

- **POST /api/sync/structure**（强制同步结构）会请求 FB 的 `/ads`，终端会看到 SOCKS → graph.facebook.com、`GET /v24.0/act_xxx/ads` 等；
- 其他接口或前端也可能调 FB。

因此步骤 7 要**单独、只发一次** `GET /api/structure/ads`，再根据这次请求看日志。

### 推荐操作（隔离验证）

1. **清空或留意控制台**：重启后端（`npm run dev:server`），或记下当前时间，便于只看之后新产生的日志。
2. **只发一个请求**：用 Thunder Client 只发一次  
   `GET /api/structure/ads?account_id=act_你的账户&limit=20`  
   （Headers 带 `Authorization: Bearer <token>`），**不要同时或随后立刻发** `POST /api/sync/structure` 或其他会调 FB 的请求。
3. **看控制台**：
   - **应有**：一行 **`[2.2] GET /api/structure/ads from structure_ads only, no FB request`**，后面跟 `account_id`、`q`、`limit`、`count`、`has_next` 等（具体格式以代码为准）。
   - **这次请求不应出现**：同一时刻附近没有新的 `graph.facebook.com`、`GET .../act_xxx/ads`、SOCKS 连接、`getStructurePage`、`makeRequest` 等。  
   若在「只发了 GET /api/structure/ads」的前提下仍看到 FB 请求，才说明 2.2 走了 FB，验收不通过。

### 期望日志（针对这一次 GET /api/structure/ads）

- 出现一行类似：  
  `[2.2] GET /api/structure/ads from structure_ads only, no FB request`  
  且带有 `account_id`、`q`、`limit`、`count`、`has_next` 等。
- 该**单次**请求**没有**伴随：
  - `getStructurePage`、
  - `makeRequest`、
  - 或向 `graph.facebook.com` 请求 `/ads` 的日志。

若有上述 FB 列表相关日志（且在**仅**调用 GET /api/structure/ads 时出现），则说明 2.2 的 ads 分支仍走了 FB，验收不通过。
