# 顺序2 阶段 2.1：structure_ads 表建表与迁移 — 落地方案

> **约定**：本方案中的**所有 SQL 与终端命令均由你亲自在本地执行**，Cursor 不代为运行。执行前请先备份数据库。

---

## 一、本次目标

- **交付物**：新增结构镜像表 `structure_ads`（仅广告层级），并创建所需索引，为 2.2「选择器查库」提供数据表。
- **范围**：本阶段只做 **structure_ads**，不做 structure_campaigns / structure_adsets（按 TASKS 最短路径「先只做 Ad 级就够用」）。
- **验收**：同账户下按 `ad_id` / `name` / `effective_status` 查询可走索引、稳定毫秒级；默认过滤（排除 DELETED/ARCHIVED）可通过 `(account_id, effective_status)` 索引支持。

---

## 二、变更点（具体到文件/内容）

| 类型 | 路径 | 说明 |
|------|------|------|
| 迁移 SQL（你执行） | `server/db/migrations/014_create_structure_ads.sql` | 建表 + 索引，幂等写法（存在则跳过） |
| Schema 定义（可选） | `server/db/schema.js` | 若你后续用 Drizzle 查 structure_ads，可在此增加表定义；2.1 阶段也可先只用原生 SQL 查，不强制 |

**表结构概要**（与 TASKS 1.2、DEV_PLAN 口径一致）：

- **共同字段**：`account_id`, `ad_id`（对应 TASKS 中的 id）, `name`, `effective_status`, `status`, `updated_time`, `last_synced_at`
- **ads 独有**：`adset_id`, `campaign_id`；`configured_status` 可选
- **唯一约束**：`(account_id, ad_id)` → 方便后续 upsert（INSERT ... ON DUPLICATE KEY UPDATE）
- **索引**：`(account_id, effective_status)`、`(account_id, name)`、唯一即覆盖 `(account_id, ad_id)` 的按 id 查询

**职责边界**（与 ad_snapshots 区分）：

- **structure_ads**：仅用于选择器列表/搜索/翻页、resolve 回显；只存「结构+状态」，不存 spend/cpc 等指标。
- **ad_snapshots / daily_stats**：仅用于规则引擎指标计算与规则评估；不用于「可选对象列表」。

## 三、风险点与注意事项

1. **备份**：执行迁移前请先备份数据库（例如 `mysqldump -u root -p <dbname> > backup_YYYYMMDD_HHmmss.sql`）。
2. **库名**：迁移脚本内使用 `DATABASE()`，请在**正确的库**下执行（与现有 `ad_snapshots` 同一库）。
3. **字符集**：与现有表一致使用 `utf8mb4`。
4. **ID 长度**：所有 ID 字段 `VARCHAR(50)`，与现有 schema.js / ad_snapshots 一致，避免 BIGINT 溢出问题。
5. **幂等**：脚本为「表不存在则创建」；若你已手动建过表，再次执行不会报错，可安全重复执行。

## 四、验证步骤（请你亲自执行）

以下步骤均在**你本机**完成，不依赖 Cursor 执行。

### 4.1 执行迁移

在项目根目录或 `server/db/migrations` 下，用 MySQL 客户端执行：

```bash
# 示例：MySQL 命令行
mysql -u <你的用户名> -p <你的数据库名> < server/db/migrations/014_create_structure_ads.sql
```

或在 MySQL 客户端内：

```sql
SOURCE d:/projects/FB-Ad-Logic-Engine/server/db/migrations/014_create_structure_ads.sql;
```

（请按你的实际路径与数据库名替换。）

### 4.2 验证表与索引

在 MySQL 中执行：

```sql
-- 1) 表是否存在
SHOW TABLES LIKE 'structure_ads';

-- 2) 表结构
DESCRIBE structure_ads;

-- 3) 索引是否齐全（应看到 uk_account_ad、idx_account_effective_status、idx_account_name）
SHOW INDEX FROM structure_ads;
```

### 4.3 验收标准（自检）

- [ ] 表 `structure_ads` 存在，且包含字段：`account_id`, `ad_id`, `adset_id`, `campaign_id`, `name`, `effective_status`, `status`, `configured_status`, `updated_time`, `last_synced_at`。
- [ ] 存在唯一索引 `uk_account_ad` on `(account_id, ad_id)`。
- [ ] 存在索引 `idx_account_effective_status`、`idx_account_name`。
- [ ] 同账户下按 `ad_id` / `name` / `effective_status` 的查询可由上述索引覆盖（验收时表可为空，只要索引存在即可）。
