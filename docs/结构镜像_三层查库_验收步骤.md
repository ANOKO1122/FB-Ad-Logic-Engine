# 结构镜像：三层查库（campaigns/adsets/ads）验收步骤

顺序2 扩展：campaigns、adsets、ads 三层均查库，0 FB 列表调用；resolve 支持三类回显。

---

## 一、前置：执行迁移

1. **备份数据库**（PowerShell 示例）：
   ```powershell
   mysqldump -u root -p <数据库名> > backup_$(Get-Date -Format 'yyyyMMdd_HHmmss').sql
   ```

2. **执行迁移 023、024**（由你亲自在本地执行）：
   ```bash
   mysql -u 你的用户名 -p 你的数据库名 < server/db/migrations/023_create_structure_campaigns.sql
   mysql -u 你的用户名 -p 你的数据库名 < server/db/migrations/024_create_structure_adsets.sql
   ```

3. **验证表存在**：
   ```sql
   SHOW TABLES LIKE 'structure_%';
   -- 应看到 structure_ads, structure_adsets, structure_campaigns, structure_sync_status
   DESCRIBE structure_campaigns;
   DESCRIBE structure_adsets;
   ```

---

## 二、单测

```bash
npm test
```

应全部通过（含 structureResolveSync、ruleEngineDispatcher 等）。

---

## 三、手动同步：campaigns + adsets + ads

1. 启动后端。
2. 登录后，对你有权限的账户调用（Thunder Client / curl）：
   ```
   POST /api/sync/structure
   body: { "account_id": "act_你的账户ID" }
   ```
3. 响应应包含 `synced_count`、`synced_campaigns`、`synced_adsets`、`duration_ms`。
4. 服务端日志应出现：
   - `[2.4] campaigns 同步 account=act_xxx count=N`
   - `[2.4] adsets 同步 account=act_xxx count=M`
   - `✅ [2.4] 结构同步完成 account=act_xxx ... campaigns=N adsets=M ads=K`

5. **查库确认**：
   ```sql
   SELECT COUNT(*) FROM structure_campaigns WHERE account_id = 'act_你的账户ID';
   SELECT COUNT(*) FROM structure_adsets   WHERE account_id = 'act_你的账户ID';
   SELECT COUNT(*) FROM structure_ads     WHERE account_id = 'act_你的账户ID';
   ```
   三个表应有数据。

---

## 四、列表接口：0 FB 调用

1. 发请求（替换为你的 account_id）：
   - `GET /api/structure/campaigns?account_id=act_xxx&limit=20`
   - `GET /api/structure/adsets?account_id=act_xxx&limit=20`
   - `GET /api/structure/ads?account_id=act_xxx&limit=20`

2. 期望：
   - 均 200，返回 `items`、`paging`、`meta`
   - `meta.source` 分别为 `structure_campaigns`、`structure_adsets`、`structure_ads`
   - 服务端日志出现 `[2.2] GET /api/structure/campaigns from structure_campaigns only, no FB request` 等，且本次请求**无** `graph.facebook.com`、`getStructurePage` 调用。

---

## 五、resolve 回显：campaign / adset / ad 三类

1. **广告**：`GET /api/structure/resolve?ids=某ad_id` → 应返回 `{ id, name, type: 'ad', ... }` 或等价，无 `missing`。

2. **广告组**：插入测试数据后测试：
   ```sql
   INSERT INTO structure_adsets (account_id, adset_id, campaign_id, name, effective_status, status)
   VALUES ('act_你的账户ID', '999001', NULL, '测试广告组', 'ACTIVE', 'ACTIVE');
   ```
   `GET /api/structure/resolve?ids=999001` → 应返回 `{ id: '999001', name: '测试广告组', type: 'adset', ... }`。

3. **广告系列**：类似插入 structure_campaigns，再 resolve 对应 campaign_id。

4. **混合**：`GET /api/structure/resolve?ids=ad1,campaign1,adset1`（替换为真实 id）→ 应分别返回 name 和 type；不存在的 id 返回 `{ id, missing: true }`。

---

## 六、RuleManager 端到端

1. 规则管理 → 新建规则。
2. 目标层级选 **广告系列 (Campaign)** 或 **广告组 (Adset)**。
3. 选择列表中应有数据（来自 structure_campaigns / structure_adsets）。
4. 保存后点 **编辑规则**，已选对象应显示名称，不应出现「id (未同步)」。
5. 若首次同步后 campaigns/adsets 为空，点「刷新列表」再试。

---

## 七、风险与排查

| 现象 | 可能原因 | 排查 |
|------|----------|------|
| campaigns/adsets 列表为空 | 未执行手动同步 | 点「刷新列表」或 POST /api/sync/structure |
| resolve 返回 missing | 结构表无该 id | 先执行手动同步；确认 id 属于该账户 |
| 单测失败 | 迁移未执行 | 执行 023、024；确认表存在 |
