# 采纳清单与具体修改说明（完整版）

> 包含：多账户上限 + 首轮 p-limit、名称类条件大表优化（原则+索引+文档+推广）、**状态不等于**配置完善、防恢复不采纳。以下为需采纳项及逐项具体改法。

---

## 一、采纳项总览

| 项 | 是否采纳 | 说明 |
|----|----------|------|
| Smart Mute（mute_until） | ✅ 已有 | 无需改，保持现状。 |
| 防恢复守护 | ❌ 不采纳 | 不做。 |
| account_ids 上限 + 管理员放宽 | ✅ 采纳 | multi 单次请求限制账户数，管理员放宽至 50。 |
| 首轮 p-limit(4) | ✅ 采纳 | multi 首轮多账户并行查库，最多 4 个并发。 |
| **名称类条件大表优化** | ✅ 采纳 | 原则+索引确认+文档+WHERE 顺序注释；推广到「名称包含」「名称不包含」。 |
| **状态不等于** | ✅ 采纳 | 监控范围「状态」行支持「不等于」；后端支持 `scope_status_exclude`，前端传参与列表请求透传。 |
| 依赖 p-limit | ✅ 采纳 | package.json 增加 p-limit（若无）。 |

## 二、具体修改清单

### 1. account_ids 上限 + 管理员放宽

| 项目 | 内容 |
|------|------|
| **文件** | `server/index.js` |
| **位置** | `GET /api/structure/objects/multi`，在得到 `allowedIds` 之后、首轮/游标逻辑之前。 |
| **逻辑** | 定义常量 `MULTI_ACCOUNT_IDS_MAX = 20`，`MULTI_ACCOUNT_IDS_MAX_ADMIN = 50`；`const isAdmin = req.user && req.user.role === 'admin'`；`const maxAccounts = isAdmin ? MULTI_ACCOUNT_IDS_MAX_ADMIN : MULTI_ACCOUNT_IDS_MAX`；若 `allowedIds.length > maxAccounts` 则 `return res.status(400).json({ error: '单次请求最多选择 ' + maxAccounts + ' 个广告账户', code: 'TOO_MANY_ACCOUNTS' })`。 |

---

### 2. 首轮 p-limit(4)

| 项目 | 内容 |
|------|------|
| **文件** | `server/index.js` |
| **位置** | multi 首轮分支（无 `afterRaw`），将 `for (const accountId of allowedIds) { ... await listStructureObjectsFromDb(...) }` 改为用 p-limit 并发。 |
| **实现** | 顶部或该路由内 `const pLimit = (await import('p-limit')).default`，`const limit = pLimit(4)`；`const results = await Promise.all(allowedIds.map((accountId, index) => limit(async () => ({ index, accountId, result: await listStructureObjectsFromDb(accountId, { type, q, limit: perAccountLimit, after: null, include_paused: includePausedEffective, scope_status, name_exclude, scope_status_exclude }) }))))`；按 `results[].index` 排序后，顺序填充 `perAccountAfter`、合并 `allItems`（去重、slice(0, limit)），生成 `nextAfter` 逻辑与现有一致。 |
| **依赖** | `package.json` 增加 `"p-limit": "^x.x.x"`（若尚无），并 `npm i p-limit`。 |

### 3. 名称类条件大表优化（原则 + 索引 + 文档 + 推广）

| 项目 | 内容 |
|------|------|
| **原则** | 所有 structure 列表查询保持「WHERE account_id = ? AND effective_status IN (...) 先写，再 AND name LIKE / name NOT LIKE」；名称包含（q）、名称不包含（name_exclude）统一按此顺序，便于用 `idx_account_effective_status` 先缩小再扫 name。 |
| **索引** | 确认迁移 014/023/024 已建 `idx_account_effective_status (account_id, effective_status)`（structure_ads / structure_campaigns / structure_adsets）；无需新增迁移。 |
| **代码注释** | 在 `server/services/structureSyncService.js` 中三个 list 方法（listStructureAdsFromDb、listStructureCampaignsFromDb、listStructureAdsetsFromDb）的 WHERE 构建处增加注释：`// 顺序：先 account_id + effective_status（可走索引），再 name LIKE/NOT LIKE（大表较贵），便于大表性能。` |
| **文档** | 在 `docs/监控范围条件_后端SQL过滤与优先展示方案.md` 或 `docs/采纳清单与具体修改说明.md` 中注明：名称包含、名称不包含在大表上为较贵操作；若出现慢查询可考虑两段查询或应用层过滤。 |

---

### 4. 状态不等于（后端 + 前端 + 接口透传）

#### 4.1 后端：支持 scope_status_exclude

| 项目 | 内容 |
|------|------|
| **文件** | `server/services/structureSyncService.js` |
| **方法** | `listStructureAdsFromDb`、`listStructureCampaignsFromDb`、`listStructureAdsetsFromDb` 三个方法。 |
| **入参** | `opts.scope_status_exclude`（可选）：字符串，逗号分隔的状态码，如 `'PAUSED'` 或 `'ACTIVE,PAUSED'`。 |
| **逻辑** | 在现有「根据 scope_status / include_paused / q 得到 statusList」之后，若 `opts.scope_status_exclude` 非空：解析为数组 `excludeList`（split 逗号、trim、去空），`statusList = statusList.filter(s => !excludeSet.has(s))`（excludeSet = new Set(excludeList)）。若过滤后 statusList 为空，可直接返回 `{ items: [], paging: { after: null } }` 或仍用 `IN ()` 查询（结果为空）。 |
| **JSDoc** | 在三个方法的 @param opts 中增加 `scope_status_exclude` 说明。 |
| **listStructureObjectsFromDb** | 将 `opts.scope_status_exclude` 传入 listOpts，下发给三个 list 方法。 |

#### 4.2 接口透传 scope_status_exclude

| 项目 | 内容 |
|------|------|
| **文件** | `server/index.js` |
| **位置** | 1）单账户：`GET /api/structure/objects` 或 `GET /api/structure/:level` 从 `req.query.scope_status_exclude` 读取，传入 `listStructureObjectsFromDb` 的 opts；2）multi：`GET /api/structure/objects/multi` 从 `req.query.scope_status_exclude` 读取，在首轮与带 after 分支中均传入 `listStructureObjectsFromDb` 的 opts。 |

#### 4.3 前端：参数生成与请求透传

| 项目 | 内容 |
|------|------|
| **文件** | `src/views/RuleManager.vue` |
| **状态行与「不等于」** | 前端已有「状态」行的 operator「不等于」（value 仍为 active_only / paused_only / active_and_paused）。需要：当状态行为「不等于」时，不把 row.value 当作 scope_status（等于）传给后端，而是把「要排除的状态」作为 scope_status_exclude 传给后端。 |
| **映射** | active_only → 排除 ACTIVE；paused_only → 排除 PAUSED；active_and_paused → 排除 ACTIVE,PAUSED。即 `scope_status_exclude = row.value === 'active_only' ? 'ACTIVE' : row.value === 'paused_only' ? 'PAUSED' : row.value === 'active_and_paused' ? 'ACTIVE,PAUSED' : ''`。 |
| **applyScopeConditions** | 在计算 `scopeStatus` 时区分 equals / not_equals：若 statusRow.operator === 'equals'，则 `scopeStatusFilter = statusRow.value`，`scopeStatusExcludeFilter = ''`；若 statusRow.operator === 'not_equals'，则 `scopeStatusFilter = ''`（或不传 scope_status，用后端默认 statusList），`scopeStatusExcludeFilter = statusValueToExclude(statusRow.value)`（如上述映射）。需新增 ref：`scopeStatusExcludeFilter`。 |
| **列表请求** | 所有调用结构列表的地方（单账户 / multi、首屏与加载更多）在已有 `scope_status`、`name_exclude` 基础上，增加 `scope_status_exclude: scopeStatusExcludeFilter.value || undefined`（或仅当有值时才带）。 |
| **初始/清空** | 在清空 scopeStatusFilter、scopeNameExclude 的地方同时清空 scopeStatusExcludeFilter。 |

## 五、验收要点（简要）

1. **account_ids 上限**：非管理员传 21 个账户 → 400、code TOO_MANY_ACCOUNTS；管理员传 36 个 → 200。
2. **p-limit**：multi 首轮 36 账户时，首屏响应时间明显短于改前（可看日志或计时）。
3. **状态不等于**：前端选「状态 不等于 所有已暂停」并应用条件 → 列表仅返回非 PAUSED（如 ACTIVE 等）；选「不等于 所有投放中」→ 仅返回非 ACTIVE；后端带 `scope_status_exclude=PAUSED` 的请求返回与 SQL 一致。
4. **名称类**：WHERE 顺序与注释一致；文档中已注明名称包含/不包含大表注意点。

以上为完整采纳清单与具体修改说明；名称类条件已推广到包含/不包含，状态不等于已纳入并给出前后端与透传的改法。
