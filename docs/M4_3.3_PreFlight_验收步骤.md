# M4 3.3 Pre-Flight 验收步骤

本验收确认：**本地 status 已达成则跳过、不调 API**；**FB 返回 "already" 类错误时记 skipped**。

---

## 场景一：本地 status 已是 PAUSED，触发暂停规则

**预期**：不调用 `pauseAd`，日志与 automation_logs 为 skipped。

1. 选一条**已暂停**的广告（在 FB 后台或 ad_snapshots 中 status=PAUSED）。
2. 确保该广告在规则的作用对象内，且规则条件会命中（如花费>0）。
3. 执行该规则（「运行此规则」或等定时）。
4. **验收**：
   - 日志出现：`⏭️  Pre-Flight: 广告 xxx 已处于 PAUSED，无需再 pause，跳过`。
   - `automation_logs` 中该 ad_id 本条记录：`status=skipped`，`error_message` 含「目标已达成」或 `api_response` 含 `reason: 'already_paused'`。
   - 无 `🔧 执行: 暂停广告` 及后续 `pauseAd` 请求（可看日志或抓包）。

**推荐做法**：用一条在 **FB 后台已经暂停** 的广告来测。这样同步会把 `ad_snapshots.status` 写成 PAUSED，规则执行时读到 PAUSED 就会 Pre-Flight 跳过。

**若在 DB 手动改 status**：定时同步（每 10/15 分钟）会从 Facebook 拉回最新状态并**覆盖** `ad_snapshots`。你改完 PAUSED 后若等到下一次同步再跑规则，同步可能已把 status 写回 ACTIVE（因为 FB 上仍是 ACTIVE），执行时就会再次调 API。因此：要么在 **改完 DB 后 1～2 分钟内、下次同步前** 立刻点「运行此规则」；要么直接用在 FB 已暂停的广告，避免被同步覆盖。

---

## 场景二：FB 返回 "Ad is already paused"（或含 already/duplicate）

**预期**：catch 后记 status=skipped，不抛错。

1. 构造「规则要暂停的广告在请求发出前已被其他途径暂停」的情况（例如手快在 FB 后台先暂停，或双端同时操作）。
2. 触发规则，执行层会调用 `pauseAd`，FB 可能返回已暂停错误。
3. **验收**：
   - 日志出现：`⏭️  FB 容错: 广告 xxx 已暂停，记 skipped`。
   - `automation_logs` 中该条：`status=skipped`，`api_response` 含 `already_in_state` 或类似说明。

若难以稳定复现，可暂时跳过场景二；场景一通过即可认为 Pre-Flight 本地判断验收通过。

---

## 通过标准

- 场景一通过：Pre-Flight 本地 status 判断生效，不重复调用 pause API。
- 场景二通过（可选）：FB already-state 容错生效。

验收通过后在 TASKS 3.3 对应验收项打勾。
