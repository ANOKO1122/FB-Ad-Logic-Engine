# 规则字段判断 + 作用对象选择（合并方案）

## 0. 目标与非目标

目标：解决

- 选不准/选到废弃广告/选得累
- 近 N 天发布（created_time）
- 4 个漏斗单次成本指标（CP_ATC/CP_IC/CP_API/单次购买成本）
- 命名驱动（广告名包含/不包含，新增同类广告自动覆盖）
- 可解释、可复盘（执行时可还原“命中集合 + 条件口径”）

非目标：

- 暂不做复杂表达式树（AND/OR 任意嵌套），保留演进路径

## 1. 数据与同步（必须先打底）
### 1.1 ad_snapshots 增加 created_time
- ad_snapshots.created_time （UTC 时间戳，广告创建时间）
### 1.2 同步侧：Graph ids 批量拉 created_time（50/批）并回写
同步任务每次拿到一批广告后：

- 先去重 ad_id
- 用 Facebook Graph 的 ids 批量查询 created_time（每批 50 个）
- 写回 ad_snapshots（重复写无所谓，值不变）

这样实现“收集时顺便收集回来”，同时避免把请求数放大到不可控。

### 1.3 层级上下文（分两阶段）
- 第 1 阶段：先显示 ad_set_id（你已有），立刻提升识别度
- 第 2 阶段（你们要 AdsPolar 级才做）：补齐 adset_name/campaign_name 的离线来源（维度表或冗余字段），让识别达到“看一眼就知道是哪条”
## 2. Where（作用对象）设计（核心体验）

targetLevel：campaign / adset / ad

- campaign：系列级控盘（活动整体止损/扩量）
- adset：受众/版位/预算组控盘（运营最常用）
- ad：素材级精细化（对“身份证信息”要求最高）

### 2.1 targetMode：IDS vs FILTER（默认 FILTER）
- 手动选择（IDS） ：继续支持勾选具体广告/广告组/系列，兼容旧习惯
- 按条件筛选（FILTER，推荐默认） ：运营主力入口
### 2.2 FILTER 三件套（status + createdWithinDays + name include/exclude）
1. 状态过滤（默认 ACTIVE）
2. 创建时间过滤：近 N 天发布 （created_time）
3. 命名过滤：广告名包含/不包含
- 包含支持：包含任意（OR）/必须都包含（AND）作为一个高级开关
- 不包含：黑名单词列表
并提供：

- 命中预览 ：展示“命中数量 + Top 50 示例”，按 spend 降序最符合运营确认逻辑
- 一键生成规则覆盖策略 ：运营通过改名即可控制纳入/排除范围（你要的效果）
### 2.3 IDS 止痛升级（默认 ACTIVE + 身份证信息）
- 默认只显示 ACTIVE（可切 ACTIVE+PAUSED / ALL）
- 每行必须显示“身份证信息”：
  - 状态（可视化）
  - 名称 + ID
  - Today Spend / ROAS（来自快照）
  - created_time（新）+ synced_at（确认数据新鲜度）
  - ad_set_id（至少先有它）
## 3. When（触发条件）与指标口径
### 3.1 CP_ATC / CP_IC / CP_API / 单次购买成本（中文展示）
- 单次加购成本 = spend / add_to_cart_count
- 单次结账成本 = spend / initiate_checkout_count
- 单次添加支付信息成本 = spend / add_payment_info_count
- 单次购买成本 = spend / purchases（你现 CPA 明确成这个）
### 3.2 分母为 0 的统一语义（null + 默认不满足）
这条是“完美方案”的核心：

- 所有成本类指标（CPA/CP_ATC/CP_IC/CP_API）分母=0 → 指标值 = null（不可用）
- 规则比较时：
  - metricValue == null → 默认“不满足”（无论 gt/lt/gte/lte/eq）
- UI 提示一行即可：
  - “当次数为 0 时，单次成本不可用，本条件不会命中。”
这样你不会再遇到“没出单反而 CPA=0 被判定很优秀”的灾难。

## 4. 规则结构与执行闭环（保证“可解释可复盘”）
### 4.1 规则保存结构（兼容旧 targetIds）
- 旧规则： targetIds + targetLevel 继续可用
- 新增：
  - targetMode: 'IDS' | 'FILTER'
  - targetFilter: { statusMode, createdWithinDays, nameInclude, nameExclude, includeMode }
### 4.2 执行闭环（FILTER 动态命中 + 可复盘审计）
当 targetMode=FILTER：

1. 执行开始时，先用 targetFilter 动态查询 出本次命中的 ad_id 列表（命中集合会随时间变化，这是 FILTER 的价值）
2. 再对命中集合跑 When 条件（用你们统一计算后的指标）
3. 写日志时必须能回答两个问题：
   - “这次规则为什么影响了这些广告？”（命中集合）
   - “当时筛选条件是什么？”（targetFilter）
最小可行复盘 （不一定要存全量列表）：

- 存 matched_target_count
- 存 matched_target_sample （例如前 50 个 id + name）
- 未来需要更强审计再存全量/哈希摘要都可以
## 5. 演进路线（V1–V4，包含“闭环”落点）

- V1：IDS 止痛
  - 默认 ACTIVE；列表显示状态/名称+ID/spend/roas/created_time/synced_at/ad_set_id
- V2：FILTER 三件套 + 预览 + 保存条件
  - status + createdWithinDays + name include/exclude（include 支持 any/all）
  - 命中预览：命中数 + Top50 示例（按 spend 降序）
  - 执行审计：写 matched_target_count + matched_target_sample
- V3：4 指标 + 口径/比较语义落地
  - CP_ATC/CP_IC/CP_API/单次购买成本
  - 分母为 0 → null；比较时默认不满足；UI 一句提示
- V4：层级名称/级联筛选/可选素材缩略图
  - 补齐 adset_name/campaign_name（维表或冗余）
  - campaign → adset → ad 级联筛选，进一步降低误选率

（复杂 AND/OR 表达式树不在当前版本目标内；后续可将 targetFilter 升级为表达式树结构）
