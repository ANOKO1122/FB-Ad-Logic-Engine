# 规则执行日志记录实现方案

## 📋 本次目标

实现完整的规则执行日志记录功能，记录规则执行的完整生命周期，包括：
1. 规则评估阶段（匹配/不匹配）
2. 动作执行阶段（成功/失败/跳过）
3. 完整的审计信息（指标快照、API 请求/响应）

---

## 第一步：拆解概念（Conceptual Breakdown）

### 1) 一句话比喻

**规则执行日志就像"行车记录仪"**：
- 记录规则什么时候执行、执行了什么、结果如何
- 出问题时可以回放整个过程，快速定位原因
- 为系统提供"可追溯性"和"可审计性"

### 2) 为什么要学

**外企 WLB 角度**：
- **可追溯性**：出问题时能快速定位，不用熬夜排查
- **可审计性**：满足合规要求，证明系统行为符合预期
- **团队协作**：多人协作时，日志是沟通的桥梁

**可维护性角度**：
- **问题排查**：快速定位规则执行失败的原因
- **数据分析**：分析规则执行效果，优化规则配置
- **系统监控**：通过日志监控系统健康状态

**面试高频点**：
- 如何设计审计日志系统？
- 如何保证日志的完整性和可靠性？
- 如何处理日志写入失败的情况？

### 3) 面试怎么问

**问题1**：如何设计一个审计日志系统？
**回答要点**：
- 记录关键操作（谁、什么时候、做了什么、结果如何）
- 使用数据库持久化（不要只写文件）
- 日志写入失败不应该影响主流程（try-catch 包裹）
- 支持查询和筛选（按账户、规则、时间等）

**问题2**：如何保证日志的完整性？
**回答要点**：
- 在关键节点都记录日志（评估、执行、结果）
- 使用事务或批量写入提高性能
- 记录完整的上下文信息（指标快照、API 请求/响应）

**问题3**：日志写入失败怎么办？
**回答要点**：
- 日志写入失败不应该中断主流程（try-catch 包裹）
- 记录警告日志，便于后续排查
- 可以考虑异步写入或消息队列（高级方案）

### 4) 本节学习目标（可检验）

学完本节后，你应该能够：
1. ✅ 理解审计日志的设计原则和最佳实践

3. ✅ 实现日志查询 API，支持筛选和分页
4. ✅ 处理日志写入失败的情况（不影响主流程）
5. ✅ 验证日志记录的完整性和准确性

### 5) 最小可行方案（MVP）

**本次实现范围**：
1. ✅ 完善动作执行阶段的日志记录（已部分实现）

### 3) 可观测性：日志字段

**应该记录的日志字段**：
- `account_id`：账户ID
- `ad_id`：广告ID
- `ad_name`：广告名称（快照）
- `rule_id`：规则ID
- `rule_name`：规则名称（快照）
- `owner_id`：负责人ID
- `metrics_snapshot`：指标快照（JSON，包含 spend、cpc、roas 等）
- `action_type`：动作类型
- `action_payload`：动作参数（JSON）
- `is_simulation`：是否为 Dry Run 模式
- `api_request`：API 请求原文
- `api_response`：API 响应原文
- `status`：执行状态（success/fail/skipped）
- `error_message`：错误信息（如果有）
- `triggered_at`：触发时间

### 4) 最容易出的 3 个 Bug

**Bug 1**：日志写入失败导致主流程中断
- **现象**：规则执行失败，但实际动作已执行
- **原因**：日志写入没有 try-catch 包裹
- **快速定位**：查看错误日志，找到 `写入审计日志失败` 的警告

**Bug 2**：日志记录不完整（缺少某些字段）
- **现象**：日志中某些字段为 NULL
- **原因**：数据获取时机不对，或字段映射错误
- **快速定位**：检查 `metricsSnapshot` 的构建逻辑，确认所有字段都有值

**Bug 3**：日志查询性能问题
- **现象**：查询日志很慢，或超时
- **原因**：没有索引，或查询条件不当
- **快速定位**：检查 `automation_logs` 表的索引，优化查询 SQL
