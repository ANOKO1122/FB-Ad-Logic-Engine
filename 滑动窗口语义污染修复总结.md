# 滑动窗口语义污染修复总结

## ✅ 修复完成

### 问题描述

**原问题：**
- `syncAccountSlidingWindow` 函数拉取 `today` + `last_7d` 数据
- `last_7d` 是过去7天的聚合值（多天合计）
- 这些数据被写入 `ad_snapshots` 表
- 当查询 `today` 时，可能会把 `last_7d` 的聚合值当作"今天的数据"
- **语义污染**：今天没有花费，但过去7天有花费，查询 today 时可能误判

**示例：**
- 某广告今天没有花费，但过去 7 天合计花费 300
- 滑动窗口把 `last_7d` 聚合写入 `ad_snapshots` 后
- `today` 查询会捡到这条快照
- 规则"今天花费>0"可能误判为真
- 这与"今天的真实值"为 0 的业务语义相悖

---

## 🔧 修复方案

### 核心改进

**严格按自然日语义：**
1. **`ad_snapshots` 只存"今天"的实时快照**
   - 只使用 `date_preset=today` 的数据
   - 不混入 `last_7d` 聚合值

2. **`daily_stats` 存"按自然日"的历史归档**
   - 使用 `time_increment=1` + `since/until` 拉取按日数据
   - 对每个"广告-自然日"更新 `daily_stats`
   - 把迟到的归因补齐到正确日期

3. **查询路由保持不变但更干净**
   - `today` → `ad_snapshots`（最后快照，只包含 today 数据）
   - `yesterday` → `daily_stats`，06:00 前空则降级到 `ad_snapshots` 的"昨日最后快照"
   - `last_X_days` → `daily_stats`（历史段）+ `ad_snapshots`（今天段）合并后动态聚合

## 📝 实现细节

### 1. 新增函数：`fetchInsightsByDay`

**功能：** 使用 `time_increment=1` 拉取按日数据

**参数：**
- `accountId`: 账户ID
- `adIds`: 广告ID列表
- `facebookApi`: Facebook API 客户端
- `sinceDate`: 开始日期（YYYY-MM-DD，账户时区）
- `untilDate`: 结束日期（YYYY-MM-DD，账户时区）

**实现：**
```javascript
// 使用 time_increment=1 获取按日数据
const batchRequests = chunk.map(adId => ({
  method: 'GET',
  relative_url: `${adId}/insights?fields=${fields}&time_increment=1&since=${sinceDate}&until=${untilDate}&use_account_attribution_setting=${useAccountAttributionSetting}`
}))
```

**返回：** 按日数据数组，每个元素包含 `date_start` 字段（YYYY-MM-DD）

---

### 2. 新增函数：`updateDailyStatsFromInsights`

**功能：** 将按日数据更新到 `daily_stats` 表

**实现：**
- 解析每个按日数据的字段（spend, purchases, link_clicks 等）
- 使用 `ON DUPLICATE KEY UPDATE` 更新 `daily_stats`
- 唯一键：`uk_account_ad_date(account_id, ad_id, date)`

**特点：**
- 幂等更新：如果记录已存在，更新字段；如果不存在，插入新记录
- 修复迟到归因：将"迟到事件"补齐到正确的自然日

---

### 3. 重构函数：`syncAccountSlidingWindow`

**核心逻辑：**

1. **Today 数据 → `ad_snapshots`**
   ```javascript
   // 只拉取 today 数据
   const todayInsights = await fetchInsightsInBatches(accountId, adIds, facebookApi, 'today')
   // 写入 ad_snapshots（只存 today 数据）
   const todayCount = await saveSnapshotsToDb(todayInsights, ...)
   ```

2. **过去 N 天的按日数据 → `daily_stats`**
   ```javascript
   // 计算日期范围（账户时区）
   const untilDate = now.minus({ days: 1 }).toFormat('yyyy-MM-dd')  // 昨天（不包含今天）
   const sinceDate = now.minus({ days: daysBack }).toFormat('yyyy-MM-dd')  // N 天前
   
   // 拉取按日数据
   const dailyInsights = await fetchInsightsByDay(accountId, adIds, facebookApi, sinceDate, untilDate)
   
   // 更新 daily_stats（按日修复迟到归因）
   const dailyStatsCount = await updateDailyStatsFromInsights(dailyInsights, ...)
   ```

**返回值：**
```javascript
{
  success: true,
  todayCount: number,        // ad_snapshots 写入的记录数
  dailyStatsCount: number,  // daily_stats 更新的记录数
  sessionId: string
}
```

---

### 4. 更新函数：`syncAllAccountsSlidingWindow`

**新增功能：**
- 分布式锁：防止多实例重复执行
- 使用 `GET_LOCK('sync:sliding_window', 0)` 获取锁
- 执行完成后释放锁

**返回值：**
```javascript
{
  success: true,
  totalAccounts: number,
  successCount: number,
  totalTodayCount: number,      // 所有账户的 today 数据总数
  totalDailyStatsCount: number,  // 所有账户的按日数据总数
  results: Array
}
```

## 📊 数据流对比

### 修复前（语义污染）

```
滑动窗口同步：
  today + last_7d → 合并 → ad_snapshots
  
today 查询：
  ad_snapshots (synced_at 在今天的记录)
  → 可能包含 last_7d 聚合值 ❌
```

### 修复后（严格按自然日）

```
滑动窗口同步：
  today → ad_snapshots (只存 today 数据) ✅
  过去 N 天按日数据 → daily_stats (按日修复) ✅
  
today 查询：
  ad_snapshots (synced_at 在今天的记录)
  → 只包含 today 数据 ✅
```

  account_id,
  ad_id,

  spend,

**预期结果：**

## 🎯 总结

### 修复效果

1. **语义清晰**
   - `today` 查询只返回"今天"的真实数据
   - `last_X_days` 查询使用"历史多天 + 今天最后快照"合并后动态聚合
   - 迟到归因修复到正确的自然日

2. **数据一致性**
   - `ad_snapshots` 只存 today 数据（实时快照）
   - `daily_stats` 存按自然日的历史归档（修复迟到归因）
   - 查询路由与数据源严格对应

3. **性能优化**
   - 分布式锁防止多实例重复执行
   - 按日更新 `daily_stats`，避免重复数据
   - 幂等更新，支持多次执行
