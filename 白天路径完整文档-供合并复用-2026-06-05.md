### 5.4 重新设计：心跳"占槽"调度模型

核心思想：**15 分钟热数据同步是主旋律（默认任务）。双窗口归档（≥02:00 初步、≥12:00 深度对账）每天各占一个槽。归因按间隔占槽（7天/14天各一次，≥02:00/≥04:00，全量重拉）。其余槽全是热同步。**

```
每个账户每 15 分钟心跳触发一次（基于账户本地时间）
                        │
          ┌─────────────┼─────────────┐
          ↓             ↓             ↓
    双窗口归档检查    归因窗口检查   以上都不满足
    (基于daily_archive (基于daily_archive (默认路径)
     _status表判断)    _status表判断)      │
          │             │             │
          ↓             ↓             ↓
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ 双窗口归档 │  │ 全量归因  │  │ 热数据同步 │
    │≥02 ARCHIVED│  │ 7日或14日 │  │ Insights  │
    │≥12 FINALIZED│ │ →daily_stats│ │ First     │
    │每天各1次  │  │ 7日每7天  │  │ →ad_snap  │
    │           │  │ 14日每14天│  │ 默认每15分钟│
    └──────────┘  └──────────┘  └──────────┘
          │             │             │
          ↓             ↓             ↓
    下个15分钟      下个15分钟      下个15分钟
    恢复热同步      恢复热同步      继续热同步
```

#### 为什么这样设计

| 设计决策 | 理由 |
|----------|------|
| 归档双窗口：≥02:00 ARCHIVED（初步）、≥12:00 FINALIZED（深度对账） | 初步归档尽早让昨天数据进入 daily_stats 可用；深度对账补充 FB 归因延迟到达的转化数据 |
| 7日归因在本地时间 ≥02:00，**每 7 天一次，全量重拉** | 全量重拉避免新广告在历史日期上的数据被增量逻辑永久遗漏；每周一次控制 API 用量 |
| 14日归因在本地时间 ≥04:00，**每 14 天一次，全量重拉** | 极端延迟场景兜底，两周一次全量拉取可接受 |
| 各只占一个槽 | 归因全量拉取可能耗时较长，占一个槽跑完即恢复热同步 |
| **全量归因** | 每次触发时拉取完整 7 天/14 天窗口数据。全量拉取虽有少量重复 API 调用，但保证了数据完整性，消除了增量逻辑遗漏新广告的隐患 |
| **热同步默认路径含整点 3 天回补** | 当 `local.minute >= 0`（即本轮心跳为整点前后任意时刻）且最近一次 3 天回补距现在 ≥60 分钟时，执行 `daysBack=3` 滑动窗口回补。`>=` 而非 `===` 确保心跳被跳过时不会漏补 |

#### 归因的全量逻辑

```
不是: 记录上次覆盖日期、只拉增量（会遗漏新广告在已覆盖日期的历史数据）
而是: 每次触发时拉取完整 N 天窗口，用 INSERT ... ON DUPLICATE KEY UPDATE 写入 daily_stats，幂等安全

示例（7日归因，每7天触发一次，全量重拉）:
  Day1 凌晨 02:00: 距上次触发≥7天 → 拉 Day1-7 到 Day1-1（共7天全量）
  Day8 凌晨 02:00: 距上次触发≥7天 → 拉 Day8-7 到 Day8-1（共7天全量）
  Day15凌晨 02:00: 距上次触发≥7天 → 拉 Day15-7 到 Day15-1（共7天全量）
  ...

示例（14日归因，每14天触发一次，全量重拉）:
  Day1 凌晨 04:00: 距上次触发≥14天 → 拉 Day1-14 到 Day1-1（共14天全量）
  Day15凌晨 04:00: 距上次触发≥14天 → 拉 Day15-14 到 Day15-1（共14天全量）
  ...

对比增量方案的缺陷:
  增量: Day3 创建的新广告，Day1-2 的 spend 数据在 Day1 归因时未覆盖，Day8 增量只拉 Day2-7，Day1 永久遗漏 ❌
  全量: Day8 全量重拉 Day1-7，新广告的 Day1-2 数据被完整捕获 ✅
```

#### 完整调度伪代码

```javascript
// 每 15 分钟心跳对每个账户执行（基于账户本地时间）
function heartbeatSlotForAccount(account) {
  const local = getLocalTime(account.timezone)  // ⚠️ 账户本地时间，不是UTC
  const today = local.toFormat('yyyy-MM-dd')
  const yesterday = local.minus({ days: 1 }).toFormat('yyyy-MM-dd')

  // --- 从 DB 查询当天归档状态（复用 daily_archive_status 表）---
  const archiveRow = await queryArchiveStatus(account.id, yesterday)
  // archiveRow: { status: 'PENDING'|'ARCHIVED'|'FINALIZED', last_7d_attribution_date, last_14d_attribution_date }

  // --- 优先级 1：双窗口归档 ---
  //   ≥02:00 且状态=PENDING → 初步归档 ARCHIVED
  //   ≥12:00 且状态=ARCHIVED → 深度对账 FINALIZED
  const archResult = await checkAndExecuteArchive(account, local, archiveRow)
  if (archResult.archived || archResult.finalized) {
    return  // 归档完成，本槽结束，下个15分钟恢复热同步
  }

  // --- 优先级 2：7 日全量归因（本地时间 ≥02:00，每 7 天一次）---
  const daysSince7d = archiveRow.last_7d_attribution_date
    ? daysBetween(archiveRow.last_7d_attribution_date, today)
    : 999  // 从未执行过 → 立即触发
  if (local.hour >= 2 && daysSince7d >= 7) {
    const sinceDate = today.minus({days: 7})
    const untilDate = yesterday
    // ★ 全量重拉完整 7 天窗口
    await backfillDailyStats(account, { sinceDate, untilDate })
    // 更新归因触发日期到 DB
    await updateAttributionDate(account.id, yesterday, 'last_7d_attribution_date', today)
    return
  }

  // --- 优先级 3：14 日全量归因（本地时间 ≥04:00，每 14 天一次）---
  const daysSince14d = archiveRow.last_14d_attribution_date
    ? daysBetween(archiveRow.last_14d_attribution_date, today)
    : 999
  if (local.hour >= 4 && daysSince14d >= 14) {
    const sinceDate = today.minus({days: 14})
    const untilDate = yesterday
    // ★ 全量重拉完整 14 天窗口
    await backfillDailyStats(account, { sinceDate, untilDate })
    await updateAttributionDate(account.id, yesterday, 'last_14d_attribution_date', today)
    return
  }

  // --- 默认：15 分钟热数据同步 ---
  // Insights First → ad_snapshots
  // 附：整点前后（minute >= 0 任意时刻）且距上次 3 天回补 ≥60 分钟，补一次 last_3d
  if (shouldRun3dBackfill(account)) {
    await syncAccountSlidingWindow(account, daysBack=3)
    mark3dBackfillDone(account)
  } else {
    await syncTodayStats(account)
  }
}

// 判断是否执行 3 天回补（整点回补，>= 防止心跳被跳过时漏补）
function shouldRun3dBackfill(account) {
  const minutesSinceLast = account.last3dBackfillAt
    ? (Date.now() - account.last3dBackfillAt) / 60000
    : 999
  return minutesSinceLast >= 60  // 距上次回补 ≥60 分钟
}
```

#### 计时逻辑说明

```
归因不是"每天一次"，而是"每隔N天一次"：
  - 7日归因: 距上次执行 ≥7 天 → 触发（约每周一次），全量重拉完整 7 天
  - 14日归因: 距上次执行 ≥14 天 → 触发（约每两周一次），全量重拉完整 14 天

用 >= 而非 === 的原因：
  - 心跳可能因前轮未完成而跳过整点槽位
  - >= 配合 daysSince >= N 的计时判断，容错性最强
  - 即使连续跳过多天，只要条件满足就会在第一个可用槽执行

整点 3 天回补：
  - 不再使用 minute===0（可能被跳过的整点卡死）
  - 改用距上次 3 天回补 ≥60 分钟即触发（>= 逻辑）
  - 每次回补完成后记录时间戳，下次判断间隔

双窗口归档：
  - ≥02:00: PENDING → ARCHIVED（初步归档，数据尽早可用）
  - ≥12:00: ARCHIVED → FINALIZED（深度对账，补充归因延迟数据）
  - 状态通过 daily_archive_status 表持久化，服务器重启不丢失
```

#### 各时区归因实际触发时间

| 账户时区 | 7日归因（本地 02:00） | 14日归因（本地 04:00） |
|----------|----------------------|------------------------|
| Asia/Shanghai (+8) | UTC 18:00 | UTC 20:00 |
| America/Los_Angeles (-7) | UTC 09:00 | UTC 11:00 |

**每个账户按自己的本地时间触发，不同时区错开，不会扎堆。**

#### 与方案A 夜间滑动窗口的关系

**方案A 将被删除。** Insights First 已天然覆盖新广告发现，全量归因保证了历史数据完整性，不再需要独立的夜间嗅探通道。`syncAllAccountsSlidingWindow` 定时任务、`sync:sliding_window` 锁、以及相关的 `filterActiveAds` 调用一并移除。

### 5.5 合并后的最终架构

```
cronService.js (简化后)
│
├─ */15 * * * *  →  统一心跳（每15分钟）
│   └─ 遍历账户，按账户本地时间 + "占槽"模型调度:
│       ├─ 本地≥02:00 状态=PENDING        →  初步归档 ARCHIVED
│       ├─ 本地≥12:00 状态=ARCHIVED       →  深度对账 FINALIZED

│       ├─ 本地≥04:00 距上次≥14天         →  14日全量归因 (完整14天窗口)
│       └─ 默认                           →  热数据同步 (Insights First → ad_snapshots)
│                                           + 整点前后 3 天回补 (距上次≥60分钟)

├─ * * * * *     →  规则执行（每分钟，不变）
│
└─ ❌ 删除: 夜间滑动窗口 cron（方案A，被 Insights First + 全量归因替代）
    ❌ 删除: syncAccountSlidingWindow（被统一函数替代）
    ❌ 删除: filterActiveAds（不再需要）
    ❌ 删除: 嗅探修复代码（不再需要）
    ❌ 删除: sync:sliding_window 锁（不再需要）
```

```

  └──────────────────────────────────────────────────────────────────┘
```
