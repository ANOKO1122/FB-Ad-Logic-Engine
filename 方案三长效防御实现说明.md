# 方案三：长效防御实现说明

## 🎯 实现目标

按照 GPT5.2 的建议，实现"长效防御"机制：
- **规则必须绑定 accountId**（不允许 NULL/空字符串）
- **权限校验**：普通用户只能绑定自己负责的账户，管理员可以绑定任意账户
- **防止反向索引退化**：确保规则必须绑定账户，避免在所有账户上执行

---

## ✅ 已实现的修改

### 1. 路由层：POST /api/rules（创建规则）

**文件**：`server/routes/rules.js`

**修改内容**：
1. ✅ 导入 `pool`（用于权限校验）
2. ✅ 接收 `accountId` 字段
3. ✅ `accountId` 必填校验（不允许 NULL/空字符串）
4. ✅ 普通用户权限校验（accountId 必须属于该用户的负责人）
5. ✅ 管理员权限校验（账户必须存在且激活）
6. ✅ 传递给 `createRule` 函数

**关键代码**：
```javascript
// ✅ accountId 必填校验
if (!accountId || accountId.trim() === '') {
  return res.status(400).json({ 
    error: '缺少必填字段: accountId（请选择广告账户）',
    code: 'MISSING_ACCOUNT_ID'
  })
}

// ✅ 普通用户权限校验
if (!isAdmin) {
  if (!ownerId) {
    return res.status(400).json({ 
      error: '当前用户未绑定负责人(owner_id)，无法创建规则',
      code: 'MISSING_OWNER'
    })
  }
  const [rows] = await pool.execute(
    `SELECT 1 FROM account_mappings 
     WHERE fb_account_id = ? AND owner_id = ? AND is_active = 1 
     LIMIT 1`,
    [String(accountId), ownerId]
  )
  if (rows.length === 0) {
    return res.status(403).json({ 
      error: '无权访问该广告账户，无法创建规则',
      code: 'ACCOUNT_FORBIDDEN'
    })
  }
}

// ✅ 管理员权限校验
else {
  const [rows] = await pool.execute(
    `SELECT 1 FROM account_mappings 
     WHERE fb_account_id = ? AND is_active = 1 
     LIMIT 1`,
    [String(accountId)]
  )
  if (rows.length === 0) {
    return res.status(400).json({ 
      error: '广告账户不存在或未激活(account_mappings)，无法创建规则',
      code: 'ACCOUNT_NOT_FOUND'
    })
  }
}
```

3. ✅ `accountId` 必填校验（不允许 NULL/空字符串）
4. ✅ 普通用户权限校验
5. ✅ 管理员权限校验

**关键代码**：
```javascript
// ✅ 字段白名单（防止全透传）
const allowed = [
  'ruleName',
  'accountId',
  'conditions',
  'actions',
  'enabled',
  'targetLevel',
  'targetIds',
  'logicOperator',
  'timezoneName',
  'isSimulation'
]

const updates = {}
for (const k of allowed) {
  if (Object.prototype.hasOwnProperty.call(body, k)) {
    updates[k] = body[k]
  }
}

// ✅ 如果更新 accountId，必须做权限校验
if (Object.prototype.hasOwnProperty.call(updates, 'accountId')) {
  if (!updates.accountId || updates.accountId.trim() === '') {
    return res.status(400).json({ 
      error: '缺少必填字段: accountId（请选择广告账户）',
      code: 'MISSING_ACCOUNT_ID'
    })
  }
  
  // 权限校验（同 POST 规则）
```

## 🎯 预期效果

### 1. 防止反向索引退化

**优化前**：
- 规则 `accountId` 为 `null`
- 代码认为规则没有指定账户
- 仍然在所有36个账户上执行

**优化后**：
- 规则必须绑定 `accountId`（NOT NULL 约束）
- 代码只在实际需要的账户上执行
- 从36个账户减少到1个账户

---

### 2. 防止权限绕过

**优化前**：
- 普通用户可能创建规则时没有指定账户
- 或者指定了不属于自己的账户

**优化后**：
- 普通用户只能绑定自己负责的账户
- 管理员可以绑定任意账户，但账户必须存在且激活
- 路由层和数据库层双重校验

---

### 3. 防止字段遗漏

**优化前**：
- 后端路由没有接收 `accountId`
- 导致规则创建时 `accountId` 为 `null`

**优化后**：
- 后端路由强制接收 `accountId`
- 数据库层 NOT NULL 约束
- 双重保障，防止字段遗漏

## 📝 注意事项

### 1. 历史数据清洗

如果现有规则中有 `account_id` 为 NULL 的记录，需要先清洗：

```sql
-- 检查 NULL 值
SELECT COUNT(*) FROM rules WHERE account_id IS NULL;

-- 清洗数据（根据实际情况调整）
UPDATE rules 
SET account_id = 'act_xxx'  -- 替换为实际的账户ID
WHERE account_id IS NULL;
```

### 2. 前端配合

前端创建规则时，必须传递 `accountId`：

