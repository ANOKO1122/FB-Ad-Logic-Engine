项目问题与解决方案全景汇总（问题清单 + 根因 + 方案）
下面按“领域维度”把你这几个月踩过的坑系统化整理，方便做复盘报告和教学。
一、数据与指标口径相关问题
1.1 成效数据“7 倍”问题
现象：前端显示的购买次数是 Facebook Ads Manager 的 7 倍。
根因：
解析 actions 时用 includes('purchase')，把 offsite_conversion.fb_pixel_purchase、purchase、omni_purchase 等多个 action_type 全部累加。
解决方案：
在 extractPurchaseCount 中使用精确匹配 offsite_conversion.fb_pixel_purchase，并在必要时按优先级选择一个权威 action_type，而不是累加。
经验：
与第三方 API 对齐时，一定要找到“权威字段”，避免多个统计纬度叠加。
1.2 ROAS 口径不统一（方案 A 之前）
现象：
有时 spend>0、purchase_value=0 的广告 roas 被当成 0，有时为 null，规则行为难以预期。
根因：
写库时混合存了 API roas 和计算 roas，读侧又混合使用，导致不同路径出现不一致。
解决方案（方案 A）：
写侧：
ad_snapshots / daily_stats 不持久化派生 cpc/roas 等比值。
roas 列只存 API 兜底值或 null。
读侧：
单日：spend=0→roas=null；value>0→value/spend；否则若 API roas>0→用 API 值；再否则 roas=0。
多日：统一用 SUM(purchase_value)/SUM(spend)。
经验：
对比值类字段，写库只存“可校验原始量”（分子/分母）+ API 兜底；所有业务判断在读侧统一计算。
1.3 Route1 Insights 字段缺失导致 purchase_value 恒为 0
现象：Today 热同步走 Route1 时，purchase_value=0，但 Graph Explorer 查同一广告当天有 action_values。
根因：
getAdInsights() 的 fields 未包含 action_values/purchase_roas/website_purchase_roas。
解决方案：
补齐 fullFields，并在 fallbackFields 中保留 ROI 核心字段。
mapItem 透传这些字段给 Ingestor。
经验：
“今日心跳”如果走了一条不带 ROI 字段的路径，会直接污染所有后续决策。
1.4 冷数据归档 SUM 快照导致“翻倍”
现象：
同一日多次快照，归档 SUM 得出远大于实际的值。
根因：
把快照当“增量”去 SUM，而实际上快照本身就是“累计值”。
解决方案：
归档时只取每广告当日最后快照：
MySQL 8：ROW_NUMBER() OVER (PARTITION BY ... ORDER BY synced_at DESC)
兼容版：max(synced_at) + join 原表。
经验：
“快照”语义与“增量”语义完全不同，一旦混用，所有汇总都会错。
二、结构同步与限流相关问题
2.1 FB updated_time 过滤不可靠
现象：
/ads?filtering=updated_time > since 实测不符合预期，返回数据覆盖范围不稳定。
根因：
FB 官方在实际生产环境中 updated_time 过滤行为不稳定，不足以作为增量保证。
解决方案：
文档明确禁用这种增量作为主路径。
增量主策略改为：本地伪增量（recentActiveIds − activeAdIds）+ Piggyback + 全量轮转兜底。
经验：
外部系统的“增量 API”如果无法给出强 SLO，就不要作为主链路。
2.2 结构伪增量触发频率过高 / 永远打同一批
现象：
每 15 分钟同一批 200 个广告被伪增量反复查。
根因：
recentActiveIds 使用 last_synced_at DESC 限定 24h，upsert 后 last_synced_at 更新为 NOW()，下一轮仍是“最新的一批”。
解决方案：
增加“双窗口”：
24h 内候选 + 至少 1h 未同步；
ORDER BY last_synced_at ASC（最旧优先）。
NULL last_synced_at 特别放行，让异常/历史数据能被优先修复。
经验：
“最近活跃”与“最近被同步”是两个维度，伪增量要综合考虑。
2.3 全局限流缺失，结构任务抢占止损规则配额
现象：
Track2/结构全量与 Today/规则执行并发时，出现 User request limit reached，止损请求可能被延后或失败。
根因：
每类任务内部有自己的 p-limit，但没有全局队列保证“谁优先占用配额”。
解决方案：
引入全局请求队列 enqueueFacebookRequest({priority, label}, task)：
并发上限 FB_GLOBAL_REQUEST_CONCURRENCY。
优先级：action > today > track2 > cold。
超时 FB_GLOBAL_QUEUE_TIMEOUT_MS。
经验：
多条定时任务 + 手动执行时，必须由一个“统一调度器”控制对外吞吐。
三、时区与窗口相关问题
3.1 Today 查询“返回空数组”
现象：
/api/rule-data?time_window=today 返回空，但 DB 中有当天快照。
根因：
数据同步按 Asia/Shanghai 写 data_date，账户时区/查询逻辑按 UTC 计算“today”，导致窗口错位。
另有 adIds IN (null) 的 SQL 构造问题。
解决方案：
所有时间字段存 UTC，账户时区在计算窗口时使用。
查询时不再构造 IN (null) 过滤；adIds 为空时不加过滤。
文档与 SQL 检验配套（CONVERT_TZ + 测试）。
经验：
“写数据使用一个时区，读数据又假设另一个时区”是时区 bug 的根源。
3.2 滑动窗口语义污染导致 Today 混入 last_7d
现象：
today 查询结果含有 last_7d 聚合值。
根因：
滑动窗口同时拉 today + last_7d 封装入 ad_snapshots。
解决方案：
严格分层：
ad_snapshots 只存 today；
daily_stats 只存按自然日聚合；
last_7_days 查询：daily_stats + today 合并。
经验：
不同时间粒度的数据必须分库/分表，在查询层再合并。
四、规则引擎与多账户相关问题
4.1 target_by_account 空数组账户仍参与执行
现象：
target_by_account 为 { act_1: [...], act_2: [] } 时，act_2 仍会被加锁并跑 evaluateRule。
根因：
getRuleAccountIds / ruleAppliesToAccount 没有把“空数组”视为“不适用”。
解决方案：
getRuleAccountIds 只返回 value 非空数组的账户。
ruleAppliesToAccount 遇到空数组账户直接返回 false。
经验：
在多账户设计中，“是否适用”最好由一个统一函数管理，避免逻辑散落。
4.2 单条执行与调度执行口径不一致
现象：
手动运行某条规则时会执行排除名单中的广告，定时执行时不会。
根因：
单条执行走旧 evaluateRule(insights, ads) 路径，没走 Dispatcher 及 dynamicScope 快照逻辑。
解决方案：
POST /api/rules/:id/execute 改为始终走 RuleEngineDispatcher + dynamicScope。
经验：
“手动执行”和“定时执行”一旦有不同代码路径，迟早会出现不一致行为。
五、动作执行与预算相关问题
5.1 预算被错误地设成 50 美元
现象：
规则期望从 30 调整到 33，但实际变成 50。
根因：
把 FB GET 回来的 daily_budget 当美元再乘 100，导致 3000 分被当成 300000 分。
解决方案：
读写预算统一为“分”：
GET：直接把 FB 返回值视作分；
计算：用分计算；
POST：仍用分。
经验：
外部 API 文档中单位约定非常关键，尤其是金额相关。
5.2 冷却期内预算规则仍被每分钟执行
现象：
预算规则执行后，日志每分钟仍出现 GET 预算/Pre-Flight。
根因：
冷却表只写 budget_campaign:/budget_adset:，调度过滤查的是 ad:adId，查不到冷却记录。
解决方案：
双写：写入 (ruleId, cooldownKey) 和 (ruleId, 'ad:'+adId)。
经验：
冷却维度与执行维度分离时，必须额外写一条“广告级别”的冷却键。
六、动态筛选与多账户相关问题
6.1 Oversize 清空快照破坏“上一次可用结果”
现象：
动态筛选命中超上限（大于 max_dynamic_matches）后，后续执行无快照可用。
根因：
刷新逻辑是“先删后建”，哪怕 ERROR_OVERSIZE 也先把旧快照清空。
解决方案（策略 B）：
只对 NORMAL 更新快照；ERROR_* 规则仅更新状态，不删数据。
经验：
在预计算缓存类场景，“错误时保留上一版结果”要写成硬约束。
6.2 多账户规则只刷新主 accountId
现象：
规则多选账户后保存，仅主账户触发动态刷新，其它账户不刷新。
根因：
TriggerB/TriggerC 都按 accountId 设计，未按“规则涉及所有账户”展开。
解决方案：
新增解析函数：从 target_by_account / target_account_ids / account_id 解析完整账户集；
TriggerB：规则保存后对所有账户逐个防抖刷新；
TriggerC：仅传 ruleId 时自动刷新全部账户。
经验：
多账户语义必须在“保存 → 刷新 → 执行 → 日志”四个链路都统一。
七、结构镜像与选择器相关问题
7.1 选择器仍穿透调用 FB 导致慢且易限流
现象：
RuleManager 打开结构列表时会直接调用 FB /ads，速度慢且配额消耗明显。
根因：
历史版本选择器直接调 FB，无结构镜像。
解决方案：
建立 structure_* 三张表；
所有列表/搜索/翻页/resolve 完全查库，不穿透 FB；
手动“刷新列表”按钮调用结构同步 API，作为唯一触达 FB 的入口。
经验：
UI 列表应尽量读本地镜像，避免每次交互都打外部 API。
八、日志与可观测性相关问题
8.1 只看 execution_logs 不知道「规则有没有被评估」
现象：
有些规则“不动”，但无法区分是“未匹配”还是“未执行”还是“账号权限不符”。
解决方案：
引入 rule_execution_summaries 表：
每条规则每轮写一行摘要，带 matched_count/executed_count/skip_reason。
经验：
将“系统层可观测性”（summaries）与“审计算级日志”（automation_logs）区分，有助于排查和教学。
8.2 TLS/代理错误掩盖实际成功
现象：
日志中频繁出现 ❌ TLS连接错误，但业务上数据已写入成功。
根因：
SOCKS5/TLS 流程中 body 收完后连接关闭触发 error。
解决方案：
在 TLS error handler 中先判断 body 是否已收完整且 JSON 解析成功，若是则视为成功，仅记录 debug。
经验：
日志层要区分“真正的错误”和“成功后的噪音”，否则会引导误判。
九、权限与执行入口相关问题
9.1 Data API IDOR 与 execute-all 越权风险
现象：
带任意 account_id 调 /api/ads、/api/insights 等接口，在没有统一 assertAccountAccess 的情况下存在 IDOR 风险。
解决方案：
所有数据接口统一调用 assertAccountAccess(user, accountId)：
admin 放行；
staff 必须在 account_mappings 中有 is_active=1 的映射，否则 403。
Health 接口与 execute-all、单条执行同样按 owner_id 过滤。
经验：
鉴权应该集中在一个函数里，并在所有入口复用。
9.2 执行中重复触发规则导致竞态
现象：
用户连点“立即运行所有规则”，产生并发执行，依赖锁兜底导致复杂日志。
解决方案：
使用全局 runningCount 控制执行状态；执行中再次请求直接返回 409/ALREADY_RUNNING。
经验：
对于重型任务，“正在执行”这一状态必须被暴露出来并受到保护。
