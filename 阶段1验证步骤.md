## 📋 验证目标

验证阶段1的修复是否正确：
1. 权限控制统一（summary/list/detail 都使用 `users.owner_id` 过滤）

3. `include_all_status` 参数只对 admin 生效

### 1. 准备测试数据

确保数据库中有以下测试数据：

```sql
-- 查看当前用户和 owner 关系
SELECT u.id AS user_id, u.username, u.role, u.owner_id, o.owner_name
FROM users u
LEFT JOIN owners o ON u.owner_id = o.id
WHERE u.status = 'active';

-- 查看 automation_logs 数据分布
SELECT 
  al.owner_id,

  COUNT(*) AS count

```

## ✅ 验证步骤

### 验证1：权限验证（summary 接口）

**目标**：验证非 admin 用户只能看到自己 owner_id 的统计

**步骤**：
1. 使用非 admin 账号登录，获取 token
2. 调用 `GET /api/automation-logs/stats/summary`
3. 检查返回的统计是否只包含该用户 owner_id 的日志

**测试命令**（使用 curl 或 Postman）：
```bash
# 替换 YOUR_TOKEN 为实际 token
curl -X GET "http://localhost:3000/api/automation-logs/stats/summary" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

**预期结果**：
- 返回的 `today` 和 `week` 统计只包含该用户 owner_id 的日志
- 如果该用户 owner_id = 1，则统计中不应该包含 owner_id = 2 的日志

**验证 SQL**（手动验证）：
```sql
-- 假设测试用户 user_id = 2, owner_id = 1
-- 应该只统计 owner_id = 1 的日志
SELECT 
  COUNT(*) AS total,
  SUM(CASE WHEN status = 'success' THEN 1 ELSE 0 END) AS success_count,
  SUM(CASE WHEN status = 'fail' THEN 1 ELSE 0 END) AS fail_count,
  SUM(CASE WHEN status = 'skipped' THEN 1 ELSE 0 END) AS skipped_count
FROM automation_logs
WHERE owner_id = 1  -- 用户的 owner_id
  AND DATE(triggered_at) = CURDATE();
```

### 验证3：权限验证（详情接口）

**目标**：验证非 admin 用户只能查看自己 owner_id 的日志详情

**步骤**：
1. 使用非 admin 账号登录
2. 获取一条属于该用户 owner_id 的日志 ID（例如：log_id = 123）
3. 调用 `GET /api/automation-logs/123`
4. 检查是否能正常返回
5. 尝试访问一条不属于该用户 owner_id 的日志 ID
6. 检查是否返回 404 或权限错误

**测试命令**：
```bash
# 测试1：访问自己的日志（应该成功）
curl -X GET "http://localhost:3000/api/automation-logs/123" \
  -H "Authorization: Bearer YOUR_TOKEN"

# 测试2：访问其他 owner_id 的日志（应该失败）
curl -X GET "http://localhost:3000/api/automation-logs/456" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

**预期结果**：
- 自己的日志：返回 200，正常显示详情
- 其他 owner_id 的日志：返回 404 "日志不存在或无权访问"

### 验证4：展示层过滤验证（列表接口 - 非 admin）

**目标**：验证非 admin 用户默认只看 success，且 `include_all_status` 参数不生效

**测试用例1**：非 admin 用户，不传参数
```bash

  -H "Authorization: Bearer NON_ADMIN_TOKEN"
```
**预期**：只返回 `status=success` 的日志

**测试用例2**：非 admin 用户，传 `include_all_status=true`

```bash
  -H "Authorization: Bearer NON_ADMIN_TOKEN"
```
**预期**：仍然只返回 `status=success` 的日志（参数不生效）

**测试用例3**：非 admin 用户，传 `include_all_status=true&status=fail`
```bash
curl -X GET "http://localhost:3000/api/automation-logs?include_all_status=true&status=fail" \
  -H "Authorization: Bearer NON_ADMIN_TOKEN"
```
**预期**：仍然只返回 `status=success` 的日志（参数不生效）

## 🔍 手动 SQL 验证

如果 API 测试不方便，可以直接用 SQL 验证逻辑：

```sql
-- 验证1：检查某个用户的 owner_id
SELECT id, username, role, owner_id FROM users WHERE id = ?;

-- 验证2：检查该用户应该看到的日志（owner_id 匹配）
SELECT COUNT(*) FROM automation_logs WHERE owner_id = ? AND status = 'success';

-- 验证3：检查该用户不应该看到的日志（其他 owner_id）
SELECT COUNT(*) FROM automation_logs WHERE owner_id != ?;

-- 验证4：检查不同状态的日志分布
SELECT status, COUNT(*) FROM automation_logs WHERE owner_id = ? GROUP BY status;
```

## ⚠️ 注意事项

1. **测试前确保有足够的数据**：至少要有不同 owner_id 的日志，以及不同 status 的日志
2. **测试 token 有效性**：确保测试时使用的 token 未过期
3. **测试环境隔离**：建议在测试环境进行，避免影响生产数据
4. **记录测试结果**：建议将测试结果记录在上面的表格中，便于追踪问题

## 🐛 常见问题排查

### 问题1：非 admin 用户能看到其他 owner_id 的日志
**排查**：
- 检查 `users.owner_id` 是否正确查询
- 检查 SQL 查询条件是否正确应用

### 问题2：include_all_status 参数不生效
**排查**：
- 检查用户 role 是否为 'admin'
- 检查参数值是否为字符串 'true'（注意大小写）
- 检查代码逻辑是否正确

### 问题3：列表接口返回了 fail/skipped 日志
**排查**：
- 检查是否传了 `include_all_status=true`（且用户是 admin）
- 检查默认过滤逻辑是否正确应用
