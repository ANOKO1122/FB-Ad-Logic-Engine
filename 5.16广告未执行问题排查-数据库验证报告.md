## 四、根因分析

### 4.1 直接原因：`outside_window` 状态错误地重置了 24h 冷却时钟

关键代码在 `server/services/ruleExecutionStateService.js` 的 `upsertRuleAdExecutionStateBatch` 函数：

```sql
INSERT INTO rule_ad_execution_state (rule_id, scope_key, last_executed_at, last_status, ad_id)
VALUES (?, ?, UTC_TIMESTAMP(), ?, ?)
ON DUPLICATE KEY UPDATE
  last_executed_at = VALUES(last_executed_at),    -- ❌ outside_window 也更新此字段！
  last_status = VALUES(last_status)
```

**`outside_window` 本意是"暂时不执行，等下次再说"，但代码把它和 `success`/`fail` 一样处理——都更新 `last_executed_at`，都触发 24h 冷却。**

### 4.2 触发条件：规则创建时间在窗口外 + 20 分钟极窄窗口

```
规则创建: 2026-05-13 16:32 北京时间
执行窗口: 00:00~00:20 北京时间（仅 20 分钟！）
冷却间隔: 1440 分钟（24 小时）
```

**规则在 16:32 创建，距离最近的执行窗口还有约 7.5 小时。创建后 1 分钟内 cron 触发首次评估，16:33 在窗口外 → 所有 8 个作用域同时被打上 `outside_window` 标记，`last_executed_at` 锁死在 16:33。**

### 4.3 死锁机制推演

```
Day 1 16:33  首次评估 → outside_window → last_executed_at = 16:33（冷却开始）
Day 1 16:34  冷却未到期（1分钟 < 1440分钟）→ 跳过
...          （每分钟都因冷却跳过，直到 24h 后）
Day 2 16:33  冷却刚好到期 → 评估 → 16:33 不在 00:00~00:20 → outside_window
             → last_executed_at 重置为 16:33 → 又是 24h 冷却 ...
Day N 16:33  永远死锁 ♾️
```

**因为执行窗口只有 20 分钟，冷却刚好 24 小时，只要首次评估不在窗口内，冷却到期的时间点也必然不在窗口内，形成完美闭环死锁。**

### 4.4 为什么有 2 个作用域能正常工作？

那两个成功的作用域（`budget_campaign:120244980370650600` 和 `budget_campaign:120244116579330600`）**在规则首次评估（16:33）时不在系统中**——它们的广告系列数据是后来才进入数据库的，首次评估恰好在 00:03~00:05（窗口内），因此逃过了死锁。

### 4.5 附带 Bug：`budget_adset` scope key 使用了 ad_id

```
budget_adset:120244468210020600  ← 120244468210020600 是广告ID，不是广告组ID！
```

问题广告的真实广告组 ID 是 `120244468210010600`，但 `resolveBudgetTargetContext` 在某些路径下取了错误的 ID 作为 adset_id，导致 scope key 与实际广告组不对应。

## 五、回答：这个规则该怎么创建？

### 5.1 问题本质

```
规则创建时间（16:32 BJ） ≠ 执行窗口（00:00~00:20 BJ）
        ↓
  首次评估落在窗口外 → outside_window → 24h 冷却重置 → 永久死锁
```

### 5.2 正确的做法

**方案 A（当前机制下的 Workaround）：在目标执行窗口内创建/启用规则**

- 比如在 **00:00~00:20 北京时间** 期间创建规则
- 这样首次评估就在窗口内 → `success` → 冷却时钟对齐窗口 → 后续每天同一时间正常执行
- ⚠️ 注意事项：确保数据已就绪（数据同步每 15 分钟一次，建议在窗口的后半段如 00:10~00:15 创建）

**方案 B（推荐）：修复代码 Bug，`outside_window` 不应更新 `last_executed_at`**

- `outside_window` 的含义是"现在不在执行时间，等会再试"
- 应该用短冷却（如 5 分钟）或**不设冷却**，让作用域每分钟重试，直到进入执行窗口
- 这样无论规则什么时候创建，所有作用域都能在下一个执行窗口正常触发

**方案 C（配置层面）：加宽执行窗口**

- 如果业务允许，将执行窗口从 20 分钟扩大到 1-2 小时，降低对齐失败的概率
- 但这只是降低概率，不能根治（如果冷却到期时间恰好卡在窗口外仍会死锁）

### 5.3 推荐组合方案

| 措施 | 类型 | 优先级 |
|------|------|--------|
| 修复 `outside_window` 冷却逻辑（不更新 `last_executed_at`） | 代码修复 | 🔴 P0 |
| 在窗口内创建/启用规则 | 操作规范 | 🟡 P1 |
| 加宽执行窗口（20分钟 → 1小时） | 配置建议 | 🟢 P2 |

## 六、修复建议

### 6.1 临时修复（立刻让卡死的作用域恢复）

```sql
-- 将规则 1179 所有 outside_window 记录的 last_executed_at 设为很久以前
-- 这样下次 cron 评估时冷却全部到期，有机会在窗口内重试
UPDATE rule_ad_execution_state
SET last_executed_at = '2020-01-01 00:00:00'
WHERE rule_id = 1179 AND last_status = 'outside_window';
```

⚠️ **执行后需等到下一个 00:00~00:20 窗口（即今晚凌晨），所有作用域会重新评估。**

### 6.2 根本修复（代码改动）

修改 `server/services/ruleExecutionStateService.js` 中 `upsertRuleAdExecutionStateBatch` 函数，让 `outside_window` 状态使用短冷却（如 5 分钟）而不是规则配置的完整冷却：

```js
// outside_window 应使用短冷却而非完整冷却，避免死锁
if (lastStatus === 'outside_window') {
  // 方案1：不更新 last_executed_at（保留原来的时间）
  // 方案2：用一个较短的冷却（如 5 分钟），让作用域能快速重试
}
```

或更根本地，在 `server/services/cronService.js` 执行窗口检查处（约第 930-940 行），对 `outside_window` 使用独立的冷却逻辑，不依赖 `rule_ad_execution_state` 的 `last_executed_at`。
